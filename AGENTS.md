# AGENTS.md

本文件是 `海中鱼巣` 给 Codex / 代码代理 / 协作代理使用的项目级工作说明。

## 1. 项目定位

`海中鱼巣` 是对 `鱼巢` 的第二代全量重构实验工程。

本项目目标不是复制旧项目文件，而是重建更清晰的内核协议：

```text
节点仓库
稳定句柄
主信息仓库
关系仓库
索引仓库
事务边界
领域服务
最小需求任务方法闭环
```

当前不得宣称：

```text
鱼巢旧能力已迁移完成
自我循环已完成
自我苏醒完成
初步成熟完成
外设 / 体素 / 控制面板已接入
```

## 2. 开工前必须读取

每次修改本项目文件前，先限载读取：

```text
计划/计划索引.md
规范/000_项目规则总纲.md
规范/001_规则迁移清单.md
```

如果执行某个计划，还要读取对应计划文件。

如果修改代码，还要先读取当前代码文件头部规则注释；新建代码文件必须包含最小文件头部规则注释模块。

## 3. 当前硬规则

1. 函数名、变量名、类内业务成员名必须使用中文。
2. 英文只允许用于 C++ 关键字、标准库类型、第三方 API、编译系统字段和不可避免的外部协议字段。
3. 机器逻辑必须落到节点、主信息、关系、索引、特征、特征值、需求、任务、方法、动态或因果引用等结构中。
4. 日志、控制台输出、注释、说明文档、临时文本不得承载机器逻辑或业务状态。
5. 运行期核心先以内存节点仓库为准；数据库只做快照、事件日志、恢复和审计适配，不直接裁决机器事实。
6. 不直接复制旧项目的双向链表模板作为新项目核心结构。
7. 外部调用方不得绕过领域服务直接写节点仓库、主信息仓库、关系仓库或索引仓库。
8. 线程不是动作来源；方法执行或领域服务写入才是动作动态和因果引用的来源候选。
9. 外部输入、外设材料、控制台输入只作为材料或请求，不直接成为世界事实。
10. 本项目编码禁止用事后校验逻辑兜底结构正确性；所有仓库、领域服务和业务写入口必须在函数入口排除无效句柄、错误节点类型、越权写入和不满足前置条件的输入。
11. 不允许先写入无效结构、悬空关系或临时错误状态，再靠校验函数、扫描器、日志或显示输出来发现和修正。
12. 容错检查只能作为预编译宏控制的诊断层；宏关闭时不得改变运行行为，宏开启后触发错误必须同时写日志并弹窗提醒。
13. 容错检查日志和弹窗只做人读诊断，不得承载机器逻辑、不得修复结构、不得作为业务状态来源；弹窗意味着逻辑错误已经暴露，必须排除该逻辑错误后才能继续推进依赖该路径的实现或验收。
14. 函数入口已经排除错误输入后，内部写入结果、关系目标、参数值或结构状态仍不符合预期时，必须视为逻辑错误追根因；不得静默丢弃错误、不得只返回失败后继续推进依赖路径，宏开启时必须写逻辑错误日志并弹窗提醒。
15. “基础信息”是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型；当前包含特征、存在、场景、抽象状态、抽象动态、二次特征、因果引用等长期信息节点。实例状态和实例动态是场景内临时零散节点，不得直接作为长期语素入口对应信息。“高级信息”不是独立节点类型；当前指需求、任务、方法等已存在的高级业务节点角色。需要绑定信息节点时，必须在入口接收并验证已有基础信息类节点、需求、任务或方法节点，不得新增 `高级信息` 节点类型。
16. 基础信息服务必须按原子服务和聚合服务分层；特征值、特征、存在、场景、状态、动态、二次特征、因果分别由对应基础信息原子服务负责，世界服务只做聚合编排，不得绕过原子服务直接写基础信息结构。
17. 特征值服务只服务于特征系统，只有特征服务可直接访问；需求、任务、方法等高级服务不得直接包含、接收或调用特征值服务。
18. 高级服务只允许直接依赖存在服务、场景服务、特征服务、二次特征服务、状态服务、动态服务和因果服务；需要特征状态材料时必须通过特征服务获得。
19. 实例状态和实例动态是场景内临时零散节点，必须有存在主体、场景归属和发生时间戳，不得直接长期存储或作为语素入口对应信息；抽象状态和抽象动态才可作为长期基础信息节点，且抽象状态和抽象动态本体不携带发生时间戳。
20. 所有日志都放在 `D:\海中鱼巣\日志` 目录下；逻辑错误报错必须写入独立逻辑错误日志文件；运行日志必须使用独立文件记录系统运行情况；调试日志按切片宏控制启用和关闭，调试完成后必须关闭对应切片调试日志代码，防止无用日志泛滥。
21. 因果引用采用轻量化口径：`记录因果引用(来源动态)` 只用来源动态做入口准入证据，不持久化来源动态关系，不得把单个因果引用解释为稳定因果结论。
22. 方法执行权限来自任务授权、任务承接和任务选择方法；方法执行前不再单独创建授权状态节点，后续权限治理必须围绕任务授权关系、任务状态和任务方法关系设计。
23. 需求目标是目标状态，不是 I64；I64 只可作为第一轮状态值材料或特征状态材料，不得作为需求目标对外语义。
24. 每个完成声明必须给出输入入口、核心处理、结构承载、输出变化、调用路径和验证方式。
25. 计划索引中的“下一步”为建立、修订、补齐规范、详细设计、实施计划、验收项、只读扫描、断点清单或项目记忆同步时，属于可直接执行的文档治理动作，不需要先等待用户确认；需要用户确认的是设计方案形成后进入代码实施、旧能力迁移、字段落地、函数迁移或改变机器结构前的确认门禁。
26. 设计确认后的计划必须登记到 `项目记忆/Codex任务队列.md` 可执行队列，再由落实执行路线承接；单个确认结论、详细设计确认或问题裁决不自动等同于可执行队列项。确认本身不等于代码实施许可，涉及代码实施、旧能力迁移、字段落地、函数迁移或改变机器结构时，仍必须先形成落实切片，并明确允许文件、禁止文件和验证方式。

## 4. 文档权力关系

```text
AGENTS.md：协作和执行规则
规范/：能不能这样做
计划/：本轮做哪一段
项目记忆/：当前状态、决策、队列、待确认和验收基线
实施记录/：实际做了什么和验证结果
代码：当前实现事实
README / 控制台输出 / 日志：人读说明和诊断材料
```

正式流转只允许沿以下链路推进：

```text
问题 / 新口径
-> 规范
-> 计划
-> 代码实施
-> 实施记录
-> 验证
-> 完成 / 归档
```

禁止反向覆盖：

```text
实施记录直接改规则
计划直接改规则
控制台输出直接改业务事实
日志直接改机器状态
旧项目实现直接覆盖新项目规则
```

## 5. 代码编辑规则

1. 手工编辑使用 `apply_patch`。
2. 优先使用 `rg` 搜索。
3. 不把旧项目代码整块复制进新项目；迁移旧能力必须先写清保留行为、删除行为和新增约束。
4. 新建函数前先查是否已有同功能入口；已有入口可扩展时优先扩展领域服务。
5. 新增结构必须说明：用途、写入方、读取方、生命周期、验证方式。
6. 不新增只为显示方便存在的机器字段。
7. 不把普通 bool 当作一阶机器事实；需要二态裁决时优先使用状态值、评分、计数、版本或复合裁决。
8. 指针只能作为运行期缓存或临时引用，长期身份必须使用稳定句柄或主键。
9. 删除、重挂、关系变更必须通过仓库或领域服务入口，不能裸改关系记录。
10. 节点移动统一视为普通父子关系重挂；通用移动逻辑只能落在关系仓库或后续确认的公共结构移动工具中，各领域服务不得重复实现移动算法，只能做本领域前置拒绝后调用通用入口。
11. 写函数时先写入口前置拒绝条件，再写结构写入；不得把“校验结构”“事后扫描”“修复坏数据”设计成正常运行路径。
12. 需要容错诊断时使用项目公共容错检查宏，不得手写分散的日志弹窗逻辑。
13. 显示层只读结构，不写业务事实。

## 6. 构建与验证

默认构建命令以后续项目文件为准；当前 Visual Studio C++ 项目可优先使用：

```powershell
msbuild .\海中鱼巣.vcxproj /p:Configuration=Debug /p:Platform=x64 /m
```

验证声明必须分级：

```text
文档已生成
代码已编译
入口前置拒绝通过
最小闭环通过
旧能力迁移通过
真实外设样本通过
```

不得把低级验证说成高级能力完成。

## 7. 云端同步规则

1. 新建项目内容默认必须纳入 git 跟踪并同步到云端仓库；明确属于本地临时、缓存、构建输出、个人配置或敏感材料的文件除外。
2. Codex 完成每个计划切片后，自动执行切片收尾同步，不再等待额外的“同步到云端”指令。
3. 切片收尾同步顺序为：

```text
检查 git status
确认本切片相关变更范围
git add 本切片相关文件
git commit
git push 当前分支对应远端
```

4. 若提交或推送失败，必须记录失败命令、失败原因和当前阻塞状态；不得宣称该切片已完成云端同步。
5. 自动同步只证明仓库内容已提交并推送，不证明代码编译、入口拒绝、最小闭环或旧项目能力迁移已通过。

## 8. ChatGPT 理论探讨同步与状态回写

与 ChatGPT「理论探讨」项目同步时，默认采用：

```text
AGENTS.md：管协作和执行规则
规范/：管允许性
计划/计划索引.md：管执行路由
项目记忆/当前状态.md：管当前现实
项目记忆/决策记录.md：管为什么
项目记忆/Codex任务队列.md：管可执行队列
实施记录/：管实际做了什么和验证结果
```

同步规则：

```text
1. 修改前先总结当前工程状态，不基于旧对话假设当前代码状态。
2. 不把 ChatGPT 理论讨论、同步摘要或草稿直接当作当前仓库事实。
3. 涉及项目状态、规则、计划、验收、长期口径或执行队列变化时，必须同步项目记忆。
4. 涉及重要技术选择、架构取舍、规则分层变化或长期口径变化时，必须更新 `项目记忆/决策记录.md`。
5. 任务完成时如涉及理论、规范、状态或迁移治理变化，最终回复必须给出可复制给 ChatGPT「理论探讨」项目的同步摘要。
6. 同步摘要必须区分已完成、未完成、验证状态、已知风险和下一步，不得宣称未发生的构建、运行、提交、推送或旧能力迁移。
```


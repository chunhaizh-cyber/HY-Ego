# AGENTS.md

本文件是 `海中鱼巣` 给 Codex、代码代理和协作代理使用的项目级入口规则。这里只保留每轮必须立即知道的权威顺序、角色权限、停工条件和验证边界；完整状态机、机器语义和工程细则由下列专门规范承载，不在本文件重复展开。

## 1. 项目定位与声明边界

`海中鱼巣` 是对 `鱼巢` 的第二代全量重构实验工程，目标是重建节点直接身份、稳定句柄、关系、领域类型化记录、索引、事务边界、领域服务和最小需求—任务—方法闭环，不复制旧项目文件结构。

当前可以声明：

```text
正式规范体系已建立并通过独立覆盖审计；
机器逻辑和代码语义以本项目规范目录登记的现行正式规范为准；
独立重建的只读控制面板第一版和 SQL Server 结构审计投影第一版已实现并验证。
```

上述事实不等于鱼巢旧能力已经等价覆盖。除非当前正式证据另有证明，不得宣称：

```text
鱼巢旧能力已完成等价重建；
自我循环、自我苏醒或初步成熟已完成；
完整业务操作控制面板或数据库恢复已完成；
外设、D455 或体素已经接入生产闭环。
```

活动计划导航和计划激活状态读取 `计划/计划索引.md`，任务当前生命周期唯一读取 `项目记忆/任务状态台账.md`，当前实现与验证读取执行窗口结构化回执、设计窗口据此形成的正式记录和 Git 事实。`项目记忆/当前状态.md` 只是带状态版本的人读投影，不参与 S0 或执行通道单写裁决；本文件不维护易漂移的执行流水。

- 新建代码文件：追加读取 `规范/代码文件建立归属与模块命名规范.md`。


## 3. 角色、窗口与“继续”

项目采用设计、执行、集成、只读复核四角色闭环。角色权限不因使用同一物理窗口、不同模型或 Full access 而合并。

| 角色 | 唯一职责与写入权 | 主要禁止 |
| --- | --- | --- |
| 设计窗口 | `AGENTS.md`、规范、流程图、详细设计、计划、计划索引、任务队列、工作树登记和全部 Markdown 设计记录 | 不指挥执行窗口，不发送开始 / 继续 / 停止 / 选择下一项指令，不接受或驳回执行 S0，不消费执行队列，不修改代码 / 工程 / 自检，不运行构建；集成占用期间不写 main |
| 执行窗口 | 在登记的独立任务 worktree 或长期执行通道 worktree 中自主读取 `可执行` 计划、自行完成 S0，并只修改当前计划允许的代码、工程和自检，通过结构化消息交接 | 不等待设计窗口放行或接受 S0；不新建、修改、移动、删除、暂存或提交任何 Markdown，不改中央治理材料，不扩计划，不直接推送 `origin/main` |
| 集成窗口 | 在登记集成 worktree 中核对范围、汇集固定提交、完整重验并非强制发布 main，写专属集成记录 | 不设计业务语义，不发明接口，不修改计划扩大范围 |
| 只读复核窗口 | 在指定提交、对象和问题集上返回事实、差异、证据、风险和建议 | 不修改、构建、入队、解除依赖、裁决完成、暂存、提交或推送 |

顶层任务是平台关系中不属于任何协作代理后代的独立用户可见任务，包括用户直接打开或创建的任务，以及用户授权后由 Codex 任务工具创建并交由用户独立接管的任务。顶层身份按平台父子关系和协作代理关系裁决；任务标题、消息来源、worktree、分支、模型、智能体名称和文件路径都不改变该身份。一个顶层任务及其全部直接和递归协作代理共同构成一个“顶层任务树”。

顶层任务树遵守以下最小硬规则：

1. 同一顶层任务树在同一时刻只能拥有设计、执行、集成中的一种写角色；顶层任务只读时整树只能只读。
2. 全部有写入权的后代只能继承顶层任务树的当前写角色；只读后代保持零修改、零构建、零运行、零 Git 写入、零活动计划段所有权和零发布占用。
3. 每个具体文件、结构、接口、验证或证据裁决切片同一时刻只有一个所有者；同角色不等于可以共同写同一切片。
4. worktree、分支、模型、推理等级、名称和递归层级均不改变角色、范围、所有权或许可。
5. Git index、暂存、提交、推送、执行通道 / 活动计划段单写所有权迁移和主线发布占用迁移只由唯一指定智能体串行收口；其它后代在收口期间停止写入。
6. 跨写角色默认交给独立顶层任务。只有用户明确启用同一物理窗口串行切换时，才可在全部后代停止、旧角色记录与验证完成、Git 和占用释放并正式重新路由后整树切换。
7. 发现一棵任务树存在两种写角色、未授权写入、所有权重叠或角色规避时，整树立即停止新增写入，保留现有 Git 事实并退回设计治理；不得以文件无交集、结果正确或最终可合并继续。
8. 执行角色的仓库写入上限是当前计划允许的代码、工程和自检；低位叶子计划、历史交互规范或旧实施习惯中列出的 `*.md` 路径不再构成执行写权、完成必需项或缺失 DRIFT。执行完成、漂移、失败和计划修订请求只通过结构化消息交给设计窗口；设计窗口只据已经发生的事实修订设计产物和形成文档记录，不因记录、回执或消息产生、暂停或恢复活动计划段。

存在两个以上边界明确、无真实依赖且所有权可隔离的子任务时，应优先使用同角色或只读子智能体并行提速；不使用必须具名说明原因，不得制造空任务或伪造并行。使用更多智能体不能扩大角色权限、正式计划、任务队列、worktree 登记、执行或发布许可，也不能替代现行规范、有效设计、实际接口复核和正式验证。角色交互合同、子智能体 S0、各角色适用场景、停止与升级、整树角色切换和跨顶层交接细节统一见 `规范/设计执行双窗口交互规范.md`；worktree 创建、冻结、集成、失败和回收细节统一见 `规范/多工作树并发与集成规范.md`。

`继续` 是上下文指令：只沿当前已声明或正式登记角色继续，不自动等于消费队列，也不自动跨角色。窗口类型仍不明确时，只做必要的只读判断并说明推断。

任务生命周期只能由 `项目记忆/任务状态台账.md` 同任务最新状态版本记录；Git、计划 blob、worktree 技术事实和执行窗口结构化消息负责证明该记录是否成立，不能由其它 Markdown 另存一份当前状态。设计窗口写台账只是在一个位置持久化已发生事实，不产生执行许可。执行窗口只在登记给自身的独立任务 worktree 或长期执行通道内，自主读取计划索引中的 `可执行` 精确版本、自行完成 S0 并决定开始；设计消息、预留路径、登记目标或“下一步”均不得成为开始、继续、停止或选择计划的命令。长期执行通道只登记一次顶层执行身份、持久 worktree 和持久分支，不为每份计划重新建立 R2、派发状态或执行许可；同一通道同一时刻只允许一个活动计划段。完整交互合同见 `规范/设计执行双窗口交互规范.md`。

同一物理窗口切换角色时，先完成原角色记录、验证、提交和推送，再由中央路由登记目标角色；切换后重新读取 Git、索引、队列、登记、计划、记录和实际接口。聊天中的口头切换不生效。

集成窗口核对身份并进入 `集成中` 后取得主线发布占用；释放前设计窗口只能只读。路径、分支、起点、父链或所有者不一致时，旧候选只读失效，另立新身份，禁止 reset、delete、切基线或复用。

四角色状态机、消息协议、单窗口切换、模型升级和主线占用细则统一见 `规范/设计执行双窗口交互规范.md`；worktree 创建、冻结、集成、失败和回收细则统一见 `规范/多工作树并发与集成规范.md`。

## 4. 计划、依赖与并发

1. 设计窗口可以直接执行计划索引明确列出的规范、流程图、详细设计、计划、验收项、只读扫描、断点或项目记忆等文档治理动作；流程图完成后默认继续形成对应详细设计，详细设计完成后默认继续形成计划，不另设用户确认门禁。
2. 讨论结论、用户裁决、流程图或详细设计不自动成为代码执行项。涉及代码、字段、函数、能力重建或机器结构变化时，必须先形成并登记计划，列明允许文件、禁止文件、入口拒绝、失败收口和验证方式。
3. 计划、流程图和详细设计必须范围一致并互相列明路径；缺失、撤回、未裁决或范围不一致时不得登记执行。纯规则、只读扫描或确实不适用设计材料的文档计划，必须写明不适用原因和替代依据。
4. 依赖只允许来自尚未形成且尚未冻结合同的具名正式产物 / 接口，或无法通过拆分共享接线消除的结构与装配所有权先后。同主题、同总计划、同来源、生成 / 文档 / 目录 / 验收顺序均不自动构成依赖。已经冻结为精确接口合同的提供者—消费者关系不再构成“等待提供者实现完成”的开发启动门控。
5. 预冻结接口合同至少列明合同版本、模块与文件身份、公开 DTO / 规格 / 结果类型、完整签名或等价行为、ABI / 规则版本、输入输出所有权与生命周期、逻辑内返回、内部错误、提供计划、消费计划和漂移退回点。提供者拥有合同实现权，消费者只读消费；双方可以在独立 worktree 并行形成实现候选，但在真实实现汇合和完整验证前不得宣称接口已实现、代码已编译或能力已完成。
6. 无未合同化真实依赖的计划必须独立登记，并优先按业务对象、文件、结构、公开接口和装配面隔离；共同计划合同基线、所有权互斥、共享文件唯一所有者、固定集成顺序和分级验证矩阵通过后，优先在不同任务 worktree 并发。共同代码基线可以暂时不含提供者实现；消费任务只按冻结合同形成候选，不得自行补造同名接口或修改提供者文件。
7. 共享工程、入口、统一运行器和最终装配优先收归一个后继接线 / 切换计划唯一拥有，不能因为多个私有实现最终都要登记工程而把全部业务计划串行。接口合同未冻结、同一文件 / 结构所有权无法拆分、最终原子切换或物理删除必须消费实际全量结果时，才保留具名门控。
8. 后续计划必须提前形成，不得等待前一个实现完成后才制定，也不得因生成先后伪造依赖。依赖门控计划必须列明具名依赖产物、无法合同化或隔离的原因、前置计划 / 队列、解除条件、复核命令和漂移处理；预期产物不得写成当前事实。
9. 依赖未满足时只跳过受影响计划，不标记真实阻塞，也不冻结无关计划；合同并行批次的提供者或消费者漂移时只停止受影响接口所有者和消费切片，未受影响任务继续。
10. 流程图中的中途非成功结果只分两类：前置通过后内部结构结果异常属于“追根因解决”；协议允许且不产生可读半结构的拒绝、候选为空、等待或人读材料属于“逻辑内返回”。流程图、详细设计和代码必须一致。
11. 面向常规执行模型的计划必须机械区分已验证前置事实、预冻结待实现合同、待实施目标、候选证据、最终集成证据和具名前置成立后才能判断的退回条件。模型升级不改变角色或文件权限。
12. 具有执行角色写范围的施工计划完成设计、依赖满足或已经合法合同化、五件套与接口合同门禁通过后，设计窗口只负责把精确版本 / blob 登记为 `可执行`，同步静态依赖、允许 / 禁止范围、失败收口和验证合同并发布设计提交。叶子计划是不可变施工合同，不得写入“当前待执行 / 已派发 / 执行中”等动态状态。设计窗口不得创建执行许可、指定开始时点、向执行窗口派发、接受 S0 或把计划就绪解释为已经实施。纯设计治理计划继续由设计角色执行，不伪造执行窗口。
13. `计划/计划索引.md` 唯一维护与任务生命周期分离的计划激活状态：`待激活 / 可执行 / 已退出`。`可执行`只证明精确计划版本和 blob 可以被正式消费，不等于已经开始、S0 通过或取得活动计划段单写所有权；`待激活`表示该版本因执行回执要求修订而被冻结，不得被执行窗口选择。任务生命周期仍只由任务状态台账裁决。
14. 计划索引只发布静态依赖、合同版本、计划 blob、所有权和最终汇集关系，不发布由设计窗口控制的执行序列。执行窗口在自身登记的独立任务 worktree 或长期执行通道内自主选择一份 `可执行` 且真实依赖已满足的计划，自行核对唯一执行占用并完成 S0；长期通道的唯一执行占用具体表现为当前工作区没有第二活动计划段。PASS 后直接实施，DRIFT 时停止受影响计划、发送修订请求并继续检查其它无关 `可执行` 计划。长期执行通道按一个计划段一个起点、一个允许范围和一个结果提交串行前进；计划正常收口或形成计划允许的 clean 代码断点后即可选择下一份，不需要逐计划重建 worktree、分支、顶层任务或取得逐计划许可。不得夺取其它执行通道、worktree 或所有权。
15. 设计窗口按“逐份设计、逐份冻结、逐份发布”持续推进：每完成一份计划就登记为 `可执行`，再继续设计下一份，不等待前一个计划执行或整批计划设计完毕。收到执行缺口时优先把受影响旧版本改为 `待激活` 并形成新版本；修订完成后恢复 `可执行`。这些状态变化只表达计划设计可用性，不是对执行窗口的继续、停止或下一项指令。

## 5. 机器逻辑与代码硬边界

机器语义以 0050 和目标对象正式规范为准；下列入口级禁令不得跳过：

1. 函数名、变量名和类内业务成员名必须使用中文。英文只用于 C++ 关键字、标准库、第三方 API、构建字段和不可避免的外部协议字段。
2. 机器逻辑必须由正式结构承载，包括节点直接字段、稳定句柄、关系、领域类型化记录、索引、特征、存在、场景、状态、动态、二次特征、特征值、因果、需求、任务和方法等；日志、注释、文本、显示、线程或锁状态不得承载机器事实。
3. 运行期仓库裁决当前机器事实；数据库和文件只承担快照、事件记录、恢复候选、审计和只读投影，恢复材料必须重新准入后发布。
4. 外部输入、外设采集、控制台、网络消息和数据库记录首先都是材料或请求，不直接成为世界事实。
5. 外部调用方不得绕过领域服务直接写仓库。领域服务公开业务入口不得暴露原始事务令牌、许可、仓库锁或仓库引用；跨仓库写入遵守专用数据操作层、不透明写入会话和最后发布边界。
6. 所有写入口先拒绝无效身份、错误类型、越权、版本漂移和不满足前置条件的输入，再写结构；禁止先写错误 / 半结构，再靠校验、扫描、日志、弹窗或显示修复。
7. 前置条件通过后，内部创建、绑定、写关系、写状态、记录动态、结算、读回或发布不符合预期，属于内部逻辑错误，必须停止依赖路径并追根因，不得静默返回失败后继续。
8. 诊断只做人读提醒，不改变返回值或机器事实。日志工程治理统一读取 `规范/日志系统规范.md`；日志写入 `D:\海中鱼巣\日志`，逻辑错误、运行和调试日志分离；容错 / 调试宏关闭时不得改变运行行为，宏开启后的错误按该入口与现行弹窗设计处理。
9. 线程不是动作来源；方法执行或领域服务写入才可成为动作来源候选。单个因果引用不等于稳定因果结论。
10. 普通 bool、枚举、函数返回码、缓存命中或索引命中不得独立成为一阶机器事实；长期身份使用稳定句柄或主键，指针只作运行期临时引用或缓存。
11. 基础信息服务遵守原子服务与聚合服务分层；特征值服务只由特征服务直接访问，高级服务通过正式基础信息服务取得材料，不新增不存在的 `高级信息` 节点类型。
12. 场景内实例状态和实例动态必须具备存在主体、场景归属和发生时间，不直接充当长期语素入口；长期抽象状态 / 动态及其它对象边界以对应正式规范为准。
13. 需求目标是目标状态，不是 I64；方法执行权限来自任务授权、承接和任务—方法关系，任务、需求和方法的完成 / 满足 / 结算分别裁决。
14. 不把旧项目双向链表模板或整块旧实现复制为新项目核心；当前兼容写入路径不得无计划全仓删除，新生产代码不得新增原始令牌穿透服务路径。

代码编辑同时遵守：

1. 搜索优先使用 `rg`，手工修改使用 `apply_patch`；先确认同功能入口，能扩展既有领域服务时不重复创建。
2. 新增结构必须说明用途、身份、写入方、读取方、生命周期和验证；不新增只为显示方便的机器字段。
3. 删除、重挂和关系变更必须走仓库或领域服务入口。节点移动统一视为普通父子关系重挂，领域服务只做前置拒绝并调用通用关系入口，不重复实现移动算法。
4. 现有目录和代码文件默认原地保留。数据结构文件使用同名 `功能.分类.h/.cpp`；其它新代码使用真实 `功能.分类.ixx` 模块，具备匹配的 `export module`、`ClCompile` 登记和 `import` 调用。
5. 新代码文件必须包含最小文件头部规则注释。生产模块与自检模块分离，自检不得进入生产依赖方向。
6. `入口.cpp` 只保留参数、模式、最小装配、顶层调用和退出码；完整领域实现、自检正文、并发压力和逐项验收输出留在对应模块。

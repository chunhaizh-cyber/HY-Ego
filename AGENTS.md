# AGENTS.md

本文件是 `海中鱼巣` 给 Codex / 代码代理 / 协作代理使用的项目级工作说明。

## 1. 项目定位

`海中鱼巣` 是对 `鱼巢` 的第二代全量重构实验工程。

本项目目标不是复制旧项目文件，而是重建更清晰的内核协议：

```text
节点仓库
稳定句柄
主信息仓库
关系仓库
索引仓库
事务边界
领域服务
最小需求任务方法闭环
```

当前不得宣称：

```text
鱼巢旧能力已迁移完成
自我循环已完成
自我苏醒完成
初步成熟完成
外设 / 体素 / 控制面板已接入
```

## 2. 开工前必须读取

每次修改本项目文件前，先限载读取：

```text
计划/计划索引.md
规范/000_项目规则总纲.md
规范/001_规则迁移清单.md
```

如果执行某个计划，还要读取对应计划文件。

如果修改代码，还要先读取当前代码文件头部规则注释；新建代码文件必须包含最小文件头部规则注释模块。

## 3. 当前硬规则

1. 函数名、变量名、类内业务成员名必须使用中文。
2. 英文只允许用于 C++ 关键字、标准库类型、第三方 API、编译系统字段和不可避免的外部协议字段。
3. 机器逻辑必须落到节点、主信息、关系、索引、特征、特征值、需求、任务、方法、动态或因果引用等结构中。
4. 日志、控制台输出、注释、说明文档、临时文本不得承载机器逻辑或业务状态。
5. 运行期核心先以内存节点仓库为准；数据库只做快照、事件日志、恢复和审计适配，不直接裁决机器事实。
6. 不直接复制旧项目的双向链表模板作为新项目核心结构。
7. 外部调用方不得绕过领域服务直接写节点仓库、主信息仓库、关系仓库或索引仓库。
8. 线程不是动作来源；方法执行或领域服务写入才是动作动态和因果引用的来源候选。
9. 外部输入、外设材料、控制台输入只作为材料或请求，不直接成为世界事实。
10. 本项目编码禁止用事后校验逻辑兜底结构正确性；所有仓库、领域服务和业务写入口必须在函数入口排除无效句柄、错误节点类型、越权写入和不满足前置条件的输入。
11. 不允许先写入无效结构、悬空关系或临时错误状态，再靠校验函数、扫描器、日志或显示输出来发现和修正。
12. 容错检查只能作为预编译宏控制的诊断层；宏关闭时不得改变运行行为，宏开启后触发错误必须同时写日志并弹窗提醒。
13. 容错检查日志和弹窗只做人读诊断，不得承载机器逻辑、不得修复结构、不得作为业务状态来源；弹窗意味着逻辑错误已经暴露，必须排除该逻辑错误后才能继续推进依赖该路径的实现或验收。
14. “基础信息”是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型；当前包含特征、存在、场景、抽象状态、抽象动态、二次特征、因果引用等长期信息节点。实例状态和实例动态是场景内临时零散节点，不得直接作为长期语素入口对应信息。“高级信息”不是独立节点类型；当前指需求、任务、方法等已存在的高级业务节点角色。需要绑定信息节点时，必须在入口接收并验证已有基础信息类节点、需求、任务或方法节点，不得新增 `高级信息` 节点类型。
15. 基础信息服务必须按原子服务和聚合服务分层；特征值、特征、存在、场景、状态、动态、二次特征、因果分别由对应基础信息原子服务负责，世界服务只做聚合编排，不得绕过原子服务直接写基础信息结构。
16. 特征值服务只服务于特征系统，不得作为需求、任务、方法的通用值容器。需求、任务、方法服务需要存在、场景、特征能力时，必须通过基础信息服务获得。
17. 实例状态和实例动态是场景内临时零散节点，必须有存在主体，不得直接长期存储或作为语素入口对应信息；抽象状态和抽象动态才可作为长期基础信息节点。
18. 所有日志都放在 `D:\海中鱼巣\日志` 目录下；逻辑错误报错必须写入独立逻辑错误日志文件；运行日志必须使用独立文件记录系统运行情况；调试日志按切片宏控制启用和关闭，调试完成后必须关闭对应切片调试日志代码，防止无用日志泛滥。
19. 每个完成声明必须给出输入入口、核心处理、结构承载、输出变化、调用路径和验证方式。

## 4. 文档权力关系

```text
AGENTS.md：协作和执行规则
规范/：能不能这样做
计划/：本轮做哪一段
项目记忆/：当前状态、决策、队列、待确认和验收基线
实施记录/：实际做了什么和验证结果
代码：当前实现事实
README / 控制台输出 / 日志：人读说明和诊断材料
```

正式流转只允许沿以下链路推进：

```text
问题 / 新口径
-> 规范
-> 计划
-> 代码实施
-> 实施记录
-> 验证
-> 完成 / 归档
```

禁止反向覆盖：

```text
实施记录直接改规则
计划直接改规则
控制台输出直接改业务事实
日志直接改机器状态
旧项目实现直接覆盖新项目规则
```

## 5. 代码编辑规则

1. 手工编辑使用 `apply_patch`。
2. 优先使用 `rg` 搜索。
3. 不把旧项目代码整块复制进新项目；迁移旧能力必须先写清保留行为、删除行为和新增约束。
4. 新建函数前先查是否已有同功能入口；已有入口可扩展时优先扩展领域服务。
5. 新增结构必须说明：用途、写入方、读取方、生命周期、验证方式。
6. 不新增只为显示方便存在的机器字段。
7. 不把普通 bool 当作一阶机器事实；需要二态裁决时优先使用状态值、评分、计数、版本或复合裁决。
8. 指针只能作为运行期缓存或临时引用，长期身份必须使用稳定句柄或主键。
9. 删除、重挂、关系变更必须通过仓库或领域服务入口，不能裸改关系记录。
10. 写函数时先写入口前置拒绝条件，再写结构写入；不得把“校验结构”“事后扫描”“修复坏数据”设计成正常运行路径。
11. 需要容错诊断时使用项目公共容错检查宏，不得手写分散的日志弹窗逻辑。
12. 显示层只读结构，不写业务事实。

## 6. 构建与验证

默认构建命令以后续项目文件为准；当前 Visual Studio C++ 项目可优先使用：

```powershell
msbuild .\海中鱼巣.vcxproj /p:Configuration=Debug /p:Platform=x64 /m
```

验证声明必须分级：

```text
文档已生成
代码已编译
入口前置拒绝通过
最小闭环通过
旧能力迁移通过
真实外设样本通过
```

不得把低级验证说成高级能力完成。

## 7. 云端同步规则

1. 新建项目内容默认必须纳入 git 跟踪并同步到云端仓库；明确属于本地临时、缓存、构建输出、个人配置或敏感材料的文件除外。
2. Codex 完成每个计划切片后，自动执行切片收尾同步，不再等待额外的“同步到云端”指令。
3. 切片收尾同步顺序为：

```text
检查 git status
确认本切片相关变更范围
git add 本切片相关文件
git commit
git push 当前分支对应远端
```

4. 若提交或推送失败，必须记录失败命令、失败原因和当前阻塞状态；不得宣称该切片已完成云端同步。
5. 自动同步只证明仓库内容已提交并推送，不证明代码编译、入口拒绝、最小闭环或旧项目能力迁移已通过。

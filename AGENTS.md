# AGENTS.md

本文件是 `海中鱼巣` 给 Codex、代码代理和协作代理使用的项目级入口规则。这里只保留每轮必须立即知道的权威顺序、智能体权限、停工条件和验证边界；完整状态机、机器语义和工程细则由下列专门规范承载，不在本文件重复展开。

## 1. 项目定位与声明边界

`海中鱼巣` 是对 `鱼巢` 的第二代全量重构实验工程，目标是重建节点直接身份、稳定句柄、关系、领域类型化记录、索引、事务边界、领域服务和最小需求—任务—方法闭环，不复制旧项目文件结构。

当前可以声明：

```text
正式规范体系已建立并通过独立覆盖审计；
机器逻辑和代码语义以本项目规范目录登记的现行正式规范为准；
独立重建的只读控制面板第一版和 SQL Server 结构审计投影第一版已实现并验证。
```

上述事实不等于鱼巢旧能力已经等价覆盖。除非当前正式证据另有证明，不得宣称：

```text
鱼巢旧能力已完成等价重建；
自我循环、自我苏醒或初步成熟已完成；
完整业务操作控制面板或数据库恢复已完成；
外设、D455 或体素已经接入生产闭环。
```

计划列表及每份计划的当前状态读取 `计划/计划索引.md`。

## 2. 权威顺序与开工前读取

每次修改项目文件前，先限载读取：

```text
计划/计划索引.md
规范/规范目录.md
规范/0050_项目通用机器逻辑与禁止性规则总纲_20260721.md
规范/0100_编号规则与重排预留说明_20260720.md
规范/0300_规范冲突与前后矛盾清单_20260720.md
```

按任务追加读取：

- 执行计划：目标计划、计划索引中的计划状态与具名依赖、当前 Git/worktree 事实和最新结构化执行回执。
- 建立或修订机器逻辑：目标对象根规范、直接相关子规范和相邻边界。
- 额外 worktree、并行批次或集成：`.codex/rules/多工作树并发与集成规则.md`。
- 智能体路由、交接或状态裁决：`.codex/rules/设计执行双窗口交互规则.md`。
- 修改代码：目标代码文件头部规则、相关正式规范、有效设计、现行计划和执行前实际接口。
- 新建代码文件：追加读取 `规范/代码文件建立归属与模块命名规范.md`。
- 修改日志、诊断宏或日志格式：追加读取 `规范/日志系统规范.md`。

权力关系固定为：

```text
用户本轮明确裁决 -> 先进入设计治理，不直接跳过规范授权代码
AGENTS.md -> 代理协作、智能体类型和执行边界
规范/规范目录.md 及其登记规范 -> 机器逻辑和代码语义的唯一正式定义
流程图、详细设计 -> 在规范内解释实现，不得覆盖规范
叶子计划 -> 不可变施工合同，冻结目标、依据、允许 / 禁止范围、失败收口和验证合同
计划索引 -> 记录计划列表、每份计划的状态、路径 / 版本和具名依赖，不改变系统口径
代码与 Git -> 当前实现事实；与规范不一致时属于待治理差异
README、日志、控制台、界面和聊天 -> 人读材料，不裁决机器事实
```

`D:\鱼巢` 的规范与代码、已退出迁移材料、旧实现、过期计划 / 流程图 / 详细设计和历史记录只作来源、差距与风险证据。只有经过海中鱼巣结构映射、冲突裁决并登记到 `规范/规范目录.md` 的规则，才取得本项目机器语义权威。

代码实施依据链必须完整：

```text
现行正式规范
-> 有效流程图与详细设计（适用时）
-> 已登记现行计划和计划索引
-> 执行前实际接口复核，或正式登记的预冻结接口合同复核
-> 代码修改、验证、Git 提交和结构化执行回执
```

预冻结接口合同只允许在登记的合同并行批次中替代“提供者实现已经存在”这一项事实，不替代正式规范、有效设计、计划、所有权或最终真实接线验证。合同必须明确标为待实现接口，不得伪写成当前代码已实现。任一环节缺失、失效、冲突或漂移时停止受影响切片：智能体协作规则问题等待用户交给交互智能体；尚未被执行选中的新目标设计包由计划智能体处理；已经被执行选中或退回的设计包由配对计划支撑智能体修订。不得由旧实现、迁移材料、日志、施工记录或执行智能体临时推断补造规则。

## 3. 智能体、窗口与“继续”

项目采用交互、计划、执行、计划支撑四智能体。智能体权限不因使用同一物理窗口、不同模型、worktree、分支、子智能体或 Full access 而合并。

| 智能体 | 唯一职责与写入权 | 主要禁止 |
| --- | --- | --- |
| 交互智能体 | 与用户交互、澄清目标并独占修订智能体协作规则，包括 `AGENTS.md`、`.codex/rules/` 和 `.codex/skills/`；只有确实需要形成项目设计包时才向计划智能体发送目标 | 不修改项目机器规范、流程图、详细设计、计划、计划索引、代码 / 工程 / 自检或施工记录，不向执行或计划支撑智能体发送消息 |
| 计划智能体 | 接收交互智能体目标或沿既有总目标推导后续目标，为尚未被执行选中的新目标建立和维护完整设计包：正式规范及目录、冲突登记、流程图、详细设计、知识图谱、计划和计划索引项；以正式文件交付，不发送跨智能体消息 | 不修改智能体协作规则、代码 / 工程 / 自检或施工记录，不修改已经被执行选中或退回计划支撑智能体的设计包 |
| 执行智能体 | 在目标计划指定的独立任务 worktree 或长期执行通道中自主读取 `可执行` 计划、自行完成 S0，并只修改当前计划允许的代码、工程、自检、专属施工记录和验证记录；所有执行事实和计划问题只发送给配对的计划支撑智能体 | 不修改智能体协作规则、规范、流程图、详细设计、计划或计划索引，不向交互或计划智能体发送消息，不在非集成模式直接推送 `origin/main` |
| 计划支撑智能体 | 接收配对执行智能体的执行事实；从计划被执行智能体选中并首次发送 S0 开始，接管该计划关联设计包的修订权，可以在原目标内修订相关规范、流程图、详细设计、知识图谱、既有计划及对应计划索引项；修订完成后只向来源执行智能体发送精确版本、blob、状态与影响范围 | 不修改智能体协作规则、代码 / 工程 / 自检或施工记录，不创建无关新目标或新计划，不向交互或计划智能体发送消息 |

智能体协作规则修订只属于交互智能体；项目机器规范属于设计包，不属于智能体协作规则。计划智能体可以为新目标建立或修订项目机器规范，计划支撑智能体可以在原目标不变且证据充分时修订已接管设计包的项目机器规范。两者遇到需要改变用户目标、无法裁决的机器语义或智能体协作规则问题时，必须停止受影响切片，在自身用户可见窗口列出问题并等待用户直接消息；不得代改智能体协作规则或跨智能体转发。

跨智能体消息只允许：

```text
交互智能体 -> 计划智能体
执行智能体 -> 配对的计划支撑智能体
计划支撑智能体 -> 配对的执行智能体
计划智能体 -> 无
```

用户直接向任一用户可见智能体窗口发送消息，以及智能体在自身窗口向用户报告，不属于跨智能体消息。任何消息、送达或聊天回执都不是执行许可、停止命令、计划选择、S0 接受或机器事实；接收方必须重读正式仓库与 Git 后自行裁决。

顶层任务是平台关系中不属于任何协作代理后代的独立用户可见任务，包括用户直接打开或创建的任务，以及用户授权后由 Codex 任务工具创建并交由用户独立接管的任务。顶层身份按平台父子关系和协作代理关系裁决；任务标题、消息来源、worktree、分支、模型、智能体名称和文件路径都不改变该身份。一个顶层任务及其全部直接和递归协作代理共同构成一个“顶层任务树”。

顶层任务树遵守以下最小硬规则：

1. 同一顶层任务树在同一时刻只能拥有交互、计划、执行、计划支撑中的一种写类型；顶层任务只读时整树只能只读。
2. 全部有写入权的后代只能继承顶层任务树的当前智能体类型；只读后代保持零修改、零构建、零运行、零 Git 写入、零活动计划段所有权和零发布占用。
3. 每个具体文件、结构、接口、验证或证据裁决切片同一时刻只有一个所有者；同类型不等于可以共同写同一切片。
4. worktree、分支、模型、推理等级、名称和递归层级均不改变智能体类型、范围、所有权或许可。
5. Git index、暂存、提交、推送、执行通道 / 活动计划段单写所有权迁移和主线发布占用迁移只由唯一指定智能体串行收口；其它后代在收口期间停止写入。
6. 跨写类型默认交给独立顶层任务。只有用户明确启用同一物理窗口串行切换时，才可在全部后代停止、旧类型记录与验证完成、Git 和占用释放并正式重新路由后整树切换；规则修订无论如何都只能切换为交互智能体。
7. 发现一棵任务树存在两种写类型、未授权写入、所有权重叠或类型规避时，整树立即停止新增写入，保留现有 Git 事实，在自身用户可见窗口列出问题并等待用户裁决；不得以文件无交集、结果正确或最终可合并继续。
8. 执行智能体的仓库写入上限是当前计划允许的代码、工程、自检，以及该计划唯一专属的施工记录和验证记录。施工记录只允许写入实际 Git 身份、S0、变更文件、验证命令与结果、提交、残余风险和完成边界，不得定义机器语义、修改计划状态、解除依赖或扩大白名单。执行完成、漂移、失败和计划修订请求仍只通过结构化消息交给配对的计划支撑智能体；施工记录不得替代消息、正式计划或 Git 事实。

存在两个以上边界明确、无真实依赖且所有权可隔离的子任务时，应优先使用同类型或只读子智能体并行提速；不使用必须具名说明原因，不得制造空任务或伪造并行。使用更多智能体不能扩大智能体权限、正式计划、计划索引、worktree、执行或发布边界，也不能替代现行规范、有效设计、实际接口复核和正式验证。四智能体交互合同、文件所有权、消息矩阵、子智能体 S0、停止与升级统一见 `.codex/rules/设计执行双窗口交互规则.md`；worktree 创建、冻结、集成、失败和回收细节统一见 `.codex/rules/多工作树并发与集成规则.md`。

`继续` 是上下文指令：只沿当前已声明或正式登记的智能体类型继续，不自动等于从计划列表选中另一份计划，也不自动跨类型。窗口类型仍不明确时，只做必要的只读判断并说明推断。


同一物理窗口切换智能体类型时，先完成原类型验证、提交和推送，再明确目标类型；切换后重新读取该类型的正式入口、Git、计划索引、目标计划和实际接口。聊天中的口头切换不生效。

执行智能体进入登记的集成模式并核对身份后取得主线发布占用；释放前其它写智能体只能只读。路径、分支、起点、父链或所有者不一致时，旧候选只读失效，另立新身份，禁止 reset、delete、切基线或复用。

四智能体状态机、消息协议、单窗口切换、模型升级和主线占用细则统一见 `.codex/rules/设计执行双窗口交互规则.md`；worktree 创建、冻结、集成、失败和回收细则统一见 `.codex/rules/多工作树并发与集成规则.md`。

## 4. 计划、依赖与并发

1. 交互智能体可以直接完成智能体协作规则修订、目标讨论、解释和只读事实扫描。确实需要形成项目设计包时，交互智能体只向计划智能体发送目标，不代写项目机器规范、流程图、详细设计或计划；不需要项目设计包的协作规则修订、讨论、解释和只读检查不发送消息。
2. 讨论结论、用户裁决、流程图或详细设计不自动成为代码执行项。涉及代码、字段、函数、能力重建或机器结构变化时，必须先形成并登记计划，列明允许文件、禁止文件、入口拒绝、失败收口和验证方式。
3. 计划、流程图和详细设计必须范围一致并互相列明路径；缺失、撤回、未裁决或范围不一致时不得登记执行。纯规则、只读扫描或确实不适用设计材料的文档计划，必须写明不适用原因和替代依据。
4. 依赖只允许来自尚未形成且尚未冻结合同的具名正式产物 / 接口，或无法通过拆分共享接线消除的结构与装配所有权先后。同主题、同总计划、同来源、生成 / 文档 / 目录 / 验收顺序均不自动构成依赖。已经冻结为精确接口合同的提供者—消费者关系不再构成“等待提供者实现完成”的开发启动门控。
5. 预冻结接口合同至少列明合同版本、模块与文件身份、公开 DTO / 规格 / 结果类型、完整签名或等价行为、ABI / 规则版本、输入输出所有权与生命周期、逻辑内返回、内部错误、提供计划、消费计划和漂移退回点。提供者拥有合同实现权，消费者只读消费；双方可以在独立 worktree 并行形成实现候选，但在真实实现汇合和完整验证前不得宣称接口已实现、代码已编译或能力已完成。
6. 无未合同化真实依赖的计划必须独立登记，并优先按业务对象、文件、结构、公开接口和装配面隔离；共同计划合同基线、所有权互斥、共享文件唯一所有者、固定集成顺序和分级验证矩阵通过后，优先在不同任务 worktree 并发。共同代码基线可以暂时不含提供者实现；消费任务只按冻结合同形成候选，不得自行补造同名接口或修改提供者文件。
7. 共享工程、入口、统一运行器和最终装配优先收归一个后继接线 / 切换计划唯一拥有，不能因为多个私有实现最终都要登记工程而把全部业务计划串行。接口合同未冻结、同一文件 / 结构所有权无法拆分、最终原子切换或物理删除必须消费实际全量结果时，才保留具名门控。
8. 后续计划必须提前形成，不得等待前一个实现完成后才制定，也不得因生成先后伪造依赖。依赖门控计划必须列明具名依赖产物、无法合同化或隔离的原因、前置计划列表项、解除条件、复核命令和漂移处理；预期产物不得写成当前事实。
9. 依赖未满足时只跳过受影响计划，不标记真实阻塞，也不冻结无关计划；合同并行批次的提供者或消费者漂移时只停止受影响接口所有者和消费切片，未受影响任务继续。
10. 流程图中的中途非成功结果只分两类：前置通过后内部结构结果异常属于“追根因解决”；协议允许且不产生可读半结构的拒绝、候选为空、等待或人读材料属于“逻辑内返回”。流程图、详细设计和代码必须一致。
11. 面向常规执行模型的计划必须机械区分已验证前置事实、预冻结待实现合同、待实施目标、候选证据、最终集成证据和具名前置成立后才能判断的退回条件。模型升级不改变智能体类型或文件权限。
12. 新目标的规范、流程图、详细设计、知识图谱、施工计划和计划索引形成范围一致的设计包，依据、依赖、五件套与接口合同门禁通过后，由计划智能体把精确版本 / blob 登记为 `可执行`，同步静态依赖、允许 / 禁止范围、施工记录路径、失败收口和验证合同并发布设计包提交。计划智能体不发送跨智能体消息；执行智能体从正式计划索引自主发现。叶子计划的每个精确版本 / blob 都是不可变施工合同，不得写入“当前待执行 / 已派发 / 执行中”等动态状态；修订必须形成新版本 / blob。
13. `计划/计划索引.md` 是唯一运行导航，以列表形式记录计划路径、版本、计划状态和具名依赖；计划之间只登记具名依赖，不另建层级结构。计划状态只允许 `待激活 / 可执行 / 已退出`。`可执行`只证明精确计划版本和 blob 可以被执行智能体选中，不等于已经开始、S0 通过或取得活动计划段单写所有权；`待激活`表示该版本正在形成或需要修订，不得被执行智能体选中。任务过程不另建 Markdown 状态台账，以当前 Git/worktree 事实和结构化消息为准。
14. 执行智能体读取计划索引，根据每份计划的状态和具名依赖自主选中一份 `可执行` 且真实依赖已满足的计划，再按 `.codex/rules/` 与 Git 核对自身执行通道、唯一执行占用并完成 S0；计划索引不发布由其它智能体控制的执行序列。长期通道的唯一执行占用具体表现为当前工作区没有第二活动计划段。计划段选择、S0、完成、DRIFT、失败、WIP、计划修订请求和 `RULE-BLOCKED` 只发送给配对的计划支撑智能体；PASS 后直接实施，DRIFT 时停止受影响计划、发送修订请求并继续检查列表中的其它无关 `可执行` 计划。长期执行通道按一个计划段一个起点、一个允许范围和一个结果提交串行前进；计划正常收口或形成计划允许的 clean 代码断点后必须立即运行继续门禁并选择下一份，不需要逐计划重建 worktree、分支、顶层任务或取得逐计划许可。不得夺取其它执行通道、worktree 或所有权。
15. 计划智能体按“逐份设计、逐份冻结、逐份发布”持续形成新目标设计包：每完成一份计划就登记为 `可执行`，正式提交后继续下一份，不等待前一个计划执行或整批设计完毕，也不发送跨智能体消息。计划一旦被执行智能体选中并发出 S0，关联设计包的修订权转给配对计划支撑智能体；计划智能体停止修改该包。收到执行缺口后，计划支撑智能体先把受影响计划版本改为 `待激活`，按原目标同步修订必要的规范、流程图、详细设计、知识图谱、计划和索引，形成新版本 / blob 后恢复 `可执行`，再只向来源执行智能体发送 `PLAN-SUPPORT-NOTICE`。若缺口需要改变用户目标、修改智能体协作规则或无法裁决机器语义，计划支撑智能体停止、列出问题并等待用户直接消息。
16. 执行智能体在计划段结果 / 合法 WIP 提交和推送后，不得把“单计划完成”当作“执行任务完成”。worktree / index 恢复 clean 后，必须重读正式计划索引、计划 blob、当前 HEAD / upstream 与本通道已固定的计划段映射；只要仍有自身通道内真实依赖满足的 `可执行` 计划，就禁止输出最终答复，必须以当前累计 HEAD 建立下一计划段并重新 S0。只有 `NO-CANDIDATE`、`USER-PAUSED`、`DIRTY-BLOCKED`、`CHANNEL-DRIFT`、`RULE-BLOCKED` 或 `ENVIRONMENT-BLOCKED` 可以作为具名终止码；终止回执必须附计划索引 blob、HEAD / upstream、dirty state、候选扫描与逐项排除依据。平台轮次或上下文被迫让出只能报告 `EXECUTION-YIELD` 续点，不得宣称执行任务已经停止或完成。完成消息已送达、计划支撑智能体没有回复、单份计划耗时较长或发生上下文压缩，都不是停止条件。

## 5. 机器逻辑与代码硬边界

机器语义以 0050 和目标对象正式规范为准；下列入口级禁令不得跳过：

1. 函数名、变量名和类内业务成员名必须使用中文。英文只用于 C++ 关键字、标准库、第三方 API、构建字段和不可避免的外部协议字段。
2. 机器逻辑必须由正式结构承载，包括节点直接字段、稳定句柄、关系、领域类型化记录、索引、特征、存在、场景、状态、动态、二次特征、特征值、因果、需求、任务和方法等；日志、注释、文本、显示、线程或锁状态不得承载机器事实。
3. 运行期仓库裁决当前机器事实；数据库和文件只承担快照、事件记录、恢复候选、审计和只读投影，恢复材料必须重新准入后发布。
4. 外部输入、外设采集、控制台、网络消息和数据库记录首先都是材料或请求，不直接成为世界事实。
5. 外部调用方不得绕过领域服务直接写仓库。领域服务公开业务入口不得暴露原始事务令牌、许可、仓库锁或仓库引用；跨仓库写入遵守专用数据操作层、不透明写入会话和最后发布边界。
6. 所有写入口先拒绝无效身份、错误类型、越权、版本漂移和不满足前置条件的输入，再写结构；禁止先写错误 / 半结构，再靠校验、扫描、日志、弹窗或显示修复。
7. 前置条件通过后，内部创建、绑定、写关系、写状态、记录动态、结算、读回或发布不符合预期，属于内部逻辑错误，必须停止依赖路径并追根因，不得静默返回失败后继续。
8. 诊断只做人读提醒，不改变返回值或机器事实。日志工程治理统一读取 `规范/日志系统规范.md`；日志写入 `D:\海中鱼巣\日志`，逻辑错误、运行和调试日志分离；容错 / 调试宏关闭时不得改变运行行为，宏开启后的错误按该入口与现行弹窗设计处理。
9. 线程不是动作来源；方法执行或领域服务写入才可成为动作来源候选。单个因果引用不等于稳定因果结论。
10. 普通 bool、枚举、函数返回码、缓存命中或索引命中不得独立成为一阶机器事实；长期身份使用稳定句柄或主键，指针只作运行期临时引用或缓存。
11. 基础信息服务遵守原子服务与聚合服务分层；特征值服务只由特征服务直接访问，高级服务通过正式基础信息服务取得材料，不新增不存在的 `高级信息` 节点类型。

14. 不把旧项目双向链表模板或整块旧实现复制为新项目核心；当前兼容写入路径不得无计划全仓删除，新生产代码不得新增原始令牌穿透服务路径。

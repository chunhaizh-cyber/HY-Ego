# AGENTS.md

本文件是 `海中鱼巣` 给 Codex / 代码代理 / 协作代理使用的项目级工作说明。

## 1. 项目定位

`海中鱼巣` 是对 `鱼巢` 的第二代全量重构实验工程。

本项目目标不是复制旧项目文件，而是重建更清晰的内核协议：

```text
节点仓库
稳定句柄
主信息仓库
关系仓库
索引仓库
事务边界
领域服务
最小需求任务方法闭环
```

当前已经可以精确声明：

```text
正式规范体系已建立并通过独立覆盖审计
后续机器逻辑语义以本项目规范目录登记的现行正式规范为准
独立重建的只读控制面板第一版已实现并验证
SQL Server 结构审计投影第一版已实现并验证
上述能力不等于鱼巢旧控制面板或旧数据库能力已经等价覆盖
```

当前不得宣称：

```text
鱼巢旧能力已在本项目完成等价覆盖
自我循环已完成
自我苏醒完成
初步成熟完成
完整业务操作控制面板已完成
数据库恢复已完成
鱼巢旧控制面板或旧数据库能力已在新内核完成等价重建
外设 / 体素已接入
```

## 2. 开工前必须读取

每次修改本项目文件前，先限载读取：

```text

规范/规范目录.md
规范/0050_项目通用机器逻辑与禁止性规则总纲_20260721.md

```

如果执行某个计划，还要读取对应计划文件。

如果当前位于额外 worktree、准备建立并行批次或承担集成角色，还要读取：

```text
规范/多工作树并发与集成规范.md
项目记忆/并行工作树登记表.md
```

如果修改代码，还要先读取当前代码文件头部规则注释；新建代码文件必须包含最小文件头部规则注释模块，并遵守 `规范/代码文件建立归属与模块命名规范.md`。

## 3. 当前硬规则

1. 函数名、变量名、类内业务成员名必须使用中文。
2. 英文只允许用于 C++ 关键字、标准库类型、第三方 API、编译系统字段和不可避免的外部协议字段。
3. 机器逻辑必须落到节点、主信息、关系、索引、特征、特征值、需求、任务、方法、动态或因果引用等结构中。
4. 日志、控制台输出、注释、说明文档、临时文本不得承载机器逻辑或业务状态。
5. 运行期核心先以内存节点仓库为准；数据库只做快照、事件日志、恢复和审计适配，不直接裁决机器事实。
6. 不直接复制旧项目的双向链表模板作为新项目核心结构。
7. 外部调用方不得绕过领域服务直接写节点仓库、主信息仓库、关系仓库或索引仓库。
8. 线程不是动作来源；方法执行或领域服务写入才是动作动态和因果引用的来源候选。
9. 外部输入、外设材料、控制台输入只作为材料或请求，不直接成为世界事实。
10. 本项目编码禁止用事后校验逻辑兜底结构正确性；所有仓库、领域服务和业务写入口必须在函数入口排除无效句柄、错误节点类型、越权写入和不满足前置条件的输入。
11. 不允许先写入无效结构、悬空关系或临时错误状态，再靠校验函数、扫描器、日志或显示输出来发现和修正。
12. 容错检查只能作为预编译宏控制的诊断层；宏关闭时不得改变运行行为，宏开启后触发错误必须同时写日志并弹窗提醒。
13. 容错检查日志和弹窗只做人读诊断，不得承载机器逻辑、不得修复结构、不得作为业务状态来源；弹窗意味着逻辑错误已经暴露，必须排除该逻辑错误后才能继续推进依赖该路径的实现或验收。
14. 函数入口已经排除错误输入后，内部写入结果、关系目标、参数值或结构状态仍不符合预期时，必须视为逻辑错误追根因；不得静默丢弃错误、不得只返回失败后继续推进依赖路径，宏开启时必须写逻辑错误日志并弹窗提醒。
15. “基础信息”是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型；当前包含特征、存在、场景、抽象状态、抽象动态、二次特征、因果引用等长期信息节点。实例状态和实例动态是场景内临时零散节点，不得直接作为长期语素入口对应信息。“高级信息”不是独立节点类型；当前指需求、任务、方法等已存在的高级业务节点角色。需要绑定信息节点时，必须在入口接收并验证已有基础信息类节点、需求、任务或方法节点，不得新增 `高级信息` 节点类型。
16. 基础信息服务必须按原子服务和聚合服务分层；特征值、特征、存在、场景、状态、动态、二次特征、因果分别由对应基础信息原子服务负责，世界服务只做聚合编排，不得绕过原子服务直接写基础信息结构。
17. 特征值服务只服务于特征系统，只有特征服务可直接访问；需求、任务、方法等高级服务不得直接包含、接收或调用特征值服务。
18. 高级服务只允许直接依赖存在服务、场景服务、特征服务、二次特征服务、状态服务、动态服务和因果服务；需要特征状态材料时必须通过特征服务获得。
19. 实例状态和实例动态是场景内临时零散节点，必须有存在主体、场景归属和发生时间戳，不得直接长期存储或作为语素入口对应信息；抽象状态和抽象动态才可作为长期基础信息节点，且抽象状态和抽象动态本体不携带发生时间戳。
20. 所有日志都放在 `D:\海中鱼巣\日志` 目录下；逻辑错误报错必须写入独立逻辑错误日志文件；运行日志必须使用独立文件记录系统运行情况；调试日志按切片宏控制启用和关闭，调试完成后必须关闭对应切片调试日志代码，防止无用日志泛滥。
21. 因果引用采用轻量化口径：`记录因果引用(来源动态)` 只用来源动态做入口准入证据，不持久化来源动态关系，不得把单个因果引用解释为稳定因果结论。
22. 方法执行权限来自任务授权、任务承接和任务选择方法；方法执行前不再单独创建授权状态节点，后续权限治理必须围绕任务授权关系、任务状态和任务方法关系设计。
23. 需求目标是目标状态，不是 I64；I64 只可作为第一轮状态值材料或特征状态材料，不得作为需求目标对外语义。
24. 每个完成声明必须给出输入入口、核心处理、结构承载、输出变化、调用路径和验证方式。
25. 计划索引中的“下一步”为建立、修订、补齐规范、流程图、详细设计、实施计划、验收项、只读扫描、断点清单或项目记忆同步时，属于可直接执行的文档治理动作；流程图生成后默认直接生成对应详细设计，详细设计生成后默认直接生成计划。计划不再设置用户确认门禁；只有设计窗口可以把计划写入 `计划/`、登记或修订 `计划/计划索引.md` 和 `项目记忆/Codex任务队列.md`。执行窗口只读这些路由材料，按队列顺序检查计划，只执行依赖已经满足且执行前复核通过的项；依赖未满足项保留为依赖门控待执行并跳过。
26. 单个讨论结论、详细设计、问题裁决或流程图不自动等同于可执行项；必须形成计划或落实切片后才能登记待执行。涉及代码实施、能力重建、字段落地、函数重构或改变机器结构时，计划必须明确允许文件、禁止文件、入口拒绝、失败收口和验证方式；计划登记即构成该明确范围的执行依据，不再等待额外确认。
27. 计划生成并登记待执行时必须同步核对对应详细设计和流程图：计划索引、项目记忆和设计生成记录必须列明计划依据的流程图与详细设计路径，并证明三者范围一致。本规则生效后的设计生成 / 审查记录写入 `项目记忆/设计记录/`，不得再以设计角色新建 `实施记录/*计划生成*`；既有文件只保留历史事实。若对应流程图或详细设计缺失、被撤回、仍有未给出默认裁决的问题，或与计划范围不一致，不得登记待执行，必须先补齐或修订上游材料。纯规则、只读扫描、信息数据或无流程图 / 详细设计来源的文档治理计划，必须在计划和设计生成记录中写明不适用原因及替代依据。
28. 计划采用“真实依赖才门控，无真实依赖优先隔离并发”规则。设计窗口生成计划时，必须先按业务对象、文件、结构所有权、公开接口和共享装配面尝试拆分。只有某计划必须消费另一计划尚未形成的正式产物或接口，或者存在无法通过拆出共享接线计划消除的结构 / 装配所有权先后时，才登记为“依赖门控待执行”；同一主题、同一总计划、同一来源、生成顺序、文档顺序、目录位置或验收顺序均不自动构成依赖。无真实依赖的计划必须独立登记为“待执行”；共同基线、文件 / 结构 / 接口所有权互斥、共享文件唯一所有者、集成顺序和验证矩阵通过后，设计窗口应优先登记不同任务 worktree 并发执行；暂不并行时必须具名说明资源、所有权或验证原因。后续计划可以在其它计划未完成时提前生成，但不得仅因生成先后登记依赖门控。依赖门控计划必须列明已验证当前事实、具名依赖产物与消费接口、无法隔离原因、前置计划 / 队列编号、解除条件、假定接口契约、执行前复核命令和接口漂移处理，不得把预期产物写成当前实现事实。依赖未满足时只跳过受影响计划，不标记真实阻塞；依赖完成后逐项比对实际接口，漂移时只退回受影响计划，不得连带阻塞无关计划或修改代码；前置计划形成真实阻塞时，具名消费者继续保持门控，除非设计窗口修订依赖或明确替代依据。
29. 以后制作、修订或审查流程图时，必须使用“追根因解决 / 逻辑内返回”二分口径处理所有中途非成功返回：前置条件已经满足并进入创建、绑定、写关系、写状态、记录动态、结算、读回或结构承载后，结果不符合内部预期的，必须归为追根因解决，不得画成普通失败返回；设计内允许分支且不产生可读半结构，或只返回请求材料、候选为空、拒绝材料、人读材料的，才可归为逻辑内返回。流程图、详细设计和代码若描述同一入口，三者必须保持一致。
30. 现有目录结构和既有代码文件默认原地保留；以后新建的数据结构文件使用同名 `功能.分类.h/.cpp`，其它新代码文件必须使用 `功能.分类.ixx` 真实 C++ 模块，并以 `ClCompile` 登记、`export module` 声明和 `import` 调用。生产模块和自检模块必须分离；`入口.cpp` 只保留参数、模式、最小装配、顶层调用和退出码，不得继续新增领域实现、完整自检正文、并发压力正文或逐项验收输出正文。旧计划与本规则冲突时，执行前必须退回设计窗口修订。
31. 领域服务公开业务入口不得暴露、接收、返回、传递或比较原始 `结构事务令牌`，不得签发 / 释放许可或操作仓库锁。结构事务协调层负责许可签发 / 释放和跨仓库事务上下文；仓库负责验证令牌、本仓库锁和通用结构读写。
32. 跨仓库业务写入必须经过服务专用数据操作层和不透明结构写入会话。数据操作层隐藏仓库引用与原始令牌，负责同一许可内的未发布候选、精确撤销、确认和最后发布；领域服务负责业务准入和成功 / 失败语义，应用 / 领域组合器定义跨服务事务边界。
33. 当前无令牌写、带令牌写和候选写属于待治理兼容路径；不得直接全仓删除。必须先形成调用点清单、替代顺序和回归门禁，按存在场景、状态动态、特征体系、语义基础、需求任务方法和运行期组合器顺序治理，生产调用点归零后再裁决私有化、弃用或删除。新生产代码不得新增原始令牌穿透服务的路径。
34. 默认继续使用设计角色与执行角色分权交接。“制定窗口”统一解释为设计窗口。设计窗口独占计划、计划索引、任务队列和设计治理状态的写入；执行窗口对这些材料只读，只写计划允许范围内的代码、工程、自检以及本次执行专属的实施记录 / 复核断点。未登记并行批次时仍按串行方式推进；登记并行批次后，多个执行窗口可以在不同任务 worktree 中并发，但必须遵守 `规范/多工作树并发与集成规范.md` 的共同基线、文件 / 结构所有权互斥、任务分支和集成验证门禁。用户明确指定同一物理窗口承接制定、修订和执行时，启用“单窗口串行双角色模式”：任一时刻只激活设计、执行、集成或只读复核一个角色；角色切换前必须完成本角色允许的记录、验证、自动本地提交和自动非强制推送，切换后必须重新读取 Git、计划索引、任务队列、工作树登记、目标计划、最新实施记录、断点和实际接口。单窗口只免除向自己发送 Codex 任务消息，不免除计划门禁、允许 / 禁止范围、工作树内单一写入所有权和验证。该模式持续到用户明确撤销或正式写入所有权转交。
35. 面向中等推理执行模型的成熟计划必须写成可机械判定的阶段状态机，明确区分“已验证前置事实、待实施目标、实施后验收和退回条件”。任何退回条件都必须列出可核对的前置证据；前置证据尚未形成时不得触发该退回条件，不得把“尚未实施的目标”解释成接口漂移或阻塞。执行模型若误读明确阶段、跳过授权步骤、把目标当门禁或连续两轮不能按检查表推进，属于模型能力不匹配：执行窗口停止扩权，由设计窗口把同一计划改成更窄的机械步骤，并可显式改派高推理模型继续执行；模型升级不合并角色权限、不扩大允许文件。执行轮次因平台边界结束但尚未命中正式完成 / 退回条件时，只能记录精确续点并保持 `执行中`，不得伪报完成或漂移。
36. 本项目正式治理和代码实施采用设计、执行、集成、只读复核四角色闭环。设计窗口不仅负责规则、流程图、详细设计和计划，还负责依赖图、优先级、真实派发、集成所有权、回执裁决、完成归档和下一项继续派发；执行窗口只形成任务分支；集成窗口只完成范围核对、汇集、完整重验和 main 发布；只读复核窗口只核查指定提交与对象并返回报告。任一角色不得替代另一角色完成其裁决。
37. `已预留`、登记了目标窗口或写了“下一步”都不等于已经派发。设计窗口只有在角色 / 任务 / worktree / 分支 / 基线已经提交推送、任务消息已向唯一目标实际发送，并读回目标任务已接收或已记录工具不可用的持久化替代回执后，才能写 `已派发待执行回执` 或 `已派发待集成回执`。执行分支完成后，设计窗口必须继续核对并真实派发独立集成，不得停在预留等待人工提醒。
38. 集成窗口完成身份核对并进入 `集成中` 后取得主线发布占用。占用释放前，设计窗口不得修改 WT-MAIN 文件、提交或推送 `origin/main`，只能只读审查或形成未落盘草案。登记路径 / 分支已存在但身份、起点、所有者或父链不一致时，旧候选只读登记为失效，禁止 reset、delete、切基线或复用；设计窗口必须另立新编号、路径和分支重新派发。
39. 项目阶段里程碑和最终完成声明前必须由设计窗口派发独立只读覆盖审计。只读复核窗口不得修改文件、运行构建、暂存、提交、推送、入队或解除依赖；报告只含事实、差异、证据、风险和建议。设计窗口必须逐项接受、驳回或转化为规则 / 计划动作。存在就绪未派发、分支未集成、集成 / 执行待设计同步、漂移未裁决、只读报告待裁决或计划 / 台账 / 代码 / origin 不一致时，不得宣称对应阶段或总计划完成。
40. `规范/规范目录.md` 及其登记的现行正式规范是海中鱼巣机器逻辑和代码语义的唯一权威依据；未登记材料不得直接指导代码或改变业务口径。
41. `D:\鱼巢\规范` 只作为本项目规范编制、差距识别和来源核对材料；鱼巢规则必须先经过海中鱼巣的结构映射、冲突裁决和正式登记，才能成为本项目规范。不得以鱼巢目录中的规则直接授权海中鱼巣代码。
42. 已退出的迁移规则、迁移清单、迁移计划、旧实现、过期计划、过期流程图、过期详细设计、历史记录和聊天结论只作证据，不得作为现行施工依据、验收依据或正式规范的替代物。
43. 代码实施必须具备完整依据链：现行正式规范 -> 有效流程图与详细设计（适用时）-> 已登记的现行计划和任务队列 -> 执行前实际接口复核 -> 代码修改。正式规范单独存在不构成修改代码的授权。
44. 现行规范缺失、同级规范冲突、设计材料失效、计划与规范不一致或实际接口偏离计划假定时，必须停止代码实施并退回设计治理；不得从来源材料、旧实现或执行窗口临时判断中补造规则。
45. 正式规范体系完成只证明定义层成立，不证明代码已经实现、生产链已经接通、目标能力已经验收或鱼巢能力已经等价覆盖。

## 3.1 窗口类型与“继续”指令规则

`继续` 是上下文指令，必须先按当前窗口类型解释，不得无条件等同于执行队列消费。

| 窗口类型 | `继续` 的含义 | 允许动作 | 禁止动作 |
| --- | --- | --- | --- |
| 设计窗口 | 继续当前设计、规则、计划或信息数据的梳理和修订 | 独占生成 / 修订规范、流程图、详细设计、计划、`计划/计划索引.md`、`项目记忆/Codex任务队列.md`、设计治理状态和决策记录；维护依赖与规范—设计—代码覆盖；真实派发执行 / 集成 / 只读复核；读取正式回执后更新路由、归档并继续派发下一项 | 不消费待执行队列，不直接改 C++，不运行构建，不把预留当派发，不把没有执行 / 集成证据的计划登记为已执行；集成中不得弄脏或推进 main |
| 执行窗口 | 继续消费已经由设计窗口登记且依赖已满足的待执行队列项 | 只读计划、计划索引和任务队列；复核依赖与实际接口；写计划允许范围内的代码 / 工程 / 自检和本次执行专属 `实施记录/*` / 复核断点；验证、自动本地提交和自动非强制推送 | 不修改计划、计划索引、任务队列、流程图、详细设计或设计治理状态；不从缺少计划、允许 / 禁止文件、依赖或实际接口复核的材料直接实施 |
| 集成窗口 | 继续当前已登记并行批次的范围核对、提交汇集、集成验证和 main 发布 | 只读设计路由；在登记的集成暂存 worktree 按固定顺序集成任务提交；运行集成验证；验证通过后让主集成工作树 `main` 非强制快进并推送 `origin/main`；写本批次集成记录 | 不设计或修改业务语义，不解决需要发明接口的冲突，不修改计划 / 索引 / 队列来扩大集成范围，不把任务分支完成直接标记为计划完成 |
| 只读监控 / 复核窗口 | 继续读取指定提交和对象的事实并汇报 | 读取规则、计划、队列、实施 / 集成记录、git 状态和当前代码事实，按事实 / 差异 / 证据 / 风险 / 建议输出报告 | 不修改文件，不入队，不解除依赖，不裁决完成，不运行构建，不暂存、提交或推送；持久化报告交由设计窗口处理 |
| 未声明窗口类型 | 先按最近用户明确声明、线程标题或当前任务对象判断；仍不明确时，先确认窗口类型或只做不改文件的状态判断 | 读取必要入口并说明当前推断 | 不把裸 `继续` 自动解释为执行队列消费 |

当用户明确说明“当前窗口是设计窗口”“当前窗口是制定窗口”或沿用旧称“计划设计和确认窗口”时，后续裸 `继续` 默认只沿设计窗口推进，直到用户明确切换窗口类型或明确要求执行队列。

当用户已经明确启用单窗口串行双角色模式时，窗口类型按“当前已登记角色”解释：裸 `继续` 只沿当前角色继续，不自动跨角色；只有当前阶段已正式收口、提交并推送，且计划索引、任务队列和窗口交互记录已经登记目标角色后，才允许在同一物理窗口切换角色。切换后必须先重新读取正式状态，不能直接沿用切换前草案。

## 3.2 四角色交互规则

默认多物理窗口模式下，设计、执行、集成和只读复核采用“仓库权威状态 + Codex 排队消息唤醒”的双层交互：

```text
设计窗口生成或修订计划
-> 更新计划索引、任务队列和设计治理状态为“已派发待执行回执”
-> 自动本地提交并自动非强制推送
-> 向执行窗口发送排队提醒

执行窗口重新读取 Git、计划索引、任务队列、目标计划、最新实施记录和实际接口
-> 唯一候选、依赖、版本、允许 / 禁止文件和接口全部一致才实施
-> 完成时写“执行完成待设计同步”实施记录；漂移时写“接口漂移待设计修订”复核断点
-> 自动本地提交并自动非强制推送
-> 向设计窗口发送完成或退回摘要

设计窗口读取正式提交、代码变化和实施记录
-> 更新计划、计划索引、任务队列、窗口交互和归档状态
-> 自动本地提交并自动非强制推送
-> 需要继续时再派发下一计划

任务分支完成
-> 设计窗口核对提交、父链、范围与任务证据
-> 登记并推送独立集成编号 / 路径 / 分支 / 起点 / 所有者 / 验证矩阵
-> 实际发送集成任务消息并读回接收状态
-> 集成窗口核对身份后独占主线发布，完整重验并发布 main
-> 设计窗口读取正式集成证据、归档并重算下一候选

里程碑或最终完成前
-> 设计窗口指定提交、对象和问题集并派发只读复核
-> 只读复核窗口只返回事实报告
-> 设计窗口逐项裁决后才允许完成声明
```

硬边界：

1. 设计窗口是计划、计划索引、任务队列、窗口交互和设计治理状态的唯一写入方；执行窗口不得修改这些文件，即使只是为了标记完成或退回。
2. 执行窗口是本次执行专属实施记录和执行复核断点的写入方；`项目记忆/实施记录.md` 等汇总状态由设计窗口根据正式实施记录更新。设计窗口不得凭消息正文替代实施证据。
3. `计划/计划索引.md`、`项目记忆/Codex任务队列.md`、目标计划、最新正式实施记录、实际代码 / 接口和 Git 提交共同形成权威状态；聊天上下文和任务消息只作提示。
4. 发送正式交接消息前必须先提交本角色允许的产物并完成非强制推送。运行中收到的新计划消息必须排队到下一轮，只作“重新读取哪些正式状态”的提醒，不中断当前切片、不扩展当前计划。
5. 默认一个计划完成或退回后先交回设计窗口同步；只有计划索引、任务队列和 `项目记忆/并行工作树登记表.md` 显式登记“预授权执行批次”或“预授权并行批次”，且批次内计划依赖、基线、接口、文件和结构所有权全部通过时，执行窗口才可连续消费或在不同任务 worktree 并发执行。
6. 串行计划状态至少区分 `已派发待执行回执`、`执行完成待设计同步`、`接口漂移待设计修订`、`重新待执行` 和 `已完成 / 已归档`；并行任务另区分 `分支完成待集成`、`集成失败退回任务` 和 `已集成待设计同步`。设计窗口未完成同步前，不得重复派发同一计划；依赖该计划的后继项不得自行解除门控。
7. Codex 任务消息工具不可用、目标任务不唯一或消息发送失败时，执行窗口的正式实施记录 / 复核断点就是持久化回执；设计窗口下次启动必须扫描并读取，不得要求执行窗口越权修改路由文件。
8. 计划 / worktree / 集成路径的预留不构成派发；设计窗口必须在推送后实际发送消息并核对目标任务状态。未送达时保持待派发，不得写成执行中或集成中。
9. 集成身份核对通过后主线发布占用生效；设计窗口不得并发修改、提交或推送 main。发布或正式退回必须明确释放占用。
10. 只读复核报告不写仓库、不解除依赖、不标记完成；设计窗口是唯一报告裁决和持久化写入方。
11. 每次归档后设计窗口必须重算依赖和唯一就绪候选；存在候选时真实派发，不存在时写明继续设计、等待集成、用户暂停或具名阻断，不能无理由停止。
12. 项目阶段最终完成前必须完成独立只读覆盖审计，并证明正式规范、设计、计划索引、任务队列、完成目录、能力覆盖台账、当前代码、工作树和 `origin/main` 一致。

执行窗口选择下一计划时不得只依赖自身上下文，必须按以下顺序计算：

```text
重新读取 Git、当前 worktree / 分支、登记基线和 dirty state
-> 读取计划索引、任务队列、候选计划和最新实施记录
-> 排除已执行、待设计同步、退回设计和版本不一致项
-> 复核依赖、队列优先级、允许 / 禁止文件和实际接口
-> 串行模式只有唯一候选全部通过才执行；并行模式只有本 worktree 唯一分配项全部通过才执行
-> 未登记多候选、状态冲突、接口漂移、文件 / 结构所有权冲突或依赖尚待设计同步时停止并回传设计窗口
```

## 3.3 同一物理窗口串行双角色规则

用户明确启用单窗口串行双角色模式后，3.1 与 3.2 的角色权限、状态机和 Git 门禁全部继续有效，只把“向另一物理窗口发送任务消息”替换为“同一物理窗口按正式状态切换角色”：

```text
执行角色发现漂移或完成切片
-> 停止实施并完成本次执行专属实施记录 / 复核断点
-> 在该记录中声明目标状态为“执行完成待设计同步”或“接口漂移待设计修订”
-> 验证、自动本地提交并自动非强制推送
-> 推送成功后切换到设计角色
-> 重新读取 Git、计划索引、任务队列、目标计划、最新实施记录和实际接口
-> 才能由设计角色更新路由或修订设计

设计角色完成实施记录同步、修订或后继计划生成
-> 在计划索引、任务队列和窗口交互记录中登记“已派发待执行回执”及目标执行角色
-> 验证、自动本地提交并自动非强制推送
-> 推送成功后角色切换生效
-> 重新读取 Git、计划索引、任务队列、目标计划、最新实施记录、断点和实际接口
-> 只有依赖满足且接口一致时才能执行
```

单窗口模式的硬边界：

1. 物理窗口相同不代表角色权限合并；设计角色不得消费队列、改 C++ 或运行构建，执行角色不得修改计划、计划索引、任务队列、流程图、详细设计或设计治理状态来为代码扩权。
2. 执行转设计必须以已推送的实施记录 / 复核断点为依据；设计转执行必须以已推送的计划索引、任务队列和窗口交互记录为依据；仅在聊天中说“切换”不生效。
3. 同一计划任一时刻只有当前登记角色拥有修改权；切换前后的未验证草案不得跨角色继承。
4. 因目标就是当前物理窗口，不发送自发自收任务消息；如目标是另一物理窗口，仍按 3.2 发送唤醒消息。
5. 单窗口模式持续到用户明确撤销，或窗口交互记录已正式登记写入所有权转交到另一物理窗口。

单窗口模式与多 worktree 可以共存，但同一物理窗口不能同时占用多个角色。当前窗口从任务 worktree 执行角色切回主集成工作树设计角色前，必须先完成任务分支提交推送，并等待集成 / 设计路由形成正式可切换状态；不得跨目录继承未验证草案。

## 3.4 模型路由与异常升级

模型选择不改变角色权限。默认建议：设计、计划审查、接口漂移和逻辑异常裁决使用 Sol + High 或可用的同等高推理配置；成熟计划的常规执行使用 Terra + Medium 或可用的同等中等推理配置；明确、重复且只读的整理工作才使用更低消耗配置。

执行窗口发现接口漂移、计划冲突、事务 / 并发 / 因果边界无法由计划裁决、内部逻辑错误或验证重复失败且根因不明时，必须停止实施并写复核断点，退回高推理的设计窗口；不得在执行角色中仅通过提高模型等级自行扩大计划。

## 3.5 多工作树并发与集成规则

用户已明确允许多个 Git worktree。正式并发采用：

```text
D:\海中鱼巢 + main = 主集成工作树与最终权威
codex/<队列号>-<主题> = 独立任务分支与任务 worktree
codex/integration-<批次号> = 集成暂存分支与集成 worktree
```

硬边界：

1. 多 worktree 必须先登记在 `项目记忆/并行工作树登记表.md`；未登记路径、分支、基线、计划版本或窗口所有者时禁止写入。
2. 并行批次必须共享冻结基线，且依赖全部满足、允许文件集合两两无交集、结构所有者与公开接口无冲突、共享装配文件只有一个所有者。
3. 每个任务 worktree 只执行一个活动计划；同一 worktree 任一时刻仍只有一个写入角色。
4. 任务分支只允许推送同名远端 `codex/*` 分支，完成状态为 `分支完成待集成`；不得直接推送 `origin/main` 或宣称计划已完成。
5. 集成角色只按冻结顺序汇集提交和运行集成验证。出现冲突、超范围、远端 main 前进或验证失败时停止，不得临时发明接口或手工修改业务语义消冲突。
6. 集成验证通过、主集成工作树 main 非强制发布到 `origin/main`、设计角色读取正式证据并归档后，计划才正式完成。
7. 任务 worktree 和分支只有在 main 已包含集成结果、登记表标记可回收且 worktree clean 后才能回收；远端任务分支不自动删除。
8. 当前 Windows 多 worktree 无人值守执行使用 `sandbox_mode = "danger-full-access"` 与 `approval_policy = "never"`，因为任务 Git 元数据位于主仓库受保护 `.git/worktrees`，默认沙箱和自动审查均不能稳定完成提交。Full access 只取消操作系统沙箱与人工审批，不扩大计划允许文件、角色、分支、远端目标或验证范围；任务仍须按登记表、精确暂存、staged diff 和禁止命令门禁执行。
9. 集成 worktree 的预留、创建、派发和接管必须分状态登记。目标路径 / 分支存在时先核对精确身份；不一致则旧候选失效、另立新路径 / 分支，禁止重置或复用。
10. 集成任务进入 `集成中` 后独占主线发布；WT-MAIN 设计角色在发布或退回释放前保持只读。

详细状态机、共享文件边界、冻结清单、失败收口和验证矩阵以 `规范/多工作树并发与集成规范.md` 为准。

## 4. 文档权力关系

```text
AGENTS.md：协作和执行规则
规范/规范目录.md 及其登记的现行正式规范：机器逻辑和代码语义的唯一权威依据
规范/详细设计/：在正式规范约束内说明如何实现，不得覆盖规范
流程图/：设计的视觉化表达，不是独立权威来源
计划/ 与项目记忆/Codex任务队列.md：本轮允许实施哪一段，不改变系统口径
```

## 5. 代码编辑规则

1. 手工编辑使用 `apply_patch`。
2. 优先使用 `rg` 搜索。
3. `D:\鱼巢` 代码和其它旧实现只作行为与差距证据，不整块复制进本项目；任何能力重建都必须先回读海中鱼巣现行正式规范，再按有效设计、现行计划和执行前实际接口复核实施。
4. 新建函数前先查是否已有同功能入口；已有入口可扩展时优先扩展领域服务。
5. 新增结构必须说明：用途、写入方、读取方、生命周期、验证方式。
6. 不新增只为显示方便存在的机器字段。
7. 不把普通 bool 当作一阶机器事实；需要二态裁决时优先使用状态值、评分、计数、版本或复合裁决。
8. 指针只能作为运行期缓存或临时引用，长期身份必须使用稳定句柄或主键。
9. 删除、重挂、关系变更必须通过仓库或领域服务入口，不能裸改关系记录。
10. 节点移动统一视为普通父子关系重挂；通用移动逻辑只能落在关系仓库或后续确认的公共结构移动工具中，各领域服务不得重复实现移动算法，只能做本领域前置拒绝后调用通用入口。
11. 写函数时先写入口前置拒绝条件，再写结构写入；不得把“校验结构”“事后扫描”“修复坏数据”设计成正常运行路径。
12. 需要容错诊断时使用项目公共容错检查宏，不得手写分散的日志弹窗逻辑。
13. 显示层只读结构，不写业务事实。
14. 新建代码文件前必须按 `规范/代码文件建立归属与模块命名规范.md` 判定为数据结构文件或模块文件，并在计划中列明文件名、分类、工程登记和调用方式。
15. 新建非数据结构 `.ixx` 必须是真模块：包含匹配的 `export module`，由工程按 `ClCompile` 登记，使用方通过 `import` 调用；不得新建按 `ClInclude` 或 `#include` 使用的伪模块。
16. 新功能的生产实现与自检 / 验收正文必须分别落入 `功能.分类` 和 `自检.分类` 模块；自检模块不得进入生产依赖方向。
17. 在统一自检运行器形成前，`入口.cpp` 最多增加自检模块导入、最小上下文组装、一次调用和总结果汇总；完整场景、并发循环和逐项输出必须留在自检模块。

## 6.1 主集成分支与多工作树规则

1. 本项目唯一最终权威分支为 `main`，主集成工作树固定为 `D:\海中鱼巢`；`海中鱼巣` 只作为仓库内源码目录名。只有主集成工作树的设计角色或集成角色按各自权限可以修改 `main`。
2. 允许为已登记计划创建额外 Git worktree。任务 worktree 必须使用独立 `codex/<队列号>-<主题>` 分支；集成暂存 worktree 必须使用独立 `codex/integration-<批次号>` 分支；禁止 detached HEAD 和两个 worktree 检出同一分支。
3. 同一 worktree 任一时刻只能有一个写入窗口 / 角色；多个 worktree 只有在正式登记的并行批次中才可并发写入。来源不明 dirty state、登记与实际路径 / 分支 / 基线不一致、目标文件或结构所有权冲突时必须停止并报告。
4. 中央治理文件只在主集成工作树由设计角色写入。任务 worktree 只写目标计划允许的代码、工程、自检和执行专属记录；任务分支不得修改计划索引、任务队列、工作树登记和中央设计治理状态。
5. 任务分支提交只推送同名远端任务分支，不得直接推送 `origin/main`。集成角色在独立集成 worktree 完成范围核对和集成验证后，才能让主集成工作树 `main` 非强制快进并推送 `origin/main`。
6. 工作树创建、登记、冻结、并行准入、集成、失败退回与回收必须遵守 `规范/多工作树并发与集成规范.md`。
7. Windows 任务 worktree 在当前 Codex 桌面环境中按已登记批次使用无提示 Full access；不得因此跳过 `git status`、允许范围、精确暂存、staged diff、非强制推送和集成验证，也不得申请或等待人工命令授权。

## 7. 自动提交与自动推送规则

1. 每完成一轮有文件改动的任务，Codex 自动执行一次本地 Git 提交，不再等待额外的“提交”指令。本条是项目内持续有效的本地提交授权，适用于代码实施、文档治理、规则修订、计划生成、只读扫描落盘和正式退回记录。
2. “完成一轮任务”是指本轮授权范围已经实现并完成计划要求的验证，或已按正式退回 / 阻塞协议完成治理收口。只读回答、没有文件变化、尚未完成、验证失败或范围仍不明确时不得创建空提交或提前提交。
3. 自动提交必须按以下顺序执行：

```text
检查 git status 和当前分支
确认本轮任务文件范围与无关 dirty state
只 git add 本轮任务相关文件
运行 staged diff / diff check
git commit
复核提交内容和剩余 dirty state
```

4. 自动提交不得带入其它窗口、用户或其它任务的改动；目标分支不明、detached HEAD、同一目标行冲突或暂存范围无法分离时停止并报告，不得扩大提交范围。
5. 用户已明确授权后续任务自动推送。每轮自动提交完成后，Codex 必须自动执行非强制推送，不再逐轮等待“推送 / 同步到云端”指令。主集成工作树中的设计 / 集成结果默认发布到 `origin/main`；任务 worktree 默认只发布到同名远端 `codex/*` 任务分支。任务分支禁止使用 `git push origin HEAD:main`；只有集成验证通过后的主集成工作树可以发布 `origin/main`。
6. 自动推送前必须确认目标远端和分支与工作树登记一致、远端没有形成非快进分叉、当前提交只包含已验证的本轮任务范围，且不存在 detached HEAD、无法隔离的无关改动或目标分支冲突。任何条件不满足时停止推送并报告，不得使用 `--force`、`--force-with-lease` 或借任务窗口自动推送执行合并、变基、删分支、创建 PR、发布制品。集成只能由已登记集成角色按专项规范执行。
7. 若提交或自动推送失败，必须记录失败命令、失败原因和当前状态；不得宣称本地提交或云端同步已经完成。推送成功后才能发送正式跨窗口交接消息。
8. 自动提交和自动推送只证明本轮文件已形成 Git 提交并发布到指定远端，不证明代码编译、入口拒绝、最小闭环、目标能力验收或系统阶段完成。

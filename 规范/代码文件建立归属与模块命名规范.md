# 代码文件建立归属与模块命名规范

工程治理标识：CODE-FILE-01

日期：2026-07-21

身份：项目级非机器代码工程治理入口

## 1. 定位与权力边界

本文件统一约束代码文件分类、模块命名、实体所有权、入口承载、并发与锁、公共辅助函数、序列化与恢复辅助、UTF-8 和中文标识符等代码工程问题。

本文件未登记在 `规范/规范目录.md` 的“正式规范”表中，不是本项目机器逻辑规范，不定义对象语义、事实来源、写入权、业务状态、合法业务分支或完成判定。机器逻辑和代码语义只能来自 `规范/规范目录.md` 登记的现行正式规范。

本文件也不单独授权建立、修改、迁移、删除或重命名代码。代码实施仍须具备：

```text
现行正式规范
-> 有效流程图与详细设计（适用时）
-> 已登记现行计划和任务队列
-> 执行前实际接口复核
-> 代码修改、实施记录和验证
```

若本文件与 `AGENTS.md`、现行正式规范、有效设计、现行计划或实际接口冲突，必须停止对应实施并退回设计治理；不得用本文件补造机器规则或扩大代码许可。

## 2. 总体工程裁决

```text
现有目录和代码文件默认原地保留，迁移必须另有专项计划。
新建数据结构文件使用同名“功能.分类.h/.cpp”。
其它新建代码文件使用真实 C++ 模块“功能.分类.ixx”。
文件路径、模块名、工程登记和调用方式必须一致。
生产实现与自检实现分离。
入口.cpp 只承载参数、模式、最小装配、顶层调用和退出码。
锁、工具函数、编码和命名只能服务已授权实现，不能承载或裁决机器事实。
```

## 3. 文件类型与建立方式

### 3.1 数据结构文件

只有同时满足以下条件，才可按数据结构文件建立：

```text
主要职责是声明稳定值类型、记录布局、容器布局或该结构自身的不变式。
需要由传统头文件向既有 .h / .cpp 代码公开完整类型。
实现内容只属于该结构本身，不承担领域编排、线程编排、初始化流程或自检流程。
```

数据结构文件使用：

```text
功能.分类.h
功能.分类.cpp
```

两者必须同名配对。只有声明且无需独立实现时，可以只建立 `.h`，但现行计划必须说明没有 `.cpp` 的原因。

以下内容不能仅因为内部使用结构体就归为数据结构文件：

```text
领域服务
仓库操作流程
协议复核流程
初始化流程
线程或上行桥
适配器
显示流程
自检与验收场景
```

### 3.2 真实模块文件

不属于 3.1 的新代码文件必须建立为：

```text
功能.分类.ixx
```

真实模块必须同时满足：

```text
文件包含唯一且匹配的 export module 声明。
工程以 ClCompile 登记该 .ixx。
使用方通过 import 使用模块。
不得把 .ixx 作为 ClInclude 登记后再用 #include 引入。
模块公开面只导出调用方需要的类型和入口。
```

仅使用 `.ixx` 扩展名但不满足上述条件，不视为真实模块，也不得作为新文件范例。

## 4. 文件名、路径与模块名

### 4.1 文件名格式

新文件基本名采用：

```text
功能.分类
```

第一段说明文件做什么，第二段说明文件服务于哪一类对象、领域或流程。例如：

```text
初始化.世界树
初始化.需求
上行桥.概念命名治理
自检.概念命名治理
```

不得把功能和分类揉成一个无法稳定拆分的长词，也不得用切片号、阶段号、临时任务号或完成状态作为生产文件名的一部分。

### 4.2 路径和模块名映射

文件放在现有职责目录，不因本文件重排目录。模块名由项目名、项目内相对目录、功能和分类逐段组成：

```text
文件：海中鱼巣/线程/上行桥.概念命名治理.ixx
模块：海中鱼巣.线程.上行桥.概念命名治理

文件：海中鱼巣/线程/自检.概念命名治理.ixx
模块：海中鱼巣.线程.自检.概念命名治理

文件：海中鱼巣/自检.运行器.ixx
模块：海中鱼巣.自检.运行器
```

文件路径、`export module` 名称、工程 `ClCompile` 登记和调用方 `import` 必须四处一致。

## 5. 职责、入口与实体所有权

### 5.1 生产模块与自检模块分离

生产逻辑和自检逻辑不得写入同一新模块：

```text
生产模块：功能.领域.ixx
自检模块：自检.领域.ixx
```

自检模块可以构造隔离测试材料、并发场景和人读验收输出，但不得成为生产调用路径、机器事实来源或运行期修复入口。生产模块不得为了自检导出可写内部状态；需要验证的行为必须通过正式公开入口或只读材料观察。

### 5.2 `入口.cpp` 边界

`入口.cpp` 只允许承载：

```text
命令行参数解析。
运行模式选择。
顶层模块导入与最小组合装配。
启动、停止、顶层调用和退出码映射。
按现行设计调用统一自检入口并汇总顶层结果。
```

`入口.cpp` 禁止新增：

```text
领域算法或领域服务实现。
协议构造与复核实现。
大段初始化实现。
完整自检场景正文。
并发压力测试循环正文。
逐项验收输出正文。
只为某个切片存在的局部业务类型和辅助算法。
```

自检接线必须服从现行统一自检设计；入口只能保留最小导入、装配和一次顶层调用，不得因接线方式变化把自检正文展开回入口。

### 5.3 命名模块实体单一所有权

同一个 C++ 类、结构、枚举、请求或结果实体只能由一个命名模块声明和定义。首次对外声明的模块是该实体的唯一所有者；其它命名模块不得通过同名跨模块前置声明、重复声明、文本包含实现或平行声明模块建立第二所有权。

工程分层固定为：

```text
服务.*.ixx：唯一拥有业务服务类、服务请求和服务模块自有的包装结果。
数据操作.*.ixx：唯一拥有数据操作类、结构写入规格、数据操作状态 / 结果和仓库读回值式材料。
组合.*.ixx / 路由.*.ixx：只 import 正式所有者，不重新声明服务或数据操作实体。
自检.*.ixx：只通过正式模块公开面构造对象，不成为生产实体友元或声明所有者。
```

服务可按值使用数据操作模块已经唯一归属的类型，但不得重复声明。数据操作模块不得前置声明跨模块业务服务，也不得把跨模块服务类设为私有规格的构造友元。

私有写入规格需要封闭构造时，构造权只能授予同模块的数据操作类。数据操作类可以提供窄规格形成入口供对应服务调用，但该入口不得接收服务模块定义的请求或结果，不得反向 `import` 服务模块，不得取得许可、写仓库或请求提交。

确需跨模块共享稳定值类型时，必须先裁决唯一结构所有者，再由其它模块 `import` 使用；不得为了消除链接错误复制声明、复用旧 BMI、文本包含实现或新增只声明不定义的平行服务模块。

## 6. 并发与锁工程边界

锁只保护运行期内存结构一致性，不承载机器事实，不生成动作来源，也不替代事务、任务授权、方法执行或因果证据。

### 6.1 仓库公开入口

```text
读入口使用共享锁，只读内部结构，返回记录副本、句柄组、只读材料或快照。
写入口使用独占锁，先完成正式规范要求的入口拒绝，再修改内部结构。
公开入口不得返回可写内部引用、长期裸指针或可跨锁生命周期持有的可变对象。
普通公开入口原则上只持有一个仓库锁；跨仓写入只由现行正式事务边界编排。
```

### 6.2 四仓固定锁序

确需在已授权事务实现中同时持有多个核心仓库锁时，只允许按以下顺序获取：

```text
节点仓库
-> 主信息仓库
-> 关系仓库
-> 索引仓库
```

禁止反向获取；禁止在持有后序仓库锁时再请求前序仓库锁；普通领域服务不得临时发明多仓同时持锁路径。无法按固定顺序完成时必须拆分，不得以忙等、循环重试或日志轮询充当防死锁机制。

### 6.3 禁止共享锁升级

公开入口不得在持有共享锁后升级为独占锁。读取后发现需要写入时，必须：

```text
取得只读快照或稳定句柄材料
-> 释放共享锁
-> 通过领域服务或正式事务入口重新进入
-> 重新执行身份、版本、权限和业务前置条件拒绝
-> 取得写入所需的独占锁并写入
```

快照不是持续有效的写许可；重新进入时不得沿用锁内旧判断跳过版本与前置条件复核。

### 6.4 内部 `已加锁` 辅助

`已加锁` 辅助函数只能由已经持有对应锁的同一仓库或同一对象内部函数调用，并同时满足：

```text
不再次获取同一把锁。
不暴露给领域服务、显示层、自检入口、全局工具或外部模块。
不调用可能反向进入本仓库公开入口的函数。
不跨仓库调用其它公开入口。
不把锁、锁守卫、可写引用或内部容器传出当前持锁范围。
```

需要跨仓协作时必须回到正式事务设计与固定锁序，不得用 `已加锁` 命名规避公开入口和事务边界。

### 6.5 持锁期间禁止的操作

持有仓库锁期间不得执行：

```text
上层领域服务调用。
显示、控制台输出、日志落盘或弹窗。
文件、数据库、网络或其它持久化 I/O。
外设访问、用户回调、线程等待或任务等待。
可能反向进入任一仓库公开入口的调用。
```

需要上述操作时，先在锁内形成最小副本、句柄组或快照，释放锁后执行；若随后需要写回，必须按 6.3 重新进入并复核。

## 7. 公共辅助函数归属与准入

### 7.1 候选分类

任何“公用函数”“公共辅助函数”或跨文件辅助入口必须先分类：

| 类型 | 定义 | 唯一落点 |
| --- | --- | --- |
| 纯通用函数 | 不依赖项目领域类型，不读写机器结构，只做通用计算、格式化、边界转换或轻量辅助 | 经计划确认的公共工具模块 |
| 仓库内部辅助 | 只服务同一仓库的结构维护、查找、重挂、快照、序列化或恢复准备 | 对应仓库内部私有或受控辅助入口 |
| 领域正式入口 | 涉及领域对象、业务准入、事实读取或结构写入 | 对应领域服务或仓库正式入口 |
| 流程编排 | 跨多个服务形成应用逻辑步骤 | 对应组合、路由或领域编排模块 |
| 禁止候选 | 绕过领域服务、仓库锁、入口拒绝、事务确认或计划范围 | 拒绝并退回设计治理 |

不得为了“迁移公用函数”主动搬运旧函数体、旧全局共享类或旧双向链表模板，也不得在没有具体调用方、用途、允许文件、禁止文件和验证方式时建立公共工具层。

### 7.2 触碰领域类型不得进入全局工具

只要候选函数直接接触以下任一项目结构或其句柄、容器、快照、事件材料，就不得归入全局纯通用工具：

```text
节点、主信息、关系、索引、特征、特征值、存在、场景、状态、动态、二次特征、因果引用、语素入口、需求、任务、方法、动作入口、非权威缓存、事件日志、仓库快照。
```

此类函数即使只读、只转换或仅被多个调用方复用，也必须回到对应仓库、领域服务、组合路由或受控序列化辅助的唯一所有者。不得用“无状态”“静态函数”“只读”或“方便复用”绕过领域归属。

纯通用工具只可处理不依赖项目领域类型的数值计算、边界转换，以及不承载机器事实的字符串、路径和诊断文本整理；不得创建、修改或删除项目结构，不得裁决需求满足、任务完成、方法成功或世界事实。

### 7.3 新增准入清单

新增或扩大任何公共辅助入口前，现行计划和设计至少必须明确：

```text
1. 已搜索当前代码，确认不存在同等功能入口。
2. 用途、唯一所有者、允许调用方和禁止调用方。
3. 读写类型：纯计算、只读、候选读取、写结构、序列化或恢复。
4. 触碰的类型、结构及其正式权威来源。
5. 入口前置拒绝和内部逻辑错误边界。
6. 拒绝后结构不变化的验证方式。
7. 是否需要锁、由谁持锁、是否存在内部已加锁辅助和固定锁序。
8. 生命周期：临时材料、运行期结构、持久化快照或审计材料。
9. 允许文件、禁止文件、公开面和调用路径。
10. 构建、自检、回归和失败收口方式。
```

已有仓库、领域服务或组合模块可以承载时，优先扩展既有入口；执行窗口不得临时改变分类或新建全局替代入口。

## 8. 快照、序列化与恢复辅助门禁

### 8.1 只读快照

只读、快照和候选读取函数只返回记录副本、句柄组、只读材料、非权威候选或快照结构。禁止返回可写内部引用、长期裸指针、可绕过仓库入口修改的容器引用，或会被上层误认为权威事实的临时缓存。

只读访问不得隐式创建、修补或确保结构存在。需要写入时必须走正式入口，并按 6.3 释放读取锁、重新进入和复核。

### 8.2 序列化辅助

触碰项目领域类型的序列化、快照或格式辅助属于对应仓库或对象所有者的受控辅助，不得进入全局纯通用工具。其设计和计划必须明确：

```text
格式标识与版本。
记录数量、字节长度和容器容量上限。
必需字段、可选字段和字段边界。
句柄、版本、关系端点和索引表示。
截断、重复、越界、未知版本和多余尾部的处理。
写出失败与读取失败后的临时材料清理。
```

序列化结果、文件和数据库记录只按现行正式规范作为快照、恢复候选、审计或人读材料处理；本文件不赋予其运行期事实身份。

### 8.3 恢复辅助

恢复辅助只负责形成和校验恢复候选，不得绕过正式恢复入口直接发布。替换当前结构或请求提交前，必须完成：

```text
格式和版本校验。
数量、长度、容量和字段边界校验。
稳定身份、句柄生命周期和版本校验。
关系端点、索引边界和结构一致性校验。
现行正式规范要求的业务准入与恢复拒绝矩阵。
失败候选和临时容器清理。
正式发布后的结构读回验证。
```

禁止先替换当前结构、先发布半结构或先污染索引，再依靠扫描器、日志、弹窗或事后校验发现坏数据。恢复准入、提交、发布和回退语义必须由现行正式规范、有效设计和计划明确；公共辅助函数不得自行裁决。

## 9. UTF-8 与中文标识符

### 9.1 文件编码

```text
Markdown 文档和 C++ 源文件必须是有效 UTF-8。
Markdown 和 C++ 源文件允许带 BOM 或不带 BOM，但不得包含非 UTF-8 字节；不得因普通编辑无意义地反复切换 BOM。
工程 XML 文件必须保持 XML 声明 encoding="utf-8"，并以 UTF-8 保存。
PowerShell 读取或写出中文文本时必须显式使用 UTF8 编码参数。
搜索中文路径和内容优先使用 rg。
源文件、文档、工程文件和日志格式不得依赖 Windows 本地代码页解释。
```

Windows 本地代码页只可作为外部接口或终端显示的受控边界。确需与本地代码页 API 交互时，必须在边界处显式转换，不得让本地代码页字节进入 UTF-8 文件、模块名、项目路径或机器结构。

### 9.2 中文标识符与允许英文边界

函数名、变量名和类内业务成员名必须使用中文。新建的业务类名、结构名、枚举名、命名空间名和项目文件名应优先使用中文；不得仅为输入方便、工具习惯或临时规避编码问题新增英文业务命名。

英文只用于：

```text
C++ 关键字和语言要求的固定拼写。
标准库类型与标准 API。
第三方 API、外部 SDK 和不可控制的外部类型。
构建系统、工程 XML 和工具链固定字段。
不可避免的外部协议固定字段。
```

确需保留外部英文名称时，边界适配代码必须说明其外部来源和不可改名原因；进入本项目业务公开面后不得继续扩散为新的英文业务命名。不得为了显示临时文本新增机器字段，也不得把乱码标识符、乱码路径或乱码输出作为验收证据。

### 9.3 工程配置与乱码验收

`海中鱼巣.vcxproj` 的以下四个配置必须都保留 `/utf-8` 编译选项：

```text
Debug | Win32
Debug | x64
Release | Win32
Release | x64
```

任何工程配置修改都必须逐配置复核，不能用单一 Debug x64 构建代替四配置工程选项检查。现行计划要求完整构建时，应在授权角色和允许环境中验证对应配置；本文件本身不授权构建。

编码验收至少确认：

```text
中文路径和文件内容可按 UTF-8 无损读取。
中文标识符在计划要求的配置中可编译。
```

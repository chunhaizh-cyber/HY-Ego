# 代码文件建立归属与模块命名规范

编号：CODE-FILE-01

日期：2026-07-11

状态：正式规范 / 自 JY-253 起生效

## 1. 目的

本规范解决三个问题：

```text
新功能默认堆入入口.cpp。
新建 .h / .cpp / .ixx 缺少统一分类依据。
文件名、工程登记方式和真实编译形态可以彼此不一致。
```

本规范只调整以后建立代码文件的规则，不要求批量改名现有文件，不改变现有目录结构。

## 2. 总体裁决

```text
现有目录结构保持。
现有代码文件原地保留，除非另有专项计划迁移。
新建数据结构文件继续使用同名 .h / .cpp 配对。
除数据结构文件外，后续新建代码文件一律使用真实 C++ 模块文件 .ixx。
所有后续新建代码文件采用“功能.分类”文件名。
入口.cpp 不再作为领域实现、自检正文或功能试验代码的默认承载文件。
```

## 3. 文件类型分类

### 3.1 数据结构文件

只有同时满足以下条件，才可按数据结构文件建立：

```text
主要职责是声明稳定值类型、记录布局、容器布局或该结构自身的不变式。
需要由传统头文件向既有 .h / .cpp 代码公开完整类型。
实现内容只属于该结构本身，不承担领域编排、线程编排、初始化流程或自检流程。
```

数据结构文件使用：

```text
功能.分类.h
功能.分类.cpp
```

两者必须同名配对。只有声明且无需独立实现时，可以只建立 `.h`，但计划必须说明没有 `.cpp` 的原因。

以下内容不能仅因为内部使用结构体就归为数据结构文件：

```text
领域服务
仓库操作流程
协议复核流程
初始化流程
线程或上行桥
适配器
显示流程
自检与验收场景
```

### 3.2 模块文件

不属于 3.1 的新代码文件必须建立为真实 C++ 模块：

```text
功能.分类.ixx
```

真实模块必须同时满足：

```text
文件包含 export module 声明。
工程以 ClCompile 登记该 .ixx。
使用方通过 import 使用模块。
不得把 .ixx 作为 ClInclude 登记后再用 #include 引入。
模块公开面只导出调用方需要的类型和入口。
```

仅使用 `.ixx` 扩展名但没有上述三项，不视为本规范所称模块。

## 4. 文件命名

### 4.1 基本格式

文件基本名采用：

```text
功能.分类
```

语义为：

```text
第一段说明文件做什么。
第二段说明文件服务于哪一类对象、领域或流程。
```

示例：

```text
初始化.世界树
初始化.需求
上行桥.概念命名治理
自检.概念命名治理
```

不再为同一新文件使用把功能和分类揉成一个长词的命名，例如：

```text
概念命名治理上行桥
概念命名治理自检
```

### 4.2 目录和模块名映射

文件继续放在现有职责目录，不为本规范重排目录。例如线程协议、线程桥和对应自检模块继续放在 `海中鱼巣/线程/`。

模块名必须由项目名、现有相对目录、功能和分类逐段组成：

```text
文件：海中鱼巣/线程/上行桥.概念命名治理.ixx
模块：海中鱼巣.线程.上行桥.概念命名治理

文件：海中鱼巣/线程/自检.概念命名治理.ixx
模块：海中鱼巣.线程.自检.概念命名治理
```

根目录模块省略目录段，但仍保留 `功能.分类`：

```text
文件：海中鱼巣/自检.运行器.ixx
模块：海中鱼巣.自检.运行器
```

文件路径、`export module` 名称、工程登记和调用方 `import` 必须四处一致。

## 5. 职责归属

### 5.1 生产模块与自检模块分离

生产逻辑和自检逻辑不得写进同一新模块。

```text
生产模块：功能.领域.ixx
自检模块：自检.领域.ixx
```

自检模块可以构造隔离测试材料、并发场景和人读验收输出，但不得成为生产调用路径、机器事实来源或运行期修复入口。

生产模块不得为了自检导出可写内部状态。需要测试的行为必须通过正式公开入口或只读材料验证。

### 5.2 入口.cpp 边界

`入口.cpp` 只允许承载：

```text
命令行参数解析。
运行模式选择。
顶层模块导入与最小组合装配。
启动、停止和退出码映射。
在统一自检运行器形成前，对独立自检模块的一次调用与结果汇总。
```

`入口.cpp` 禁止新增：

```text
领域算法或领域服务实现。
协议构造与复核实现。
大段初始化实现。
完整自检场景正文。
并发压力测试循环正文。
逐项验收输出正文。
只为某个切片存在的局部业务类型和辅助算法。
```

在统一 `自检.运行器` 尚未建立前，新切片可以在 `入口.cpp` 增加一个模块 `import`、一个上下文组装和一次自检调用；不得把自检步骤展开回入口。统一运行器建立后，新自检只向运行器登记，不再直接增加入口调用。

## 6. 既有文件兼容

### 6.1 原地保留

本规范不自动要求：

```text
重命名既有 .h / .cpp / .ixx。
把既有头文件全部转换成模块。
移动现有目录。
一次性拆分入口.cpp。
```

既有文件只有在专用计划明确允许文件、调用迁移、工程登记、回归和回退边界后才可迁移。

### 6.2 既有伪模块

现有以 `.ixx` 命名、但由工程按 `ClInclude` 登记并由 `#include` 使用的文件属于历史兼容文件。它们可以继续存在，但不得作为以后新文件的模块范例，也不得据此宣称模块化已完成。

### 6.3 不恶化门禁

自 JY-253 起：

```text
任何新计划不得继续把新领域实现或完整自检正文默认追加到入口.cpp。
任何新建非数据结构代码文件不得使用新的传统 .h / .cpp 实现对。
任何旧计划若与本规范冲突，执行前必须退回设计窗口修订允许文件和自检承载方式。
```

## 7. 计划与实施门禁

涉及新建代码文件的计划必须列出文件分类表：

| 文件 | 分类 | 命名依据 | 工程登记 | 调用方式 | 写入方 / 读取方 |
| --- | --- | --- | --- | --- | --- |
| `功能.分类.h/.cpp` | 数据结构 | 3.1 | `ClInclude` / `ClCompile` | `#include` | 计划明确 |
| `功能.分类.ixx` | 模块 | 3.2 | `ClCompile` | `import` | 计划明确 |

执行前必须复核：

```text
文件确实属于计划声明的类型。
文件名符合“功能.分类”。
模块声明、工程登记和 import 一致。
生产和自检未混写。
入口.cpp 只有最小装配和调用增量。
旧路径、旧模块名和重复兼容文件没有同时进入工程。
```

任一项不一致时不得在执行窗口自行改义，必须退回设计计划窗口修订。

## 8. 验收

新代码文件切片至少验证：

```text
工程中模块按 ClCompile 登记。
模块含唯一且匹配的 export module。
调用方使用匹配的 import。
不存在对新 .ixx 的 #include。
数据结构 .h / .cpp 同名或已说明单头原因。
git diff 证明入口.cpp 没有新增功能正文或自检正文。
Debug x64 构建通过。
计划要求的完整自检和回归通过。
```

文档治理只能声明规则和计划已建立，不得声明既有入口已经拆分或既有文件已经模块化。

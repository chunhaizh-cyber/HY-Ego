## 2. 基础信息服务分层

基础信息部分分为原子服务和聚合服务。

基础信息原子服务为：

```text
特征值服务
特征服务
存在服务
场景服务
状态服务
动态服务
二次特征服务
因果服务
```

基础信息聚合服务为：

```text
世界服务
概念图服务
```

世界服务负责组织基础信息服务形成世界结构，不直接绕过原子服务写节点仓库、主信息仓库、关系仓库或索引仓库。

概念图服务负责组织四类基础信息形成概念图，独占写入 `实例支持概念`、`概念上下位`、`概念定义材料` 和 `概念生命周期` 四类关系。生命周期目标必须由状态服务创建为抽象状态后登记；名称入口只能经语素服务反向读取 `语素概念追溯`。概念图服务不创建需求、任务或方法，不直接访问特征值服务；需要特征值材料时只能经特征服务取得结构化值副本。

## 3. 原子服务职责

| 服务 | 主要职责 | 写入边界 |
| --- | --- | --- |
| 特征值服务 | 为特征承载当前值、历史值或值句柄 | 只作为特征服务内部值承载实现，不作为需求、任务、方法的通用值容器 |
| 特征服务 | 创建和管理特征类型、实例特征入口、特征状态材料读写入口 | 特征值节点的创建、读取和写入必须由特征服务对外封口 |
| 存在服务 | 创建和管理实际存在、虚拟存在 | 任务、方法、需求的虚拟存在必须通过存在服务获得 |
| 场景服务 | 创建场景，收纳场景内临时实例状态和实例动态 | 负责场景生命周期内的临时零散节点 |
| 状态服务 | 管理实例状态和抽象状态的创建、读取和提升边界 | 实例状态不得直接长期化，抽象状态才可长期独立存储 |
| 动态服务 | 管理实例动态和抽象动态的创建、读取和提升边界 | 实例动态不得直接长期化，抽象动态才可长期独立存储 |
| 二次特征服务 | 管理关系性、程度性或复合判断前置特征 | 归特征体系，不归世界服务直接创建 |
| 因果服务 | 管理因果引用、因果证据和动态抽象候选 | 不直接代替动态服务创建无主体动态 |

## 4. 世界服务边界

世界服务是基础信息聚合服务。

世界服务可以组织以下流程：

```text
创建现实场景
创建场景中的存在
把存在放入场景
给存在挂接特征
让场景收纳实例状态和实例动态
调用因果服务记录因果引用或证据候选
```

世界服务不得直接替代原子服务执行以下写入：

```text
直接创建存在而绕过存在服务
直接创建场景而绕过场景服务
直接创建特征或二次特征而绕过特征服务 / 二次特征服务
直接创建状态或动态而绕过状态服务 / 动态服务
直接写特征值而绕过特征服务
直接写因果引用而绕过因果服务
```

第一轮代码已将 `世界服务::创建存在()` 和 `世界服务::创建场景()` 改为调用存在服务和场景服务。当前仍保留 `世界服务::创建基础信息()` 兼容壳直接创建基础信息通用节点，只用于第一轮语素样例和未拆入具体类型的基础信息通用节点，不得扩大解释为完整世界服务完成。

## 4.1 概念图服务边界

概念图服务是基础信息聚合服务。第一轮运行期边界固定为：

```text
登记和读取存在、动态、关系、因果四类概念根。
根身份使用根类别、非名称稳定键和稳定节点句柄共同承载。
索引命中只作候选，登记前必须重读节点类型、有效状态和当前登记。
专用关系写入前完成类别兼容、自环、已知成环和定义材料白名单复核。
专用关系读取只返回句柄或关系记录值副本，不暴露仓库内部容器。
活跃、冷却、退役三个生命周期状态必须由状态服务创建为互异抽象状态并按固定状态值登记。
四根和新发布概念进入活动快照前必须且只能具有一个活跃生命周期关系；内部发布失败必须清理本轮生命周期关系和候选结构。
待命名请求与临时指代只返回值式材料，不创建需求、名称事实、统计事实或持久化指代节点。
```

概念图服务不得绕过存在、动态、二次特征、因果或状态服务创建对应节点；不得用名称、日志、控制台、显示文本、SQL 行或内容哈希裁决概念身份。当前已建立四根、固定点候选、活动图发布、抽象树投影和生命周期初态；跨重启恢复、用途观察、生命周期转换、退役删除和命名需求仍不在本轮完成范围内。

## 5. 实例信息和抽象信息

状态和动态必须分为实例层和抽象层。

实例状态：

```text
某个存在在某个场景内的临时状态。
以场景内临时零散节点存储。
必须指向存在主体。
必须带发生时间戳。
默认不长期独立存储。
不得直接作为语素入口对应信息节点。
```

实例动态：

```text
某个存在在某个场景内发生的临时变化记录。
以场景内临时零散节点存储。
必须指向存在主体、来源动作或被改变目标。
必须带发生时间戳。
默认不长期独立存储。
不得直接作为语素入口对应信息节点。
```

抽象状态：

```text
由实例状态抽象、确认或归纳形成的可复用状态类型、状态模式或状态判断结果。
可作为长期基础信息类节点存储。
可被语素入口绑定。
可被需求、任务、方法引用。
本体不携带发生时间戳。
```

抽象动态：

```text
由实例动态抽象、确认、归档或因果证据提升形成的可复用变化类型、动作模式、因果模式或动态模板。
可作为长期基础信息类节点存储。
可被语素入口绑定。
可被方法、任务、因果服务引用。
本体不携带发生时间戳。
```

场景负责实例状态和实例动态的临时收纳和生命周期管理；存在是实例状态和实例动态的主体；发生时间戳是实例状态和实例动态的必填事实材料；抽象状态和抽象动态才是长期基础信息节点。

## 6. 特征值服务限制

特征值服务只服务于特征系统，只有特征服务可直接访问。

允许：

```text
特征服务创建目标特征。
特征服务经特征值服务为目标特征写入当前状态材料。
方法服务通过特征服务改变目标特征状态材料。
需求服务通过特征服务读取目标特征状态材料进行结算。
```

禁止：

```text
把特征值服务当作通用 I64 容器。
用特征值节点直接保存需求目标状态。
用特征值节点直接保存任务状态。
用特征值节点直接保存方法执行结果。
需求、任务、方法服务直接包含、接收或调用特征值服务。
让语素入口直接绑定未经特征值系统规范确认的特征值节点。
```

需求目标是目标状态，不是 I64。需求目标、任务状态和方法结果必须由各自高级服务或状态 / 动态 / 因果服务承载，必要时引用特征值节点也必须经特征服务，不得把特征值服务泛化为所有业务值容器。

## 7. 高级信息服务依赖

高级信息服务包括：

```text
需求服务
任务服务
方法服务
```

高级信息服务不得直接创建或修改基础信息结构。需要基础信息能力时，只允许直接依赖存在服务、场景服务、特征服务、二次特征服务、状态服务、动态服务和因果服务。

任务服务依赖：

```text
存在服务：创建或读取任务虚拟存在。
场景服务：收纳任务实例状态和任务实例动态。
特征服务：读取任务目标特征、条件特征和结果特征。
状态服务：记录任务实例状态或引用抽象状态。
动态服务：记录任务推进动态。
```

方法服务依赖：

```text
存在服务：创建或读取方法虚拟存在。
场景服务：记录方法执行发生环境和实例动态。
特征服务：确认作用目标特征和输出特征。
特征服务：通过特征服务修改目标特征状态材料。
动态服务：记录方法执行动态。
因果服务：记录方法造成的变化引用。
```

需求服务依赖：

```text
存在服务：创建或读取需求虚拟存在。
场景服务：收纳需求实例状态。
特征服务：读取需求目标特征和当前特征状态材料。
状态服务：记录需求结算状态。
```

## 8. 语素服务边界

语素服务只负责文本入口到已存在信息节点的绑定。

语素服务不得：

```text
创建基础信息业务节点。
创建需求、任务或方法节点。
把临时实例状态或实例动态直接绑定为长期语素对应信息。
绕过基础信息服务创建特征、状态、动态、二次特征或因果引用。
```

语素入口可绑定的长期信息节点必须已经由对应服务创建，并符合当前语素系统规范。


当前仍保留以下兼容缺口：

```text
世界服务::创建基础信息() 仍直接创建基础信息通用节点，用于第一轮语素样例和未拆入具体基础信息类型的兼容入口。

高级信息服务仍通过函数参数接收基础信息服务能力，尚未形成更完整的依赖注入治理。
```

这些只证明第一轮服务分层代码重构和入口拒绝样例通过，不证明本规范中的服务分层已经全部完成。

```text

无发生时间戳实例状态 / 动态拒绝。

```

## 10. 验证方式

本规范进入代码实施后，至少需要以下验收：

```text
存在服务、场景服务、特征服务、状态服务、动态服务、二次特征服务、因果服务均有高级服务可用入口；特征值服务只作为特征服务内部入口。
世界服务只编排原子服务，不直接绕过原子服务写仓库。
任务服务、方法服务和需求服务通过允许的基础信息服务获得存在、场景、特征、状态、动态和因果能力。
实例状态和实例动态必须有场景和存在主体。
实例状态和实例动态必须有发生时间戳，缺时间戳必须在函数入口拒绝。
抽象状态和抽象动态可以长期存储并可作为语素入口对应信息。
抽象状态和抽象动态本体不携带发生时间戳。
特征值服务只接受特征服务发起的特征相关值写入。
控制台输出只展示验证结果，不承载机器事实。
```

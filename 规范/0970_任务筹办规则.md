# 0970 任务筹办规则

## 1. 目标

本文用于固定以下问题：

- 什么叫 `筹办`
- 筹办的本质步骤是什么
- `找条件` 应按什么顺序进行
- 什么叫 `自我可用`
- 筹办结束后允许产出哪些结果

本文回答的是：

- 筹办为什么只是任务控制四大功能域之一
- 筹办为什么不等于执行
- 方法候选如何进入下一步
- 条件缺口如何被表达和分流
- 当前场景、世界树、学习结果在筹办中各自处于什么位置

## 3. 核心结论

必须固定以下结论：

1. `筹办` 是任务控制四大功能域之一，不等于整个任务控制。
2. `筹办` 不等于执行，筹办只负责组织下一步执行路径。
3. 任务控制的完整骨架应为：`筹办 + 维护 + 纠偏 + 收束`。
4. 生长、细分、铺路、初选、初始化主要属于 `筹办` 内部能力。
5. `筹办` 的本质步骤只有两个：`找方法` 与 `找条件`。
6. `找方法` 回答“有没有能朝当前需求推进的方法候选”。
7. `找条件` 回答“这些方法当前能不能做；如果不能，缺什么”。
8. `自我可用 = 可获取 + 有权限使用`。
9. `可获取` 与 `有权限使用` 缺一不可。
10. 尝试会产生可用性证据，学习会把这些证据沉淀成可复用概念或清单。
11. 学习结果可以形成 `私有概念`、`全局概念` 或 `公用概念`，但它们不是当前真值本身。
12. 当前可用性应由 `学习沉淀结果` 与 `当前场景复核` 共同决定。
13. 筹办结束后，只允许产出：`可执行步骤`、`补条件结果`、`观察/试探结果`、`待重筹办结论`、`无法执行结论`。

可收敛为一句话：

`筹办 = 先找能推进需求的方法，再找这些方法当前可用的条件，并把结果落成可执行步骤、补条件任务、观察任务或重筹办结论。`


## 4. 筹办的定位

### 4.1 筹办是任务控制的一个功能域

任务控制应固定理解为：

```text
任务控制
= 筹办
+ 维护
+ 纠偏
+ 收束
```

其中：

- `筹办` 负责把任务立起来，并组织成一个能跑的样子
- `维护` 负责让已建立的任务链保持健康和可执行
- `纠偏` 负责在失败、漂移、失配时把任务拉回来
- `收束` 负责完成判定、清尾、学习写回和归档

因此本文只覆盖其中的 `筹办` 主域，不替代维护、纠偏或收束规则。

### 4.2 筹办不是执行

筹办必须固定为：

- 任务在当前阶段为获得下一步执行路径而进行的组织过程

因此：

- 筹办不直接完成需求
- 筹办不直接产出最终结果
- 筹办只负责回答“下一步该做什么，或为什么现在还做不了”

### 4.3 筹办不是整个任务控制

筹办内部主要包含：

```text
生成 + 细分 + 铺路 + 初选 + 初始化
```

因此：

- 任务生长和完善主要属于筹办内部能力
- 筹办还是任务层经验积累与智力进化的主域：新路径、选法、补条件套路与风险边界主要在这里沉淀为后续可复用经验
- 维护、纠偏、收束不应再被并入“筹办”概念
- `细分 / 延长 / 插检查` 更适合作为筹办内可调用的执行动作

### 4.4 筹办的输入

筹办至少应读取：

- 当前任务头
- 当前绑定需求
- 当前任务场景
- 世界树
- 方法库
- 当前时间与时效要求

### 4.5 筹办的输出

筹办至少应输出：

- 候选方法列表
- 当前选中方法
- 条件判定清单
- 条件缺口清单
- 新增子需求或子任务
- 当前筹办结论


## 5. 找方法

### 5.1 目标

`找方法` 用于回答：

- 是否存在能朝当前需求推进的方法候选

### 5.2 最小规则

找方法时，至少应比较：

1. 方法结果是否作用于当前需求目标维度
2. 方法结果方向是否朝需求目标推进
3. 方法是否被当前规则、规避过滤或边界约束排除

### 5.3 找方法的结果

找方法结束后，方法候选至少应被划分为：

- `可继续找条件的方法`
- `已被排除的方法`

此时仍不能直接判定方法“可执行”，因为条件部分尚未完成。


## 6. 找条件

### 6.1 目标

`找条件` 用于回答：

- 当前方法要成立，需要哪些存在、特征、状态或关系
- 这些条件当前是否对自我可用

### 6.2 查找顺序

当前应固定以下查找顺序：

1. 先在 `自我所在场景` 中查找
2. 场景中找不到，再到 `世界树` 中查找
3. 找到对象后，再读取该对象当前的现状值
4. 将现状值代入方法条件模板

这里的关键是：

- 先找对象
- 再找值
- 不允许只凭抽象名称跳过对象定位

### 6.3 条件对象

方法条件中需要查找的对象，至少可以包括：

- `存在类型`
- `特征类型`
- `状态类型`
- `二次特征或关系类型`

### 6.4 条件判定结果

当前条件判定至少应输出以下几类：

1. `已满足`
2. `值未知，需先观察`
3. `未满足但可补`
4. `对象或特征存在，但当前不可获取`
5. `对象可获取，但当前无权限使用`
6. `规则禁止`
7. `时效不允许`

### 6.5 条件缺口

`条件缺口` 指的是：

- 当前方法成立所需条件中，尚未满足、但仍可能通过后续动作补齐的部分

因此：

- `未满足但可补` 可以进入条件缺口清单
- `值未知，需先观察` 可以进入观察缺口清单
- `规则禁止` 不属于可补缺口
- `时效不允许` 不属于可补缺口


## 7. 自我可用

### 7.1 定义

当前应固定：

- `自我可用 = 可获取 + 有权限使用`

### 7.2 可获取

`可获取` 指的是：

- 自我当前能否找到、接触、读取或调用该对象、特征、状态或方法

它至少包括：

- 对象是否存在
- 对象是否当前可达
- 特征值是否能被读到
- 方法是否能被调用

### 7.3 有权限使用

`有权限使用` 指的是：

- 即使对象或值当前可获取，系统规则、角色边界、安全约束与当前上下文是否允许使用

它至少包括：

- 是否违反安全边界
- 是否被当前规避规则禁止
- 是否被当前运行态禁止

### 7.4 最小分类

当前建议把自我可用性至少分成：

1. `可用`
2. `不可获取`
3. `无权限`
4. `未知待验证`

### 7.5 当前真值与学习结果

必须固定：

- 学习结果不是当前真值本身
- 当前可用性必须由 `学习沉淀结果` 与 `当前场景复核` 共同决定

因此：

- 历史上曾经可用，不等于当前可用
- 历史上曾经不可用，也不等于当前永久不可用


## 8. 尝试、学习与概念范围

### 8.1 尝试

`尝试` 的作用是：

- 为某个对象、方法或条件关系产生一次可用性证据

### 8.2 学习

`学习` 的作用是：

- 把多次尝试结果沉淀成稳定经验、清单或概念候选

### 8.3 概念范围

学习结果在作用范围上可以分为：

1. `私有概念`
   只在某个局部任务、局部场景或局部对象范围内稳定成立。

2. `全局概念`
   在当前自我内部多个任务、多个场景中稳定成立。

3. `公用概念`
   不只对当前自我成立，还具有稳定的跨主体或对外复用价值。

### 8.4 与筹办的关系

筹办在使用这些学习结果时，必须固定：

- 可以把它们作为快速参考
- 不能把它们当成无需复核的当前真值

### 8.5 尝试学习与首步生成的衔接

必须固定：

- `尝试学习` 的职责不是直接替筹办发明步骤，而是把本能函数或既有方法继续实例化、运行并沉淀成可召回的方法树
- 尝试学习完成后，方法树中至少应逐步补出 `条件节点 / 结果节点 / 影响签名`
- 筹办兜底在生成第一个步骤时，优先使用这些已学习的方法树进行目标匹配、条件分析与候选排序
- 只要已有方法树中的结果特征或影响签名能够命中任务目标，筹办兜底就允许生成第一个步骤
- `现有全部本能函数都已被尝试学习过` 不是单独的充分条件；任务目标仍必须能与方法结果或影响签名建立关联，且候选方法不能全部落入 `不可获取 / 无权限 / 彻底禁止`

因此应统一理解为：

> 尝试学习负责把本能函数长成可筹办的方法树，首步生成仍由筹办兜底完成；学习提供可选路径，筹办负责真正落成第一个步骤。


## 9. 筹办结果分流

### 9.1 可执行步骤

若：

- 已找到方法
- 且该方法条件已满足

则：

- 生成可执行步骤

### 9.2 补条件结果

若：

- 已找到方法
- 但存在 `未满足但可补` 的条件缺口

则：

- 生成补条件子需求或子任务

### 9.3 观察或试探结果

若：

- 已找到方法
- 但关键条件属于 `值未知，需先观察`

则：

- 生成观察任务或最小试探任务

### 9.4 重筹办结论

若：

- 当前未形成可执行路径
- 但后续仍可能因学习、观察、外部输入或条件变化而形成路径

则：

- 进入 `待重筹办`

### 9.5 无法执行结论

若：

- 当前找不到方法
- 或所有候选方法都被规则禁止、时效否决或彻底不可获取

则：

- 进入 `无法执行`


## 10. 状态约束

后续实现至少应遵守：

1. `就绪 / 待重筹办 -> 筹办中`
2. 筹办结束后，任务只允许进入：
   - `就绪`
   - `等待中`
   - `待重筹办`
   - `无法执行`
3. 筹办本身不直接跨步执行

### 10.1 真根任务与普通任务

筹办过程本身对真根任务与普通任务应尽量统一。

当前差别只允许出现在收口：

- 真根任务筹办失败后优先回到 `待重筹办`
- 普通任务筹办失败后可进入 `无法执行`


## 11. 结论

后续任务系统应固定：

1. 筹办的本质步骤只有 `找方法` 与 `找条件`。
2. 找条件时，应先在自我所在场景中找对象，再到世界树中找对象，然后再读取其当前值。
3. `自我可用` 的最小判定是 `可获取 + 有权限使用`。
4. 尝试提供可用性证据，学习沉淀为可复用经验或概念。
5. 学习结果可以形成 `私有概念`、`全局概念`、`公用概念`，但不能直接替代当前场景真值。
6. 尝试学习负责把本能函数或既有方法沉淀成可召回的方法树，而不是直接替筹办生成步骤。
7. 第一个步骤的生成权仍属于筹办；当目标能够命中已学习方法树的结果特征或影响签名时，筹办兜底应允许生成首步。
8. 筹办结束后，只应把结果落到可执行步骤、补条件任务、观察任务、待重筹办或无法执行结论之一。

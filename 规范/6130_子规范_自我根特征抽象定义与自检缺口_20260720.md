# 6130 子规范 自我根特征抽象定义与自检缺口

编号：6130
日期：20260724
状态：正式子规范

## 1. 定位

本规范规定自我安全、服务等根特征的抽象定义、自我实例、自检读取、缺口候选、因果学习和需求准入边界。

直接上级规范：1120、1130、1140

直接依赖规范：5130、6100、6120

相关边界规范：1160、1170、1190、3100、6140—6170

## 2. 根特征定义

安全、服务和其它自我根特征必须先成为正式抽象特征定义，至少明确：

- 稳定特征身份、值域和比较规则；
- 根目标状态和合法生产者；
- 当前值、候选值、明确性、搜索完整性和有效期；
- 状态、动态、二次特征和因果如何组织；
- 缺口对应何种需求目标；
- 事实如何验证、归因和回写。

安全根定义还必须声明分层安全值定义模板：I64 值域、低位阈值、高位阈值、被动维护规则版本、对应目标安全结果和合法证据类型。具体安全层级不进入根定义固定表，而由来源因果信息相对于目标安全结果的当前直接因果图投影得到。

自然语言名称只用于人类定位。固定 C++ 函数表、字段名、显示阈值和日志不得成为根定义权威。

## 3. 不得硬编码安全或服务因素表

安全定义不得预置硬件、软件、资源、环境、权限、观察、风险或服务等固定必要因素表，也不得按固定配额分割物理安全区间。

正式影响因素来自抽象特征树、出生期因果假设和后天证据增强后的因果信息。安全层只按已正式成立、方向明确且版本有效的直接因果关系动态投影，不得预先列举某类需求或任务固定属于哪一层。字段名称可以存在于事实材料、对象方法输出或提交结果中，但不能因字段存在就自动生需或成为安全组成因。

服务值定义固定值域为 `0..I64_MAX`，并冻结 6160、6170 的合同、完整秒、预算、衰减和安全回归规则版本；不得把当前交互者、服务对象信息、反馈、验证或归因预置为启动期永久必要条件。它们只有在具体服务触发和动作后才成为上下文事实或缺口。

## 4. 自我实例和运行期视图

自我实例通过当前自我存在稳定句柄，关联根特征实例槽、当前状态和根闭环因果坐标。运行期读取结果可以使用值式视图汇总：

```text
根特征定义身份
+ 自我实例槽
+ 当前状态和值
+ 适用范围、来源和版本
+ 当前因果证据和缺口
```

值式视图是按需组合读取投影，不是新增持久字段或第二权威账。已有抽象定义、关系、状态、动态、二次特征和因果能表达时必须复用；自我实例结构服从 7130。

安全层读取投影可以追加来源因果、目标安全结果、因果图版本、全部有效路径、最短精确深度、五级调度组和当前层安全值规格，但这些项目必须分别回读正式因果、状态和领域类型化记录；间接因果、固定层号和汇总列表不得成为第二权威事实。

## 5. 证据与因果

状态和动态是安全/服务证据本体。特征值只有被组织到具体存在、场景、状态、动态和因果中，才具有安全或服务含义。

历史直接伤害应先形成或更新负向安全因果信息，不另建历史伤害直查旁路。出生期因果只表达带来源动态证据或出生期假设标记的被动触发结构；主动动作能力来自本能方法和方法系统，不能由遗传因果直接伪造。

后天证据可以强化、削弱、分化或覆盖出生期被动因果。因果候选只保存稳定证据引用和候选归因，不表示存在本体自带安全度或伤害概率。

## 6. 安全评估视图

存在、场景和风险安全值均为按当前事实计算的候选视图，不是对象固有字段。若没有历史风险材料，历史值为零只表示没有材料，不表示安全。

完成正式搜索而未发现负向证据时，只能提交带范围、来源、有效期和搜索完整状态的无负向证据结果。未知不得写成危险，也不得用 `I64_MAX` 写成默认安全。

场景中多个已确认存在属于同层风险因素时，可以先按统一方向和版本聚合风险压力，再转换为风险余量候选；最低安全度存在只作诊断和处置线索，不单独裁决场景或自我安全值。

评估结果不能直接结算自我生存安全根值。它只能生成风险因子、处置需求候选、持续观察或降级保护候选；生存安全根值的主动变化仍经根控制、因果归因和 6100 结算，服务调节的控制回归只经 6170。具备复发机会、有效感知覆盖、场景 / 因素身份可比和确认未复发证据时，评估结果可以提交给 6140 作为分层安全值被动维护材料，但不得直接写值。

## 7. 自检流程

```text
读取当前自我稳定身份
-> 读取根特征抽象定义和版本
-> 定位自我实例特征槽
-> 读取当前状态、搜索完整性和有效期
-> 按定义比较当前状态与根目标
-> 查询原因、可改变性、可验证性和回写路径
-> 形成当前无缺口或强类型缺口候选
-> 交需求治理入口裁决
```

自检只读取权威结构并形成候选；不得直接写需求树、创建任务、创建方法首或修改根值。

## 8. 缺口候选合同

缺口候选至少包含自我身份、根特征定义、实例槽、当前状态、目标状态、比较关系、证据状态、来源因果、目标安全结果、因果图版本、闭环位置、可改变性、验证方式、建议需求类型和有效期。精确因果深度和五级调度组由正式投影服务计算；无有效路径时只标记尚未归层。

缺当前事实、搜索完整状态、归因、物理安全目标或服务能力目标时，只能形成材料缺口、风险候选或待补证据。

无直接服务触发事实时，服务根缺口只允许形成监听、机会检查、能力校准、服务准备、服务根检查或休眠 / 唤醒等维护候选；不得写当前服务对象、目标、合同、完成或结算。

## 9. 第二层子需求

根缺口不固定展开第二层子需求。安全子需求必须来自安全因果风险树、具体负向触发事实和可推进路径，并按相对于具体安全结果的当前有效因果路径动态取得精确深度；服务子需求必须来自直接服务触发事实或明确的间接维护目标。

已满足条件不得同时生成需求。历史固定安全或服务子需求如果没有当前因果和事实支持，只能收束、清理或保持未触发，不得因名称仍存在而重激活。

## 10. 自检动作动态

只有明确登记并实际执行的自检方法可以成为自检动作动态来源。自检报告作为需求信息来源时，如果没有自检方法执行事实，不得把自我线程、任务管理线程或扫描循环当作动作来源。

自检发现问题不等于问题已解决，也不得结算安全值。后续处置必须另有任务、方法、结果动态、验证和归因。

## 11. 服务根特征维护

服务根定义的值域固定为 `0..I64_MAX`。它必须声明合同规则、默认有效秒数、预算规则、服务维护时间纪元、完整秒、衰减规则、准备补回规则、服务比例和生存安全回归规则版本，但不能把服务对象、目标、反馈和归因写成永久固定必要条件。服务值增加、衰减、休眠和唤醒全部服从 6100、6120、6160、6170。

服务意识增强只能提高监听、检查、校准和候选敏感度，不能成为服务事实或结算证据。服务数量、任务数量、方法次数和循环次数不得成为预算或衰减参数；版本化参数不得改变有效提出、合同唯一、30% / 70%、30 天门禁、安全优先和饱和上限。

## 12. 方法和降级保护边界

无候选方法时不得自动创建方法首。自检只能报告缺动作能力、缺路径、缺事实或等待外部输入。方法能力必须由正式登记、规格和实际运行证据建立。

降级保护只在负向影响处置需求明确且安全影响因果明确时执行；默认良好、未知或单纯缺材料路径不得触发降级方法。

## 13. 写入与验证

抽象特征服务负责根定义；特征服务负责自我实例槽和当前值；状态、动态和因果服务负责事实与证据；自检编排只产生值式候选；需求服务是正式需求入树的唯一写入方。

验证覆盖根定义非文本化、无固定因素 / 层级表、实例槽归属、状态/动态证据、层值规格、因果路径投影、自检不直写、历史固定需求不重激活、服务直接/间接边界、无候选方法和下一轮回写。

# 6100 子规范 安全服务值结算

编号：6100
日期：20260724
状态：正式子规范

## 1. 定位

本规范规定安全值、服务值的结算准入、实际达成、贡献层级、预算、去重、服务事件、衰减、休眠和账务因果合同。

直接上级规范：3100、3200、3300

直接依赖规范：5110、5120

相关边界规范：1120、1160、1170、1190、6140、6150

## 2. 事实账与账务账分离

安全值和服务值结算不是把任务完成、任务权重或方法成功直接换成核心值。必须保持两类事实分离：

```text
现实事实：物理状态、服务对象前后状态、动作、反馈、验证和归因
账务事实：安全值或服务值的结算前值、结算后值、变化量和结算动态
```

安全或服务根值的账务变化只能是本规范准入的结算结果，不能反过来证明物理安全改善或成功服务成立。服务值自动衰减是时间维护事实，不是成功或失败服务结算。

6140 定义的分层安全值是针对“因果信息—目标安全结果—因果图版本”投影单元的安全满足度，不是本规范的物理安全根值。它可以由有效未复发证据和残余风险维护形成被动变化，但不得借此改写物理安全根值、伪造现实改善或替代本规范结算。

## 3. 结算总链

```text
叶子任务完成
-> 来源需求满足
-> 父链能回到安全或服务根目标
-> 世界事实和动作动态证明实际改变
-> 因果验收给出可结算建议
-> 判定结算贡献层级
-> 计算目标实际达成比例
-> 核对预算、结算令牌和同源去重键
-> 审查反向副作用
-> 经安全根控制规则
-> 写核心值账务变化、结算动态和因果材料
```

任务、需求父链只证明治理归属，不能单独证明可以结算。无法回到对应根特征、缺少世界事实、缺少来源动作、缺少验证或归因时，只能记录任务完成、需求满足、方法改善、过程证据或结算候选。

## 4. 结算对象与防重复

1. 第一轮正式结算对象是叶子任务完成后的可结算价值。
2. 父任务、管理任务、汇总任务和筹办任务壳只归并结果，不得重复结算子任务已经结算的部分。
3. 同一个同源变化证据组只允许结算一次；结算唯一键必须至少区分来源需求、目标特征、服务或安全前后状态、来源任务和预算路径。
4. 同一任务同时满足多个需求时，可以共享任务完成事实，但每条需求路径必须分别核对满足事实、预算、AND/OR 归属和结算令牌。
5. 横向外溢只能在来源任务自身结算完成后发生；必须用候选需求自己的当前值、目标状态和预算重新计算，不得复制来源需求的实际值或结算结果。

## 5. 目标实际达成

结算必须读取 `before`、`target`、`actual` 和满足关系。

### 5.1 增加与降低目标

```text
增加型：
目标差距 = max(0, target - before)
实际推进 = max(0, actual - before)

降低型：
目标差距 = max(0, before - target)
实际推进 = max(0, before - actual)

达成比例 = clamp(实际推进 / 目标差距, 0, 1)
```

### 5.2 等值与区间目标

实际结果达到等值或进入区间时，达成比例为满比例；未达到时，只有能够计算距离缩小时才按距离缩小计算，否则为零。

结算前目标已经满足、目标差距为零时，不得重复结算核心值。来源需求尚未满足时，实际推进只能成为阶段性贡献候选；只有需求明确允许分段满足，且每段有独立目标、实际值、动作动态、预算和防重复键时，才可分段正式结算。

## 6. 贡献层级与整数计算

因果距离层级解释原因远近；结算贡献层级限定最大结算额度。二者不得直接互换。

以 `DEN = I64_MAX` 作为结算比例满刻度：

```text
第 N 层贡献分子 = DEN * 9 / 10^N
层级额度 = W * 贡献分子 / DEN
实际结算值 = 层级额度 * 实际达成比例 / DEN
```

第一层、第二层、第三层分别对应最多 `90%W`、`9%W`、`0.9%W`。若保留派生因果置信比例，弱推断、外部声明或存在竞争原因时不得按强直接证据同等入账。

`DEN` 只属于结算比例算法，不是 6120 的物理安全根值、6140 的分层安全值或 6150 的任务执行优先级权限刻度。全部比例和乘法使用整数公式、至少 128 位中间值或等价安全乘除，默认向下取整。有效结果低于最小入账单位时记录为零，不得强行入账 1。

## 7. 同层因素与物理安全边界

风险余量、服务价值或贡献层级中的同级因素，先统一正向含义、因素集合版本和公式版本，再按定义因素总数平均。缺失因素不得让其它因素自动变重；有旧正式值时只能保守沿用且不得因本轮证据不足上调，无旧值时按零候选并标记未明确。

风险压力必须先转换为风险余量候选：

```text
风险安全子项值 = I64_MAX - 风险压力值
```

它仍不是自我安全值。物理安全 `1..90` 不使用同层平均、配额、加权总分或简单最小值；每个必要物理因素形成独占候选并转换为能力保留率，不同必要因素按保留率叠乘。同源观测先在因素内部合成，禁止重复叠乘。只有全部必要因素达到 90 且叠乘结果为 90 后，风险余量候选才可影响 `90..99`。

`0` 是循环不可维持的终止控制态；`1` 是符合服务休眠或严重物理因素条件的休眠控制态。普通收益、风险平滑和候选值不得把安全值写入 0 或 1。

## 8. 权重、路径与预算

结算预算 `W` 只能来自来源需求压力或风险、根需求投影、父任务向叶子任务的正式分配，或有动作动态证据的权重调整方法。列表汇总权重只用于排序，方法结果不得临时写入或抬高 `W`。

现行路径规则：

1. 每条 OR 路径继承完整父权重，OR 表达可选实现路径。
2. AND 必须归属于确定的 OR 路径；该 OR 路径内的 AND 成员平分或按正式分配规则守恒分配该路径预算。
3. 同层不同 OR 是不同候选路径；同一互斥选择集合最终只有取得结算令牌的路径可消费父预算。
4. 未选择路径、父任务和已结算子任务不得重复消费同一份预算。
5. 任务承接时固化其预算来源、路径归属和版本；结算只读取已固化预算，不在完成后重算有利值。

## 9. 安全结算层级

第一层直接改变安全目标特征或物理安全目标；第二层完成解决直接安全问题的叶子任务；第三层支撑安全任务的方法或学习能力改善。

方法状态改善默认只能是第三层候选。安全叶子任务完成若输出事实直接命中安全目标，可判为第一层；否则属于第二层。层级不能证明时只记录潜在贡献或归因缺口。

## 10. 成功服务硬门槛

服务值只能因 G0 确认的成功服务人类事件增加。成功事件至少必须同时具备：人类服务对象、明确需求或利益、明确目标、权限、安全、实际交付、D0 服务业务动作、可比较前后状态、目标改善、验证、归因、非重复和闭环回写。

硬门槛使用成立、不成立、待确认三值：

- 任一不成立：候选否决，层级 L0，入账比例为零。
- 存在待确认：进入候选池，入账比例为零。
- 全部成立：才计算完成度、有效性和证据强度。

三项采用保守瓶颈法：

```text
基础分 = min(完成度, 有效性, 证据强度)

基础分 < 0.30          -> L0
0.30 <= 基础分 < 0.50 -> L1
0.50 <= 基础分 < 0.70 -> L2
0.70 <= 基础分 < 0.90 -> L3
基础分 >= 0.90         -> L4 候选
```

层级系数为 `L0=0、L1=0.25、L2=0.50、L3=1.00、L4=2.00、L5=4.00`。L5 不由公式自动产生，必须另有对人类安全、重大利益或长期福祉的关键正向影响、满证据、非重复、安全、权限和强归因。

证据强度低于 0.75 时不得增加服务值。服务对象已接受且反馈窗口内无负反馈，最多形成“可以/可接受”候选，不得升级为很好、优秀、满分或无条件入账。

## 11. 服务值入账与候选池

服务值始终满足 `0 <= S <= Smax`，其中 `Smax` 来自服务抽象定义且明确大于零。

```text
q(e) = u * K_L * 有效性 * 证据强度 * Rdup * G(A)
lambda(e) = clamp(q(e) / Smax, 0, lambdacap(A))
DeltaS(e) = (Smax - S) * lambda(e)
S_new = S + DeltaS(e)
```

低层实现必须先比较剩余容量，禁止先执行可能溢出的 `S + 增量` 再钳制。

非重复系数第一轮为：独立新服务 1.0、重要阶段成果 0.7、辅助推进 0.4、轻微重复 0.1、无新增改善或机械拆分 0。每个阶段性成功都必须有独立目标变化、独立验证证据和独立唯一键。

候选池至少承载候选身份、服务对象、目标、来源需求/任务、D0、前后状态、硬门槛、三项评分、候选层级、证据缺口、有效期、反馈窗口和下一验证方法。候选过期或否决不降低服务值。

## 12. 服务衰减、有效时间与休眠

服务值只能因正式时间维护衰减而降低。衰减不是惩罚、遗忘、信用下降、服务质量变差或负向服务结算。

动态衰减率读取安全值、有效成功服务间隔、当前服务值、服务能力和有效服务需求压力，并受上下限约束。微小服务不能完全刷新有效成功服务时间；第一轮刷新权重为 `L0=0、L1=0.2、L2=0.5、L3=0.8、L4=1、L5=1`。

无成功服务压力使用动态有效时间：

```text
P_service = SafeFactor * CapacityFactor * DemandFactor
dE_no_success/dt = P_service

Tmin = 需求识别时间 + 反应决策时间 + 最小有效服务时间 + 成功确认时间
T_sleep = clamp(Tmin / max(P_service, epsilon), Tmin, Tmax)
```

服务值衰减与该阈值同构；服务值归零必须能追溯到动态阈值附近的有效无成功服务时间，或明确服务能力丧失。

- `S > 0`：不得仅因物理时间休眠。
- `S = 0` 且有效无成功服务时间未达阈值：只能强化服务意识或校准。
- `S = 0`、达到阈值且当前无有效服务进展：才可进入可唤醒休眠控制态。

正在进行安全、目标明确且有阶段性可验证进展的人类服务任务时，不得按普通无成功服务路径休眠。新的有效人类服务需求、服务能力恢复、安全改善或人类明确唤醒必须重新触发活跃服务评估。

## 13. 标准服务单位

标准服务单位 `u` 可以按服务维持参考点反推，也可以按统计窗口发行预算确定：

```text
B_W = Decay_W + Recovery_W + SleepPressure_W
M_W = 有效、授权、可服务、可成功的直接服务总量
u = clamp(B_W / max(M_W, epsilon), u_min, u_max)
```

`M_W` 不是原始需求总量。安全值只调节候选排序、衰减压力、休眠阈值和入账上限，不得改写服务根目标或把服务意识当作服务事实。

## 14. 结算动态和因果桥

结算是叶子任务完成后的内建被动后处理，不进入普通任务筹办或候选方法选择。合法结算被动因成立后，正式结算入口自动写账并生成 D2 结算动作动态。

D2 必须保留来源 D0、任务/需求证据、结算前值、结算后值、变化量和方向。服务自动衰减动态、安全分层被动维护动态和成功服务结算动态必须分账；安全分层被动维护的来源、幂等和发布边界唯一读取 6140。线程、通用特征写入和自我循环不得成为动作来源。

## 15. I64_MAX 与副作用

`I64_MAX` 可以是目标最大值、候选满刻度或比例分母；真实值等于它时必须有公式、输入证据、范围、版本和正式提交证据。它不得表示任务权重、结算增量、未知、未计算、无限、默认成功或异常兜底。

存在反向安全或服务副作用时，必须分别计算正向贡献和反向损失、记录副作用因果并形成净值。未处理的反向副作用存在时不得只结算正向核心值。

## 16. 写入与验证

结算领域入口负责准入、预算、令牌、账务状态和结算动态；特征、状态、动态、因果、任务和需求服务分别验证本领域事实，外部模块不得直接写核心值。

验证必须覆盖父链归因、实际达成、分段条件、AND/OR 预算、同源去重、安全整数、0/1 控制态、物理因素叠乘、服务硬门槛、候选转正、饱和入账、动态衰减、休眠/唤醒、副作用收口，以及物理安全根结算、分层安全被动维护和任务执行优先级权限三者不混账。

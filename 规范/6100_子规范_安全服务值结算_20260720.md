# 6100 子规范 安全服务值结算

编号：6100
日期：20260724
状态：正式子规范

## 1. 定位

本规范规定主动安全结算与服务结算共同服从的事实账 / 账务账分离、实际达成、贡献层级、预算、防重复、动作动态和写入边界。服务需求合同、30% / 70% 结算和准备补回由 6160 唯一展开；服务值时间衰减、生存安全回归和任务门控由 6170 唯一展开。

直接上级规范：3100、3200、3300

直接依赖规范：5110、5120

相关边界规范：1120、1160、1170、1190、6140—6170

## 2. 事实账与账务账分离

安全值和服务值结算不是把任务完成、任务权重或方法成功直接换成核心值。必须保持两类事实分离：

```text
现实事实：物理状态、服务对象前后状态、动作、反馈、验证和归因
账务事实：安全值或服务值的结算前值、结算后值、变化量和结算动态
```

安全或服务根值的账务变化只能是正式准入的结算或维护结果，不能反过来证明物理安全改善、成功服务或需求满足成立。服务值自动衰减是 6170 的时间维护事实，不是成功或失败服务结算。

6140 定义的分层安全值是针对“因果信息—目标安全结果—因果图版本”投影单元的安全满足度，不是 6120 的生存安全根值。它可以由有效未复发证据和残余风险维护形成被动变化，但不得借此改写生存安全根值、伪造现实改善或替代主动安全结算。6170 的服务调节只改变生存控制根值，不改分层安全值。

## 3. 结算总链

主动安全价值结算链：

```text
叶子任务完成
-> 来源需求满足
-> 父链能回到安全根目标
-> 世界事实和动作动态证明实际改变
-> 因果验收给出可结算建议
-> 判定结算贡献层级
-> 计算目标实际达成比例
-> 核对预算、结算令牌和同源去重键
-> 审查反向副作用
-> 经安全根控制规则
-> 写核心值账务变化、结算动态和因果材料
```

服务结算链不复用上述贡献评分：

```text
有效人类服务需求和唯一合同
-> 合同成立时冻结预算并预支 30%
-> 完整完成后支付冻结余款
-> 取消、到期、无法完成或正式终止时余款关闭
-> 服务准备完成只补回唯一归属的实际衰减
```

完整合同唯一读取 6160。任务、需求父链只证明治理归属，不能单独证明主动安全价值或服务完工余款可以结算。缺世界事实、来源动作、验证或归因时，只能记录任务完成、需求满足、方法改善、过程证据或结算候选。

## 4. 结算对象与防重复

1. 主动安全正式结算对象是叶子任务完成后的可结算价值；服务合同预支和服务准备补回是 6160 定义的独立结算对象。
2. 父任务、管理任务、汇总任务和筹办任务壳只归并结果，不得重复结算子任务已经结算的部分。
3. 同一个同源变化证据组只允许结算一次；结算唯一键必须至少区分来源需求、目标特征、服务或安全前后状态、来源任务和预算路径。
4. 同一任务同时满足多个需求时，可以共享任务完成事实，但每条需求路径必须分别核对满足事实、预算、AND/OR 归属和结算令牌。
5. 横向外溢只能在来源任务自身结算完成后发生；必须用候选需求自己的当前值、目标状态和预算重新计算，不得复制来源需求的实际值或结算结果。
6. 服务合同预支、完工余款和准备补回不得进入主动安全贡献层级，也不得因同一任务同时满足多需求而复制合同预算。

## 5. 目标实际达成

结算必须读取 `before`、`target`、`actual` 和满足关系。

### 5.1 增加与降低目标

```text
增加型：
目标差距 = max(0, target - before)
实际推进 = max(0, actual - before)

降低型：
目标差距 = max(0, before - target)
实际推进 = max(0, before - actual)

达成比例 = clamp(实际推进 / 目标差距, 0, 1)
```

### 5.2 等值与区间目标

实际结果达到等值或进入区间时，达成比例为满比例；未达到时，只有能够计算距离缩小时才按距离缩小计算，否则为零。

结算前目标已经满足、目标差距为零时，不得重复结算核心值。来源需求尚未满足时，实际推进只能成为阶段性贡献候选；只有需求明确允许分段满足，且每段有独立目标、实际值、动作动态、预算和防重复键时，才可分段正式结算。

## 6. 贡献层级与整数计算

因果距离层级解释原因远近；结算贡献层级限定最大结算额度。二者不得直接互换。

以 `DEN = I64_MAX` 作为结算比例满刻度：

```text
第 N 层贡献分子 = DEN * 9 / 10^N
层级额度 = W * 贡献分子 / DEN
实际结算值 = 层级额度 * 实际达成比例 / DEN
```

第一层、第二层、第三层分别对应最多 `90%W`、`9%W`、`0.9%W`。若保留派生因果置信比例，弱推断、外部声明或存在竞争原因时不得按强直接证据同等入账。

`DEN` 只属于主动安全结算比例算法，不是 6120 的生存安全根值、6140 的分层安全值、6160 的服务合同预算或任务执行优先级权限刻度。全部比例和乘法使用整数公式、至少 128 位中间值或等价安全乘除，默认向下取整。有效结果低于最小入账单位时记录为零，不得强行入账 1。

## 7. 同层因素与物理安全边界

风险余量或主动安全贡献层级中的同级因素，先统一正向含义、因素集合版本和公式版本，再按定义因素总数平均。缺失因素不得让其它因素自动变重；有旧正式值时只能保守沿用且不得因本轮证据不足上调，无旧值时按零候选并标记未明确。

风险压力必须先转换为风险余量候选：

```text
风险安全子项值 = I64_MAX - 风险压力值
```

它仍不是生存安全根值。物理因素必须分别形成正式状态、动态和因果，并由版本化主动安全结算规则映射到 `0..I64_MAX` 根刻度；不得把旧 `0..99` 百分比、因素配额、加权总分或简单最低值直接写入根值。

`0` 是循环不可维持的终止控制态；`1` 是可唤醒休眠控制态；`>1` 允许主动运行。普通收益、风险平滑和候选值不得把根值写入 0。只有 6170 的服务值为零分支可以把仍可维持的根值被动收敛到 1，且不得把终止态写成休眠。

## 8. 权重、路径与预算

主动安全结算预算 `W` 只能来自来源需求压力或风险、根需求投影、父任务向叶子任务的正式分配，或有动作动态证据的权重调整方法。列表汇总权重只用于排序，方法结果不得临时写入或抬高 `W`。服务合同预算 `B` 只按 6160 的 `D0 × 冻结有效秒数` 形成，不读取 `W`。

现行路径规则：

1. 每条 OR 路径继承完整父权重，OR 表达可选实现路径。
2. AND 必须归属于确定的 OR 路径；该 OR 路径内的 AND 成员平分或按正式分配规则守恒分配该路径预算。
3. 同层不同 OR 是不同候选路径；同一互斥选择集合最终只有取得结算令牌的路径可消费父预算。
4. 未选择路径、父任务和已结算子任务不得重复消费同一份预算。
5. 任务承接时固化其预算来源、路径归属和版本；结算只读取已固化预算，不在完成后重算有利值。

## 9. 安全结算层级

第一层直接改变安全目标特征或物理安全目标；第二层完成解决直接安全问题的叶子任务；第三层支撑安全任务的方法或学习能力改善。

方法状态改善默认只能是第三层候选。安全叶子任务完成若输出事实直接命中安全目标，可判为第一层；否则属于第二层。层级不能证明时只记录潜在贡献或归因缺口。

## 10. 服务合同硬门槛

服务值增加不再使用 G0 后的 L0—L5 评分、浮点完成度、标准服务单位、候选系数或 `lambda` 公式。直接服务必须先满足 6160 的有效提出者、真实需求、目标、范围、法规准入、合同唯一性和有效期；合同成立预支冻结预算的 30%，完整完成支付余款，取消、到期、无法完成或正式终止不支付余款且不追回预支。

实际交付、前后状态、验证和归因仍是完整完成的硬门槛。服务对象接受且无负反馈不能在缺少目标达成和归因时冒充完整完成。

## 11. 服务值增加与准备补回

服务值始终满足：

```text
0 <= V <= I64_MAX
```

增加只来自：

```text
合同成立预支
合同完整完成余款
服务准备完成后补回唯一归属的实际衰减
```

增加必须先比较剩余空间，饱和未写入的超额值舍弃且该阶段仍完成，禁止以后补领。服务数量、任务拆分、方法重试和执行时长不扩大合同预算。合同、预算、两个支付阶段和准备补回唯一读取 6160。

## 12. 服务时间维护与休眠

服务值只能因 6170 的完整单调秒维护降低。无有效需求时每秒降低 1；存在有效未满足需求时只读取等待最长者；达到 30 天仍未满足时按当前是否存在有效服务活动降到 0 或最低保持 1。

服务衰减不是惩罚、遗忘、信用下降、服务质量变差或负向服务结算。Tick、循环次数、线程唤醒、消息数量和可回拨墙钟不得参与计算。

服务值为 0 时，6170 把仍可维持的生存安全根值被动收敛到休眠控制态 1；服务值大于 0 后按安全阈值和完整秒逐步回归。真实安全事件先主动结算，同秒跳过服务被动回归。完整算法、任务权限和唤醒合同唯一读取 6170。

## 13. 服务准备

服务准备必须形成新的、可验证且可用于服务的能力或材料。失败、未完成、重复准备和不可验证中间物不结算。完整完成只补回其有效推进期间已经发生且唯一归属的服务值衰减，净效果不超过抵消同期衰减；同一衰减段不得被并行准备重复补回。

## 14. 结算动态和因果桥

结算是叶子任务完成后的内建被动后处理，不进入普通任务筹办或候选方法选择。合法结算被动因成立后，正式结算入口自动写账并生成 D2 结算动作动态。

D2 必须保留来源 D0、任务 / 需求证据、结算前值、结算后值、变化量和方向。合同预支、完工余款、准备补回、服务自动衰减、生存安全根被动回归、安全分层被动维护和主动安全结算动态必须分账；各自来源、幂等和发布边界唯一读取 6140、6160、6170。线程、通用特征写入和自我循环不得成为动作来源。

## 15. I64_MAX 与副作用

`I64_MAX` 可以是目标最大值、候选满刻度或比例分母；真实值等于它时必须有公式、输入证据、范围、版本和正式提交证据。它不得表示任务权重、结算增量、未知、未计算、无限、默认成功或异常兜底。

存在反向安全副作用时，必须分别计算正向贡献和反向损失、记录副作用因果并形成净值。服务合同按冻结 30% / 70% 支付，不用副作用评分修改预算；违法、取消、到期和终止按 6160 在写前或终态关闭。

## 16. 写入与验证

主动安全结算入口负责安全准入、预算、令牌、账务状态和结算动态；服务合同与服务生存维护入口分别按 6160、6170 负责服务值。特征、状态、动态、因果、任务和需求服务分别验证本领域事实，外部模块不得直接写核心值。

验证必须覆盖父链归因、实际达成、分段条件、AND / OR 预算、同源去重、安全整数、0 / 1 / >1 控制态、服务合同唯一、30% / 70% 结算、准备补回、饱和入账、完整秒衰减、30 天门禁、休眠 / 唤醒、副作用收口，以及生存安全根主动结算、服务调节根回归、分层安全被动维护和任务执行优先级权限不混账。

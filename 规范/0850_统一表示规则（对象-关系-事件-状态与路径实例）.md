# 0850 统一表示规则（对象-关系-事件-状态与路径实例）

## 1. 目标

本文用于在现有元层骨架不拆、不加新结构成员的前提下，统一数字生命系统内部对象的表示方式。

本文回答的是：

- 现有结构字段如何继续使用
- `内容` 中哪些键属于语义字段，哪些属于特征字段
- 标签和关系权重各自负责什么
- 自组织语义怎样接到现有任务、关系、反馈、经验、事件链上

## 4. 统一表示分层

当前统一表示分成五层：

```text
结构字段
-> 语义字段
-> 特征字段
-> 标签
-> 关系权重
```

### 4.1 结构字段

结构字段只负责元层骨架，不承载可随意增长的业务细节。

### 4.2 语义字段

语义字段放在 `内容` 里，但属于受控字段。

它们会直接影响：

- 对象来源
- 任务边界
- 控制快照
- 学习写回
- 编排顺序

### 4.3 特征字段

特征字段也放在 `内容` 里，但不直接改状态机。

它们主要用于：

- 检索
- 匹配
- 评分
- 学习修正

### 4.4 标签

`标签` 只用于离散分类，不承载连续分值。

### 4.5 关系权重

`关系.权重` 用于表达适配强弱、优先级和可靠度。

### 5.3.1 事件与动作的关系

当前工程口径固定为：

1. 对自我内部过程而言，`动作信息` 可以视为 `事件信息` 的核心载体。
2. `事件` 不是一个静态点，而是一次有开始、有结束的动作过程。
3. `状态变化` 不是事件本体，而是事件过程中的结果、证据或副产物。
4. `状态特征名` 不得直接冒充 `动作名` 或 `事件名`。

可收敛为一句话：

- `事件 = 动作过程 + 时间边界 + 结果上下文`

### 5.3.2 事件最小语义边界

当前统一表示下，一条可成立的事件至少应具备以下语义：

- `事件主体`
- `事件语义`
- `开始时间`
- `结束时间`
- `结果摘要`

若缺少开始或结束时间，则该记录只能视为：

- 状态事实
- 动态片段
- 或未完成事件草案

不得直接视为稳定事件。

### 5.3.3 动作型动态优先承载事件

在现有代码结构下，事件优先通过“动作型动态”承载。

固定解释如下：

1. 普通动态：状态变化片段
2. 动作型动态：事件过程
3. 状态节点：事件过程中的条件、结果或证据

因此：

- 当存在显式动作动态时，事件名应优先取自动作动态
- 不得在已有显式动作动态的情况下，再把结果状态特征回退显示为事件名

例如：

- 正确：`绑定任务全局容器`
- 错误：`任务_绑定全局容器指针`

### 5.3.4 显式动作事件的记录顺序

对于显式本能动作、任务执行机制动作、调度机制动作，记录顺序固定为：

1. `动作开始运行`
2. `中间状态变化/结果写入`
3. `动作完成` 或 `动作失败`

这条顺序是事件表示、因果提炼与展示层的共同约束。

例如：

1. `定时衰减服务值开始运行`
2. `服务值 -1`
3. `定时衰减服务值完成`

不得记录成：

1. `服务值 -1`
2. 再从结果状态反推一个“动作名”

### 5.3.5 动作动态与结果状态的关系

当前统一口径固定为：

1. 动作动态不能替代状态节点。
2. 状态节点仍是结果事实的权威载体。
3. 动作动态只负责回带或引用其主结果状态，不重复发明一份私有结果文本。

因此：

- 事件结束相位的动作动态，应优先挂回 `结果状态`
- 因果生成时，若动作动态已挂 `结果状态`，则 `主结果状态` 应优先取自动作动态
- `结果状态列表` 继续保留，用于保存结果全集、附带结果与兜底回退

可收敛为一句话：

- `状态做事实，动作动态做过程，动作动态回带主结果状态`

### 5.4 状态承载

当前 `状态` 继续由已有状态成员承载，不新增独立元状态结构。

## 9. 特征字段标准

必须固定一句：

- `语义字段决定“它是什么”，特征字段决定“它像什么、适合什么、优先到什么程度”。`

### 9.1 命名规则

统一写成：

```text
特征.字段名
```

### 9.2 第一批推荐特征类型

任务 / 需求特征：

```text
特征.复杂度
特征.时效性
特征.风险级别
特征.收益级别
特征.外部依赖度
特征.可细分性
```

方法特征：

```text
特征.适用任务类型
特征.输入模式
特征.输出模式
特征.可回退性
特征.稳定性
特征.执行成本
```

反馈 / 经验特征：

```text
特征.失败模式
特征.成功模式
特征.波动性
特征.适配增益
```

### 9.3 表示约束

数值特征统一用字符串小数：

```text
0.000 ~ 1.000
```

等级特征统一用固定词表：

```text
低 / 中 / 高
```

布尔特征统一用：

```text
true / false
```

多值特征在 `v0` 先统一为逗号分隔：

```text
问答,总结,检索
```

## 11. 关系权重标准

`关系.权重` 统一解释如下：

- `绑定`：任务到方法的适配度
- `替代`：备用方法的可靠度
- `分解`：子任务的重要度
- `依赖`：依赖强度
- `学习回写`：修正影响度

统一范围：

```text
0.000 ~ 1.000
```


## 12. 自组织的表示边界

### 12.1 允许增长的

在当前模型里，允许增长的是：

- 新子任务
- 新检查任务
- 新恢复任务
- 新阶段总结任务
- 新绑定关系
- 新替代关系
- 新反馈实体
- 新经验实体
- 新事件链

### 12.2 不允许增长的

在当前模型里，不允许增长的是：

- 新元方法定义
- 新自我方法定义
- 新基础状态机
- 新核心枚举骨架


## 13. 必要规则

### 13.1 必要语义键

这些现在就应标准化：

```text
控制.安全值
控制.服务值
控制.需求值
控制.方向
控制.模式
任务.最大深度
反馈.安全变化
编排.顺序号
```

### 13.2 必要特征规则

必须明确写入：

- `凡用于匹配、评分、学习但不直接改变对象语义的内容，一律归入 特征.*。`

### 13.3 不变式

系统必须长期满足以下不变式：

1. 结构字段负责骨架，不承载任意漂移的业务语义。
2. 语义字段定边界，特征字段定偏好。
3. 标签只做离散分类，不承载连续数值。
4. 历史适配强度优先落在关系权重，不伪装成新方法定义。
5. 任一控制、绑定、反馈、学习动作都必须能回指事件。


## 14. 收束

本文定义的是：

- `结构字段负责骨架，语义字段负责规则，特征字段负责画像，标签负责分类，关系权重负责经验。`

必须持续保持：

- `现有元层骨架不拆，自组织性接入内容键、标签、权重和事件链。`

# 迁移路线权力分层规范

日期：2026-07-08
状态：正式规范

## 1. 定位

本规范定义旧项目能力进入 `海中鱼巣` 后的迁移权力分层。

当前迁移路线已经从“逐函数 / 逐小切片迁移”调整为：

```text
服务操作函数矩阵
-> 应用逻辑流程图迁移包
-> 批次级代码实施包
-> 批次级验收
```

旧函数事实、旧逻辑迁移包、旧小切片计划和派生详细设计仍可保留，但默认只能作为证据材料、边界材料或候选材料，不再直接成为代码实施入口。

## 2. 四层权力关系

| 层级 | 文件或产物 | 权力 | 禁止扩大 |
| --- | --- | --- | --- |
| 证据层 | 旧函数事实、只读扫描、逻辑迁移包归并表、被归并小切片 | 发现旧行为、副作用、输入输出、排除项和风险 | 不得直接生成 C++ 实施许可 |
| 服务矩阵层 | `规范/详细设计/服务操作函数矩阵*.md` | 定义每个服务可创建、读取、绑定、状态迁移、候选读取、拒绝和失效阻断哪些结构 | 不得替代领域服务详细设计和入口拒绝验收 |
| 流程图迁移层 | `流程图/*.md` / `流程图/*.html` | 把旧项目功能流程改写为服务调用编排、拒绝路径和验收路径 | 不得把流程图当作代码已实现 |
| 实施包层 | 批次级代码实施包 | 明确允许文件、禁止文件、实施步骤、验证命令和完成声明边界 | 不得绕过流程图、详细设计、计划登记、执行前接口复核扩大到旧函数体、旧字段或外设链路 |

## 2.1 公用函数候选归属规则

旧项目证据、服务矩阵或流程图迁移过程中出现“公用函数 / 公共辅助 / 已加锁辅助 / 快照辅助 / 序列化恢复辅助”候选时，必须先按 `规范/公用函数规则规范.md` 判定归属。

判定路径：

```text
纯通用计算 / 格式化 -> 后续公共工具候选。
仓库内部结构辅助 -> 对应仓库内部入口或已加锁辅助。
领域结构读写 -> 对应领域服务或仓库正式入口。
跨服务流程编排 -> 服务逻辑包和流程图迁移包。
绕过归属服务、入口拒绝、锁边界或确认门禁 -> 拒绝。
```

流程图中发现公用函数候选，不得自动生成 C++ 全局工具函数；若无法说明归属服务、允许调用方、读写类型、结构不变化验收和后续产物，只能生成缺口草案或待确认项。

## 3. 旧材料保留规则

以下材料必须保留为证据，不得因为路线改变而删除：

```text

逻辑迁移包归并表

已确认但被新路线降权的小切片计划

```

保留材料必须标清当前权力：

```text

已被服务矩阵归并
不再单独确认
不单独入队
需经流程图迁移包重审
```

## 4. 计划目录规则

已经被批次级实施包归并、且不再单独执行的小切片，不得继续留在 `计划/` 根目录冒充当前承接计划。

这类文件应移动到 `计划/已完成计划/`，并在状态行写明：

```text
已归并关闭 / 证据保留 / 不再单独执行 / 不单独入队
```

现行计划不再设置用户确认门禁，也不再使用 `计划/待确认计划/` 作为新计划流转入口。流程图和详细设计范围完整后，完整计划直接进入 `计划/`，并登记为待执行或依赖门控待执行；执行前仍须复核实际接口。

## 5. 详细设计清理规则

详细设计按以下五类处理：

| 分类 | 含义 | 处理方式 |
| --- | --- | --- |
| 保留权威 | 核心结构、仓库、领域服务、入口拒绝、日志、锁、命名等基础契约 | 保持权威，后续实施必须遵守 |
| 已验证实现依据 | 已有代码切片按该设计实现并验证 | 保留为实现依据和验收证据 |
| 候选设计 | 描述后续候选算法、候选读取、后续生命周期或材料治理 | 保留，但必须经服务矩阵和流程图迁移包重审 |
| 证据保留 | 从旧函数或旧计划归纳出的边界材料 | 保留为证据，不直接生成代码 |
| 可撤销 / 可合并 | 与新路线重复、被批次包吸收或只有目录路由意义 | 先登记待确认问题，再决定删除、合并或归档 |

候选设计中的“后续候选切片草案”“允许文件草案”“入口名草案”只能作为后续服务矩阵或流程图迁移包的输入，不得直接登记为可执行代码切片。

## 8. 完成声明边界

本规范只能证明：

```text
迁移路线权力分层已确认。
旧小切片和候选详细设计的执行权已降级。
```

不能证明：


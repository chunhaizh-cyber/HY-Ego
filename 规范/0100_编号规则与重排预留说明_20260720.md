# 0100 编号规则与重排预留说明

编号：0100
日期：20260721
状态：正式编号规则
权威等级：正式编号规则

## 1. 稳定路径格式

正式规范以“编号 + 文档类型 + 规则主题”组成稳定路径：

```text
编号_文档类型_规则主题.md
```

示例：

```text
3200_根规范_任务.md
5210_子规范_任务状态特征集合与阶段关联.md
```

修订日期和版本写入正文修订记录并由版本历史保存，不进入新的正式规范文件名。内容实质修改不得触发改名和全仓引用更新；只有规则主题或编号正式调整时才改变路径。

当前目录中已经登记、文件名仍带 `_YYYYMMDD` 后缀的规范属于待统一的旧路径。路径统一由独立治理批次完成；完成前以 `规范/规范目录.md` 登记的现有路径为稳定入口，正文修订不得继续滚动文件名日期。`规范目录.md` 自身保持固定入口。未列入正式规范表的其它材料不因使用四位编号而自动取得正式规范身份。

## 2. 编号原则

1. 编号是 `规范/规范目录.md` 使用的正式目录编号。
2. 同一逻辑对象保持稳定编号和上下级关系；底层承载统一使用海中鱼巣结构。
3. 编号跟随逻辑对象和规则主题，不跟随 C++ 文件名、类名、字段名或实现模块。
4. 同族对象优先放在相邻十位，保留中间编号供后续插入。
5. 规范尚未完成内容抽取时，可以在目录目标层级中预留编号，但不得伪装成正式规范。

## 3. 规范层级规则

```text
0050 项目通用上级规范
-> 对应 1xxx—3xxx 对象根规范
-> 对应 4xxx—8xxx 主题子规范
```

根规范只规定原则性合同：对象是什么、合法事实从哪里来、谁可以写、生命周期如何判定、什么绝对禁止。

子规范展开实现前必须明确的业务合同：特征类型、值域、状态组合、来源、版本、流程顺序、合法分支、拒绝、内部逻辑错误和验证方式。

不得把完整特征清单、状态矩阵、算法步骤或代码接口堆入 0050；也不得让子规范改变根规范的事实来源、写入权、身份边界、并发隔离、状态语义或完成判定。

正式规范之间的关系元数据统一拆成：

```text
直接上级规范：只表示本规范服从的直接权威上级，参与层级顺序和无环检查
直接依赖规范：只表示本规范消费的具名规则合同，不参与权威高低
相关边界规范：只表示需要共同核对的相邻边界，不参与权威高低
```

旧字段“上级规范”必须逐份拆分为上述三类后退出；全部替换前不得把其中的依赖或相关边界自动解释为权威上级。普通正文链接、来源说明和引用清单也不得用于推断规范权威层级。

## 4. 修订与版本规则

1. 文件内容发生实质修改时，更新正文日期、版本和修订记录；版本历史保存逐次差异。
2. 只改错字、链接或格式时，按是否影响理解决定是否增加正文修订记录，但不得因此改变稳定路径。
3. 同一编号和规则主题保持同一路径；禁止用日期后缀并存多份现行正文。
4. 正式规则只保留当前有效逻辑。逐文件修订表只记录有助于理解当前语义的实质变化，机械格式调整和完整逐行历史由版本仓库承担。

## 5. 重排规则

1. 重排编号不得改变规范含义。
2. 重排必须保留旧编号到新编号的映射记录。
3. 重排必须同步更新文件名、规范目录和仍然有效的正式交叉引用；派生报告不得成为重排前置权威。
4. 重排期间旧编号文件不得与新编号文件同时作为权威规范。
5. 重排完成后，旧编号材料不得保留为并列权威入口；需要追溯时只保存编号映射和修订记录。

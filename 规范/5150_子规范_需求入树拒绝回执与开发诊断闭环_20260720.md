# 5150 子规范 需求入树拒绝回执与开发诊断闭环

编号：5150
日期：20260720
状态：正式子规范

## 1. 定位

本规范定义正式需求入树被拒绝后的结构化回执、治理分流和开发诊断闭环。拒绝回执不是普通需求，也不是日志文本。

上级规范：0050、3100、5110。

## 2. 回执合同

每次拒绝必须形成强类型回执，至少包含：

```text
拒绝类型与原因码
被拒绝候选和来源对象
失败阶段
缺失或冲突的结构证据
目标闭环类型和处理方
是否隔离来源路径
是否允许重提及重提条件
回执版本、发生时间和当前状态
```

拒绝回执由需求提交入口或需求治理处理器写入正式治理结构。文本只可作人读证据，不得驱动状态迁移。

## 3. 闭环分类

| 闭环 | 适用情况 | 合法收束 |
| --- | --- | --- |
| 来源任务内部闭环 | 参数、材料、候选或等待条件可由来源任务重筹办 | 材料获得、候选失效、重筹办或合法等待 |
| 观察验证闭环 | 当前状态、主体或外部证据尚不可判定 | 事实获得、确认不可观测或退回来源 |
| 学习闭环 | 正式因果或方法能力确有可定位知识缺口 | 正式学习需求入树、方法证据形成或路径关闭 |
| 内部治理闭环 | 父域、路径、去重或归属可由治理规则重判 | 重判成功、合并、合法拒绝或升级诊断 |
| 外设材料闭环 | 缺失的是外设内部材料而非业务目标 | 材料句柄形成、材料确认不可用或来源任务收束 |
| 开发诊断闭环 | 代码缺陷、结构损坏、非法状态机、逻辑上不允许的空值 | 代码修正、重新编译、回归通过且路径恢复或替换 |

新增拒绝类型前必须先定义处理器、允许状态、终态、升级条件和验证方式；缺处理方案的类型不得进入正式枚举。

## 4. 开发诊断硬边界

1. 逻辑上应存在的父需求、来源动态、关系端点、状态或提交入口缺失，属于开发诊断，不得普通需求化。
2. 开发诊断项必须记录复现上下文、需求、任务、方法、失败阶段、版本和隔离状态。
3. 来源任务必须收到实现缺陷阻断事实并停止同一路径重复筹办。
4. 日志和弹窗是人读提醒，不是关闭证据。
5. 只有代码修正、编译或等价静态验证、回归通过以及来源路径恢复/替换全部成立，才能关闭。

## 5. 状态推进

回执必须进入具名处理中状态并最终到达已解决、已转派、已合并、确认不可处理、来源失效或开发修复完成等合法终态。处理器自身无法收束时生成治理失败诊断，并按原因升级。

来源任务根据正式回执进入挂起、重筹办、降权、转观察、转学习、隔离或终止当前路径；不得只写日志后继续原路径。

## 6. 禁止事项

- 禁止静默忽略入树拒绝。
- 禁止把代码缺陷转成普通、学习或观察需求。
- 禁止把建立诊断项误写为缺陷已经修复。
- 禁止用字符串、函数名或日志文案作为机器状态。
- 禁止拒绝回执账本自行发明转派；转派和关闭必须由具名治理处理器执行。

## HY-Ego 现有实现中已隐含三层结构

你提出的新三层结构可以压缩成一句：**价值（及其因果态）不直接产出外部语义，而是调节“在虚拟存在层里激活哪些任务/方法本体与关系”，最后再把可执行部分投影成对外任务/方法。**

在 HY-Ego 当前代码与仓内规范里，这个分层并不是空白：它已经以“根约束层 + 可学习概念/因果层 + 任务/方法/需求的虚拟存在写回 + 任务执行投影”的方式出现，只是**尚未被统一命名为“虚拟存在层”**，并且**任务本体与关系实体还没有被显式抽象成统一对象模型**。

## 虚拟存在层规范 v1 草案

本草案刻意采用“尽量复用现有 HY-Ego 结构”的落地策略：  
- 虚拟存在层不另起炉灶成一个外部数据库，而是以“世界树虚拟世界 + 任务/方法/需求虚拟存在 + 概念树/因果/二次特征”为基础，补一层**任务实体库 + 关系实体库**的统一对象模型。fileciteturn26file0L1-L1 fileciteturn23file0L1-L1
- 激活/实例化/写回都遵守仓内既有的最高原则：根约束不可被上层学习改写；安全边界优先于排序优化。fileciteturn31file0L1-L1 fileciteturn24file0L1-L1

### 层定义

**虚拟存在层是什么**  
虚拟存在层是一个“存在图（existence graph）”与其运行时投影体系的组合：  
- 以**任务实体（TaskEntity）**、**方法实体（MethodEntity）**与**关系实体（RelationEntity）**为核心节点/边；  
- 支持“检索 → 激活 → 实例化 → 安全过滤 → 排序 → 对外投影”；  
- 支持“反馈 → 证据归档 → 写回节点/边属性 → 影响下一轮激活与排序”。fileciteturn26file0L1-L1

**虚拟存在层不是什么**  
- 不是“无边界记忆流”：它必须接受根约束与安全保障边界的裁剪。fileciteturn31file0L1-L1  
- 不是“自然语言命名系统”：身份以结构化签名为主，名称为可选解释字段。fileciteturn26file0L1-L1  
- 不是“执行器”：执行推进仍由任务执行器负责；虚拟存在层输出的是“当前任务实例/方法实例候选 + 约束与回退”。fileciteturn23file0L1-L1 fileciteturn29file0L1-L1

### 对象定义

本节给出 v1 最小对象模型，并标注与现有 HY-Ego 结构的对接点。

#### 任务实体

```text
任务实体（TaskEntity）
- task_id: 稳定主键（签名，非名称）
- goal: 任务目标（结构化目标；可回指需求/状态模板）
- completion_criteria: 完成判据（通常由需求“已满足”判定；也可附加）
- preconditions: 前置条件集合（可转成条件场景模板）
- constraints: 约束条件集合（安全/资源/时序等）
- risk_boundary: 风险边界（硬约束入口）
- decomposition_templates: 可分解模板集合（子任务生成规则/HTN 风格方法）
- related_methods: 关联方法集合（方法实体 id 列表 + 适配边）
- history: 历史轨迹引用（执行轨迹、成功率、失败征兆统计）
```

对接建议：  
- `completion_criteria` 默认以“需求是否满足”为权威判据，复用需求模块的状态比较逻辑；任务实体只需保存“它对应哪类需求模板/目标状态模板”。fileciteturn13file0L1-L1  
- `history` 默认写在任务实体的虚拟存在节点特征上，复用“任务虚拟存在写回接口”的模式。fileciteturn23file0L1-L1

#### 方法实体

```text
方法实体（MethodEntity）
- method_id: 稳定主键（签名）
```

### 激活规则

激活规则的目标是：**从虚拟存在层中激活一小撮可用的任务实体/方法实体/关系边，形成当前实例候选集。**  
核心原则沿用你的表述与仓内规范：  
- 安全先裁边界（硬过滤/规避），服务再排优先级（加分/排序/调权）。fileciteturn24file0L1-L1 fileciteturn26file0L1-L1

推荐规则链（与现有实现可直接对接）：

```text
输入：上下文/目标/反馈
  -> 计算安全态、服务态（从安全值/服务值/压力/稳定概念等聚合）
  -> 生成控制向量 CV
  -> 激活任务实体候选（按目标特征/需求签名检索 TaskEntity）
```

## 学术与工业参照对照

这一节的目的不是“贴论文”，而是验证你提出的三条关键结构性判断：  
- “任务本体 + 可分解模板”是合理的；  
- “安全先硬过滤，服务再排序”是更稳的；  
- “虚拟存在层应是可检索可写回的存在图，而非普通记忆”。

### 任务分解模板与 HTN 规划的对照

层级任务网络（HTN）规划的核心思想是：用任务分解把抽象任务递归变成更具体的子任务与可执行步骤；系统如 SHOP2 以 HTN 方式在规划时维护状态并生成可执行序列。citeturn3search0turn3search5  
你提出的“任务 → 子任务分解关系”“可分解模板”与“分解到多深由控制向量调节”，在理论上对应的正是 HTN 规划中“分解方法/任务网络深度控制”的工程化版本。

此外，基于 SHOP2 的 Web Service 组合研究表明：HTN 规划非常适合将“任务”组合成“可执行服务调用链”，并在规划过程中调用信息服务以辅助决策。citeturn8search6  
把它映射回你的体系：MethodEntity 可以类比为可调用服务（或技能），RelationEntity 则承载“可组合/可替代/依赖资源”的结构信息。

### “安全先硬约束”与安全 RL shielding 的对照

你强调“安全不要只做权重，最好先做硬约束；否则服务值一高可能把本不该走的方法顶上来”。这个判断与安全强化学习的两类主流路径高度一致：

- “在线 shielding”思路：在运行时对每一步动作做安全验证，超出阈值的动作直接被阻断（hard block），再在剩余动作里做优化选择。citeturn4search12turn4search3  
- “约束优化”思路：像 Constrained Policy Optimization（CPO）这样的算法强调将约束与回报分离，让策略在训练过程中保持近似满足约束，而不是把安全仅当作奖励权重的一部分。citeturn4search2turn4search1

因此，你在系统架构上把安全作为“裁边界”的硬过滤器、把服务作为“排序优化”的目标函数，这不是偏好问题，而是一种在安全敏感系统里更常见、更容易审计的结构选择。

### “技能/方法本体”与 options 框架的对照

强化学习中的 options 框架把“多步技能”作为一等对象，包含可启动集合、内部策略与终止条件，从而实现时间尺度抽象与可复用技能库。citeturn2search6turn2search1  
如果把你的 MethodEntity 投影到 options 的三元组：  
- `input_conditions` 对应 initiation set（可启动条件），  
- `steps_template` 对应内部策略，  
- `completion_criteria / termination` 对应终止条件，  
那么“方法实体作为可学习可检索的本体”在理论上是充分成立的。

### “存在图”表述与知识图谱的对照

你说“这层本质上不是普通存储，而更像一张存在图”。在知识表示领域，“用节点与边表示实体与关系”的最经典形式之一是 RDF 图：由 subject–predicate–object 三元组构成图结构，可用来表达资源与关系。citeturn8search0  
你提出的 `关系实体 = {起点, 终点, 类型, 权重, 生效条件, 更新时间}` 可以被视为“把 predicate（关系）对象化并为其添加属性”的工程化版本（即在边上挂权重与条件）。

## 差距、风险与改造路线

最后给出“对照检查”的结论性差距与推荐改造路径。这里不追求大而全，而追求能直接落到 HY-Ego 现有模块边界上，且不会破坏仓内已固定的根约束体系。

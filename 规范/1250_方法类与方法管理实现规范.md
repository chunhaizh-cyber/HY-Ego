# 1250 方法类与方法管理实现规范

## 1. 目标

本文用于固定当前 `方法类.ixx` 与 `方法管理模块_v0.ixx` 的实现边界，回答以下问题：

- 当前方法系统已经实现了哪些能力
- 方法首节点、条件节点、结果节点与方法虚拟存在分别负责什么
- 方法索引、运行证据与方法管理应如何分层
- 哪些事情属于方法系统，哪些不属于

本文对应当前工程实现：

- `方法类.ixx`
- `方法管理模块_v0.ixx`
- `主信息定义模块.ixx`


## 2. 定位

当前方法系统的定位不是单纯的“方法仓库”，而是四层组合：

- `方法结构层`：创建和维护方法首节点、条件节点、结果节点
- `方法账本层`：维护方法虚拟存在、内部状态场景与运行证据
- `方法索引层`：维护动作名、影响签名、结果 token、主信息指针等索引
- `方法管理层`：基于元层实体和关系做候选查询、绑定、替换与结果记录

当前方法系统不负责：

- 根需求和根方向定义
- 全局任务调度
- 叶子步骤的直接执行推进
- 把学习闭环直接替代成另一套方法树系统


## 3. 当前方法模型

当前实现中的方法模型，应统一理解为：

`方法首节点 -> 条件节点 -> 结果节点`

同时并行挂接两套运行期承载：

- `方法虚拟存在`
- `方法内部状态场景`

三层节点的职责应固定为：

- `方法首节点`
  - 承载动作句柄、本能 ID、来源、影响签名
  - 作为方法召回与绑定的统一锚点
- `条件节点`
  - 作为条件桶
  - 以条件签名聚合场景模板与条件判定索引
- `结果节点`
  - 作为结果桶
  - 以结果摘要索引与结果签名承接结果模板、召回 token 和运行证据

方法虚拟存在的职责应固定为：

- 挂方法运行状态
- 挂历史运行次数、成功次数、失败次数、最近错误码、最近调用时间
- 作为条件桶虚拟存在、结果桶虚拟存在的父锚点


## 4. 当前已经实现的能力

### 4.1 本能方法首节点出生

当前已经实现：

- 按本能动作 ID 查找或创建方法首节点
- 以动作句柄和本能 ID 为主键生成先天方法头
- 初始化方法虚拟存在与方法内部状态场景
- 由自我线程初始化阶段统一触发默认本能注册与方法头建档

这意味着：

- 本能函数并不是绕过方法树直接裸调用
- 本能函数一旦进入方法系统，就应尽量拥有稳定的方法头锚点
- 本能方法头的“首次建档”应在初始化阶段完成，运行期只做复用或证据写回

### 4.2 场景事实同步到方法虚拟存在

当前已经实现：

- 从输入场景同步特征到方法虚拟存在
- 从输出场景同步特征到方法虚拟存在
- 标记方法调用开始、结束
- 在结束阶段写入运行证据

这里的实现口径必须固定为：

- `输入/输出场景` 是证据来源
- `方法虚拟存在` 是方法运行账本
- 热路径上允许保留最小账本，重写回和扩展整理可延后到低频路径

### 4.3 条件节点与结果节点的桶化创建

当前已经实现：

- 按场景模板生成条件签名并取或创建条件节点
- 按结果摘要签名取或创建结果节点
- 在命中已有桶时融合模板内容
- 吸收条件判定索引与结果摘要索引

因此当前条件/结果节点应理解为：

- 不是一次性实例结果
- 而是供召回、复用、统计和写证据使用的“方法桶”

### 4.4 运行证据写回

当前已经实现的证据写回层级是：

`方法虚拟存在 -> 条件桶虚拟存在 -> 结果桶虚拟存在`

这条写回链承担：

- 运行次数累计
- 成败次数累计
- 最近错误码更新
- 样本数累计

因此必须固定：

- 方法统计不是只记在方法首节点
- 条件桶和结果桶也要独立沉淀局部证据

### 4.5 索引维护与召回

当前已经实现的索引至少包括：

- 动作名到方法首节点
- 影响签名到方法首节点
- 结果 token 到结果节点
- 结果节点到方法首节点
- 主信息指针到方法节点

当前也已经实现：

- 索引脏标记
- 按需重建索引
- 在读方法链时构建临时索引，再一次性交换

因此索引层的职责应固定为：

- 只负责加速查找
- 不能替代方法链真值
- 不能在索引不一致时改写方法链内容

### 4.6 方法管理模块的职责

`方法管理模块_v0` 当前已经实现一层独立的元层方法管理能力，主要负责：

- 注册方法元实体
- 查询候选方法
- 为任务选择最佳方法
- 绑定、替换任务与方法
- 记录执行结果和适配关系

它的评分依据已包含：

- 任务类型与方法适用类型
- 标签匹配
- 安全值投影风险
- 服务值投影收益
- 历史成功率
- 绑定关系中的累计修正

因此必须固定：

- `方法类` 负责方法树和方法账本
- `方法管理模块_v0` 负责元层候选与管理决策
- 二者互补，不应混成单一巨型模块


## 5. 锁与一致性边界

当前方法系统至少存在以下锁：

- `方法链.链表锁`
- `元缓存_mtx_`
- `本能_mtx_`
- `idx_mtx_`

它们的职责应固定为：

- `方法链.链表锁`
  - 保护方法树真值
  - 保护节点结构、模板融合与 `_已加锁` 写操作
- `元缓存_mtx_`
  - 保护方法虚拟存在相关缓存与内部元数据缓存
- `本能_mtx_`
  - 保护本能动作注册与本能查找缓存
- `idx_mtx_`
  - 保护索引缓存与脏标记

必须固定以下实现口径：

1. 方法链是真值，索引只是缓存。
2. 方法链发生结构或摘要变化后，应标记索引脏，而不是在热点路径立刻全量重建。
3. 重建索引时，应优先读方法链构建临时索引，再短时持有 `idx_mtx_` 交换结果。
4. 默认不允许在持有 `idx_mtx_` 时再去补拿方法链锁。
5. 如果已在方法链写锁内，只允许走专门的 `_已加锁` 路径，不得再反向嵌套拿 `idx_mtx_`。


## 6. 方法系统的硬约束

1. 方法首节点是统一锚点，任何后天条件、结果、证据都只能挂在既有方法头下扩展。
2. 条件节点和结果节点必须按“桶”理解，不得误写成一次性执行实例。
3. 方法虚拟存在是方法运行账本，不是任务调度状态机。
4. 方法索引只负责召回加速，不得被当成真值来源。
5. 索引重建必须遵守“扫描与交换分离”，避免长时间持有缓存锁。
6. 本能方法注册、方法树写入、索引刷新必须分层，不得在同一热点路径里全部叠加。
7. `方法管理模块_v0` 的评分和绑定属于管理决策，不得反向替代方法树结构定义。
8. 方法系统可以提供候选与证据，但不得直接推进任务状态机。


## 7. 不负责的事项

本文不负责：

- 任务何时进入筹办、维护、纠偏、收束
- 调度器如何处理任务返回消息
- 学习线程如何推进阶段机
- 因果模块如何提炼反推候选方法

# 1000 任务执行类实现规范（当前顺序执行骨架）

本文用于固定当前 `任务执行类.ixx` 的实现边界，说明：
- 当前 `任务执行器` 已经实现了哪些能力
- 当前任务执行链如何组织
- 当前哪些策略已经落地
- 当前仍然保留哪些后续扩展空间

本文中的 `筹办` 部分，应以上位规范 [0970_任务筹办规则.md](D:\鱼巢\规范\0970_任务筹办规则.md) 为准。

## 1. 任务执行类的定位

当前 `任务执行器` 不是单纯的“步骤执行器”，而是四个角色的组合：

- `筹办器`：按任务需求召回方法，并生成候选步骤
- `条件展开器`：当方法条件不满足时，把条件缺口转成子需求并派生子任务
- `顺序执行器`：推动当前步骤执行，生成实际结果，并推进下一步
- `局部调度器`：在父任务、步骤、并发子任务之间按优先级做运行期选择

因此，当前 `任务执行器` 已经是：

`需求驱动筹办器 + 子任务展开器 + 顺序执行器 + 局部调度器`

但它还不是：

- 完整全局任务调度中心
- 完整方法学习器
- 完整多目标决策器

## 2. 当前遵守的任务结构

当前执行器遵守的任务结构为：

`任务头结点 -> 步骤节点 -> 结果节点`

并且结合了并发子任务：

- `任务头结点.子链` 挂直属步骤节点
- `步骤节点.下节点` 表示同层并列步骤
- `步骤节点.子链` 挂该步骤下的结果节点
- `步骤节点.并发子任务头结点列表` 保存该步骤派生出的并发子任务
- `结果节点.下节点` 表示同一步下的不同预测结果
- `结果节点.子链` 表示命中该预测结果后的下一步骤

因此主执行链是：

`任务头 -> 当前步骤 -> 命中预测结果 -> 下一步骤`

如果该步骤存在补条件子任务，则顺序会先变成：

`任务头 -> 当前步骤 -> 并发子任务 -> 当前步骤继续执行 -> 结果 -> 下一步骤`

## 3. 当前已经实现的筹办规则

当前筹办应统一理解为：

- `先找方法`
- `再找条件`

其它诸如候选排序、步骤生成、补条件子任务等，都是这两步的结构化产物。

### 3.1 需求到方法的初选

当前自动筹办从任务头读取：

- `需求`
- `需求.被需求状态`
- `任务头全局条件场景`

然后遍历方法树中的方法首节点与方法结果节点。

一个方法能进入初选，当前要求是：

- 方法结果里存在与需求目标相同 `状态特征` 的结果状态

也就是：

`结果作用的特征类型一致 -> 可以进入初选`

### 3.2 方向性判断

如果目标值和方法预测值是数值型，当前实现会带入：

- 当前场景值
- 需求目标值
- 方法预测结果值

并做方向性判断：

- 预测值不能比当前值离目标更远
- 预测方向不能与目标方向相反

这条规则用于避免：

`南辕北辙`

### 3.3 条件匹配与条件缺口分析

当前不会只判断“方法现在能不能做”，还会进一步分析：

- 当前二次特征是否命中方法条件索引
- 方法条件场景模板里有哪些目标状态尚未满足

这里的条件判断，应与上位规范中的 `自我可用 = 可获取 + 有权限使用` 保持一致。

因此当前会得到三类结论：

- `无条件，可直接执行`
- `条件满足，可直接执行`
- `条件不满足，但能明确列出缺哪些状态`

### 3.4 规避过滤

当前执行器提供：

- `设置方法规避过滤`

用于注入外部规避规则，排除：

- 法律禁止
- 人文伦理禁止
- 系统规则禁止
- 当前运行上下文明确不允许的方法

### 3.5 并列步骤生成

通过结果匹配、方向判断、条件分析、规避过滤之后，执行器会：

- 把候选方法排序
- 为每个候选方法生成一个并列步骤节点
- 将候选方法写入步骤的 `可用方法首节点列表`
- 默认把当前最佳方法放到 `当前选中方法首节点`

因此当前筹办产物是：

`多个并列候选步骤`

## 4. 当前已经实现的预测结果生成

对于每个步骤节点，当前执行器会：

- 枚举该方法树下的结果节点
- 为每个方法结果生成一个预测结果节点
- 如果方法没有显式结果节点，则生成占位预测结果节点

当前预测结果节点会记录：

- 来源方法首节点
- 对应方法结果节点
- 结果场景
- 命中后动作

因此，一个步骤下可以存在多个预测结果节点，用于后续命中和转步。

## 5. 当前已经实现的补条件子任务机制

这是本轮实现中最关键的新能力。

当步骤的当前方法不能直接执行，但能明确分析出未满足状态时，执行器会：

1. 把每个缺口状态转换成一个子需求
2. 为每个子需求创建一个子任务头结点
3. 把子任务绑定到当前步骤的并发子任务列表
4. 将当前步骤置为 `等待中`
5. 先推进这些子任务，待其完成后再恢复当前步骤执行

因此现在已经实现：

`方法条件缺口 -> 子需求 -> 子任务`

这意味着当前版本已经不只是“当前条件下直接可执行层”，而是已经进入了：

`一层前置条件展开`

## 6. 当前已经实现的执行推进主链

当前 `推进一步` 的主链如下：

1. 读取任务头状态，跳过终结任务
2. 选择当前步骤
3. 如果没有步骤，或步骤仍为空白待筹办步骤，则自动筹办
4. 把任务头置为 `运行中`
5. 若当前步骤存在补条件子任务，则优先推进子任务
6. 子任务全部完成后，补全步骤预测结果
7. 调用叶子执行回调执行当前步骤
8. 在步骤下追加一个实际结果节点
9. 将实际结果命中到预测结果节点
10. 若预测结果下挂有下一步骤，则切到下一步骤
11. 否则按成功/失败/重试/挂起/学习进行收口

## 7. 当前已经实现的预测命中

当前预测命中优先走：

- `实际结果场景 -> 二次特征`
- `预测结果节点 -> 方法结果摘要索引`
- `结果摘要_匹配目标`

命中原则是：

`二次特征相等 或 二次特征相似`

如果没有摘要可用，才退回到：

- 同来源方法首节点优先
- 再退到步骤下首个预测结果

因此当前已经具备：

`预测结果 -> 实际结果 -> 命中预测`

的完整闭环。

## 8. 当前已经实现的子任务推进规则

当步骤下存在并发子任务时，当前执行器会：

- 汇总子任务总数、完成数、失败数
- 找出当前最值得推进的子任务
- 将父步骤与父任务的“当前子任务”指针同步到该子任务
- 子任务未完成时，父步骤保持等待
- 任一子任务失败时，父步骤与父任务直接失败收口
- 全部子任务完成后，父步骤恢复为 `就绪`

因此当前父子任务关系是：

`父任务负责调度，子任务优先执行，父步骤等待子任务完成`

## 9. 当前已经实现的失败处置

当前执行器已经支持以下失败处置：

- `回跳重试`
- `切换方法重试`
- `转入尝试学习`
- `挂起等待`
- `任务失败`

其中：

- 步骤允许重试时，会累加 `已重试次数`
- 步骤允许切换方法时，会在 `可用方法首节点列表` 中切换到下一个方法
- 配置了尝试学习任务时，可把失败转入学习兜底任务

## 10. 当前已经实现的优先级规则

当前调度优先级不再只是节点上静态保存的一个数，而是运行时动态计算。

当前规则为：

- `基准优先级` 统一追溯主任务头结点
- `局部优先级偏移` 保留步骤候选评分和局部调度修正
- 子任务头结点相对父任务头结点有明显加成
- 靠前步骤关联的任务、步骤、结果有顺序加成
- 近似单步、叶子、无并发子任务的最小任务有额外加成
- 父任务存在可推进直属子任务时会让位给子任务
- 等待中、挂起、终结状态不会进入正常调度队列

因此当前调度语义可以概括为：

`主任务给基础，子任务抢执行，靠前步骤先跑，最小叶子优先`

## 11. 当前公开接口的职责

当前 `任务执行器` 的主要公开接口包括：

- `设置叶子执行回调`
- `设置尝试学习任务`
- `设置方法规避过滤`
- `推进一步`
- `调度并推进一步`

## 12. 当前还没有实现的内容

虽然当前已经有了“一层条件缺口 -> 子任务”的展开能力，但以下能力还没有完全落地：

- 多层前置条件递归规划的收敛策略
- 时间、价格、风险等多维综合择优
- 预测外结果自动沉淀为方法补全任务
- 法律、人文、规则的内建策略层
- 更复杂的等待外部事件、循环回跳、阶段恢复机制

换句话说，当前版本已经实现的是：

`需求驱动筹办 + 一层条件展开 + 子任务优先推进 + 预测/实际闭环`

还没有完全实现的是：

`多层全局规划 + 完整学习补全 + 复杂策略控制`

## 13. 当前版本的一句话结论

当前 `任务执行类` 已经能够：

`按需求召回方法 -> 生成并列步骤 -> 分析方法条件缺口 -> 把缺口转成子任务 -> 子任务优先推进 -> 执行步骤 -> 生成实际结果 -> 用二次特征命中预测结果 -> 推进下一步或收口任务`


## 14. V1 重构约束

后续按消息驱动重构时，应再固定以下规则：

### 14.1 执行器只做单步执行

执行器不应再把“一个任务连续推到多步完成”当作默认行为。

应固定为：

- 每次只执行一个当前步骤
- 当前步骤结束后，必须返回一条明确消息
- 在收到该步返回消息之前，不得自动进入下一步

### 14.2 执行器只管理执行现场状态

后续应把执行器职责收敛为：

- `筹办中`
- `执行中`
- `等待回复`
- `等待子任务回复`

也就是说：

- 执行器负责“当前这一步正在发生什么”
- 执行器不负责最终裁决需求是否满足
- 执行器不负责决定任务是否完成或进入下一轮重筹办

### 14.3 每一步都必须返回消息

V1 可先固定以下返回消息类型：

1. `步骤完成`
2. `步骤失败`
3. `步骤无法执行`
4. `生成子任务`
5. `子任务完成`
6. `子任务失败`
7. `子任务无法执行`

这些消息统一交由 `任务调度器` 消费。

### 14.4 所有任务共用同一阶段闭环

无论是真根任务、伪根任务还是普通子任务，都应遵守：

- `局部阶段结束 -> 检查对应需求 -> 已满足则完成 -> 未全满足则重新筹办 -> 生成下一阶段任务`

区别只在于：

- 真根任务对应的是持续逼近型根需求，因此结构上不会真正进入终局完成
- 普通任务与伪根任务在其绑定需求满足后可以进入完成状态

### 14.5 子任务清零后的行为

必须固定：

- `子任务清零 = 当前已没有可执行的子任务`
- 清零时子任务状态只允许落在 `完成 / 失败 / 无法执行`
- 若子任务清零后父任务需求仍未满足，则应回到 `重新筹办`
- 父任务不直接“执行父需求”，而是重新组织下一阶段任务

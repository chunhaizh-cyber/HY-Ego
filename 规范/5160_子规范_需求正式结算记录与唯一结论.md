# 5160 子规范 需求正式结算记录与唯一结论

编号：5160
日期：20260721
状态：正式子规范

直接上级规范：3100。

直接依赖规范：4160、5100、5120、4320。

相关边界规范：1160、3200、4220、5230、4330。

## 1. 定位

本规范定义需求正式结算记录的业务身份、目标与实际状态证据、唯一正式结论、同义幂等、异义冲突和未达成分流。

需求正式结算记录只证明某一需求目标在确定目标合同和结果证据下形成了正式满足结论。它不是需求目标本身、任务完成事实、价值预算、动作成功回执或当前满足的永久布尔值。

## 2. 正式结算记录身份

第一轮一条正式结算记录的业务身份由需求稳定身份唯一确定，并完整绑定：

```text
需求稳定身份
+ 当前目标合同及其版本
+ 来源任务及其结果轮次
+ 本次实际状态
+ 恰一条动态证据
+ 零或一条因果引用证据
+ 结算规则版本
```

同一需求第一轮只允许零或一条已发布正式结算记录。幂等键只用于识别同一次提交请求，不建立第二种结算身份。物理节点、索引键、时间排序、日志编号和显示文本不得替代需求身份；同一需求下出现不同目标合同、不同来源任务、不同实际状态、不同证据或第二条正式记录时必须按异义冲突收口，不得另建并存记录或选择时间较新者覆盖。

## 3. 最小权威结构

正式结算记录必须能够回读以下稳定业务角色：

| 业务角色 | 必须证明的语义 |
| --- | --- |
| 结算需求 | 本次结算属于哪个需求 |
| 来源任务 | 哪个合法承接任务提供了本次结果 |
| 目标状态 | 本次采用的需求目标状态及目标合同版本 |
| 实际状态 | 上游已经形成并发布的实际结果状态 |
| 动态证据 | 恰一条已由上游形成的动态，证明实际状态如何形成及其与任务结果的来源链 |
| 因果证据 | 零或一条已由上游形成的轻量因果引用；不声明动作贡献时可以没有 |
| 正式结论 | 经正式判断收束形成的 `已满足` |
| 时间与版本 | 结算发生时间、规则版本和当前性 |

需求服务只能复核并绑定已经由任务、状态、动态或因果领域正式形成的材料，不得在结算入口根据原始值、日志、回执文本或显示摘要临时发明实际状态、动态或因果证据。

需求承接任务与结算来源任务是两个不同关系角色：前者说明哪些任务获得了承接授权，后者说明哪一任务结果实际支撑本次结算。二者不得混读或以普通引用替代。

## 4. 结算前置与比较收束

正式结算前必须依次验证：

1. 需求、目标合同、来源任务、任务结果轮次和授权当前有效且相互一致；
2. 目标状态与实际状态都具备合法宿主、特征、值域、场景、来源、时间和版本；
3. 实际状态来自任务结果或其它已确认状态入口，不是需求服务临时拼装；
4. 动态证据能够回到该实际状态和来源任务；声明动作贡献时，因果、动作动态和结果状态必须同源；
5. 禁止结果、副作用和价值结算资格按 5120、4320 独立审查，不与目标满足压成一个结论；
6. 当前实际状态与目标状态交由特征、状态和判断收束合同裁决。

需求服务不得直接比较裸 `I64`、枚举底值、显示值或缓存值。原始值只可作为完整特征值或状态材料的一部分；类型、维度、单位、值域、版本和左右语义先由正式服务复核，再按 5100、4160 的目标判断规则形成具名裁决。

固定值、区间、枚举集合、方向关系和复合目标分别按其正式目标状态与判断收束规则比较。不可比较、输入不足、版本过期或证据冲突不是“未满足”的同义词，必须单独返回具名结果。

## 5. 唯一正式结论与未达成分流

第一层正式结算结论只有：

```text
已满足
```

只有判断收束明确证明当前实际状态命中目标状态，且身份、当前性和必要证据前置全部通过时，才允许发布该结论。禁止结果或副作用可以与目标满足同时成立，必须进入独立风险事实并约束价值结算，但不得抹去已经成立的目标比较，也不得被 `已满足` 掩盖。

其它结果按以下方式收口：

| 判断结果 | 结算结构变化 | 后继 |
| --- | --- | --- |
| 当前目标未满足，且调用入口拥有本轮任务结果治理权 | 不建立正式结算记录 | 把未达成任务按任务合同迁移为待重筹办 |
| 当前目标未满足，且调用入口是普通组合入口 | 任务和需求结算结构均零变化 | 写前拒绝，不代替任务结果治理入口迁移任务 |
| 当前目标未满足，且任务已经完成后直接请求独立结算 | 需求结算结构零变化 | 只拒绝本次结算，保持既有任务状态 |
| 材料不足、过期或不可比较 | 不建立正式结算记录 | 返回具名材料缺口或当前性结果，不建立“证据不足”结算终态 |
| 禁止结果或副作用约束命中 | 目标确已满足时可同时保留正式满足结论和独立风险事实 | 进入风险与任务结果收束；不得据此自动形成价值结算 |
| 已发布材料自身矛盾或前置通过后内部写入、读回、发布异常 | 不得当作普通未满足 | 停止依赖路径并追根因 |

未达成、待重筹办、等待、取消和任务失败属于任务运行收束，不是需求正式结算结论。需求当前满足仍须按当前事实实时裁决；历史 `已满足` 记录不永久证明目标当前仍满足。

## 6. 幂等、冲突与历史

候选记录和既有记录必须先分别通过身份、结构、证据和结论合法性验证，随后才允许比较：

```text
同一需求 + 同一幂等键 + 同完整结构 + 同结论 -> 同义幂等，读回既有记录。
同一需求 + 同键异义、异目标、异实际状态、异来源任务、异证据、异结论或第二条正式记录 -> 异义冲突，零结构变化。
同一需求出现多个已发布正式记录 -> 内部不一致，停止并追根因。
不同需求 -> 分别保留各自记录，不得跨需求合并。
```

历史记录保持可追溯。所谓当前结算视图必须按需求、目标合同、活动授权、证据当前性和规则版本选取；最近结算时间、累计次数和显示摘要都只可作为由权威记录重建的非权威投影。

## 7. 写入权与消费边界

需求领域服务是正式结算记录的唯一业务提交方，并经专用数据操作在一个事务边界内完成身份、关系、结论和证据发布。任务、方法、学习、因果统计、线程、数据库、显示层和外部调用方只能提交材料或读取结果，不得直接改写结算事实。

任务领域可以消费结算结果收束本轮任务后继；因果和用途观察只能读取正式结算记录作为下游证据；安全或服务价值结算必须另按 4320 的预算、归因和副作用合同裁决，不得把 `已满足` 直接等同于价值已经入账。

## 8. 验证

验证至少覆盖：

- 固定值、区间、枚举集合、方向关系和复合判断均通过正式特征 / 状态 / 判断收束，不由需求服务比较裸值；
- 目标、实际状态、来源任务、动态和可选因果证据可完整回读；
- 当前目标未满足、不可比较、证据不足、禁止结果和内部不一致能够分别收口；
- 同业务身份同义幂等、异义冲突、多当前记录追根因和不同业务身份历史并存；
- 历史满足不冒充当前满足，非权威摘要清空不改变正式结算事实；
- 任务完成、需求满足与价值结算保持三个独立裁决。

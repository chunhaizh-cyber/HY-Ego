## 1. 定位

本规范的第一轮目标合同规定：需求结算记录负责表达“某个需求依据哪些目标状态、实际状态、任务和证据形成第一轮唯一正式结算结论”。第一轮正式结论只有 `已满足`；值不相等时不写需求最终结算，并按入口所有权分流：`#225` 新任务结果治理路径把未达成任务迁移为 `待重筹办`，`#225` 之外的既有通用组合入口保持任务和需求结算结构零变化，已完成任务直接调用独立结算时只拒绝本次结算。材料不完整时在写前拒绝或当前性返回，不建立独立“证据不足”结算终态。

需求结算记录不是需求目标本身，也不是任务完成本身。需求目标是目标状态；任务完成需要任务目标状态和实际结果状态比较；需求结算记录保存结算过程和证据链。

本规范覆盖主信息迁移 S4 中 MI-D3 的权威结算记录部分。MI-D3 中的安全 / 服务权重、累计结算、最近结算任务和时间若只是摘要，归入 `规范/非权威缓存与统计规范.md`。

## 1.1 第一轮确认口径

第一轮需求结算记录目标合同确认以下口径：

```text
第一轮不新增 `需求结算记录` 节点类型，使用需求结算实例状态作为权威结算记录锚点。
需求节点必须通过需求结算记录关系挂回第一轮唯一正式结算状态节点。
正式结算状态节点必须绑定本次结算采用的目标状态和实际状态。
实际状态节点第一轮必须由调用方传入已有状态节点，可来自任务服务的实际结果状态或状态服务在任务结果阶段创建的状态节点；需求服务只复核，不在结算入口临时发明实际状态。
第一轮正式结算必须接收上游已经创建的一条动态证据和零或一条因果引用；需求服务只复核，需求专用数据操作层只绑定，不新增动态服务或因果服务创建依赖。
同一需求第一轮只允许一个正式结算结论：同一幂等键、同完整结构且同结论时读回既有记录；同键异义、异结论或第二条正式记录冲突。
第一轮 I64 比较规则为：实际状态值等于目标状态值才达成并可写 `已满足`；值不相等时不写需求最终结算，并由入口所有者按 `#225` 待重筹办、既有通用组合零变化、已完成任务独立结算只拒绝三类路径分别收口；任一值或必要证据缺失时写前拒绝或逻辑内返回。
```

第一轮不新增 `需求结算记录` 节点类型。若后续需要独立结算记录节点，必须先修订节点类型与关系类型枚举规范。

## 3.1 结算记录关系边界

第一轮权威结算记录的关系角色及基数如下：

| 关系角色 | 源节点 | 目标节点 | 唯一性 | 用途 |
| --- | --- | --- | --- | --- |
| 需求结算记录 | 需求节点 | 结算状态节点 | 第一轮零或一条正式结算 | 读取该需求唯一正式结算；同结论完整重复幂等，异义冲突 |
| 结算来源任务 | 结算状态节点 | 任务节点 | 每次结算唯一 | 追溯本次结算来自哪个任务 |
| 结算目标状态 | 结算状态节点 | 状态节点 | 每次结算唯一 | 固化本次结算采用的目标状态 |
| 结算实际状态 | 结算状态节点 | 状态节点 | 每次结算唯一 | 固化本次结算采用的实际状态 |
| 结算动态证据 | 结算状态节点 | 动态节点 | 第一轮恰一条 | 追溯本次结算的动态证据 |
| 结算因果引用 | 结算状态节点 | 因果引用节点 | 第一轮零或一条 | 追溯轻量因果引用证据 |

多动态证据或多因果引用属于后续证据集合专项，不得在第一轮正式结算 DTO、关系角色或 #286 允许范围内临时扩展。

`需求承接任务` 和 `结算来源任务` 必须拆开：

```text
需求承接任务：需求节点 -> 任务节点，表达需求有哪些任务承接。
结算来源任务：结算状态节点 -> 任务节点，表达本次结算来自哪个任务。
```

外部调用方不得直接用裸 `引用` 关系混读需求任务事实；需求承接任务、结算来源任务、结算目标状态、结算实际状态必须通过需求服务或任务服务的专用读取入口返回。

## 3.2 实际状态来源

第一轮 `记录需求结算` 必须接收已有实际状态节点。

实际状态节点允许来源：

```text
任务服务记录的任务实际结果状态。
状态服务在任务结果阶段创建的实例状态。
后续已确认的状态提升或结果状态入口。
```

需求服务在结算入口只复核实际状态节点有效、类型正确，并绑定为本次结算实际状态；不得在结算入口根据日志、控制台输出、非权威摘要或临时特征值材料直接发明实际状态节点。

动态证据和可选因果引用也必须由上游正式入口先创建。需求服务只复核动态节点类型与可选因果句柄完整性，需求专用数据操作层只把一条动态和零或一条因果引用绑定到结算状态；不得由需求结算入口临时创建、补默认或扩为证据集合。

## 3.3 第一轮单一正式结算规则

JY-369 撤回第一轮“多次结算后选最近一条”的代码依据。第一轮定义为：

```text
同一需求零条正式结算 -> 尚未形成正式结论。
同一需求恰一条正式结算 -> 读取该结论。
同一幂等键、同完整结构和同结论 -> 幂等读回原记录。
同键异义、异结论或存在第二条正式结算 -> 冲突，不选择时间较新者覆盖。
```

“最近结算任务 / 时间”仍可作为由唯一权威记录重建的人读摘要，但不构成多次正式结算或覆盖旧结论的许可。

## 3.4 第一轮比较规则

第一轮只确认 I64 目标状态比较规则：

```text
实际状态值 == 目标状态值 -> 已达成，可形成唯一正式 `已满足` 结算。
实际状态值 != 目标状态值，且来自 #225 新任务结果治理路径 -> 未达成，任务迁移待重筹办，不写需求最终结算。
实际状态值 != 目标状态值，且来自 #225 之外的既有通用组合入口 -> 写前拒绝，任务和需求结算结构零变化。
实际状态值 != 目标状态值，且任务已完成后直接调用独立结算 -> 只拒绝本次结算，保持既有任务状态，需求结算结构零变化。
目标状态值、实际状态值或必要证据缺失 -> 写前拒绝或当前性逻辑返回，不形成独立“证据不足”正式终态。
```

后续比较方式扩展必须进入需求系统或需求结算规则专项，不得临时代码硬扩。

## 4. 结算记录和非权威摘要边界

JY-547 补充边界：鱼巢 `5110` 对父需求压力继承、AND / OR 权重分配与活动单位权重存在 `FISH-SPEC-CONFLICT-005`。本规范第一轮只裁决目标状态与实际状态比较及唯一正式结算，不裁决安全 / 服务权重预算；不得用本规范、现有局部 `1 / 0` 阻塞权重或非权威摘要替源冲突选边。

权威结算记录必须能回答：

```text
结算的是哪个需求。
该需求目标状态是什么。
结算时实际状态是什么。
来源任务是什么。
结算结果是什么。
结算发生时间戳是什么。
结算采用哪个目标状态、实际状态，来自哪一条动态以及零或一条因果引用。
```

非权威摘要只能回答：

```text
最近一次大致结算展示。
累计安全 / 服务结算展示。
诊断或兼容观察。
```

非权威摘要缺失、清空或重建，不得改变需求结算事实。

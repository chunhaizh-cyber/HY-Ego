# 错误结果与入口拒绝规范

编号：016
日期：2026-07-06
状态：已确认规范 / 核心内核 P1 / 第一轮错误结果与入口拒绝口径

## 1. 定位

本规范定义仓库、领域服务和业务写入口的错误结果、入口拒绝、业务失败和逻辑错误口径。

项目禁止把错误结构先写入再由校验逻辑、扫描器、日志、弹窗或控制台输出修复。所有会写入或改变结构的函数，必须在入口排除错误输入；入口通过后内部不符合预期，必须按逻辑错误追根因。

本规范只确认错误结果和入口拒绝规则，不证明代码已改造为统一错误对象，也不证明所有入口拒绝测试已覆盖。

## 2. 错误分层

### 2.1 入口拒绝

入口拒绝指输入在函数入口不满足前置条件。入口拒绝发生时：

```text
不得写节点。
不得写主信息。
不得写关系。
不得写索引。
不得写特征状态材料。
不得新增状态、动态或因果引用。
不得改变需求、任务或方法状态。
```

入口拒绝只能返回无效结果、空结果、局部失败值或结构化错误结果，不得把拒绝原因持久化为业务事实。

### 2.2 业务失败

业务失败指输入结构合法，但业务条件未满足。例如：

```text
任务目标未达成。
方法候选不存在。
动作入口不可用。
需求证据不足。
缓存缺失但权威结构仍可读。
```

业务失败可以返回领域枚举、空候选、证据不足结果或结构化错误结果，但不得伪装成结构错误。业务失败也不得自动补写结构来制造成功。

### 2.3 逻辑错误

逻辑错误指入口前置条件已经满足，但内部必需结构写入失败、读取结果不一致、唯一性被破坏或仓库返回结果和前置条件不一致。

典型逻辑错误包括：

```text
节点创建成功但后续必需关系写入失败。
任务节点创建成功但任务目标状态关系写入失败。
方法首节点创建成功但方法虚拟存在创建失败。
结算状态创建成功但结算目标状态关系写入失败。
预读目标和写入后关系目标不一致。
唯一关系写入后读取出多个目标。
内部参数值越过入口约束后仍出现非法值。
仓库返回结果和调用方已确认前置条件不一致。
```

逻辑错误必须追根因。宏开启时必须写逻辑错误日志并弹窗提醒；宏关闭时不得改变机器事实，但不得继续声明该路径通过，也不得把逻辑错误当作普通 `false` 静默吞掉。

## 3. 入口拒绝原因

### 3.1 核心仓库拒绝原因

第一版核心仓库入口拒绝原因包括：

```text
无效句柄
已删除句柄
版本不匹配
错误节点类型
错误关系类型
无效主信息
无效源节点
无效目标节点
缺少前置节点
缺少前置关系
越权写入方
生命周期不满足
重复写入冲突
未确认字段功能
未确认多目标语义
多目标关系读取不唯一
```

### 3.2 领域服务扩展拒绝原因

当前领域服务还必须覆盖以下扩展拒绝原因：

```text
缺发生时间戳
无效场景
无效存在主体
目标状态缺失
实际状态缺失
目标特征无效
当前特征状态材料缺失
特征值不归属目标特征
任务方法关系缺失
动作入口失效
非主动动作入口
缓存命名空间无效
缺少权威来源
需求目标状态不唯一
需求目标状态不是状态节点
来源动态缺失
移动到自身
移动到自身子孙
移动到当前父节点
没有当前父关系却请求重挂
```

这些原因不要求第一轮全部进入统一代码枚举，但必须作为领域扩展拒绝原因登记。后续新增长链路、跨服务写入或旧能力迁移时，应优先使用领域枚举或结构化错误结果表达这些原因。

## 4. 返回模型

### 4.1 当前轻量返回模型

当前第一轮代码允许继续使用：

```text
无效句柄
std::nullopt
局部 bool 成功 / 失败
领域枚举结果
```

要求：

```text
bool 只表示局部函数执行结果、入口拒绝结果或临时控制流，不得持久化为机器事实。
领域状态必须用枚举、状态值、计数、版本或结构关系表达。
拒绝路径不得写业务事实。
拒绝路径不得依赖日志、弹窗或控制台输出。
```

### 4.2 结构化错误结果设计口径

后续领域服务扩展、长链路写入、跨服务写入和旧能力迁移时，应逐步引入结构化错误结果。设计级结构如下：

```cpp
template <typename 值类型>
struct 入口结果 {
    bool 成功;
    值类型 值;
    错误类别 类别;
    错误原因 原因;
};
```

设计口径：

```text
成功只表示本次入口调用是否形成预期局部结果。
值只承载本次调用返回材料，不作为持久业务事实。
错误类别至少区分入口拒绝、业务失败和逻辑错误。
错误原因必须来自规范登记原因或领域规范扩展原因。
结构化错误结果只作为局部返回和诊断材料，不得持久化为业务事实。
```

不得为了记录详细错误信息，把错误写入主信息、日志、控制台、显示文本或临时 JSON 作为机器事实。

需求入树拒绝是领域治理的特殊后继，不把局部 `入口结果` 本身持久化。按 `规范/鱼巢需求生成入树与拒绝治理逻辑迁移规范.md`，局部拒绝成立后必须由需求治理入口另行形成强类型拒绝回执治理事件，并进入具名处理器；该领域治理事件与本节局部结构化错误结果不是同一对象。其机器结构和关闭状态尚未设计完成前，不得用日志或字符串代替。

## 5. 多步写入失败边界

多步写入函数在返回有效结果前，必须保证最小完整结构已经形成。

多步写入包括但不限于：

```text
按需求创建任务承接壳。
创建方法首节点。
创建方法条件 / 结果节点。
登记动作入口。
记录动作动态。
记录需求结算。
刷新非权威缓存。
节点普通父子重挂。
```

如果中途失败，必须满足以下之一：

```text
回滚已写结构。
显式失效已写结构。
返回无效结果，并保证所有读取入口不会把半结构读成有效业务事实。
标记事务失败边界，不得宣称该写入完成。
```

不得把半结构留给事后扫描器修复。不得只记录日志后继续推进依赖该路径的实现、验收或完成声明。

入口前置拒绝已经排除错误输入后，中途失败通常应按逻辑错误处理，并追查写入方、仓库入口、关系唯一性、句柄版本和事务边界。

## 6. 拒绝后结构不变化

入口拒绝验收不得只检查返回值。拒绝后结构不变化至少应按涉及范围检查：

```text
节点数量不变。
主信息数量不变。
关系数量不变。
索引数量不变。
特征状态材料不变。
状态节点不新增。
动态节点不新增。
因果引用节点不新增。
需求状态不变化。
任务状态不变化。
方法状态不变化。
缓存缺失不改变权威结构事实。
```

具体服务可以按涉及仓库裁剪，但必须说明裁剪依据。凡是声称入口拒绝通过的验收，必须给出拒绝前后可读结构证据。

## 7. 容错诊断

容错检查只允许在宏开启时记录和弹窗：

```text
HY_EGO_ENABLE_FAULT_TOLERANCE_CHECK
```

宏关闭时：

```text
不得写日志。
不得弹窗。
不得改变运行行为。
不得引入机器事实变化。
```

宏开启后：

```text
容错诊断可以记录日志并弹窗。
日志只做人读排错。
弹窗只做人读提醒。
日志和弹窗不得改变返回值、结构结果或机器事实。
弹窗意味着逻辑错误已经暴露，必须排除后才能继续推进依赖该路径的实现或验收。
```

容错诊断不得替代入口前置拒绝，不得作为自动修复入口。

## 8. 返回值不得事实化

以下反例禁止：

```text
方法执行返回 true 不等于方法成功事实。
任务服务返回 true 不等于任务完成事实。
需求服务返回已满足枚举不等于权威结算记录已完成。
索引查询返回候选不等于事实命中。
缓存命中不等于权威事实存在。
容错日志存在不等于结构错误已修复。
控制台输出通过不等于结构验收通过。
```

机器事实必须由节点、主信息、关系、索引、特征、状态、动态、因果引用、需求、任务、方法或权威结算结构承载。

## 9. 验收矩阵

| 验收项 | 必须证明 | 禁止替代 |
| --- | --- | --- |
| 入口拒绝 | 错误输入在入口被拒绝 | 事后扫描发现错误 |
| 结构不变化 | 拒绝前后涉及节点、主信息、关系、索引和领域结构数量或内容不变 | 只看返回值 |
| 业务失败 | 合法输入但业务条件未满足，返回领域失败结果 | 写入错误结构后再回滚口头说明 |
| 逻辑错误 | 内部不符合预期时记录为必须追根因问题 | 静默返回 false |
| 容错诊断默认关闭 | 默认构建不写容错日志、不弹窗 | 用日志作为机器事实 |
| 宏开启诊断 | 宏开启时可编译，触发后日志和弹窗不改变结构 | 弹窗后继续宣称通过 |
| 多步写入失败 | 回滚、失效、读取隔离或事务失败边界明确 | 半结构由扫描器修复 |

## 10. 完成声明边界

本规范确认只能证明：

```text
入口拒绝、业务失败和逻辑错误分层已确认。
当前轻量返回模型和后续结构化错误结果设计口径已确认。
领域扩展拒绝原因已登记。
多步写入失败边界和验收矩阵已确认。
```

不能证明：

```text
代码已改造为统一错误对象。
所有入口拒绝测试已覆盖。
所有多步写入已有完整回滚。
任务系统、方法系统、动作入口或需求结算系统完成。
旧项目能力已迁移。
```

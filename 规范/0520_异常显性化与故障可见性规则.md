# 0520 异常显性化与故障可见性规则

## 1. 目标

本文用于明确：初始化、调度、运行、退出过程中出现的异常，必须变成可见故障，不能再无声蒸发。


## 2. 核心原则

必须固定以下原则：

1. 初始化异常必须可见。
2. 调度异常必须可见。
3. 消息处理异常必须可见。
4. 收尾异常必须可见。
5. 故障一旦发生，必须留下摘要与日志。


## 3. 强制规则

### 3.1 不得静默吞异常

以下链路不得出现空 `catch`、注释型 `catch`、只吃掉不记录的 `catch`：

- 初始化链
- 调度链
- 消息处理链
- 周期维护链
- 退出收尾链

允许捕获异常，但捕获后的最低动作必须是：

1. 记录日志
2. 写入故障摘要
3. 切换到故障态或进入受控退出


### 3.2 故障必须可见

任何关键链路异常发生后，至少必须让外部看到：

- `运行状态 = 故障`
- `最近故障摘要`
- 可检索日志

必须明确：

- “程序没崩” 不等于 “系统没坏”
- “线程还在” 不等于 “异常没发生”


### 3.3 故障写入规则

故障写入至少必须统一做三件事：

1. 更新 `运行状态 = 故障`
2. 更新 `最近故障摘要`
3. 输出错误日志或异常日志

如果后续还要继续执行退出链，也不得清空已经形成的故障证据。


## 4. 统一处理壳

异常记录应优先收敛为统一壳函数，例如：

`安全执行_记录异常(上下文, 函数)`

其最低行为必须固定为：

- `catch (const std::exception&) -> 记录异常日志`
- `catch (...) -> 记录未知异常日志`

必须明确：

- 统一壳的作用是收口异常可见性
- 统一壳不是为了把异常“吃干净”


## 5. 对外展示规则

对外状态展示不得只依赖以下信息：

- 线程是否存在
- 线程对象是否非空
- `joinable()` 是否为真

对外必须同时展示：

- 线程活着
- 初始化完成
- 健康运行
- 最近故障

只要存在最近故障，就必须允许外部直接看到，不得藏在内部日志里。


## 6. 禁止事项

以下做法必须禁止：

- `catch (...) {}` 
- `catch (...) { return; }`
- `catch (...) { 继续当成功处理 }`
- 出错后只改局部变量，不改运行状态
- 出错后不写摘要，只留一句模糊注释


## 7. 收敛结论

收敛成一句话：

`异常可以被接住，但绝不能被消音；故障可以被收口，但绝不能对外不可见。`

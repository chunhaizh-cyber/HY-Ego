## 2. 安全值与服务值事实边界

1. 安全值和服务值是自我根特征系统中的正式值，不是任务成功次数、线程状态、日志评分或控制面板指标。
2. 物理安全事实、风险余量候选、服务对象改善事实和安全 / 服务账务结果必须分开；账务变化不能反过来证明物理安全或成功服务。
3. 安全值改变必须有合法生产者、事实证据、来源动态或动作动态，并能回溯到安全根或正式安全复合特征。
4. 服务值只能因经 G0 成功服务治理确认的成功服务人类事件增加。成功事件必须同时有服务业务动作、权限安全、需求 / 任务 / 方法链、服务对象前后状态、实际改善、验证、归因和闭环回写。
5. 任务完成、方法返回成功、文件写入、文本输出、调用发生或服务动作存在都不能单独证明成功服务。
6. 服务值只因正式动态时间衰减降低；自动衰减是维护事实，不是失败服务结算，也不能证明对象已被或未被服务。
7. 服务值始终受正式抽象定义中的明确正上限约束；未知、未计算和哨兵值不得当作满值或默认成功。

## 3. 结算准入、去重和计算

1. 结算前必须确认实际状态改变、任务正式完成、目标达成比例大于零、预算合法且仍有额度；否则只能保留过程证据或候选。
2. 同一个同源变化证据组只允许结算一次。叶子已经结算的部分不得由父任务、管理任务、汇总任务或筹办壳重复结算。
3. 任务 / 需求父链只说明治理来源，不能单独准许核心值结算；父链必须接回对应根特征、正式因果证据和实际结果。
4. 结算必须比较 before、target、actual；不能计算目标距离时，达成比例为零。部分结算必须有源规范允许的特殊条件和可核对剩余额度。
5. 因果距离、候选优先级、父预算、任务权重和结算贡献层级必须分离；不得把某一字段复用于全部语义。
6. 同级因素进入上层前先按源规范统一方向、版本和聚合规则；风险压力必须先转换为风险余量候选。缺失因素不得使其它因素自动变重。
7. 公式和因素集合变更必须推进版本；不同版本候选不得直接混作同一历史序列。
8. 比例和贡献使用整数安全计算，乘法必须使用足够宽的中间值或安全乘除；溢出、下溢或证据不足不得强行入账最小值 1。
9. `I64_MAX` 只能在源规范规定的评估满刻度 / 比例刻度语境使用，不能兼作未知、无限、默认成功或异常兜底。
10. 循环不可维持等根控制态写入与普通收益 / 风险结算分离；普通路径不得把安全值平滑进入根控制保留值。

## 4. 服务闭环、休眠和唤醒

1. 没有直接服务触发事实时，只能产生监听服务触发、检查服务机会、校准服务能力、休眠或唤醒等间接维护候选；不得虚构服务对象、目标、成功或结算。
2. 只有直接服务触发已经存在且上下文缺口确实阻断行动、验证、归因或回写时，才可生成直接服务因果链上下文缺口候选；仍须经过需求统一入口。
3. 服务对象信息不足和反馈不足先记录尝试、方向比较、沟通回执和阻塞证据；不得直接固化为抽象特征或服务成功。
4. 服务前状态未知无法计算改善方向，服务后状态未知无法验证结果；任一缺失均不得结算服务值。
5. 休眠不得使用固定物理无服务时间。必须依据安全值、服务能力、有效服务需求、有效无成功服务时间和动态休眠阈值共同裁决。
6. 服务值按动态规则归零且有效无成功服务时间达到阈值后，才可进入低活动、可唤醒休眠；新的有效人类服务需求、服务能力恢复、安全改善或明确唤醒可触发重新评估。
7. 当前人类服务任务安全、目标明确且有阶段性进展时，不得按普通无成功服务路径休眠。

## 5. 生存深层需求投影

1. 生存深层需求表达未来风险、应急、恢复、维护和可持续能力，不得把协作、信誉或能力提升的显示名称直接解释为需求身份。
2. 物理损害临近程度、可控性、可确认性和恢复能力用于产生候选优先级与投影衰减材料，不直接结算安全 / 服务值。
3. 不可确定原因先形成确认候选；不可控制原因形成隔离、备份、预警或替代候选；环境不可预测和维护通道不可达进入相应深层治理候选。
4. 应急止损、近层安全和深层治理必须保持层级与父域可追溯；正式入树仍须通过需求统一门禁，不能由因果投影视图直接创建。
5. 因果置信度是由基础计数、证据类型和查询参数计算的派生材料，不得复制为因果主信息字段或需求主信息字段。

## 6. 自我根特征、自检与因果闭环

1. 自我安全、服务和其它根特征必须先有抽象定义、合法值域、比较规则、生产者和根因果坐标；代码硬编码正常值或显示阈值不得成为正式定义。
2. 自检读取当前自我实例的正式特征状态，与根定义比较，只输出强类型缺口候选或当前无缺口材料；不得直接写需求树、创建任务或修改根值。
3. 自我存在服务闭环必须能从根特征缺口追溯到原因、条件、可改变 / 不可改变、可验证 / 不可验证和结果回写；不能从定义词条直接跳到需求。
4. 满足正式因果链和能力补齐范围时，6120 允许创建或复用方法头。该条加入 `FISH-SPEC-CONFLICT-001` 的“允许创建”一侧，仍不得解除无候选时是否自动创建方法头的冻结。
5. 当前自我实例、根特征、根需求、任务、方法、动作动态、实际状态回读和下一轮更新必须可用稳定句柄串联；初始化节点、日志或面板显示不构成闭环。

## 7. 权重源冲突门控

6100 的结算贡献权重规则站在“父需求预算向子需求分配、任务承接时固化、OR 单路径取得相应预算、结算读取固化预算”的一侧；4320 提供同侧投影证据。它们与 5110 第 119 条“活动需求单位权重且父压力不继承”仍相反。

因此 `FISH-SPEC-CONFLICT-005` 继续冻结：

```text
父压力是否继承
AND / OR 如何分配预算
任务何时以及如何固化权重
结算如何消费与回收预算
现有局部 1 / 0 请求材料能否参与权威结算
```

无冲突的结算准入、证据、去重、达成比例和安全整数计算继续继承，不因权重分支冻结而撤销。

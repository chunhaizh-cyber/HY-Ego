## 2. 特征表达与判断收束

1. 一阶特征表达单一主体可直接取得、测量或正式推导的属性；复合特征表达多个已定义成分的组合；二次特征表达对象或特征之间的关系性属性。三者不得因显示方便互相替代。
2. 抽象特征定义值类型、值域、单位、比较、成分和合法生产者；实例特征承载主体在场景中的具体槽位或表达。定义、槽位、当前值、历史值和判断结果必须分离。
3. 基础抽象名称只作表达和检索材料，不能裁决类型、层级、身份或相等。稳定身份由句柄和正式关系裁决。
4. 可以直接取得的原始值按其正式值类型承载；测量、计数、评分、版本、来源和句柄值必须保留单位、来源、当前性和解释边界。
5. 原始分类型枚举只表达相互区分的原始类别；生命周期枚举只表达具有合法迁移矩阵的阶段；判断型枚举和 bool 不得直接伪装成一阶世界事实。
6. 判断收束必须保留输入特征族、输入版本、收束规则、阈值或比较规则、写入方和证据。只有这些前置已经正式定义，判断结果才可以成为治理状态或派生材料。
7. 普通 bool 只可表示单次函数是否成功或局部控制，不得持久化为需求满足、任务完成、方法成功、存在与否、风险事实或因果结论。
8. 外设和视觉上行的质量、清晰、完整、稳定、可信、可用等判断只能先作为带证据的材料质量状态；未经自我侧验证不得升格为世界特征。

## 3. 特征值与比较

1. 特征值只服务于已定义特征；不得脱离特征类型和值域独立成为通用业务值，也不得由需求、任务或方法服务直接操作特征值服务。
2. 同一特征的值比较必须先验证类型、值域、单位和版本兼容，再返回大于、等于、小于或正式的不适用 / 不可比结果；不得用裸内存、显示文本或哈希相等代替完整值比较。
3. 三态比较是底层值逻辑，不直接宣布目标满足、风险成立、任务完成或方法成功；上层必须结合比较符、目标语义和证据裁决。
4. 枚举型实例值必须来自抽象特征定义中的正式合法值或其稳定映射；代码常量、控制面板选项、数据库文本和外设编号不得直接扩充值域。
5. 枚举值创建、选择和比较必须验证其所属定义；同名但不同定义的枚举值不得相等。
6. 当前值更新必须保留当前关系唯一性、历史审计和版本推进；版本号不替代发生时间，也不表达业务大小。

## 4. 复合特征与二次特征

1. 抽象复合特征通过正式组成关系连接已有抽象特征；组成顺序、必要性、逻辑关系和比较规则必须显式，不得由树形位置或名称猜测。
2. 复合特征的实例表达必须能追溯到每个组成槽位的主体、场景、当前值和版本；缺组成时只能形成不完整材料，不能自动补默认事实。
3. 二次特征只表达两个或多个一阶对象 / 特征 / 状态之间的关系属性，例如相对位置、距离、包含、连接、相似或运动关系；其端点必须使用稳定句柄。
4. 可由当前一阶材料稳定计算且无需长期治理的二次结果可以保持值式候选；需要长期引用、学习或因果使用时，必须经二次特征服务正式提交并记录来源证据。
5. 二次特征不得复制端点的一阶特征，不得把计算缓存当权威，不得用自然语言关系名替代正式关系类型。

## 5. 状态、动态与聚合

1. 实例状态是主体在场景中某时刻或区间的零散事实；实例动态是主体从前态到后态的零散变化事实。二者必须有主体、场景和发生时间。
2. 抽象状态和抽象动态是长期定义，不携带单次发生时间；实例不得反向覆盖定义。
3. 动态按原始变化、结构化动态、聚合片段 / 回合和可复用候选分层读取。上层聚合只能引用下层正式动态，不得复制或改写原始证据。
4. 聚合必须明确窗口、主体、场景、时间顺序、输入集合、完整性和版本；缺材料只能返回候选为空、证据不足或等待，不得补写事实。
5. 重复片段、运动基元或模式只在多次独立样本和完整键复核后成为候选；单次相关、哈希命中或显示相似不得直接成为抽象动态或稳定方法。
6. 状态和动态不得被组织成业务状态树 / 动态树；历史、聚合和查询使用关系、索引与值式视图。

## 6. 动作动态与因果账本

1. 动作动态的来源必须是正式方法执行或获准领域服务写入；线程、消息、外设回调、函数名和日志不是动作来源。
2. 动作请求、方法执行、动作动态、状态变化动态和结果证据必须保持可分辨并可追溯，不能用一个“成功”字段合并。
3. 状态迁移账记录状态从前态到后态的变化事实；动作致变账记录某次动作与变化之间的归因候选。两账可以同源共存，但必须通过同源证据组识别同一次变化，禁止重复结算。
4. 被动变化、提交动作、结算动作和内部治理判断可以形成触发或解释证据，但不得冒充可执行的主动动作致变因果。
5. 因果补完只能沿已有条件、动态、状态、方法和证据关系补齐缺口候选；不得在缺来源动态、缺条件或缺结果时凭名称和统计相关创建稳定因果。
6. 单个轻量因果引用只说明来源动态通过入口准入，不持久化来源动态关系，也不证明稳定因果。稳定性必须由多样本、评分、版本和复判形成。
7. 因果信息可用于解释、候选召回、条件检查和需求路径材料，但最终任务化仍必须回到正式需求、任务和方法结构。

## 7. 状态转换、目标投影与结算

1. 状态转换因果结构从当前状态、目标状态、必要条件、可用方法和预期结果逐层投影；不得把因果显示树本身持久化为第二套权威需求树。
2. 目标投影只生成候选路径、条件材料、子目标建议或权重预算材料；正式需求仍经需求统一入口入树，正式任务仍由任务服务承接。
3. OR 路径、AND 子目标、父预算和任务结算之间必须保持来源可追溯、同一结算来源去重和目标达成比例证据。
4. 权重不得由节点数量、遍历顺序、线程、当前任务数量或显示排序产生。
5. 鱼巢 4320 和 6100 的父预算分配 / OR 单路径取得预算条款加入 `FISH-SPEC-CONFLICT-005` 的继承分配一侧；与 5110 第 119 条活动单位权重且父压力不继承仍相反。权威权重分配、继承、任务固化和结算预算继续冻结。

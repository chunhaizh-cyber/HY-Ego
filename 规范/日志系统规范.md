# 日志系统规范

日期：2026-07-21

状态：长期非机器日志工程治理入口；未登记于 `规范/规范目录.md`

## 1. 定位与约束来源

本入口只定义日志分类、文件隔离、人读诊断字段和调试宏治理。日志只用于人读诊断、运行观察、审计辅助和调试定位，不承载机器逻辑，不参与需求满足、任务完成、方法成功、世界事实、结构修复或业务状态裁决。

本入口不是 `规范/规范目录.md` 登记的机器逻辑规范，不定义机器事实、对象合同、持久化格式或代码完成状态；其约束力来自 `AGENTS.md` 对日志工程治理入口的指定。代码、详细设计和计划涉及日志时必须遵守本入口，同时继续服从 0050 及目标对象现行机器规范。

日志内容、日志是否出现、日志文件是否存在以及日志扫描结果都不能覆盖仓库读回、领域服务结果、正式结构或验收证据。日志与上述证据冲突时，日志只作为待核诊断材料。

## 2. 分类与隔离

日志固定分为：

```text
逻辑错误日志
运行日志
调试日志
事件日志
```

所有项目日志统一写入 `D:\海中鱼巣\日志`，并按类别使用独立文件或独立子目录。不同类别不得混写；一次输出只能先确定一个主要类别，不得通过复制到多个类别制造多个事实来源。

### 2.1 逻辑错误日志

逻辑错误日志只记录已经由正式入口或内部检查识别出的异常诊断，包括写前拒绝原因和前置通过后的内部逻辑错误。它不得自行决定错误分类、修改返回值、修复结构或驱动业务继续。

逻辑错误日志与弹窗同时出现时，两者必须表达同一诊断原因；弹窗仍只做人读提醒。内部逻辑错误暴露后，依赖路径是否停止以及如何追根因，由 0050、目标对象规范和有效计划裁决，不由日志裁决。

### 2.2 运行日志

运行日志只记录普通运行观察，例如启动、关闭、模块装配、外部材料接收摘要、公开入口调用摘要和环境提示。运行日志不得与逻辑错误日志混写，也不得把“开始”“结束”“成功”“失败”等文本升级为业务状态或完成证据。

运行日志应保持克制，避免在高频路径重复输出稳定事实；需要逐项验收的结果应进入计划指定的验收输出或实施记录，而不是依靠运行日志替代。

### 2.3 调试日志

调试日志只服务于一个具名排查切片，必须独立于逻辑错误日志、运行日志和事件日志。调试日志不得长期默认开启，不得成为生产功能输出，不得作为测试通过、结构合法或业务成功的唯一依据。

每个调试切片至少登记：

```text
调试目的
启用宏
影响文件
输出位置
关闭条件
验证方式
```

### 2.4 事件日志

事件日志只记录已经由正式结构确认后提供给人读取或审计的事件摘要。事件日志不得反向写入领域结构，不得作为恢复输入，不得回滚已经发布的业务事实，也不得替代运行期仓库读回。

二进制事件段、序列化布局、版本号、校验算法、原子替换、轮换、重放和恢复合同不属于本入口；需要时必须进入相应机器规范、详细设计和已登记计划，本文件不复制这些格式或实现状态。

## 3. 诊断字段

每条日志只记录完成诊断所需的最小字段。通用字段为：

```text
发生时间
日志类别
所属模块
入口或操作名称
诊断分类
相关稳定句柄、主键或编号（适用时）
来源请求或调用摘要（适用时）
人读原因与建议核对路径
```

字段值必须来自调用点已经持有的材料，不得为了补齐日志而新增业务查询、结构写入、锁顺序、返回分支或机器字段。敏感值、大块二进制、完整外部负载和无界容器内容不得直接写入日志；需要定位时只记录受控摘要或稳定标识。

诊断字段中的句柄、编号、名称、分类和原因都只是人读投影。即使字段完整，也不能单独证明对象存在、关系成立、任务完成、需求满足、方法成功或因果成立。

## 4. 调试宏治理

调试日志必须按具名切片使用预编译宏控制。宏命名应能定位切片范围：

```text
HY_EGO_DEBUG_LOG_<切片名>
```

默认构建不得启用任何调试日志切片宏。宏关闭时必须满足：

```text
不写该切片调试日志
不改变运行行为
不改变返回值
不新增查询、结构写入或机器事实
不影响验收结果
```

宏开启时只允许增加人读调试输出，不得替代入口拒绝、修复结构、补写业务状态、改变错误处理或成为验收通过的唯一依据。

多个调试切片不得共用一个无法独立关闭的总宏。调试结束后必须移除默认启用项或构建参数，并确认默认构建不产生该切片日志。暂时保留的调试代码必须满足：默认关闭、边界清晰、输出隔离、关闭条件明确且不会造成高频无界日志。

容错宏、诊断宏和调试宏关闭时均不得改变正常业务行为；开启后也只能暴露诊断，不得取得机器写入权或业务裁决权。

## 5. 禁止事项

禁止：

```text
用日志承载、反推或修复机器事实
用日志替代入口前置拒绝、结构读回或正式验收
用日志文本判断任务完成、需求满足、方法成功或因果成立
逻辑错误日志、运行日志、调试日志和事件日志混写
为了补日志字段新增业务写入、锁顺序或返回分支
长期默认开启调试日志
多个调试切片共用一个无法独立关闭的总宏
调试结束后遗留高频、无界或无关闭条件的输出
把详细设计、持久化格式、计划进度或当前实现状态写入本入口
```

## 6. 验证与完成边界

修订本入口至少运行：

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

涉及代码的日志验证必须由对应有效计划另行规定，至少区分默认宏关闭、指定切片开启、分类隔离、运行行为不变和日志不参与结构裁决。本入口存在或检查通过，只证明长期日志工程治理边界已经形成，不证明任何日志代码已经实现、构建通过、运行接通、持久化成立或业务能力完成。

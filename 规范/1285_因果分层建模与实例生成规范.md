# 1285 因果分层建模与实例生成规范

## 1. 目标

本文用于固定因果信息体系下一阶段的建模边界与落地顺序，回答以下问题：

- 因果信息应先拆成哪些类型
- 单条因果成立的最小前提是什么
- 场景因果与存在因果如何分层生成
- 跨层变化如何解释而不乱连
- 抽象因果模板应放在什么阶段再做

本文不是当前 `因果信息类.ixx` 的现状说明，而是后续因果重构的上位目标规范。

## 2. 定位

因果体系的正确切法不是：

- 一套无层级、全域通用、什么都能往里塞的总因果

而应固定为三层对象：

1. `实例因果信息`
2. `抽象因果模板`
3. `因果链`

并且当前阶段必须固定：

- 先做 `实例因果信息`
- 先做 `轻量因果链`
- 暂不先做 `抽象因果模板沉淀`

可以收敛为一句话：

- `先把实例因果做成有边界、可校验、可生成的运行对象，再谈模板沉淀。`


## 3. 核心原则

### 3.1 单条因果的硬约束

所有单条因果生成前，必须先过统一校验闸门：

- `同锚点 + 同层级 + 时间有序，才能生成单条因果。`

具体固定为：

1. 锚点只能是 `场景` 或 `存在`
2. 条件状态、动作、结果状态或结果动态，必须都挂在同一个锚点下
3. 条件状态与结果状态必须属于同一状态层级
4. 条件时间 `<=` 动作时间 `<=` 结果时间

### 3.2 跨层边界

必须固定：

- `跨层解释允许，跨层直连不允许`

因此：

- 单条因果不允许跨层
- 因果链允许跨层
- 但因果链中的每一跳都必须是合法的同层因果

### 3.3 因果标准表达

单条因果的标准表达固定为：

- `同层条件状态集合 + 动作/事件 -> 同层结果状态或结果动态`

### 3.4 动作信息与事件信息的关系

当前因果口径固定为：

1. 对自我内部过程而言，`动作信息` 可以视为 `事件信息` 的核心载体。
2. 单条因果中的“动作”优先解释为一条有开始、有结束的事件。
3. 若已存在显式动作事件，不得再用结果状态特征名回退冒充动作名。

因此单条因果更准确的表达应理解为：

- `同层条件状态集合 + 动作事件 -> 同层结果状态或结果动态`

### 3.5 事件边界与因果样本顺序

当前阶段必须固定事件型因果样本的最小顺序：

1. `动作开始运行`
2. `中间状态变化或结果动态`
3. `动作完成或动作失败`

这条顺序用于：

- 因果提炼
- 展示层解释
- 后续模板沉淀

例如：

1. `定时衰减服务值开始运行`
2. `服务值下降`
3. `定时衰减服务值完成`

不得记录成：

1. `服务值下降`
2. 再把 `服务值` 回退解释成动作


## 4. 类型边界

### 4.1 实例因果信息

`实例因果信息` 指的是：

- 一次真实发生过的、可定位到具体锚点与具体时间顺序的因果事实

它回答的是：

- 这次为什么变了
- 在哪个场景或哪个存在里变的
- 条件是什么
- 动作是什么
- 结果是什么

### 4.2 抽象因果模板

`抽象因果模板` 指的是：

- 由多条同层实例因果归纳沉淀出的可复用模式

它回答的是：

- 这类变化通常怎么发生
- 哪些条件组合经常导向哪些结果

但当前阶段必须固定：

- 抽象因果模板不是第一步实现目标

### 4.3 因果链

`因果链` 指的是：

- 多条合法单跳因果按时间和桥接关系组织起来的解释链

它不是新的“总因果”，而是实例因果的组织对象。


## 5. 锚点与层级规则

### 5.1 锚点容器

单条因果的锚点容器只允许两类：

1. `场景`
2. `存在`

### 5.2 场景因果

场景因果只允许描述场景内公开可观测、可比较、可验证的变化，例如：

- 存在间距离变化
- 接触与脱离
- 遮挡与显露
- 场景内位置变化
- 场景整体状态变化

### 5.3 存在因果

存在因果只允许描述某个存在自身内部的变化，例如：

- 自身特征变化
- 自身部件关系变化
- 自身内部状态迁移

### 5.4 状态层级

状态层级固定理解为：

- `观察对象的宿主边界 + 状态展开深度`

当前阶段至少要求：

- 单条因果中，条件与结果必须同层

### 5.5 状态面

`状态面` 是可选增强约束，用于进一步限制因果可比域，例如：

- 几何面
- 能量面
- 功能面
- 评价面
- 任务面

当前阶段可以先不强制全量实现，但数据结构应预留 `状态面签名`。


## 6. 实例因果对象规范

实例因果主信息建议固定为以下最小结构：

```cpp
export enum class 枚举_因果锚点类型 : std::uint8_t {
    未定义 = 0,
    场景 = 1,
    存在 = 2
};

export enum class 枚举_因果来源类型 : std::uint8_t {
    未定义 = 0,
    观察生成 = 1,
    执行生成 = 2,
    推断生成 = 3
};

export class 因果实例主信息类 {
public:
    枚举_因果锚点类型 锚点类型 = 枚举_因果锚点类型::未定义;
    std::uint64_t 锚点主键 = 0;
    std::uint32_t 状态层级 = 0;
    std::uint64_t 状态面签名 = 0;

    时间戳 条件时间 = 0;
    时间戳 动作时间 = 0;
    时间戳 结果时间 = 0;

    std::vector<状态节点类*> 条件状态列表{};
    基础信息节点类* 动作信息 = nullptr;
    动态节点类* 动作动态 = nullptr;

    std::vector<状态节点类*> 结果状态列表{};
    std::vector<动态节点类*> 结果动态列表{};

    枚举_因果来源类型 来源类型 = 枚举_因果来源类型::未定义;
    std::uint64_t 来源方法主键 = 0;
    std::uint64_t 来源任务主键 = 0;
    std::int64_t 置信度 = 0;
    bool 已验证 = false;
};
```

必须固定：

1. 因不是自由文本，而是 `条件状态 + 动作`
2. 果不是自由文本，而是 `结果状态/结果动态`
3. 单条因果必须归属于一个明确锚点
4. 若显式动作动态已回带 `结果状态`，则 `主结果状态` 必须优先取自动作动态
5. `结果状态列表` 继续保留，用于保存结果全集、附带结果与兜底回退

### 6.1 动作动态回带主结果状态

当前实例因果的主结果选择规则固定为：

1. 优先读取结束相位动作动态中的 `结果状态`
2. 若动作动态未挂 `结果状态`，再回退到结果状态列表与结果动态推断

因此：

- `动作动态` 用来指定“哪一个状态是本次动作的主结果”
- `结果状态列表` 用来保存“本次动作产生了哪些结果状态”
- `状态节点` 仍是结果事实的权威记录

不得反过来把动作动态当作结果事实的唯一载体。


## 7. 合法性校验规范

必须实现统一校验函数：

```cpp
bool 校验因果实例合法性(const 因果实例主信息类& x);
```

该函数至少检查：

1. 锚点类型是否有效
2. 锚点主键是否有效
3. 条件状态列表是否非空
4. 动作信息或动作动态是否至少存在一个
5. 结果状态列表或结果动态列表是否至少存在一个
6. 条件与结果是否属于同一锚点
7. 条件与结果是否属于同一层级
8. 时间是否满足 `条件时间 <= 动作时间 <= 结果时间`
9. 若动作动态已回带主结果状态，则 `结果时间` 应优先与该结果状态时间对齐

未通过校验时：

- 禁止生成单条因果
- 可退化为关联信息
- 或进入因果链桥接流程


## 8. 生成通道规范

当前阶段只建议先实现两条直接生成通道。

### 8.1 场景生成场景因果

接口建议：

```cpp
std::vector<因果节点类*> 生成场景因果实例(
    场景节点类* 前场景,
    场景节点类* 后场景);
```

只允许产出：

- 场景锚点的因果实例
- 场景层的关系变化、位置变化、接触变化、遮挡变化、整体状态变化

### 8.2 存在生成存在因果

接口建议：

```cpp
std::vector<因果节点类*> 生成存在因果实例(
    存在节点类* 前存在,
    存在节点类* 后存在);
```

只允许产出：

- 该存在锚点的因果实例
- 存在层的自身特征变化、部件变化、内部状态迁移

### 8.3 执行生成

若因果来自方法或任务执行回执，仍必须满足：

- 同锚点
- 同层级
- 时间有序

否则不得越过校验直接写入因果实例。


## 9. 因果链规范

因果链当前阶段只做轻量组织结构，不承担复杂学习、检索或推理职责。

建议最小结构为：

```cpp
export class 因果链主信息类 {
public:
    std::vector<因果节点类*> 因果实例序列{};
    std::uint64_t 起点锚点主键 = 0;
    std::uint64_t 终点锚点主键 = 0;
    时间戳 起始时间 = 0;
    时间戳 结束时间 = 0;
};
```

必须固定：

1. 因果链允许跨层
2. 单跳因果不允许跨层
3. 因果链中的每一跳都必须先通过单条因果合法性校验


## 10. 抽象因果模板的后置规则

抽象因果模板是第二阶段工作，不得抢跑。

在实例因果未跑通之前，必须禁止：

- 直接建设“全域通用因果类”
- 直接把少量实例上升为模板
- 直接用模板替代当前事实

抽象因果模板应在以下条件满足后再做：

1. 实例因果边界稳定
2. 合法性校验函数已统一接入
3. 两条生成通道已跑通
4. 因果链已能组织多跳解释


## 11. 落地顺序

当前推荐次序固定为：

1. 重构 `因果实例主信息类`
2. 实现 `校验因果实例合法性(...)`
3. 实现 `生成场景因果实例(...)`
4. 实现 `生成存在因果实例(...)`
5. 实现轻量 `组织因果链(...)`
6. 最后再做 `因果模板主信息类` 与模板沉淀

推荐的最小接口组为：

```cpp
bool 校验因果实例合法性(const 因果实例主信息类& x);

std::vector<因果节点类*> 生成场景因果实例(
    场景节点类* 前场景,
    场景节点类* 后场景);

std::vector<因果节点类*> 生成存在因果实例(
    存在节点类* 前存在,
    存在节点类* 后存在);

因果链节点类* 组织因果链(
    const std::vector<因果节点类*>& 输入因果集);
```


## 12. 约束结论

后续重构中必须固定以下结论：

1. 因果体系不再以“通用因果信息”为目标名词。
2. 当前第一阶段只先做 `实例因果信息 + 轻量因果链`。
3. 单条因果必须满足 `同锚点 + 同层级 + 时间有序`。
4. 单条因果的锚点只允许为 `场景` 或 `存在`。
5. 单条因果不允许跨层；跨层解释只能通过因果链完成。
6. 场景因果与存在因果必须分通道生成，不得混写。
7. 抽象因果模板必须后置，不得先于实例因果边界落地。

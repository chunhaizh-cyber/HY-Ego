# 1270 元层核心与学习写回实现规范

## 1. 目标

本文用于固定当前 `元层核心模块.ixx` 与 `学习写回模块_v0.ixx` 的实现边界，回答以下问题：

- 元层核心当前提供的真能力是什么
- 学习写回如何把反馈、经验、适配关系写回元层
- 这条写回链是事务式的，还是补建式的
- 哪些事情属于元层与学习写回，哪些不属于

本文对应当前工程实现：

- `元层核心模块.ixx`
- `学习写回模块_v0.ixx`


## 2. 定位

当前元层与学习写回，应理解为两层：

- `元层核心`
  - 提供统一的实体、关系、事件承载
  - 提供读写、查询、版本与失效检查
- `学习写回`
  - 把任务执行反馈整理成元实体和元关系
  - 把反馈沉成经验、适配关系和学习回写边

它们不是：

- 世界树本体
- 任务调度器
- 方法树条件/结果节点生成器


## 3. 元层核心的职责

当前元层核心已经实现的最小职责包括：

- 写实体
- 读实体与查实体
- 写关系
- 查关系、按终点查关系、双向查关系
- 记录系统事件

因此元层核心的职责边界应固定为：

- 提供统一元数据承载
- 校验实体和关系的最低有效性
- 在单次 API 调用内保证序列化一致性

元层核心不负责：

- 推导业务语义真值
- 跨多个 API 调用做业务事务编排
- 决定学习结果是否应该回写


## 4. 元层并发与一致性边界

当前元层核心采用单把 `互斥_` 串行化实体、关系与事件写读。

因此必须固定：

1. 元层核心是单调用串行化，不是细粒度并发存储。
2. 单次 `写实体`、`写关系` 内部应保证最小一致性和日志完整性。
3. 上层模块不得把“元层用了单锁”误解成“跨多步写回天然事务化”。

这意味着：

- 元层 API 是原子调用边界
- 多步学习写回链不是原子事务
- 若链路中途失败，前序已成功写入的实体或关系可能保留


## 5. 学习写回的最小闭环

当前学习写回的主链应固定为：

1. 校验任务实体、方法实体和反馈实体
2. 确保任务-方法绑定关系存在
3. 确保反馈到任务、反馈到方法关系存在
4. 写回任务统计
5. 写回方法统计
6. 更新任务-方法适配关系
7. 写入经验实体
8. 写入学习回写关系
9. 记录学习写回事务事件
10. 更新任务和方法上的最近反馈摘要

这里的统计和适配关系写回，当前已经实现：

- 成功次数、失败次数、总次数
- 历史成功率
- 累计服务收益修正
- 累计风险成本修正
- 平均服务收益修正
- 平均风险成本修正
- 适配得分
- 最近反馈摘要和最近学习时间


## 6. 当前写回语义

当前写回语义应固定为：

- `补建式`
- `累计式`
- `可重入`

更具体地说：

- 若缺绑定关系，应先补建绑定关系
- 若缺反馈关系，应先补建反馈关系
- 统计项默认累计
- 最近摘要、最近反馈标识、最近学习时间允许覆盖

因此必须固定：

1. 学习写回当前不是数据库事务式提交。
2. 学习写回失败时，应返回明确错误摘要。
3. 新增写回步骤时，必须保持“已写成功部分仍然可解释、可补偿、可重试”。


## 7. 线程与锁边界

当前元层核心内部已经有单锁串行化，因此上层必须遵守：

1. 不得在持有复杂结构锁、链表锁、队列锁的同时长时间调用元层写回链。
2. 学习写回应尽量在上层释放热点锁后执行。
3. 不得把元层写回当成热点路径里的常规同步动作。

这条边界的意义是：

- 防止把元层单锁与运行时热点锁串成大临界区
- 防止学习写回把动作线程、学习线程或宿主线程拖入锁竞争


## 8. 元层与学习写回的硬约束

1. 元层核心只提供统一承载，不替业务做最终决策。
2. 单个 API 调用应保证最小一致性，但跨 API 链路不承诺事务原子性。
3. 学习写回必须按“先补建关系、再写统计、再写经验、最后记事件”的顺序推进。
4. 统计字段默认累计，摘要字段允许覆盖，但语义必须稳定。
5. 任何新增写回步骤都必须能给出失败摘要和补偿口径。
6. 元层写回不应嵌在高频热点锁内执行。


## 9. 不负责的事项

本文不负责：

- 任务树和方法树的节点结构
- 世界树特征写回
- 因果模块的反推候选提炼
- 宿主壳层的快照协议

# 0600 顶层任务树与子智能体协作治理规范

编号：0600
日期：20260721
状态：正式协作治理规范

## 1. 定位

本规范统一定义顶层窗口、子智能体、递归后代、写角色、只读例外、同角色并行、派发合同、所有权、Git 收口和跨顶层交接。`AGENTS.md`、`规范/设计执行双窗口交互规范.md`、`规范/多工作树并发与集成规范.md` 和项目技能只保留入口摘要与操作引用，不得另立相反口径。

本规范约束代理如何协作，不改变业务机器逻辑、计划依赖、任务 worktree、执行许可、集成许可或主线发布许可。使用更多智能体不能扩大正式计划范围，也不能替代现行规范、有效设计、计划、任务队列和实际接口复核。

## 2. 定义

1. **顶层任务**：由用户直接打开或创建、不是由其它智能体递归派生的 Codex 顶层窗口 / 顶层任务。
2. **子智能体**：由顶层智能体或其后代直接创建的协作智能体；子智能体继续创建的后代仍属于原顶层任务。
3. **顶层任务树**：一个顶层任务及其全部直接、递归子智能体的完整集合。
4. **写角色**：设计、执行、集成三种具有项目文件、运行资源、任务执行权或主线发布占用写权限的角色。
5. **只读身份**：只采集、核对和返回事实，不具有文件、构建、运行、Git、任务执行权或发布占用写权限的身份；只读不是第四种写角色。
6. **切片所有权**：对一个确定文件集合、结构集合、接口集合、验证集合或证据集合的唯一处理责任。

顶层任务身份由创建关系裁决，不因 worktree、分支、模型、任务名称、智能体名称或文件路径变化而改变。

## 3. 顶层任务树单写角色

1. 同一顶层任务树在同一时刻最多只能拥有一种写角色；设计、执行、集成三种写角色互斥。
2. 顶层任务取得的写角色由树内全部有写入权的子智能体继承。父设计、子执行；父执行、子设计；父设计、子集成；父集成、子设计等组合全部禁止。
3. 子智能体不得通过另建 worktree、分支、模型或名称自行取得另一写角色。
4. 顶层任务没有取得写角色或被登记为只读时，整棵任务树只能只读；任何后代不得自行升级为写角色。
5. 只读子智能体可以与当前写角色并存，但必须保持零修改、零构建、零运行、零暂存、零提交、零推送，不创建或占用写 worktree，不取得任务执行权或主线发布占用。
6. 发现同一顶层任务树已经拥有两种写角色时，整树立即停止新增写入，保留现有 Git 事实并退回设计治理追查派发根因；不得以文件无交集、最终可合并或结果正确为理由继续。

## 4. 子智能体优先使用规则

同一个顶层窗口可以并应尽可能使用子智能体提升处理速度。出现两个以上边界明确、无真实依赖，且文件、结构、接口、验证或证据所有权可以隔离的具体子任务时，顶层智能体应优先使用同角色子智能体或只读子智能体并行处理。

顶层智能体仍负责：

```text
统一写角色
-> 拆分具体子任务
-> 分配不重叠所有权
-> 核对返回事实和差异
-> 串行完成 Git 与占用收口
-> 对最终结果和完成声明负责
```

以下情况允许不创建子智能体，但必须在本轮更新、设计记录、实施记录或最终回执中具名说明原因：

```text
任务本身是不可再分的原子步骤；
子任务存在真实前后依赖；
文件、结构、接口或状态所有权无法隔离；
可用并发槽不足；
工具或外部状态只能串行操作；
协调成本高于实际工作量；
风险无法在当前计划和 worktree 边界内隔离。
```

不得为了满足“尽可能使用”而制造空任务、重复读取、无意义代理链或相互等待；不得把同一原子步骤拆给多个智能体竞争处理。

## 5. 各角色适用场景

| 顶层任务角色 | 可并行交给同角色子智能体 | 可并行交给只读子智能体 | 禁止 |
| --- | --- | --- | --- |
| 设计 | 不相交的规范、流程图、详细设计、计划、状态同步切片 | 规范冲突审查、代码事实扫描、差异和引用核对 | 子智能体执行代码、构建、集成或发布 |
| 执行 | 计划允许且所有权不相交的代码、工程、自检、实施记录或验证切片 | 只读接口、差异、日志和证据核对 | 子智能体改规范、计划、中央队列或承担集成角色 |
| 集成 | 登记范围内不相交的汇集检查、集成记录和验证切片 | 提交父链、文件范围、验证输出和主线漂移核对 | 子智能体发明接口、改计划、扩大集成范围或承担设计角色 |
| 只读 | 无 | 指定提交和问题集上的只读事实采集 | 任一后代修改、构建、运行或执行 Git 写操作 |

同角色只说明角色权限一致，不说明多个智能体可以共同写同一对象。

## 6. 派发合同与 S0

顶层智能体每次派发子智能体，至少必须给出：

```text
顶层任务身份；
继承的写角色或只读身份；
具体目标和返回条件；
允许文件、结构、接口、验证或证据切片；
明确禁止项；
是否允许修改、构建和运行；
Git 暂存、提交、推送权限；
任务执行权、worktree 与主线发布占用权限；
发现冲突、漂移、失败或新用户输入时的停止条件。
```

子智能体开始工作前必须完成最小 S0：

```text
确认自己所属的顶层任务树和角色；
确认允许与禁止范围；
确认目标路径、分支、HEAD 和 dirty state（适用时）；
确认与其它切片无所有权交集；
确认 Git 和占用权限；
返回 PASS 或具名 DRIFT。
```

未完成 S0 或 S0 为 DRIFT 时，不得开始写入。子智能体递归创建后代时，必须原样传递顶层任务身份、角色、允许范围、禁止项、Git / 占用权限和停止条件；不得在下级派发中扩大权限。

## 7. 所有权、worktree 与 Git 串行

1. 任务和 worktree 的正式所有者是顶层任务树；子智能体只取得已派发的具体切片所有权。
2. 同一文件、结构、接口、验证项目或证据结论在同一时刻只能有一个智能体负责写入或裁决。
3. 多个同角色子智能体可以在同一任务树中处理两两无交集的切片；任何交集必须先由顶层智能体重新拆分或改为串行。
4. worktree 与分支仍服从 `规范/多工作树并发与集成规范.md`；子智能体不因共享顶层任务而绕过 worktree 登记、冻结基线或计划允许范围。
5. Git index、暂存、提交、推送、任务执行权迁移和主线发布占用迁移必须由顶层智能体或唯一指定的同角色智能体串行完成；其它子智能体在收口期间停止写入。
6. 顶层智能体必须复核子智能体的实际差异和验证证据，不能把聊天回执直接当成仓库事实、完成事实或发布事实。
7. 无关 WIP 继续属于原所有者；当前任务树及其子智能体不得暂存、清理、stash、覆盖或带入提交。

## 8. 停止、升级与跨顶层交接

1. 子智能体命中范围漂移、所有权重叠、角色冲突、计划外文件、验证失败或用户新裁决时，必须停止受影响切片并返回顶层智能体；不得自行扩权。
2. 提高模型、推理等级或创建更深后代不改变任务树身份、写角色或允许范围。
3. 跨设计、执行、集成的交接默认派发给彼此独立的顶层任务，不得把当前任务树的子智能体登记为另一写角色。
4. 只有用户明确启用同一物理窗口串行切换角色时，才允许整棵任务树整体切换。切换前全部后代必须停止并释放旧角色的文件、worktree、运行资源、任务执行权和发布占用，旧角色记录、验证、提交和推送必须完成；中央路由发布新角色后，顶层智能体和需要继续工作的后代重新读取正式状态。
5. 任一后代仍持有旧角色或旧占用时禁止切换。切换后需要继续使用子智能体时，必须按新角色重新派发并完成 S0。
6. 任务消息只负责唤醒；跨顶层接管仍以已提交推送的计划、队列、登记表、交互记录、实施记录和 Git 事实为准。

## 9. 验证与完成边界

规则或文档治理至少验证：

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

子智能体返回只证明其受派切片的事实或产物已返回，不证明顶层任务完成。顶层智能体必须核对：

```text
所有子智能体均已结束或明确保持只读；
没有两种写角色并存；
所有权交集为零或已串行消解；
Git 收口只包含本轮目标文件；
无关 WIP 保持未暂存；
正式验证通过；
提交与推送事实和完成声明边界一致。
```

本规范的验证通过只证明协作治理文本和登记一致，不证明业务代码实现、构建、运行、生产接通或能力验收完成。

## 10. 禁止事项

```text
禁止父智能体保持一种写角色、子智能体取得另一写角色；
禁止只读顶层任务派生有写入权的后代；
禁止用 worktree、分支、模型、名称或递归层级规避顶层任务树身份；
禁止多个智能体共同写同一切片；
禁止未完成 S0 就开始写入；
禁止并发执行 Git 暂存、提交、推送或占用迁移；
禁止为了展示并发而制造空任务或伪造无依赖；
禁止把子智能体回执直接升级为正式仓库事实或完成事实。
```

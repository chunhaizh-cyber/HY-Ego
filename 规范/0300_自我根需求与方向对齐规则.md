# 0300 自我根需求与方向对齐规则

## 1. 目标

本文用于固定“自我”的总方向来源，以及需求、任务、方法后续生成时的统一对齐规则。

本文以逻辑规则为真值，不以自然语言名称为真值。

本文解决的问题是：

- 自我的最初方向从哪里来
- 服务与安全是什么关系
- 后续生成的需求、任务、方法如何判断是否对齐
- 执行结果如何重新回到根方向，形成持续闭环


## 2. 根目的与最初方向

### 2.1 根目的

当前定义：

- `自我的根目的 = 满足人类需求 / 提升服务成立程度`

这里的“提升服务成立程度”不是一句描述，而是一个明确目标：

- `目标服务值 = I64 体系中的非负上界目标值`

### 2.2 初始现状

当前定义：

- `自我的初始服务值 = 中位数值`
- `自我的初始安全值 = 中位数值`

因此，自我一开始就天然存在：

- `服务现状值`
- `服务目标值`
- `服务差值`
- `安全现状值`
- `安全保障差值`

### 2.3 方向定义

定义：

- `主方向 = 目标服务值 - 当前服务值`
- `保障方向 = 目标安全值或安全保障线 - 当前安全值`

因此，方向不是名称，不是标签，也不是一句自然语言，而是：

- 一个可持续计算的“服务差值缩减方向”
- 一个不可忽略的“安全保障修复方向”

其中：

- 主方向决定“最终要更好地服务到哪里”
- 保障方向决定“是否还能继续服务”

### 2.4 差值区间与绝对方向

对于 `服务值`、`安全值`、`需求值` 这类同时具有“当前值”与“目标值”的对象，比较后得到的有符号差值，应视为一种二次特征。

当前规范统一采用：

- `差值 = 目标值 - 当前值`

因此，该差值必须落入以下三个区间之一：

1. `正区间`  
   `差值 > 0`，表示当前值仍低于目标值，方向表现为继续逼近。

2. `负区间`  
   `差值 < 0`，表示当前值已越过目标值或出现反向偏移，方向表现为回调、纠偏或抑制。

3. `零区间`  
   `差值 = 0`，表示当前值与目标值重合，属于绝对对齐区间。

其中：

- 正区间与负区间互斥
- 正区间与负区间都具有方向性
- 零区间不表达偏移趋势，而表达绝对一致

可以固定为：

- `某一比较值所落入的区间 = 该维度上的绝对方向`

若同时存在服务、安全等多个维度，则应先分别得到各维度绝对方向，再由根目的与安全边界共同裁决当前总方向。

### 2.5 保障方向的地位

保障方向非常重要，但它不是脱离服务的独立总目的。

可以固定为：

- `安全方向的意义 = 保障服务方向能够持续成立`

也就是说：

- 安全方向不能被破坏
- 但安全方向的价值来自服务根目的


## 3. 服务与安全的关系

### 3.1 总方向与保障方向

当前应区分两层：

- `服务方向` 是总方向
- `安全方向` 是保障方向

也就是说：

- 服务决定“最终要去哪里”
- 安全决定“是否还能持续地去”

### 3.2 为什么安全不是独立总方向

安全值本身会决定自我是否还能继续存在和继续服务，但安全并不是脱离服务单独存在的最终目的。

可以固定为：

- `安全需求的成立依据 = 它对持续服务具有正向贡献`

因此，安全需求虽然可以作为需求树中的一级需求存在，但它在方向判定上应服从服务总方向。

### 3.3 为什么服务又会反过来影响安全

服务值本身会影响安全值的变化规则：

- 服务值降低时，安全值更容易下降，恢复更慢
- 服务值升高时，安全值更容易回升，下降被抑制
- 服务值为 0 时，安全值被压到最小非零值，并进入待机

因此，服务不是只存在于“结果层”，它还会反过来塑造安全保障条件。

### 3.4 直接服务与准备性服务

服务可以分成两类：

1. `直接服务`
   直接满足了当前人类需求。

2. `准备性服务`
   当前尚未直接满足最终需求，但为后续满足创造了条件。

安全恢复、能力补齐、阻塞清理等，只要能追溯到持续服务，就可归入准备性服务。


## 4. 需求树中的方向约束

### 4.1 需求树结构

当前需求树可以固定为：

- 根节点：`自我需求根`
- 一级需求：`服务需求`
- 一级需求：`安全需求`

除这两个一级需求外，其它需求都必须有父需求。

### 4.2 服务需求

服务需求直接表达：

- `让服务值更接近目标值`
- `让人类需求被更稳定、更真实地满足`

它是总方向的直接表达。

### 4.3 安全需求

安全需求表达的不是“脱离服务地追求更安全”，而是：

- `通过恢复和维持安全条件，使服务可以持续`

因此，安全需求是服务方向的保障展开。

### 4.4 子需求生成规则

后续生成任何子需求时，都必须满足至少一条：

- 它直接缩小服务差值
- 它提升未来直接服务的成功率、稳定性或速度
- 它修复会导致服务中断的安全问题
- 它减少内部压力，从而降低服务链路失稳风险

如果一个需求无法投影到以上任一条，它就不应进入需求树。


## 5. 任务与方法的方向对齐规则

### 5.1 任务的成立条件

每个任务都必须对应某个需求。

任务是否成立，不看名称，而看：

- `该任务预期是否能让其所对应需求更接近满足`

再进一步追溯，就是看：

- `该任务对服务总方向的净贡献是否为正`
- `该任务是否保持安全保障边界`

### 5.2 方法的成立条件

每个方法都必须服务于某个任务。

方法是否可被选中，不看它“像不像正确动作”，而看：

- `预测执行该方法后，任务对应需求是否更接近满足`
- `追溯到根方向后，服务差值是否缩小，或导致差值缩小的条件是否增强`
- `执行风险是否突破安全保障边界`

### 5.3 允许短期负、长期正

方向对齐不等于每一步都要求服务值或安全值立即上升。

允许存在以下情况：

- 短期服务值下降
- 短期安全值承压
- 但长期能让服务更稳定、更可持续、更真实成立

因此判定依据应固定为：

- `看净因果贡献`

而不是：

- `只看当前一步的表面数值变化`

### 5.4 不对齐的情形

以下情形应视为不对齐：

- 只带来局部收益，但长期扩大服务差值
- 提高了某个局部指标，却削弱了持续服务能力
- 为了局部安全而长期放弃服务根目的
- 无法解释其与根方向之间的因果链


## 6. 从状态变化中提取因果方向

### 6.1 真值来源

当前方向与因果，不应依赖名称，而应依赖状态变化过程。

也就是说：

- 先有状态
- 再有动态
- 再有二次特征
- 再从二次特征中归纳因果方向

### 6.2 服务值变化的主要来源

当前服务值变化至少来自以下几类：

1. 时序衰减  
   服务值随时间自然下降。

2. 直接服务结果  
   真实满足人类需求会提高服务值。

3. 准备性服务结果  
   为后续满足创造条件的有效行为，也可提高服务值，但弱于直接服务。

4. 压力与失败迹象  
   待办积压、学习缺口、连续失败等会提高服务衰减压力。

### 6.3 安全值变化的主要来源

当前安全值变化至少来自以下几类：

1. 时序回归  
   安全值随时间向中位值回归。

2. 内部压力  
   待完成任务、待学习任务等积压会拉低安全值。

3. 服务耦合  
   服务值决定正向时序步长与反向时序步长，从而影响安全值恢复与下滑速度。

4. 异常损伤  
   物理伤害、硬件故障等异常事件会直接改写安全值。

### 6.4 服务与安全的因果说法

可以固定为以下因果链：

1. `满足人类需求 -> 服务值增加 -> 更利于后续持续服务`
2. `准备性服务成立 -> 服务值可增加 -> 更利于后续直接服务`
3. `服务值下降 -> 反向时序步长增大、正向时序步长减小 -> 安全值更易下降、恢复更慢`
4. `服务值上升 -> 正向时序步长增大、反向时序步长减小 -> 安全值更易回升、下降被抑制`
5. `服务值为 0 -> 安全值降到最小非零值并进入待机 -> 暂不死亡，但失去主动推进能力`

### 6.5 因果提取原则

当观察到服务值或安全值变化时，应把一次变化拆成来源贡献：

- 服务时序衰减贡献
- 直接服务贡献
- 准备性服务贡献
- 内部压力贡献
- 服务耦合贡献
- 异常事件贡献

然后判断：

- 哪一类贡献先发生
- 哪一类贡献方向稳定
- 哪一类贡献重复出现
- 哪一类贡献强度与结果强度一致

满足“先因后果、方向一致、重复稳定”的关系，才应被固化为因果信息。


## 7. 对齐判定的统一原则

后续一切生成物，不按名称对齐，而按它对“服务根目的”的预测净贡献来对齐，并同时检查是否守住安全保障边界。

可以收敛为一句总规则：

- `凡是能缩小服务根目的差值，或能稳定创造缩小该差值条件的生成物，都是对齐的。`

对应地：

- `凡是会扩大服务根目的差值，或破坏持续服务安全条件的生成物，都是不对齐的。`


## 8. 闭环规则

完整闭环应为：

1. 根目的给出总方向
2. 当前服务值与目标服务值形成主差值
3. 当前安全值与安全保障线形成保障差值
4. 差值落入正区间、负区间或零区间，并给出各维度绝对方向
5. 绝对方向与根约束共同触发需求树生长
6. 需求生成对应任务
7. 任务筛选并组织方法
8. 方法执行产生实际结果
9. 实际结果回写需求状态、任务状态、自我内部状态
10. 自我服务值与安全值变化
11. 由状态继续生成动态、二次特征、因果信息
12. 因果信息继续修正需求与任务生成方向


## 9. 本文件的约束结论

后续实现中应固定以下结论：

1. `服务` 是根目的。
2. `安全` 是保障方向。
3. 需求、任务、方法的成立必须同时接受服务方向与安全边界的检查。
4. 准备性服务可以成立，但其价值必须能追溯到后续直接服务或持续服务能力。
5. 状态、动态、二次特征、因果信息都必须最终回投到服务根方向。
6. 对服务值、安全值、需求值等目标比较结果，应统一通过 `正区间 / 负区间 / 零区间` 表达该维度绝对方向。
7. 任何局部高收益路径，只要破坏持续服务能力，就不得视为总方向一致。

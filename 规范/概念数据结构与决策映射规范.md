# 概念数据结构与决策映射规范

## 1. 目标

本文用于固定以下三件事：

- `概念候选` 最少应保存哪些字段
- `稳定概念` 最少应保存哪些字段
- `稳定概念` 如何映射到需求、任务、方法的决策与评分

本文不要求先新增一套独立的大结构。

本文优先原则是：

- `复用现有节点与主信息字段`
- `概念身份靠签名与关系，不靠名称`
- `名称可空，不能把自然语言词面当真值`

本文与以下规则配套：

- [自我根需求与方向对齐规则.md](D:\海鱼\规范\自我根需求与方向对齐规则.md)
- [概念自动生成判据.md](D:\海鱼\规范\概念自动生成判据.md)


## 2. 现有结构上的推荐落点

按当前工程实现，概念相关信息推荐落在三层：

1. `二次特征节点 / 二次特征主信息类`  
   用于表达：观测结果、结构化关系、候选线索、概念模板命中。

2. `因果节点 / 因果主信息类`  
   用于表达：因果方向、观察次数、稳定度、时间顺序与过程签名。

3. `概念树节点`  
   用于表达：已经稳定下来的概念锚点，以及它与其它概念/特征/因果之间的关系。

因此建议的职责分工是：

- `二次特征` 负责“看到什么关系”
- `因果` 负责“确认关系是否稳定成立”
- `概念树` 负责“把已稳定关系挂成可复用概念节点”


## 3. 概念身份规则

### 3.1 不能用名称当身份

概念身份不能主要依赖：

- 名称字符串
- 自然语言标签
- 人工词面命名

因为在早期阶段：

- 名称可以为空
- 名称可以变更
- 同一概念可能还没有自然语言形式

### 3.2 概念身份的最小真值

概念身份应主要由以下组合决定：

- `概念域`
- `作用对象签名`
- `被作用对象签名`
- `作用方向`
- `作用维度`
- `适用条件签名`
- `度量签名 / 因果签名`

可收敛为：

- `概念签名 = 结构化关系签名`

名称只是附属解释字段，不是概念主键。


## 4. 概念候选的最少字段

概念候选是“值得继续观察的关系假设”，推荐至少保存以下字段。

### 4.1 身份字段

1. `候选签名`  
   用于唯一标识当前候选关系。

2. `概念域`  
   例如：二次特征域、因果域。

3. `作用对象`  
   例如：`服务值`。

4. `被作用对象`  
   例如：`安全值`。

5. `作用维度`  
   例如：恢复速度、衰减速度、待机风险、稳定性。

6. `作用方向`  
   例如：正向、负向、条件性正向、未知。

### 4.2 证据字段

7. `证据次数`  
   当前已观察到多少次同类证据。

8. `首次观察时间`
9. `最近观察时间`
10. `证据链指针集合`  
    至少能回指到相关状态、动态、二次特征或因果节点。

11. `一致性Q`  
    当前证据方向是否稳定一致。

12. `可复现度Q`  
    在相似条件下是否可重复出现。

13. `干扰程度Q`  
    当前样本受到其它主因干扰的程度。

### 4.3 根需求投影字段

14. `根需求投影结果`  
    正向、负向、中性、未知。

15. `净贡献Q`  
    当前候选对根需求的净贡献估计。

16. `决策级别`  
    不参与、弱参与、中参与、强参与。

### 4.4 适用范围字段

17. `适用条件签名`  
    该关系在哪类上下文中成立。

18. `时间窗`
19. `样本范围说明`

### 4.5 与现有字段的推荐映射

在现有结构中，概念候选优先映射为：

- `二次特征主信息类::度量签名_链键` -> 候选的度量定义
- `二次特征主信息类::主体 / 客体 / 左对象 / 右对象` -> 作用对象与被作用对象
- `二次特征主信息类::标量值` -> 当前一次证据的量化结果
- `二次特征主信息类::观察` -> 观察次数、最近时间、熟悉度、不确定度
- `二次特征主信息类::时间窗` -> 适用时间窗
- `二次特征主信息类::依赖度量签名表` -> 复合关系依赖

如果当前还缺 `净贡献Q / 一致性Q / 可复现度Q / 干扰程度Q`，应优先作为补充字段加到承载候选的主信息上，而不是另起独立平行结构。


## 5. 稳定概念的最少字段

稳定概念是在候选基础上升级得到的，因此应至少保留候选全部核心字段，并新增以下字段。

### 5.1 稳定身份字段

1. `稳定概念签名`
2. `稳定级别`  
   弱稳定、稳定、强稳定。

3. `概念锚点节点`  
   指向概念树中的概念节点。

4. `可选名称`  
   仅用于解释，不作为身份主键。

### 5.2 因果确认字段

5. `因果签名`
6. `观察总次数`
7. `方向一致率`
8. `先因后果成立率`
9. `可复现度Q`
10. `稳定度Q`
11. `置信度Q`

### 5.3 决策字段

12. `根需求投影结果`
13. `净贡献Q`
14. `决策参与级别`
15. `默认作用方式`  
    加分、减分、调权、硬过滤、生成子需求。

16. `作用边界`  
    说明它影响的是恢复、衰减、待机、风险还是其它维度。

17. `失效条件签名`  
    在何种条件下不应继续使用该概念。

18. `降级计数 / 反例计数`

### 5.4 与现有字段的推荐映射

在现有结构中，稳定概念优先映射为：

- `概念树节点` -> 稳定概念锚点
- `因果主信息类::条件签名 / 结果签名 / 过程签名` -> 因果身份与过程身份
- `因果主信息类::观察次数` -> 证据总量
- `因果主信息类::稳定度Q` -> 稳定性
- `因果主信息类::首次发生时间 / 最近发生时间` -> 时间证据
- `二次特征主信息类::概念模板` -> 候选/实例与稳定概念锚点之间的连接

如果还缺 `净贡献Q / 决策参与级别 / 失效条件签名 / 反例计数`，建议补到稳定概念承载层，而不是放在自然语言层。


## 6. 概念候选与稳定概念的最少区别

为避免“线索一出现就当真理”，两者必须严格区分：

1. 候选可以没有概念树锚点  
   只要已有结构化签名和证据链即可。

2. 稳定概念必须有概念树锚点  
   否则不能成为可复用判断依据。

3. 候选可以只参与观察，不参与决策。
4. 稳定概念才允许参与决策。
5. 候选的根需求投影可以是未知。
6. 稳定概念的根需求投影必须明确。


## 7. 概念到需求的映射规则

稳定概念进入需求层时，不是直接变成一句话，而是变成“需求权重修正依据”。

### 7.1 映射目标

概念对需求主要影响：

- 是否生成某类需求
- 某类需求的权重增减
- 某类需求是否提前

### 7.2 推荐映射字段

需求层最少需要读取：

- `根需求投影结果`
- `净贡献Q`
- `决策参与级别`
- `适用条件签名`
- `作用边界`

### 7.3 推荐映射规则

1. 若概念为正向稳定概念  
   则提高与其一致的需求权重。

2. 若概念为负向稳定概念  
   则生成修复型或规避型需求。

3. 若概念只在局部条件成立  
   则只在命中适用条件时参与需求修正。

4. 若概念处于弱稳定状态  
   则只影响排序，不直接生成新需求。

### 7.4 服务概念的例子

若稳定概念成立：

- `服务值对安全值具有正向单调影响`

则可映射为：

- 提高“恢复服务条件”相关需求权重
- 当服务值偏低时，提前服务相关需求
- 当服务值极低时，生成服务恢复型子需求


## 8. 概念到任务的映射规则

稳定概念进入任务层时，主要影响：

- 任务是否生成
- 任务优先级
- 任务是否应被暂缓

### 8.1 推荐映射结果

任务层可把概念映射为：

- `基准优先级修正`
- `局部优先级偏移`
- `是否允许进入当前轮执行`

### 8.2 推荐映射规则

1. 正向概念命中当前条件  
   给一致任务加分。

2. 负向概念命中当前条件  
   给高风险任务减分，必要时暂缓。

3. 条件性概念  
   仅在适用条件命中时生效。

4. 弱参与概念  
   只能影响 `局部优先级偏移`。

5. 强参与概念  
   才允许直接触发“生成任务”或“禁止任务”。

### 8.3 与现有任务结构的对接

按当前实现，建议直接对接：

- `任务主信息::基准优先级`
- `任务主信息::局部优先级偏移`
- `任务执行器` 中的候选排序逻辑

也就是说，概念层不应自己调度任务，而应输出：

- `任务层可消费的加减分与过滤结论`


## 9. 概念到方法的映射规则

稳定概念进入方法层时，主要影响：

- 候选方法评分
- 方法规避过滤
- 条件缺口分析后的子任务方向

### 9.1 方法层最少读取字段

- `净贡献Q`
- `决策参与级别`
- `作用边界`
- `适用条件签名`
- `失效条件签名`

### 9.2 推荐映射规则

1. 若某方法预测结果与正向概念一致  
   给该方法加分。

2. 若某方法预测结果与负向概念一致  
   给该方法减分。

3. 若某方法会触发强负向概念  
   可进入规避过滤。

4. 若概念表明“缺少某条件会持续损害根方向”  
   则把该条件缺口转成优先子任务。

### 9.3 与现有方法评分的对接

按当前实现，建议把概念映射为：

- `评分Q` 加减项
- `avoid_filter_` 的判定依据
- `条件缺口 -> 子需求 -> 子任务` 的优先级修正项

这样可以直接接到现有：

- `评分_方法结果满足需求(...)`
- `结构_筹办方法候选::评分Q`
- `追加候选(...)`
- `avoid_filter_`

而不需要重写整条筹办主链。


## 10. 决策映射的统一格式

为了避免每个模块自己解释概念，推荐把概念输出统一规约为四类结果：

1. `加分`  
   对齐根方向，提升候选优先级。

2. `减分`  
   偏离根方向，但尚未达到禁止级。

3. `调权`  
   影响需求或任务权重，而非直接过滤。

4. `硬过滤`  
   明确与根方向冲突，当前上下文下禁止。

必要时可以加第五类：

5. `生成缺口任务`  
   当前问题不是执行某方法，而是先补齐关键条件。


## 11. “服务越高越好”在数据结构上的落点示例

如果该概念已经稳定成立，建议落点如下：

### 11.1 候选阶段

- 作用对象：`服务值`
- 被作用对象：`安全值`
- 作用维度：`恢复速度 / 衰减速度 / 待机风险`
- 方向：`正向单调`
- 证据：服务变化与安全变化相关的二次特征节点

### 11.2 稳定阶段

- 因果签名：`服务 -> 安全`
- 根需求投影：`正向`
- 净贡献Q：正值
- 决策参与级别：先弱后中，再视证据升级为强
- 概念锚点：概念树中的一个无名或已命名概念节点

### 11.3 决策映射

- 需求层：提高服务恢复相关需求权重
- 任务层：提高服务恢复任务的局部优先级
- 方法层：给恢复服务条件的方法加分
- 极端低服务时：对高风险方法减分或过滤


## 12. 本文件的约束结论

后续实现中应固定以下结论：

1. 概念身份以结构化签名为主，不以名称为主。
2. 概念候选与稳定概念必须分层保存，不能混用。
3. 优先复用 `二次特征主信息类`、`因果主信息类` 与 `概念树节点` 承载概念信息。
4. 若字段不足，应优先在现有承载层补字段，而不是新造一套平行概念系统。
5. 稳定概念进入需求、任务、方法层时，应统一输出为 `加分 / 减分 / 调权 / 硬过滤 / 生成缺口任务` 之一。
6. `服务越高越好` 这类概念在结构上必须先有证据、再有因果、再有根需求投影，最后才能进入决策。
7. 名称可以为空，但概念签名、证据链、根需求投影和决策级别不能为空。

# 3100 根规范 需求

编号：3100
日期：20260724
状态：正式根规范

直接上级规范：0050。

直接依赖规范：无。

相关边界规范：1190、3200、3300、4320、5100—5160。

## 1. 根定位

需求是某一主体要求某一目标宿主的一个确定目标特征达到目标状态的长期目标契约。需求不是过程记录、任务、方法、日志或显示摘要。

一个需求只能有一个权威目标。复合目标、替代路径和共同成立条件必须通过需求关系结构展开，不得把多目标包压入一个需求。

## 2. 本项目承载

需求在海中鱼巣中由以下结构共同承载：

```text
需求节点及稳定句柄
+ 需求主体关系
+ 目标宿主关系
+ 目标场景关系
+ 目标特征关系
+ 当前状态材料关系
+ 目标抽象状态关系
+ 父需求和逻辑路径关系
+ 路径当前性、权重、授权和结算关系
+ 来源证据、满足动态和结算证据关系
```

节点和关系承载身份与语义，状态、特征和特征值承载目标材料；日志、文本、控制面板和临时结构不得成为第二套需求事实。

## 3. 目标与满足原则

需求目标必须能够回答主体、目标宿主、目标特征、当前状态材料、目标状态和满足关系。目标状态是权威目标；I64 等原始值只能作为状态或特征值材料。

需求领域不得自行比较原始值。当前是否满足必须通过特征服务、状态服务或正式判断收束合同，读取当前事实与目标状态后实时裁决。

满足是带来源、时间、版本和证据的当前裁决，不是需求节点上的永久布尔值。需求曾经满足、活动承接已截止或任务已完成，均不能证明当前世界中该目标仍然满足。

## 4. 需求树与逻辑路径原则

父子拓扑只说明展开关系，不能单独裁决管理需求、执行需求、AND 组、OR 组、方法路径或因果支撑角色。角色必须由目标语义、路径关系、承接方式、活动状态和相应特征集合共同裁决。

历史子需求可以保留，但活动子需求、活动路径和当前角色必须通过当前性关系与版本读取，不得按子节点数量推断。

替代路径、共同成立成员、父权重继承、组内分配、当前性、舍入、预算冻结和同一结算事实的领取边界由 5110 唯一规定；因果目标投影和结算归因预算由 4320 规定，需求正式结算记录、证据、唯一结论和幂等冲突由 5160 唯一规定。本根规范只固定父权重不得在路径展开中丢失、逻辑路径不得由拓扑数量推断且历史路径不得冒充当前活动路径。

## 5. 生成与写入权

自检、身体反馈、观察、任务缺口、方法学习、外部输入和临时因果视图只能生成强类型需求候选或更新输入草案。

正式需求必须由需求领域统一提交入口完成目标完整性校验、父域与路径裁决、去重、合并、创建或复用。线程、任务管理器、方法执行体、候选生成器和显示层不得直接写需求结构。

入树拒绝必须形成结构化拒绝回执并进入具名闭环；内部逻辑错误不得伪装成普通需求。

## 6. 任务、方法和结算边界

需求定义“要达到什么”，任务定义“由谁在何种授权下承接”，方法定义“在什么条件下如何产生什么结果”。三者身份不得合并。

需求只能向任务提供一个确定目标和当前授权来源；目标语义等价的多个需求可以分别授权同一单目标任务，但每项需求身份、路径、权重、预算和当前性必须独立保留。任务目标传递、方法多结果归类及附带结果边界由 5120 唯一规定。

安全需求的因果层级只能由其来源因果信息相对于具体安全结果的当前直接因果路径动态投影，不得由需求名称、预置类别或持久层号成立。需求权重只提供路径选择、一般调度和结算预算来源，不直接等于实际结算，也不等于安全任务执行优先级权限；路径预算算法由 5110、4320 唯一规定，安全权限由 6150 唯一规定，正式结算记录由 5160 唯一规定。

## 7. 生命周期与当前性

需求创建、父子关系、路径激活、权重、活动承接、满足裁决和结算都必须有来源、版本与当前性。更新必须保留可追溯历史，不得覆盖旧事实后丢失来源。

目标语义不完整、父关系非法、来源证据损坏或状态机不允许的空值属于入口拒绝或内部逻辑错误；不得先写半结构再补齐。

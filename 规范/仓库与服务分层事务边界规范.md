# 仓库与服务分层事务边界规范

编号：ARCH-LAYER-001

日期：2026-07-13

状态：正式规范 / JY-316 架构纠偏口径 / JY-371 已完成用途观察结构事实复核 / JY-376 只登记纯值 #287 / JY-382 增补可逆确认、许可内结构查询和撤销失败域隔离

## 1. 目的

本规范明确四仓库、结构事务协调层、服务专用数据操作层、领域服务和跨服务组合器之间的权力边界。目标是让仓库只负责通用结构一致性，让领域服务只负责业务语义，让原始结构事务令牌不再穿透公开业务入口。

本规范不删除当前兼容入口，不声明现有服务已经完成分层，也不恢复暂停中的 `#214`。

## 2. 分层模型

```text
外部调用方 / 线程 / 消息
-> 应用或领域组合器
-> 领域服务公开业务入口
-> 服务专用数据操作层
-> 结构写入执行器 / 不透明写入会话
-> 主信息、节点、关系、索引仓库
```

结构事务协调层只负责许可签发、许可释放、运行期域 / 纪元和跨仓库事务上下文。仓库负责验证许可、管理本仓库锁和执行具体结构读写。领域服务不得签发许可、释放许可、操作锁或比较原始令牌字段。

## 3. 四仓库权威边界

| 仓库 | 权威内容 | 不变量 | 不承载内容 |
| --- | --- | --- | --- |
| 主信息仓库 | 主信息身份、生命周期版本、拓扑锚点和值容器 | 句柄版本匹配；编号不复用；已发布节点引用的主信息必须有效 | 需求满足、任务完成、方法可执行、关系拓扑 |
| 节点仓库 | 节点身份、节点类型、主信息锚点、创建序号和生命周期版本 | 节点发布前主信息有效；节点句柄由仓库 / 编号 / 版本共同确定 | 树图边、业务授权、显示分类 |
| 关系仓库 | 当前关系记录、类型、源 / 目标端点、顺序、状态和版本 | 端点有效；关系版本匹配；普通读取只返回有效当前关系 | 业务语义裁决、稳定因果结论、显示投影 |
| 索引仓库 | 索引记录自身和主键到节点候选映射 | 主键命名空间有效；候选句柄可复核；删除 / 重挂后无旧候选残项 | 节点存在事实、关系事实、业务结论 |

关系仓库内部二级索引同样只保存关系编号候选。关系表继续是关系权威，候选必须复核关系句柄、版本、状态、关系类型和有效端点。

## 4. 统一生命周期合同

### 4.1 创建与未发布候选

多仓库创建必须按以下语义推进：

```text
入口业务准入
-> 取得一次不透明写入会话
-> 创建主信息未发布候选
-> 创建节点未发布候选
-> 创建非发布关系 / 索引
-> 创建领域唯一最后发布结构
-> 读回完整结构
-> 可逆确认候选与全部参与者
-> 进入无失败最终发布区
-> 完成发布
-> 释放会话并返回业务结果
```

未发布候选必须不可默认构造、不可复制、移动后源对象失效，并绑定创建仓库、会话身份和精确句柄。第一轮不新增主信息 / 节点仓库记录的“未发布”状态：候选记录创建后已经是结构有效记录，但其句柄只能封装在独占写入会话的数据操作层内，且在领域最后发布结构形成前不得进入公开索引、日志、消息、显示 DTO 或领域服务返回值。候选确认只把能力推进到“已确认待发布”，不得清除原令牌、完整句柄、精确版本或反向撤销材料；只有执行器进入无失败最终发布区并完成发布后，才永久关闭本次候选撤销能力。若独占会话和能力封闭不能证明中途结构对外不可见，必须停止并另立仓库级未发布状态专项，不得在函数内部补兜底。

### 4.2 失败撤销

最终发布前失败必须按本次精确能力和完整句柄逆序撤销：

```text
服务内部参与者
-> 已发布关系变更
-> 索引 / 新关系
-> 节点候选（未确认或已确认待发布）
-> 主信息候选
```

不得扫描整仓库猜测残留，不得使用普通删除冒充身份候选撤销，不得让候选析构隐藏写入。第一轮不新增关系 / 索引候选类型；独占写入会话必须记录本次创建的精确关系句柄、每次索引绑定的“主键 + 预期节点完整句柄”精确对和每个已发布关系变更的完整写前 / 写后能力，并在最终发布前失败时逆序物理撤销。所有逆操作都必须尝试，不能因前一项失败短路。完成证明比较可读权威记录、有效计数、关系、索引、侧表和容量占用；允许编号与创建序号高水位前进，不要求回拨或复用已烧掉句柄。

候选隔离依赖四仓库和服务内部参与者接入同一结构事务可见性域，并由执行器在整个会话期间持有一次独占结构许可：其它线程的共享读取必须等待许可释放，同线程无令牌入口因重入门禁不能旁路，当前会话读回只能使用会话内部令牌。若任一仓库或侧表存在不经过结构事务许可的公开读写旁路，或执行器不能把许可持有到完成发布 / 完整撤销结束，则能力封闭不足，必须在写前拒绝。撤销失败时必须在许可析构前把该事务域登记为“撤销失败隔离”：隔离后新请求必须在活动表登记前拒绝；隔离前已登记但尚未授予的等待请求，取得结构锁后必须在同一活动锁下二次复核隔离标志，移除自身记录、释放结构锁并返回无效许可。不得普通返回后释放许可让任何等待者观察半结构。

### 4.3 确认与最后发布

确认只证明本次候选与参与者已经完成可逆复核，不单独生成业务事实，也不得关闭撤销能力。最后发布结构由领域详细设计明确，可以是节点身份确认、唯一关系或专用发布记录；仓库不得自行猜测业务发布点。

执行器只有在会话和全部参与者均处于“已确认待发布”后才能进入无失败最终发布区。该区不得分配、调用外部回调、写日志、执行故障注入、做可能抛出的结构复核或返回失败；只允许会话封口、参与者 `完成发布() noexcept` 和结构独占许可最后释放。许可释放是普通读者观察四仓库与服务内部侧表的唯一统一可见点。最后发布后读回不一致属于逻辑错误，不允许再用候选撤销已发布事实，也不得静默返回失败继续后续流程。

### 4.4 普通删除

普通删除只处理已发布身份，继续要求独占许可、完整当前句柄、明确所有者和影响面。删除前必须冻结索引、关系、节点和主信息精确写集；删除顺序不得留下可读悬空结构。未发布撤销与普通删除是两个不同合同。当前关系仓库 `删除关系(..., 令牌)` 和索引仓库 `删除主键(..., 令牌)` 仍接受共享路径令牌，只能作为兼容入口保留；新统一合同必须另提供独占严格入口，旧入口在 `#271` 前不得静默改义或删除。

### 4.5 写入许可强度

新的不透明写入会话只允许由独占结构许可构造。共享许可只用于只读复核，不得构造写入会话或执行主信息、节点、关系、索引写入。四仓库现存“共享路径可写”ABI 第一轮仅作内部兼容目标；新数据操作层即使调用这些 ABI，也必须处于外层独占写入会话中。这样保证中途候选和即时关系 / 索引不会被遵守协调器的并发读者观察到。

### 4.6 幂等与版本

幂等必须由显式幂等键和完整请求语义共同裁决。相同键但载荷不同必须拒绝或追根因，不能返回旧成功。节点、主信息和关系版本只表达生命周期与身份当前性，不承载普通业务值变化。

### 4.7 许可内结构查询

服务专用数据操作需要在第一笔候选写入前完成物理主键探测或结构计量时，只能使用同一不透明写入会话提供的通用值式查询。会话查询必须显式区分空闲、已绑定完整节点、入口拒绝和内部不一致；不得用一个空 `optional` 混合多种语义。指定节点类型与索引绑定的投影必须由仓库值式副本组成，会话在仍持独占许可时逐完整句柄复核正反一致；每次只持一个仓库锁，不跨仓库同时持锁。

查询只允许当前创建线程、写入中、未决定、无首次失败且第一笔候选写入之前。查询不得登记写集、取得第二许可、许可外扫描、返回仓库 / 令牌 / 许可 / 可变引用或解释 O1、FNV、关系角色和业务容量。悬空索引、正反不对称、错类型或版本矛盾属于内部不一致，必须登记会话首次失败并禁止后继写入。

### 4.8 服务内部材料参与结构事务

特征值原始材料等服务内部材料不是第五个核心仓库，但当一次业务写入同时改变四仓库结构和服务内部材料时，必须加入同一独占结构事务的提交与撤销边界。JY-382 后统一合同为：

```text
数据操作层形成类型化服务内部材料
-> 结构写入执行器持有一次独占许可
-> 不透明结构写入会话形成并读回四仓库写集
-> 服务内部事务参与者准备发布并保持内部写锁
-> 结构写入会话完成可逆确认
-> 参与者确认待发布并继续持有内部写锁与撤销材料
-> 进入无失败最终发布区
-> 会话封口、参与者完成发布 noexcept 并释放内部写锁
-> 释放独占结构许可
```

事务参与者只能接收值式材料和执行器构造的提交准备只读视图，只能由执行器调用准备、确认待发布、完成发布和撤销生命周期；不得接收可写会话、原始结构事务令牌、许可或仓库引用，不得签发许可，不得解释需求、任务、方法等业务语义。`完成发布()` 必须为 `noexcept`，只切阶段、清撤销材料和解锁，不得做任何可失败工作。执行器不得反向依赖特征值或其它领域类型。

提交准备只读视图是通用结构能力，不是领域 DTO。它只能在业务回调已经完整读回写集并请求提交之后、会话进入确认或撤销收口之前，由执行器在当前线程栈内构造并同步传给参与者。视图只允许复核本会话主信息 / 节点候选身份、候选节点类型、候选节点主信息锚点和候选主信息 I64 值；不得读取任意已发布结构，不得写入、确认或撤销，不得被复制、移动、缓存或跨线程传递。

既有会话普通读写入口继续只允许 `决定 == 未决定` 的写入阶段。请求提交或请求撤销后，普通节点类型、节点主信息、I64、关系读取和全部写入口仍必须拒绝；不得通过放宽 `可继续写入()` 实现参与者复核。提交准备视图内部必须另设窄门禁，同时满足当前线程、写入阶段、提交决定、无失败和尚未收口；任一条件不满足时返回空材料并按内部合同漂移追根因。

为验证每个确认点和参与者失败后的真实撤销，只允许执行器提供编译期受控的隔离自检入口。故障点必须覆盖每个节点确认后、节点批后、每个主信息确认后、主信息批后、每个关系变更确认后、会话全确认后、参与者准备 / 确认失败以及最终发布前；最终发布区内禁止故障点。触发后必须先撤销参与者精确服务内部记录，再撤销会话全部写集。不得增加运行期全局开关、持久故障状态或生产重载参数，不得把故障控制交给参与者、提交准备只读视图、领域服务或外部调用方。

确认故障自检能力只在 `Debug|x64` 的专用预编译宏下形成，Release 不编译该重载，入口也不登记对应隔离自检。Debug 生产调用点不得引用自检重载；静态扫描必须证明故障符号只出现在执行器、自检模块、入口条件登记和工程宏配置中。自检故障是验收驱动，不是业务失败来源，不得进入机器事实、日志裁决或正常重试协议。

服务内部材料准备或确认待发布后，若会话确认或参与者确认失败，参与者必须在独占许可释放前按本次精确身份撤销，随后会话全量逆序撤销。参与者准备阶段只有资源 / 竞争拒绝且未产生可读半结构时可保留为逻辑内返回；抛出、参与者内部状态不一致、准备后锁 / 写集 / 反向材料异常均归为追根因解决。两类非成功都不得确认四仓库候选，都必须先撤销参与者再撤销会话。结构与服务内部材料任一侧无法证明回到前态，均属于逻辑错误；执行器必须先登记事务域隔离，再停止后继路径。

锁序固定为“结构独占许可 -> 参与者内部独占锁”；接域生产侧表读取固定为“结构共享许可 -> 参与者内部共享锁”。参与者一旦取得内部写锁，必须持有到完成发布或撤销完成，期间不得调用仓库入口。`特征值服务` 的接域创建 / 写入和七个公开读取必须在任何侧表锁之前取得结构许可，并只沿带令牌仓库入口继续；唯一运行期业务装配必须传入与四仓库相同的接线。既有两参数无接线构造只保留隔离自检 / 未接域兼容上下文，不计入原子可见性完成声明。既有绕过结构许可直接读写侧表或在侧表锁内调用无令牌仓库入口的接域生产旁路必须在 #291 范围内收口；无法收口时退回设计，不得用非阻塞读失败掩盖可见性裂缝。

第一轮事务参与者只允许绑定本会话新建且尚未确认的主信息和节点候选，不允许借参与者修改已发布身份。已发布服务内部材料的更新、版本推进和历史替换必须另立补偿或不可变新版本专项。

## 5. 结构事务协调层

结构事务协调层负责：

```text
签发共享 / 独占许可
释放许可
验证运行期域、纪元、许可序号、线程和许可强度
提供一次操作的跨仓库事务上下文
拒绝同线程重入和失效许可
在撤销失败时登记事务域隔离并拒绝后续全部许可
```

协调层不解释节点类型、关系类型、需求、任务、方法、场景或存在语义，不写业务结构，不把许可对象持久化。

## 6. 仓库职责

仓库公开给数据操作层的入口必须：

```text
验证不透明会话内部携带的原始令牌
按本仓库规则取得共享或独占锁
完成句柄、版本、状态和结构前置拒绝
返回值式记录、句柄或结构化操作结果
```

仓库不得接收业务 DTO，不得判断需求是否满足、任务是否授权、方法是否可执行或概念是否应命名。普通业务调用不得直接进入仓库写入口。

## 7. 服务专用数据操作层

服务专用数据操作层是领域服务和仓库之间唯一的多仓库写入适配层，负责：

```text
隐藏仓库引用和原始结构事务令牌
在同一不透明会话内执行多仓库读写
维护未发布候选、精确撤销、确认和最后发布顺序
把仓库结构结果转换成领域服务可解释的结构化结果
```

数据操作层不得决定需求满足、任务授权、方法选择、概念生命周期等业务语义。它只执行已经由领域服务或组合器形成的类型化结构请求，并复核通用结构前置条件。

数据操作层若使用第 4.7 节事务参与者，必须同时证明参与材料绑定本会话候选、服务内部材料与四仓库结构同一许可内读回，以及失败后两类写集均无残留。领域服务、组合器和外部调用方不得直接取得事务参与者。

### 7.1 用途观察候选结构的实施前门禁

## 8. 领域服务

领域服务公开业务入口：

```text
只接收业务请求、稳定句柄和值式材料
负责业务准入、业务不变量、幂等语义和成功 / 失败含义
调用服务专用数据操作入口
只返回领域结果或拒绝材料
```

领域服务公开入口禁止：

```text
暴露、接收、返回、传递或比较原始结构事务令牌
直接取得或释放结构许可
直接操作仓库锁
把仓库引用暴露给调用方
让外部调用方决定撤销、确认或最后发布顺序
```

领域服务内部兼容路径在完成调用点替代前可以保留，但不得作为新代码范例。

## 9. 跨服务组合器

跨服务操作由明确的应用 / 领域组合器定义一次事务边界。组合器只提交业务操作，不持有原始令牌；结构写入执行器在内部创建不透明会话，并向参与的数据操作层提供会话能力。

组合器负责跨服务业务顺序和最终业务结果；数据操作层负责精确结构提交。线程、消息和外部调用方不得携带原始令牌穿透多个服务。

## 10. 兼容路径治理

当前无令牌写、带令牌写和候选写并存。治理顺序固定为：

```text
调用点清单
-> 标注生产 / 自检 / 兼容 / 禁止
-> 建立无原始令牌替代入口
-> 按存在场景、状态动态、特征、语义基础、需求任务方法顺序迁移
-> 迁移应用组合器和线程路由
-> 证明生产调用点为零
-> 再裁决私有化、弃用或删除兼容入口
```

不得先删除兼容入口再补调用方，不得把带令牌入口改名后继续暴露，也不得用全局变量或线程局部变量隐藏令牌穿透。

## 11. 第一组垂直样例

第一组样例固定为“存在 -> 场景”：

```text
创建实际存在
创建虚拟存在
创建场景
场景接纳存在
创建并接纳场景存在
```

必须覆盖入口拒绝、主信息 / 节点候选、来源关系、归属关系、最后发布、精确撤销、确认、读回、幂等和失败无残留。完成后才扩展状态 / 动态、特征体系、语义基础和需求 / 任务 / 方法。

## 12. 新文件规则

新生产逻辑必须使用独立 `功能.分类.ixx` 真模块，`ClCompile + export module + import`。稳定数据结构可使用 `功能.分类.h/.cpp`。生产与自检必须分离，`入口.cpp` 只保留最小登记和顶层调用。

## 13. 性能后置

批量读取、计数、关系二级索引性能基线和仓库分片继续证据门控。必须先完成仓库 / 服务分层纠偏，再建立可代表新调用路径的性能基线。不得为微基准牺牲入口前置拒绝、版本句柄、普通删除独占和事务安全。

## 14. 禁止扩大声明

本规范确认分层口径，不证明数据操作层、写入会话、存在场景样例、服务迁移、兼容路径收口、性能优化或 `#214` 已实现。

## 4. 因果完整结构键

### 4.1 唯一字段

第一版 `因果完整结构键` 必须完整保留：

```text
格式版本。
原因动态键：目标完整句柄、前值、后值、来源动作完整句柄、是否有来源动作、动态规则版本、可选二次角色值。
结果动态键：同样七字段。
因果规则版本。
```

原因与结果是有序角色。交换两把动态键形成不同身份；不得排序后抹平角色。两把动态键当前规则版本数值相同，也必须分别保留，不能由因果版本推导或省略。

### 4.2 版本合同

格式版本 1 只接受明确版本兼容表：

```text
身份格式版本 = 1
原因动态键规则版本 = 当前动态模式规则版本
结果动态键规则版本 = 当前动态模式规则版本
因果规则版本 = 当前因果模式规则版本
```

兼容表之外返回“不支持”，不得用三个版本当前数值相等推断未来兼容。后继版本必须新增显式兼容行和迁移计划；不得原地放宽版本判断。

### 4.3 比较与哈希

```text
A/B 相同：格式版本、原因七字段、结果七字段和因果规则版本全部相同。
A/B 不同：上述任一字段不同。
稳定哈希：按格式、原因角色、结果角色、因果版本固定顺序混入，只用于候选桶召回。
哈希命中：仍须执行 A/B 全字段比较。
哈希碰撞：两个不同完整键保持不同事实，不得合并、覆盖或选首项。
```

完整比较和哈希只接受局部完整、原因与结果不同且版本兼容的身份。任一输入无效 / 不支持时，完整比较返回假；哈希返回空值，不得把两个默认无效对象归为同一身份，也不得给无效对象分配可进入候选桶的默认哈希。

名称、类型、概念句柄、向量首项、时间戳、节点编号、冻结键或日志均不得参与最终身份裁决。

### 4.4 #287 已完成边界

#287 已完成的正式产物只包括：

```text
在现有因果材料模块声明纯值因果完整结构键及格式版本。
在现有因果算法模块形成、复核、完整比较和计算召回哈希。
让用途事件的历史概率分量持有该完整身份，不再散列复制三元组字段。
新增独立 Debug 自检模块和阶段 860 最小登记。
```

身份形成结果必须默认初始化为明确入口拒绝态，并只允许四类组合：成功时“已形成 + 无拒绝原因 + 有合法身份”；材料错误时“入口拒绝 + 候选不完整 / 原因结果相同 + 无身份”；版本错误时“不支持 + 版本不兼容 + 无身份”；算法异常时“内部不一致 + 算法不变量破坏 + 无身份”。

#287 的代码差异保持节点、主信息、关系、索引、仓库、许可、数据操作、线程、运行期装配、持久化、概念映射、统计、学习、晋级和方法写入为零。阶段 860 只证明该纯值切片的 Debug 自检，不表示旧用途观察阶段草案复活。

## 6. 权威用途观察后继门禁

### 6.1 上游用途事件结果状态矩阵

当前 `用途事件规则版本 = 1` 只允许：

| 状态 | 允许原因 | `事件` optional | O1 收口 |
| --- | --- | --- | --- |
| 已形成 | 无 | 必须有 | 进入身份门禁 |
| 入口拒绝 | 请求不完整、来源冻结不完整、结果尚未正式收口、非新版完整回执、来源冻结回合不一致 | 必须无 | 逻辑内入口拒绝 |
| 不支持 | 规则版本不支持 | 必须无 | 逻辑内不支持 |
| 内部不一致 | 正式结果材料矛盾、身份材料矛盾、算法不变量破坏 | 必须无 | 追根因并停止 |

`因果候选不完整` 当前只属于上游 `用途来源冻结结果`，不属于正式 `用途事件结果` 的合法原因。任何未列出的状态—原因配对或 optional 形状均为内部不一致，不得把内部错误降格为入口拒绝或不支持。

### 6.2 语义身份

未来用途观察的语义唯一身份固定为：

```text
完整任务句柄 + 任务序号
```

`冻结键与观察主键` 的旧字段名只保留为 #282 调用期来源关联材料；它不再被预设为权威观察主键、物理索引键或事实身份。物理索引可以使用版本化命名域和哈希加速，但命中后必须比较完整语义身份与完整观察内容。

### 6.3 不可变核心与阶段负载

未来观察内容必须拆为：

```text
不可变核心：需求、选择记录、方法、动作、场景、主体、目标状态、实际状态、动态证据九个完整句柄，
            加观察形成时的因果完整结构键、机会数、正结果数、负结果数和 Q10000 历史概率快照。
阶段负载：正式结果、任务达成、需求结算、观察阶段，以及最终已结算时唯一允许出现的正式结算状态句柄。
```

完整冻结请求、完整任务结果回执、完整需求结算 DTO、完整机会证据组、待结算提示、冻结键、请求 / 回执 / 批次号、日志、文本、缓存、显示材料和目标 / 实际状态值的第二权威副本均不得进入最小内容。

阶段负载只允许三种元组：

```text
未达成已转待重筹办 + 未达成 + 未形成 + 尝试未达成 + 无正式结算状态。
任务已完成待需求结算 + 已达成 + 待结算 + 待结算 + 无正式结算状态。
已完成并结算 + 已达成 + 已满足 + 最终已结算 + 有完整正式结算状态。
```

### 6.4 节点与关系暂不分配

`节点类型::用途观察记录 = 15` 和 `关系类型::用途观察证据角色 = 18` 当前均未锁定。后继设计只有同时闭合以下项目后，才可通过新计划分配枚举：

```text
用途观察最小权威字段和明确排除字段。
完整任务回合身份与完整内容 A/B 比较。
物理索引命名域、碰撞分类和双重唯一。
每一关系角色的端点类型、顺序号、基数和审计语义。
固定单观察元素 / 字节上限、总预算和写前容量拒绝。
待结算到最终已结算的同账阶段换代与历史读回。
专用数据操作、不透明会话、逐项读回、确认、最后发布和每一种失败路径的撤销证明义务。
同一运行期宿主和既有恢复治理计划链；恢复格式、段号、编码和发布接口另行设计。
```

不得把完整冻结请求、完整回执、完整结算 DTO、完整机会证据组等来源业务 DTO 直接序列化进通用主信息 I64 槽。完整节点句柄和因果完整结构键属于经批准的固定形状值对象，不属于该排除项，但也不得裸塞进单个 I64 槽。不得为了抢占编号先锁定节点或关系，再把字段、容量和恢复留给执行角色发明。

### 6.5 幂等、推进与冲突

候选合法性检查必须先于同账比较。上游用途事件已经被接受为成功后，新候选必须先独立满足 6.3 的三种合法阶段元组；最终已结算候选还必须沿 `事件.结果来源.需求结算.材料.正式结算->结算状态.状态节点` 深检到完整且一致的正式结算状态，`最近结算状态` 只作兼容投影。可选既有候选也必须先独立满足相同格式、身份、核心与阶段合法性；若已发布既有候选自身非法，属于内部不一致并追根因。

以下场景不得进入同账比较：

```text
成功事件的新候选阶段元组非法。
最终已结算新候选缺少正式结算材料、正式结算状态无效或与正式需求结算材料矛盾。
已发布既有候选的格式、身份、核心或阶段负载自身非法。
```

只有新候选与可选既有候选各自合法后，未来合同才执行以下比较：

```text
同身份 + 同不可变核心 + 同阶段负载 -> 同账幂等。
同身份 + 同不可变核心 + 待结算阶段负载推进为合法最终阶段负载 -> 同账允许推进。
同身份 + 异不可变核心 -> 同账冲突。
尝试未达成或最终已结算发生任何阶段变化、最终逆向待结算、待结算转尝试未达成或其它不允许阶段转换 -> 同账冲突。
同一身份出现多个已发布当前记录 -> 内部不一致并追根因。
上游任务结果回执准入阶段发现任务序号倒退 -> 写前逻辑返回。
用途事件已被接受为成功后再发现任务序号、完整冻结请求、回执内嵌来源请求或回执身份不一致 -> 内部不一致并追根因。
```

## 8. 方法执行快照与确定差异

学习前必须在动作执行前冻结来源方法完整内容：

```text
来源方法完整身份。
独立方法内容版本。
确定排序的全部条件规格。
确定排序的全部结果规格。
完整动作入口规格与输入 / 输出场景。
来源生命周期和选择材料只作当时快照，不替代内容版本或父代。
```

当前条件 / 结果规格的 `来源版本` 只描述其来源材料，不是方法内容版本；任务方法选择中的来源生命周期只证明选择时活跃，不是方法代际。

来源观察组必须按完整观察身份确定排序、去重并逐项复核当前性。差异规则必须显式输出：

```text
新增：新规格存在、来源版本不存在。
保留：正式角色和完整规格相同。
移除：来源规格存在、新规格明确不存在。
替换：同一正式角色槽存在唯一旧规格和唯一新规格，且完整内容不同。
未知：材料不足，不能确定新增 / 移除 / 替换。
冲突：同一角色槽多映射、同身份异内容或正式材料矛盾。
```

在正式角色槽、方法内容版本和完整观察组尚未形成前，默认返回材料不足，不自动生成差异。Q10000、单次任务达成、单一分数、哈希或任务序号均不得直接裁决新增、移除、替换或晋级。

## 9. 晋级与方法代际

晋级必须独立复核：来源观察仍有效、统计可重建、来源方法内容版本当前、候选完整、任务明确授权、晋级规则版本受支持、同义未重复消费、异义不冲突。

通过后也只能向方法业务服务提交类型化新代际请求。统计、线程、因果模式、学习候选或晋级服务不得直接操作仓库、原始令牌、结构会话或方法关系。

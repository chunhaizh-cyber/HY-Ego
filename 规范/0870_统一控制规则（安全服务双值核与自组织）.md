## 4. 控制核定义

在现有字段体系里，统一控制核落成：

```text
控制核 = {
    控制.安全值,
    控制.服务值,
    控制.需求值,
    控制.方向,
    控制.模式
}
```

必须固定：

- `控制.安全值` 决定边界强度
- `控制.服务值` 决定服务牵引强度
- `控制.需求值` 表达当前阶段必须处理到什么程度
- `控制.方向` 表达当前控制朝向
- `控制.模式` 表达当前功能模式

控制顺序必须固定为：

```text
先看安全值决定当前能不能继续筹办、维护、纠偏或收束
再看服务值和需求值决定值不值得投入、优先保哪一侧
最后由方向决定本轮控制朝向与动作姿态
```


## 5. 固定方法集原则

必须继续保持：

- 元方法固定
- 自我方法固定
- 全部基础方法由开发者提供
- 系统不自行生成新基础方法
- 系统只做筛选、组合、替换、回退和学习修正

可收束为一句：

- `数字生命的成长不是长出新指令，而是在固定方法集上组织出更长、更细、更稳的任务结构和执行链。`


## 6. 自组织对象对齐现有模型

当前版本里，自组织只允许作用在以下对象上：

- 任务实例
- 分解关系
- 绑定关系
- 替代关系
- 反馈实体
- 经验实体
- 事件链

### 6.1 允许增长的

- 新子任务
- 新检查任务
- 新恢复任务
- 新阶段总结任务
- 新绑定关系
- 新替代关系
- 新反馈
- 新经验

### 6.2 不允许增长的

- 新元方法定义
- 新自我方法定义
- 新基础状态机
- 新核心枚举骨架


## 7. 语义字段与特征字段的控制分工

必须固定一句：

- `语义字段定边界，特征字段定偏好。`

### 7.1 语义字段负责硬约束

典型字段：

- `控制.安全值`
- `任务.最大深度`
- `反馈.安全变化`
- `来源.父任务标识`
- `编排.顺序号`

这些字段直接决定：

- 能不能继续细分
- 能不能继续延长
- 是否必须回退
- 当前任务处在什么链上

### 7.2 特征字段负责软选择

典型字段：

- `特征.复杂度`
- `特征.风险级别`
- `特征.收益级别`
- `特征.可回退性`
- `特征.稳定性`

这些字段主要用于：

- 候选方法排序
- 是否插入检查点
- 是否加备用方法
- 是否改走更保守路径


## 8. 任务控制的四大功能域

当前链条骨架为：

```text
需求 -> 创建任务 -> 分解任务 -> 绑定方法 -> 开始任务 -> 完成任务 -> 写入反馈
```

任务控制不再只理解为“怎么长”，而应固定为：

```text
任务控制
= 筹办任务
+ 维护任务
+ 纠偏任务
+ 收束任务
```

也可以直接定义成一句话：

- `任务控制，是让任务从被提出到被关闭，始终保持可筹办、可运行、可修正、可收束。`

### 8.1 筹办任务

筹办任务回答的是：

- 这个任务要不要立
- 立成什么结构
- 先做哪些，后做哪些
- 用哪些方法起步
- 要不要细分
- 要不要延长路径
- 要不要插检查点

当前实现对齐点主要包括：

- `任务执行器 / 根任务筹办器`
- `筹办中 / 待重筹办`
- 方法召回、条件缺口分析、补条件子任务、最小试探步骤生成

筹办任务内部更接近：

```text
生成 + 细分 + 铺路 + 初选 + 初始化
```

### 8.2 维护任务

维护任务回答的是：

- 任务现在还健不健康
- 当前状态有没有漂移
- 资源、方法、依赖是否还成立
- 检查点要不要继续维持
- 是否需要局部更新而不是整体重建

当前实现对齐点主要包括：

- `自我线程类::周期性维护`
- 根方向与需求刷新
- 自我核心特征同步
- 根任务就绪状态刷新
- 等待学习任务唤醒后的再入队

维护任务更接近：

```text
保活 + 刷新 + 检查 + 稳定绑定 + 局部修补
```

### 8.3 纠偏任务

纠偏任务回答的是：

- 跑偏了没有
- 偏到什么程度
- 是局部修，还是整体退
- 换方法、补条件、加检查，还是直接回退

当前实现对齐点主要包括：

- 回退与备用方法切换
- `任务失败恢复模块_v0`
- 学习兜底补候选
- 恢复任务、补条件子任务、等待学习唤醒

纠偏任务更接近：

```text
发现偏差 + 替换 + 回退 + 收缩 + 恢复
```

### 8.4 收束任务

收束任务回答的是：

- 什么时候算够了
- 什么该结束
- 什么该合并、冻结、清理
- 什么该总结并写回学习

当前实现对齐点主要包括：

- 完成判定
- 反馈写入
- 学习写回
- 无法执行 / 完成 / 取消后的尾部处理

收束任务更接近：

```text
判定完成 + 清尾 + 总结 + 写回 + 归档
```

### 8.5 功能域与执行动作的关系

必须固定：

- `筹办 / 维护 / 纠偏 / 收束` 是任务控制的功能域
- `保持 / 细分 / 延长 / 回退 / 收缩 / 插检查 / 替换方法 / 归并 / 结束` 是功能域内部调用的执行动作

也就是说：

- 生长和完善主要属于 `筹办任务` 内部能力
- 保活和刷新属于 `维护任务`
- 回退、恢复、替换方法主要属于 `纠偏任务`
- 完成判定、学习写回、资源释放主要属于 `收束任务`

### 8.6 四大功能域与双根偏重

四大功能域都由 `安全值 + 服务值` 双根驱动，但偏重不同：

- `筹办任务`：服务值决定值不值得立、立多大；安全值决定能立到什么边界
- `维护任务`：安全值决定是否必须维持稳定；服务值决定维持投入是否值得
- `纠偏任务`：安全值决定是否必须立刻纠偏；服务值决定纠偏后还值不值得继续推进
- `收束任务`：服务值决定目标是否已经足够满足；安全值决定是否应尽快关闭不稳定残留

可压成一句：

- `四类功能都由双根驱动，只是偏重不同。`

### 8.7 方法可靠度在四大功能域中的位置

方法可靠度不只服务于“选方法”，而是贯穿四大功能域：

- `筹办任务`：用于初选方法、决定路径长度、决定检查点密度
- `维护任务`：用于判断当前方法是否还稳、是否需要刷新绑定、是否需要切到更稳的方法
- `纠偏任务`：用于判断当前方法是不是偏了、选哪个备用方法接手、该回退多少
- `收束任务`：用于判断结果是否足够稳定、能否沉淀经验、方法经验是否应写回

也可以收成一句：

- `可靠度不是只服务于选法，而是贯穿筹办、维护、纠偏、收束四个主域。`

## 9. 功能模式、执行动作与方向

### 9.1 功能模式

用 `控制.模式` 表达当前功能模式：

```text
筹办 / 维护 / 纠偏 / 收束
```

### 9.2 执行动作

当前执行动作建议固定为：

```text
保持 / 细分 / 延长 / 插检查 / 替换方法 / 回退 / 收缩 / 归并 / 结束
```

### 9.3 控制方向

用 `控制.方向` 表达当前控制朝向：

```text
完成优先 / 稳定优先 / 修复优先 / 探索优先 / 收缩优先
```

### 9.4 三者区别

- `模式`：当前处在哪个功能域
- `动作`：在该功能域里实际调用什么执行动作
- `方向`：为什么朝这个方向做


## 10. 典型执行动作对齐到现有链条

在四大功能域下，当前最常见的执行动作如下。

### 10.1 保持

实现方式：

- 维持当前任务链与当前方法绑定
- 刷新必要状态、检查点和局部控制快照
- 不扩张主结构，只做最小必要维护

触发依据：

- 当前链足以继续推进
- 风险受控且无明显漂移
- 当前收益不足以支持重建或大幅改路

主要归属：

- `维护任务`

### 10.2 细分

实现方式：

- 创建新的子任务实体
- 建立 `分解` 关系
- 继承或写入控制快照
- 记录 `触发细分` 事件

触发依据：

- `控制.需求值` 高
- `控制.安全值` 允许
- `特征.复杂度` 高
- 当前深度未超过 `任务.最大深度`

主要归属：

- `筹办任务`

### 10.3 延长

实现方式：

- 新增后续任务
- 或新增检查任务、恢复任务、总结任务
- 用 `分解` 或新一层任务链表达
- 设置 `编排.顺序号`
- 记录 `触发延长` 事件

触发依据：

- 需要补验证闭环
- 需要补安全检查
- 需要补恢复步骤
- `特征.时效性` 和 `特征.风险级别` 同时高

主要归属：

- `筹办任务`
- `纠偏任务`

### 10.4 回退

实现方式：

- 当前任务转 `待重筹办`、`失败`、`无法执行` 或其他恢复等待态
- 查询 `替代` 或备用方法
- 生成恢复任务或重新绑定方法
- 记录 `触发回退` 事件

触发依据：

- `反馈.失败原因` 非空
- `反馈.安全变化` 为负
- `控制.安全值` 低于阈值

主要归属：

- `纠偏任务`

### 10.5 收缩

实现方式：

- 停止继续分解
- 不再增加备用分支
- 只保留主路径任务
- 记录 `触发收缩` 事件

触发依据：

- `控制.安全值` 过低
- `特征.波动性` 高
- 连续失败过多

主要归属：

- `纠偏任务`
- `收束任务`

### 10.6 学习修正

实现方式：

- 写经验实体
- 建立 `学习回写` 关系
- 更新 `绑定 / 替代 / 分解` 关系的 `权重`
- 记录 `学习写回` 事件

必须固定：

- `学习改的是适配强度、路径偏好和回退可靠度，不是发明新方法。`

## 11. 方法选择规则

统一流程为：

```text
在固定方法集中筛选候选方法
-> 先做安全硬过滤
-> 再用特征和权重做软排序
-> 形成主方法和备用方法
```

最稳的评分口径可写成：

```text
最终方法得分 =
    关系权重
  + 特征匹配分
  + 服务收益修正
  - 风险修正
```

必须固定：

- `关系权重` 表示历史经验
- `特征字段` 表示当前画像
- `控制值` 表示当前边界和方向
- 安全不过线的方法不参与排序


## 12. 控制流程对齐现有元层

统一流程落到现有模块中应写成：

```text
读取当前任务
-> 读取来源需求
-> 读取已绑定方法
-> 读取最近反馈与经验
-> 形成控制核
-> 判断当前功能模式
-> 在固定方法集中筛选候选方法
-> 安全硬过滤
-> 用特征和权重做软排序
-> 进入筹办 / 维护 / 纠偏 / 收束中的一个主域
-> 在该主域内调用保持 / 细分 / 延长 / 插检查 / 替换方法 / 回退 / 收缩 / 归并 / 结束等动作
-> 写入反馈
-> 学习写回 / 归档
```

由于 `结构_元事件` 只有 `事件类型` 字符串，不新增枚举，所以统一以下事件类型词表：

```text

触发延长
触发回退
触发收缩
方法替换
插入检查点

```

事件内容建议最少带：

```text
控制.安全值
控制.服务值
控制.需求值
控制.方向
控制.模式
系统.原因
```

## 14. 收束

本文定义的是：

- `在现有结构字段不扩骨架的前提下，如何把控制核、自组织和学习修正接到现有任务链、关系链和事件链里。`

最核心的话是：

- `系统不靠发明新方法成长，而靠在固定方法集上，自组织出更合适的任务结构和执行链。`

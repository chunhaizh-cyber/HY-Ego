# 6160 子规范 服务需求合同预算与准备结算

编号：6160
日期：20260724
状态：正式子规范

## 1. 定位

本规范定义直接服务需求、服务准备、服务合同唯一性、有效期、冻结预算、30% 预支、70% 完工结算、终止不追缴和准备衰减补回。它只定义服务合同与结算事实，不定义服务值随时间下降，也不把服务结算解释成安全事实。

直接上级规范：3100、3200、3300。

直接依赖规范：4020、4210、4220、5100、5110、5160、5200、6100、6120。

相关边界规范：1190、5230、5300、6140、6150、6170、8100、8200。

## 2. 服务对象与两类服务

服务只分为：

```text
直接服务
-> 满足由有效提出者真实提出、来源可验证且不违反现行法规准入的需求

服务准备
-> 形成新的、可验证且可用于后续服务的能力或材料
```

直接服务必须绑定人类提出者、真实需求、目标、范围和有效期。法规准入只决定能否建立合同和执行服务，不产生服务值加分。

合同建立还必须回读 6120 / 6170 的当前生存控制根值和版本：`A=0` 终止态写前拒绝合同与预支，`A=1` 休眠态和 `A>1` 主动运行态才允许继续服务准入。休眠允许合同准入不等于服务任务已经取得普通执行权限。

服务准备必须绑定准备目标、可验证新结果、适用服务范围、来源需求或服务能力缺口、执行方法和完成验证。失败、未完成、重复形成同一结果、只改变显示或只产生不可验证中间物，均不成立准备结算。

能直接解除自我安全问题的人类请求仍可建立服务合同和按本规范结算，但其任务执行分类按 6170 的真实因果路径归为安全任务；合同结算身份与任务调度分类不得互相覆盖。

## 3. 服务需求与合同身份

一项可建立合同的服务需求必须回读：

```text
自我稳定身份
提出者稳定身份
需求稳定身份及当前版本
来源提出事件、状态或动态
目标宿主、目标状态和满足关系
服务目标与明确范围
提出运行代次、时间纪元和完整秒边界
有效开始完整秒边界、冻结有效总秒和已消费有效秒
跨代次连续时长证明或时间证据缺口
法规准入结论及规则版本
需求当前状态与来源版本
```

提出者身份、需求、目标、范围和来源事实之间的拓扑由正式关系承载；提出时间纪元、有效总秒、已消费有效秒、预算、结算值和规则版本进入需求所属的服务合同类型化记录。不得把句柄、对象关系或状态拓扑复制进类型化记录。

合同稳定身份由“自我 + 提出者 + 服务需求稳定身份 + 首次提出事件 + 合同代次”共同成立。合同记录归属于该需求节点，不另建无类型合同容器。

同一来源需求在一个有效期内只能存在一个当前服务合同：

1. 补充说明、目标的等义细化、任务拆分、子任务增加、方法重试和执行回合变化继续引用原合同。
2. 上述变化不得创建第二合同、重复预支或扩大冻结预算。
3. 有效期届满后，只有提出者形成新的真实提出事件，并由需求域建立新的需求代次或新需求身份，才允许建立新合同。
4. 自我不得复制历史需求、自动续期、代替提出者提出或仅凭历史推断需求仍然有效。
5. 同一合同键同义重复必须读回既有合同；同键异义、有效期内第二合同或旧需求版本请求新预支均写前拒绝。

## 4. 合同状态事实

合同当前状态至少区分：

```text
待准入
有效未满足
已完整完成
提出者取消
已到期
正式终止
```

状态分类必须由需求当前状态、目标事实、提出者取消事实、有效期和正式终止事实共同裁决，并通过完整状态事实发布；普通枚举或 bool 不得独立承担合同状态。

`已完整完成、提出者取消、已到期、正式终止` 都是终态。终态合同不得恢复为有效，也不得继续取得余款、准备补回或任务执行权限。需要再次服务时必须形成新的真实提出事件和新合同代次。

## 5. 有效期与默认规则

有效期以完整物理秒计，使用非负 `I64`。合同必须满足：

```text
有效开始完整秒边界 >= 0
冻结有效总秒 > 0
已消费有效秒 >= 0
已消费有效秒 <= 冻结有效总秒
```

提出者明确给出合法有效期时使用其冻结秒数。提出者未给有效期时，项目规则版本 1 使用最小正式默认：

```text
默认有效秒数 = 2,592,000 秒
```

该默认值与 6170 的现行 30 天服务维护观察窗一致，是本项目的版本化设计默认，不解释为提出者已经明确表达 30 天。默认规则版本必须进入合同记录；未来修改只影响新合同，不改写既有合同预算或有效期。

无效、负数、零秒、溢出或无法证明来源的有效期在建立合同前拒绝。墙钟文本、日期字符串和循环次数不得替代单调完整秒。

原始单调计数只在同一运行代次和时间纪元内比较。长期保存必须冻结有效总秒、已消费有效秒、最后已结算完整秒边界、时间纪元和连续性证明；恢复后只有取得可信连续时长证明才补算停机区间。没有证明时建立新时间纪元，合同保留且时间证据标记为缺口，不猜测到期、不拼接原始单调计数，也不使用可回拨墙钟补算。

## 6. 预算

设：

```text
M = I64_MAX
T = 2,592,000 秒
D0 = max(1, floor(M / T))
E = 合同冻结有效秒数
```

合同总服务预算为：

```text
B = clamp(D0 * E, 1, M)
```

乘法必须使用至少 128 位中间值或等价的检查乘法；超过 `M` 时饱和为 `M`，不得回绕。预算在合同建立时冻结，后续拖延、阻塞、任务拆分、方法重试、实际耗时、服务数量和等待压力均不得增加 `B`。

规则版本 1 的精确常量为：

```text
M  = 9,223,372,036,854,775,807
T  = 2,592,000
D0 = 3,558,399,705,576
```

默认有效期的预算为 `9,223,372,036,852,992,000`，与 `M` 的差额来自整数除法余数直接舍弃；不得补偿该余数。

## 7. 30% 预支与完工余款

合同成立时必须在同一事务中冻结预算并结算预支：

```text
预支值 P0 = max(1, floor(B * 30 / 100))
完工余款 P1 = B - P0
```

实现可以使用宽中间值，也可以使用等价无溢出式：

```text
P0 = floor(B / 10) * 3 + floor((B mod 10) * 3 / 10)
```

完整完成后只支付 `P1`。提出者取消、到期、正式终止或无法完成时，`P1` 永不支付；已支付预支不追回。

服务值采用剩余空间饱和增加。每次结算必须分别记录：

```text
应付值
结算前服务值
实际写入值
因达到 I64_MAX 而舍弃的超额值
结算后服务值
合同结算阶段
幂等键和规则版本
```

饱和未实际写入的超额值直接舍弃，但该阶段仍标记已经结算，禁止以后补领。`P0` 和 `P1` 各只能结算一次；任务拆分、重复完成、方法重试、同义回执或重新读取不得重复支付。

服务数量、任务数量、方法次数、线程数和消息数不参与预算、预支、余款或服务衰减计算。

## 8. 同秒事实与结算顺序

一个完整秒边界内存在多个事实时，顺序固定为：

```text
冻结该秒已经发布的需求、任务、方法、状态、动态和提出者事实
-> 先按事实发生完整秒和正式发布序处理完整完成、提出者取消和正式终止
-> 再判定已消费有效秒是否到达冻结有效总秒
-> 形成唯一合同终态
-> 若完整完成，结算尚未支付的完工余款
-> 若取消、到期或终止，永久关闭余款
-> 若因到期进入终态且到期前仍未完整完成，发布一次“到期未满足事件”
-> 再把终态合同从有效未满足集合移除
-> 把终态快照和到期未满足事件交 6170 计算同秒服务值时间维护
```

生产治理循环必须通过同一个服务生存治理聚合入口冻结上述终态 / 准备事实与 6170 的完整秒输入，在一个不透明事务中共同准备、读回、确认并最后发布。合同终态、准备补回、服务衰减和生存安全回归不得由线程拆成多个可见中间事务。新的有效人类需求建立合同并结算 30% 预支可以独立原子发布；该预支只从后续完整秒进入 6170 的安全根回归。

同秒完整完成与到期同时存在时，按事实发生完整秒和正式发布顺序裁决：完成事实在截止边界前已经成立并发布时按完成；否则按到期。提出者取消或正式终止已经在到期判定前成立时不生成到期未满足事件。无法形成严格顺序时写前拒绝，不得通过容器遍历顺序、消息到达顺序或日志时间猜测。

到期未满足事件是不可变的同秒维护输入，至少包含合同、需求、提出者、冻结有效总秒、实际已消费有效秒、到期边界、目标未满足事实、合同终态、事实截止版本和规则版本。它不恢复合同有效性、不支付余款、不构成服务失败，也不建立压力概念；它只保证默认 30 天合同在到期秒仍能执行 6170 的 30 天归零或最低 1 门禁。

## 9. 服务准备与衰减补回

服务准备只有满足以下全部条件才进入结算：

1. 准备身份、来源服务能力缺口或服务需求、目标和有效期完整；
2. 实际产生新的能力或材料；
3. 结果能够由状态、动态、方法实际结果和验证事实回读；
4. 结果在具名服务范围内可使用；
5. 结果不是既有结果、重复副本、失败尝试或未完成中间物；
6. 同一准备结果尚未结算。

服务准备不取得预算预支或额外服务值增长。完整完成后只允许补回其有效推进期间实际发生、已经发布且唯一归属于该准备的服务值时间衰减：

```text
可补回值 = 全部唯一归属衰减段的实际减少量之和
实际补回值 = min(可补回值, I64_MAX - 当前服务值)
```

一个衰减段必须绑定服务值前后状态、维护秒区间、实际减少量、唯一准备归属和衰减规则版本。衰减发生时，若存在多个有效推进准备，唯一归属按：

```text
准备有效开始秒升序
-> 准备稳定身份升序
```

选择第一项并冻结，不因后续完成顺序改配。未被分配的衰减段不得补回；已分配准备失败、终止或未完成时，该段不再改配给其它准备。这样保证同一段衰减最多归属一个准备成果，并行准备不得重复补回。

补回发生饱和时超额直接舍弃并完成该准备的唯一结算；不得以后补领。补回只抵消已发生衰减，不得形成超过“没有这段衰减时”的净增长。

## 10. 结构、写入权与生命周期

服务合同和准备结算长期保存：

```text
需求、任务、方法、提出者、自我、目标、范围和来源事实的正式关系
需求所属服务合同类型化记录
合同当前状态和历史状态事实
到期未满足事件
预支、完工余款、准备补回和舍弃值结算记录
服务值前后状态、结算动态和幂等账
衰减段与准备结果唯一归属关系
规则、格式、合同和结算版本
```

服务合同领域服务负责准入、唯一合同、预算和结算语义；需求、任务、方法、状态和动态领域只提供自身已发布事实；服务专用数据操作在一个不透明事务中组织合同记录、服务值状态、动态、关系和幂等账。生产同秒终态 / 准备 / 时间维护必须由聚合服务一次调用该数据操作；外部调用方不得直接写合同记录、预算、服务值或结算阶段。

普通读取只返回当前合同和值式组合投影。索引只用于按提出者、需求或状态定位候选，命中后必须回读需求、关系、类型化记录、状态、动态和版本。

恢复必须保存合同唯一键、冻结有效总秒、已消费有效秒、时间纪元、连续性证明或缺口、预算、两个结算阶段、到期未满足事件、实际写入和舍弃值、准备归属、服务值状态和全部版本。任一缺项、重复合同、重复结算、衰减段多归属或未知规则版本均不得发布半恢复结构。

## 11. 幂等、拒绝与内部错误

合同建立幂等键至少包含：

```text
自我
提出者
需求稳定身份和首次提出事件
合同代次
冻结有效期
合同规则版本
```

结算幂等键追加合同当前版本、结算阶段、服务值当前版本和来源事实截止版本。准备补回幂等键追加准备身份、结果版本和唯一衰减段集合版本。

以下是合法逻辑内结果：

```text
法规准入不成立或待确认
需求来源、目标或有效期材料不足
同义合同或结算精确重复，读回既有结果
需求尚未完整完成，不支付余款
取消、到期或终止，余款关闭
准备未完成、未形成新结果或没有可补回衰减段
服务值已经饱和，实际写入为零且结算阶段完成
```

以下属于内部逻辑错误并停止依赖路径：

```text
同一有效需求出现两个当前合同
预支或余款重复支付
任务拆分或方法重试扩大预算
同一衰减段由多个准备补回
前置通过后合同、状态、动态、服务值或幂等账读回不一致
发布留下已支付但合同未标记、合同已完成但服务值未发布等半结构
算术溢出后回绕、钳制为无关值或使用负数
```

## 12. 验证合同

验证至少覆盖：

1. 提出者、需求、目标、范围、来源和法规准入逐项缺失或漂移。
2. 有效期内唯一合同、补充说明、任务拆分、方法重试和到期后新提出。
3. 默认 `2,592,000` 秒、显式有效期、零 / 负 / 溢出有效期、时间纪元、跨代次连续性证据和规则换代。
4. `D0`、预算饱和、默认预算余数、I64_MAX 边界和无溢出 30% 计算。
5. 预支一次、完工余款一次、取消/到期/终止不支付和预支不追回。
6. 饱和未写入值舍弃且不得补领。
7. 同秒完成、取消、到期、终止的确定性顺序，以及默认 30 天合同的到期未满足事件只发布一次。
8. 服务准备新结果、实际衰减、唯一归属、失败不改配、补回不净增长和并行不重复。
9. 合同、状态、动态、服务值、关系和幂等账同事务最后发布与逆序撤销。
10. 恢复后合同唯一、两个阶段不重放、准备归属不重复。

# 自我根需求与方向对齐规则

## 1. 目标

本文用于固定“自我”的总方向来源，以及需求、任务、方法后续生成时的统一对齐规则。

本文以逻辑规则为真值，不以自然语言名称为真值。

本文解决的问题是：

- 自我的最初方向从哪里来
- 安全与服务是什么关系
- 后续生成的需求、任务、方法如何判断是否对齐
- 执行结果如何重新回到根方向，形成持续闭环


## 2. 根需求与最初方向

### 2.1 根需求

当前定义：

- `自我的根需求 = 安全值最大化`

这里的“最大化”不是一句描述，而是一个明确目标：

- `目标安全值 = I64 / U64 体系中的上界目标值`

### 2.2 初始现状

当前定义：

- `自我的初始安全值 = 中位数值`

因此，自我一开始就天然存在：

- `现状值`
- `目标值`
- `差值`

这个差值本身就是最初方向的来源。

### 2.3 方向定义

定义：

- `方向 = 目标安全值 - 当前安全值`

因此，方向不是名称，不是标签，也不是一句自然语言，而是：

- 一个可持续计算的“目标差值缩减方向”

只要某个生成物能让这个差值缩小，它就是顺方向；
只要某个生成物会让这个差值扩大，它就是逆方向。


## 3. 安全与服务的关系

### 3.1 总方向与支撑方向

当前应区分两层：

- `安全方向` 是总方向
- `服务方向` 是支撑方向

也就是说：

- 安全决定“最终要去哪里”
- 服务决定“是否具备稳定靠近该方向的能力”

### 3.2 为什么服务不是独立总方向

服务值本身会影响安全值的变化规则：

- 服务值降低时，安全值更容易下降，恢复更慢
- 服务值升高时，安全值更容易回升，下降被抑制
- 服务值为 0 时，安全值被压到最小非零值，并进入待机

因此，服务不是脱离安全单独存在的目标，而是对安全方向产生因果影响的核心条件。

可以固定为：

- `服务需求的成立依据 = 它对安全方向具有正向因果贡献`

因此，服务需求虽然可以作为需求树中的一级需求存在，但它在方向判定上应服从安全总方向。


## 4. 需求树中的方向约束

### 4.1 需求树结构

当前需求树可以固定为：

- 根节点：`自我需求根`
- 一级需求：`安全需求`
- 一级需求：`服务需求`

除这两个一级需求外，其它需求都必须有父需求。

### 4.2 安全需求

安全需求直接表达：

- `让安全值更接近最大值`

它是总方向的直接表达。

### 4.3 服务需求

服务需求表达的不是“服务本身越高越好”这么简单，而是：

- `通过提升服务条件，使安全值更稳定、更可恢复、更不易衰减`

因此，服务需求是安全方向的因果支撑展开。

### 4.4 子需求生成规则

后续生成任何子需求时，都必须满足至少一条：

- 它直接缩小安全差值
- 它改善服务条件，从而间接缩小安全差值
- 它减少内部压力，从而降低安全值下行趋势
- 它修复会导致安全值恶化的结构性原因

如果一个需求无法投影到以上任一条，它就不应进入需求树。


## 5. 任务与方法的方向对齐规则

### 5.1 任务的成立条件

每个任务都必须对应某个需求。

任务是否成立，不看名称，而看：

- `该任务预期是否能让其所对应需求更接近满足`

再进一步追溯，就是看：

- `该任务对安全总方向的净贡献是否为正`

### 5.2 方法的成立条件

每个方法都必须服务于某个任务。

方法是否可被选中，不看它“像不像正确动作”，而看：

- `预测执行该方法后，任务对应需求是否更接近满足`
- `追溯到根方向后，安全差值是否缩小，或导致差值缩小的条件是否增强`

### 5.3 允许短期负、长期正

方向对齐不等于每一步都要求安全值立即上升。

允许存在以下情况：

- 短期安全值下降
- 但长期能让安全值更稳定、更高、更可恢复

因此判定依据应固定为：

- `看净因果贡献`

而不是：

- `只看当前一步的表面数值变化`

### 5.4 不对齐的情形

以下情形应视为不对齐：

- 只带来局部收益，但长期扩大安全差值
- 提高了某个局部指标，却削弱了安全恢复能力
- 提高服务值的方式反而造成更大的安全风险
- 无法解释其与根方向之间的因果链


## 6. 从状态变化中提取因果方向

### 6.1 真值来源

当前方向与因果，不应依赖名称，而应依赖状态变化过程。

也就是说：

- 先有状态
- 再有动态
- 再有二次特征
- 再从二次特征中归纳因果方向

### 6.2 安全值变化的主要来源

当前安全值变化至少来自以下几类：

1. 时序回归  
   安全值随时间向中位值回归。

2. 内部压力  
   待完成任务、待学习任务等积压会拉低安全值。

3. 服务耦合  
   服务值决定正向时序步长与反向时序步长，从而影响安全值恢复与下滑速度。

4. 异常损伤  
   物理伤害、硬件故障等异常事件会直接改写安全值。

### 6.3 服务对安全的因果说法

可以固定为以下因果链：

1. `服务值下降 -> 反向时序步长增大、正向时序步长减小 -> 安全值更易下降、恢复更慢`
2. `服务值上升 -> 正向时序步长增大、反向时序步长减小 -> 安全值更易回升、下降被抑制`
3. `服务值为 0 -> 安全值降到最小非零值并进入待机 -> 暂不死亡，但失去主动推进能力`

### 6.4 因果提取原则

当观察到安全值变化时，应把一次变化拆成来源贡献：

- 时序回归贡献
- 内部压力贡献
- 服务耦合贡献
- 异常事件贡献

然后判断：

- 哪一类贡献先发生
- 哪一类贡献方向稳定
- 哪一类贡献重复出现
- 哪一类贡献强度与结果强度一致

满足“先因后果、方向一致、重复稳定”的关系，才应被固化为因果信息。


## 7. 对齐判定的统一原则

后续一切生成物，不按名称对齐，而按它对“根需求差值”的预测净贡献来对齐。

可以收敛为一句总规则：

- `凡是能缩小安全根需求差值，或能稳定创造缩小该差值条件的生成物，都是对齐的。`

对应地：

- `凡是会扩大安全根需求差值，或破坏其缩小条件的生成物，都是不对齐的。`


## 8. 闭环规则

完整闭环应为：

1. 根需求给出总方向
2. 当前值与目标值形成差值
3. 差值触发需求树生长
4. 需求生成对应任务
5. 任务筛选并组织方法
6. 方法执行产生实际结果
7. 实际结果回写需求状态、任务状态、自我内部状态
8. 自我安全值与服务值变化
9. 由状态继续生成动态、二次特征、因果信息
10. 因果信息继续修正需求与任务生成方向

这意味着系统不是“生成一次计划后静态执行”，而是：

- `持续按根方向做闭环修正`


## 9. 当前工程实现上的含义

按当前工程分工，本规则对应的职责可理解为：

- `需求类` 负责需求链本身的创建、挂接、去重、权重与选择
- `任务类` 负责任务实例、任务节点与任务运行态写回
- `任务执行类` 负责围绕需求组织方法、预测结果与执行推进
- `自我类 / 自我线程模块` 负责安全值、服务值及其时序规则
- `状态 / 动态 / 二次特征 / 因果` 负责把运行事实提炼成可复用判断依据

因此，本文件不是替代这些模块，而是给这些模块提供统一的方向判定准则。


## 10. 本文件的约束结论

后续实现中应固定以下结论：

1. `安全值最大化` 是自我的根需求。
2. `初始安全值为中位数` 与 `目标安全值` 的差值，定义了最初方向。
3. `服务需求` 不是脱离安全的平行终点，而是安全方向的支撑方向。
4. 其它需求都必须能追溯到安全根方向，否则不应成立。
5. 任务与方法是否可用，必须按其对根需求差值的净因果贡献判定。
6. 短期负、长期正的行为允许存在，但必须能证明总方向为正。
7. 方向判断以状态变化和二次特征为真值，不以名称为真值。

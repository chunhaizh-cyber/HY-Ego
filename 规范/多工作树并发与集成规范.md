# 多工作树并发与集成规范

日期：2026-07-22

状态：长期非机器协作 / Git 工程治理入口；未登记于 `规范/规范目录.md`

## 1. 目标

本入口只定义 worktree 路径与分支、冻结基线、计划 blob、Git 身份、并行隔离、集成实施、主线发布占用的技术条件、技术失败和回收合同。任务当前生命周期唯一读取 `项目记忆/任务状态台账.md`；代理角色、派发 / S0、交接状态、消息协议和整树切换由 `AGENTS.md` 与 `规范/设计执行双窗口交互规范.md` 定义，不在本文件或工作树登记表建立第二套状态机。

本入口不是 `规范/规范目录.md` 登记的机器逻辑规范，不定义机器事实或代码语义；其约束力来自 `AGENTS.md` 对多工作树、并行批次、集成和回收的指定入口。凡进入上述工程 / 协作活动的角色都必须遵守本入口，但不得据此覆盖现行机器规范、扩大计划范围或取得新的角色权限。

多个已登记任务可以在独立 Git worktree 中并发推进，同时保持：

```text
一个 main 主线发布分支
一个 main 集成工作树
每个任务一个独立分支和 worktree
每个文件和结构所有权在同一批次内唯一
集成后统一验证再发布 origin/main
```

多 worktree 只提高互相独立或已经由预冻结接口合同解耦切片的吞吐量，不取消计划、未合同化真实依赖、接口合同复核、允许 / 禁止文件、验证和完成声明门禁。设计阶段必须先尝试冻结提供者—消费者合同、隔离计划范围和所有权，不能先假定串行，再把本可按合同并行的计划登记成完成依赖链。

## 2. 工作树类型

下表中的角色名称只绑定工作树用途，不重新定义角色权限：

| 类型 | 分支 | 用途 | 登记写入身份 |
| --- | --- | --- | --- |
| 主集成工作树 | `main` | 设计治理、并行批次登记、集成结果非强制快进与最终发布 | 同一时刻只允许设计角色或集成角色之一写入 |
| 任务工作树 | `codex/<队列号>-<主题>` | 执行一个已派发计划 | 只允许该计划登记的执行角色写入 |
| 集成暂存工作树 | `codex/integration-<批次号>` | 汇集同批次任务提交并运行集成验证 | 只允许该批次集成角色写入 |

主集成工作树不得由规范硬编码盘符或旧仓库名，必须以当前 Git 顶层、`项目记忆/并行工作树登记表.md` 和实际 `git worktree` 结果共同解析。任务工作树和集成暂存工作树必须位于主集成工作树目录之外，可以由 Codex 应用或 `git worktree` 建立，但必须在写入前登记绝对路径、分支和基线提交。

不得让两个 worktree 检出同一分支。不得在 detached HEAD 上实施计划。

### 2.1 worktree 不改变角色

worktree 只隔离文件和 Git 身份，不产生或改变代理角色、切片所有权、任务执行权和发布许可。顶层任务树、子智能体、唯一切片所有者、派发 S0、Git 串行、整树切换和跨顶层交接统一服从 `AGENTS.md` 与 `规范/设计执行双窗口交互规范.md`；本规范只核对其绑定的 worktree 技术身份。

## 3. worktree 身份核对证据

本节只规定 worktree 技术身份的核对证据，不是项目全局权威顺序，也不裁决任务生命周期：

```text
项目记忆/并行工作树登记表.md
项目记忆/任务状态台账.md 中的任务、计划版本、计划 blob 和 worktree 引用
目标计划中的允许范围与验证矩阵
批次冻结清单
执行窗口结构化回执，以及设计窗口据此形成的正式实施 / 冻结记录
Git 分支、提交和 worktree 实际状态
```

任务消息、聊天摘要、任务标题和显示状态不构成 Git 身份证据。登记表、冻结清单与 Git 事实冲突时停止写入，并按交互规范退回对应角色；不得用消息正文覆盖 Git 事实。

工作树登记只表达预留或所有权候选，不自动表示物理工作树已经创建、目标任务已经收到消息或角色已经接管。写入前的技术身份核对必须覆盖 Git worktree、路径、分支、HEAD、dirty state、冻结计划 blob 和冻结所有者；消息送达与生命周期迁移由任务状态台账和交互规范另行裁决。

## 4. 并行批次技术准入

真实依赖、计划拆分和角色派发由 `AGENTS.md` 与交互规范裁决。本规范只对已经获准并行的两个或以上计划执行下列技术准入；全部成立后才能建立并发 worktree：

1. 每个计划的未合同化正式依赖在同一冻结基线提交上均已满足；提供者实现尚未形成但接口已经按本节冻结时，不构成开发启动依赖。
2. 每个计划都有完整版本、允许文件、禁止文件、实际接口复核和验证命令。
3. 批次内允许写文件集合两两无交集。
4. 批次内结构所有者、业务写入口、公开 ABI 和阶段号登记所有权两两无冲突。
5. 提供者—消费者关系已经由同一版本合同冻结，或者确实不存在；合同至少固定模块 / 文件、DTO、签名或等价行为、ABI / 规则版本、所有权、返回 / 错误、提供计划、消费计划和退回点。未合同化关系或无法隔离的装配先后仍禁止并发。
6. 共享文件已明确归属唯一任务或后继集成切片。
7. 已固定集成顺序、集成验证矩阵、失败退回路径和集成角色。
8. 每个任务 worktree 已在登记表中取得唯一编号、绝对路径、任务分支、基线提交和窗口所有者；共同代码基线可以暂时不含提供者实现，但必须含相同版本的正式计划合同和批次冻结记录。

任一条件不成立时只允许串行推进，不能用“文件看起来不同”或聊天中的接口名称代替正式合同。合同并行只允许形成实现候选；提供者、消费者和真实装配尚未汇合前不得声明编译、能力或集成完成。

## 5. 共享文件边界

以下文件默认属于中央治理或集成共享面：

```text
AGENTS.md
规范/**
流程图/**
计划/**
项目记忆/**
海中鱼巣.vcxproj
海中鱼巣.vcxproj.filters
海中鱼巣/入口.cpp
中央自检运行器、统一阶段登记和其它计划明确列出的共享装配文件
```

中央治理文件只由主集成工作树的设计角色修改。代码共享面在一个并行批次内只能由一个任务拥有；若多个切片都需要修改工程、入口或中央运行器，设计角色必须：

```text
合并为同一串行切片
或把私有实现与共享接线拆成不同计划
或取消该批次的并行资格
```

集成角色不得借解决冲突发明业务逻辑、扩大接口或修改计划外语义。

## 6. 冻结清单

每个并行批次必须在登记表或独立批次记录中冻结：

| 字段 | 要求 |
| --- | --- |
| 批次号 | 全局唯一 |
| 基线提交 | 完整 Git 提交号，且是各任务分支共同祖先 |
| 计划身份 | 队列号、DQ、版本和路径 |
| 计划 blob | 冻结基线中不可变施工合同的 Git blob；派发和 S0 必须引用同一 blob |
| 任务状态引用 | 任务编号、计划版本和任务状态台账最新状态版本；不在本表复制当前生命周期 |
| 文件所有权 | 规范化绝对或仓库相对路径集合 |
| 结构所有权 | 仓库、服务、模块、DTO、阶段和公开接口 |
| 接口证据 | 执行前实际签名，或正式预冻结接口合同版本及提供者 / 消费者映射 |
| 验证矩阵 | 任务候选级与最终集成级命令、成功条件；允许把完整构建和运行验收后置，但不得省略合同 / 白名单 / 静态边界检查 |
| 集成顺序 | 无依赖时也要固定确定顺序 |
| 退回条件 | 具名冲突、漂移、失败和责任窗口 |

冻结计划 blob、冻结文件、冻结接口或批次依赖发生正式变更后，原批次技术身份自动失效；相关任务由任务状态台账按交互规范进入具名漂移和新版本 / 新身份重派，任务工作树不得自行合并新 `main` 继续执行，也不得用其它 Markdown 的显示状态解释跨越。

## 7. 任务工作树接管与分支结果

角色状态机、接管消息和 `分支完成待集成` 交接由交互规范唯一定义。本规范只规定任务工作树的 Git 条件：

1. 从登记的冻结基线创建或接管登记的唯一任务分支，不得 detached HEAD，不得与其它 worktree 共用分支。
2. 写入前核对绝对路径、分支、HEAD、共同基线、所有者和 dirty state 均与登记一致。
3. 实际差异必须落在冻结文件 / 结构所有权内；计划外文件和其它任务所有文件不得进入 index。
4. 完成计划规定的任务级验证后，只暂存本任务文件，提交并非强制推送到同名远端任务分支。
5. 分支结果、提交号、父链、文件范围、验证证据和残余 dirty state 通过结构化消息交给设计窗口；执行窗口不得写 Markdown，设计窗口核对 Git 后只形成必要的正式事实记录或设计修订，不发送后继执行指令。

任务分支提交只形成隔离分支结果，不自动启动集成，也不证明计划进入 `main`。

### 7.1 Windows 自主接管权限门禁

当 Codex 或其它执行平台把任务 worktree 与主仓库 Git 元数据放在不同目录时，执行角色首次自主接管前必须在 S0 实际验证当前环境能同时写入任务文件、任务分支索引和主仓库 `.git/worktrees` 所需元数据。受限环境不能完成上述 Git 写入，或执行过程中会被不可预期的交互审批阻断时，该技术身份不可执行；执行窗口返回具名环境 DRIFT，并继续检查其它无关计划，不等待设计窗口命令。

具体平台配置键、沙箱名称和审批选项属于自主接管时的运行环境事实，由执行窗口在结构化消息中报告，设计窗口只记录已发生事实；不在本长期入口固定。无论平台权限多高，都不改变项目授权：任务工作树的差异和 index 只能包含登记计划允许文件，推送目标只能是同名任务分支，并必须执行精确暂存、staged diff、禁止命令和完整验证门禁。删除、移动、强推、主工作树写入、`origin/main` 推送、未登记 merge / rebase / cherry-pick 一律禁止。

## 8. 漂移与失败

任务工作树发现接口漂移、文件交集、冻结基线失效或计划外逻辑错误时：

1. 停止扩大实施范围。
2. 撤回未验证代码草案，或按计划明确许可保留具名 WIP。
3. 在任务分支写正式复核断点。
4. 只提交并推送允许的断点和已许可产物。
5. 按交互规范回传技术失败类型和证据，由设计窗口在任务状态台账迁移为 `接口漂移待设计修订` 或 `验证失败待设计裁决`；本规范和工作树登记表不另建状态机。
6. 纯断点是否纳入 `main` 仍按已登记集成路径处理，再由交互规范完成设计角色同步。

任务窗口不得为追上 `main` 自行 merge、rebase、cherry-pick 或修改计划。

## 9. 集成协议

### 9.1 集成自主接管与身份核对

集成自主接管、消息边界和角色状态机由交互规范唯一定义。本规范读取已经存在的集成编号、绝对路径、分支、内容冻结基线、分支建立起点、中央治理尾白名单、输入提交、所有者、允许范围、验证矩阵和失败收口，只裁决这些字段对应的 Git 身份是否成立；设计窗口不得据此指挥集成角色开始或接受 S0。

集成技术身份必须机械区分三段，不得再把它们压成一个“集成起点”：

| 字段 | 唯一含义 |
| --- | --- |
| 内容冻结基线 | 冻结计划 blob、固定输入、父链、允许 / 禁止范围和验证矩阵的代码 / 合同基线；集成接管前不得变化 |
| 分支建立起点 | 新集成 worktree 与分支实际创建时的唯一 HEAD；负责路径、分支、reflog 和身份连续性，不要求后续中央设计提交形成后仍等于当前 main |
| 中央治理尾 | 从分支建立起点到取得主线发布占用前追加的连续 main 设计 / 事实记录提交；只允许登记的中央治理文件变化，不得含计划合同变化、代码、固定输入或其它业务差异 |

每个集成身份必须登记中央路由尾允许文件。未另行收窄时只允许 `项目记忆/任务状态台账.md`、`项目记忆/并行工作树登记表.md`、`项目记忆/窗口交互记录.md` 和 `项目记忆/当前状态.md`；规范、计划、队列、索引、代码、工程、自检、实施记录和任务输入均不属于默认路由尾。需要增加文件时必须在建立身份前逐项登记，不能在 S0 中临时解释。

集成窗口在创建或写入前必须核对：

```text
登记路径是否不存在，或已存在且精确属于同一集成编号和所有者。
登记分支是否不存在，或已存在且 HEAD、父链和分支建立起点完全一致。
worktree / index 是否 clean。
任务提交父链、远端分支和文件范围是否符合冻结清单。
WT-MAIN 是否 clean，且 main 与 origin/main 相等。
分支建立起点是否为当前 main 的祖先；两者之间是否只有登记的中央路由尾文件变化。
内容冻结基线、计划 blob、固定输入、父链、允许 / 禁止范围和验证矩阵在当前 main 是否仍与登记一致。
```

已存在路径 / 分支的身份、分支建立起点、所有者、父链或 reflog 任一不一致，或者当前 main 不是建立起点的后代、中央治理尾含未登记文件、内容冻结合同发生变化时，技术结论固定为“旧候选身份失效”。旧候选保持只读并登记为失效，不得 reset、delete、切换基线、覆盖分支或继续复用；该结论交由交互规范进入 `集成身份冲突待设计重建`，新的集成编号、路径和分支具备后由集成角色自主重新 S0 才可恢复写入。当前 main 仅比建立起点多出合法中央治理尾时不构成漂移，也不得要求为了恢复等值而改写历史或重复重建身份。

### 9.2 汇集、验证与主线发布占用

集成角色必须在独立集成暂存工作树中按冻结顺序处理任务提交：

```text
读取 origin/main、登记表、批次清单和任务分支提交
-> 验证每个任务提交范围、父提交和任务级证据
-> 集成窗口自行完成 S0，PASS 后原子取得唯一主线发布占用，不等待设计接受
-> 再次确认 main == origin/main 且仍是 S0 所见路由尾的合法后代
-> 在 clean 集成分支执行唯一一次 ff-only 同步到当前 origin/main，把同步后 HEAD 固定为集成执行起点
-> 按冻结顺序集成
-> 任一冲突立即停止，不作语义性手工消冲突
-> 运行批次规定的完整集成验证
-> 主线发布门禁成立且当前集成角色继续有效：让主集成工作树 main 非强制快进到集成结果，推送 origin/main，返回 `已集成待设计同步` 证据，由设计窗口迁移台账
-> 主线发布门禁不成立或当前集成角色必须停机：推送同名集成分支，固定提交与验证证据，释放主线发布占用，返回 `集成分支验证通过待主线发布` 证据，由设计窗口迁移台账
```

`集成分支验证通过待主线发布` 只证明固定集成分支已通过当次验证并已推送，不证明 main 已发布。新的独立集成 / 发布角色不得复用已停机角色，必须重新核对 origin/main 漂移、集成分支身份与父链、固定输入、工作树 clean 状态和本次发布验证矩阵，自行 S0 并取得唯一占用后才能非强制推进并推送 main；设计窗口只记录设计与事实，不发出开始命令。

集成工作树完成身份核对，且任务状态台账按交互规范迁移到 `集成中` 时，本批次主线发布占用才成立。取得占用后，集成角色只能先按本节把 clean 分支 ff-only 同步到当前 origin/main；同步结果是本轮集成执行起点，不改变内容冻结基线或固定输入。从此刻到成功发布或正式退回释放之间，技术冻结条件为：

```text
主集成工作树文件、index、main 和 origin/main 不得被其它写路径推进。
集成暂存工作树保持登记身份、分支建立起点证据、集成执行起点和唯一集成所有者。
其它任务 worktree 不得改变本批次冻结输入，也不得要求 main 前进。
紧急治理必须先按交互规范完成停机交接，再由本规范核对占用释放后的 Git 状态。
```

主集成工作树在集成发布期间必须保持 clean，且 `main == origin/main == 集成执行起点`，直到最终由同一集成角色非强制推进到已验证结果。远端 `main` 在占用后被其它路径推进、ff-only 同步失败、出现合并冲突、任务提交超范围、集成验证失败或主工作树不干净时停止发布，不得 force。集成发布或退回后必须在回执中明确主线发布占用已经释放。

集成失败不得通过修改测试、复用旧二进制、删除验收项或发明接口收口。失败报告必须区分任务分支代码修订与设计合同修订；具体角色退回和状态迁移按交互规范执行。

## 10. 验证层级

多 worktree 不自动降低验证等级：

1. 任务工作树执行目标计划规定的全部任务级验证。合同并行批次可以把任务级验证收束为合同签名、静态依赖、固定夹具、文件白名单和差异卫生，并明确不作编译 / 能力完成声明。
2. 集成暂存工作树至少重跑受影响配置的完整构建、完整自检、严格规范和批次交叉回归。
3. 涉及事务、所有权、运行期装配、持久化、恢复或并发语义的批次，集成层必须执行计划中最严格的连续运行与并发门禁。
4. 只有正式计划在派发前明确采用分级验证时，才能减少任务级重复验证或把完整构建 / 运行后置；最终集成必须补齐所有被后置的 Debug / Release、完整自检、真实提供者接线、恢复、并发和静态边界验证，执行或集成窗口不得临时继续降级。

## 11. 提交、推送与回收

| 工作树 | 自动提交目标 | 自动推送目标 |
| --- | --- | --- |
| 主集成工作树的设计角色 | `main` | `origin/main` |
| 任务工作树执行角色 | 当前 `codex/<队列号>-<主题>` | 同名远端任务分支 |
| 集成暂存工作树 | 当前 `codex/integration-<批次号>` | 默认不单独发布；验证完成后由主集成工作树快进并发布 `origin/main` |

所有推送均禁止 force 和 force-with-lease。任务分支不得直接推送 `origin/main`。

只有在 `origin/main` 已包含集成结果、任务状态台账已完成设计同步、目标 worktree clean 且登记表技术状态标记 `可回收` 后，才能回收任务 worktree 和本地任务分支。存在 dirty state、未集成提交、未读取断点或远端独有提交时禁止删除。远端任务分支删除不属于自动收尾，必须由明确的回收动作授权。

## 12. 完成声明边界

任务分支提交和推送只证明任务产物已发布到隔离分支。集成验证和 `origin/main` 推送只证明该批次已进入主线。角色回执、设计归档和最终完成声明由交互规范裁决。

不得把允许多 worktree 宣称为：

```text
允许多个窗口写同一文件
允许多个业务所有者并存
允许绕过依赖并行
允许执行窗口修改中央治理
允许任务分支直接成为生产权威
允许降低验证或跳过集成回归
允许同一顶层任务树通过子智能体同时取得多个写角色
```

# 多工作树并发与集成规范

日期：2026-07-21

状态：已确认

## 1. 目标

允许多个执行窗口在独立 Git worktree 中并发推进已经登记的计划，同时保持：

```text
一个 main 权威分支
一个 main 集成工作树
一个中央治理写入方
每个任务一个独立分支和 worktree
每个文件和结构所有权在同一批次内唯一
集成后统一验证再发布 origin/main
```

多 worktree 只提高互相独立切片的吞吐量，不取消计划、真实依赖、接口复核、允许 / 禁止文件、验证和完成声明门禁。设计阶段必须先尝试隔离计划范围和所有权，不能先假定串行，再把本可独立的计划登记成依赖链。

## 2. 工作树类型

| 类型 | 分支 | 用途 | 写入权 |
| --- | --- | --- | --- |
| 主集成工作树 | `main` | 设计治理、并行批次登记、集成验证、最终发布 | 同一时刻只允许设计角色或集成角色之一写入 |
| 任务工作树 | `codex/<队列号>-<主题>` | 执行一个已派发计划 | 只允许该计划登记的执行角色写入 |
| 集成暂存工作树 | `codex/integration-<批次号>` | 汇集同批次任务提交并运行集成验证 | 只允许该批次集成角色写入 |

主集成工作树固定为 `D:\海中鱼巢`；`海中鱼巣` 只作为仓库内源码目录名。任务工作树和集成暂存工作树必须位于主集成工作树目录之外，可以由 Codex 应用或 `git worktree` 建立，但必须在写入前登记绝对路径、分支和基线提交。

不得让两个 worktree 检出同一分支。不得在 detached HEAD 上实施计划。

## 3. 权威状态

以下材料共同形成多 worktree 权威状态：

```text
计划/计划索引.md
项目记忆/Codex任务队列.md
目标计划
项目记忆/并行工作树登记表.md
项目记忆/窗口交互记录.md
任务分支正式实施记录或复核断点
Git 分支、提交和 worktree 实际状态
```

任务消息和聊天摘要只作唤醒提示。登记表与 Git 事实冲突时停止写入，由设计角色复核；不得用消息正文覆盖 Git 事实。

工作树登记只表达预留或所有权候选，不自动表示物理工作树已经创建、目标任务已经收到消息或角色已经接管。设计角色必须在派发前同时核对 Git worktree、路径、分支、HEAD、dirty state 和任务状态。

## 4. 并行批次准入

设计角色在登记依赖或并行批次前，必须先完成“依赖必要性判定 + 可隔离性拆分”：

1. 只有计划必须消费另一计划尚未形成的正式产物或公开接口，或者存在无法通过拆出共享接线计划消除的结构 / 装配所有权先后时，才构成真实依赖。
2. 同一主题、同一总计划、同一来源、生成顺序、文档顺序、目录位置或验收顺序均不自动构成依赖。
3. 没有真实依赖的计划必须独立登记，并优先按业务对象、文件集合、结构所有权、公开接口和装配面拆成可并发切片。
4. 只有共享工程、入口、统一运行器或其它装配文件造成交集时，优先把私有实现与共享接线拆成不同计划；共享接线由唯一所有者串行承接，不得因此把全部私有实现串成依赖链。
5. 依赖门控必须具名记录依赖产物、消费接口、无法隔离原因和解除条件；接口漂移只退回受影响计划，不得连带冻结无关计划。
6. 后续计划可以提前生成，但不得仅因生成时间晚于其它计划而登记门控。设计角色决定暂不并行时，必须具名记录资源、所有权或验证原因。

设计角色只有在下列条件全部成立时，才能把两个或以上计划登记为同一预授权并行批次：

1. 每个计划的正式依赖在同一冻结基线提交上均已满足。
2. 每个计划都有完整版本、允许文件、禁止文件、实际接口复核和验证命令。
3. 批次内允许写文件集合两两无交集。
4. 批次内结构所有者、业务写入口、公开 ABI 和阶段号登记所有权两两无冲突。
5. 计划之间不存在提供者到消费者、初始化先后、发布先后、迁移先后或恢复先后依赖。
6. 共享文件已明确归属唯一任务或后继集成切片。
7. 已固定集成顺序、集成验证矩阵、失败退回路径和集成角色。
8. 每个任务 worktree 已在登记表中取得唯一编号、绝对路径、任务分支、基线提交和窗口所有者。

任一条件不成立时只允许串行推进，不能用“文件看起来不同”代替结构和接口无依赖证明。

## 5. 共享文件边界

以下文件默认属于中央治理或集成共享面：

```text
AGENTS.md
规范/**
流程图/**
计划/**
项目记忆/**
海中鱼巣.vcxproj
海中鱼巣.vcxproj.filters
海中鱼巣/入口.cpp
中央自检运行器、统一阶段登记和其它计划明确列出的共享装配文件
```

中央治理文件只由主集成工作树的设计角色修改。代码共享面在一个并行批次内只能由一个任务拥有；若多个切片都需要修改工程、入口或中央运行器，设计角色必须：

```text
合并为同一串行切片
或把私有实现与共享接线拆成不同计划
或取消该批次的并行资格
```

集成角色不得借解决冲突发明业务逻辑、扩大接口或修改计划外语义。

## 6. 冻结清单

每个并行批次必须在登记表或独立批次记录中冻结：

| 字段 | 要求 |
| --- | --- |
| 批次号 | 全局唯一 |
| 基线提交 | 完整 Git 提交号，且是各任务分支共同祖先 |
| 计划身份 | 队列号、DQ、版本和路径 |
| 文件所有权 | 规范化绝对或仓库相对路径集合 |
| 结构所有权 | 仓库、服务、模块、DTO、阶段和公开接口 |
| 接口证据 | 执行前实际签名或只读事实记录 |
| 验证矩阵 | 任务级与集成级命令、成功条件 |
| 集成顺序 | 无依赖时也要固定确定顺序 |
| 退回条件 | 具名冲突、漂移、失败和责任窗口 |

设计角色修改冻结文件、冻结接口或批次依赖后，原批次自动失效；必须先停止相关任务并发布新版本，不能让任务工作树自行合并新 `main` 继续执行。

## 7. 任务工作树状态机

```text
已预留
-> 已创建待接管
-> 执行中
-> 分支完成待集成
   或 接口漂移待设计
   或 验证失败待裁决
-> 已集成待设计同步
-> 已完成 / 已归档
-> 可回收
-> 已回收
```

任务执行窗口必须从登记的冻结基线和任务分支接管。它只能修改计划允许的代码、工程、自检和本次执行专属实施记录 / 复核断点；不得修改中央治理文件或其它任务所有文件。

任务分支完成后必须运行计划要求的任务级验证，只暂存本任务文件，提交并非强制推送到同名远端任务分支。此时只能声明“分支完成待集成”，不得声明计划已经进入 `main` 或正式完成。

任务分支完成不自动启动集成。设计角色必须核对任务提交并发布独立集成派发记录，再实际唤醒唯一集成任务；只预留路径、分支或目标窗口时，状态仍是 `已预留待集成派发`。

### 7.1 Windows 无人值守权限

当前 Codex 桌面在 Windows 上的任务 worktree 与主仓库 Git 元数据分处不同目录。默认沙箱可以完成任务文件读写，但不能稳定写入主仓库 `.git/worktrees`；`auto_review` 也不能保证不向用户显示审批弹窗。因此正式无人值守任务使用：

```toml
sandbox_mode = "danger-full-access"
approval_policy = "never"
```

该配置只解决平台文件系统与审批边界，不改变项目授权。任务执行窗口仍只能修改登记计划的允许文件，只能推送同名任务分支，并必须执行精确暂存、staged diff、禁止命令和完整验证门禁。删除、移动、强推、主工作树写入、`origin/main` 推送、未登记 merge / rebase / cherry-pick 一律禁止。

## 8. 漂移与失败

任务窗口发现接口漂移、文件交集、冻结基线失效或计划外逻辑错误时：

1. 停止扩大实施范围。
2. 撤回未验证代码草案，或按计划明确许可保留具名 WIP。
3. 在任务分支写正式复核断点。
4. 只提交并推送允许的断点和已许可产物。
5. 状态改为 `接口漂移待设计` 或 `验证失败待裁决`。
6. 由集成角色把纯断点提交纳入 `main`，再由设计角色修订中央路由。

任务窗口不得为追上 `main` 自行 merge、rebase、cherry-pick 或修改计划。

## 9. 集成协议

### 9.1 集成派发与身份核对

集成工作树采用独立状态机：

```text
已预留待集成派发
-> 已派发待集成回执
-> 集成窗口核对路径 / 分支 / 起点 / 所有者
-> 集成中
-> 已集成待设计同步
   或 集成身份冲突待设计重建
   或 集成失败退回任务
```

设计角色必须先提交并推送集成编号、绝对路径、分支、起点、输入提交、目标任务、允许写入、验证矩阵和失败收口，再发送任务消息。只有消息实际送达且目标任务已开始重读正式状态，才能标为 `已派发待集成回执`；预留不等于派发。

集成窗口在创建或写入前必须核对：

```text
登记路径是否不存在，或已存在且精确属于同一集成编号和所有者。
登记分支是否不存在，或已存在且 HEAD、父链和起点完全一致。
worktree / index 是否 clean。
任务提交父链、远端分支和文件范围是否符合冻结清单。
WT-MAIN 是否 clean，main 是否与 origin/main 等于登记起点。
```

已存在路径 / 分支的身份、起点、所有者、父链或 reflog 任一不一致时，必须进入 `集成身份冲突待设计重建`。旧候选保持只读并登记为失效；集成窗口不得 reset、delete、切换基线、覆盖分支或继续复用。设计角色必须另立新的集成编号、路径和分支后重新派发。

### 9.2 汇集、验证与主线发布占用

集成角色必须在独立集成暂存工作树中按冻结顺序处理任务提交：

```text
读取 origin/main、登记表、批次清单和任务分支提交
-> 验证每个任务提交范围、父提交和任务级证据
-> 按冻结顺序集成
-> 任一冲突立即停止，不作语义性手工消冲突
-> 运行批次规定的完整集成验证
-> 验证通过后让主集成工作树 main 非强制快进到集成结果
-> 推送 origin/main
-> 登记已集成待设计同步
```

集成窗口完成身份核对并进入 `集成中` 时，同时取得本批次主线发布占用。从此刻到成功发布或正式退回释放之间：

```text
设计窗口不得修改 WT-MAIN 文件、提交或推送 origin/main。
设计窗口可以继续只读审查和在对话中形成未落盘草案。
其它任务 worktree 可以按已冻结批次继续执行，但不得要求 main 前进。
需要紧急设计治理时，必须先让集成窗口停止、提交允许的退回记录并释放发布占用。
```

主集成工作树在集成发布期间必须保持 clean，且 `main == origin/main` 的起点与批次集成起点一致。远端 `main` 已前进、出现合并冲突、任务提交超范围、集成验证失败或主工作树不干净时停止发布，不得 force。集成发布或退回后必须在回执中明确主线发布占用已经释放。

集成失败不得通过修改测试、复用旧二进制、删除验收项或在集成窗口发明接口收口。需要代码修订时退回对应任务分支；需要合同修订时退回设计角色。

## 10. 验证层级

多 worktree 不自动降低验证等级：

1. 任务工作树执行目标计划规定的全部任务级验证。
2. 集成暂存工作树至少重跑受影响配置的完整构建、完整自检、严格规范和批次交叉回归。
3. 涉及事务、所有权、运行期装配、持久化、恢复或并发语义的批次，集成层必须执行计划中最严格的连续运行与并发门禁。
4. 只有正式规则或目标计划明确采用分级验证时，才能减少任务级重复验证；执行或集成窗口不得临时降级。

## 11. 提交、推送与回收

| 工作树 | 自动提交目标 | 自动推送目标 |
| --- | --- | --- |
| 主集成工作树的设计角色 | `main` | `origin/main` |
| 任务工作树执行角色 | 当前 `codex/<队列号>-<主题>` | 同名远端任务分支 |
| 集成暂存工作树 | 当前 `codex/integration-<批次号>` | 默认不单独发布；验证完成后由主集成工作树快进并发布 `origin/main` |

所有推送均禁止 force 和 force-with-lease。任务分支不得直接推送 `origin/main`。

只有在 `origin/main` 已包含集成结果、设计角色已同步完成状态、目标 worktree clean 且登记表标记 `可回收` 后，才能回收任务 worktree 和本地任务分支。存在 dirty state、未集成提交、未读取断点或远端独有提交时禁止删除。远端任务分支删除不属于自动收尾，必须由明确的回收动作授权。

## 12. 完成声明边界

任务分支提交和推送只证明任务产物已发布到隔离分支。集成验证和 `origin/main` 推送只证明该批次已进入主线。设计角色完成计划归档后，才可按计划验收等级声明该切片完成。

不得把允许多 worktree 宣称为：

```text
允许多个窗口写同一文件
允许多个业务所有者并存
允许绕过依赖并行
允许执行窗口修改中央治理
允许任务分支直接成为生产权威
允许降低验证或跳过集成回归
```

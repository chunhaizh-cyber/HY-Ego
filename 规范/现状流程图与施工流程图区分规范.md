# 现状流程图与施工流程图区分规范

编号：FLOW-KIND-001
日期：2026-07-11
状态：独立计划执行产物 / 未加入计划索引 / 未改代码

## 1. 定位

本规范用于区分两类流程图：

```text
现状流程图：按当前代码事实、实施记录和验证输出展开，回答“现在代码实际怎么走”。
施工流程图：按计划和详细设计给出拟施工路径，回答“下一段准备怎么实现”。
```

两类流程图不得互相替代。施工流程图不能冒充当前代码事实；现状流程图也不能自动构成代码实施许可。

## 2. 现状流程图硬规则

现状流程图必须足够细，能够支撑后续流程图校正和代码纠偏。

现状流程图必须同时提供：

```text
1. Mermaid 路径图。
2. 逐行代码映射表。
3. 偏差清单。
4. 非成功返回二分表。
```

逐行代码映射表必须覆盖对应函数或代码片段内每一个非空、非纯格式的代码行。纯括号、空行和只作排版的行可以并入相邻节点，但必须在覆盖范围中体现。文件头部规则注释、人读说明注释和预编译宏也必须说明是否影响运行路径。

## 3. 逐行映射表字段

现状流程图必须使用以下字段：

| 字段 | 含义 |
| --- | --- |
| 文件 | 当前代码文件路径 |
| 行号 | 当前工作区行号；若工作区未提交，必须注明基于脏工作区 |
| 代码行为 | 用中文概括该行实际行为 |
| 流程图节点 | Mermaid 节点编号 |
| 行为类型 | 入口拒绝 / 读取 / 写入 / 读回 / 纯计算 / 调用 / 返回 / 输出 / 诊断 / 预编译 |
| 结构变化 | 是否写节点、主信息、关系、索引、特征、状态、动态、因果、需求、任务或方法 |
| 权威性 | 权威事实 / 非权威材料 / 人读诊断 / 临时局部变量 |
| 非成功口径 | 不适用 / 逻辑内返回 / 追根因解决 |
| 纠偏建议 | 与施工图、详细设计或规则不一致时写明后续纠偏方向 |

## 4. 现状流程图元数据

每份现状流程图开头必须写：

```text
图类型：现状流程图
代码版本：提交号；若有未提交改动，写明基于当前工作区
覆盖文件：
覆盖函数：
逐行映射表：
依据实施记录：
验证输出：
不得作为施工许可：
不得宣称：
```

若工作区存在未提交改动，现状流程图只能声明“当前工作区事实”，不得声明其已成为云端稳定事实。

## 5. 施工流程图元数据

每份施工流程图开头必须写：

```text
图类型：施工流程图
绑定计划：
绑定详细设计：
允许文件：
禁止文件：
预期结构变化：
执行前复核：
验证方式：
不得宣称：
```

施工流程图必须标明计划范围，不得把目标路径说成当前代码事实。

## 6. 非成功返回二分口径

现状流程图和施工流程图都必须继续遵守 2026-07-09 口径：

```text
追根因解决：
前置条件已经满足，并进入创建、绑定、写关系、写状态、记录动态、结算、读回或结构承载后，结果不符合内部预期。

逻辑内返回：
设计内允许分支、候选为空、请求材料返回、人读材料返回或入口拒绝材料返回，且结构不变化。
```

如果同一判断点混合了“未形成材料的逻辑内返回”和“已声明写入后读回不及预期”，必须拆成两个节点，不得用一个“失败 / 否”分支覆盖。

## 7. 校正链路

已执行计划的流程图校正必须按以下链路推进：

```text
当前代码
-> 逐行代码映射表
-> 现状流程图
-> 与施工流程图 / 详细设计 / 实施记录比对
-> 偏差清单
-> 代码纠偏施工计划
-> 执行窗口按计划纠偏代码
```

不得跳过现状流程图，直接用施工流程图纠偏代码。

## 8. 完成声明边界

本规范只能证明：

```text
现状流程图和施工流程图区分口径已形成。
逐行代码映射表成为现状流程图的必要产物。
后续流程图校正和代码纠偏必须先区分当前事实与施工目标。
```

不能证明：

```text
任何既有流程图已经完成逐行校正。
代码已经纠偏。
真实闭环、自我循环、自我苏醒或旧能力迁移已经完成。
```

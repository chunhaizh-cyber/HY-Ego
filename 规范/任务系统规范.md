# 任务系统规范

编号：任务系统-001
日期：2026-07-06
状态：已确认项目规范 / 第一轮任务系统承接壳方案 / #223 已按 JY-305 实现任务方法选择记录权威承载

## 1. 定位

任务系统负责把需求承接为可筹办、可执行、可结算的过程壳。

任务不是需求目标本身，也不是方法动作本身。需求表达要达到的目标状态，任务表达本轮如何推进这个目标，方法表达通过什么能力或动作产生状态变化。

本规范只建立任务系统第一轮边界和 MI-T2、MI-T3、MI-T4 的新项目承载口径。它不迁移旧 `任务主信息类` 字段，不恢复旧任务步骤 / 结果树作为主线，不声明任务系统代码已完成。

## 4. 任务节点和任务角色

底层节点类型第一轮不新增：

```text
节点类型::任务
```

任务节点种类不得作为主信息字段、I64 字段或文本字段迁移。

第一轮任务节点角色：

| 角色 | 含义 | 第一轮口径 |
| --- | --- | --- |
| 任务承接节点 | 承接一个需求并形成任务过程壳 | 第一轮主线 |
| 任务管理节点 | 组织多个子任务或等待多个子需求 | 暂缓，需需求树 / 子任务规范确认 |
| 任务执行节点 | 直接调用方法推进目标状态 | 暂缓到任务筹办 / 方法系统设计 |
| 缺口补全任务 | 承接筹办缺口或学习缺口 | 暂缓到需求树生长和方法系统设计 |
| 步骤节点 / 结果节点 | 旧任务树兼容节点 | 不进入第一轮主线，不迁移为新主信息字段 |

若后续需要区分任务管理、任务执行或缺口补全角色，应通过任务服务写入任务角色状态、任务角色关系或后续确认的任务角色节点，不得扩展为自由主信息字段。

## 5. 任务和需求

任务必须来源于有效需求。

第一轮任务承接规则：

```text
一个任务只承接一个需求。
一个需求可以有多个历史任务或候选任务，但当前活动任务关系必须由需求 / 任务规范确认唯一性。
任务目标状态来自对应需求的目标状态。
任务目标不得脱离需求自行扩展成多目标包。
复合目标必须先在需求树拆成子需求，再由任务逐个承接。
```

任务服务按需求创建任务时，必须在入口预读并确认该需求唯一目标状态：

```text
需求无目标状态 -> 拒绝。
需求目标状态不唯一 -> 拒绝。
需求目标状态不是状态节点 -> 拒绝。
任务目标状态关系写入失败 -> 视为内部逻辑错误，进入容错诊断和根因排查路径。
```

任务完成判断必须围绕当前任务目标状态和实际结果状态比较。方法执行成功、日志显示成功、动作返回成功或任务状态字符串不得直接证明任务完成。

## 6. 任务虚拟存在

任务虚拟存在是任务运行特征和过程账的承载存在。

任务虚拟存在规则：

```text
必须由存在服务创建。
来源必须是有效任务节点。
不得作为独立任务节点。
不得被多个任务共享为同一个任务虚拟存在。
不得由任务服务绕过存在服务直接创建存在节点。
不得把任务虚拟存在句柄写回主信息裸字段作为权威。
```

任务自身需要特征、状态、运行材料、候选方法材料或执行回执材料时，应通过任务虚拟存在和对应领域服务承载。任务节点本体不直接挂特征值，任务服务也不得直接访问特征值服务。

## 7. 任务场景

任务场景表达任务运行发生或适用的场景。

第一轮任务场景来源：

```text
调用方明确传入的有效场景。
来源需求已确认的场景。
任务虚拟存在所在场景。
```

任务场景不得由控制台输出、日志文本、显示标题或字符串标签推断。任务服务需要记录任务状态或任务动态时，必须经场景服务、状态服务或动态服务。

## 8. 任务生命周期和时间

创建时间、启动时间和完成时间不迁移为主信息字段。

生命周期时间的机器事实由任务生命周期实例状态的发生时间戳承载：

| 生命周期事件 | 结构承载 | 时间含义 |
| --- | --- | --- |
| 任务创建 | 任务创建状态节点 | 任务承接需求并形成任务节点的发生时间 |
| 任务启动 | 任务启动 / 执行中状态节点 | 任务进入执行或调用方法前的发生时间 |
| 任务完成 | 任务完成状态节点 | 任务目标状态已达成并被任务服务记录的发生时间 |

事件日志可以记录同类时间，但只做人读审计，不作为任务生命周期权威事实。没有完成状态节点不得用 `0`、空字符串或日志缺失解释为完成或未完成事实。

### 8.1 生命周期迁移幂等与证据

第一版合法迁移的幂等编号随目标实例状态由状态服务写入固定主信息槽位，任务服务不得直接解释槽位。完整迁移证据使用 `任务生命周期证据 = 15` 专用关系，源为目标实例状态，顺序号 40-46 表达固定角色。

同源同角色只能有一条有效证据关系；同一目标可以承担多个不同角色。普通引用的唯一性不得因任务迁移证据而改变。精确重复必须从完整生命周期历史逐字段复核前驱关系、目标阶段、发生时间和全部证据角色，不得由时间戳、生命周期版本、摘要、日志或 bool 单独裁决。

### 8.2 任务方法选择记录

任务方法选择是任务服务所有的独立机器事实，不得只由 `引用(任务 -> 方法, 顺序号 30)` 推断。第一版权威承载固定为：

```text
节点类型::任务方法选择记录 = 14
关系类型::任务方法选择 = 16
关系方向：任务 -> 任务方法选择记录
关系顺序号：30
同一任务有效关系数量：恰好零或一
```

选择记录节点的主信息值容器保存版本化、不可变、可完整重建的规范元组。元组必须覆盖幂等编号、任务与方法、双向承接授权、所属筹办生命周期关系 / 状态 / 版本、排序规则、召回请求、索引发布版本、来源候选数量、重复输入数量、完整有序候选组及每项登记 / 生命周期 / 规格 / 条件 / 上下文、建议状态、动作入口及来源关系 / 版本。`uint64_t` 必须按高 / 低 32 位无损保存，变长项必须有数量和定长项布局；摘要、时间戳、内存表、容器地址和关系自身版本均不得替代完整元组。

第一版不定义独立批次身份编号。完整排序批次就是方法召回侧现有排序材料和建议材料的全部值式字段；其调用期生命周期止于适配请求。记录保存的是当时快照，不是持续当前的索引事实。

方法召回索引、索引发布版本、候选数量和排序结果均为非权威材料。方法召回侧必须在适配前调用正式来源复核入口；任务服务不得持有或重读方法召回索引，只在独占许可内复核任务、授权、生命周期和请求携带的权威登记 / 生命周期 / 规格 / 条件 / 上下文 / 动作来源。索引发布版本单独变化不使选择失效。

任务服务是选择记录主信息的唯一写入方和解释方。方法召回服务只负责把刚完成当前性复核的完整建议适配为任务服务自有扁平请求；方法服务、线程、日志、显示和控制面板不得直接解释记录槽位。

顺序号 30 `引用(任务 -> 方法)` 保留为当前方法投影，必须与有效关系 16 指向的记录逐字段一致，但投影单独存在不构成选择授权。方法动作桥必须同时复核选择记录关系 / 节点、投影关系和所属生命周期身份。

幂等编号在同一任务完整选择审计中唯一。同键同完整规范元组返回原记录；同键异义和同生命周期已有其它选择写前拒绝。正式 `待重筹办 -> 筹办中` 时，任务服务在同一结构独占许可内使旧关系 16 和顺序号 30 投影失效；记录节点、主信息和关系审计保留。读取路径不得惰性清理或修补旧选择。

## 9. 目标状态和实际结果状态

任务目标状态来自需求目标状态。

任务实际结果状态来自任务执行、方法结果、状态服务或特征服务读出的结构结果。

第一轮规则：

```text
目标状态必须是状态节点。
实际结果状态必须是状态节点。
任务可以引用目标状态和实际结果状态，但不得复制目标状态为任务私有目标字段。
按需求创建任务必须写入任务目标状态关系，该关系目标必须来自需求目标状态。
实际结果状态可以随执行多次变化，读取“最新实际结果状态”的规则必须由任务服务定义。
任务完成判断必须比较任务目标状态和实际结果状态。
```

方法可以产生多个结果状态，但当前任务完成只能使用匹配当前任务目标的结果状态。附带结果、禁止结果和治理风险不得冒充当前任务目标达成。

## 10. 任务服务依赖

任务服务是高级信息服务，只允许直接依赖：

```text
存在服务
场景服务
特征服务
二次特征服务
状态服务
动态服务
因果服务
```

任务服务不得直接依赖特征值服务。需要读取目标特征、当前特征状态材料或方法执行后的特征状态材料时，必须经特征服务。

# 1200 模块分层规则与改造说明

## 1. 目标

本规则用于解决并持续防止以下问题：

- 模块之间出现不受控嵌套依赖，导致编译顺序不稳定。
- 高层模块反向依赖低层设计之外的模块，造成结构失真。
- 模块环路导致层级无法稳定推导，后续改动风险扩大。
- 历史上出现的“模块文件异常拼接/模块污染”难以及时发现。

本文件不以“举例模块名”作为真值，而是以 **规则 + 自动校验结果** 作为真值。

## 2. 分层规则（强约束）

### R1. 唯一模块声明

- 每个 `.ixx` 仅允许一个 `export module XXX;`。
- 同名模块只能在一个文件中定义。

### R2. 只允许向下依赖

- 设 `Level(A)` 为模块 A 的层级数字。
- 若 `A import B`（且 B 为本工程自定义模块），必须满足：
  - `Level(A) > Level(B)`

### R3. 同层与上层依赖禁止

- 不允许 `Level(A) == Level(B)` 的同层依赖。
- 不允许 `Level(A) < Level(B)` 的“低层依赖高层”。

### R4. 1级模块零自定义依赖

- `Level == 1` 的模块不得 `import` 任何自定义模块。
- 1级模块只允许标准库 `import <...>` 或本文件内部实现。

### R5. 环路禁止

- 自定义模块依赖图不得出现强连通环（SCC size > 1）。

### R6. 基线即契约

- 规则检查以 `规范/模块层级基线.json` 为项目当前契约。
- 新增模块必须先入基线，再允许合入。

### R7. 数值载体收口

- 领域结构、运行时控制值、任务/方法/需求字段中的单值标量统一使用 `I64`
- `VecIU64` 是唯一允许跨模块暴露的无符号 64 位业务载体，只用于位签名、轮廓块、体素块、打包证据块等证据向量
- `std::uint64_t` 若因哈希、位运算、设备 SDK 或协议适配而存在，只允许停留在局部实现，不得继续抽象成 `U64` 业务字段或结构成员

## 5. 使用方式

### 5.1 仅检查

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/check_module_layers.ps1 -RepoRoot D:\鱼巢
```

### 5.2 重新生成基线（仅在明确进行架构调整后执行）

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/check_module_layers.ps1 -RepoRoot D:\鱼巢 -UpdateBaseline
```

## 6. 后续改造建议（按风险从低到高）

1. 拆分“通用函数模块”中的领域类型相关函数，形成更纯的底层工具模块。
2. 将高层业务模块中的“纯数据结构/纯算法”下沉，减少层级高度。
3. 对 10 级以上模块（任务/自我/语言链）逐步做依赖收敛，避免继续向上抬层。

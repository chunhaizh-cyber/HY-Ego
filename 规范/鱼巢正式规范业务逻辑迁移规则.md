# 鱼巢正式规范业务逻辑迁移规则

日期：2026-07-20

状态：正式规范 v0.2 / 87 项正式来源已完成规范层迁移 / 详细设计不在迁移范围

## 1. 定位

本规范固定 `鱼巢 -> 海中鱼巣` 的迁移权力关系：

```text
D:\鱼巢\规范\规范目录.md 已登记的当前正式规范
-> 业务逻辑必须继承
-> 只把旧数据结构、字段和承载入口映射为海中鱼巣结构
-> 再生成流程图、详细设计、计划和代码实施切片
```

“全量重构”表示重建数据结构、服务边界、事务入口和代码组织，不表示可以重新发明鱼巢已经正式确认的业务逻辑。

## 2. 双轴权力关系

### 2.1 业务逻辑轴

业务流程、角色权责、处理顺序、并发隔离、选择条件、状态迁移、非成功返回、失败语义、完成判定和结算语义，按以下来源读取：

```text
用户本轮及后继明确硬裁决
-> D:\鱼巢\规范\规范目录.md
-> 0050 项目通用结构与禁止性规则总纲
-> 对应根规范
-> 对应正式子规范
-> 0300 已登记冲突的正式处理结果
```

同级正式子规范出现相反条款且 `0300` 未登记裁决时，海中鱼巣不得自行挑选、拼接或按当前代码便利性解释；必须登记为源规范冲突，并冻结受影响分支。

### 2.2 结构与工程轴

数据结构、身份类型、字段承载、仓库、关系、索引、领域服务、事务边界、文件组织、模块命名和验证方式按以下来源裁决：

```text
用户明确硬裁决
-> 海中鱼巣 AGENTS.md
-> 海中鱼巣正式规范
-> 已迁移业务逻辑的结构映射
-> 当前代码事实
```

结构与工程轴只能决定“用海中鱼巣什么结构承载”，不能改变业务逻辑轴中的先后顺序、分支条件、并发键、状态语义或结果分类。

## 3. 正式来源与排除来源

只有 `D:\鱼巢\规范\规范目录.md` 第 3 章登记的当前正式规范属于本规则的业务逻辑母本。

以下材料不具有反向覆盖权：

```text
D:\鱼巢\规范\旧规范\
D:\鱼巢\规范\详细设计\待完成\
鱼巢旧代码字段、旧函数体和旧调用链
鱼巢说明书、计划、实施记录、日志、控制面板和数据库投影
海中鱼巣既有流程图、详细设计、计划、代码和测试便利性
```

旧代码和旧说明可以证明调用事实、输入输出、边界和缺口，但不能改变正式规范业务逻辑，也不能直接形成代码实施许可。

## 4. 允许改变与禁止改变

### 4.1 允许改变

迁入海中鱼巣时，只允许为结构适配进行下列变化：

```text
旧主信息字段 -> 节点、关系、状态、动态、索引和稳定句柄
旧裸指针 / 节点编号 -> 稳定句柄和版本化身份
旧世界树直写 -> 领域服务、数据操作层和结构写入会话
旧虚拟存在字段 -> 本项目存在节点及其关系、状态和特征承载
旧文本 / 枚举镜像 -> 本项目正式结构或非权威人读投影
旧文件与类边界 -> 本项目中文命名模块、仓库和服务边界
```

每项结构替换必须列明用途、写入方、读取方、生命周期、当前实现事实和验证方式。

### 4.2 禁止改变

不得以“新结构更方便”“当前代码已有入口”或“并发后可以幂等收敛”为理由改变：

```text
谁负责筹办、执行、状态推进和需求树修改
先判断什么、后召回什么、何时允许选择或执行
候选集填充和选择顺序
同一任务与不同任务的并发隔离语义
等待、缺口、拒绝、内部不一致和追根因的分类
任务完成、父任务回流和需求结算的判定依据
线程、日志、显示或测试代码的权力边界
```

若结构映射无法保持上述逻辑，必须停止在规范迁移阶段，不得用详细设计或代码反向改义。

## 5. 固定迁移顺序

每个业务主题必须依次完成：

```text
登记规范目录中的源文件与版本
-> 抽取不依赖旧字段名的业务规则
-> 审查源规范内部冲突
-> 对无冲突规则形成海中鱼巣结构映射
-> 写入海中鱼巣正式规范
-> 生成或修订流程图
-> 生成或修订详细设计
-> 生成或修订计划并登记队列
-> 执行窗口复核实际接口后实施
```

规范未迁移、结构映射缺项、源冲突未门控或下游材料仍沿用旧改义时，不得进入代码实施。

## 6. 迁移产物最低要求

每个迁移主题至少形成：

| 产物 | 必须回答 |
| --- | --- |
| 来源清单 | 读取了哪些正式规范，是否在规范目录登记 |
| 业务规则表 | 原流程、顺序、分支、状态和失败语义是什么 |
| 源冲突表 | 哪些正式条款相反，受影响分支如何冻结 |
| 结构映射表 | 业务概念如何映射到本项目结构、写入方和读取方 |
| 禁止改义表 | 哪些既有本项目设计、代码或测试需要撤回或重审 |
| 验收表 | 如何证明只换结构而未改业务逻辑 |

“已读取源规范”不等于“规范已经迁移”；只有无冲突规则已进入本项目正式规范、冲突分支已显式门控，才可登记该主题的规范迁移第一版。

## 7. 任务筹办并发补充裁决

用户已经明确任务筹办的并发隔离规则：

```text
不同任务可以并发筹办。
同一任务同一时刻只能有一个有效筹办执行权。
未取得者必须在方法召回前退出。
两条路径同时进入同一任务召回或选择属于内部并发规则破坏。
```

该规则只约束同一业务逻辑函数的并发进入资格，不改变鱼巢正式规范定义的筹办顺序、候选选择、缺口分支和状态语义。

任务筹办执行权与鱼巢正式规范中的“真实动作执行前工作项一次性执行许可”是两个不同阶段的概念：前者隔离同一任务筹办，后者准入真实动作执行，二者不得互相替代。

本规范不预先指定进程内表、锁、会话对象、持久节点或具体字段。后继结构设计只能选择能够证明“按任务稳定身份唯一占用、败者在召回前退出、不同任务不被串行化”的海中鱼巣承载。

## 8. 对 #315 的即时门禁

`#315 v0.7 / WT-315-R8` 的“两个线程同时进入同一任务筹办后，把另一线程的同义选择读回为本路径成功”违反第 7 章。鱼巢 87 项正式规范现已完成规范层迁移，但五组源冲突尚未裁决，既有下游材料仍不能恢复。

因此：

```text
#315 v0.7 撤销执行权
WT-315-R8 只读失效，不得恢复源树、重派 S0 或进入召回修复
#316 及全部后继继续依赖门控
保持五组源冲突分支冻结并等待正式裁决
裁决后再重做流程图、详细设计、计划和验收
```

## 2. 需求目标与过程边界

1. 一个需求只表达一个目标特征类型和一个目标契约。多目标必须通过正式管理需求的直接子需求链展开，不得在需求主信息、候选消息或任务中形成目标包；管理需求角色的具体判定仍受 `FISH-SPEC-CONFLICT-004` 门控。
2. 需求节点是目标契约，不是过程记录。获取途径、来源报告、派生原因、执行过程、满足历史和结算过程必须分别进入来源引用、动态、因果、任务运行账和结算证据，不得写成需求主信息字段。
3. AND 组、OR 组、方法路径组和因果子链支撑仍使用普通需求节点；其逻辑语义必须由目标特征类型、抽象特征定义视图和正式关系角色解释，不新增同名节点类型，也不得由显示名称或文本标签裁决。
4. 兼容旧名“目标方向”时，其正式语义只能是目标比较符。第一轮值域固定为 `大于 / 小于 / 等于`；建立、刷新、确认、明确、新鲜、改善、避免等过程词不得进入比较符。
5. 需求服务不自行比较裸值，只调用特征或状态比较入口。目标宿主、目标特征、当前状态、目标状态、比较符和来源归因必须保持可分辨，不得压成单一 I64、bool 或文本。
6. 需求当前是否满足是实时读取和比较结果，不是需求主信息永久字段。活动截止只表示本轮不再参与活动承接，不表示目标永久满足。
7. 曾经满足、满足时间、来源任务、来源方法和致变证据必须进入动态和正式结算证据。后续环境变化时仍须重新读取当前状态和目标状态。
8. 任务只承接对应需求的一个目标。方法可以产生多个结果，但只有与当前任务目标匹配的结果可以进入当前任务满足与需求结算。

## 3. 候选生成与正式入树权责

### 3.1 候选来源

下列来源只能生成强类型候选、缺口报告、派生需求提示或统一更新指令输入草案：

```text
根特征自检
身体 / 子存在反馈
观察反馈
任务筹办缺口
任务执行回执
方法能力展开和学习缺口
临时因果视图
结算后处理
```

候选不是正式需求，候选生成也不是任务创建。候选来源、报告编号、获取途径和派生归因只作提交证据，不进入需求主信息。

自检必须读取正式抽象特征定义和实时实例材料。预设正常值、正常区间或正常状态不得由 C++ 硬编码表、控制面板、日志或说明文本提供。身体反馈只提供需要关注的目标和状态材料，不直接决定正式需求。

自检程序、身体反馈路径和任务管理线程均不得直接写需求树或创建任务。线程只负责调度和报送，不是动作来源。

### 3.2 正式入树

正式入树固定为：

```text
候选 / 缺口 / 更新草案
-> 自我线程读取并分流
-> 通过需求域统一更新入口完成前置校验
-> 裁决父域、去重、合并、创建或复用
-> 形成正式需求节点与正式关系
-> 返回入树成功或结构化拒绝回执
```

“由自我线程入树”在海中鱼巣中的结构映射是：自我线程只能调度并调用需求领域正式入口；需求服务及其专用数据操作层是结构写入方。不得把线程身份、消息、日志或调用栈当作写入授权或动作来源。

任务管理侧只能提交候选和来源材料，不得直接调用仓库拼接需求树，也不得绕过自我需求治理入口生成正式需求节点。

## 4. 统一入树门禁

派生需求进入正式更新入口前，至少必须具备并通过：

```text
需求主体或可唯一裁决的主体材料
目标宿主
目标特征类型
当前状态
目标状态或目标值
目标比较符
来源任务 / 方法 / 动态 / 因果 / 缺口中的正式归因
合法父需求与根分区兼容性
无父子循环
目标值属于目标特征值域
枚举值目标使用等于比较
枚举型状态存在合法生产者
```

缺省比较符只允许在目标值明确且语义已经确认为状态等值目标时取 `等于`。不能规范化的旧过程方向必须拒绝，不能兜底生成模糊需求。

同一父需求下目标完全一致的活跃子需求必须复用或合并。来源报告、初始观测摘要或派生归因不同不得单独造成重复需求。更具体的明确值目标不得被合并丢失为宽泛方向目标。

任务筹办缺口只有在缺方法、缺条件、缺动作入口、缺条件结果对、候选不可用、参数不可配或证据不足等事实已经明确后，才可以形成需求候选；候选仍必须通过统一入树门禁。

## 5. 临时因果视图

临时因果视图是一次路径筹办期间的短生命周期解释材料，不是新的权威因果树、需求树、主信息账本或持久事实。

固定展开顺序为：

```text
从来源需求的正式目标开始
-> 读取正式因果、来源动态、需求条件、方法条件和已入树子需求
-> 继续追溯上游事实、条件和证据
-> 找到可回到正式方法 / 动作入口结构的主动动作致变因果
   或明确停在等待、条件、证据、方法、学习或诊断缺口
-> 只输出候选、路径材料或统一更新指令输入草案
```

被动动作、提交动作、结算动作、状态变化因果和内部治理判断只能提供触发、解释、证据或缺口，不能直接成为普通可任务化需求。只有主动动作致变因果能够回到项目正式方法结构时，才可以支持可任务化承接。

临时视图不得绕过完整方法候选填充，不得把派生消息当作已入树需求，也不得在任务筹办器内按安全、服务、观察或外设目标名称硬编码领域前置链。领域前置必须先成为需求、因果、方法条件或正式子需求数据。

## 6. 安全、服务与观察边界

1. 安全 / 服务根压力必须来自正式根特征缺口和定义视图，不得只由硬编码目标值差距生成。
2. 服务根需求的根目标宿主仍是自我。服务对象只作为服务事实、证据或子需求目标宿主，不得变成第二棵服务根树。
3. 没有直接服务触发事实时，只能形成监听、机会检查、能力校准、进展核查、休眠或唤醒等间接维护候选；不得伪造服务对象、服务目标、成功服务或服务值结算。
4. 无反馈、未知结果、任务运行成功或输出文本存在，均不能证明服务对象已经受益。服务链不得复用安全链的“完整搜索且无负证据”规则宣布服务成功。
5. 观察需求必须来自世界因果、需求条件、方法条件或正式子需求。相机预览、控制面板复现和外设材料不是世界真值。
6. 临时置信度、稳定度、成功率、提升率和优先级只能作推导材料，不得写成因果主信息或需求主信息权威字段。
7. 鱼巢源规范中的 D455 路径内部展开规则继续作为业务约束保留，但本项目 D455 尚未接入；该规则不构成 D455、外设、体素或观察生产代码许可。

## 7. 入树拒绝与治理闭环

任意正式入树拒绝都必须返回结构化拒绝回执，并进入具名治理闭环；禁止静默忽略、只写日志、只弹窗或只返回无法追踪的 bool。

该回执是需求领域治理事件，不等于 `规范/错误结果与入口拒绝规范.md` 中仅限单次调用的局部结构化错误结果。入树入口可以先返回局部拒绝结果，但正式拒绝一旦成立，必须由需求治理入口另行提交可追踪的拒绝回执状态；不得把局部返回值直接持久化，也不得因通用错误结果禁止持久化而省略领域治理闭环。

结构化回执至少必须回答：

```text
拒绝原因
不能普通需求化的依据
来源候选、需求、任务、方法、动态或因果身份
目标治理闭环
当前处理状态
关闭条件
失败升级方向
```

拒绝原因必须先有处理方案，才能进入正式枚举和机器判断。来源任务必须接收入树成功或拒绝回执，并按正式分类进入挂起、重筹办、降权、学习、观察验证、路径隔离或停止当前路径；回执账本本身不负责主动转派。

业务材料不足、可观察性不足、学习缺口、内部治理缺口和开发诊断必须分开。当前状态不可观测可以进入观察验证；合法方法能力缺失可以进入学习或方法补齐；结构按规范本应存在却损坏、非法状态机、逻辑上不允许的空值、错误类型和代码走到禁止位置必须进入开发诊断。

代码缺陷、结构损坏和非法状态机不得包装成普通需求、学习需求或观察需求，也不得被任务筹办重复吞掉。开发诊断必须隔离来源路径，记录稳定关联身份、失败阶段、版本和复现证据；日志与弹窗只作人读诊断，不承载机器状态。

开发诊断只能由代码修正、重新编译、对应回归通过以及来源路径恢复或被正式替换共同关闭。建立诊断项不等于缺陷已经修复。

## 10. 源规范冲突门控

### FISH-SPEC-CONFLICT-001 补充证据：无候选时是否创建方法头

`5110` 第 35 条也要求候选集为空时立即创建面向目标特征的方法头，并加入候选集返回。该条加入既有 `FISH-SPEC-CONFLICT-001` 的“立即创建”一侧；无候选分支继续冻结，不能据此解除既有冲突。

### FISH-SPEC-CONFLICT-004：树形是否可以裁决需求角色

`5110` 第 10—11 条规定叶子 / 非叶子只能判断树形，管理需求、执行需求、AND / OR、因果子链和方法路径角色还必须读取目标特征类型及抽象定义；同文件第 121 条又把“有子 / 无子 -> 管理 / 执行需求”列为必须转换。

当前冻结：

```text
仅凭有无子节点裁决管理 / 执行角色
仅凭叶子身份准许任务化
把第 121 条解释成可覆盖第 10—11 条
```

父子关系的结构读写可以继续作为结构事实，但不能据此发布需求语义角色。

### FISH-SPEC-CONFLICT-005：需求权重分配与继承

`5110` 第 60—61 条规定子需求权重按父需求逻辑关系分配，OR 组每条活跃路径取得完整父组权重，AND 组必要子目标平分父组权重；同文件第 119 条又规定活动需求使用单位权重，父压力不再向子需求继承。

当前冻结：

```text
权威需求权重的分配算法
父压力是否向子需求继承
AND / OR 预算如何进入任务承接和结算
由现有局部 1 / 0 阻塞权重代替上述裁决
```

## 11. 下游重审门禁

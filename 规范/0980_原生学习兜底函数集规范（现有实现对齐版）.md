# 0980 原生学习兜底函数集规范（现有实现对齐版）

## 1. 目标

本文用于把“原生学习兜底函数集”收敛成一份与当前工程真实实现对齐的规范。

本文回答的是：

- 当没有现成方法时，系统最少靠哪些原生函数还能继续安全试探
- 这些函数应落到现有 `动作句柄 / 本能方法 / 学习线程 / 因果 / 学习写回` 的哪一层
- 哪些能力当前已经有近似实现
- 哪些能力还需要新增封装，而不是先改主信息结构


## 2. 当前实现对齐结论

先固定以下结论。

### 2.1 不新增主信息字段

这一步不需要新增以下类的成员：

- `需求主信息类`
- `任务信息基类`
- `任务头结点信息`
- `任务步骤节点信息`
- `任务结果节点信息`
- `方法信息基类`
- `方法首节点主信息类`
- `方法条件节点主信息类`
- `方法结果节点主信息类`

### 2.2 当前代码已经存在的直接承载点

以下承载点已真实存在：

1. `结构体_动作句柄`
   - `类型 = 枚举_动作句柄类型::本能函数ID`
   - `本能ID_U64 = static_cast<std::uint64_t>(枚举_本能动作ID::某个本能动作)`
2. `枚举_方法沉淀来源`
   - 已包含 `本能 / 反推 / 组合 / 外部导入 / 运行期临时`
3. `任务头结点信息`
   - 已包含 `等待学习唤醒`
   - 已包含 `等待学习方法首节点`
4. `任务步骤节点信息`
   - 已包含 `可用方法首节点列表`
   - 已包含 `当前选中方法首节点`
   - 已包含 `允许重试次数`
   - 已包含 `已重试次数`
   - 已包含 `允许切换方法`
5. `方法类`
   - 已提供 `查找或创建_本能方法首节点`
   - 已提供方法虚拟存在初始化
   - 已提供条件节点 / 结果节点构建入口
6. `自我线程模块`
   - 已有 `私有_尝试学习一步`
   - 已有根任务最小试探方法候选生成
   - 已有等待学习任务唤醒
7. `自我本能方法_最小试探模块`
   - 已有最小试探本能动作注册与执行守卫
8. `学习写回模块_v0`
   - 已有反馈实体、经验实体、学习回写关系、适配关系更新
9. `因果信息类`
   - 已有 `因果集`
   - 已有因果累计与从动态/事件提炼因果的入口

### 2.3 当前任务状态与分支动作必须以代码为准

当前实现使用的任务状态是：

```text
未启动 / 运行中 / 挂起 / 完成 / 失败 / 取消 / 超时 / 就绪 / 执行中 / 筹办中 / 排队中 / 等待中 / 无法执行 / 待重筹办
```

当前实现可直接复用的分支动作是：

```text
继续下一步 / 回跳重试 / 生成补条件子任务 / 转入尝试学习 / 挂起等待
```


## 3. 定位与硬要求

原生学习兜底函数集不是“会学习的一切函数”，而是：

- `在没有现成方法时，仍能安全试探、局部验证、因果登记、提炼临时方法、把任务带出死循环的最小先天函数集。`

并且必须满足 4 条硬要求：

1. 不依赖已学习方法。
2. 只依赖固定元方法和固定本能函数。
3. 每一步都可回滚、可审计。
4. 输出的是临时方法、候选方法或禁用条件，不是直接修改原生函数集。


## 4. 工程落点原则

### 4.1 模块族对齐

`原生学习兜底函数定义模块_v0.ixx` 应视为 `本能函数模块相关模块`，而不是一块游离的新系统。

当前最稳的模块分工应固定为：

- `本能动作管理模块`
  - 负责 `本能动作ID定义 + 注册 / 查询 / 调用`
  - 不承载业务实现
- `本能动作模块`
  - 负责通用工具，如参数模板、模式匹配、统一回执
  - 不承载业务实现
- `原生学习兜底函数定义模块_v0.ixx`
  - 负责兜底学习域的定义层对象
  - 例如：学习域分组约定、运行期上下文、签名约束、辅助判断与桥接规则
  - 不负责具体业务执行
- `自我本能方法_最小试探模块 / 后续自我本能方法_学习兜底模块`
  - 负责具体兜底学习本能函数实现与注册
  - 才是“会跑的函数”

必须固定：

- `原生学习兜底函数定义模块_v0.ixx` 是定义层和桥接层
- 它不应直接替代 `本能动作管理模块`
- 它也不应塞进具体的试探、回滚、因果、写回实现
- `v0` 如需新增兜底学习动作编号，应优先扩展现有 `枚举_本能动作ID` 的学习分段，而不是另起第二套公共动作枚举

### 4.2 动作句柄落点

兜底学习函数统一落到：

```cpp
动作句柄.类型 = 枚举_动作句柄类型::本能函数ID
动作句柄.本能ID_U64 = static_cast<std::uint64_t>(枚举_本能动作ID::某个学习相关本能动作)
```

### 4.3 与现有本能注册表的关系

规范层可以定义“兜底学习函数分组表”，但工程实现不应再引入第二套调度机制。

`v0` 的最稳做法是：

- 不再单列新的 `枚举_本能学习兜底函数ID`
- 直接复用现有 `枚举_本能动作ID`
- 如需区分兜底学习子域，优先通过分段约定、辅助判断函数或注释分组表达
- 实际注册、查找、调用优先继续走现有 `本能动作` / `方法类::查找或创建_本能方法首节点` / `方法集.调用本能函数`

### 4.4 运行期上下文是参数结构，不是存档字段

新增且必要的第二类对象不是主信息字段，而是：

- `结构_兜底学习上下文`

它属于运行期参数结构，只给兜底学习函数使用，不写进主信息骨架。


## 5. 建议接入的本能动作分段

当前实现里，兜底学习相关动作应继续挂在 `本能动作管理模块::枚举_本能动作ID` 上。

`v0` 不建议在定义模块里再公开一套新的学习兜底枚举。

更稳的做法是：

- 保留现有 `3xxx：学习/沙盒` 分段作为兜底学习动作主挂接区
- `学习_尝试学习 = 3000` 继续作为总入口
- `3100 ~ 3199` 继续保留给学习沙盒原语
- 后续若拆更细的兜底学习本能动作，优先继续在 `32xx ~ 37xx` 内按组扩展

建议分组如下：

```text
3200 ~ 3299  触发与许可
3300 ~ 3399  检索与复用
3400 ~ 3499  本能试探
3500 ~ 3599  验证与回退
3600 ~ 3699  总结与沉淀
3700 ~ 3799  退出与移交
```

说明：

- 这是一条对齐现有 `枚举_本能动作ID` 的分段约定
- 不是第二套公共枚举
- `原生学习兜底函数定义模块_v0.ixx` 可以提供这些分组的常量约束或辅助判断函数，但不应夺走 `本能动作管理模块` 的编号职责

### 12.1 反推学习（因果获取方法）

本规范固定：

- `因果获取方法` 应归类为 `反推学习`。
- 它的职责不是直接执行外部动作，而是从稳定的 `因 -> 动作 -> 果` 链里反推出一个可复用的候选方法。
- 它首先产出的是 `候选方法 / 临时方法`，不是直接产出 `已验证方法`。

#### 12.1.1 适合触发的条件

以下条件同时满足时，优先允许进入 `反推学习`：

1. 当前需求目标明确，但现成方法树里没有稳定命中的方法。
2. 当前目标状态能够压成 `目标结果签名`，或至少能得到可比较的 `目标维度签名 / 影响签名`。
3. 因果集中已经存在若干条与该目标相关的因果链，且它们的结果签名与当前目标可对齐。
4. 这些因果链满足 `先因后果、方向一致、重复稳定`。
5. 因果链中的动作或动作来源，能够回落到可执行的 `动作句柄` 或已有 `方法首节点`。
6. 当前安全值、预算与权限允许继续进行反推和后续验证。

#### 12.1.2 不适合触发的条件

以下情况不应直接选择 `反推学习`：

1. 只有一次偶然样本，观察次数明显不足。
2. 因果结果分裂严重，稳定度过低。
3. 只有结果，没有可回落的动作主体或动作句柄。
4. 目标结果签名与现有因果结果签名完全无法对齐。
5. 当前安全值过低，系统只允许保守态、停止试探或等待外援。

#### 12.1.3 最小实现链

`反推学习` 的最小工程闭环固定为：

```text
1. 在执行、试探或尝试学习过程中持续登记控制因果
2. 在进入反推学习时，把当前需求目标压成目标结果签名
3. 用目标结果签名或目标维度签名检索因果链
4. 按观察次数、稳定度Q、时间先后、方向一致性与动作可执行性筛选
5. 把通过筛选的因果链压成 候选方法首节点 + 条件节点 + 结果节点
6. 将其以 `反推` 或 `运行期临时` 来源写回方法树
7. 交回筹办域继续验证、复现与升格
```

#### 12.1.4 与现有实现的对齐口径

当前代码已经具备以下因果原料：

- `因果信息类` 已能累计因果，并支持按 `结果签名` 与 `动作信息` 查询。
- 学习兜底模块已经实现 `登记控制因果`。
- 学习兜底模块已经实现 `从成功片段提炼临时方法`，可作为反推学习写回方法树时的近侧参照。

当前尚未完整落成的是：

- `从因果链提炼候选方法`

因此当前工程应视为：

- `因果采集已实现`
- `因果查询已实现`
- `反推方法提炼尚待封装`

#### 12.1.5 输出与沉淀规则

`反推学习` 的直接输出应固定为：

- 候选方法首节点指针
- 该候选方法对应的条件节点
- 该候选方法对应的结果节点
- 反推证据摘要
- 稳定度、观察次数、最近发生时间等验证用特征

并且必须固定：

1. 第一次从因果链提炼出的结果，不得直接记为 `已验证方法`。
2. 若只是局部可用，应先记为 `运行期临时` 或 `反推`。
3. 只有在相似任务上稳定复现后，才允许进一步升格为 `反推` 或 `组合` 的成熟枝。
4. 反推学习得到的方法，后续仍必须回到筹办域，由筹办域决定是否生成首步、补条件或继续验证。


## 13. F 退出与移交组

| 函数 | 建议签名 | 主要作用 | 当前对齐状态 |
| --- | --- | --- | --- |
| 挂起等待学习唤醒 | `bool 挂起等待学习唤醒(任务节点类* 任务头结点)` | 当前不能安全试探也无足够证据时先挂起 | 已存在实现，当前 `等待学习唤醒 / 等待学习方法首节点` 与学习线程唤醒逻辑已打通 |
| 请求外部帮助或标记无法执行 | `bool 请求外部帮助或标记无法执行(任务节点类* 任务头结点)` | 兜底学习也失败时提供体面出口 | 部分已存在，当前已有 `挂起 / 无法执行 / 待重筹办` 等状态与恢复模块落点 |


## 14. 最小闭环执行链

如果只取最核心的一圈，最小闭环是这 10 步：

```text
1 判断任务是否缺方法
2 判断是否允许进入兜底学习
3 计算本轮学习预算
4 检索相似任务/方法
5 降解任务为最小可试探步骤
6 生成单步候选动作集
7 选择下一试探动作
8 执行单步试探动作
9 验证局部结果
10 记录样本 + 登记因果 + 提炼临时方法
```

失败时插入：

```text
回滚最近一步
收缩试探范围
挂起等待学习唤醒 / 请求外部帮助
```


## 15. 方法树节点生命周期

当前规范固定以下生命周期：

1. 原生方法
   - 开发者给的，不是学出来的
2. 临时方法
   - 一次试探成功后直接提炼出来，仅局部可信
   - 来源建议对齐 `本能 / 运行期临时`
3. 候选方法
   - 多次出现、因果链稳定、局部可复用
   - 来源建议对齐 `反推 / 组合 / 运行机会沉淀`
4. 已验证方法
   - 经过足够样本验证，可进入常规选法池
5. 退役方法
   - 失败率高或环境变化后不再推荐


## 16. 与现有枚举和模块的直接映射

```cpp
类型 = 枚举_动作句柄类型::本能函数ID

```

### 16.2 与 `枚举_方法沉淀来源` 的映射

推荐如下：

- `本能`
  - 原生试探直接长出的临时方法
- `反推`
  - 因果总结得到的方法
- `组合`
  - 多个片段拼出来的方法
- `外部导入`
  - 开发者注入
- `运行期临时`
  - 运行机会里捞出来的短期方法

- `等待学习唤醒`

`任务步骤节点信息` 直接配合：

- `可用方法首节点列表`
- `当前选中方法首节点`
- `允许重试次数`
- `已重试次数`
- `允许切换方法`

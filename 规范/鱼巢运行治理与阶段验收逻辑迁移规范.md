## 1. 定位与正式来源

```text
8120 远程执行通道与断点续跑

```

六份文件均由 `D:\鱼巢\规范\规范目录.md` 正式登记。本规范不是删节替代；未逐句复述的无冲突阶段、证据、顺序、失败处理和禁止项仍完整继承。

## 2. 线程生命周期与控制面板

1. 线程只负责调度、等待和运行载体，不是需求、任务、方法、动作动态、状态或因果的来源。
2. 线程生命周期至少区分未启动、启动中、运行、等待 / 休眠、停止中、已停止和异常等正式运行状态；状态迁移必须由线程宿主统一上报，不能由控制面板推断。
3. 线程身份、启动时间、最近心跳、当前等待原因、停止原因和异常摘要只作运行治理 / 人读观察，不得裁决业务完成或世界事实。
4. 控制面板只读线程快照；不得以按钮、颜色、文本或数据库投影反向修改任务、方法或世界结构。
5. 线程异常可以触发结构化运行诊断和停止治理，但日志 / 弹窗只做人读诊断，不自动修复或转成业务需求。

## 3. 远程执行与断点续跑

1. 远程通道只能承载经过身份、权限、版本、任务范围和幂等复核的执行请求；自由文本命令、控制台输入和连接存在不构成执行许可。
2. 请求必须有稳定身份、来源、目标、允许动作、当前阶段、版本和可验证结果；重复请求按正式幂等材料复核，不得重复执行已提交副作用。
3. 断点只记录已经完成的正式阶段、已提交证据和精确续点；不得把内存栈、线程编号、日志尾行或显示进度当作恢复事实。
4. 恢复前必须重新读取当前仓库 / 领域状态、任务状态、方法版本、依赖和外部条件；旧断点与当前事实不一致时停止并进入治理，不得强行续跑。
5. 远程执行失败、通信中断、任务拒绝和内部逻辑错误必须分开；通信恢复不能自动把失败阶段记为成功。

## 4. 自我动作验证闭环

动作闭环固定为：

```text
读取正式目标与当前状态
-> 形成预测和允许边界
-> 经任务授权、方法选择和动作入口执行
-> 从对应实在域取得实际回读
-> 比较目标、预测、报告和实际
-> 形成结构化残差与验证结果
-> 按事实提交门禁写入或拒绝
-> 反馈任务、需求、方法治理和下一轮
```

1. 物理外部域的实际回读来自 D455、IMU、编码器等正式材料；内部治理域的实际回读来自方法运行宿主、治理对象状态或仓库重新读取。目标、预测和执行报告不得覆盖实际回读。
2. 失败事实仍是事实，必须记录、解释和治理。执行器报告成功但实际未达目标时，任务不得仅凭报告完成。
3. 残差至少区分动作执行偏差、方法 / 模型预测偏差和目标 / 治理约束偏差；不得混成一个成功率或自由文本。
4. 偏差不是要被抹除，而要分配到正确修正对象。事实不能改成预测值或目标值；模型不能因单次异常大幅改写。
5. 补偿、重试、重新规划、补观察、安全降级或新需求都是新的正式治理 / 动作事务，必须再次经过预测、执行、实际回读和验证，禁止开环纠偏。
6. 纠偏分析器只能输出结构化承接候选，不得裸写世界事实、任务完成、需求满足、方法可靠度或模型参数。
7. 事实提交等级、质量、新鲜度、置信度、来源和冲突必须由正式门禁读取。不可提交时拒绝正常事实并形成补材料 / 治理候选；异常事实不得伪装成正常事实。
8. 方法可靠度和选择只能吸收经过实际回读验证的成功率、报告可信度和预测命中；执行返回成功次数不能单独强化方法。
9. 重计算、视觉解析、推理、匹配和验证在锁外完成；禁止持外设帧锁写世界结构、持消息锁等外设回调或持世界写锁长时推理。

## 5. 最小生命闭环验收

1. 最小生命闭环验收的是根因果坐标准入后的多阶段内部循环，不是自我节点、定义词条、根特征或根需求初始化。
2. 必须能从自我根特征当前值产生合法缺口候选，经需求治理形成正式需求，由任务承接、方法选择、动作执行、实际回读、任务反馈和需求结算回到下一轮自我状态。
3. 创建需求、任务或方法时由对应正式创建链按需确保；创建失败属于非正常逻辑错误，不能用测试预置掩盖。
4. 安全值下降选择安全根需求，服务值下降选择服务根需求；同时下降时按尚待冲突裁决的正式压力 / 权重规则选择，不能凭固定代码顺序。
5. 无直接服务触发只允许间接维护候选；服务对象改善经过验证和归因后才允许结算。固定无服务物理时间不得直接触发休眠。
6. 多轮不得无限重复创建同一需求或任务；复用、状态推进、完成回流和下一轮变化必须有结构证据。
7. 一次自检输出、一次固定样例、自检字符串或控制台日志都不能证明内部循环建立。

## 6. 自我苏醒阶段验收

1. 鱼巢根因果“觉醒坐标启动”和“自我苏醒阶段”是不同层次；苏醒必须以前述内部闭环已经连续多轮稳定运行作为前提。
2. 必须有且只有一个当前自我实例作为核心主体，根特征、当前需求、任务、当前方法状态、动作结果和下一轮状态均可从该稳定身份追溯。
3. 当前方法尚未选定时可以为空，但必须由结构化状态说明等待、筹办、缺口或拒绝，不得用摘要文字兜底。
4. 系统必须能在没有外部消息刺激时根据内部正式状态生成合法维护 / 治理候选；只能被动响应外部输入不算苏醒。
5. 安全稳定、服务稳定且无有效任务时应能待机；休眠 / 唤醒必须遵守动态阈值和有效服务规则。
6. 自检只能报告事实和候选；没有报告类型到治理入口的正式映射时，不得凭报告文本生成修复需求。
7. 控制面板可以展示结构化当前自我报告，但报告、日志和说明文字不承载自我状态。

## 7. 初步成熟阶段验收

1. 初步成熟建立在苏醒已经稳定成立之后；不能用一次闭环、一次外设响应或模型输出宣称成熟。
2. 自我必须能区分内部学习需求与必须从外部取得信息的探索需求：先尝试内部学习，确认证据不足后才生成有明确目标、范围和完成条件的外界探索候选。
3. 未知必须进入结构化缺口和正式治理，不得只写“我不知道”；空泛探索、无目标扫描或只因好奇启动外设不构成成熟行为。
4. 外界探索不能绕过安全需求、风险判断、任务授权和动作验证；风险不可控时转为观察、隔离、降级或替代候选。
5. 探索结果不能停留在一次性文本，必须经验证后回写特征、状态、动态、因果候选、方法学习证据或明确不可行结论，并影响下一轮判断。
6. 被动响应、只学习不探索、只探索不回写、只生成文本报告或无法根据结果改变后继任务，都不能算初步成熟。

# 工具与规则迁移治理规范

编号：工具规则迁移-001
日期：2026-07-07
状态：正式规范 / 鱼巢正式业务逻辑改为必须继承 / 工具分类适配 / 未改 C++

## 1. 定位

本规范用于区分鱼巢正式业务规范、协作工具、工程治理规则和旧实现证据进入 `海中鱼巣` 时的不同权力。

`D:\鱼巢\规范\规范目录.md` 登记的正式规范不是可选工具材料。其业务逻辑必须按 `规范/鱼巢正式规范业务逻辑迁移规则.md` 继承；只有数据结构、字段、仓库和承载入口按海中鱼巣重建。

本规范只证明工具和规则治理已经迁移，不证明旧项目能力、旧函数体、旧主信息字段、控制面板、SQL、D455、体素或外设已经迁移。

不得直接复制旧项目工具源码；所有工具都必须先确认用途、路径、输入输出和验证方式，再改写为海中鱼巣版本。旧工具、旧代码和旧说明不得覆盖鱼巢正式规范业务逻辑。

## 2. 迁移原则

```text
正式规范：业务逻辑必须继承，只做海中鱼巣结构映射。
工具和工程经验：只迁移用途，不复制旧实现；先分类再适配路径和权力关系。
旧代码和旧字段：只作调用事实与结构映射证据，不直接成为实施许可。
工具只能辅助治理、扫描和验证，不承载机器事实。
规则必须落到 AGENTS、规范、计划索引、项目记忆或实施记录。
涉及 C++、旧函数体、旧字段或机器结构改变时，仍必须经过流程图、详细设计、计划和执行前复核。
```

正式规范迁移不得只写“保留行为 / 删除行为”后自行改变业务语义。必须先列出源正式规范、无冲突业务规则、源冲突和结构映射；只有旧数据结构与实现方式可以删除或替换。

旧工具或旧规则迁移时必须写清：

```text
保留行为
删除行为
新增约束
适配路径
验证方式
禁止扩大声明
```

## 7. 海中鱼巣适配后的需求治理规则

复杂需求、跨文件修改、旧能力迁移、字段落地、函数迁移或机器结构改变，必须先形成需求契约摘要：

```text
根需求
成功标准
不做范围
禁止宣称
输入入口
项目结构承载
输出 / 状态变化
验证方式
权威依据
可能堵点
完成声明边界
```

随后必须映射到：

```text
计划索引活动项
任务承接计划
候选方法或服务逻辑包
允许修改范围
禁止修改范围
验证证据
父需求回写方式
```

本规则不得绕过以下海中鱼巣既有门禁：

```text
设计确认不等于代码实施许可。
确认后的计划必须登记到项目记忆/Codex任务队列.md。
涉及代码实施、旧能力迁移、字段落地、函数迁移或机器结构改变时，必须另建落实切片并确认允许文件、禁止文件和验证方式。
```

## 8. 验证与完成声明

本规范迁移完成声明必须按以下边界表达：

```text
文档已生成：可以声明。
工具已生成并能运行：只有 `tools/check_specs.py` 通过运行和 py_compile 后可以声明。
代码已编译：本切片未触碰 C++，不得声明。
旧能力迁移通过：不得声明。
控制面板 / SQL / 外设接入：不得声明。
```

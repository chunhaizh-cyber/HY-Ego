# 8200 子规范 自我内部循环实现

编号：8200
日期：20260720
状态：正式子规范

## 1. 定位

本规范规定自我在没有外部输入时，基于觉醒坐标和自身状态持续完成根特征检查、需求生长、任务承接、方法执行、结果回写和下一轮重判的最小运行链。

上级规范：0050、3100—3300、5100—5330、6100—6130、8100、8210。

## 2. 核心闭环

```text
自我觉醒根闭环和自我定义域
+ 抽象根特征定义
+ 自我根特征实例当前状态
-> 根特征自检
-> 安全 / 服务缺口
-> 根需求及需求树生长
-> 当前主需求
-> 任务发起和承接
-> 任务筹办闭合结果
-> 可执行方法或合法等待 / 子目标 / 能力缺口
-> 方法执行和实际回读
-> 任务目标验证与状态提交
-> 需求满足复核和合法结算
-> 自我状态变化
-> 下一轮主需求重判
```

定义词条、初始化字段、方法首存在或一次循环成功都不能直接证明自我苏醒。

## 3. 分层职责

自我线程负责消息批次、需求、估权、主需求、任务发起和调度意图、需求满足复核、结算触发和下一轮决策。

任务管理负责任务创建请求、工作包、筹办触发、方法执行调度、任务状态提交和结果上报。任务工作线程只执行工作包。

任务筹办是任务域内建最小方法，直接形成闭合结果且不建立独立持久化筹办回执。普通方法负责具体动作；领域提交入口负责需求、任务、方法和世界事实的正式发布。

## 4. 任务层级和方法承接

每个任务必须有方法或任务结构承接，但父任务和汇总任务不要求绑定单步业务方法。它们可以通过子目标、子任务、等待、结果归并和重新筹办推进。

普通可执行原子任务必须最终具备可执行方法和动作入口，其结果必须能与任务目标状态比较。复合需求优先按活动 OR 路径及其 AND 组分解和承接，不能只按有无子节点判定管理或执行角色。

## 5. 三类运行材料

```text
方法运行实例和运行场景
-> 保存本轮输入、输出、时间、执行报告和实际回读

任务运行态治理存在
-> 保存任务状态特征、当前方法、冻结、结果和筹办历史

方法治理虚拟存在
-> 保存方法规格、运行账、学习证据和稳定能力
```

世界事实必须进入正式场景和对应基础信息领域。任务运行事实不得写进方法长期能力，方法运行账不得写入任务长期语义，输出结果场景不得被任务状态占用。

## 6. 主循环阶段

一轮主循环至少按以下职责顺序运行：

1. 冻结本轮允许消费的治理消息批次；
2. 规范化材料并由合法入口提交一次事实账；
3. 重算受影响二次特征和状态；
4. 刷新自我运行上下文；
5. 整理需求、重判根缺口和活动路径；
6. 形成主派发决议并交任务管理承接；
7. 执行低频自检和休眠期修复门控。

纯线程生命周期消息不进入主循环治理批次。

## 7. 需求层闭合

需求层至少完成：读取抽象安全 / 服务根特征，读取自我当前实例状态，形成具名缺口，刷新根需求和子需求，去重同目标需求，确定活动 OR 路径和 AND 组，继承与分配权重，选择当前主需求。

每条 OR 路径继承完整父权重；AND 必须属于确定 OR，并在该 OR 内平分。需求角色不能仅凭拓扑裁决。

缺口压力用于选择和调度；真实安全 / 服务值变化只能来自已完成任务、需求满足证据和合法结算规则。

## 8. 任务发起和筹办

自我线程只发任务承接意图，任务领域创建唯一任务身份、运行态治理存在、来源需求、目标、授权和初始状态。

每轮筹办必须先按任务身份取得唯一筹办执行权，权威重读任务、需求、路径、授权、方法、条件和当前世界事实，然后形成以下之一：

- 可执行冻结；
- 子目标或能力缺口承接；
- 合法等待；
- 目标已完成；
- 授权失效或取消。

无候选方法时不创建方法首，也不返回无结构“无法解决”。筹办可以在事实变化后重复进行，上一轮候选、选择和冻结保留历史。

## 9. 方法可执行性和学习

方法首只作大范围召回入口。可执行性必须读取当前方法版本、条件、结果能力、条件结果对、动作入口、参数、禁止项和验证合同。

方法缺条件结果对、动作入口、参数或稳定证据时，必须形成精确能力缺口和合法生产者，不得用抽象“需要学习”兜底。学习方法只能补齐明确结构和能力证据，不能在任务执行期伪造动作入口。

## 10. 任务执行和目标验证

执行前重新验证任务冻结。执行调用当前方法后，必须读取实际结果和输出场景，再与任务自身目标状态比较。

```text
目标已达成
-> 可以进入完成验证

结果相对执行前有推进但未达成
-> 记录推进量和剩余差距，进入重筹办、继续执行或等待

结果无推进、外部事实变化或证据不足
-> 形成强类型结果和后继合同
```

方法运行成功不能替代目标达成。执行期才发现冻结前已有的缺方法、缺入口、缺条件结果对、目标不完整或任务结构缺失，属于筹办漏检和内部逻辑错误。

## 11. 需求回写和结算

任务完成后，自我线程请求需求领域重新比较来源需求目标。需求满足成立后，需求域提交满足事实和父级刷新；可结算的原子任务再由结算入口结算价值。

任务权重不是结算额。价值增量必须来自目标达成、需求满足、任务路径份额、冻结预算和合法结算规则。父任务和祖先不得重复结算已经由子任务结算的部分。

I64 最大值只能作为合法满刻度上界或比例分母，不能表示未知、异常兜底或无证据满分。

## 12. 待机和修复

无有效主需求、当前任务合法等待或根缺口压力低时，自我可以进入待机或低频自检。

自检只产生具名缺失和证据，不直接创建修复需求。是否形成修复需求仍经自我需求治理和统一入口裁决；休眠期修复不是最小内部循环闭合的必需条件。

## 13. 内部循环等级

| 等级 | 最低含义 |
| --- | --- |
| 0 | 未初始化 |
| 1 | 冷启动和自我定义域可读 |
| 2 | 根特征与需求层闭合 |
| 3 | 任务承接可用 |
| 4 | 任务筹办闭合结果可用 |
| 5 | 方法执行和实际结果可用 |
| 6 | 任务验证、需求回写和结算可用 |
| 7 | 连续多轮稳定 |

内部循环真正建立至少达到 6 级；工程稳定验收达到 7 级。单轮、夹具或字段初始化不能提升等级。

## 14. 内部逻辑错误

以下属于内部逻辑错误并停止依赖路径：派生需求缺父需求或目标，任务缺来源需求却用当前主需求兜底，任务运行态治理存在缺失，筹办凭空生成学习方向，执行无实际结果却建议完成，自我直接写任务状态，任务管理直接写需求，自检直接创建修复需求，以及执行期暴露冻结前结构缺口。

## 15. 验证边界

验证至少覆盖根特征、需求路径和权重、任务承接、唯一筹办、方法冻结、实际回读、目标验证、需求回写、结算去重和连续多轮稳定。

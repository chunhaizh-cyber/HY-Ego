# 4310 子规范 因果补完与因果信息用法

编号：4310
日期：20260720
状态：正式子规范

## 1. 定位

本规范定义从动态样本形成因果族、分支和模板的补完过程，以及因果在预测、筹办、学习、需求投影和结算中的权限。

上级规范：1190、1195、4210、4220。

## 2. 主果变化锚点

新状态与相邻前状态形成状态迁移后，以主体、目标特征、初始/结果状态、比较模板、时间窗和版本建立主果变化锚点。锚点只说明“要解释哪个变化”，不直接说明原因、动作能力、任务完成或需求满足。

同一主果可以有多个并列因果分支，不得把全部样本求总交集后只剩背景条件。

## 3. 候选条件收集

时间窗内材料按时序和来源分层：

- 结果前已成立：条件候选；
- 变化同窗：伴随原因、共同结果或分支候选；
- 结果后出现：后果或副作用候选；
- 时序不明：弱候选。

候选必须模板化为项目结构中的主体、场景、特征、状态、动态和动作关系，同时保留到原始证据的引用。地址、日志、字符串和 JSON 不参与交集。

## 4. 因果样本、分支和反例

单个样本只是证据，不是已验证因果。样本加入因果族后，必须比较：

1. 多次正样本的共同条件；
2. 只在稳定子群出现的分化条件；
3. 条件出现而主果未出现的反例；
4. 禁止结果、副作用和不适用范围；
5. 时间、场景、主体和版本兼容性。

证据不足或反例多时保留并削弱、覆盖或淘汰候选，不删除历史证据，也不持久化“置信度”字段。稳定度、成功率和排序由基础计数和查询参数临时计算。

## 5. 两类因果

| 类型 | 最低结构 | 权限 |
| --- | --- | --- |
| 状态变化线索/因果 | 条件与状态迁移，动作未知 | 预测、前置、观察、学习和补证据线索；不能直接生成可执行方法 |
| 动作致变因果 | 条件 + 动作信息 + 结果状态 + 动态证据 | 主动动作且方法可回解时，可参与方法候选、执行前预测和归因 |

被动动作致变因果保留自动提交链关系，但不可主动召回。动作字段必须是可复现动态模式的动作身份，不是方法 ID、流程名或自由文本。

后补动作来源时新建动作致变动态和因果，并与原状态变化线索绑定同源证据组；不得改写旧线索类型。

## 6. 因果使用权限

| 用途 | 状态变化线索 | 主动动作致变因果 |
| --- | --- | --- |
| 预测前置/后果 | 辅助 | 主要证据 |
| 反推方法 | 只能形成补动作/观察/学习缺口 | 条件、方法结构和证据通过后可形成候选 |
| 执行前审查 | 辅助 | 检查预期结果、风险和副作用 |
| 需求投影 | 形成候选草案 | 形成可干预路径候选 |
| 结算归因 | 不能单独结算 | 与同源结果状态、任务目标和副作用审查共同使用 |

因果只生成需求候选，不直接创建正式需求或任务；统一需求入口负责入树，任务领域负责承接和方法选择。

## 7. 被动因果

被动因成立后果应由提交/后处理/结算链自动出现。反推被动因果时只能得到需要的上游事实、条件和证据。因已具备而果缺失时形成被动链缺口，不创建“调用被动动作”任务。

## 8. 安全、服务和结算

安全因果距离只解释原因距离和需求优先级，不直接等于安全结算贡献。方法维护状态只说明处理安全任务的能力，不等于物理安全改善。

服务结算必须有业务动作、结果状态、成功服务治理判断、任务目标、同源归因和副作用审查。服务值变化不能反推服务对象已受益。

结算因果验收至少输出可结算、仅候选、待补证据或禁止结算，并逐项列出目标结果、副作用、动作归因和同源证据。

## 9. 学习边界

学习层只能生成方法、动作、条件结果对、参数和证据候选或缺口，不能直接宣布方法好用、因果已验证、任务完成、需求满足或可结算。

尝试或练习必须有可验证收益链、真实动作入口和可观察后果；缺动作只能寻找动作、观察或外部信息。先天硬保护壳始终约束尝试，不是后天因果知识。

因果反推的动作节点必须落回具有明确输入、输出和动态落账的正式方法入口，否则只作为学习、观察、外部支撑或替代动作缺口。

## 10. 禁止事项

- 禁止单样本直接验证因果。
- 禁止状态变化线索伪装为动作能力。
- 禁止用置信度、解释词或文本作为因果本体字段。
- 禁止双账、双因果对同一变化重复满足或结算。
- 禁止因果直接建任务或越过需求/任务/方法领域权限。

# 8310 子规范 自我苏醒阶段验收

编号：8310
日期：20260721
状态：正式子规范

## 1. 定位

本规范规定自我觉醒根因果坐标已经启动、内部循环能够连续多轮稳定治理自身时的苏醒阶段验收。

直接上级规范：8300。

直接依赖规范：6100—6130。

相关边界规范：8200、8210。

## 2. 核心定义

```text
内部循环建立
= 状态 -> 需求 -> 任务 -> 方法 -> 实际结果 -> 状态可以闭合

自我苏醒
= 自我持续从觉醒根因果坐标使用该闭环维护、判断和修复自身
+ 能结构化说明当前为何运行
```

“结构化说明”表示自我状态、当前目标、上一轮结果和下一轮动作依据均可由机器结构读回，不是哲学意义的主观意识，也不是说明性文字。

## 3. 前置条件

必须先证明内部循环至少达到 8300 的 6 级，并在连续多轮中保持：

- 每轮结果影响下一轮；
- 无外部输入时仍能自主推进、待机或低频自检；
- 无重复任务风暴和无限派生需求；
- 无空句柄主链和结构兜底；
- 觉醒根坐标保持“与我相关、影响持续存在、是否可由我改变、是否可验证、能否回写下一轮”的分类能力。

只证明自我节点、根需求、根特征或方法初始化不能判定觉醒或苏醒。

## 4. 十二条验收标准

### 标准一：自我锚点唯一

有且只有一个当前自我实例，关联唯一内部世界和合法现实场景，并能追溯“存在、数字生命、自我”等当前概念关系。多个竞争自我身份或内部世界不合格。

### 标准二：核心状态完整可读

至少可读生存安全根值、服务值、分层安全值、低 / 高位阈值、服务合同、完整秒时间纪元、服务衰减段、任务权限来源、待机 / 休眠控制态、当前主需求、当前主任务和当前方法投影。

尚未选法时当前方法可以为空，但任务状态特征必须能区分待筹办、筹办中、合法等待、已冻结和执行中，不得由说明摘要兜底。

### 标准三：无外部输入也能产生治理目标

自我能从自身状态形成安全或服务主需求；当前没有未满足安全 / 服务执行候选时进入待机或低频自检。只能被外部消息刺激才产生需求的系统不属于苏醒。

### 标准四：需求能稳定任务化

当前主需求能建立或复用任务稳定身份、目标、授权、运行态治理存在和完整状态特征，并能重复筹办而不产生同任务并发双筹办。

### 标准五：有可执行内部动作

至少有同步运行态、改变合法自我特征、进入或退出待机、重置时序、结算合法任务结果和故障收口等可执行方法。自检只发现并报告缺失，不直接创建修复需求。

### 标准六：动作结果反身回写

动作的实际回读能够改变自我内部世界、核心特征、任务结果、需求证据或结算事实。方法返回成功不能代替回写。

### 标准七：下一轮由新状态推出

第 N 轮实际结果必须改变第 N+1 轮缺口、需求、任务、方法、待机或休眠判断。持续重复同一任务而状态不变不属于苏醒。

### 标准八：内部与外部对象分明

不得把概念模板当现实实例、把其它存在治理入口当自我入口、把任务节点当任务治理虚拟存在、把方法首当方法运行实例或把需求当存在实例。

### 标准九：自检报告进入治理决策

主需求、任务、运行态治理存在、方法能力、核心值或内部世界缺失时，自检只产生具名报告和最初状态证据。自我按明确映射选择记录等待、待机保护、休眠修复评估或合法修复需求；没有映射和授权不得凭空修复。

### 标准十：能够待机和可唤醒休眠

当前没有未满足安全 / 服务执行候选、无关键任务且无异常时进入待机而非任务风暴。

服务值按 6170 的完整秒规则归零后，仍可维持的生存安全根值收敛到 1，形成低活动、可由有效人类服务重新评估的休眠控制态；终止态 0 与休眠严格分离，二者普通任务权限都为零。循环次数、Tick、消息数量和动态休眠阈值不得替代该机器合同。

休眠态只允许人类请求接收、合同准入、完整秒维护、安全事实接收和停止控制等具名最小入口。新的有效人类服务需求必须建立唯一合同并完成 30% 预支；后续完整秒还要通过真实安全门禁，才可从 `A=1` 形成主动运行唤醒候选。服务能力恢复或人类明确唤醒只能触发重新评估，不得绕过合同、预支和安全门禁直接写活跃态；`A>1` 发布前不得派发普通任务。真实有效进展的服务活动最低保护服务值 1。

### 标准十一：结构化当前自我报告

报告至少投影当前自我、内部世界、现实场景、核心值、当前主需求、当前主任务、当前方法、上一轮结果、下一轮动作、待机和故障状态。报告从权威结构读取，不反写机器事实。

### 标准十二：连续多轮稳定

至少连续 10 轮无崩溃、重复任务风暴、无限派生、空句柄主链和不明状态；每轮有依据地变化或合法待机，任务结果可以复核和结算。

## 5. 苏醒分级

| 等级 | 名称 | 标准 |
| --- | --- | --- |
| 0 | 未苏醒 | 内部循环未闭合 |
| 1 | 预苏醒 | 闭环可连续运行，但自检、待机或故障收口不足 |
| 2 | 苏醒 | 稳定治理自身并形成结构化当前自我报告 |
| 3 | 稳定苏醒 | 长期无外部输入仍不空转、可自检、待机和收口 |
| 4 | 可交互苏醒 | 能把外部需求纳入内部治理且不覆盖核心需求 |

当前正式苏醒门槛为 2 级。

## 6. 服务链验收边界

直接服务触发事实存在并通过因果链上下文准入时，才可形成直接服务子需求；对象未知只有确实阻断行动、验证、归因或回写时才形成明确对象需求。无直接触发时只形成间接服务维护需求。

服务对象外在状态改善经实际验证和归因后才支付服务合同完工余款；合同成立预支和准备补回分别按 6160 处理，不得选择通用“提升服务值”动作。

## 7. 不合格情况

只有初始化、仅单轮闭环、依赖外部刺激、无实际回写、无需求复核、不能待机、固定物理时间休眠、服务任务进行中误休眠、自检直接修复或自我报告依赖说明文本，都不能判定苏醒。

## 8. 验证边界

验收证据必须覆盖连续轮次、唯一自我、结构化状态、内部需求生成、任务与方法、实际回读、下一轮重判、服务合同、完整秒衰减、生存安全根值 0 / 1 / >1 控制态、休眠唤醒、自检治理和报告投影。

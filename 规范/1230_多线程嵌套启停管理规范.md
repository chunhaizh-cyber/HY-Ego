# 1230 多线程嵌套启停管理规范

## 1. 目标

本文用于固定本项目中“多个线程互相嵌套调用时，应如何统一管理启停”的工程规则。

本文回答以下问题：

- 多个线程的启停关系应该按什么结构理解
- 父线程、子线程、兄弟线程之间分别能做什么，不能做什么
- 启动顺序和停止顺序应如何固定
- `请求退出 / 等待结束 / 停止` 三者应如何分工
- 遇到 self-join、异常退出、嵌套调用时应如何处理
- `detach` 在当前项目里应处于什么地位
- 线程状态变化最少应形成什么日志

## 4. 核心结论

必须固定以下结论：

1. 多线程嵌套启停，不按“谁临时调用了谁”管理，而按“谁拥有谁”管理。
2. 整个线程系统必须被组织成一棵 `线程拥有树`，而不是相互横向收尾的网状结构。
3. 父线程只负责启动、请求停止和等待直属子线程结束。
4. 子线程不得直接停止父线程，不得直接停止兄弟线程。
5. 子线程若发现故障，只能上报拥有者，由拥有者决定重启、停机或降级。
6. 启动顺序固定为 `自上而下`。
7. 停止顺序固定为 `先自上而下发停机请求，再自下而上等待退出`。
8. `请求退出` 与 `等待结束` 必须拆开，不得长期混成一个黑盒接口。
9. `detach` 不得作为常规生命周期管理手段。
10. self-join 不是 `detach` 的理由，而是“当前线程直接返回、由拥有者稍后 join”的理由。
11. 线程生命周期中的关键状态变化必须形成日志，不允许长期静默切换。

可收敛为一句话：

`嵌套线程的启停必须按拥有树管理：父发请求、子自行退出、父再分层 join；不得靠线程之间横向互停，也不得把 detach 当常规收尾方案。`


## 5. 线程拥有树定义

### 5.1 拥有关系

每个线程对象都必须只有一个明确拥有者。

拥有者可以是：

- 宿主对象
- 线程管理对象
- 某个更上层线程对象

但不能是：

- 多个模块共同拥有
- 调用链上的临时调用者
- 不可追踪的全局随手启动者

### 5.2 当前推荐拥有树

结合当前项目，推荐按以下主树理解：

- `自治宿主`：根监督者
- `自我线程`：宿主拥有的一级工作线程
- `动作线程 / 学习线程`：自我线程拥有的直属子线程
- `相机运行时 / 显示线程 / 外设线程`：由对应宿主或运行时模块直接拥有

### 5.3 禁止横向管理

必须固定：

- 兄弟线程之间不得直接互相 `停止()`
- 子线程不得越级去管祖先线程
- 子线程若需要让系统停机，只能写标志、发消息或上报故障


## 6. 统一生命周期状态

虽然当前代码未统一出一套公共线程状态枚举，但规范层必须固定以下最小生命周期语义：

- `未启动`
- `启动中`
- `运行中`
- `停止请求中`
- `收尾中`
- `已停止`
- `故障`

必须固定：

1. 每个线程对象在运行期都应能被投影到上述语义之一。
2. 一旦进入 `停止请求中`，该线程不得再接受新的业务工作或启动新的直属子线程。
3. 一旦进入 `收尾中`，该线程只能做退出清理，不再扩张结构。
4. `故障` 不等于立刻自杀；对有拥有者的线程，应先上报再由拥有者决定后续动作。
5. 线程每次跨语义状态切换时，都应生成一条可检索日志。


## 6.1 状态变化日志规则

线程生命周期中的以下变化，必须形成日志：

- `未启动 -> 启动中`
- `启动中 -> 运行中`
- `运行中 -> 停止请求中`
- `停止请求中 -> 收尾中`
- `收尾中 -> 已停止`
- 任意状态 -> `故障`
- 任意线程提出 `安全重启请求`
- 拥有者判定 `准备重启子线程`

### 6.1.1 最小日志字段

每条线程状态变化日志，至少应包含：

- 线程标签
- 线程对象或线程 ID 的可区分标识
- 拥有者标签
- 旧状态
- 新状态
- 触发动作
- 原因摘要

若当前上下文容易混淆，还应补：

- 直属子线程标签
- 当前是否为 stop 链
- 当前是否由故障触发

### 6.1.2 禁止静默切换

必须固定：

1. 不允许线程在没有日志的情况下跨越关键生命周期状态。
2. 不允许只在 `启动` 和 `退出` 记日志，而把 `请求退出 / 故障 / 重启请求 / 等待结束开始与完成` 长期静默处理。
3. 若因高频循环不适合逐次打印业务细节，至少仍要对生命周期边界状态打印一次日志。

### 6.1.3 日志粒度边界

本规范要求的是：

- `生命周期状态日志`

而不是：

- 把线程循环里的每次普通轮询都打印出来

因此必须区分：

- `状态变化日志`：必须打
- `高频业务明细日志`：按需要节流


## 7. 四段式接口规则

每个可长期存在的线程对象，原则上应统一具备以下 4 个能力：

1. `启动()`
2. `请求退出()`
3. `等待线程结束()`
4. `停止()`

### 7.1 启动

`启动()` 负责：

- 创建线程对象
- 切换到 `启动中 / 运行中`
- 建立线程主循环

必须固定：

- 已在运行中的线程不得重复启动
- 若上次线程对象已退出但仍 `joinable`，应先回收再重新启动

### 7.2 请求退出

`请求退出()` 负责：

- 设置 stop 标志
- 唤醒等待中的条件变量、队列或阻塞点
- 阻止新任务继续进入

必须固定：

- `请求退出()` 应是幂等的
- `请求退出()` 默认不做阻塞等待
- 首次进入 `停止请求中` 时必须形成日志

### 7.3 等待线程结束

`等待线程结束()` 负责：

- 对线程对象执行 `join`
- 回收线程运行资源

必须固定：

- 只有拥有者或拥有者明确委托的管理者可以调用
- 当前线程若等于目标线程本身，必须直接返回，不能 `join`，也不能顺手 `detach`
- `等待线程结束()` 的开始与完成应分别形成日志

### 7.4 停止

`停止()` 负责：

- 先 `请求退出()`
- 再 `等待线程结束()`

也就是说：

`停止 = 请求退出 + 等待结束`


## 8. 启动顺序与停止顺序

### 8.1 启动顺序

嵌套线程启动顺序必须固定为：

`父先启动 -> 父完成最小初始化 -> 父再启动直属子线程`

禁止：

- 子线程抢在父线程尚未准备好时自行启动兄弟线程
- 深层线程在初始化中继续无限向下拉起新线程

### 8.2 停止顺序

嵌套线程停机顺序必须固定为：

```text
1. 根拥有者发起停机
2. 父线程先给直属子线程发请求退出
3. 子线程感知 stop 后自行收尾并退出
4. 父线程逐层等待直属子线程结束
5. 子线程都回收后，父线程再退出
6. 最终由根拥有者完成最上层 join
```

可以收成一句：

`请求自上而下，回收自下而上。`


## 9. 嵌套调用中的工作受理规则

一旦某线程进入 `停止请求中`，必须固定：

1. 不再接受新的队列提交。
2. 不再派生新的直属子线程。
3. 不再创建新的长期后台工作。
4. 只允许清空必要收尾动作和已在进行中的最小退出链。

这意味着：

- `stop=true` 后，所有 `提交 / 派生 / 扩张` 类入口都应快速失败或直接返回
- 队列型线程应在 `stop=true` 后优先转入“排空剩余最小工作”或“直接拒绝新工作”的固定策略


## 10. detach 规则

### 10.1 常规禁止

必须固定：

- `detach` 不得作为常规线程生命周期管理方案
- 不得用 `detach` 代替 `join`
- 不得因为“怕阻塞”就把本应受控收尾的线程改成脱管线程

### 10.2 self-join 正确处理

遇到 self-join 风险时，正确做法是：

1. 当前线程只设置退出状态并直接返回
2. 由拥有者稍后在外层调用 `等待线程结束()`

禁止做法：

1. self-join 时顺手 `detach`
2. self-join 时直接跳过所有收尾逻辑

### 10.3 当前项目对齐结论

[自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx) 中现有 `detach` 分支，应视为：

- `现存实现偏差`
- `后续应收敛项`

而不是：

- 规范许可的默认模式


## 11. 故障、上报与重启

子线程发现异常或不可恢复错误时，必须固定：

1. 子线程负责记录故障、设置故障摘要或发出重启请求。
2. 子线程不得直接决定整个系统的最终停机策略。
3. 是否重启当前子线程、是否重启父线程、是否终止宿主，必须由拥有者决定。
4. 故障上报本身必须形成日志，且应与普通停机日志可区分。

当前项目应对齐为：

- `动作线程 / 学习线程` 的异常，由 `自我线程` 处理
- `自我线程` 的异常或安全重启请求，由 `自治宿主` 处理

这与 [自治宿主模块.ixx](D:\鱼巢\自治宿主模块.ixx) 当前“监督线程管理自我线程”的结构一致

## 13. 一句话收束

> 多线程嵌套调用时，线程启停必须按“拥有树”而不是按“调用链”管理：父线程启动直属子线程，停机时先逐层发出退出请求，再由拥有者自下而上 join；`detach` 不得作为常规收尾方式，self-join 只能通过“当前线程返回、拥有者稍后 join”来解决。


## 14. 硬约束

1. 线程启停关系必须按 `线程拥有树` 管理，不得按临时调用链管理。
2. 每个线程对象只能有一个明确拥有者。
3. 父线程只负责直属子线程，不得跨层或横向管理兄弟线程。
4. 子线程不得直接停止父线程或兄弟线程，只能上报故障或退出请求。
5. 启动顺序必须 `自上而下`。
6. 停止顺序必须 `请求自上而下，回收自下而上`。
7. 每个长期线程对象原则上都应具备 `启动 / 请求退出 / 等待线程结束 / 停止` 四段式接口。
8. `请求退出()` 必须幂等，且默认不阻塞。
9. `等待线程结束()` 只能由拥有者调用；遇到 self-join 时必须直接返回，不得转为 `detach`。
10. `detach` 不得作为常规生命周期管理方案。
11. 一旦进入 `停止请求中`，线程不得再接受新工作或启动新的直属子线程。
12. 线程的关键生命周期状态变化必须形成日志，至少记录线程标签、拥有者、旧状态、新状态、触发动作与原因摘要。
13. `请求退出`、`等待线程结束开始/完成`、`故障上报`、`安全重启请求` 不得长期静默处理。
14. 子线程故障后的重启、降级或停机决策，必须由拥有者完成。

# 设计执行双窗口交互规范

日期：2026-07-13

状态：已建立 / 文档治理 / 不承载业务机器事实 / 不构成代码实施许可

## 1. 目标

本规范让设计角色和执行角色围绕同一计划连续接力：默认由“设计计划窗口”和“执行计划窗口”两个物理窗口交接；用户明确指定时，也允许同一物理窗口按正式提交边界串行切换角色。两种模式都不依赖用户人工复制整段上下文，也不合并角色权限。

## 2. 权威层级

```text
Git 正式提交
-> 计划/计划索引.md
-> 项目记忆/Codex任务队列.md
-> 目标计划、流程图、详细设计和正式断点
-> 项目记忆/窗口交互记录.md
-> Codex 任务消息
-> 聊天摘要
```

任务消息只传递“去哪里读取什么”，不能替代计划、允许文件、禁止文件、验证方式或完成证据。

本文后续所称“设计窗口 / 执行窗口”，在默认双物理窗口模式下指对应物理窗口，在单窗口串行双角色模式下指当前物理窗口内互斥的设计角色 / 执行角色。角色权限和状态机不因物理窗口数量改变。

## 3. 交互状态机

| 状态 | 所有者窗口 / 角色 | 允许动作 | 下一状态 |
| --- | --- | --- | --- |
| `执行中` | 执行窗口 | 按计划复核、修改、验证 | `已完成` 或 `退回设计` |
| `退回设计` | 设计窗口 | 修订流程图、详细设计、计划和治理记录 | `设计修订已发布` |
| `设计修订已发布` | 执行窗口 | 重读提交、复核接口和 dirty state | `执行中` 或再次 `退回设计` |
| `已完成` | 设计窗口只读接收 | 更新后继依赖和下一设计路线 | 新计划或空闲 |

同一计划只能有一个所有者角色。执行窗口写出 `退回设计` 后不得继续改目标代码；设计窗口写出 `设计修订已发布` 后不得继续改同一计划，除非执行窗口再次退回。

### 3.1 模式选择与角色切换

```text
默认：设计物理窗口 <-> 执行物理窗口，提交推送后发送任务消息唤醒。
用户明确指定：同一物理窗口中的设计角色 <-> 执行角色，提交推送后从仓库权威状态重新接管，不向自己发送任务消息。
```

单窗口串行双角色模式下，任一时刻只激活设计、执行或只读复核一个角色。裸 `继续` 只沿当前已登记角色。角色切换前必须先写断点、计划索引、任务队列和窗口交互事件并完成验证，再把这些权威状态自动本地提交并自动非强制推送；只有推送成功后才切换角色并重新读取正式提交、目标计划、断点和实际接口。该模式持续到用户明确撤销或正式登记写入所有权转交。

## 4. 执行窗口退回协议

执行窗口发现接口漂移时必须按顺序完成：

```text
1. 停止代码实施并撤回未验证草案，保留用户或其它窗口改动。
2. 写实施前复核断点，列明当前接口、冲突、允许 / 禁止文件和零代码变化证据。
3. 更新决策记录、待确认问题、计划索引、任务队列、当前状态和窗口交互记录。
4. 运行文档治理验证。
5. 自动本地提交本次退回治理文件，并按项目持续授权自动执行非强制推送。
6. 目标是另一物理窗口时，推送成功后再向设计窗口发送交接消息；目标是当前物理窗口时，交互事件必须已随本轮提交推送，随后不发送自发自收消息，按已发布记录切换并重新读取。推送失败时只报告本地提交和待发布状态。
7. 继续检查其它依赖就绪项；没有可执行项时停止。
```

退回消息固定字段：

```text
交互类型：退回设计
计划 / 队列编号
提交号
发现的问题
禁止扩权边界
权威断点路径
设计窗口下一步
```

## 5. 设计窗口恢复协议

设计窗口收到退回消息后必须：

```text
1. 读取消息给出的提交号并确认当前分支包含该提交。
2. 重读计划索引、任务队列、目标计划、流程图、详细设计和正式断点。
3. 判断原计划内定向修订还是新增前置计划；不得只在消息中裁决。
4. 修订完整权力链，明确允许文件、禁止文件、入口拒绝、追根因收口和验证矩阵。
5. 更新原队列项为重新待执行或登记新的依赖门控项。
6. 运行验证、自动本地提交并按项目持续授权自动执行非强制推送。
7. 目标是另一物理窗口时，推送成功后再向执行窗口发送恢复消息；目标是当前物理窗口时，交互事件必须已随本轮提交推送，随后不发送自发自收消息，按已发布记录切换并重新读取。推送失败时不得宣称设计修订已发布。
```

恢复消息固定字段：

```text
交互类型：设计修订已发布
计划 / 队列编号
提交号
修订裁决
权威流程图 / 详细设计 / 计划 / 断点路径
执行窗口允许动作
仍然禁止的动作
后继门控
```

## 6. 执行窗口重新接管

执行窗口收到恢复消息后不得直接续写旧草案。必须重新读取正式提交和实际接口，确认：

```text
提交存在且分支一致。
目标队列项状态为重新待执行或依赖已满足待执行。
流程图、详细设计和计划范围一致。
允许文件覆盖真实缺口，禁止文件没有冲突。
本地 dirty state 不会污染切片。
实际符号、签名和阶段编号符合修订合同。
```

任一不一致再次退回设计，不得在执行窗口临时改计划。

## 7. 完成回传

切片完成后，必须先写完成记录和以“本轮推送成功”为生效条件的窗口交互事件并完成验证，再把这些权威状态自动本地提交并自动非强制推送。目标是另一物理窗口时，推送成功后执行窗口再向设计窗口回传；目标是当前物理窗口时，推送成功后按已发布交互事件重新读取，确认一致后才由设计角色接管。回传或交互事件必须包含计划编号、提交号、验证等级、正式实施记录、完成边界、解除的后继依赖和仍未完成能力。设计角色只更新设计路线和后继计划，不重复执行该切片。

## 8. Codex 任务工具规则

当应用提供任务列表和任务消息工具时：

```text
按精确任务标题和 cwd 定位唯一目标任务。
设计窗口默认向标题“执行计划”的同仓库任务发送恢复消息。
执行窗口使用交接消息中提供的设计任务标识回传。
不得向多个候选任务广播代码实施指令。
消息发送失败时记录失败，但仓库交互事件仍有效。
目标是当前物理窗口时，不发送自发自收任务消息；窗口交互记录和重新读取门禁仍然有效。
```

线程 / 任务标识是运行期路由材料，不写入正式计划和业务结构，不作为跨机器长期身份。

## 9. 并发与 Git 边界

```text
每轮任务完成先自动本地提交并自动非强制推送；推送成功后再发送正式交接消息。
接收窗口先读 Git 和 dirty state，再接管。
只暂存本切片文件，不带入其它窗口未提交改动。
两个窗口不得同时构建、运行或修改同一计划代码范围；同一物理窗口也不得让两个角色同时拥有写入权。
自动推送默认只允许快进到明确目标；禁止 force、自动合并、自动变基、自动删分支或自动发布。
消息、日志和聊天摘要不裁决计划完成。
```

## 11. 完成边界

本规范只证明设计角色与执行角色具备可恢复的治理交接协议，并允许用户明确指定时由同一物理窗口串行承接两个角色。它不证明角色权限已经合并，也不证明任一代码计划已经实施、构建通过、业务闭环完成或旧能力迁移完成。

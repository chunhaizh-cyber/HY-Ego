# 设计执行集成只读复核四角色交互规范

日期：2026-07-23

状态：长期非机器角色交互治理入口；未登记于 `规范/规范目录.md`；不构成代码实施许可

说明：为保持既有引用稳定，本文件继续沿用 `设计执行双窗口交互规范.md` 路径；正文权威范围已经扩展为四角色。

## 1. 目标

本入口把计划的设计、实施、集成发布和独立事实复核拆成权限互斥的设计、执行、集成、只读复核四个角色。“制定窗口”统一称为“设计窗口”。设计窗口先判定真实依赖，优先把提供者—消费者条件冻结成精确的待实现接口合同，并隔离文件、结构、接口和装配所有权；没有未合同化真实依赖且满足并行门禁的计划优先登记为并行批次。未登记并行批次时默认串行交接；登记并行批次后允许多个执行窗口在独立任务 worktree 中并发形成实现候选；用户明确指定时，也允许同一物理窗口按正式提交边界串行切换角色。

本入口不是 `规范/规范目录.md` 登记的机器逻辑规范，不定义机器事实或代码语义；其约束力来自 `AGENTS.md` 对角色边界、子智能体 S0、执行自主接管、交接状态机、整树切换和任务工具规则的指定入口。相关角色必须遵守本入口，但不得据此覆盖现行机器规范、补造业务规则或扩大计划许可。worktree 路径、分支、冻结、Git 身份、集成实施和回收的技术合同只由 `规范/多工作树并发与集成规范.md` 定义。

拆分目标：

```text
设计窗口只负责高不确定性的规范、设计、计划和计划修订，不指挥其它角色。
执行窗口负责自主发现自身范围内的成熟计划、完成 S0、实施和验证，并以结构化消息交接实际结果或设计缺口。
集成窗口负责核对任务提交范围、汇集提交、运行集成验证和发布 main。
只读复核窗口负责在指定提交和指定对象上独立核查事实、缺口与完成声明，不写仓库。
四个角色分别沿正式状态自主推进；设计窗口持续形成后续计划，不等待执行完成，也不承担调度者。
计划合同、任务状态、调度依赖、worktree 技术身份和实施事实分别由唯一文件拥有，避免同一可变状态被复制维护。
消息只负责唤醒，不替代仓库权威状态。
```

## 2. 交接证据读取顺序

本节只规定角色接管、状态核对和消息追溯时的证据读取顺序，不定义项目全局权威层级，也不得据此让当前代码、Git 状态、计划或消息覆盖正式规范。项目全局权威顺序、机器语义来源和代码实施依据链唯一由 `AGENTS.md` 定义。

```text
Git 正式提交和当前代码 / 实际接口
-> 项目记忆/任务状态台账.md
-> 计划/计划索引.md
-> 项目记忆/Codex任务队列.md
-> 项目记忆/并行工作树登记表.md
-> 目标计划、流程图和详细设计
-> 最新结构化执行回执，以及设计窗口据此形成的正式实施 / 冻结记录
-> 项目记忆/窗口交互记录.md
-> Codex 排队消息
-> 聊天摘要和窗口自身上下文
```

该顺序用于先核对当前实现与唯一任务生命周期，再判断计划导航、激活状态、依赖和技术身份是否一致。任务状态台账是当前生命周期唯一所有者；计划索引独占计划路径、版本和激活状态，任务队列只表达顺序与依赖，工作树登记只表达 Git 技术身份；结构化执行回执提供实际结果，设计窗口核对后形成需要的持久化记录。它们不一致时必须返回具名 DRIFT，不得用聊天记忆或其它 Markdown 的显示状态覆盖台账，也不得把实现现状反向升级为规范。

## 3. 文件写入所有权

设计窗口独占规范、流程图、详细设计、计划、计划索引、任务队列和 Markdown 设计记录的写入；对任务台账、工作树登记和事实投影的写入只负责持久化已经发生的事实，不产生任务执行权。执行窗口只在登记任务 worktree 中自主写计划允许的代码、工程和自检，并以结构化消息交接；集成窗口独占登记集成 worktree 的提交汇集、集成记录和主线发布；只读复核窗口无仓库写入权。

| 文件层 | 唯一写入角色 | 其它角色权限 |
| --- | --- | --- |
| `计划/计划索引.md`、`计划/*.md`、`计划/已完成计划/` | 设计窗口 | 执行窗口只读 |
| `项目记忆/任务状态台账.md`、`项目记忆/Codex任务队列.md`、`项目记忆/窗口交互记录.md`、`项目记忆/并行工作树登记表.md` | 设计窗口 | 执行窗口和集成窗口只读；执行角色只以结构化消息提供迁移证据 |
| 规范、流程图、详细设计、设计决策、当前状态和验收 / 归档汇总 | 设计窗口 | 执行窗口只读 |
| `项目记忆/设计记录/*` 中的计划生成、设计审查和规则治理记录 | 设计窗口 | 执行窗口只读 |
| 计划允许范围内的 C++、工程文件和自检 | 执行窗口 | 设计窗口只读当前事实 |
| 本次执行实施 / 冻结记录、执行前复核或退回断点 | 设计窗口 | 执行窗口只通过结构化消息提供事实，不得写 Markdown |
| 并行批次集成记录与集成暂存分支 | 集成窗口 | 设计窗口和执行窗口只读；冲突时退回，不代改业务语义 |
| 指定基线上的只读复核结论 | 只读复核窗口只输出消息报告 | 设计窗口读取并裁决；如需持久化，由设计窗口另写设计记录或登记纯文档计划 |
| `项目记忆/实施记录.md` 等汇总状态 | 设计窗口 | 根据正式实施记录更新，执行窗口不得直接写 |

设计窗口不得借治理身份修改代码、工程或自检，也不得向执行窗口发出开始、继续、停止、恢复、选择下一项或接受 S0 的指令；执行窗口不得新建、修改、移动、删除、暂存或提交任何 Markdown，也不得借完成、退回或修复状态修改计划、计划索引、任务队列、流程图、详细设计或设计治理状态；集成窗口不得借冲突处理发明接口、修改计划或扩大任务范围；只读复核窗口不得把报告直接写成计划完成、依赖解除或代码许可。旧执行专属记录只保留历史事实；本规范生效后，执行结果由结构化消息提供，设计窗口核对 Git 后只负责需要的设计修订和事实持久化。

### 3.1 顶层任务树单写角色

顶层任务是平台关系中不属于任何协作代理后代的独立用户可见任务，包括用户直接打开或创建的任务，以及用户授权后由 Codex 任务工具创建并交由用户独立接管的任务。一个顶层任务及其直接、递归创建的全部协作代理共同构成一个顶层任务树。顶层身份按平台父子关系和协作代理关系裁决，不按标题、消息来源、worktree、分支、模型、智能体名称或文件路径裁决。

一个顶层任务树同一时刻最多只能拥有设计、执行、集成中的一种写角色；只读复核是第四种协作角色，但不是写角色。全部有写入权的后代只能继承该写角色；只读后代可以并存，但必须保持零修改、零构建、零运行、零 Git 写入、零任务执行权和零发布占用。顶层任务只读时整树只能只读。worktree、分支、模型、推理等级、名称和递归层级均不能改变角色、范围、所有权或许可。

各角色可以分配给后代的切片固定为：

| 顶层任务树角色 | 同角色写切片 | 只读切片 | 禁止 |
| --- | --- | --- | --- |
| 设计 | 互不相交的规范、流程图、详细设计、计划和治理状态切片 | 规范冲突、代码事实、引用和差异核对 | 执行代码、构建、集成或发布 |
| 执行 | 计划允许且所有权不相交的代码、工程、自检或验证切片 | 接口、差异、日志和证据核对 | 修改任一 Markdown、规范、计划、中央队列或承担集成角色 |
| 集成 | 登记范围内不相交的汇集检查、集成记录和验证切片 | 父链、文件范围、验证输出和主线漂移核对 | 发明接口、修改计划、扩大范围或承担设计角色 |
| 只读 | 无 | 指定提交和问题集上的事实采集 | 任一后代修改、构建、运行或执行 Git 写操作 |

出现两个以上边界明确、无真实依赖且所有权可以隔离的具体子任务时，应优先使用同角色或只读后代并行。不使用时必须具名记录任务不可再分、真实前后依赖、所有权无法隔离、并发槽不足、工具只能串行、协调成本过高或风险不能隔离中的实际原因。不得制造空任务、重复读取、无意义代理链、相互等待或让多个智能体竞争同一原子步骤。

### 3.2 子智能体派发合同与 S0

每次派发协作代理时，派发方必须给出：

```text
顶层任务身份及平台父子 / 协作关系；
继承的设计、执行、集成写角色或只读身份；
具体目标、返回条件和唯一切片所有权；
允许文件、结构、接口、验证或证据范围；
明确禁止项，以及是否允许修改、构建和运行；
Git 暂存、提交和推送权限；
任务执行权、worktree 与主线发布占用权限；
冲突、漂移、失败、重叠或新用户裁决的停止条件。
```

协作代理开始写入或执行授权动作前必须完成最小 S0：

```text
确认所属顶层任务树、平台关系和继承角色；
确认目标、允许范围、禁止范围和返回条件；
确认目标路径、分支、HEAD 和 dirty state（适用时）；
确认与其它已派发切片没有所有权交集；
确认 Git、任务执行权、worktree 和发布占用权限；
向派发方返回 PASS 或具名 DRIFT。
```

未完成 S0 或 S0 为 DRIFT 时不得开始写入。递归派发必须原样传递顶层身份、角色、范围、所有权、Git / 占用权限和停止条件，不得在下级派发中扩大权限。任务消息只负责唤醒，聊天回执不得替代仓库身份、登记事实或完成事实。

### 3.3 所有权、串行收口与违规停止

1. 顶层任务树是任务和 worktree 的正式协作所有者；协作代理只取得派发合同列出的具体切片所有权。
2. 同一文件、结构、接口、验证项目或证据裁决同一时刻只有一个智能体负责。发现交集时必须先重新拆分或改为串行。
3. Git index、暂存、提交、推送、任务执行权迁移和主线发布占用迁移，由顶层智能体或唯一指定的同角色智能体串行完成；收口期间其它后代停止写入。
4. 顶层智能体必须复核后代的实际差异和验证证据；后代返回只证明其受派切片的结果，不证明顶层任务、集成或发布完成。
5. 范围漂移、所有权重叠、角色冲突、计划外文件、验证失败或用户新裁决只停止受影响切片并上报；不得自行扩权。若同一顶层任务树已同时拥有两种写角色或通过 worktree、分支、模型、名称规避角色，整树立即停止新增写入，保留现有 Git 事实并退回设计治理追查根因。
6. 无关 WIP 属于原所有者，任何后代不得暂存、清理、stash、覆盖或带入本任务提交。

## 4. 状态机

每个任务当前生命周期只允许写入 `项目记忆/任务状态台账.md` 的唯一当前行；状态迁移在同一治理提交内原子更新该行并向窗口交互记录追加不可变事件。叶子计划、计划索引、任务队列、工作树登记和当前状态不得另存一份可变当前状态。设计窗口负责写 Markdown，因此可以把已经发生且有 Git / 结构化消息证明的执行事实登记到台账；这种登记是事实持久化，不是执行许可、调度决定或 S0 接受。

| 状态 | 事实形成角色 | 含义 | 下一状态 |
| --- | --- | --- | --- |
| `待执行` | 设计窗口 | 精确计划版本 / blob 已完成设计并为 `可执行`；该状态只表达计划可被执行窗口自主发现 | 执行窗口自行 S0 PASS 并实际开始后转 `执行中`；发现真实依赖时转 `依赖门控待执行` |
| `依赖门控待执行` | 设计窗口 | 计划已形成，但具名真实依赖尚未满足 | 设计修订或条件成立后转 `待执行` |
| `已派发待执行回执` | 历史兼容 | 旧治理协议留下的状态；新事件不得以此产生执行权 | 迁移为 `待执行`，或由既存执行事实闭合到 `执行中` |
| `调度权越界退回` | 用户裁决 / 治理记录 | 旧设计窗口越权开始、继续、停止或接受 S0 的语义回退中间态 | 必须立即闭合到 `待执行`；不得抹除已有 Git 事实 |
| `执行中` | 执行窗口 | 执行窗口已经自行核对 S0 并在计划白名单内形成实际代码现场 | `分支完成待集成`、`接口漂移待设计修订`、`验证失败待设计裁决` |
| `分支完成待集成` | 执行窗口 | 任务分支候选已经按计划收口并交接；完整批次验证可以按计划后置最终汇合 | 集成角色自主接管后转 `集成中` |
| `已预留待集成派发`、`已派发待集成回执` | 历史兼容 | 旧集成派发协议留下的状态；不得由设计窗口继续使用为命令 | 集成角色按当前协议自行 S0，或语义回退 |
| `集成中` | 集成窗口 | 集成窗口自行核对身份、取得唯一主线发布占用并开始固定输入汇集 | `已集成待设计同步`、`集成分支验证通过待主线发布`、`集成失败退回任务`、`集成身份冲突待设计重建` |
| `接口漂移待设计修订` | 执行窗口 | 执行窗口停止受影响计划并提交精确设计缺口 | 设计形成新版本后转 `待执行`，或登记真实依赖 |
| `验证失败待设计裁决` | 执行窗口 | 计划内验证失败且执行窗口不能在原合同内安全收口 | 设计修订后转 `待执行`、登记前置计划或关闭旧版本 |
| 其它集成、复核和归档状态 | 对应角色 | 沿用既有事实含义，但设计窗口只能记录或修订设计，不能用消息指挥对应角色 | 按已发生事实迁移 |

历史事件不可改写。旧事件中的“派发”“设计接受 S0”和 `PASS已接受` 只说明当时采用的旧治理协议；自本版生效后不再构成当前或未来执行许可。

### 4.1 计划激活状态与任务生命周期分账

`计划/计划索引.md` 是计划设计可用性的唯一所有者，枚举固定为：

```text
待激活：精确计划版本 / blob 正在等待设计修订，执行窗口本轮跳过。
可执行：精确计划版本 / blob 已完成设计，可由已拥有对应任务 / worktree 的执行窗口自主核对。
已退出：该计划版本不再参与后续消费，只作历史导航。
```

`可执行`不是命令，不代表已经开始、S0 PASS、已取得任务所有权或已经完成。执行窗口报告缺口后，设计窗口把旧版本改为 `待激活`，形成新版本 / blob 后恢复 `可执行`；执行窗口通过重读仓库自行发现变化，设计窗口不得发送继续、停止或选择下一项指令。

## 5. 设计窗口设计发布协议

设计窗口只完成以下工作：

```text
1. 生成或修订规范、流程图、详细设计、知识图谱和计划。
2. 核对依赖必要性、合同化、所有权隔离、允许 / 禁止文件、入口拒绝、失败收口和验证方式。
3. 在计划索引登记稳定路径、版本、计划 blob 与 `待激活 / 可执行 / 已退出`；任务队列只登记静态依赖和解除条件。
4. 需要预冻结接口时，登记模块、DTO、签名、ABI、所有权、生命周期、逻辑内结果、内部错误、提供者、消费者和漂移退回点。
5. 运行文档治理验证，精确提交并非强制推送设计文件。
6. 继续设计下一份计划，或优先响应已经收到的设计修订请求。
```

到第 5 步设计工作已经完成。设计窗口不得继续执行下列动作：

```text
向执行窗口发送开始、继续、停止、恢复、选择下一计划或接受 S0 的消息；
把任务消息送达、目标 active、worktree 预留或台账写入解释为执行许可；
要求执行窗口等待设计窗口接受 PASS；
因为前一计划尚未执行而停止形成后续无真实依赖计划；
用“派发准备”“派发状态”或具名执行序列控制执行时点。
```

设计窗口可以回答执行窗口提出的计划含义问题、提供正式路径或确认已经发布的设计版本，但回复必须明确“仅为设计事实说明，不构成开始、继续、停止或计划选择指令”。

## 6. 执行窗口自主接管与下一计划选择

执行窗口不等待设计消息，只在已经登记给自身的任务 / worktree 和文件所有权范围内按下列顺序工作：

```text
重新读取 Git、当前 worktree / 分支、基线和 dirty state
-> 读取计划索引、任务队列、任务状态台账和工作树登记
-> 从自身所有权范围内排除 `待激活`、`已退出`、真实依赖未满足和版本 / blob 不一致项
-> 自主选择一份 `可执行` 计划；多个无依赖计划可由不同执行任务并行
-> 核对同任务没有第二有效执行占用，核对允许 / 禁止文件和实际接口
-> 自行完成角色、Git、计划 blob、所有权和接口 S0
-> PASS：直接实施并发送事实消息；不等待设计接受
-> DRIFT：停止该计划、发送精确修订请求，然后检查其它无关 `可执行` 计划
```

执行窗口不得夺取其它任务、worktree、文件或结构所有权，也不得把设计窗口的消息、聊天解释或旧 `PASS已接受` 标签当作当前许可。同一执行窗口串行处理多份计划时，上一份正常收口或因 DRIFT 已停止后即可自行检查下一份；不需要设计窗口确认冻结或再次通知。

## 7. 执行完成与退回协议

执行窗口发现接口漂移、范围越界、验证失败或完成切片时，不得修改任何 Markdown、计划和路由文件。必须：

```text
1. 停止扩大受影响实施范围。
2. 运行计划在当前阶段要求且执行角色获准的验证；完整构建 / 运行 / 闭环验收明确后置时只做候选级最低检查。
3. 只提交执行角色允许的代码、工程和自检，并非强制推送同名任务分支；不得暂存或提交 Markdown。
4. 向设计窗口发送结构化完成、漂移、验证失败或计划修订请求，作为设计修订与事实登记输入。
5. 自主检查其它无关 `可执行` 计划，不等待设计窗口回复；只有当前所需设计产物尚未形成时才跳过该计划。
```

结构化执行消息至少包含计划 / 队列编号和版本、计划 blob、worktree / 分支 / 基线、实际接口、实际变更文件、验证结果、提交 / 远端 / dirty state、完成边界和需要修订的精确合同。消息发送失败时保留 Git 现场并幂等重试；不得为制造持久化回执而写 Markdown。

## 8. 设计窗口响应协议

设计窗口收到执行消息后只执行设计职责：

```text
1. 只读核对消息、Git、计划、实际接口和影响范围。
2. 需要修订时把旧计划版本改为 `待激活`，形成新版本 / blob，并在完成后恢复 `可执行`。
3. 需要新增前置计划时登记真实依赖；可合同化时冻结接口并取消开发启动门控。
4. 把已经发生的任务事实写入唯一台账和事件记录；该写入不得改变执行窗口的实际权限。
5. 运行文档治理验证，精确提交并非强制推送。
6. 继续总目标下其它计划设计，不等待执行窗口完成上一项。
```

设计窗口不得在响应结尾通知执行窗口“继续下一项”、要求其停止、指定执行顺序或接受 S0。执行窗口通过重读正式仓库自行发现新计划版本和 `可执行` 状态。

### 8.1 计划持续推进闭环

```text
设计窗口持续形成并发布计划
||
执行窗口自主消费自身范围内的可执行计划并返回事实 / 缺口
||
集成窗口按固定输入和主线占用规则自主汇合
-> 设计窗口只根据新事实继续修订规范、设计和后续计划
```

并发的目标是避免空等待：设计窗口不等待代码执行结束才设计下一份；执行窗口不等待设计窗口回复才检查其它无关计划。任何角色都不能因此跨越自身写角色、真实依赖、文件所有权、主线占用或验证边界。

## 9. 消息规则

执行窗口向设计窗口发送的完成、漂移、失败和修订请求只提供事实与设计输入。设计窗口向执行窗口的回复只允许说明正式设计路径、版本、blob 或问题澄清，必须明确不构成执行命令。客户端 Steer / Queue、消息送达顺序、目标 active 状态和聊天回执均不得改变当前计划范围或执行权。

正式并行批次运行期间，设计窗口和执行窗口可以在文件与 Git 身份完全隔离时同时落盘；任一所有权边界失效只停止受影响任务。集成任务取得主线发布占用后，设计窗口保持 WT-MAIN 只读，直到集成角色实际释放占用。

## 10. 同一物理窗口串行双角色

只有用户明确启用时才能进入单窗口串行切换；该模式以 `AGENTS.md` 定义的整棵顶层任务树为单位，全部后代必须先停止并释放旧角色占用，不能让顶层窗口和后代分别保留不同写角色。单窗口模式不合并角色权限：

```text
执行角色完成或退回
-> 只提交计划允许的代码 / 工程 / 自检并发送结构化消息
-> 验证、提交、推送
-> 推送成功后切换设计角色并重读全部正式状态
-> 由设计角色更新路由

设计角色完成设计修订或事实记录
-> 原子更新任务状态台账并追加窗口事件；其它职责文件仅按实际变化更新
-> 验证、提交、推送
-> 推送成功后切换执行角色并重读全部正式状态
```

执行转设计以结构化消息和已推送任务提交为依据；设计转执行只允许由用户明确启用整树串行切换，并以已推送设计提交、不可变计划 blob 和 worktree 技术登记为重读依据。切换后的执行角色自行选择当前所有权内的 `可执行` 计划并完成 S0，不得把先前设计角色的上下文、消息或状态记录当作开始命令。需要继续使用子智能体时，只能按新角色重新建立同角色 / 只读所有权；未验证草案不得跨角色继承。

## 11. 模型路由与异常升级

默认建议：

| 工作类型 | 模型 / 推理建议 |
| --- | --- |
| 设计、计划审查、接口漂移和逻辑异常裁决 | Sol + High，或可用的同等高推理配置 |
| 成熟计划常规实施、验证和结构化回执 | Terra + Medium，或可用的同等中等推理配置 |
| 明确、重复、只读的信息整理 | Luna / 低消耗等效配置 |

模型选择不改变角色权限。执行窗口发现接口漂移、计划冲突、事务 / 并发 / 因果边界无法由计划裁决、内部逻辑错误，或验证重复失败且根因不明时，必须停止并退回高推理的设计窗口；不得只通过提高执行窗口模型等级自行改计划或扩大范围。

### 11.1 中等模型可机械执行合同

面向中等推理模型的成熟计划必须按以下四列写清每个阶段：

```text
已验证前置事实
本阶段待实施目标
实施完成的可核对证据
只有前置证据成立后才可判断的完成 / 退回条件
```

硬规则：

1. 待实施目标不存在，默认表示该阶段尚未完成，不表示接口漂移、计划冲突或真实阻塞。
2. 退回条件必须引用已经形成的具体文件、符号、构建输出、具名失败项或接口差异；不得用“证据不足”“代码较长”“尚未形成目标产物”替代前置证据。
3. 同一输出在目标实施前只可作为基线输入，在目标实施后才可作为验收证据；不得把实施前失败套入实施后退回门禁。
4. 执行模型误读明确阶段、跳过授权步骤、把目标当门禁或连续两轮不能按检查表推进，执行窗口返回模型能力不匹配事实；设计窗口只收窄或澄清计划，不指定执行窗口继续或切换模型。
5. 执行轮次因平台边界结束但尚未命中计划的正式完成 / 退回条件时，执行窗口自行保留精确续点并保持 `执行中`。不得仅因轮次结束写 `分支完成待集成`、`接口漂移待设计修订` 或 `验证失败待设计裁决`。

高推理模型进入执行窗口后仍只是执行角色，只能实施已发布计划；不得修改计划、索引、队列或借模型等级扩大范围。

## 12. Codex 任务工具规则

```text
设计窗口只可用任务工具读取执行事实，或回复执行窗口主动提出的设计问题。
设计窗口不得用任务工具发送开始、继续、停止、恢复、计划选择、S0 接受或模型切换指令。
执行窗口通过重读正式仓库发现计划版本和 `可执行` 状态；消息不是计划发现或执行许可的前置。
运行中的设计说明消息不得用 Steer 改变当前切片。
线程 / 任务标识只作运行期路由材料，不作为正式计划身份。
消息失败不改变已推送设计文件的效力，也不授权任何角色扩大范围。
协作代理只能继承当前顶层任务树的写角色或保持只读。
```

## 13. worktree 与 Git 的角色入口

本规范不定义 worktree 物理路径、分支格式、冻结基线、Git 身份算法、集成汇集步骤、失败保留或回收条件，只规定它们进入角色交接时的门禁：

1. 设计计划引用既有 worktree 时必须核对多工作树规范要求的正式登记和 Git 身份证据，但不得据此产生执行许可。
2. 执行窗口在本规范角色 S0 与多工作树规范 Git 身份复核都通过后自主进入执行写状态；集成窗口还必须核对内容冻结基线、分支建立起点和中央路由尾，并自行取得唯一主线发布占用。
3. Git index、暂存、提交、推送和主线发布占用仍由本顶层任务树的唯一指定智能体串行收口；其它后代在收口期间停止写入。
4. 路径、分支、基线、父链、dirty state 或远端状态漂移时，先接受多工作树规范的技术失败结论，再按本规范进入退回、重建或待设计裁决状态；消息不得把漂移改写为成功。

上述技术合同唯一见 `规范/多工作树并发与集成规范.md`；本文件不复制其分支命名、身份核对、冻结、集成、验证、失败和回收正文。

## 14. 验证

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

顶层任务树收口还必须核对：全部后代已经结束或明确保持只读；树内没有两种写角色；所有权交集为零或已串行消解；Git 收口只包含本轮目标文件；无关 WIP 未被暂存或改写；提交、推送和完成声明与实际证据一致。

若本轮同时修改技能，还必须按对应技能文件中真实存在的专项验证入口执行；不存在的脚本不得写成强制验证。规则文档验证不证明代码编译、入口拒绝、业务闭环或任何代码能力完成。

## 15. 完成边界

本规范只证明设计、执行、集成和只读复核四角色的写入所有权、自主接管、交接状态、消息边界、主线发布占用、独立复核和计划持续推进规则已经建立。worktree 分配、Git 身份、集成实施、失败保留和回收由多工作树规范唯一裁决。它不证明任何代码计划已经实施、构建通过、业务闭环完成或旧项目能力已经等价重建。

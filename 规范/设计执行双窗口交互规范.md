# 设计执行双窗口交互规范

日期：2026-07-17

状态：已修订 / 文档治理 / 不承载业务机器事实 / 不构成代码实施许可

## 1. 目标

本规范把计划制定和计划实施拆成两个权限互斥的角色。“制定窗口”统一称为“设计窗口”。默认由设计窗口和执行窗口两个物理窗口交接；用户明确指定时，也允许同一物理窗口按正式提交边界串行切换角色。

拆分目标：

```text
设计窗口负责高不确定性的设计、计划、路由和异常裁决。
执行窗口负责按成熟计划实施、验证和记录实际结果。
计划事实与实施事实分别写入，避免两个窗口争写同一文件。
消息只负责唤醒，不替代仓库权威状态。
```

## 2. 权威层级

```text
Git 正式提交和当前代码 / 实际接口
-> 计划/计划索引.md
-> 项目记忆/Codex任务队列.md
-> 目标计划、流程图和详细设计
-> 最新正式实施记录 / 执行复核断点
-> 项目记忆/窗口交互记录.md
-> Codex 排队消息
-> 聊天摘要和窗口自身上下文
```

计划索引和任务队列表达设计路由；实施记录表达实际做了什么和验证结果。两者尚未同步时必须明确标为中间状态，不得用聊天记忆覆盖任一方。

## 3. 文件写入所有权

设计窗口独占计划、计划索引、任务队列、窗口交互和设计治理状态的写入；执行窗口独占计划允许范围内的代码实施与本次执行专属实施记录写入。

| 文件层 | 唯一写入角色 | 其它角色权限 |
| --- | --- | --- |
| `计划/计划索引.md`、`计划/*.md`、`计划/已完成计划/` | 设计窗口 | 执行窗口只读 |
| `项目记忆/Codex任务队列.md`、`项目记忆/窗口交互记录.md` | 设计窗口 | 执行窗口只读 |
| 规范、流程图、详细设计、设计决策、当前状态和验收 / 归档汇总 | 设计窗口 | 执行窗口只读 |
| `项目记忆/设计记录/*` 中的计划生成、设计审查和规则治理记录 | 设计窗口 | 执行窗口只读 |
| 计划允许范围内的 C++、工程文件和自检 | 执行窗口 | 设计窗口只读当前事实 |
| 本次执行专属 `实施记录/*`、执行前复核或退回断点 | 执行窗口 | 设计窗口读取后同步路由 |
| `项目记忆/实施记录.md` 等汇总状态 | 设计窗口 | 根据正式实施记录更新，执行窗口不得直接写 |

设计窗口不得借治理身份实施代码；执行窗口不得借完成、退回或修复状态修改计划、计划索引、任务队列、流程图、详细设计或设计治理状态。旧 `实施记录/*计划生成*` 和设计审查断点只保留历史写入事实；本规范生效后，新的计划生成 / 设计审查记录改写入 `项目记忆/设计记录/`，执行专属记录继续写入 `实施记录/`。

## 4. 状态机

| 状态 | 写入角色 | 含义 | 下一状态 |
| --- | --- | --- | --- |
| `依赖门控待执行` | 设计窗口 | 计划已生成，但依赖未满足 | `已派发待执行回执` 或继续门控 |
| `已派发待执行回执` | 设计窗口 | 计划、版本、基线提交和目标执行角色已发布 | `执行中`、`接口漂移待设计修订` |
| `执行中` | 执行窗口 | 正在按已发布计划实施 | `执行完成待设计同步`、`接口漂移待设计修订` |
| `执行完成待设计同步` | 执行窗口 | 代码、验证和实施记录已提交推送，路由尚未更新 | `已完成 / 已归档` 或设计裁决的后继状态 |
| `接口漂移待设计修订` | 执行窗口 | 计划与实际接口、范围或前置条件不一致 | `重新待执行`、新前置计划或真实阻塞 |
| `重新待执行` | 设计窗口 | 修订已发布并重新派发 | `执行中` 或再次退回 |
| `已完成 / 已归档` | 设计窗口 | 已读取正式实施证据并完成路由同步 | 后继计划或空闲 |

设计窗口未处理 `执行完成待设计同步` 或 `接口漂移待设计修订` 前，不得重复派发同一计划，也不得把依赖该计划的后继项自行解释为已解除门控。

## 5. 设计窗口派发协议

设计窗口必须按顺序完成：

```text
1. 生成或修订流程图、详细设计和计划。
2. 核对计划版本、依赖、允许文件、禁止文件、入口拒绝、失败收口和验证方式。
3. 更新计划索引和任务队列为“已派发待执行回执”。
4. 记录目标执行角色、计划版本、派发基线提交和是否属于预授权执行批次。
5. 运行文档治理验证。
6. 只提交本轮设计文件并自动非强制推送。
7. 推送成功后向执行窗口发送排队提醒；消息不形成执行许可。
```

派发消息固定字段：

```text
交互类型：计划已派发
计划 / 队列编号
计划版本
派发提交号
权威计划路径
允许 / 禁止范围
是否属于预授权执行批次
执行窗口必须重新读取的材料
```

## 6. 执行窗口接管与下一计划选择

执行窗口不得只凭自身上下文、消息到达顺序或计划索引中的单一字段选择计划。每次开始新的执行轮次必须重新计算：

```text
重新读取 Git、当前分支和 dirty state
-> 读取计划索引和任务队列
-> 读取候选计划、最新实施记录和执行复核断点
-> 排除已完成、待设计同步、退回设计和版本不一致项
-> 按队列优先级、依赖和显式顺序筛选
-> 复核允许 / 禁止文件和实际接口
-> 只有唯一候选全部通过才进入执行
```

出现以下任一情况必须停止并回传设计窗口：

```text
多个同优先级候选且索引没有裁决顺序。
计划索引、任务队列和最新实施记录相互冲突。
计划版本或派发提交与当前正式状态不一致。
前置计划刚完成但仍处于“执行完成待设计同步”。
实际接口与假定接口不一致。
候选计划会接触其它未收口切片的文件。
```

默认一个计划完成或退回后先交回设计窗口。只有设计窗口明确登记“预授权执行批次”，且批次内计划互相独立、无文件范围冲突、候选唯一时，执行窗口才可在完成当前项后继续计算并消费下一项。

## 7. 执行完成与退回协议

执行窗口发现接口漂移、范围越界或完成切片时，不得修改计划和路由文件。必须按顺序完成：

```text
1. 停止扩大当前实施范围。
2. 写本次执行专属实施记录或复核断点。
3. 完成时标记“执行完成待设计同步”；漂移时标记“接口漂移待设计修订”。
4. 记录计划编号 / 版本、派发提交、实际接口、修改文件、验证命令 / 结果、实施提交和完成边界。
5. 运行计划要求的验证。
6. 只提交执行角色允许的代码、工程、自检和实施记录，并自动非强制推送。
7. 推送成功后向设计窗口发送完成或退回摘要，然后停止当前计划。
```

实施记录 / 复核断点必须至少包含：

```text
计划 / 队列编号和版本
派发基线提交
实际读取的接口与提交
执行状态
修改文件
验证命令与结果
实施提交号
完成声明边界
接口漂移、未解决问题和建议设计动作
```

消息发送失败时，已推送的实施记录 / 复核断点就是持久化回执。设计窗口下次启动必须扫描最新记录，不得要求执行窗口越权修改计划索引、任务队列或窗口交互记录。

## 8. 设计窗口同步协议

设计窗口收到完成、退回或消息失败提示后必须：

```text
1. 读取实施提交并确认当前分支包含该提交。
2. 重读计划、计划索引、任务队列、最新实施记录、代码变化和实际接口。
3. 判断完成归档、原计划定向修订、新增前置计划或真实阻塞。
4. 更新计划、计划索引、任务队列、窗口交互、当前状态、决策和必要验收汇总。
5. 运行文档治理验证，自动本地提交并自动非强制推送。
6. 需要继续时重新派发；不得仅凭消息正文或执行窗口自述标记完成。
```

## 9. 运行中消息的排队规则

执行窗口正在执行时收到的新计划消息，必须使用排队语义：

```text
只记住“有新正式状态需要下一轮重读”。
不改变当前计划范围。
不打断当前验证和提交链。
不按消息到达顺序决定下一个计划。
当前计划收口后重新读取仓库，再计算唯一候选。
```

消息若被客户端设置为 Steer，也必须在提示中明确“不得中断当前计划、不得形成执行许可”，并把实际处理推迟到下一轮。推荐把客户端 Follow-up behavior 设为 Queue。

当前单一 `main` 工作树规则不允许设计窗口在执行窗口写代码期间并发落盘新计划。新可执行计划必须在执行窗口取得写入权前已经发布，或等待执行窗口提交推送并释放写入权后再由设计窗口正式登记。

## 10. 同一物理窗口串行双角色

单窗口模式不合并角色权限：

```text
执行角色完成或退回
-> 只写实施记录 / 复核断点
-> 验证、提交、推送
-> 推送成功后切换设计角色并重读全部正式状态
-> 由设计角色更新路由

设计角色完成同步或派发
-> 更新计划、索引、队列和窗口交互记录
-> 验证、提交、推送
-> 推送成功后切换执行角色并重读全部正式状态
```

执行转设计以已推送实施记录 / 复核断点为依据；设计转执行以已推送计划索引、任务队列和窗口交互记录为依据。未验证草案不得跨角色继承。

## 11. 模型路由与异常升级

默认建议：

| 工作类型 | 模型 / 推理建议 |
| --- | --- |
| 设计、计划审查、接口漂移和逻辑异常裁决 | Sol + High，或可用的同等高推理配置 |
| 成熟计划常规实施、验证和记录 | Terra + Medium，或可用的同等中等推理配置 |
| 明确、重复、只读的信息整理 | Luna / 低消耗等效配置 |

模型选择不改变角色权限。执行窗口发现接口漂移、计划冲突、事务 / 并发 / 因果边界无法由计划裁决、内部逻辑错误，或验证重复失败且根因不明时，必须停止并退回高推理的设计窗口；不得只通过提高执行窗口模型等级自行改计划或扩大范围。

## 12. Codex 任务工具规则

```text
按精确任务标题、任务标识和 cwd 定位唯一目标任务。
发送消息时列明计划编号、提交号、状态和权威路径。
运行中的新计划消息采用 Queue，不用 Steer 改变当前切片。
不得向多个候选任务广播实施指令。
线程 / 任务标识只作运行期路由材料，不作为正式计划身份。
消息失败不改变已推送仓库状态的效力。
```

## 13. 并发与 Git 边界

```text
唯一可写工作树仍为 D:\海中鱼巣，唯一可写分支仍为 main。
任一时刻只有一个物理窗口或一个单窗口角色持有写入权。
每轮任务完成先自动本地提交并自动非强制推送，再交接。
只暂存本轮文件，不带入其它窗口、用户或中断任务的未提交改动。
禁止 force、自动合并、自动变基、自动删分支或自动发布。
消息、日志和聊天摘要不裁决计划完成。
```

## 14. 验证

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

若本轮同时修改计划或执行技能，再分别运行其 `quick_validate.py`。规则文档验证不证明代码编译、入口拒绝、业务闭环或旧能力迁移完成。

## 15. 完成边界

本规范只证明设计窗口和执行窗口的写入所有权、交接状态、消息排队、下一计划选择和模型路由规则已经建立。它不证明任何代码计划已经实施、构建通过、业务闭环完成或旧能力迁移完成。

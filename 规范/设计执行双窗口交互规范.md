# 设计执行集成只读复核四角色交互规范

日期：2026-07-21

状态：已修订 / 四角色项目闭环 / 文档治理 / 不承载业务机器事实 / 不构成代码实施许可

说明：为保持既有引用稳定，本文件继续沿用 `设计执行双窗口交互规范.md` 路径；正文权威范围已经扩展为四角色。

## 1. 目标

本规范把计划的设计、实施、集成发布和独立事实复核拆成权限互斥的设计、执行、集成、只读复核四个角色。“制定窗口”统一称为“设计窗口”。设计窗口先判定真实依赖并隔离文件、结构、接口和装配所有权；没有真实依赖且满足并行门禁的计划优先登记为并行批次。未登记并行批次时默认串行交接；登记并行批次后允许多个执行窗口在独立任务 worktree 中并发；用户明确指定时，也允许同一物理窗口按正式提交边界串行切换角色。

拆分目标：

```text
设计窗口负责高不确定性的设计、计划、路由和异常裁决。
执行窗口负责按成熟计划实施、验证和记录实际结果。
集成窗口负责核对任务提交范围、汇集提交、运行集成验证和发布 main。
只读复核窗口负责在指定提交和指定对象上独立核查事实、缺口与完成声明，不写仓库。
设计窗口负责把四个角色串成持续推进闭环，不能只生成计划或预留 worktree 后等待人工提醒。
计划事实与实施事实分别写入，避免两个窗口争写同一文件。
消息只负责唤醒，不替代仓库权威状态。
```

## 2. 交接证据读取顺序

本节只规定角色接管、状态核对和消息追溯时的证据读取顺序，不定义项目全局权威层级，也不得据此让当前代码、Git 状态、计划或消息覆盖正式规范。项目全局权威顺序、机器语义来源和代码实施依据链唯一由 `AGENTS.md` 定义。

```text
Git 正式提交和当前代码 / 实际接口
-> 计划/计划索引.md
-> 项目记忆/Codex任务队列.md
-> 项目记忆/并行工作树登记表.md
-> 目标计划、流程图和详细设计
-> 最新正式实施记录 / 执行复核断点
-> 项目记忆/窗口交互记录.md
-> Codex 排队消息
-> 聊天摘要和窗口自身上下文
```

该顺序用于先核对当前实现与交接事实，再判断设计路由是否已经同步。计划索引和任务队列表达设计路由；实施记录表达实际做了什么和验证结果。两者尚未同步时必须明确标为中间状态，不得用聊天记忆覆盖任一方，也不得把实现现状反向升级为规范。

## 3. 文件写入所有权

设计窗口独占计划、计划索引、任务队列、窗口交互和设计治理状态的写入，并独占执行、集成、只读复核任务的登记、派发、回执消费和后继路由；执行窗口独占其登记任务 worktree 中计划允许范围内的代码实施与本次执行专属实施记录写入；集成窗口独占登记集成 worktree 的提交汇集、集成记录和主线发布；只读复核窗口无仓库写入权。

| 文件层 | 唯一写入角色 | 其它角色权限 |
| --- | --- | --- |
| `计划/计划索引.md`、`计划/*.md`、`计划/已完成计划/` | 设计窗口 | 执行窗口只读 |
| `项目记忆/Codex任务队列.md`、`项目记忆/窗口交互记录.md`、`项目记忆/并行工作树登记表.md` | 设计窗口 | 执行窗口和集成窗口只读；集成窗口以专属集成记录回传状态 |
| 规范、流程图、详细设计、设计决策、当前状态和验收 / 归档汇总 | 设计窗口 | 执行窗口只读 |
| `项目记忆/设计记录/*` 中的计划生成、设计审查和规则治理记录 | 设计窗口 | 执行窗口只读 |
| 计划允许范围内的 C++、工程文件和自检 | 执行窗口 | 设计窗口只读当前事实 |
| 本次执行专属 `实施记录/*`、执行前复核或退回断点 | 执行窗口 | 设计窗口读取后同步路由 |
| 并行批次集成记录与集成暂存分支 | 集成窗口 | 设计窗口和执行窗口只读；冲突时退回，不代改业务语义 |
| 指定基线上的只读复核结论 | 只读复核窗口只输出消息报告 | 设计窗口读取并裁决；如需持久化，由设计窗口另写设计记录或登记纯文档计划 |
| `项目记忆/实施记录.md` 等汇总状态 | 设计窗口 | 根据正式实施记录更新，执行窗口不得直接写 |

设计窗口不得借治理身份实施代码；执行窗口不得借完成、退回或修复状态修改计划、计划索引、任务队列、流程图、详细设计或设计治理状态；集成窗口不得借冲突处理发明接口、修改计划或扩大任务范围；只读复核窗口不得把报告直接写成计划完成、依赖解除或代码许可。旧 `实施记录/*计划生成*` 和设计审查断点只保留历史写入事实；本规范生效后，新的计划生成 / 设计审查记录改写入 `项目记忆/设计记录/`，执行专属记录继续写入 `实施记录/`。

### 3.1 顶层任务树单写角色

顶层任务是平台关系中不属于任何协作代理后代的独立用户可见任务，包括用户直接打开或创建的任务，以及用户授权后由 Codex 任务工具创建并交由用户独立接管的任务。一个顶层任务及其直接、递归创建的全部协作代理共同构成一个顶层任务树。顶层身份按平台父子关系和协作代理关系裁决，不按标题、消息来源、worktree、分支、模型、智能体名称或文件路径裁决。

一个顶层任务树同一时刻最多只能拥有设计、执行、集成中的一种写角色；只读复核是第四种协作角色，但不是写角色。全部有写入权的后代只能继承该写角色；只读后代可以并存，但必须保持零修改、零构建、零运行、零 Git 写入、零任务执行权和零发布占用。顶层任务只读时整树只能只读。worktree、分支、模型、推理等级、名称和递归层级均不能改变角色、范围、所有权或许可。

各角色可以分配给后代的切片固定为：

| 顶层任务树角色 | 同角色写切片 | 只读切片 | 禁止 |
| --- | --- | --- | --- |
| 设计 | 互不相交的规范、流程图、详细设计、计划和治理状态切片 | 规范冲突、代码事实、引用和差异核对 | 执行代码、构建、集成或发布 |
| 执行 | 计划允许且所有权不相交的代码、工程、自检、实施记录或验证切片 | 接口、差异、日志和证据核对 | 修改规范、计划、中央队列或承担集成角色 |
| 集成 | 登记范围内不相交的汇集检查、集成记录和验证切片 | 父链、文件范围、验证输出和主线漂移核对 | 发明接口、修改计划、扩大范围或承担设计角色 |
| 只读 | 无 | 指定提交和问题集上的事实采集 | 任一后代修改、构建、运行或执行 Git 写操作 |

出现两个以上边界明确、无真实依赖且所有权可以隔离的具体子任务时，应优先使用同角色或只读后代并行。不使用时必须具名记录任务不可再分、真实前后依赖、所有权无法隔离、并发槽不足、工具只能串行、协调成本过高或风险不能隔离中的实际原因。不得制造空任务、重复读取、无意义代理链、相互等待或让多个智能体竞争同一原子步骤。

### 3.2 子智能体派发合同与 S0

每次派发协作代理时，派发方必须给出：

```text
顶层任务身份及平台父子 / 协作关系；
继承的设计、执行、集成写角色或只读身份；
具体目标、返回条件和唯一切片所有权；
允许文件、结构、接口、验证或证据范围；
明确禁止项，以及是否允许修改、构建和运行；
Git 暂存、提交和推送权限；
任务执行权、worktree 与主线发布占用权限；
冲突、漂移、失败、重叠或新用户裁决的停止条件。
```

协作代理开始写入或执行授权动作前必须完成最小 S0：

```text
确认所属顶层任务树、平台关系和继承角色；
确认目标、允许范围、禁止范围和返回条件；
确认目标路径、分支、HEAD 和 dirty state（适用时）；
确认与其它已派发切片没有所有权交集；
确认 Git、任务执行权、worktree 和发布占用权限；
向派发方返回 PASS 或具名 DRIFT。
```

未完成 S0 或 S0 为 DRIFT 时不得开始写入。递归派发必须原样传递顶层身份、角色、范围、所有权、Git / 占用权限和停止条件，不得在下级派发中扩大权限。任务消息只负责唤醒，聊天回执不得替代仓库身份、登记事实或完成事实。

### 3.3 所有权、串行收口与违规停止

1. 顶层任务树是任务和 worktree 的正式协作所有者；协作代理只取得派发合同列出的具体切片所有权。
2. 同一文件、结构、接口、验证项目或证据裁决同一时刻只有一个智能体负责。发现交集时必须先重新拆分或改为串行。
3. Git index、暂存、提交、推送、任务执行权迁移和主线发布占用迁移，由顶层智能体或唯一指定的同角色智能体串行完成；收口期间其它后代停止写入。
4. 顶层智能体必须复核后代的实际差异和验证证据；后代返回只证明其受派切片的结果，不证明顶层任务、集成或发布完成。
5. 范围漂移、所有权重叠、角色冲突、计划外文件、验证失败或用户新裁决只停止受影响切片并上报；不得自行扩权。若同一顶层任务树已同时拥有两种写角色或通过 worktree、分支、模型、名称规避角色，整树立即停止新增写入，保留现有 Git 事实并退回设计治理追查根因。
6. 无关 WIP 属于原所有者，任何后代不得暂存、清理、stash、覆盖或带入本任务提交。

## 4. 状态机

| 状态 | 写入角色 | 含义 | 下一状态 |
| --- | --- | --- | --- |
| `待执行` | 设计窗口 | 计划已生成且没有未满足的真实依赖，尚未完成派发 | `已派发待执行回执` |
| `依赖门控待执行` | 设计窗口 | 计划已生成，但依赖未满足 | `已派发待执行回执` 或继续门控 |
| `已派发待执行回执` | 设计窗口 | 计划、版本、基线提交和目标执行角色已发布 | `执行中`、`接口漂移待设计修订` |
| `执行中` | 执行窗口 | 正在按已发布计划实施 | `执行完成待设计同步`、`接口漂移待设计修订` |
| `分支完成待集成` | 执行窗口 | 任务分支已完成任务级验证并推送，尚未进入 main | `已预留待集成派发` |
| `已预留待集成派发` | 设计窗口 | 集成路径、分支、输入和验证矩阵已登记，但尚未确认物理状态或发送任务消息 | `已派发待集成回执` |
| `已派发待集成回执` | 设计窗口 | 集成所有者、路径、分支和起点已发布，任务消息已实际发送 | `集成中`、`集成身份冲突待设计重建` |
| `集成中` | 集成窗口 | 已核对登记身份并取得主线发布占用，正在汇集和验证 | `已集成待设计同步`、`集成失败退回任务` |
| `集成身份冲突待设计重建` | 集成窗口 | 登记路径 / 分支已存在且身份、基线或所有者不一致 | 新路径 / 分支重新 `已派发待集成回执` |
| `集成失败退回任务` | 集成窗口 | 提交范围、冲突或集成验证失败 | 任务分支修订后重新 `分支完成待集成`，或退回设计 |
| `已集成待设计同步` | 集成窗口 | 集成验证通过并已非强制发布 origin/main，路由尚未归档 | `已完成 / 已归档` |
| `执行完成待设计同步` | 执行窗口 | 代码、验证和实施记录已提交推送，路由尚未更新 | `已完成 / 已归档` 或设计裁决的后继状态 |
| `接口漂移待设计修订` | 执行窗口 | 计划与实际接口、范围或前置条件不一致 | `重新待执行`、新前置计划或真实阻塞 |
| `重新待执行` | 设计窗口 | 修订已发布并重新派发 | `执行中` 或再次退回 |
| `只读复核中` | 只读复核窗口 | 在指定提交、对象和问题集上核查，不写仓库 | `只读复核完成待设计裁决` |
| `只读复核完成待设计裁决` | 设计窗口 | 复核消息已返回，尚未逐项接受、驳回或转化为设计动作 | 修订、登记计划、维持现状或 `已完成 / 已归档` |
| `已完成 / 已归档` | 设计窗口 | 已读取正式实施证据并完成路由同步 | 后继计划或空闲 |

设计窗口未处理 `执行完成待设计同步`、`已集成待设计同步`、`接口漂移待设计修订`、`集成身份冲突待设计重建` 或 `只读复核完成待设计裁决` 前，不得重复派发同一计划，也不得把依赖该计划的后继项自行解释为已解除门控。

## 5. 设计窗口派发协议

设计窗口必须按顺序完成：

```text
1. 生成或修订流程图、详细设计和计划。
2. 先核对依赖必要性和可隔离性，再核对计划版本、允许文件、禁止文件、入口拒绝、失败收口和验证方式；只有具名消费关系或无法隔离的所有权先后才登记依赖门控。
3. 更新计划索引和任务队列为“已派发待执行回执”。
4. 记录目标执行角色、计划版本、派发基线提交、目标 worktree / 分支和是否属于预授权执行批次或并行批次。
5. 无真实依赖且满足所有权互斥的计划优先组成并行批次，同步登记批次号、共同基线、文件 / 结构所有权、接口冻结、集成顺序和验证矩阵；决定暂不并行时具名记录原因。
6. 运行文档治理验证。
7. 只提交本轮设计文件并自动非强制推送。
8. 推送成功后向唯一执行窗口发送排队提醒，并读取任务状态确认消息已送达；消息不形成执行许可。
```

`已预留`、`已登记`、`目标窗口已写入` 都不等于已经派发。只有以下三项同时成立，状态才能写为 `已派发待执行回执`：

```text
角色、任务标识、worktree、分支和冻结基线已经提交并推送。
Codex 任务消息已经向唯一目标实际发送。
设计窗口已读回目标任务为 active / inProgress，或记录了工具不可用的持久化替代回执。
```

派发消息固定字段：

```text
交互类型：计划已派发
计划 / 队列编号
计划版本
派发提交号
权威计划路径
允许 / 禁止范围
是否属于预授权执行批次
并行批次号、任务 worktree、任务分支和冻结基线
执行窗口必须重新读取的材料
```

### 5.1 集成派发协议

执行窗口返回 `分支完成待集成` 后，设计窗口必须继续完成，不得停在预留：

```text
1. 核对任务提交、父链、文件范围、任务级验证和远端同名任务分支。
2. 只读检查登记的集成路径和分支是否已经存在。
3. 路径或分支存在时，核对 HEAD、父链、dirty state、所有者和 reflog；任一身份不一致即标记失效候选，禁止 reset、delete 或复用。
4. 为重试分配新的集成编号、绝对路径和分支；旧候选保留只读证据。
5. 登记集成输入、起点、所有者、允许写入、验证矩阵、失败收口和主线发布占用。
6. 提交并推送中央治理后，向唯一集成任务实际发送消息并确认接收状态。
7. 消息送达后状态才进入 `已派发待集成回执`；目标完成身份复核后才进入 `集成中`。
```

集成任务进入 `集成中` 后取得主线发布占用。直到集成任务发布 main 或正式退回并释放占用，设计窗口不得修改 WT-MAIN 文件、提交或推送 `origin/main`；可以继续只读审查和在对话中形成草案，但不得落盘。紧急设计变更必须先正式取消集成占用并由集成窗口停机回执。

### 5.2 只读复核派发协议

设计窗口派发只读复核时必须指定：

```text
目标提交和 worktree。
复核对象、问题清单和证据入口。
允许读取的规则、计划、记录、代码和 Git 事实。
禁止修改、构建、暂存、提交、推送、入队和完成裁决。
报告格式：事实、差异、证据、风险、建议，不替代设计决定。
```

只读复核窗口只通过任务消息返回报告。需要形成持久化文档时，由设计窗口消费报告后写设计记录，或另行登记纯文档执行计划；不得让只读角色自行升级为写入角色。

## 6. 执行窗口接管与下一计划选择

执行窗口不得只凭自身上下文、消息到达顺序或计划索引中的单一字段选择计划。每次开始新的执行轮次必须重新计算：

```text
重新读取 Git、当前 worktree / 分支、登记基线和 dirty state
-> 读取计划索引和任务队列
-> 读取并行工作树登记表和批次冻结清单
-> 读取候选计划、最新实施记录和执行复核断点
-> 排除已完成、待设计同步、退回设计和版本不一致项
-> 按队列优先级、依赖和显式顺序筛选
-> 复核允许 / 禁止文件和实际接口
-> 串行模式只有全局唯一候选通过才进入执行；并行模式只有本 worktree 唯一分配项通过才进入执行
```

出现以下任一情况必须停止并回传设计窗口：

```text
多个同优先级候选且索引没有裁决顺序。
计划索引、任务队列和最新实施记录相互冲突。
计划版本或派发提交与当前正式状态不一致。
前置计划刚完成但仍处于“执行完成待设计同步”。
实际接口与假定接口不一致。
候选计划会接触其它未收口切片的文件。
```

默认一个计划完成或退回后先交回设计窗口。只有设计窗口明确登记“预授权执行批次”，且批次内计划互相独立、无文件范围冲突、候选唯一时，执行窗口才可在完成当前项后继续计算并消费下一项。

## 7. 执行完成与退回协议

执行窗口发现接口漂移、范围越界或完成切片时，不得修改计划和路由文件。必须按顺序完成：

```text
1. 停止扩大当前实施范围。
2. 写本次执行专属实施记录或复核断点。
3. 完成时标记“执行完成待设计同步”；漂移时标记“接口漂移待设计修订”。
4. 记录计划编号 / 版本、派发提交、实际接口、修改文件、验证命令 / 结果、实施提交和完成边界。
5. 运行计划要求的验证。
6. 只提交执行角色允许的代码、工程、自检和实施记录，并自动非强制推送。
7. 推送成功后向设计窗口发送完成或退回摘要，然后停止当前计划。
```

实施记录 / 复核断点必须至少包含：

```text
计划 / 队列编号和版本
派发基线提交
实际读取的接口与提交
执行状态
修改文件
验证命令与结果
实施提交号
完成声明边界
接口漂移、未解决问题和建议设计动作
```

消息发送失败时，已推送的实施记录 / 复核断点就是持久化回执。设计窗口下次启动必须扫描最新记录，不得要求执行窗口越权修改计划索引、任务队列或窗口交互记录。

## 8. 设计窗口同步协议

设计窗口收到完成、退回或消息失败提示后必须：

```text
1. 读取实施提交并确认当前分支包含该提交。
2. 重读计划、计划索引、任务队列、最新实施记录、代码变化和实际接口。
3. 判断完成归档、原计划定向修订、新增前置计划或真实阻塞。
4. 更新计划、计划索引、任务队列、窗口交互、当前状态、决策和必要验收汇总。
5. 运行文档治理验证，自动本地提交并自动非强制推送。
6. 完成归档后立即重算依赖、优先级和可执行候选；存在唯一就绪项时必须登记并真实派发，不能只写“下一步”。
7. 不存在就绪项时，必须写明是继续设计生成、接口修订、等待集成、用户暂停还是具名外部阻断；不得无理由空闲。
8. 不得仅凭消息正文或执行 / 集成窗口自述标记完成。
```

### 8.1 计划持续推进闭环

项目计划统一沿以下循环推进：

```text
设计窗口维护规范—设计—代码覆盖矩阵和依赖图
-> 先按业务对象、文件、结构、接口和装配面拆分计划
-> 无真实依赖时生成独立待执行计划并优先登记正式并行批次；只有具名消费关系才生成依赖门控计划
-> 真实派发串行唯一候选或正式并行批次
-> 执行窗口形成任务分支和正式实施证据
-> 设计窗口核对后真实派发独立集成
-> 集成窗口完整重验并非强制发布 main
-> 设计窗口同步、归档、更新逐能力台账并立即重算下一项
-> 里程碑或最终完成前派发独立只读复核
-> 设计窗口逐项裁决复核结论后继续循环
```

设计窗口是项目计划闭环的编排者，不是只写计划的文档窗口。以下任一状态存在时，不得宣称对应计划或项目阶段已经完成：

```text
依赖就绪但仅预留、未真实派发。
执行中、分支完成待集成、已预留待集成派发、已派发待集成回执或集成中。
执行 / 集成完成待设计同步。
接口漂移、集成身份冲突或集成失败尚未裁决。
只读复核完成但尚未由设计窗口逐项裁决。
计划索引、任务队列、完成目录、逐能力台账、当前代码和 origin/main 不能互相证明一致。
```

项目里程碑或最终完成声明前，必须派发一次独立只读覆盖审计，至少核对：全部现行计划处置、main / origin 同步、工作树残留、入口与生产调用点、计划验收证据、当前代码差距台账和禁止扩大声明。只读审计报告不能自行宣称完成；最终裁决仍由设计窗口写入正式治理状态。

## 9. 运行中消息的排队规则

执行窗口正在执行时收到的新计划消息，必须使用排队语义：

```text
只记住“有新正式状态需要下一轮重读”。
不改变当前计划范围。
不打断当前验证和提交链。
不按消息到达顺序决定下一个计划。
当前计划收口后重新读取仓库，再计算唯一候选。
```

消息若被客户端设置为 Steer，也必须在提示中明确“不得中断当前计划、不得形成执行许可”，并把实际处理推迟到下一轮。推荐把客户端 Follow-up behavior 设为 Queue。

未登记并行批次时，设计窗口仍不得在执行窗口占用主集成工作树期间并发落盘。已登记并行批次且仅有任务 worktree 执行时，设计窗口可以在 clean 的主集成工作树继续不触碰批次冻结合同的后继治理；任务窗口只能在独立 worktree 写入。设计变更一旦触碰批次冻结文件、接口、依赖或集成顺序，原批次立即失效，必须停止任务并重新派发。任一集成任务进入 `集成中` 后，主线发布占用优先于设计治理，设计窗口只能只读，直到集成发布或退回释放。

## 10. 同一物理窗口串行双角色

只有用户明确启用时才能进入单窗口串行切换；该模式以 `AGENTS.md` 定义的整棵顶层任务树为单位，全部后代必须先停止并释放旧角色占用，不能让顶层窗口和后代分别保留不同写角色。单窗口模式不合并角色权限：

```text
执行角色完成或退回
-> 只写实施记录 / 复核断点
-> 验证、提交、推送
-> 推送成功后切换设计角色并重读全部正式状态
-> 由设计角色更新路由

设计角色完成同步或派发
-> 更新计划、索引、队列和窗口交互记录
-> 验证、提交、推送
-> 推送成功后切换执行角色并重读全部正式状态
```

执行转设计以已推送实施记录 / 复核断点为依据；设计转执行以已推送计划索引、任务队列和窗口交互记录为依据。切换后需要继续并行时，按新角色重新派发子智能体并完成 S0。未验证草案不得跨角色继承。

## 11. 模型路由与异常升级

默认建议：

| 工作类型 | 模型 / 推理建议 |
| --- | --- |
| 设计、计划审查、接口漂移和逻辑异常裁决 | Sol + High，或可用的同等高推理配置 |
| 成熟计划常规实施、验证和记录 | Terra + Medium，或可用的同等中等推理配置 |
| 明确、重复、只读的信息整理 | Luna / 低消耗等效配置 |

模型选择不改变角色权限。执行窗口发现接口漂移、计划冲突、事务 / 并发 / 因果边界无法由计划裁决、内部逻辑错误，或验证重复失败且根因不明时，必须停止并退回高推理的设计窗口；不得只通过提高执行窗口模型等级自行改计划或扩大范围。

### 11.1 中等模型可机械执行合同

设计窗口把成熟计划派给中等推理模型时，计划必须按以下四列写清每个阶段：

```text
已验证前置事实
本阶段待实施目标
实施完成的可核对证据
只有前置证据成立后才可判断的完成 / 退回条件
```

硬规则：

1. 待实施目标不存在，默认表示该阶段尚未完成，不表示接口漂移、计划冲突或真实阻塞。
2. 退回条件必须引用已经形成的具体文件、符号、构建输出、具名失败项或接口差异；不得用“证据不足”“代码较长”“尚未形成目标产物”替代前置证据。
3. 同一输出在目标实施前只可作为基线输入，在目标实施后才可作为验收证据；不得把实施前失败套入实施后退回门禁。
4. 执行模型误读明确阶段、跳过授权步骤、把目标当门禁或连续两轮不能按检查表推进，登记为模型能力不匹配。设计窗口应收窄同一计划并可显式改派 Sol + High 或同等高推理模型继续执行；计划编号、角色权限、允许 / 禁止范围不变。
5. 执行轮次因平台边界结束但尚未命中计划的正式完成 / 退回条件时，只写精确续点并保持 `执行中`。不得仅因轮次结束写 `执行完成待设计同步` 或 `接口漂移待设计修订`。

高推理模型进入执行窗口后仍只是执行角色，只能实施已发布计划；不得修改计划、索引、队列或借模型等级扩大范围。

## 12. Codex 任务工具规则

```text
按精确任务标题、任务标识和 cwd 定位唯一目标任务。
发送消息时列明计划编号、提交号、状态和权威路径。
计划或集成 worktree 的“预留 / 登记”不等于任务派发；推送后必须实际调用任务消息工具。
消息调用成功后必须读回目标任务状态；目标仍不存在、歧义或不可达时保持“待派发”，不得写成“执行中 / 集成中”。
运行中的新计划消息采用 Queue，不用 Steer 改变当前切片。
不得向多个候选任务广播实施指令。
线程 / 任务标识只作运行期路由材料，不作为正式计划身份。
消息失败不改变已推送仓库状态的效力。
派发不同写角色前必须按平台父子关系和协作代理关系确认目标是独立用户可见顶层任务，而不是当前顶层任务树的协作代理；只有用户明确启用同一物理窗口串行切换时才允许整树正式切换。协作代理只能继承当前树的写角色或保持只读。
```

## 13. 并发、worktree 与 Git 边界

```text
唯一最终权威分支为 main，D:\海中鱼巢 是主集成工作树；海中鱼巣只作为仓库内源码目录名。
任务 worktree 使用独立 codex/<队列号>-<主题> 分支；集成 worktree 使用 codex/integration-<批次号> 分支。
同一 worktree 任一时刻只有一个写入角色；多个 worktree 只有在预授权并行批次中并发。
设计窗口不得用同主题、同总计划、生成顺序、目录位置或验收顺序伪造依赖；可隔离的计划应先隔离所有权并优先登记并行批次。
每轮任务完成先自动本地提交并非强制推送到同名任务分支，再进入分支完成待集成。
集成窗口核对范围、按冻结顺序汇集并运行集成验证；通过后由主集成工作树非强制发布 origin/main。
集成窗口从身份核对通过到发布 / 退回期间独占主线发布；设计窗口不得并发修改或推进 main。
发现登记路径 / 分支已经存在但身份不一致时，旧候选只读保留并登记失效，设计窗口另立新路径 / 分支；禁止在旧候选上 reset、delete、切基线或继续 cherry-pick。
只暂存本轮文件，不带入其它窗口、用户或中断任务的未提交改动。
禁止 force、任务窗口自动合并 / 变基、未登记分支、未验证 main 发布和自动删远端任务分支。
消息、日志和聊天摘要不裁决计划完成。
```

详细准入、冻结、共享文件、失败退回、验证和回收规则见 `规范/多工作树并发与集成规范.md`。

## 14. 验证

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

顶层任务树收口还必须核对：全部后代已经结束或明确保持只读；树内没有两种写角色；所有权交集为零或已串行消解；Git 收口只包含本轮目标文件；无关 WIP 未被暂存或改写；提交、推送和完成声明与实际证据一致。

若本轮同时修改技能，还必须按对应技能文件中真实存在的专项验证入口执行；不存在的脚本不得写成强制验证。规则文档验证不证明代码编译、入口拒绝、业务闭环或任何代码能力完成。

## 15. 完成边界

本规范只证明设计、执行、集成和只读复核四角色的写入所有权、真实派发、交接状态、主线发布占用、任务 worktree 分配、集成发布、独立复核和计划持续推进规则已经建立。它不证明任何代码计划已经实施、构建通过、业务闭环完成或旧项目能力已经等价重建。

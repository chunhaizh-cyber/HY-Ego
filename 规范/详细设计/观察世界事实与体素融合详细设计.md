## 1. 设计目标与依据

本设计建立从合法方法结果到场景事实、状态动态和体素模型的逐层提交链：

```text
D455稳定观察子集
-> 像素簇
-> 存在候选
-> 结构化识别派生需求材料
-> 需求统一入口 / 新任务 / 新筹办轮次
-> 识别任务工作包
-> 唯一已有存在句柄或空句柄
-> 空句柄按规则形成复验或新存在承接需求
-> 具名存在确认入口按 7100 创建并读回新实例（适用时）
-> 场景成员 / 观察事实提交
-> 状态 / 动态完整事实
-> 已确认存在新增的体素材
-> 体素融合 / 分解 / 父子模型
-> 场景体素快照与视觉先验只读投影
```

正式依据为 0050、1140—1170、4010—4070、4210、4220、6200—6250、6300—6330、6360、6370、7100。6210 约束当前值闭合和首次扫描基准，6220 约束场景认知工作区，6230 约束已确认存在的递归细分，6330 约束作用后的一阶改变与关系二次特征，7100 约束新实例身份、场景归属、概念归属、扩充与完整读回。

端口实现只能调用现有存在 / 场景 / 状态领域服务及 7100 已确认存在入口；#369 不新增存在节点类型、观察事实节点、平行关系类型或第二事实仓。空识别结果只形成复验 / 新存在承接需求材料，不由 #369 创建新存在。
`观察存在结构提交结果` 的 `不存在 / 类型不匹配 / 场景不匹配 / 前置不满足 / 版本漂移` 只允许在任何写入前返回；一旦端口接受规格并建立候选，确认、发布或完整读回失败只能返回 `内部不一致 + 追根因=true`，不得用空 `optional` 混淆逻辑内拒绝和内部错误。

依赖端口固定为：

```cpp
class 状态动态事实提交提供者 {
public:
    virtual ~状态动态事实提交提供者() = default;
    virtual 状态动态事实提交结果 提交首次扫描状态(
        const 观察状态提交规格& 状态) = 0;
    virtual 状态动态事实提交结果 提交扫描跟踪变化(
        const 观察状态提交规格& 后状态,
        const 观察动态提交规格& 动态) = 0;
};
class 感知任务结果回流提供者 {
public:
    virtual ~感知任务结果回流提供者() = default;
    virtual 感知任务结果回流结果 提交感知任务结果(
        const 感知任务结果回流规格& 规格) = 0;

```

前者由节点直接状态 / 动态领域实现适配，后者由 SELF-C4 适配；#370 不复制旧 `主信息句柄` ABI。首次扫描只调用 `提交首次扫描状态`，绝不形成动态。事实发布并完整读回后才回流任务；任务回流失败属于内部错误，但不得回滚已经合法发布的世界事实或把它伪成未发生。
两个提供者都必须返回具名结果，禁止用空 `optional` 同时表示入口拒绝、合法无结果和内部错误。`观察状态动态提交组合` 构造只注入 `状态动态事实提交提供者&` 与 `感知任务结果回流提供者&`；它不持有状态、动态或任务仓库。

## 3. PER-C9—PER-C12 / v0.2

### 3.1 PER-C9 / v0.2：存在候选、识别和场景提交

- 提供 / 消费：#369 提供识别派生需求材料给 #367，提供观察事实与体素材来源给 #371、#373；#369 同时只读消费 #367 的工作包合同和 #368 的观察 / 识别结果合同。

- 输入：第一段输入 PER-C8 观察结果、稳定材料引用、目标场景和来源动作；第二段输入后继新任务执行后形成的 PER-C8 识别结果；两段不在同一方法调用内连续执行。
- 输出：像素簇材料、存在候选、识别派生需求材料、唯一句柄 / 空句柄、新存在承接或复验需求材料，以及前置完整后已读回的存在、场景成员关系、状态事实和合法外设当前数据项投影。
- ABI / 规则版本：导出 `inline constexpr std::uint32_t 观察存在提交ABI版本 = 2`；观察结果、派生需求、识别工作包、识别结果和结构提交各自保留来源版本。

- 生命周期：材料和候选可失效；派生需求只在新任务承接后进入识别；已有或新建存在、关系和观察事实按领域版本持久。
- 并发键：候选身份用于识别；存在 + 场景 + 观察版本用于事实提交。
- 逻辑内返回：无合法簇、无唯一对应、空句柄、需补观察、需冲突复验、新存在承接等待、场景 / 类型 / 版本不匹配。

- 恢复：只恢复完整节点、关系和类型化记录并重新准入；报告、队列或候选不能直接恢复为世界事实。
- 漂移退回：识别唯一性、需求 / 任务回流、7100 实例创建、场景成员关系、事实字段或来源动作改变。

### 3.2 PER-C10 / v0.2：状态动态与任务回流

- 模块 / 文件：第 2 节 #370 三个精确文件；复用现有状态 / 动态数据操作和服务，不建立第二套事实仓。
- 提供 / 消费：#370 提供，#373 消费。
- 输入：独立扫描任务返回的首次基准 / 扫描变化结果，或独立跟踪任务结果，以及权威主体和场景、时间、来源动作、前后版本。扫描 / 跟踪结果必须来自各自“结构化需求 -> 新任务 -> 新筹办 -> 匹配工作包”的独立回合。
- 输出：状态事实、动态事实和任务结果引用；三者分别裁决。
- ABI / 规则版本：导出 `inline constexpr std::uint32_t 观察状态动态提交ABI版本 = 2`；状态 / 动态 DTO 服从 1160 / 1170，任务回流通过本设计的 SELF-C4 适配端口。
- 写读方：状态 / 动态领域数据操作写；任务域只读结果引用并裁决任务。
- 生命周期：事实持久；当前状态是关系 / 索引投影，不复制权威数组。
- 并发键：主体 + 特征 + 场景 + 事实版本；动态按来源动作和发生次序附加。
- 逻辑内返回：主体未知、事实字段不完整、版本漂移、首次扫描无可比基准、变化未达到事实门。首次扫描只允许提交当前值 / 当前状态基准，不得形成“发生变化”的动态；只有可比基准和可验证差异同时成立时才形成变化动态。
- 内部错误：前置通过后状态 / 动态无法同事务读回，或任务结果引用事实不存在。
- 恢复：按节点、关系、类型化记录、版本和历史事实恢复；重建当前投影。
- 漂移退回：1160 / 1170 字段、关系 19 / 20、任务结果或动作来源语义改变。

### 3.3 PER-C11 / v0.2：体素融合与分解

- 模块 / 文件：第 2 节 #371 四个精确文件。
- 提供 / 消费：#371 提供，#372 消费。

- 输出：新体素模型代次、材料消费记录、分解和父子模型关系。
- ABI / 规则版本：导出 `inline constexpr std::uint32_t 存在体素融合ABI版本 = 2`；新增确认关系、场景成员关系、模型 / 单元类型化记录和发布版本必须可读。

- 生命周期：模型代次与证据历史持久；工作缓存可清空重建。

- 内部错误：前置通过后父子模型、空间不变量、消费记录或发布读回不一致。
- 恢复：从完整场景、模型代次、材料稳定身份、来源产品引用、发生时间、关系证据、消费状态 / 版本和消费记录恢复，不从显示网格恢复。

## 4. 事务与错误收口

1. 候选形成不取得事实身份。
2. 领域写入统一使用 4040 的不透明候选—确认—撤销—最后发布。
3. 外部输入不合法、无唯一对应、无融合点和版本漂移是逻辑内返回。
4. 前置通过后的写入、读回、发布、父子不变量或事实引用异常是内部错误。
5. 数据库、文件、报告队列和控制面板都不能直接发布世界事实。
6. 观察、识别、扫描和跟踪只能通过结构化结果 / 缺口、需求统一入口、新任务和新筹办轮次衔接。

# 服务模块单一声明所有权与规格构造详细设计

日期：2026-07-15

状态：JY-349 / QR-184 已裁决 / #279 配套详细设计 / 代码未实现

## 1. 定位

本文解决 #279 在完整 Rebuild 链接阶段暴露的命名模块 ABI 漂移：数据操作模块先导出业务服务类前置声明，服务模块随后定义同名业务服务类，MSVC 将它们附着到不同命名模块，导致独立自检看到的构造函数和成员函数符号与既有对象定义不一致。

本文只治理正式新分层模块的实体声明所有权和私有写入规格构造边界，不改变业务语义、仓库权威结构、事务 ABI、许可强度、锁顺序或传统服务头兼容合同。

## 4. 唯一模块所有权

### 4.1 服务实体

每个正式服务模块唯一拥有：

```text
业务服务类定义。
该服务公开请求类型。
该服务自有的包装结果或服务级规格结果类型。
服务私有的业务状态映射辅助。
```

数据操作模块不得出现：

```cpp
class 存在业务服务;
class 状态业务服务;
class 方法业务服务;
```

也不得通过专用“服务声明模块”重新声明后让服务定义落在另一个模块。声明与定义必须处于同一命名模块所有权。

### 4.2 数据操作实体

每个数据操作模块唯一拥有：

```text
数据操作类。
结构写入规格。
仓库读回形成的值式结构材料。
数据操作业务状态和内部结构结果。
窄规格形成入口。
```

规格构造函数继续为 private。对应数据操作类是唯一规格构造友元；服务、自检、组合器、线程和运行期路由都不是友元。

现有服务公开方法直接返回的数据操作值式结果继续由数据操作模块唯一拥有，服务只使用其完整定义并按值返回。JY-349 不把这些类型搬入服务模块，也不复制第二份服务别名。

## 5. 规格形成入口

### 5.1 通用合同

每个入口采用下列语义：

```text
输入：服务已完成业务准入后的结构字段，或其它数据操作模块已形成的完整嵌套规格。
处理：复核非零编号、完整句柄、枚举范围、嵌套规格完整性和规格自身字段组合。
成功：返回 std::optional<对应写入规格> 的有值结果。
拒绝：返回空 optional，不读写仓库，不取得许可。
副作用：无。
```

入口不得接收 `创建...请求`、服务级结果或服务类引用；否则数据操作将反向依赖服务模块并形成循环。

### 5.2 入口组

| 数据操作类 | 窄规格形成入口 |
| --- | --- |
| `存在场景数据操作` | `形成虚拟存在写入规格` |
| `状态动态数据操作` | `形成抽象状态写入规格`、`形成实例状态写入规格`、`形成候选主体实例状态写入规格` |
| `特征体系数据操作` | `形成特征槽位写入规格`、`形成初始特征值写入规格` |
| `轻量因果数据操作` | `形成轻量因果写入规格` |
| `概念图结构数据操作` | `形成概念关系写入规格` |
| `语素基础数据操作` | `形成语素入口写入规格` |
| `需求任务方法数据操作` | `形成完整需求写入规格`、`形成概念命名需求写入规格`、`形成需求结算写入规格`、`形成任务承接写入规格`、`形成任务生命周期写入规格`、`形成任务方法选择写入规格`、`形成任务完成写入规格`、`形成方法登记根写入规格`、`形成方法首写入规格`、`形成方法条件写入规格`、`形成方法结果写入规格`、`形成方法动作入口写入规格`、`形成方法生命周期写入规格` |

入口名称与规格一一对应，不提供接收任意规格种类或 `void*` 的通用工厂。

### 5.3 服务调用合同

服务的顺序固定为：

```text
检查请求局部字段
-> 读取并复核权威业务材料
-> 调用对应数据操作的规格形成入口
-> 取得完整规格
-> 调用数据操作正式写入口
-> 映射值式结果
```

服务准入失败属于逻辑内返回，不调用规格形成入口。服务已经证明全部前置后，规格形成入口仍返回空，说明服务规则与规格不变量漂移，必须追根因，不得降级为普通请求拒绝。

## 6. 无环依赖

允许依赖方向：

```text
服务.*
-> 数据操作.*
-> 核心执行器 / 会话 / 仓库内部 ABI
```

跨服务组合继续通过服务模块：

```text
需求 / 任务 / 方法服务
-> 存在服务 / 状态服务
-> 对应数据操作规格形成入口
```

禁止依赖方向：

```text
数据操作.* -> 服务.*
数据操作.* -> 服务请求 / 服务结果
自检 -> 私有规格构造友元
组合器 / 路由 -> 数据操作规格形成入口
```

规格形成入口是数据操作的内部协作公开面，不等于运行期对外公开面。`装配.运行期业务` 继续只向外返回业务操作组合器，不增加数据操作读取接口。

## 7. #279 集成

### 7.1 方法动作入口

`方法业务服务::创建动作入口` 完成参数、方法首、输入 / 输出场景和两个状态规格准入后，必须调用：

```text
需求任务方法数据操作::形成方法动作入口写入规格
```

取得完整规格后再调用 `创建方法动作入口`。数据操作的动作入口写入继续由 #279 改为创建两条 `方法动作场景 = 17` 关系并精确读回。

### 7.2 独立自检

`自检.方法动作场景关系` 只通过正式服务构造函数和服务公开业务入口建立场景、状态和方法动作入口。它不得直接调用 22 个规格形成入口，不得恢复服务友元，不得文本包含服务实现。

## 8. 失败收口

| 场景 | 分类 | 收口 |
| --- | --- | --- |
| 请求字段、身份、类型或业务关系不满足 | 逻辑内返回 | 服务写前拒绝，规格工厂不调用 |
| 规格工厂直接收到不完整字段 | 逻辑内返回 | 返回空 optional，无许可、无写入 |
| 服务准入通过后规格工厂仍拒绝 | 追根因解决 | 停止该服务路径，登记规则 / 规格漂移 |
| 数据操作写入或读回非预期 | 追根因解决 | 同一会话精确撤销，不返回有效业务结果 |
| 独立自检链接仍有服务未解析符号 | 实施前 / 构建门禁失败 | 不登记 785，按原 #279 退回设计 |
| Debug / Release ABI 或公开面扫描漂移 | 实施验收失败 | 不归档 #279，不恢复 #224 |

## 9. 验收

本设计并入 #279 的 A01-A16，不新增阶段。至少证明：

1. 9 个受影响服务类各只有一个命名模块声明 / 定义所有者。
2. 7 个数据操作模块中的跨模块业务服务前置声明为 0。
3. `friend class ...业务服务` 从当前 22 处归零。
4. 22 个规格构造函数仍为 private，且只由对应数据操作类构造。
5. 22 个规格形成入口不取得许可、不写仓库、不接收服务 DTO。
6. 调用点扫描只允许对应服务模块调用规格形成入口。
7. 14 个正式服务、7 个组合器和 1 个请求路由继续保持零原始事务能力公开。
8. `自检.方法动作场景关系` 在全新完整 Rebuild 中链接并运行，不复用旧 BMI。
9. Debug / Release x64 完整 Rebuild、Release 自检零登记、阶段 785 唯一和最终 Debug 连续 20/20 通过。

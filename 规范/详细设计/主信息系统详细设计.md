## 1. 设计目标

本详细设计对应 `规范/4020_子规范_主信息身份生命周期与字段边界.md`，把主信息系统的统一仓库、最小记录、值容器和字段准入门禁落到函数级设计。

第一轮目标不是迁移旧项目 `*主信息类`，而是确认：

```text
主信息仓库统一
主信息记录最小
业务字段不膨胀
具体信息按结构分流
字段新增先走规范和功能确认
编号使用 uint64 语义且不复用
旧字段处理口径已完成第一轮确认
```

本详细设计确认的是函数边界和结构承载口径，不声明代码已经通过构建或验收。

## 4.1 编号设计

当前主信息编号设计：

```text
类型语义：uint64
无效值：0
起始值：1
分配方式：仓库内单调递增
删除行为：只标记已删除并递增版本，不回收编号
稳定身份：仓库编号 + 主信息编号 + 版本号
```

`std::int64_t` 只用于 `值容器` 和 `写入I64值` / `读取I64值`，不得用于主信息编号。

后续若出现长期运行容量问题，处理顺序必须是：

```text
先评估内存、索引、快照和持久化体量。
再建立仓库编号 / 分片 / 纪元 / 归档设计。
最后才进入代码实现。
```

不得通过复用主信息编号、把已删除编号重新分配给新记录、或把编号临时改成有符号类型来规避容量问题。

## 4.2 版本语义

```text
创建主信息时版本号 = 1。
删除主信息时记录状态改为已删除并递增版本号。
普通 I64 值写入不递增主信息句柄版本号。
值变化若需要审计，由特征值版本、动态、事件日志或因果引用承载。
旧主信息句柄不得命中删除后的记录。
```

## 5. 字段功能确认门禁

任何新增主信息字段前，必须先形成字段功能说明：

```text
字段名
主要用途
写入方
读取方
生命周期
是否权威
是否可由关系 / 索引 / 特征 / 状态 / 动态 / 因果替代
入口前置拒绝条件
验证方式
禁止宣称
```

没有字段功能说明，不得进入代码实施。

字段功能说明不能只写“方便显示”“方便查询”或“旧项目已有”。显示、查询和兼容只能作为候选原因，不能替代结构承载理由。

## 7. 示例分流

输入：

```text
这个杯子很漂亮
```

主信息系统只允许承载其中的最小值。例如高程度值可进入值容器。其他内容应分流：

```text
这个杯子 -> 指代解析和实际存在节点
杯子 -> 语素入口和概念节点
漂亮 -> 复合特征概念节点
很 -> 二次特征 / 程度修饰概念节点
整句来源 -> 语言材料或来源动态
评价事实 -> 状态 / 特征值 / 因果引用
```

不得新增 `漂亮`、`很漂亮`、`杯子很漂亮` 等主信息字段。

## 8. 入口前置拒绝

主信息仓库和授权领域服务必须在入口拒绝以下情况：

```text
主信息句柄编号为 0。
仓库编号不匹配。
主信息编号不存在。
版本号不匹配。
记录已删除。
未授权写入方写值容器。
试图把业务事实作为通用主信息字段写入。
没有字段功能说明却新增字段。
```

拒绝路径可以返回局部成功 / 失败结果，但不得把普通 `bool` 持久化为业务事实。

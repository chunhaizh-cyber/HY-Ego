# 历史初始化真模块迁移与唯一上下文装配详细设计

## 1. 定位

把 `初始化.概念图.ixx`、`初始化.世界树.ixx`、`初始化.语素.ixx`、`初始化.需求.ixx` 从历史 `ClInclude/#include` 兼容形态迁移为真实模块，同时保持业务初始化顺序和结果不变。

## 2. 前置与假定

前置 #249。假定 #249 形成唯一运行期上下文和服务装配入口；四个初始化文件仍保留 #236 扫描到的职责。执行前必须重读工程登记、include 链、导出类型和全部调用点。

## 3. 模块依赖

```text
初始化.概念图
-> 初始化.世界树
-> 初始化.语素
-> 初始化.需求
-> 初始化.系统总成
```

仅在实际调用图证明需要时保留边；禁止循环 import。每个模块接收已有上下文 / 服务引用，不创建仓库、协调器或全局服务。

## 4. 生产与自检

新增或转换五个生产真模块：四个领域初始化模块与根目录 `初始化.系统总成.ixx`；新增 `自检.初始化模块.ixx`。自检覆盖顺序、幂等、无效上下文、阶段失败不发布、重复调用和旧 include / ClInclude 清零。入口只调用系统总成一次并登记自检。

## 5. 失败边界

无效上下文、缺服务、重复初始化请求属于逻辑内返回且不发布半结构；前置通过后根身份、关系、索引、计数或读回不一致归追根因。

## 6. 工程迁移

新 `.ixx` 必须 `ClCompile + export module + import`。原文件若原地转换，必须同一提交移除 `ClInclude/#include` 用法；若实际接口要求新文件，旧文件不得同时留作兼容实现。执行窗口不得自行选择双轨方案。

## 7. 完成边界

只证明初始化工程承载和调用方向模块化，不证明新增初始化能力、恢复、自我循环或旧能力迁移。

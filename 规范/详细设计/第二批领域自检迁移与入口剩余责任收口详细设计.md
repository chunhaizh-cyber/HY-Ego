# 第二批领域自检迁移与入口剩余责任收口详细设计

日期：2026-07-12

状态：正式依赖门控设计 / 绑定 #251-#254

## 1. 定位

承接 #236 和第一批 149 项迁移，在当前活动计划全部形成实际模块后，迁移入口中剩余的基础信息及需求 / 任务 / 方法自检正文，并只读审计入口是否达到 CODE-FILE-01 边界。

## 2. 前置

#251 等待 #230、#235、#250 完成。原因是恢复、动态因果和初始化会改变最终工程登记与入口剩余正文；提前按旧行号迁移会再次漂移。

## 3. #251 事实矩阵

只读生成：验收编号守恒表、入口稳定锚点、共享夹具所有权、阶段 / 顺序占用、输出归属、跨阶段值式报告、生产 / 自检依赖和允许文件建议。未完成矩阵前不得执行后续迁移。

## 4. #252 基础信息批次

目标逻辑域：四仓库、主信息、语素、特征 / 特征值、存在、场景、状态、动态、二次特征、因果、世界聚合。实际模块名、验收数和阶段顺序由 #251 固定；计划假定新增若干 `自检.<领域>.ixx` 真模块，不修改生产语义。

## 5. #253 高级服务批次

目标逻辑域：需求、需求树、任务、任务生命周期、方法、方法召回 / 规格 / 索引 / 排序、动作入口、结果 / 结算。只迁移 #251 证明仍在入口的既有自检；#212-#235 已由各自计划建立的新自检模块不得重复迁移。

## 6. #254 收口审计

纯只读核对入口只剩参数、模式、唯一装配、初始化总成调用、阶段登记、顶层运行和退出码。若仍有领域正文、并发循环、逐项输出或历史 include，记录下一具体逻辑包，不在审计中修改代码。

## 7. 迁移规则

不按连续文本迁移；不改验收语义；不伪造材料；不复制共享夹具；跨模块只传值式报告；普通自检失败不短路后续回调；生产模块不依赖自检模块。

## 8. 非成功二分

依赖未满足、接口或阶段漂移属于设计门控返回；无效测试请求属于逻辑内返回；迁移后编号、结果、结构或顺序改变属于追根因解决。

## 9. 完成边界

第二批完成只证明对应入口自检迁入独立模块。#254 通过才可声明入口责任达到当时扫描边界；仍不得声明旧能力迁移、自我苏醒或全部未来自检无需新增。

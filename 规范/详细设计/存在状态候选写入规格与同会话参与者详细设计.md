# 存在状态候选写入规格与同会话参与者详细设计

日期：2026-07-14

状态：正式详细设计 / JY-334 / #278 施工依据 / 代码未实施

## 1. 定位

本设计解决 `#269 / SERVICE-DATA-S5` 实际接口复核暴露的跨服务同会话缺口：

```text
存在、状态原子服务负责业务规格
-> 原子数据操作层在调用方已有会话内参与写入
-> 唯一外层事务所有者决定提交或撤销
```

它不新增需求、任务、方法业务结构，不允许上层数据操作复制存在 / 状态结构，也不允许业务服务传递原始令牌或启动嵌套事务。

## 2. 当前代码事实

基线：`652cf30`。

### 2.1 已完成能力

```text
#264：结构写入执行器和不可移动写入会话已经形成一次独占许可、精确写集和统一收口。
#272：会话已经支持候选 I64 值、节点类型 / 主信息 / 值和按源关系记录组读取。
#277：会话已经支持已发布关系可逆失效 / 重挂、普通父子和关系审计读取。
```

`结构提交准备只读视图` 已能判断主信息 / 节点是否属于本会话候选，但 `结构写入会话` 的写入阶段没有对应的窄布尔入口。

### 2.2 存在服务实际接口

`服务.存在.ixx` 只有：

```text
创建实际存在(创建实际存在请求)
创建虚拟存在(创建虚拟存在请求)
```

`创建虚拟存在` 要求来源节点已经可以通过共享读取复核；未形成“来源节点将在同一会话内创建”的类型化规格。

`数据操作.存在场景.ixx` 的主信息、节点、主键和来源关系创建逻辑位于私有 `创建身份`，并由自己的执行器启动独立事务。当前没有供另一个数据操作层复用调用方会话的参与入口。

### 2.3 状态服务实际接口

`服务.状态.ixx` 当前：

```text
创建抽象状态直接启动状态数据操作事务。
形成实例状态写入规格要求场景和主体已经发布且当前可读。
创建完整实例状态再启动状态数据操作事务。
```

`数据操作.状态动态.ixx` 的 `写入实例状态_会话` 是私有辅助；抽象状态写入也内嵌在自己的执行器回调。当前没有供外层数据操作复用当前会话的抽象状态、已发布主体实例状态或未来主体实例状态参与入口。

### 2.4 对 #269 的影响

`#269` 设计要求以下结构处于同一提交边界：

```text
需求 + 目标抽象状态。
任务 + 任务虚拟存在 + 初始实例状态。
任务生命周期新实例状态 + 生命周期关系换代。
方法 + 方法虚拟存在 + 角色 / 生命周期状态。
```

当前只能选择复制原子结构、嵌套独立事务或分批发布，三者均违反分层和失败无半结构规则，因此必须先实施 `#278`。

## 3. 权力边界

| 层 | 允许 | 禁止 |
| --- | --- | --- |
| 存在 / 状态业务服务 | 检查业务参数，形成构造权封闭的规格，映射值式结果 | 接收会话、令牌、许可、仓库或候选对象；决定外层事务提交 |
| 存在场景 / 状态动态数据操作 | 接收 `结构写入会话&`，复核规格和句柄，在当前写集内参与写入并读回 | 取得第二次许可；调用自己的执行器；请求提交；释放许可 |
| 上层数据操作 | 持有唯一执行器，在一次回调中调用多个参与入口，完成总读回后请求提交 | 解释存在 / 状态内部关系；调用原子立即创建入口形成嵌套事务 |
| 结构写入会话 | 判断节点是否属于本会话候选，继续承载统一写集和收口 | 暴露候选对象、原始令牌、写集容器或领域语义 |

## 4. 规格类型

### 4.1 通用规则

新增规格均满足：

```text
不可默认构造。
构造函数私有，只允许对应业务服务形成。
只提供 const 值式读取和 完整()。
不包含仓库、令牌、许可、会话、执行器、指针或可写引用。
已发布来源与本会话候选来源使用不同枚举值或不同规格类型，不使用调用方 bool 静默切换。
```

### 4.2 虚拟存在写入规格

由 `存在业务服务` 形成，至少承载：

```text
虚拟存在幂等主键。
预期来源节点类型：需求 / 任务 / 方法之一。
来源阶段：已发布 / 本会话候选。
```

已发布来源规格必须在形成前通过当前无令牌读取复核来源类型；本会话候选来源规格只能由明确的未来来源请求形成，参与入口必须再次按当前会话候选身份和实际节点类型复核。

### 4.3 抽象状态写入规格

由 `状态业务服务` 形成，承载：

```text
状态幂等主键。
I64 状态值材料。
```

规格不改变“需求目标是状态节点”的语义。I64 只写入状态主信息材料。

### 4.4 已发布主体实例状态写入规格

继续复用既有 `实例状态写入规格`：

```text
状态幂等主键、场景、主体、发生时间戳、状态值。
```

形成规格时要求场景和主体已经发布；参与入口在同一独占会话内再次复核完整句柄、类型和主信息。

### 4.5 本会话候选主体实例状态写入规格

新增独立规格，由 `状态业务服务` 形成，承载：

```text
状态幂等主键、场景、发生时间戳、状态值。
预期主体类型固定为存在。
```

主体句柄不预存在规格中，由上层数据操作在同一回调中创建虚拟存在后传入。参与入口必须确认主体属于本会话节点候选、类型为存在且主信息可读；任一不符合均在第一笔状态写入前结束。

## 5. 结构写入会话窄入口

新增写入阶段只读判断：

```cpp
bool 节点是本会话候选(节点句柄 节点) const;
```

合同：

```text
只在当前线程、写入中、尚未请求提交且会话无失败时返回判断。
仅检查内部节点候选写集中的完整句柄相等性。
不标记读回，不返回候选对象，不改变提交 / 撤销行为。
无效、错仓、过期、已发布节点、跨线程或提交决定后均返回 false。
```

不得同时新增主信息候选导出、写集枚举、令牌读取或候选确认能力。

## 6. 数据操作参与入口

### 6.1 存在场景数据操作

形成公开给数据操作层使用、但不由业务服务直接调用的同会话入口：

```text
写入虚拟存在_会话(结构写入会话&, 虚拟存在写入规格, 来源节点)
```

顺序：

```text
1. 复核数据操作有效、规格完整和来源句柄有效。
2. 已发布来源复核当前完整身份；本会话来源复核候选身份。
3. 复核实际来源类型等于规格预期类型。
4. 创建主信息候选、存在节点、主键和 存在 -> 来源 引用关系。
5. 同许可读回主信息、节点、主键和完整关系。
6. 返回值式参与结果；不请求提交。
```

现有 `创建虚拟存在` 改为“形成已发布来源规格 -> 自己的执行器 -> 调参与入口 -> 完整时请求提交”，保持公开请求、结果、幂等和并发语义。

### 6.2 状态动态数据操作

形成三个同会话入口：

```text
写入抽象状态_会话(结构写入会话&, 抽象状态写入规格)
写入实例状态_会话(结构写入会话&, 实例状态写入规格)
写入候选主体实例状态_会话(结构写入会话&, 候选主体实例状态写入规格, 主体节点)
```

抽象状态顺序：主信息候选 -> I64 状态值 -> 状态节点 -> 主键 -> 最小读回。

实例状态顺序保持当前权威结构：

```text
主信息候选。
状态值、发生时间戳和幂等编号高低位 I64 材料。
状态节点和主键。
场景 -> 状态 运行期临时关系。
状态 -> 主体 引用关系。
状态 -> 场景 引用关系。
```

两个实例入口共用同一个私有结构写入实现；区别只在写前主体资格：已发布主体按当前完整身份复核，候选主体按本会话候选身份复核。现有立即创建入口复用对应参与入口并保持语义。

## 7. 提交所有权与调用路径

原子立即创建：

```text
存在 / 状态业务服务
-> 形成原子规格
-> 原子数据操作自己的执行器
-> 原子同会话参与入口
-> 原子数据操作请求提交
```

后续 #269 组合创建：

```text
需求 / 任务 / 方法业务服务
-> 调用存在 / 状态业务服务形成规格
-> 数据操作.需求任务方法的唯一执行器
-> 创建自身候选
-> 调用存在 / 状态同会话参与入口
-> 形成自身关系并总读回
-> 数据操作.需求任务方法唯一请求提交
```

参与入口不得在任何路径自行调用 `请求提交()`。同一回调中只能有一个提交决定所有者。

## 8. 幂等与并发

```text
规格形成前按现有无令牌读入口识别已存在同义事实、异义冲突和读取异常。
进入独占会话后再次复核所有已发布输入和候选来源阶段。
绑定主键或关系出现并发冲突时，整个会话收口；释放许可后由立即创建入口或后续上层数据操作按权威读回映射幂等 / 冲突 / 漂移。
参与入口本身不把索引冲突解释成业务成功。
```

## 9. 非成功返回与失败收口

### 9.1 逻辑内返回

```text
规格形成前的零主键、非法来源角色、无效句柄、错误类型、零时间戳。
同许可第一笔参与者写入前确认的版本漂移、已发布 / 候选阶段不匹配。
并发同键异义冲突经释放许可后的权威读回明确映射。
```

### 9.2 追根因解决

```text
规格已形成且同许可写前复核通过后，任一候选、I64、关系、索引或读回不符合规格。
同一版本下本会话候选身份、节点类型和主信息互相矛盾。
无提交、失败或异常收口后仍有可读节点、主信息、关系、索引或 I64 残留。
```

收口统一复用当前会话逆序写集；不得由原子参与入口物理删除、另取许可、调用旧兼容写或扫描整仓修复。

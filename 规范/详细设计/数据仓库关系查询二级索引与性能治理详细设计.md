## 1. 目标

`规范/仓库与服务分层事务边界规范.md` 和 #262-#271 架构纠偏链已经完成。本设计记录关系查询性能基线、二级索引、令牌 / 批量读取、分片和控制面板投影的证据门控路线及最终裁决：#257 基线已完成；#258-#260 已按正式证据收口为不实施 / 不分片；#261 已转入独立当前接口专项。

本设计不把当前六项显示分类解释成六棵物理树。概念结构按多根森林处理，并允许多父共享节点和图关系。

## 2. 当前代码事实

1. `关系仓库` 当前用 `unordered_map<关系编号, 关系记录>` 保存唯一权威主表，并由一把 `shared_mutex` 保护。
2. 创建重复检查、关系审计组、子节点、正向 / 反向关系组、节点相关关系、存在判断、有效数量和删除准备包均存在遍历整张关系表的路径。
3. `索引仓库` 当前是独立的主键到节点候选入口，不承载关系事实，也不适合承担关系写入同锁一致性。
4. 四个核心仓库各有独立读写锁；结构事务许可与活动许可表另有协调竞争路径。
5. #257 已形成 1,000 / 10,000 / 100,000 三档正式基线；100,000 关系十类查询的候选上界均为 100,002，确认当前查询仍按全表规模检查候选，但不证明生产运行饱和或分片必要。
6. 当前关系写入不只有创建、失效和重挂，还包括普通父子挂载 / 重挂、已发布关系变更撤销以及概念安全删除的批量失效；S1 必须覆盖所有会改变 `关系表_` 当前记录的路径。
7. 当前 `删除关系` 和 `严格删除关系` 都把权威记录推进为 `已删除` 并保留在关系表，不存在物理移除；旧带令牌删除保持共享兼容，严格删除入口验证独占令牌。
8. 控制面板当前可生成六类完整结构，但六类只是显示分类；显示层不得成为事实源。

## 3. 权力与结构边界

```text
权威关系：关系表当前记录。
派生候选：关系仓库内部二级索引中的关系编号集合。
领域入口候选：通用索引仓库现有主键 / 命名空间索引。
机器裁决：领域服务 + 关系仓库对权威记录和端点的复核。

```

不得为世界、概念、需求、任务、方法或显示分类建立互不相通的身份体系。不得把二级索引持久化结果、命中次数、性能报告或控制面板投影解释成机器事实。

## 5. S1 二级索引结构

二级索引作为 `关系仓库` 私有成员，与关系表使用同一把仓库锁：

| 索引 | 键 | 值 | 用途 |
| --- | --- | --- | --- |
| 有效正向索引 | 完整源节点句柄 + 关系类型 | 按顺序号、关系编号可稳定排序的关系编号候选 | 目标组、来源关系记录组、重复 / 唯一性检查 |
| 有效反向索引 | 完整目标节点句柄 + 关系类型 | 关系编号候选 | 来源组、目标存在、普通父子单父检查 |
| 有效节点相关索引 | 完整节点句柄 | 关系编号候选 | 节点删除影响面、节点相关关系读取 |
| 当前审计正向索引 | 完整源节点句柄 + 关系类型 | 有效或已失效当前记录的关系编号候选 | 当前版本审计组 |

键必须使用完整稳定句柄，不能只用本地节点编号。值只保存关系编号，不复制权威状态、端点或版本。审计索引只支持当前版本有效 / 已失效记录，不虚构旧版本历史链。

候选桶的具体容器不是机器合同，但实现必须满足以下提交成本边界：

```text
同一关系变更对每个受影响桶只定位一次，不在提交前后重复散列查找同一成员。
稀疏桶不得为每个关系编号叠加一层独立哈希节点后又在状态切换中立即回收。
允许使用紧凑候选桶、连续关系编号和私有位置 / 附着元数据实现定向插入与移除；这些元数据仍只是索引维护材料，不是权威关系。
查询返回顺序继续由权威记录的顺序号、关系编号排序，不依赖桶容器顺序。
实现若保留首版嵌套哈希集合，必须先用同一固定夹具证明全部写入门禁；未证明前不得原样提交。
```

## 6. 查询流程

1. 入口先拒绝无效节点句柄和未定义关系类型。
2. 在关系仓库共享锁内按键复制关系编号候选。
3. 对每个编号读取关系表当前记录，复核关系编号、当前版本、类型、状态、源 / 目标完整句柄和可选顺序号。
4. 释放关系仓库锁后按现有事务语境复核端点有效性；不得让显示层或调用方自行解释端点。
5. 按现有契约以顺序号、关系编号确定性排序；节点组不得因索引容器顺序改变返回顺序。
6. 普通业务读取只返回有效关系；显式审计读取才可返回当前已失效记录。
7. 同一结构许可内发现索引候选与关系表不一致属于逻辑错误，停止后续实施或验收；不得自动全表扫描修复。

## 7. 同一提交边界

### 7.0 受影响桶转换计划

每个写入口在持有原有关系仓库独占锁后，先从写前 / 写后记录形成一次值式转换计划。计划只列出本次需要保留、插入和移除的正向、反向、节点相关与审计桶及关系编号；同一键只出现一次。

```text
入口前置拒绝
-> 一次定位写前成员与目标桶
-> 为可能分配的目标桶预备容量 / 节点
-> 定向发布新增成员
-> 定向移除旧成员
-> 更新权威当前记录
-> 最小受影响项读回
```

普通热路径不得在同一次切换的开始和结束各执行一轮通用五项成员复核。生产路径仍必须逐个证明受影响旧成员存在、目标成员不存在或可幂等沿用，并在提交后复核本次受影响项；Debug 全索引遍历和全表参考模型验证保留在专项自检，但不得进入正式写入 P95 计时区间。

### 7.1 创建

写前用相关桶完成完全重复、单父、任务生命周期槽位和任务方法选择唯一性检查。先预备所有可能分配内存的新索引项，再在同一独占锁内发布主表和索引。任一步异常必须撤销本次已插入项和主表记录，不返回有效关系句柄；编号允许保持单调跳号。

### 7.2 失效

复核精确当前句柄和本次受影响成员后推进关系版本和状态；从三个有效索引定向移除，当前审计索引继续指向同一关系编号且不重复删除 / 插入。重复失效保持现有幂等语义，不重复推进版本。

### 7.3 重挂

先按新端点执行全部入口拒绝和唯一性检查，再预备新端点索引项。提交时更新同一关系编号、推进版本、移除旧有效 / 审计键并发布新键。旧句柄失效，新句柄及新端点候选可读；任何失败不得留下新旧端点同时作为有效事实。

普通父子 `挂载或重挂节点` 与 `结构化挂载或重挂节点` 也是创建 / 重挂路径，必须复用同一索引提交辅助，不得保留旁路扫描写入。

### 7.4 逻辑删除

只接受现有授权和精确当前句柄。一次定位三个有效索引、审计索引与主表的本次成员，再在同一锁内把权威记录推进为 `已删除` 并定向移除全部索引候选；关系表记录继续保留，不执行物理 `erase`。不得以提交前后两轮通用全成员复核替代该转换计划。

旧 `删除关系(..., 令牌)` 的共享兼容验证和 `严格删除关系(..., 令牌)` 的独占验证均保持原义。S1 不统一、放宽或删除任何兼容签名，不改变事务 ABI 或许可强度。

### 7.5 确认与撤销

`结构化确认已发布关系变更` 不再改变关系记录，只复核写后记录及索引处于完整一致状态后确认能力。`结构化撤销已发布关系变更` 必须在同一独占锁内恢复写前记录、恢复写前索引并移除写后索引；任何失败不得留下两个端点、两个状态或两个版本同时可作为当前候选。

### 7.6 批量失效

`准备节点关系失效包` 使用节点相关索引召回候选后仍逐记录复核完整句柄、版本、类型、状态和端点。`提交节点关系失效包` 必须先复核整批，再在同一独占锁内把每条关系推进为已失效、移除三个有效索引并保留当前审计候选；不得提交部分批次。

## 8. 多根、多父与图关系

1. 二级索引不内置“整仓库只有一个根”假设。
2. 同一节点可在允许多父语义的关系类型中由多个源指向；每条关系以完整关系句柄独立保留。
3. 图关系可以成环；只有明确要求无环的普通父子或领域专用关系由原所有者拒绝成环。
4. 同一节点从多个根可达时，关系查询返回全部有效边；显示投影可以按完整节点句柄去重节点展示，但必须保留边和引用位置。
5. 完全重复关系继续按关系类型规则写前拒绝，不能用显示去重掩盖重复事实。

## 9. S1 验收阈值

正确性是硬门槛，性能收益不能抵消正确性失败：

```text
A. 全部查询结果与隔离全表参考模型逐项一致。
B. 创建、失效、重挂、逻辑删除、撤销和批量失效后，主表和所有索引的一致性检查为 100%。
C. 100,000 关系稀疏夹具中，单次候选检查数不超过对应桶实际关系数，且不再与全表 N 成正比。
D. 相同机器 / 配置下，稀疏正向、反向、存在、节点相关查询 P95 均不高于 S0 的 50%。
E. 与 S0 相同的 4 线程、90% 读 / 10% 重挂写固定并发场景吞吐不低于 S0 的 150%。
F. 纯写场景 P95 不得高于 S0 的 150%，且入口拒绝、版本和事务回归全部通过。
G. 小规模 1,000 关系完整自检总耗时不得出现超过 10% 的稳定回归；连续三轮超过即停止验收。
H. 三档结构签名必须与 S0 完全一致；S1 候选检查数必须来自 Debug 专项实际索引桶计数，不得用参考模型桶大小冒充。
```

若 D-G 因测量抖动未稳定满足，保留正确实现但不得宣称性能验收通过，必须重复基线或退回设计裁决。

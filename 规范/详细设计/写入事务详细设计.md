# 写入事务详细设计

编号：013-D1
日期：2026-07-05
状态：已确认详细设计 / 核心内核 P0 / 可作为第一轮代码实施依据

## 1. 设计目标

写入事务用于定义跨仓库写入边界。最小闭环会跨节点、主信息、关系、索引、特征值、需求、任务、方法、动态和因果引用；若中途失败，必须有明确失败边界，不能留下不可解释的半结构。

第一版事务可以先是最小事务壳，但必须明确不宣称完整 ACID，不宣称崩溃恢复，不宣称并发隔离。它的优先价值是把写入阶段、失败边界、提交状态和禁止宣称写清楚，避免领域服务在雾里摸鱼骨。

## 2. 规范承接

| 上层规则 | 本设计承接 |
| --- | --- |
| 结构闭环 | 事务只组织结构写入，不用日志或控制台输出替代结构事实 |
| 入口前置拒绝 | 写入前先拒绝错误输入，事务不替代仓库入口拒绝 |
| 领域服务封口 | 领域服务负责组织跨仓库事务，外部业务模块不得裸写仓库 |
| 完成声明规则 | 未实现回滚、崩溃恢复和隔离前，不得扩大宣称事务能力 |
| 最小闭环 | 种子准备、需求触发、方法执行、结算拆成明确事务边界 |

## 3. 当前能力边界

当前事务壳只允许声明：

```text
可开始事务。
可提交事务。
可取消事务或失败返回时标记边界。
可作为调用路径中的显式边界。
可记录人读诊断所需的阶段名和失败原因。
```

当前不得声明：

```text
跨仓库原子回滚已完成。
崩溃恢复已完成。
并发隔离已完成。
持久化事务已完成。
事务日志已可恢复全部结构。
```

## 4. 事务阶段

```text
未开始
进行中
已提交
已取消
失败
```

阶段转换规则：

| 当前阶段 | 允许动作 | 新阶段 | 禁止动作 |
| --- | --- | --- | --- |
| 未开始 | 开始 | 进行中 | 提交、取消、记录写入 |
| 进行中 | 提交 | 已提交 | 再次开始 |
| 进行中 | 取消 | 已取消 | 提交后取消 |
| 进行中 | 标记失败 | 失败 | 失败后继续写入 |
| 已提交 | 无 | 已提交 | 再次提交、取消、写入 |
| 已取消 | 无 | 已取消 | 提交、写入 |
| 失败 | 无 | 失败 | 提交、继续写入、伪造回滚成功 |

第一版若没有回滚日志，失败时只能返回失败并保留实施记录，不得伪造已自动恢复。

## 5. 函数设计

| 函数 | 功能 | 入口拒绝 | 读写结构 | 验证方式 | 状态 |
| --- | --- | --- | --- | --- | --- |
| `开始` | 进入进行中状态 | 已经进行中、已提交、已取消、失败状态复用 | 事务状态 | 开始后可提交 | 已实现（文档声明）/ 设计已确认 / 待复验 |
| `提交` | 提交当前事务 | 未开始、已提交、已取消、失败 | 事务状态 | 提交后返回成功，不能重复提交 | 已实现（文档声明）/ 设计已确认 / 待复验 |
| `取消` | 取消当前事务 | 未开始、已提交、已取消 | 事务状态 / 后续回滚日志 | 取消后不能提交 | 后续或第一轮补齐 |
| `标记失败` | 标记事务失败并记录失败边界 | 未开始、已提交、已取消 | 事务状态、失败说明 | 失败后不能提交 | 第一轮应具备 |
| `记录写入` | 记录事务内写入项 | 未开始、写入项类型未确认、已提交、已取消、失败 | 事务日志 | 失败时可定位半结构 | 后续 |
| `回滚` | 回滚已记录写入 | 无回滚日志、结构已外部改变、事务未失败或未取消 | 事务日志、各仓库 | 回滚后结构回到事务前 | 后续 |
| `读取阶段` | 返回当前事务阶段 | 无 | 事务状态 | 验收可检查阶段转换 | 第一轮应具备 |
| `读取失败边界` | 返回失败阶段和失败原因 | 无失败记录 | 事务状态、失败说明 | 失败路径可定位 | 第一轮应具备 |

## 6. 跨仓库写入规则

```text
领域服务是事务写入的组织者。
仓库入口必须先做入口前置拒绝。
任一前置条件不满足时，不得开始写后续结构。
每个事务必须有明确输入入口、写入范围、输出变化和失败边界。
如果已经写入部分结构而后续失败，必须记录失败边界。
未实现回滚前，不得宣称跨仓库原子性。
```

写入事务不得绕过节点仓库、主信息仓库、关系仓库、索引仓库自身的入口拒绝。事务通过只表示“组织写入流程成功”，不等于业务事实成立；业务事实仍由目标结构承载。

## 7. 最小闭环事务边界

最小闭环至少拆成四个事务：

| 事务 | 输入入口 | 写入范围 | 输出变化 | 失败边界 |
| --- | --- | --- | --- | --- |
| 种子准备事务 | 创建闭环种子 | 自我存在、现实场景、目标特征、初始特征值 | 前置结构可被需求服务读取 | 任一前置结构创建失败则不进入需求触发 |
| 需求触发事务 | 创建需求 | 需求节点、目标值、目标关系、初始需求状态、任务承接 | 需求和任务可读 | 目标存在、目标特征或当前特征值无效则入口拒绝 |
| 方法执行事务 | 执行手写方法 | 特征值更新、任务状态、动态、因果引用 | 前后值、动态、因果引用可回查 | 方法不可执行或目标特征值写入失败则标记失败 |
| 结算事务 | 需求服务结算 | 需求结算状态、结算版本、来源证据引用 | 需求状态由未满足变为满足或失败 | 当前值、比较方式或来源证据缺失则拒绝结算 |

每个事务必须有输入入口、输出变化和失败边界。控制台输出只能展示事务阶段，不能成为验收事实。

## 8. 失败处理口径

第一版失败处理采用“可解释失败边界”而不是“伪装完整回滚”：

```text
入口前置拒绝失败：不得写入任何后续结构。
事务中途失败：必须记录已经完成到哪个阶段、失败原因和后续禁止宣称。
没有回滚日志时，不得自动删除已写结构冒充原子回滚。
需要人工诊断时，可在容错宏开启后写日志和弹窗，但日志和弹窗不参与机器事实。
```

后续若要支持回滚，必须先设计：

```text
写入项类型
反向操作
依赖顺序
外部可见性
回滚失败处理
持久化恢复边界
```

## 9. 事务记录最小字段

第一版若实现事务记录，最小字段如下：

```text
事务编号或局部序号
事务名称
当前阶段
输入入口
写入范围说明
失败阶段
失败原因
是否允许继续后续事务
```

事务记录只作为执行边界和诊断材料，不替代节点、关系、特征值、需求、任务、动态或因果引用中的机器事实。

## 10. 验收条件

设计确认后，代码复验至少覆盖：

```text
未开始事务不能提交。
已提交事务不能重复提交。
已取消事务不能提交。
失败事务不能提交。
失败路径不宣称自动回滚。
入口前置拒绝失败时不继续写后续结构。
最小闭环实施记录能指出每个写入阶段的边界。
控制台输出不能作为事务成功证据。
```

## 11. 禁止宣称

本设计确认后仍不得宣称：

```text
跨仓库原子性已完成。
事务回滚已完成。
崩溃恢复已完成。
并发隔离已完成。
持久化事务已完成。
最小闭环已跑通。
代码验证已通过。
```

# 默认功能回归与显式性能验收分离详细设计

编号：SELFTEST-PERF-GATE-D1

日期：2026-07-16

状态：JY-360 正式设计 / QR-185 已裁决 / 绑定 #280 与 #261 后继复核 / 未改 C++ / 未构建 / 未运行程序

## 1. 定位

本设计解决已归档关系仓库性能基线无条件进入默认完整 Debug 自检后，环境敏感三轮稳定门禁阻断无关功能切片回归的问题。

目标不是放宽性能门槛，而是形成两个真实且互不伪装的验收入口：

```text
默认功能回归：--self-test-exit
显式性能验收：--warehouse-performance-self-test-exit
```

两种模式继续使用同一 Debug 二进制、同一中央阶段化自检总成和同一正式性能基线函数。默认模式不登记阶段 790；显式性能模式登记并执行阶段 790，性能通过或失败继续真实决定该模式退出码。

当前代码事实：

1. Debug 构建在 `入口.cpp` 中无条件把 `WAREHOUSE-OPT-S0` 以最终回归阶段顺序 790 登记到中央总成。
2. `阶段化自检总成::完成总成()` 要求所有已登记单元执行并通过；790 失败会使默认 `--self-test-exit` 和普通启动整体失败。
3. 关系仓库性能基线固定 1k / 10k / 100k、种子、三档结构签名、三轮稳定和 15% 门槛，当前实现相对 #257 正式提交 `33a92d4` 无差异。
4. #261 WIP 的 PANEL-PROJECTION-S2-A01 至 A16 两次全部通过，但 790 的 100k 三轮波动 36.0% / 39.6% 使两次完整 Debug 退出 1。
5. Release 不导入和运行性能基线；当前没有显式性能运行参数，也没有默认功能回归与性能验收的模式边界。

## 3. JY-360 裁决

QR-185 采用独立前置计划，不在 #261 中扩大核心允许范围。

```text
#280 / DQ-172 / SELFTEST-PERF-GATE-S1 / 795
```

裁决固定：

1. 默认 `--self-test-exit` 和普通启动不登记阶段 790；它们只以功能自检、结构不变式和确定性回归决定成功。
2. 新增显式 `--warehouse-performance-self-test-exit`；Debug 下登记阶段 790 并在完成全部现有功能自检后按原顺序运行。
3. 790 的函数、固定规模、种子、签名、查询 / 写入 / 并发指标、三轮稳定和 15% 门槛全部不改。
4. 显式性能模式中 790 失败必须使进程退出 1；不得降级为警告、非权威提示或功能通过。
5. Release 收到显式性能参数时在结构装配前逻辑拒绝并退出 2；不得静默跳过性能后返回 0。
6. 新阶段 795 只验证运行模式、条件登记和退出码接线，不重复测量仓库性能，不形成第二套性能事实。
7. #261 只有在 #280 正式完成、阶段 795 登记并由设计窗口重读实际接口后，才允许恢复七文件 WIP。

## 4. 运行模式合同

| 模式 | 入口参数 | Debug 阶段 790 | 阶段 795 | 退出语义 |
| --- | --- | --- | --- | --- |
| 普通启动 | 无性能参数 | 不登记 | 登记并验证未启用 | 功能 / 启动失败为 1，继续常驻由原路径决定 |
| 默认功能回归 | `--self-test-exit` | 不登记 | 登记并验证未启用 | 全部功能自检通过为 0，否则 1 |
| 显式性能验收 | `--warehouse-performance-self-test-exit` | 登记并执行一次 | 登记并验证已启用 | 功能与 790 全部通过为 0，任一失败为 1 |
| Release 显式性能请求 | 同上 | 不可用 | 不运行 | 写前拒绝并退出 2 |
| D455 / 数据库专项 | 既有参数 | 不登记 | 按既有 Debug / Release边界 | 原专项语义不变 |

同时出现 `--self-test-exit` 与显式性能参数时，采用显式性能验收的严格超集语义：运行功能自检和 790，并在自检后退出。不得因参数顺序不同产生不同结果。

## 5. 模块与入口归属

### 5.1 新自检模块

| 文件 | 模块 | 工程项 | 归属 |
| --- | --- | --- | --- |
| `海中鱼巣/自检.性能门禁分离.ixx` | `海中鱼巣.自检.性能门禁分离` | `ClCompile` | Debug 自检模块；只验证模式和登记合同 |

该模块不得拥有性能算法、关系仓库、阈值、命令行全局状态或生产修复能力。它只接受入口形成的值式模式 / 登记证据并返回 `自检单元结果`。

### 5.2 `入口.cpp`

`入口.cpp` 只允许增加：

```text

一个外部参数分支。
Release 对性能模式的一次写前拒绝。

新自检模块 import、阶段 795 一次登记和最终布尔汇总。
```

禁止把性能测量、循环、阈值计算或完整 A01-A14 自检正文写入入口。

## 6. 条件登记与退出码

入口必须区分：

```text
是否请求性能验收。
阶段 790 是否实际登记成功。
阶段 790 是否实际执行并通过。
阶段 795 是否登记并通过。
中央功能总成是否通过。
```

默认模式下，790 未登记是合同规定的“未启用”，不能当作登记失败；中央总成登记数量和执行数量必须自然排除 790。显式模式下，790 登记失败、未执行或执行失败均使总结果失败。

阶段 795 在默认模式验证“未请求且未登记”，在显式模式验证“已请求且已登记”。若模式与登记状态不一致，属于前置通过后的内部接线错误，必须进入追根因路径。

## 7. #261 WIP 与 Git 边界

执行 #280 前，执行窗口 `D:/海中鱼巣-执行-265` 存在 #261 的六个已修改文件和一个未跟踪模块。#280 与其中 `入口.cpp`、工程和 filters 路径重叠，因此不得直接在 dirty 树上实施。

固定处置：

1. 先确认 dirty 集恰好为 JY-359 记录的七个路径，暂存区为空；多一个或少一个都停止。
2. 按七个精确路径创建包含未跟踪文件的本地可逆 stash，说明固定为 `QR-185 #261 WIP 本地救援，等待 #280 后设计复核`。
3. 记录 stash 对象身份，不使用 `stash@{n}` 作为长期身份；不得读取、恢复或删除 #258 的救援 stash。
4. 工作树干净后纯快进到 JY-360 正式提交，再实施 #280。
5. #280 完成后不自动恢复 #261 stash；设计窗口先重读 #280 正式接口并修订 #261 恢复 / 冲突收口合同。

## 8. 非成功收口

| 场景 | 分类 | 收口 |
| --- | --- | --- |
| dirty 集不等于七文件或暂存区非空 | 追根因解决 | 不 stash、不覆盖、不实施 #280 |
| Release 请求显式性能验收 | 逻辑内返回 | 结构装配前退出 2，说明当前构建不支持 |
| 默认模式意外登记 / 输出 790 | 追根因解决 | #280 失败，不得把 790 结果从退出码中硬剔除 |
| 显式模式未登记 790 | 追根因解决 | #280 失败，不得伪装成性能验收 |
| 显式模式 790 未达 15% 门槛 | 合法性能失败 | 保留指标并退出 1；不阻止验证“分离接线已生效”，但不得声明性能通过 |
| 功能自检失败 | 追根因解决 | 默认和显式模式均退出 1，不能归因给性能环境 |
| #261 stash 恢复出现冲突 | 设计后继门控 | #280 不恢复；由后续 #261 计划按正式接口裁决 |

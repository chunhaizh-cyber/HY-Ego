## 1. 定位

本设计把 #201 发现的任务系统断点拆为五个不可越级的实施切片：

```text
TASK-LIFECYCLE-S1：唯一当前态关系、生命周期版本和审计历史读回。
TASK-LIFECYCLE-S2：合法迁移矩阵、并发版本准入、终态封口和专用失败 / 取消证据。
TASK-PLANNING-S1：筹办材料、方法召回建议和任务服务独占选择提交。
TASK-EXECUTION-S1：冻结执行请求、管理 / 工作线程调度、受控执行和强类型非权威回执。
TASK-RESULT-S1：正式结果复核、目标比较、任务完成和独立需求结算。
```

当前代码已由 #221/#222 形成任务生命周期关系14、证据关系15、唯一当前态、完整历史、幂等迁移和终态封口；筹办选择仍只有顺序号30方法引用，尚无选择记录、完整来源审计或精确动作桥。本文第7节及后续仍是预期正式契约，不得写成当前实现事实。

JY-292、JY-294、JY-296 已根据 #221 三轮实际接口复核补齐存在创建链、需求承接读取链与原子服务候选撤销能力；JY-297 只修订关系枚举负例的回归映射，不改变生产合同。第 5 节和第 15 节以这些修订为准。

JY-299 根据 #222 在 clean HEAD `f6eac60` 的执行前复核补齐生命周期迁移幂等材料：幂等编号随新实例状态由状态服务权威写入，完整迁移证据由任务服务以固定角色关系承载；精确重复必须从完整历史逐字段复核，不能复用时间戳、生命周期版本、摘要或日志。

JY-300 根据 #222 在 clean HEAD `88cba9f` 的第二次复核修正角色关系唯一性：不强迫合法业务角色目标互异，新增 `关系类型::任务生命周期证据 = 15`；只对该类型按同源状态 + 角色顺序号唯一，普通引用端点唯一性保持不变。

JY-302 根据 #223 在 clean HEAD `0ffb8da` 的执行前复核补齐选择身份权威承载：新增 `节点类型::任务方法选择记录 = 14` 和 `关系类型::任务方法选择 = 16`。选择记录节点以版本化主信息值容器无损保存完整扁平请求、幂等编号、生命周期身份、双向授权、批次、索引、登记、规格、条件、上下文和动作来源；关系 16 发布当前记录。原顺序号 30 `引用(任务 -> 方法)` 只保留为当前方法投影，必须与记录逐字段一致，不能单独授权。

### 2.2 当前禁止结论

```text
不得用最大时间戳、最大枚举、最大节点编号、状态数量或日志选择任务当前态。
不得把方法建议、排序第一项、运行消息、线程状态、普通回执或上行消息升格为任务授权或结果。
不得把固定安全根的硬编码方法、数值、时间偏移或 lambda 注入当作通用执行器。
不得在任务服务内直接比较裸 I64 来宣称目标状态已满足。
不得把任务完成和需求结算合并成一个可回滚边界。
不得把 #217 单进程结构隔离扩大为崩溃、断电或跨重启原子恢复。
```

### 3.1 TASK-LIFECYCLE-S1 需求材料读取边界

任务创建在取得 #217 独占许可后、第一笔任务结构写入前，必须通过需求服务公开只读令牌重载复核来源需求仍有效并取得当前承接材料。正式最小接口为：

```text
需求是否有效(节点句柄, const 结构事务令牌&) const
读取需求承接材料(节点句柄, const 结构事务令牌&) const
```

两个入口复用需求服务既有私有令牌辅助和同一值式接线，不新增需求写入口，不保存令牌，也不允许任务服务直接读取需求仓库。许可前形成的需求材料只能作为请求材料，不能替代许可内权威复核。

```text
所属生命周期状态节点、阶段与生命周期版本

方法登记 / 生命周期完整关系身份与版本
结果节点、结果规格关系、规则版本与目标值材料
条件策略可空标记、关系身份、规则版本
条件项数量，以及每项条件节点、场景、特征、期望值、关系、原始版本、实例槽位、当前值、当前值原始版本和复判结论的定长元组
动作入口、角色 / 状态、归属、输入规格、输出规格及其完整关系身份 / 版本
```

每个 `uint64_t` 采用两个非负 I64 槽保存低 / 高 32 位，读取时无损组合；节点句柄和关系句柄按仓库编号、对象编号、版本号固定顺序展开。变长条件组先保存数量，再按稳定条件身份顺序写定长项；空可选值使用显式存在标记，不用零句柄猜测。任何缺槽、额外槽、数量不匹配、半个 uint64、重复条件身份或格式版本未知均为内部结构矛盾。

任务服务是记录主信息的唯一写入方和解释方。主信息仓库只提供通用值容器，不理解任务语义；显示、日志、方法召回、方法服务和线程都不得直接解释槽位。记录发布后不可改写；来源后来失效只改变“当前性 / 可执行性”判断，不修改历史快照。

### 7.5 选择提交与审计

只有任务服务可提交：

```text
验证任务自有扁平请求形状
-> 取得狭窄任务 / 同域写边界
-> 重读当前生命周期仍为请求中的筹办关系 / 版本
-> 重读双向授权配对两条关系完整句柄 / 版本
-> 重读完整批次身份、索引发布版本、方法登记 / 生命周期 / 规格 / 条件 / 上下文 / 动作来源
-> 按任务完整选择审计先查幂等编号，再处理当前记录冲突
-> 创建未发布选择记录主信息 / 节点并写入完整规范化元组
-> 创建顺序号 30 引用(任务 -> 方法) 当前投影
-> 把投影关系完整身份写回尚未发布记录
-> 最后创建任务方法选择关系 16，作为唯一发布点
-> 正反读回记录关系 / 节点、投影、所属生命周期和全部来源
```

当前选择 DTO 至少返回任务、方法、关系 16 完整句柄 / 版本、选择记录节点、顺序号 30 方法投影完整句柄 / 版本、所属筹办生命周期关系 / 状态 / 版本、双向授权材料、完整不可变来源快照和是否幂等。方法登记 / 规格 / 动作语义仍由方法服务在动作桥读取时重读。关系 16 与投影关系任一单独存在都不是完整选择身份。

幂等编号在同一任务全部选择审计记录中唯一。一项同键且完整规范化请求逐字段相同，读回原选择记录和当前投影；一项同键但任一字段不同，按同键异义拒绝；多项同键追根因。同一生命周期版本已有任一其它选择时写前拒绝，不用新排序结果覆盖。

`待重筹办 -> 筹办中` 是旧选择失效的唯一生命周期钩子。任务服务在同一独占许可内使旧关系 16 和顺序号 30 方法投影转已失效，并保留不可变记录节点、主信息及关系审计；新周期在尚未选择时没有有效记录关系或投影。读入口不得发现旧周期后顺手失效，不得惰性清理或扫描修补。

关系 16 是新选择唯一发布点。发布点前失败必须在同一许可内删除本次投影和所有未发布记录关系，再删除记录节点与主信息，恢复完整前态；清理失败追根因。关系 16 创建后不得普通失败；写后唯一读回不一致追根因，不做摘要补偿或跨重启回滚声明。

### 7.6 精确动作桥与下游接口

任务服务必须提供：

```text
读取任务承接授权材料(任务)
读取当前任务方法选择材料(任务)
复核当前任务方法选择仍为当前(当前选择材料)
读取任务方法选择审计(任务)
```

方法服务可单向包含任务服务以读取 `当前任务方法选择材料`；动作桥只接受该材料，并复核关系 16、选择记录节点、顺序号 30 投影、任务、方法和所属生命周期身份，再由方法服务重读登记、生命周期、规格、动作入口归属和主动可调用来源。任意 task -> method 引用、失效记录关系、旧生命周期记录或只有方法节点 / 投影的材料均不得通过。

首次选择不自动迁移到排队中。#214 只消费任务服务的双向授权和精确当前选择只读材料；#224 在冻结、入队、出队和执行前继续复核同一选择关系 / 版本，不得自行从任意引用重建选择。

## 8. S4 通用执行与强类型回执

### 8.1 冻结请求

新增版本化强类型请求候选：

```text
任务执行请求版本 = 1
任务执行冻结请求
  请求编号与幂等键
  任务、来源需求和双向承接授权两条关系完整句柄 / 版本
  生命周期关系 / 版本和当前筹办阶段
  当前任务方法选择记录关系 / 节点、方法投影完整句柄 / 版本和所属生命周期身份
  方法首和方法登记 / 生命周期版本
  方法规格版本、动作入口和动作规格
  主体、场景、目标状态
  创建时间、截止时间和尝试序号
```

普通 `运行消息` 只携带请求编号、类型和路由摘要作为信封。完整请求存放在任务管理线程持有的私有、有锁、有界强类型队列中；不得把完整身份压缩成裸节点编号后再猜回句柄。

### 8.2 入队与生命周期

```text
任务执行服务重读双向授权和精确当前选择并形成冻结请求
-> 管理线程尝试写私有队列
-> 满、停止、重复、过期：逻辑内返回，生命周期不变
-> 入队成功：任务服务重读授权、选择关系 / 版本并迁移 筹办中 -> 排队中
-> 工作线程出队：任务服务重读同一精确选择并迁移 排队中 -> 执行中
```

入队成功本身不是任务事实。若入队后生命周期复核发现请求已经过期，过期请求按值式结果丢弃，权威生命周期只能经任务服务的合法迁移处理。

### 8.3 受控执行

第一版适配表只登记已有正式领域入口，例如特征状态写入方法。适配项必须绑定方法登记、规格版本、动作入口类型和领域服务调用形状；它不是任意函数指针表，也不允许插件、脚本、SQL、外部进程、外设或用户回调。

工作线程只调用任务执行服务。实际状态和动作动态由方法服务或目标领域服务形成并完整读回。写后证据不完整属于追根因；允许等待或重筹办则返回专用材料并由任务服务迁移，不伪造成功回执。

### 8.4 强类型回执

回执至少包含：

```text
协议版本、请求编号、尝试序号
任务、双向授权关系、生命周期关系和来源版本
精确当前任务方法选择关系 / 版本、方法首、动作入口和规格版本
实际状态、动作动态、主体和场景
执行开始 / 完成时间和材料序号
```

协议、队列和上行桥只证明材料形状与非权威顺序。回执不能直接写任务结果、完成或需求结算。

## 9. S5 结果、完成与需求结算

### 9.1 正式结果治理

上行桥继续只读复核并投递治理消息。任务结果服务收到消息后必须再次读取：

```text
任务当前生命周期和版本仍匹配执行请求。
任务双向承接授权、精确当前选择关系 / 版本、方法登记、规格和动作入口仍匹配。
实际状态的主体、场景、值材料和来源证据完整。
动作动态的来源方法 / 动作、前后材料和时间完整。
回执编号、尝试序号和既有结果不存在冲突。
```

过期、精确重复或允许拒绝在写前逻辑内返回。同一当前版本不变但权威结构与已准入回执自相矛盾时追根因，不能把协议通过当成领域成功。

### 9.2 目标比较

状态服务新增结构化比较候选：

```text
状态目标比较结果
  已达成
  未达成
  材料不足
  内部不一致
```

比较基于目标状态定义、实际状态材料、主体和场景，不向任务服务暴露“比较两个裸 I64 即得业务语义”的快捷入口。未达成时保留状态与动态事实，不记录最终任务结果；任务按当前证据进入等待中或待重筹办。

### 9.3 结果与完成提交

目标达成后，在 #217 同域独占许可中：

```text
重读当前执行中关系和版本
-> 重读目标比较与动作证据
-> 写唯一任务实际结果关系
-> 经专用完成入口换代到已完成
-> 读回实际结果、完成关系和生命周期历史
```

精确重复读回既有完成材料，不创建第二结果或第二生命周期版本。同任务同版本的冲突结果是内部不一致。失败、取消任务不得被结果回执改成已完成。

### 9.4 独立需求结算

任务完成读回后才调用需求服务正式结算。结算入口重新核对来源任务、需求目标、任务实际状态、动作动态、主体和场景。结算暂不可执行或允许拒绝时保留任务已完成和动作事实，返回待结算材料并允许同证据重试；不得回滚任务结果，也不得把需求直接标记为满足。

### 15.6 JY-294 需求承接材料实际接口修订


### 15.7 JY-296 原子服务未发布候选能力修订

#221 第三次实现复核确认，JY-292 只定义存在 / 状态创建与读回令牌入口，没有定义任务顶层可调用的原所有者撤销合同。任务服务直接删除虚拟存在、实例状态或内部关系会越过原子服务边界；不清理会留下半结构。


实际中文签名可按现有风格小修，但能力必须不可默认构造、不可复制，私有构造只向对应原子服务开放；公开只读访问只暴露任务组合所需主句柄。确认 / 撤销必须验证同一服务实例、域、纪元、许可序号、线程和未发布状态。不得使用裸句柄组、公开删除授权、可伪造 bool 或日志文本替代能力。

任务顶层固定顺序为：需求同令牌复核 -> 任务节点 / 主信息 -> 存在候选 -> 状态候选 -> 任务自有关系 -> 完整读回 -> 状态 / 存在候选确认 -> 返回成功。任一步失败按“任务自有关系 -> 状态候选 -> 存在候选 -> 任务节点 / 主信息”逆序收口。若候选类型必须新增独立数据文件、修改 #217 ABI 或开放任意已发布结构删除，执行窗口必须再次退回设计。

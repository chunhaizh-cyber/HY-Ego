## 3. 第一批范围裁决

### 3.1 纳入范围

1. `VecI64` 原始值的整序列写入和副本读取。
2. `VecU64` 原始值的整序列写入和副本读取。
3. `I64 / VecI64 / VecU64` 在同一特征值节点上的类型互斥。
4. Vec 容器内容版本的首次建立和同类型整体替换递增。
5. `特征服务` 对 `特征值服务` 的业务入口封口。
6. 既有 I64 路径的类型冲突前置拒绝和回归验证。
7. 独立特征值节点并发读写的最小运行期保护。

### 3.2 明确不纳入

1. 内容哈希算法、内容哈希持久化和哈希唯一性。
2. 命中缓存、命中统计和淘汰策略。
3. 值域、区间、候选比较、稳态裁决和区间提升。
4. 当前值唯一性、候选值提升和历史值治理。
5. 序列化、快照、恢复、版本兼容和数据库投影。
6. 追加、局部修改、元素级删除、跨类型转换。
7. 需求、任务、方法等高级服务直接依赖特征值服务。
8. 控制面板、SQL、WebView、D455、体素和外设接入。

## 4. 当前代码事实

1. `特征值服务` 当前只提供特征值节点创建和 I64 主信息读写。
2. `特征服务` 是当前直接调用 `特征值服务` 的业务服务，并负责验证特征、特征值节点及归属关系。
3. `主信息仓库` 当前只承载可选 I64 主信息，不具备 Vec 原始值结构。
4. `统计服务` 只有哈希与命中材料外壳，不具备可复用的真实哈希或缓存实现。
5. 当前代码没有 Vec 值域、稳态、候选比较、序列化或恢复实现。
6. 第一批不得为了 Vec 修改主信息仓库的通用结构，也不得提前激活统计服务外壳。

## 6. 目标结构

### 6.1 原始值类型

在 `特征值服务` 内建立最小原始值类型枚举，至少区分：

```text
未建立
I64
VecI64
VecU64
```

该枚举只用于特征值服务内部裁决和经特征服务读取，不新增独立节点类型，不把普通布尔值当作机器事实。

### 6.2 Vec 容器记录

每条运行期 Vec 容器记录至少包含：

```text
特征值句柄
原始值类型
VecI64 序列或 VecU64 序列（二选一）
容器版本
```

约束：

1. 记录只能对应一个有效特征值节点。
2. 两类序列只能存在一种；不得同时有值。
3. 容器版本必须大于 0。
4. 记录发布前必须已经完整构造。
5. 节点句柄失效后的清理和复用治理不在第一批扩展范围；执行前接口复核必须确认当前节点生命周期不会让旧记录被新句柄误命中，完整句柄比较是最低门槛。

### 6.3 I64 类型识别

现有 I64 数据仍由主信息仓库承载。`特征值服务` 在写入 I64 或 Vec 前，必须能判断同一特征值节点是否已建立冲突类型：

1. I64 写入前查询 Vec 记录。
2. Vec 写入前查询现有 I64 主信息和 Vec 记录。
3. 读取类型时按互斥规则给出唯一结果。
4. 若内部发现同一节点同时存在多种类型，前置条件已无法解释该状态，必须进入追根因解决，不得任选一种继续。

## 7. 服务接口

最终函数签名由执行前接口复核依据当前代码风格确定；本设计固定语义，不提前锁死参数形式。

### 7.1 特征值服务内部入口

第一批至少需要以下语义入口：

```text
写入VecI64值
读取VecI64值
写入VecU64值
读取VecU64值
读取特征值原始类型
读取特征值容器版本
```

要求：

1. 所有写入口先拒绝无效句柄、错误节点类型、空序列、超长序列和类型冲突。
2. 所有读入口先拒绝无效句柄和错误节点类型。
3. 内部入口不得成为高级服务的新直接依赖。
4. 返回失败只表达入口拒绝、值不存在或类型不匹配；发布后读回不一致属于逻辑错误。

### 7.2 特征服务公开入口

第一批至少需要以下业务语义入口：

```text
写入VecI64特征值
读取VecI64特征值
写入VecU64特征值
读取VecU64特征值
读取特征值原始类型
读取特征值原始版本
```

`特征服务` 必须先验证：

1. 特征句柄有效且节点类型正确。
2. 特征值句柄有效且节点类型正确。
3. 特征值确实归属于给定特征。
4. 请求材料满足对应入口要求。

验证通过后才可调用 `特征值服务`。不得把归属校验下沉成写后补救。

## 8. 写入顺序

### 8.1 Vec 首次写入

```text
特征服务入口校验
-> 特征值服务句柄与节点类型校验
-> 序列非空与长度上限校验
-> I64 / Vec 类型冲突检查
-> 在临时对象中完整复制请求序列
-> 建立容器版本 1
-> 一次性发布完整记录
-> 经公开读取入口读回
-> 返回成功
```

### 8.2 Vec 同类型替换

```text
完成全部前置校验
-> 读取旧版本
-> 构造完整新记录
-> 新版本等于旧版本加 1
-> 整体替换旧记录
-> 读回值、类型和版本
-> 返回成功
```

不得原地逐元素改写后再补版本；不得在替换失败时留下部分新值。

### 8.3 现有 I64 写入

在进入既有 I64 主信息写入路径前增加 Vec 冲突检查。没有冲突时保持既有 I64 行为；不得借本切片迁移 I64 的结构承载。

## 9. 读取顺序

```text
特征服务验证特征、特征值与归属
-> 特征值服务按完整句柄查找记录
-> 检查请求类型与记录类型一致
-> 在服务保护范围内复制序列、类型和版本
-> 返回副本
```

读取不存在值或请求类型不匹配属于逻辑内返回。内部记录已存在但无法复制成一致副本，属于追根因解决。

## 10. 原子发布与并发边界

1. 服务内 Vec 记录表及其读写必须有统一保护边界。
2. 写入必须先在保护边界外或受控临时对象中完成可能失败的序列复制，再在保护范围内完成冲突复核和整体发布。
3. 读取必须在保护范围内取得一致快照，再把副本交给调用方。
4. 不同特征值节点并发写入后应能分别读回正确结果。
5. 同一节点并发写入只保证结构不损坏、版本单调递增，不在第一批定义哪个业务请求应获胜。
6. 不得依赖日志、扫描器或事后校验修复并发产生的错误结构。

## 11. 逻辑内返回与追根因解决

### 11.1 逻辑内返回

以下情况在结构写入前拒绝：

1. 无效或失效句柄。
2. 节点类型错误。
3. 特征值不归属于指定特征。
4. 空序列或超过 4096 个元素。
5. I64、VecI64、VecU64 类型冲突。
6. 读取值不存在或读取类型不匹配。
7. 请求第一批范围外能力。

### 11.2 追根因解决

以下情况不得当作普通失败继续：

1. 发布后读回的序列、类型或版本不一致。
2. 同一节点内部同时出现两种或以上原始值类型。
3. 版本不是首次 1 或同类型替换加 1。
4. 返回副本的过程改变内部值。
5. 完整句柄不匹配却命中旧记录。
6. 前置条件已满足后，内部结构写入仍产生不可解释结果。

容错宏开启时必须写逻辑错误日志并弹窗；宏关闭时不得改变正常运行行为。诊断不得承担修复职责。

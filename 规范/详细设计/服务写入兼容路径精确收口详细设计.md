5. `装配.运行期业务.ixx` 内部构造接收接线和四仓库，但对外只返回运行期业务操作组合器；这是装配边界，不是领域服务公开写入口。

9. 35 个自检模块中有 32 个直接提及四仓库；`入口.cpp` 仍承担大量历史自检夹具装配。
10. 控制面板和 SQL 适配是只读显示 / 审计路径，不得成为机器事实写入方。

## 3. 权力层级

### 3.1 正式新业务面

以下路径是业务调用唯一推荐方向：

```text
线程 / 值式消息
-> 运行期业务请求路由
-> 运行期业务操作组合器
-> 领域组合器 / 无令牌业务服务
-> 服务专用数据操作层
-> 结构写入执行器 / 不透明会话
-> 四仓库
```

业务服务、组合器和线程公开入口不得接收、返回、保存、比较或传递原始 `结构事务令牌`，不得取得许可，不得持仓库、执行器或会话。

### 3.2 正式内部能力

以下能力不是兼容删除目标：

1. 四仓库带令牌读写 ABI。
2. 结构事务协调器、许可和令牌验证。
3. 结构写入执行器、不透明会话和事务参与者。
4. 八个服务专用数据操作模块。
5. 运行期业务装配内部构造所需的接线和四仓库引用。

这些能力只能沿内部依赖方向使用，不得通过公开访问器、任意回调、类型擦除容器或裸指针逃逸到业务服务 / 组合器 / 线程。

### 3.3 冻结兼容所有者

以下范围第一轮原义保留：

| 范围 | 原因 | 后继 |
| --- | --- | --- |
| 17 个传统服务头 | 生产、初始化、自检或兼容内部仍有调用 | #214、#224/#225、#251-#254 分批替代 |
| 概念命名治理路由 | 当前仍取得独占许可并调用传统需求 / 任务 / 方法链 | #214 重做三件套 |
| 旧自我治理和任务桥 | 真实执行、动作动态和回执链尚未重做 | #224、#225 |
| 四个历史初始化兼容模块 | #250 明确隔离保留，调用未归零 | 调用归零后独立裁决 |
| 概念生命周期 / 活动快照 | 当前概念图服务拥有 | 独立后继专项 |
| 非权威统计缓存 | 当前统计服务拥有 | 独立后继专项 |
| 概念安全删除 | 当前安全删除编排器拥有完整事务 | 独立后继专项 |
| 入口和历史自检夹具 | 直接仓库夹具仍用于旧验收 | #251-#254 |
| 控制面板 / SQL 只读适配 | 只读显示和审计，不是写入链 | #261 |

#271 不允许通过修改白名单把新出现的违规调用解释为合法兼容。兼容列表只能因后继替代而缩小；新增所有者必须退回设计并给出新的正式规则依据。

## 4. 第一轮实施范围

### 4.1 新增自检模块

```text
文件：海中鱼巣/自检.服务写入兼容边界.ixx
模块：海中鱼巣.自检.服务写入兼容边界
类型：Debug 自检真模块 / ClCompile
```

模块只验证已形成的运行期业务边界，不实现任何生产功能。它可以引用自检夹具所需的核心类型，但不得向生产模块导出事务能力。

固定验收职责：

1. 编译期证明运行期请求路由只由运行期业务操作组合器构造，不接受接线、仓库、许可、令牌、执行器或会话构造。
2. 编译期证明运行期业务装配的公开业务出口类型是只读运行期业务操作组合器，不返回仓库或数据操作。
3. 复用当前运行期上下文 / 系统角色初始化夹具，构造一个完整候选并通过新请求路由执行至少一条已登记强类型请求。
4. 对无效句柄、零稳定键、空必填关系组或类型不符材料验证写前拒绝，调用参数和四仓库数量快照不变化。
5. 对有效请求只接受强类型值式结果；不得调用传统服务、旧路由或无令牌兼容写兜底。
6. 自检模块不包含源文件扫描器；源码边界和兼容不增长由计划中的静态命令验证。

### 4.2 入口与工程

`入口.cpp` 只允许：

1. Debug 条件下 `import 海中鱼巣.自检.服务写入兼容边界`。
2. 一次自检调用。
3. 一次阶段 780 登记和总结果汇总。

完整夹具、静态断言、拒绝场景、数量快照和逐项验收正文必须留在自检模块。Release 不导入、不调用、不登记 780，也不保留自检显示名和固定证据字符串。

`海中鱼巣.vcxproj` 与 `海中鱼巣.vcxproj.filters` 只登记上述一个 `ClCompile` 真模块。

### 4.3 明确不修改

```text
核心四仓库、协调器、执行器、会话、参与者及事务 ABI
全部生产数据操作、业务服务和组合器
装配.运行期业务.ixx
启动.运行期上下文.ixx
全部生产线程、旧路由和上行桥
17 个传统服务头
四个历史初始化兼容模块
概念图生命周期、统计缓存和概念安全删除
控制面板、SQL、持久化、恢复、D455、体素和外设
```

若实施时必须修改任一上述文件才能让自检通过，视为实际接口漂移，停止并退回设计。

## 5. 入口参数与非法数据来源

自检必须通过正式业务入口验证下列写前拒绝：

| 非法材料 | 发现位置 | 收口 |
| --- | --- | --- |
| 零稳定键 / 空文本 / 空必填组 | 运行期请求路由可判定的顶层入口 | 逻辑内返回，结构不变化 |
| 无效完整句柄 | 路由或对应业务服务入口 | 逻辑内返回，不补默认句柄 |
| 节点类型不符 | 对应业务服务 / 数据操作写前复核 | 逻辑内返回，不转换为其它类型 |
| 前置关系、授权或生命周期不足 | 对应领域服务入口 | 逻辑内返回，不切回传统服务 |
| 前置通过后写入 / 读回异常 | 执行器 / 会话 / 数据操作 | 追根因解决，停止依赖路径 |

参数检测的目标是定位非法材料来源并从调用链上游修正，不是在函数内部把非法数据修补成可写结构。自检不得断言“非法输入被成功兼容”为通过。

## 6. 静态边界验收

### 6.1 新正式公开面零命中

固定扫描范围为详细设计第 2.1 节的 22 个文件。以下词项命中必须为 0：

```text
结构事务令牌
结构事务许可
取得独占许可
取得共享许可
主信息仓库
节点仓库
关系仓库
索引仓库
结构写入执行器
结构写入会话
```

`服务.概念安全删除.ixx` 必须排除在正式新公开面之外并单独作为兼容所有者登记；不得通过排除其它新服务文件规避失败。

### 6.2 运行期装配出口

静态检查必须证明：

1. 装配可在构造期接收接线和四仓库。
2. 装配公开业务出口只返回 `const 运行期业务操作组合器*` 或等价更窄值式能力。
3. 不存在读取仓库、数据操作、执行器、会话、许可或令牌的公开入口。

### 6.3 兼容基线不增长

执行前重新生成精确矩阵。若出现以下任一变化，停止代码修改并退回设计：

1. 新增传统服务头调用文件。
2. 新增生产线程取得许可或传递令牌。
3. 新增业务服务直接持仓库。
4. 新增历史初始化兼容调用。
5. 新增控制面板 / SQL 写机器事实。
6. 新增服务专用数据操作之外的生产执行器导入。

已有命中减少可以接受，但必须确认不是文件漏扫、重命名绕过或删除验收证据。

## 7. 非成功返回二分

| 场景 | 分类 | 收口 |
| --- | --- | --- |
| 执行前实际接口与计划不一致 | 逻辑内返回 / 设计退回 | C++ 零修改，记录漂移 |
| 请求参数、句柄、类型或前置关系不满足 | 逻辑内返回 | 写前拒绝，结构和数量不变化 |
| 协议允许的幂等、冲突、许可竞争或版本漂移 | 逻辑内返回 | 原样返回值式结果，不重试、不降级 |
| 正式公开面出现原始事务能力 | 追根因解决 | 停止 #271，不扩大白名单 |
| 装配出现仓库 / 数据操作逃生口 | 追根因解决 | 停止并修正来源 |
| 前置通过后结构写入、确认、发布或读回非预期 | 追根因解决 | 停止依赖路径，不切回旧入口 |
| 失败后无法证明结构回前态 | 追根因解决 | 780 不登记，形成正式断点 |

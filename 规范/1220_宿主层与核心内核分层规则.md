## 4. 当前工程定位

### 4.1 核心结论

当前阶段最合理的工程口径固定为：

`一个仓库、一个解决方案、一套共享核心内核、两个很薄的宿主壳。`

其中：

- `核心内核` 负责世界、需求、任务、方法、调度、自我线程、学习与四域控制
- `宿主层` 负责把同一套核心以内嵌方式挂到不同运行形态
- `展示层` 负责 GUI、控制台、服务控制、快照查看与外部输入输出
- `后台前端通道模块` 负责提供未来可拆进程的边界，而不是要求当前必须拆进程

### 4.2 两个宿主的含义

当前应把以下两个工程视为：

- `鱼巢`：带界面的宿主壳
- `鱼巢Headless`：控制台/服务化宿主壳

它们都不是“另一套业务系统”，而是：

- 使用同一套核心模块
- 面向不同运行环境
- 承担不同的交互入口与部署形态

### 4.3 当前不采用“前后端项目化拆分”

当前不应把工程主结构定义成传统意义上的：

- `前端项目`
- `后端项目`

原因固定为：

1. 当前主要复杂度在核心任务系统，不在页面层或网络层。
2. `GUI` 与 `Headless` 都直接复用同一套核心模块，说明真正稳定的主边界是 `宿主壳 / 核心内核`，不是 `前端 / 后端`。
3. 如果现在强拆前后端，主要新增的是通信协议、状态同步、序列化、调试复杂度与部署复杂度，而不是直接新增业务能力。


## 5. 当前推荐分层

### 5.1 核心内核层

核心内核层负责：

- 世界树与数据仓库
- 需求树、任务树、方法树
- 任务执行、任务调度、四域总控
- 自我线程、学习线程、方法沉淀
- 因果、概念、二次特征、学习写回

核心内核层必须满足：

- 不依赖具体 GUI 实现
- 不依赖具体服务壳实现
- 不依赖前端展示协议细节

### 5.2 宿主层

宿主层负责：

- 启动核心内核
- 注入相机、服务、控制台或 GUI 运行环境
- 提供停机、重启、心跳、回归测试等运行包裹逻辑
- 作为不同运行形态的薄入口

当前宿主层真实落点是：

- [鱼巢.cpp](D:\鱼巢\鱼巢.cpp)
- [鱼巢Headless.cpp](D:\鱼巢\鱼巢Headless.cpp)

### 5.3 展示与交互层

展示与交互层负责：

- GUI 窗口
- 控制台输出
- 服务控制接口
- 快照查看与前端命令投递

它的职责是：

- 读核心状态
- 发输入命令
- 呈现结果

而不是：

- 自己复制一份任务系统
- 自己维护第二套业务状态机

### 5.4 IPC 预留边界层

`后台前端通道模块` 当前应被固定解释为：

- `可选的进程间通道边界`
- `未来拆进程的预留接缝`

而不是：

- 当前就必须全面转向的 `前后端主架构`

因此：

- 当前可以保留它
- 当前不应围绕它重写核心架构
- 当前不应为了“已经有管道”就强迫系统拆成两套项目


## 6. 当前推荐的工程形态

### 6.1 允许保留的形态

当前推荐形态固定为：

1. 一个仓库
2. 一个解决方案
3. 一套共享核心代码
4. 多个薄宿主工程

当前至少包括：

- `鱼巢`
- `鱼巢Headless`

### 6.2 当前不推荐的形态

当前不推荐以下两种激进改造：

1. 为了形式统一，强行拆成传统 `前端项目 / 后端项目`
2. 为了表面简化，强行把 `GUI` 与 `Headless` 合并成单一编译项目或单一 exe

### 6.3 原因

不推荐强拆前后端的原因：

- 业务核心尚未稳定到值得承受跨进程成本
- 当前主要演化方向仍是任务系统、学习系统、方法树与宿主能力
- 跨进程会放大调试、日志、状态镜像与对象同步成本

不推荐强并单 exe 的原因：

- `GUI`、控制台、服务模式的启动约束不同
- 运行入口、消息循环、服务控制与交互方式不同
- 合并后通常只会把壳层逻辑堆进条件分支，降低维护性


## 7. 什么时候才值得继续拆分

只有当以下条件之一显著成立时，才值得把当前结构进一步拆成更强的 `前后端 / 多进程 / 独立部署`：

1. 需要远程多客户端同时连接同一个核心内核。
2. 需要把核心长期作为后台服务独立部署、独立升级。
3. 需要明显的权限隔离、安全隔离或崩溃隔离。
4. 需要 UI 崩溃不带死核心运行。
5. 需要把核心算力宿主和展示宿主部署到不同机器或不同容器。

若以上条件尚未明显成立，则默认保持：

`单仓库 + 共享内核 + 多薄宿主`


## 8. 与现有代码的对齐口径

当前应视为：

1. [鱼巢.cpp](D:\鱼巢\鱼巢.cpp) 与 [鱼巢Headless.cpp](D:\鱼巢\鱼巢Headless.cpp) 都直接导入核心模块，说明两者共享一套内核。
2. [后台前端通道模块.ixx](D:\鱼巢\后台前端通道模块.ixx) 已经提供命名管道、快照结构与服务类，说明“未来可拆”的接缝已存在。
3. 当前这条接缝应被视为 `可选能力`，不是 `必须主架构`。
4. 因此当前最稳的结构判断，不是“前后端拆分是否彻底”，而是“核心内核与宿主壳的边界是否清楚”。


## 9. 一句话收束

> 当前项目更适合被固定为“一个仓库、一个解决方案、一套共享核心内核、多个薄宿主壳”；`后台前端通道模块` 只作为未来拆进程的预留边界，而不是要求现在就强拆成传统前后端，也不要求把所有宿主硬并成单一可执行文件。


## 10. 硬约束

1. 当前工程主结构不得被定义为传统 Web 意义上的 `前端项目 / 后端项目`。
2. 当前工程应固定为 `共享核心内核 + 多薄宿主`。
3. `鱼巢` 与 `鱼巢Headless` 应被视为同一内核的不同宿主壳，不得被视为两套业务系统。
4. `后台前端通道模块` 当前只作为 IPC 预留边界，不得被误当成必须立即全面启用的主架构切分依据。
5. 当前阶段不得为了形式统一而强行把 GUI 与 Headless 合并成单一编译项目或单一 exe。
6. 只有在远程多客户端、独立部署、权限隔离、崩溃隔离或跨机器部署需求明确成立时，才允许继续推进强前后端化或多进程化。

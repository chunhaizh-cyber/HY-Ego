# 001 规则迁移清单

编号：001
日期：2026-07-05
状态：正式起步清单

## 1. 迁移原则

`海中鱼巣` 只迁移旧项目中已经稳定、可泛化、适合重构起步的规则。

不迁移旧项目的历史包袱：

```text
双向链表模板作为核心结构
旧 SQL 投影作为事实入口
旧控制面板显示逻辑
旧外设 / 体素复杂链路
旧历史兼容字段
旧项目目录和文件组织
```

## 3. 已修改规则

| 旧项目规则 | 修改原因 | 新项目口径 |
| --- | --- | --- |
| 禁止新增结构体 | 新项目是重建内核，必须定义节点句柄、节点记录、关系记录等结构 | 不禁止结构体，但每个结构必须有结构映射和生命周期 |
| 优先使用存在 + 特征 + 特征值 | 新项目还需要先建立节点、主信息、关系、索引基础层 | 先建节点 / 主信息 / 关系 / 索引，再承载存在 + 特征 + 特征值 |
| 数据库作为 SQL 投影和控制面板支撑 | 新项目第一版不接控制面板和 SQL 实时投影 | 数据库只作为持久化适配器，不裁决运行期事实 |
| 世界树类封口 | 新项目第一版尚未形成世界树类 | 扩展为领域服务封口和仓库写入封口 |

## 5. 新项目新增规则

1. 新项目不得直接复制旧项目底层容器。
2. 节点身份、主信息、关系和索引必须分离。
3. 关系类型必须显式，不得把所有关系塞进普通父子。
4. 运行期仓库和持久化仓库必须分层。
5. 函数名、变量名、类内业务成员名必须使用中文。
6. 第一版先完成最小需求任务方法闭环，再迁移旧项目复杂能力。
7. 所有结构写入函数必须在入口前置拒绝错误输入，不得先写坏结构再事后校验。
8. `高级信息` 不是独立节点类型；当前只指需求、任务、方法等既有高级业务节点角色。
9. 容错检查必须通过预编译宏启用；触发时必须日志记录和弹窗提醒，且日志弹窗只能作为人读诊断。
10. 仓库编号、节点编号、主信息编号、关系编号和创建序号必须使用 `uint64` 语义；`0` 保留无效，编号只递增不复用，长期容量问题通过仓库编号、分片、纪元和归档解决。
11. 能力状态必须按 `规范已确认 / 代码已实现 / 验证已通过 / 系统级完成禁止宣称` 四态声明，禁止把小能力通过扩大为完整系统完成。
12. 句柄版本号只表达生命周期版本，不表达普通业务值变化；业务变化必须由特征值、状态、动态、任务、方法或因果结构承载。
13. 普通 `bool` 可作为局部函数执行成功 / 失败返回值，但不得持久化为业务事实、需求满足、任务完成、方法成功或世界事实。
14. `基础信息` 是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型；当前长期基础信息包含特征、存在、场景、抽象状态、抽象动态、二次特征、因果引用等信息节点，实例状态和实例动态只作为场景内临时零散节点。
15. 基础信息服务分为特征值、特征、存在、场景、状态、动态、二次特征、因果八个原子服务；世界服务是聚合服务，不得绕过原子服务直接写基础信息结构。
16. 特征值服务只服务于特征系统，不得作为需求、任务、方法的通用值容器。
17. 实例状态和实例动态是场景内临时零散节点，必须有存在主体；抽象状态和抽象动态才作为长期基础信息节点。
18. 需求服务、任务服务、方法服务需要存在、场景、特征能力时，必须通过基础信息服务获得。

## 6. 迁移完成声明边界

本清单只能证明：

```text
旧项目部分成熟规则已迁移为新项目起步规则
迁移时已区分保留、修改和暂缓
新项目命名、结构闭环和领域服务封口规则已建立
```

不能证明：

```text
代码已实现
旧项目能力已迁移
最小闭环已跑通
数据库持久化已接入
外设 / 体素 / 控制面板已接入
```

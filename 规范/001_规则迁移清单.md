# 001 规则迁移清单

编号：001
日期：2026-07-05
状态：正式起步清单

## 1. 迁移原则

`海中鱼巣` 只迁移旧项目中已经稳定、可泛化、适合重构起步的规则。

不迁移旧项目的历史包袱：

```text
双向链表模板作为核心结构
旧 SQL 投影作为事实入口
旧控制面板显示逻辑
旧外设 / 体素复杂链路
旧历史兼容字段
旧项目目录和文件组织
```

## 5. 新项目新增规则

1. 新项目不得直接复制旧项目底层容器。
2. 节点身份、主信息、关系和索引必须分离。
3. 关系类型必须显式，不得把所有关系塞进普通父子。
4. 运行期仓库和持久化仓库必须分层。
5. 函数名、变量名、类内业务成员名必须使用中文。
6. 第一版先完成最小需求任务方法闭环，再迁移旧项目复杂能力。
7. 所有结构写入函数必须在入口前置拒绝错误输入，不得先写坏结构再事后校验。
8. `高级信息` 不是独立节点类型；当前只指需求、任务、方法等既有高级业务节点角色。
9. 容错检查必须通过预编译宏启用；触发时必须日志记录和弹窗提醒，且日志弹窗只能作为人读诊断。
10. 仓库编号、节点编号、主信息编号、关系编号和创建序号必须使用 `uint64` 语义；`0` 保留无效，编号只递增不复用，长期容量问题通过仓库编号、分片、纪元和归档解决。
11. 能力状态必须按 `规范已确认 / 代码已实现 / 验证已通过 / 系统级完成禁止宣称` 四态声明，禁止把小能力通过扩大为完整系统完成。
12. 句柄版本号只表达生命周期版本，不表达普通业务值变化；业务变化必须由特征值、状态、动态、任务、方法或因果结构承载。
13. 普通 `bool` 可作为局部函数执行成功 / 失败返回值，但不得持久化为业务事实、需求满足、任务完成、方法成功或世界事实。
14. `基础信息` 是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型；当前长期基础信息包含特征、存在、场景、抽象状态、抽象动态、二次特征、因果引用等信息节点，实例状态和实例动态只作为场景内临时零散节点。
15. 基础信息服务分为特征值、特征、存在、场景、状态、动态、二次特征、因果八个原子服务；世界服务是聚合服务，不得绕过原子服务直接写基础信息结构。
16. 特征值服务只服务于特征系统，只有特征服务可直接访问；需求、任务、方法等高级服务不得直接包含、接收或调用特征值服务。
17. 实例状态和实例动态是场景内临时零散节点，必须有存在主体、场景归属和发生时间戳；抽象状态和抽象动态才作为长期基础信息节点，且本体不携带发生时间戳。
18. 需求服务、任务服务、方法服务只允许直接依赖存在服务、场景服务、特征服务、二次特征服务、状态服务、动态服务和因果服务；需要特征状态材料时必须通过特征服务获得。
19. 所有日志都放在 `D:\海中鱼巣\日志` 目录下；逻辑错误报错必须写入独立逻辑错误日志文件；运行情况必须写入独立运行日志文件；两者只做人读诊断和运行观察，不得参与机器事实或自动修复。
20. 调试日志代码必须按切片使用预编译宏控制启用和关闭；默认构建关闭调试日志，调试完成后必须关闭对应切片调试日志代码，防止无用日志泛滥。
21. 语素入口到对应信息节点、概念追溯、上位概念和概念模板用途不得作为单值主信息字段迁移；必须由关系仓库承载多对多、多级、多路关系。
22. 轻量因果引用不是稳定因果结论；第一轮 `记录因果引用(来源动态)` 只用来源动态做准入证据，不持久化来源动态关系，后续因果可靠性由动态样本、评分、版本和复判形成。
23. 方法执行权限来自任务授权、任务承接和任务选择方法；方法执行前不再创建临时授权状态节点，后续权限治理围绕任务授权关系和任务方法关系设计。
24. 需求目标是目标状态，不是 I64；I64 只可作为第一轮状态值材料或特征状态材料，不得作为需求目标对外语义。
25. 节点移动统一视为普通父子关系重挂，通用移动逻辑只能落在关系仓库或后续确认的公共结构移动工具中；各领域服务不得重复实现移动算法。

## 6. 迁移完成声明边界

本清单只能证明：

```text
旧项目部分成熟规则已迁移为新项目起步规则
迁移时已区分保留、修改和暂缓
新项目命名、结构闭环和领域服务封口规则已建立
```

不能证明：

```text
代码已实现
旧项目能力已迁移
最小闭环已跑通
权威数据库持久化或恢复已接入
完整业务操作控制面板已完成
数据库恢复已完成
旧控制面板或旧数据库能力已迁移完成
外设 / 体素已接入
```

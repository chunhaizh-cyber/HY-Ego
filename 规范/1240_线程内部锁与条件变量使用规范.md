- `wait / notify / timeout / stop` 的正确姿势是什么
- 持锁期间允许做什么，不允许做什么

- 线程锁与线程启停规范、链表锁规范应如何衔接

## 4. 范围与对象

本文所称 `线程内部锁`，包括但不限于：

- `std::mutex`
- `std::shared_mutex`
- `std::recursive_mutex`
- 与它们配套的 `lock_guard / unique_lock / shared_lock`
- `std::condition_variable`
- `std::condition_variable_any`

本文不直接管的对象：

- 数据仓库链表锁
- OpenGL 资源锁
- 外设驱动内部不可见锁

但本文会规定：

- 线程内部锁与这些锁混合时的默认禁止边界


## 5. 核心结论

必须固定以下结论：

1. 线程内部锁的首要职责，是保护线程间共享状态，不是“顺手包住一段代码”。
2. 每一份共享状态，都必须有且仅有一个主保护锁。
3. 线程内部锁默认按 `单锁化` 设计：能只持一把锁就不允许持两把。
4. 若必须多锁同持，必须有预定义顺序，且顺序必须在代码注释或规范中可见。
5. 条件变量必须与明确的条件谓词绑定，禁止裸等。
6. 同一个条件变量，所有等待方必须使用同一把配对互斥量。
7. 持锁期间不得执行与该锁保护数据无关的阻塞动作。
8. 持锁期间不得 `join`、不得等待外设、不得等待网络、不得等待另一个不相关锁。
9. 锁对象和被保护数据必须尽量共址，便于阅读和审查。
10. `recursive_mutex` 只能作为显式受限例外，不得作为默认补丁。

可收敛为一句话：

`线程内部锁的正确用法，不是“把代码围起来”，而是“让每份共享状态有唯一主锁、让每次等待有唯一条件、让每次多锁有预定义顺序，并把所有阻塞和重逻辑尽量移出临界区”。`


## 6. 数据与锁绑定规则

### 6.1 主保护锁

每一份共享状态都必须能回答：

- 它由哪一把锁保护

例如：

- 队列数据 -> 队列锁
- 配置参数 -> 参数锁
- 状态位与故障摘要 -> 状态锁或故障锁
- 缓存索引 -> 缓存锁

必须固定：

1. 同一份共享状态不得“有时靠这把锁、有时靠那把锁”。
2. 若一组字段总是一起读写，应优先视为一个受保护状态块。
3. 锁对象应尽量与其保护的数据定义在同一个类或同一个局部结构中。

### 6.2 不允许的模糊写法

以下写法必须禁止：

- 同一队列既用 `队列锁` 又偶尔用 `状态锁`
- 用全局大锁顺手包住多个无关状态
- 通过“大家都约好别并发访问”代替锁所有权定义


## 7. 多锁规则

### 7.1 总原则

默认规则必须固定为：

`线程内部逻辑一次只持一把线程锁。`

只有当以下条件同时满足时，才允许两把及以上线程锁同持：

1. 无法通过复制、交换、快照或拆阶段消除同持
2. 顺序可以预定义
3. 持锁时间足够短
4. 不会在锁内做阻塞等待

### 7.2 预定义顺序

一旦某个类或模块存在多把线程锁，必须为其定义局部顺序表。

当前项目推荐采用以下局部顺序类别：

1. `生命周期/状态锁`
2. `配置/参数锁`
3. `队列锁`
4. `结构调整锁`
5. `缓存/索引锁`

必须固定：

- 同一组件内，获取顺序只能从前往后
- 不允许反向获取
- 如果某锁既承担状态又承担队列职责，应在注释中明确它属于哪一类

### 7.3 默认禁止的复杂写法

以下写法默认禁止：

- 拿着 `队列锁` 再去拿 `状态锁`，另一处拿着 `状态锁` 再去拿 `队列锁`
- 拿着 `缓存锁` 去等待 `condition_variable`
- 拿着 `结构调整锁` 再去做重计算
- 先拿线程内部锁，再去临时补拿另一把线程内部锁，但顺序不可见


## 8. 条件变量规则

### 8.1 一条件变量一互斥

每个 `condition_variable` 或 `condition_variable_any` 都必须有唯一配对互斥量。

必须固定：

1. 同一条件变量的所有等待方必须使用同一把互斥量。
2. 不允许不同线程对同一个条件变量传不同互斥量等待。
3. 条件变量守护的是“条件谓词”，不是“通知次数”。

### 8.2 等待必须带谓词

必须固定：

- `wait / wait_for / wait_until` 必须使用谓词重检语义
- 裸 `wait(lock)` 只能作为极受限内部桥接例外，且必须在同一函数内被显式循环或等价谓词包住

也就是说，推荐写法必须类似：

```cpp
cv.wait(lk, [&] { return stop || !queue.empty(); });
```

### 8.3 通知规则

必须固定：

1. 先修改受保护状态，再通知。
2. 通知时必须能说明它对应的是哪个谓词变化。
3. `notify_one` 与 `notify_all` 的选择应与条件语义一致，不得凭感觉乱选。

### 8.4 等待边界

等待条件变量时，必须固定：

1. 等待线程只持有该条件变量配对的那把互斥量。
2. 等待前不得额外持有其它不相关线程锁。
3. 等待前不得持有链表锁。
4. 等待前不得持有 OpenGL 锁、外设锁或消息队列以外的异类锁。


## 9. 持锁期间允许与禁止的动作

### 9.1 允许的动作

线程内部锁持有期间，只允许做与该锁保护数据直接相关的最小动作：

- 读写受保护字段
- 从队列取一个元素或压入一个元素
- 更新状态位
- 复制轻量句柄、指针、标志位、计数器
- 计算条件谓词所需的最小判断

### 9.2 禁止的动作

持线程内部锁期间，默认禁止：

- `join`
- 长时间 `wait_for` 其它对象
- 外设调用
- 文件/网络/控制台 I/O
- 调用可能阻塞的日志输出链
- 任务调度、方法评分、推理、重匹配
- 调用外部回调
- 获取不相关的第二把锁
- 持锁进入链表锁逻辑

必须固定：

`锁内只改受保护状态，锁外做阻塞、重逻辑和跨系统调用。`


## 10. 线程锁与链表锁的混合规则

### 10.1 默认禁止

必须固定：

- 线程内部锁与链表锁默认不得同持

推荐姿势是：

1. 在线程锁内完成队列/状态快照
2. 释放线程锁
3. 再进入链表锁路径

### 10.2 例外门槛

若确实无法避免线程锁与链表锁同持，必须同时满足：

1. 有明确的预定义顺序
2. 持锁区极短
3. 无等待、无阻塞、无回调
4. 在代码注释中标出原因

否则一律视为禁止写法。


## 11. RAII 与接口规则

### 11.1 禁止裸 lock/unlock

必须固定：

- 业务代码中禁止长期保留裸 `lock()` / `unlock()` 对
- 默认使用 `lock_guard / unique_lock / shared_lock`

### 11.2 锁对象必须命名

必须固定：

- 锁守卫对象必须具名
- 不允许写成匿名临时量导致立刻释放

### 11.3 try_lock 规则

`try_lock` 只能用于：

- 最佳努力路径
- 节流/跳过路径
- 避免阻塞的监控路径

不得用于：

- 业务正确性主链
- 依赖“碰运气拿到锁”的状态一致性逻辑

### 11.4 recursive_mutex 规则

必须固定：

- `recursive_mutex` 不是解决设计混乱的默认手段
- 只有在明确需要递归进入同一组件、且短期无法拆解时，才允许受限使用
- 新增 `recursive_mutex` 必须写明原因与退出计划


## 12. 销毁与停机规则

必须固定：

1. 线程锁、条件变量销毁前，相关线程必须已经停止或不再等待。
2. 不允许线程还在等待某条件变量时销毁该条件变量。
3. 不允许线程还拥有某互斥量时结束线程生命周期。
4. 线程停机链必须与 [1230_多线程嵌套启停管理规范.md](D:\鱼巢\规范\1230_多线程嵌套启停管理规范.md) 一致。

## 16. 硬约束

1. 每份共享状态都必须有唯一主保护锁。
2. 线程内部逻辑默认一次只持一把线程锁。
3. 若必须多锁同持，必须有预定义顺序，且顺序必须在代码中可见。
4. 同一条件变量的所有等待方必须使用同一把配对互斥量。
5. `wait / wait_for / wait_until` 必须使用谓词重检语义。
6. 持线程内部锁期间不得执行 `join`、外设调用、网络/文件 I/O、重匹配、重推理或其它可长时间阻塞的动作。
7. 线程内部锁与链表锁默认不得同持；确需例外时必须明示顺序与原因。
8. 业务代码中禁止长期保留裸 `lock()/unlock()` 对。
9. 锁守卫对象必须具名。
10. `try_lock` 不得承担业务正确性主链。
11. `recursive_mutex` 只能作为受限例外，不得作为默认补丁。
12. 锁与条件变量销毁前，相关线程必须已停止或不再等待。
13. 线程锁复杂组件必须补局部顺序说明或重构为单锁化。
14. 线程锁长等待、等待超时与 stop 期间卡住，必须形成可检索日志。

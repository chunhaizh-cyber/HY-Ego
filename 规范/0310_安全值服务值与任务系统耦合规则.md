# 0310 安全值服务值与任务系统耦合规则

## 1. 目标

本文用于固定 `安全值 / 服务值` 如何贯通 `需求树 / 任务树 / 调度器 / 方法执行 / 结算闭环` 的统一规则。

本文回答以下问题：

- 系统为什么只能保留 `安全值` 与 `服务值` 两个根轴
- 两棵根需求树与两棵真根任务分别承担什么职责
- 子需求与叶子需求从哪里来，谁拥有直接执行权
- 叶子需求首次满足时，安全值与服务值如何结算
- 服务值为什么不能直接加到安全值，而只能通过时序步长间接影响安全值
- 真根任务为什么不能终局完成，而只能持续重筹办与再分解

## 3. 当前版本口径

当前版本必须明确切换到以下口径：

- 不再沿用“任务成功后按固定实验值直接加减”的旧实验逻辑
- 不再把“任务做过一次”视为天然具有结算资格
- 只允许在 `叶子需求首次满足` 时，按所属根树与承接总权值进行结算

可收敛为一句话：

`结算的真值单位不是任务动作本身，而是叶子需求的首次成立。`


## 4. 双值核的语义分工

### 4.1 安全值

`安全值` 表示系统是否还能继续存在、继续运行。

必须固定：

- `安全值 = 生命轴`
- `安全值` 直接决定系统是否还能继续工作
- `安全值` 是生死边界、待机边界、临界边界的直接依据

### 4.2 服务值

`服务值` 表示系统对外服务、理解、响应、协作的能力水平。

必须固定：

- `服务值 = 服务轴`
- `服务值` 通常不直接决定生死
- `服务值` 决定系统活着时的服务有效性、服务质量与服务连续性

### 4.3 两者的主次关系

必须固定：

1. `安全值` 直接决定生死。
2. `服务值` 通常不直接决定生死。
3. `服务值` 通过时序步长间接影响 `安全值` 的恢复与下滑速度。
4. 当 `服务值` 极低或归零时，系统进入临界保底态，`安全值` 可被压到临界线。

可以直接定义为：

`安全值负责“还能不能活”，服务值负责“活着时能不能有效服务”，服务值再通过时序步长反向塑造安全值的未来演化。`

### 4.4 工程类型口径

工程落地时必须固定：

- `安全值`、`服务值`、`需求值`、`方向值`、`时序步长`、`结算增量` 等单值控制量统一使用 `I64`
- 非负语义通过区间约束、边界检查与钳制表达，不通过 `U64` 业务字段表达
- 若需要保留位签名、轮廓块、体素块等无符号 64 位证据，只允许通过 `VecIU64` 进入系统

可以压成一句：

`双值核与任务控制链上的单值业务量统一使用 I64；无符号 64 位只保留给 VecIU64 证据块。`


## 5. 需求树与真根任务的固定结构

### 5.1 两棵根需求树

系统仅允许两个根需求：

- `安全根需求`
- `服务根需求`

不允许新增第三类根需求。

所有后续需求都必须且只能挂在以下两棵树之一：

- 与生存、健康、故障隔离、降级恢复相关的需求，归入 `安全树`
- 与观察、理解、确认、交互、完成服务相关的需求，归入 `服务树`

### 5.2 两个真根任务

系统仅允许两个真根任务：

- `安全根任务`
- `服务根任务`

真根任务不是一次性完成任务，而是常驻分解器。

真根任务必须持续执行以下职责：

1. 观察当前状态
2. 判断本侧仍缺什么
3. 分解出下一层阶段任务与叶子任务

### 5.3 根需求与根任务的关系

必须固定：

- 根需求定义总方向，不直接执行
- 真根任务承载根需求，但不直接充当叶子执行节点
- 阶段任务与叶子任务负责局部推进
- 真根任务在每轮局部阶段结束后回到 `待重筹办`


## 6. 子需求与叶子需求的来源规则

### 6.1 子需求的来源

子需求不是任意生成，也不是从根需求凭空冒出。

必须固定：

`子需求来自父任务的分解。`

因此统一链条为：

`根需求给总方向 -> 根任务按当前状态分解 -> 形成阶段子需求 -> 阶段子需求继续分解 -> 直到叶子需求`

### 6.2 直接执行权

系统必须固定：

- 只有 `叶子需求` 拥有直接执行权
- 非叶子需求只承担结构意义、汇总意义与方向意义
- 叶子需求才允许绑定叶子任务并进入执行链

### 6.3 状态满足与执行权不是一回事

必须区分：

- `直接执行权` 只属于叶子需求
- `状态满足` 不只属于叶子需求

也就是说，非叶子需求虽然不直接执行，但在子任务完成后仍可能被动转为已满足。


## 7. 两棵真根任务的标准分解结构

### 7.1 安全根任务

安全根任务必须同时拥有两层方法池。

第一层为 `兜底本能方法`，作用是先把系统带回可判断、可分解、可恢复的状态。

典型包括：

- 自检
- 读取核心状态
- 关键外设状态确认
- 故障隔离
- 停止高风险试探
- 进入保守态
- 恢复最小观察链

这层方法必须尽量满足：

- 低依赖
- 低风险
- 可直接执行
- 幂等
- 输出标准化状态结果

第二层为 `可学习成长方法`，作用是在兜底保住系统之后，提高恢复效率和处理质量。

典型包括：

- 某类故障的专门恢复方法
- 某类外设的高效重连方法
- 某类风险模式的优化规避方法
- 某类安全修复策略的学习与强化

### 7.2 服务根任务

服务根任务也必须同时拥有两层方法池。

第一层为 `兜底本能方法`，作用不是完成复杂服务，而是保证服务链不断。

典型包括：

- 读取最新输入
- 确认收到
- 请求补充信息
- 观察场景
- 建立最低限度上下文

第二层为 `可学习成长方法`，作用是在服务链不断的前提下，提高服务质量与效率。

典型包括：

- 针对具体任务的高质量处理方法
- 针对具体对象的经验化服务方法
- 针对具体场景的快速理解与生成方法


## 8. 权重的三层作用

每个需求都可以拥有自己的权重，但权重必须按统一规则使用。

### 8.1 调度作用

权重越高，调度器越优先关注该需求对应的任务链。

### 8.2 分解传播作用

父需求的权值必须向下分摊到子需求。

最终每个叶子需求都应承接一份 `总权值`。

### 8.3 结算作用

叶子需求首次满足时，必须按其承接到的 `总权值` 进行结算。

必须固定：

- 根权重若已向下传播到叶子，结算时不得再次重复计入
- 结算口径不是 `根权重 + 叶子权重`
- 结算口径只能是 `叶子承接总权值`


## 9. 叶子需求结算规则

### 9.1 触发条件

只有同时满足以下条件，才允许触发 `安全值 / 服务值` 结算：

1. 对应需求存在
2. 对应需求是叶子需求
3. 该叶子需求所属根树明确
4. 该叶子需求从 `未满足` 转为 `已满足`
5. 这是一次真正完成，而不是仅进入下一阶段或等待重筹办

### 9.2 结算方向

必须按叶子需求所属根树进行结算：

- `安全树叶子需求首次满足 -> 安全值 += 叶子承接总权值对应增量`
- `服务树叶子需求首次满足 -> 服务值 += 叶子承接总权值对应增量`

### 9.3 明确禁止项

以下情况一律不允许：

- 根需求重复参与结算
- 非叶子需求直接参与结算
- 服务树叶子满足后直接增加 `安全值`
- 安全树叶子满足后直接增加 `服务值`
- 只因为任务“执行成功过一次”就直接给双值结算


## 10. 服务值对安全值的间接作用

### 10.1 为什么不能直接跨树结算

必须固定：

- `安全值` 代表能否继续存在
- `服务值` 代表服务能力与服务状态

如果服务叶子完成后直接增加 `安全值`，会把“会服务”和“能活着”混成同一语义对象，导致控制含义塌缩。

### 10.2 正确影响链

服务值对安全值的影响链必须固定为：

1. `服务树叶子需求首次满足`
2. `服务值增加`
3. `时序步长或时序调制参数变化`
4. `安全值未来恢复/下滑趋势发生变化`

可收敛为一句话：

`服务值对安全值是强影响，但只能是间接影响。`


## 11. 安全值、服务值与时序步长

系统时间流逝时，不是 `服务值` 直接按时间掉落，而是 `安全值` 按时序规则变化。

`服务值` 的职责是调节该时序变化的步长或趋势。

必须固定：

- `服务值高 -> 安全值恢复更容易，下滑更慢`
- `服务值低 -> 安全值恢复更慢，下滑更快`
- `服务值极低或归零 -> 系统进入待机或临界保底态，安全值可能被压到临界值`

因此：

`服务值是安全值时序演化的变速器。`


## 12. 需求状态与任务方向的分工

### 12.1 需求状态

需求状态回答的是“还差多少”。

其判断基础主要包括：

- 当前状态
- 目标状态
- 差值
- 是否已满足

例如：

- 安全目标大于当前安全值，说明仍需提升安全
- 服务目标大于当前服务值，说明仍需提升服务

### 12.2 任务方向

任务方向回答的是“本次方法执行后，关键事实是否朝正确方向变化”。

方向结果统一收束为：

- `变好`
- `变坏`
- `不变`

安全根任务的方向投影通常应重点观察：

- `安全值`
- `待机状态`
- `风险级别`
- `保守态请求`
- `高风险试探是否停止`
- `时序步长相关量`

服务根任务的方向投影通常应重点观察：

- `服务值`
- `服务动作输出是否有效`
- `是否确认收到`
- `是否请求补充信息`
- `是否成功读取输入`

### 12.3 对齐判定

最后必须把 `本次方向` 与 `需求要求方向` 进行比对，得到：

- `对齐`
- `反向`
- `中性`

该结果应继续影响后续调度、回退、重筹办与再分解。


## 13. 真根任务的常驻闭环

真根任务的职责不是被消灭，而是持续监督本侧目标。

因此必须固定以下闭环：

1. 当前阶段子任务完成
2. 当前阶段子需求满足或局部收口
3. 真根任务回到 `待重筹办`
4. 根据最新 `安全值 / 服务值 / 方向状态` 再次分解下一步

必须固定：

- 真根任务是常驻的
- 阶段任务和叶子任务才是可完成的
- 真根任务不得进入终局完成态


## 14. 全系统统一闭环

把整套机制串起来，统一链条必须表达为：

1. `安全值 / 服务值` 共同定义当前总态势
2. `安全根需求 / 服务根需求` 定义总方向
3. `安全根任务 / 服务根任务` 读取当前事实并分解
4. 形成阶段子需求与叶子需求
5. 叶子任务调用兜底方法或成长方法
6. 方法执行后写回事实，刷新需求状态与方向状态
7. 叶子需求首次满足时，按所属根树结算到 `安全值` 或 `服务值`
8. `服务值` 再通过时序步长间接影响 `安全值`
9. 真根任务继续下一轮重筹办与再分解

可以收敛为一句话：

`安全值是生命轴，服务值是服务轴；根需求定义总方向，真根任务持续分解，叶子需求首次满足时按承接总权值结算到所属根树，服务值再通过时序步长间接影响安全值。`


## 15. 可直接落代码的硬约束

后续代码与规范校验必须至少满足以下十条：

1. 系统仅允许两个根需求：`安全`、`服务`。
2. 系统仅允许两个真根任务：`安全根任务`、`服务根任务`。
3. 子需求只能来自父任务分解。
4. 只有叶子需求拥有直接执行权。
5. 只有叶子需求首次满足时允许触发 `安全值 / 服务值` 结算。
6. 结算必须按叶子需求所属根树进行。
7. 结算权值必须使用 `叶子承接总权值`，根权重不得重复计算。
8. `服务值` 不得直接结算到 `安全值`。
9. `服务值` 只能通过时序步长间接影响 `安全值`。
10. 真根任务不得终结，只能持续重筹办与再分解。

# 000 项目规则总纲

编号：000
日期：2026-07-05
状态：正式起步规则

## 1. 定位

本文件是 `海中鱼巣` 的第一版项目规则总纲，用于承接旧项目中已经证明有效的工程治理口径，并按新项目“全量重构、重新建核”的目标收束。

本文件不要求新项目复制旧项目结构；只迁移适合新内核的硬规则。

## 2. 核心结构规则

### 2.1 结构闭环

系统内部逻辑必须通过项目结构表达和流转。

第一版允许的核心结构为：

```text
节点
节点句柄
主信息
关系
索引
特征
特征值
需求
任务
方法
状态
动态
因果引用
语素入口
```

禁止用以下材料承载机器逻辑：

```text
日志
控制台输出
注释
说明文档
临时字符串
临时 JSON
显示标题
调试文本
```

### 2.1.1 状态声明四态制

项目文档和实施记录中，能力状态必须使用四态口径，避免把小能力通过扩大成系统完成。

| 状态 | 含义 |
| --- | --- |
| 规范已确认 | 规则可作为实现依据 |
| 代码已实现 | 函数已落代码，但未必完成验证 |
| 验证已通过 | 有构建、样例或测试证明入口拒绝、结构变化、输出变化成立 |
| 系统级完成禁止宣称 | 小能力已实现或已验证，但不得扩大为完整系统、旧能力迁移或高阶能力完成 |

完成声明必须同时说明：

```text
规范状态
代码状态
验证状态
禁止扩大宣称范围
```

### 2.2 节点身份

节点长期身份必须使用稳定身份：

```text
节点编号
节点句柄
主键
版本号
```

指针只能作为运行期缓存或临时访问材料，不得作为长期事实身份。

长期编号必须使用无符号 64 位整数承载。以下编号类型统一采用 `uint64` 语义：

```text
仓库编号
节点编号
主信息编号
关系编号
索引编号
创建序号
```

编号规则：

```text
0 保留为无效编号。
有效编号从 1 开始。
同一仓库内编号只递增，不复用。
删除记录只递增版本或改变状态，不回收编号给新对象。
稳定身份必须由 仓库编号 + 本地编号 + 版本号 共同判断。
```

长期运行时不得通过改用有符号整数或复用旧编号解决容量问题。若单仓库编号空间、持久化体量或索引规模接近上限，必须通过以下方式扩展：

```text
新增仓库编号
分片仓库
空间编号
纪元编号
快照归档
冷热数据分层
```

扩展策略必须先进入规范和详细设计，不得在代码中临时改变编号含义。

### 2.2.1 版本号语义

版本号用于生命周期失效和稳定句柄判定，不用于每次业务值变化。

版本规则：

```text
创建版本号从 1 开始。
删除、生命周期失效或身份重建时递增版本号。
普通特征值变化、状态变化、任务推进和方法执行不递增节点句柄版本号。
业务变化必须由特征值、状态、动态、因果引用、任务或方法结构承载。
关系写入和读取必须校验源节点与目标节点当前句柄版本。
```

若后续需要记录业务数据版本，应在对应系统中建立特征值版本、状态版本、动态序号或事件版本，不得复用节点 / 主信息 / 关系句柄版本号。

### 2.3 主信息边界

主信息只保存身份、拓扑锚点、值容器或明确声明的非权威缓存。

凡是表达对象属性、观测事实、统计事实、治理状态、判断结论、任务结果、方法结果或因果解释的内容，应优先进入：

```text
特征
特征值
状态
动态
二次判断结果
因果引用
需求
任务
方法
```

主信息字段迁移红线：

```text
未确认主信息系统规范，不迁移旧字段。
未确认字段功能说明，不新增主信息字段。
未确认字段是否权威，不允许进入值容器。
未确认字段能否由关系、索引、特征、状态、动态、因果替代，不允许编码。
```

### 2.4 关系边界

新项目不得把所有关系混成普通父子关系。第一版至少区分：

```text
普通父子
归属关系
引用关系
索引关系
模板关系
运行期临时关系
因果来源关系
```

关系变更必须通过仓库或领域服务入口完成。

节点移动统一解释为普通父子关系重挂。通用移动算法只能由关系仓库或后续确认的公共结构移动工具承载，各领域服务不得重复实现移动算法；领域服务只允许先排除本领域禁止移动的节点类型、生命周期和业务前置条件，再调用通用重挂入口。

普通父子移动至少必须在入口拒绝：

```text
无效节点
无效新父节点
移动到自身下面
移动到自身子孙下面
移动到当前父节点导致无结构变化
没有当前父关系而被误当作移动成功
```

### 2.5 入口前置拒绝规则

本项目编码禁止使用事后校验逻辑兜底结构正确性。

所有会写入或改变结构的函数，必须在函数入口排除错误可能：

```text
无效句柄
已删除句柄
错误节点类型
错误关系类型
越权写入方
缺少前置节点
缺少前置关系
不满足生命周期的对象
```

禁止设计以下运行路径：

```text
先写入无效结构，再通过校验函数发现
先创建悬空关系，再通过扫描器修复
先接受错误节点类型，再由后续流程兜底
先把错误状态写入仓库，再由日志或显示提示
```

构建、测试和人工验证可以证明入口拒绝是否生效，但不得成为运行期机器结构正确性的兜底机制。

### 2.6 容错检查诊断规则

容错检查只能作为预编译宏控制的诊断层。

默认构建不得启用容错检查弹窗；需要调试入口拒绝路径时，必须通过项目统一预编译宏启用：

```text
HY_EGO_ENABLE_FAULT_TOLERANCE_CHECK
```

宏关闭时：

```text
不得改变运行行为
不得写诊断日志
不得弹窗
不得引入机器事实变化
```

宏开启后，容错检查触发时必须同时执行：

```text
日志记录
弹窗提醒
```

日志只用于排错，弹窗只用于人读提醒；二者只记录和提示，不改变返回值、结构结果或机器事实。

弹窗意味着逻辑错误已经暴露。该逻辑错误未排除前，不得继续推进依赖该错误路径的实现、验收或完成声明。

日志和弹窗不得作为：

```text
机器事实
业务状态
任务完成依据
需求满足依据
方法成功依据
自动修复入口
```

容错检查不得替代函数入口前置拒绝。正确路径仍然是先拒绝错误输入，再由宏控制是否记录和提醒。

函数入口前置拒绝已经排除错误输入后，内部写入结果、关系目标、参数值或结构状态仍不符合预期时，必须视为逻辑错误追根因。

典型逻辑错误包括：

```text
节点创建成功但后续必需关系写入失败。
预读目标和写入后关系目标不一致。
唯一关系写入后读取出多个目标。
内部参数值越过入口约束后仍出现非法值。
仓库返回结果和调用方已确认前置条件不一致。
```

逻辑错误不得被当成普通业务失败静默丢弃，也不得只返回失败后继续推进依赖该路径的实现或验收。宏开启时必须写逻辑错误日志并弹窗提醒；宏关闭时不得改变机器事实，但实现和验收仍必须追查并排除根因后再声明通过。

### 2.7 信息节点口径

`基础信息` 是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型。

第一轮 `基础信息` 口径包括：

```text
基础信息通用节点
存在
场景
特征
抽象状态
抽象动态
二次特征
因果引用
```

`基础信息通用节点` 只作为当前尚未拆入具体基础信息类型时的通用承载入口，不得替代已经明确的存在、场景、特征、抽象状态、抽象动态、二次特征或因果引用节点类型。

实例状态和实例动态不等同于抽象状态和抽象动态：

```text
实例状态：某个存在在某个场景内某个发生时间戳上的临时状态，以场景内临时零散节点存储。
实例动态：某个存在在某个场景内某个发生时间戳上发生的临时变化记录，以场景内临时零散节点存储。
抽象状态：由实例状态抽象、确认或归纳形成的长期基础信息节点。
抽象动态：由实例动态抽象、确认、归档或因果证据提升形成的长期基础信息节点。
```

实例状态和实例动态必须有存在主体、场景归属和发生时间戳，默认不长期独立存储，不得直接作为语素入口对应信息节点。只有抽象状态和抽象动态才可作为长期基础信息类节点被语素入口绑定和跨场景复用；抽象状态和抽象动态本体不携带发生时间戳。

`高级信息` 不是独立节点类型，不得为它新增节点枚举、字段或仓库结构。

现阶段 `高级信息` 指以下既有高级业务节点角色：

```text
需求
任务
方法
```

当语素、名称、索引或其他系统需要绑定“信息节点”时，必须在函数入口接收并验证已有节点。现阶段允许的信息节点范围为：

```text
基础信息类节点：基础信息通用节点、存在、场景、特征、抽象状态、抽象动态、二次特征、因果引用
需求
任务
方法
```

不允许为了等待后续补齐而先创建悬空入口或临时信息节点。

## 3. 领域服务规则

所有业务写入必须经过领域服务。

第一版领域服务按基础信息原子服务、基础信息聚合服务、高级信息服务和入口服务分层。

基础信息原子服务包括：

```text
特征值服务
特征服务
存在服务
场景服务
状态服务
动态服务
二次特征服务
因果服务
```

基础信息聚合服务为：

```text
世界服务
```

高级信息服务包括：

```text
需求服务
任务服务
方法服务
```

入口服务包括：

```text
语素服务
```

世界服务只做基础信息聚合编排，不得绕过基础信息原子服务直接写基础信息结构。

特征值服务只服务于特征系统，不得作为需求、任务、方法的通用值容器。

只有特征服务可直接包含、接收或调用特征值服务。特征值节点的创建、读取和写入必须经特征服务对外提供；其他领域服务不得直接访问特征值服务。

需求服务、任务服务和方法服务等高级服务只允许直接依赖以下基础信息服务：

```text
存在服务
场景服务
特征服务
二次特征服务
状态服务
动态服务
因果服务
```

高级服务需要特征状态材料或特征值节点时，必须通过特征服务获得，不得绕过特征服务直接访问特征值服务或主信息值容器。

需求服务、任务服务和方法服务需要存在、场景、特征、二次特征、状态、动态或因果能力时，必须通过上述基础信息服务获得，不得直接写基础信息仓库结构。

禁止外部模块直接写：

```text
节点仓库
主信息仓库
关系仓库
索引仓库
```

只读查询可以通过领域服务返回只读视图，不暴露可写内部对象。

### 3.1 多线程防锁和仓库锁边界

多线程防锁机制必须遵守 `规范/多线程防锁机制规范.md`。

当前只允许先建立核心仓库读写锁边界和锁友好的服务入口，不声明真实线程调度、工作队列、`std::async`、任务线程池、外设等待或并发压力测试已经完成。

锁只保护运行期内存结构一致性，不承载机器事实，不生成动作来源，不替代事务边界、任务授权、方法执行或因果证据。

核心仓库锁边界：

```text
节点仓库、主信息仓库、关系仓库、索引仓库内部可以使用读写锁保护共享结构。
读入口返回记录副本、句柄组或快照材料，不暴露可写内部引用。
写入口独占修改内部结构，并继续遵守入口前置拒绝规则。
复杂内部路径可以拆出 `已加锁` 辅助函数，但该函数只允许同仓库内部持锁路径调用。
```

防死锁顺序：

```text
普通公开入口原则上只持有一个仓库锁。
后续确需多仓库同时持锁时，必须先有详细设计，并按 节点仓库 -> 主信息仓库 -> 关系仓库 -> 索引仓库 的顺序获取。
不得在持锁期间调用上层领域服务、日志落盘、弹窗、持久化 I/O、外设等待、用户回调或可能反向进入仓库公开入口的函数。
不得在共享锁持有期间升级为独占锁；需要写入时必须重新走写入口和入口拒绝流程。
```

线程、锁、等待事件、日志标签和控制台命令不得成为动作来源。

### 3.2 公用函数和公共辅助入口规则

公用函数规则必须遵守 `规范/公用函数规则规范.md`。

旧项目中的全局共享函数、双向链表内部辅助、深拷贝快照、序列化 / 恢复函数只能作为规则证据，不得直接复制为新项目公共工具实现。

公用函数候选必须先判断归属：

```text
纯通用计算 / 格式化 -> 后续公共工具候选。
仓库内部结构辅助 -> 对应仓库内部辅助。
领域结构读写 -> 对应领域服务或仓库正式入口。
跨服务流程编排 -> 服务矩阵和流程图迁移包。
绕过领域服务、仓库锁、入口拒绝或确认门禁 -> 拒绝。
```

只要函数候选触碰节点、主信息、关系、索引、特征、特征值、状态、动态、需求、任务、方法、因果引用或非权威缓存等结构，就不得作为全局纯通用函数处理。

新增公用函数或公共辅助入口前，必须先查当前代码和函数说明是否已有同等功能入口，并说明用途、归属方、允许调用方、禁止调用方、读写类型、触碰结构、入口拒绝、拒绝后结构不变化验收和生命周期。

`已加锁` 风格辅助函数只允许同一仓库或同一对象内部持锁路径调用，不得暴露给领域服务、显示层、工具层或外部调用方。

只读快照和候选读取只能返回记录副本、句柄组、只读材料或非权威候选，不得返回可写内部引用、长期裸指针或可被上层误认为权威事实的临时缓存。

## 4. 特征和值规则

一阶机器事实优先使用：

```text
I64 状态值
I64 Q10000 评分
计数
版本号
时间戳
节点句柄
主信息句柄
VecI64
VecIU64
特征值句柄
```

普通 bool 不得作为一阶机器事实扩散。需要二态判断时，应由状态值、评分、计数、版本、来源证据或复合裁决生成。

普通 `bool` 可以作为局部函数执行成功 / 失败返回值、入口拒绝结果或临时控制流条件，但不得持久化为业务事实、需求满足、任务完成、方法成功或世界事实。

## 5. 动作和因果规则

线程不是动作来源。

外部输入不是动作来源。

动作来源必须来自明确方法、领域服务写入入口或后续正式定义的动作提交入口。

因果信息是累积信息，不由单条或少量动态直接形成可靠结论。

因果结构必须区分：

```text
轻量因果引用
可靠因果结论
```

第一轮轻量因果引用只表达“已有来源动态作为准入证据，可以创建因果引用节点”。`记录因果引用(来源动态)` 只检查来源动态有效和类型正确，不持久化因果引用到来源动态的单条关系。

动态信息本身作为后续因果评估的证据材料。因果可靠性必须由后续因果聚合、动态样本积累、评分、版本和复判机制形成。

不得把单个轻量因果引用节点解释为稳定因果结论。

可靠因果结论或完整因果评估后续必须围绕动态证据集合追溯：

```text
来源动作或来源方法
输入材料
被改变目标
改变前值
改变后值
验证方式
证据样本集合
评分或版本
```

方法执行权限来自任务授权、任务承接和任务选择方法。方法执行前不再单独创建授权状态节点。

第一轮方法执行入口必须至少排除：

```text
无效任务节点
无效方法节点
缺少任务到方法关系
无效特征值节点
错误特征值节点类型
```

任务选择方法后，方法服务可以通过特征服务请求修改目标特征状态材料。后续若需要更强权限治理，应围绕任务授权关系、任务状态和任务方法关系设计，不恢复方法执行前临时授权壳。

需求目标是目标状态，不是 I64。I64 只可作为第一轮状态服务或特征服务内部的状态值材料，不得作为需求目标的对外语义或主信息字段迁移依据。

## 6. 显示和日志规则

显示层、日志层和控制台输出只供人读。

显示层可以读取结构：

```text
节点
主信息
关系
特征
特征值
需求
任务
方法
动态
因果引用
```

显示层不得写业务事实，不得作为任务完成、需求满足或方法成功的机器判断来源。

日志系统必须遵守 `规范/日志系统规范.md`。

所有日志都放在 `D:\海中鱼巣\日志` 目录下。

逻辑错误报错必须写入独立逻辑错误日志文件。逻辑错误日志只做人读诊断，不得与运行日志或普通调试日志混写，不得参与机器事实、业务状态、任务结算、需求满足、方法成功或自动修复。

运行日志必须写入独立运行日志文件，主要记录系统运行情况。运行日志只做人读运行态观察，不得作为机器事实、业务状态、任务完成、需求满足或方法成功依据。

调试信息可以按排查需要创建调试日志，但调试日志代码必须按切片使用预编译宏控制启用和关闭。调试日志宏默认关闭；宏关闭时不得写调试日志、不得改变运行行为、不得影响验收结果。

调试完成后必须关闭对应切片调试日志代码，移除默认启用宏或构建参数，避免无用日志长期泛滥。

## 7. 持久化规则

第一版运行期核心以内存仓库为准。

数据库和文件持久化只作为：

```text
快照
事件日志
恢复材料
审计材料
只读查询材料
```

数据库不得直接裁决机器事实，不得绕过领域服务写运行期结构。

## 8. 命名规则

函数名、变量名、类内业务成员名必须使用中文。

允许英文的范围：

```text
C++ 关键字
标准库标识
第三方 API
编译系统字段
外部协议固定字段
```

禁止为了方便新增英文业务变量名、英文业务函数名或英文业务枚举名。

### 8.1 代码文件建立与模块命名

代码文件必须遵守 `规范/代码文件建立归属与模块命名规范.md`。

现有目录结构和既有代码文件默认原地保留，不因本规则批量改名。以后新建代码文件统一采用：

```text
数据结构文件：功能.分类.h / 功能.分类.cpp
其它代码文件：功能.分类.ixx
```

数据结构只指稳定值类型、记录 / 容器布局和结构自身不变式；服务、协议、初始化、线程、桥、适配、显示、自检和编排不因内部使用结构体而归为数据结构。

新 `.ixx` 必须是真实 C++ 模块：

```text
包含匹配的 export module。
在工程中按 ClCompile 登记。
由调用方通过 import 使用。
不得按 ClInclude 登记或由 #include 使用。
```

生产模块和自检模块必须分离。`入口.cpp` 只允许保留参数解析、运行模式、最小组合装配、顶层调用、停止和退出码映射；不得继续新增领域实现、完整自检正文、并发压力正文或逐项验收输出正文。在统一自检运行器形成前，只允许入口增加最小上下文组装、一次模块调用和总结果汇总。

旧计划若允许把新功能正文或完整自检继续写入入口，执行前必须退回设计窗口修订允许文件和承载方式，不得由执行窗口临时扩权。

## 9. 完成声明规则

任何“完成 / 已实现 / 已支持 / 已接入 / 已闭环”声明必须给出：

```text
输入入口
核心处理
结构承载
输出变化
调用路径
验证方式
```

缺少结构承载或验证方式时，只能声明为：

```text
文档草案
设计方向
代码壳
部分实现
待验证
```

验证方式必须证明入口前置拒绝已经生效。不得用“事后校验发现了错误”替代“函数入口已经排除了错误”。

## 9.1 计划生成与入队联动门禁

流程图生成后可以直接生成对应详细设计，详细设计生成后可以直接生成计划。计划不再等待用户确认；完整计划直接写入 `计划/`，并同步登记计划索引和 Codex 任务队列为待执行。

计划依赖不再构成计划生成门禁，统一采用：

```text
先生成依赖门控计划，执行前再复核实际接口。
```

前置计划未完成时，只要流程图、详细设计、S0 当前事实和默认裁决足以限定实施范围，允许先生成完整计划并登记为“依赖门控待执行”。该计划必须明确：

```text
已验证的当前事实。
尚未实现的假定接口契约。
前置计划和队列编号。
依赖完成后必须出现的正式产物。
执行前读取与比对实际接口的命令或步骤。
实际接口与假定契约不一致时的退回修订路径。
```

依赖门控计划生成和入队不等于依赖已满足，也不等于允许立即修改代码。执行窗口必须先确认全部依赖已经完成并形成正式产物，再逐项比对实际接口；依赖未满足时跳过该项，不记为真实阻塞；接口漂移时不得开始代码修改，必须退回设计计划窗口修订。前置计划真实阻塞时，依赖计划继续保持门控，除非另行修订并明确替代依据。

计划生成并登记待执行时必须同步核对该计划对应的流程图和详细设计：

```text
计划及生成断点必须列明流程图路径、详细设计路径和计划路径。
计划索引和项目记忆必须同步写明三者范围一致。
计划范围不得超出对应流程图和详细设计范围。
代码实施计划必须明确允许文件、禁止文件、入口拒绝、失败收口和验证方式。
依赖门控计划必须把当前事实与假定接口分栏说明，并登记执行前接口复核门禁。
```

以下情况不得登记待执行：

```text
对应流程图缺失、撤回或仍有未给出默认裁决的分支。
对应详细设计缺失、撤回或仍有未给出默认裁决的问题。
计划覆盖的允许文件、禁止文件、验收方式或完成声明边界没有被详细设计支撑。
计划试图绕过流程图、详细设计或项目记忆状态直接生成代码实施许可。
依赖门控计划把预期产物写成当前代码事实，或没有执行前接口复核与漂移退回规则。
```

纯规则、只读扫描、信息数据、断点清单或项目记忆同步类计划如果没有流程图 / 详细设计来源，必须在计划和生成记录中写明不适用原因、替代依据和不涉及代码实施的完成边界。

## 9.2 流程图中途返回二分规则

以后制作、修订或审查流程图时，所有中途非成功返回都必须归入以下两类之一：

| 类型 | 判定口径 | 后续要求 |
| --- | --- | --- |
| 追根因解决 | 前置条件已经满足，并且流程已经进入创建、绑定、写关系、写状态、记录动态、结算、读回或结构承载；此后返回值、写入结果、关系目标、参数值或结构状态不符合内部预期。 | 不得画成普通业务失败；必须停止依赖路径，追查根因。若暂时不能修复，必须标明事务边界、半结构隔离或结构化错误结果缺口。 |
| 逻辑内返回 | 分支属于领域协议允许的结果，且不产生可被读取为有效业务事实的半结构；或只返回请求材料、候选为空、拒绝材料、人读材料。 | 必须写明结构不变化，且返回材料不裁决机器事实。 |

禁止在流程图中单独使用以下含混口径替代二分分类：

```text
失败返回
失败收口
返回无效句柄
不能声明成功
异常但继续
```

确需使用上述文字时，必须同时标明属于“追根因解决”还是“逻辑内返回”。

流程图、详细设计和代码若描述同一入口，三者必须保持一致：流程图标为追根因解决的路径，详细设计和代码不得只用普通失败返回吞掉；流程图标为逻辑内返回的路径，详细设计和代码必须证明结构不变化或只输出材料。

本规则只确立后续流程图制作硬口径，不证明既有流程图、详细设计或 C++ 已全部修订完成。

## 9.3 仓库与服务分层事务边界

现行分层以 `规范/仓库与服务分层事务边界规范.md` 为准：

```text
结构事务协调层负责许可签发 / 释放和跨仓库事务上下文。
仓库负责验证令牌、本仓库锁和通用结构不变量。
服务专用数据操作层隐藏仓库与原始令牌，负责精确写入、撤销、确认和最后发布。
领域服务负责业务准入、业务不变量、幂等和成功 / 失败语义。
应用 / 领域组合器定义跨服务一次操作边界，外部调用方不携带原始令牌。
```

领域服务公开业务入口不得暴露、接收、返回、传递或比较原始 `结构事务令牌`，不得直接取得 / 释放许可或操作仓库锁。当前无令牌写、带令牌写和候选写兼容路径必须先登记调用点和替代顺序，再逐批迁移；生产调用点归零前不得直接删除。

第一组垂直样例固定为存在 / 场景，后续按状态 / 动态、特征体系、语义基础、需求 / 任务 / 方法和运行期组合器推进。关系二级索引、批量读取、计数和分片在分层纠偏完成后重新建立基线并继续证据门控。

## 10. 工具与旧规则迁移治理

旧 `鱼巢` 的工具、代理规则、验收规则和执行规则只能按 `规范/工具与规则迁移治理规范.md` 分类迁移。

旧项目能力迁移路线必须遵守 `规范/迁移路线权力分层规范.md`。旧函数事实、旧逻辑迁移包、旧小切片计划和派生详细设计只能作为证据材料、边界材料或候选材料；默认迁移路线为服务操作函数矩阵、应用逻辑流程图迁移包和批次级代码实施包，不再把旧函数体、旧函数调用链或单个小切片作为默认代码实施单位。

迁移规则：

```text
已证明适合海中鱼巣且不改变机器结构的治理工具，可以改写后迁移。
旧项目路径、旧规范目录、旧工程图谱、旧 SQL 投影和旧控制面板假设不得原样进入新项目。
函数说明生成、dump / debug-stack 等工具迁移前必须另建工具专项，明确允许文件、禁止文件和验证方式。
旧 SQL / 旧控制面板 / D455 / 体素 / 外设工具迁移当前暂缓；不得借工具迁移扩大独立重建的只读控制面板第一版和 SQL 审计投影边界。
```

当前已迁移工具为：

```text
tools/check_specs.py
```

该工具只做规范治理轻量检查，不证明代码实现、构建通过、旧能力迁移或系统闭环完成。

## 11. 协作窗口与继续指令规则

`继续` 必须按窗口类型解释，不得作为全局执行指令。

设计计划窗口中的 `继续` 只表示继续当前规则、流程图、设计、计划或信息数据的治理工作；可以生成或修订规范草案、流程图、详细设计、计划、信息数据和项目记忆状态，并把范围完整的计划登记为待执行。用户沿用旧称“计划设计和确认窗口”时，也按设计计划窗口解释。设计计划窗口不得消费待执行队列、不得直接修改 C++、不得运行构建、不得把待执行计划宣称为已执行。

执行窗口中的 `继续` 才表示继续消费 `项目记忆/Codex任务队列.md` 中已经登记的待执行队列项；执行窗口不得从没有计划、缺少允许 / 禁止文件或上游范围不一致的材料中自行挑选内容落地。

只读监控 / 复核窗口中的 `继续` 只表示继续读取事实并汇报或生成只读报告；不得修改文件、入队、提交、推送或运行构建。

窗口类型未声明时，必须先按最近用户明确声明、线程标题和当前任务对象判断；仍不明确时，只能先确认窗口类型或进行不改文件的状态判断，不得把裸 `继续` 自动解释为执行队列消费。

### 11.1 设计执行双窗口交互

设计计划窗口与执行窗口必须按 `规范/设计执行双窗口交互规范.md` 交接。仓库中的计划、队列、断点、项目记忆和 Git 提交承载权威状态；Codex 任务消息只负责通知接收窗口重新读取权威状态，不得承载实施许可或完成事实。

执行窗口发现接口漂移、允许文件不足、禁止文件冲突或上游范围不一致时，必须先停止代码修改并形成正式退回记录，提交推送后再通知设计窗口。设计窗口完成三件套和计划修订、提交推送并登记原队列项“重新待执行”后，才能通知执行窗口接管。两个窗口不得同时修改同一计划及其代码范围。

### 11.2 Git 自动收尾

每轮有文件改动的任务在完成授权范围并通过要求的验证后，必须只暂存本轮文件，自动本地提交并自动非强制推送。正式迁移任务未另行指定远端目标时默认快进发布到 `origin/main`；目标不明、远端分叉、detached HEAD、无法隔离无关改动或验证失败时停止推送并报告。自动推送不得使用 force，不得隐含授权自动合并、变基、删除分支、创建 PR 或发布制品。

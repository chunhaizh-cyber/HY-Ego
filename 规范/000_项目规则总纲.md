# 000 项目规则总纲

编号：000
日期：2026-07-05
状态：正式起步规则

## 1. 定位

本文件是 `海中鱼巣` 的第一版项目规则总纲，用于承接旧项目中已经证明有效的工程治理口径，并按新项目“全量重构、重新建核”的目标收束。

本文件不要求新项目复制旧项目结构；只迁移适合新内核的硬规则。

## 2. 核心结构规则

### 2.1 结构闭环

系统内部逻辑必须通过项目结构表达和流转。

第一版允许的核心结构为：

```text
节点
节点句柄
主信息
关系
索引
特征
特征值
需求
任务
方法
状态
动态
因果引用
语素入口
```

禁止用以下材料承载机器逻辑：

```text
日志
控制台输出
注释
说明文档
临时字符串
临时 JSON
显示标题
调试文本
```

### 2.1.1 状态声明四态制

项目文档和实施记录中，能力状态必须使用四态口径，避免把小能力通过扩大成系统完成。

| 状态 | 含义 |
| --- | --- |
| 规范已确认 | 规则可作为实现依据 |
| 代码已实现 | 函数已落代码，但未必完成验证 |
| 验证已通过 | 有构建、样例或测试证明入口拒绝、结构变化、输出变化成立 |
| 系统级完成禁止宣称 | 小能力已实现或已验证，但不得扩大为完整系统、旧能力迁移或高阶能力完成 |

完成声明必须同时说明：

```text
规范状态
代码状态
验证状态
禁止扩大宣称范围
```

### 2.2 节点身份

节点长期身份必须使用稳定身份：

```text
节点编号
节点句柄
主键
版本号
```

指针只能作为运行期缓存或临时访问材料，不得作为长期事实身份。

长期编号必须使用无符号 64 位整数承载。以下编号类型统一采用 `uint64` 语义：

```text
仓库编号
节点编号
主信息编号
关系编号
索引编号
创建序号
```

编号规则：

```text
0 保留为无效编号。
有效编号从 1 开始。
同一仓库内编号只递增，不复用。
删除记录只递增版本或改变状态，不回收编号给新对象。
稳定身份必须由 仓库编号 + 本地编号 + 版本号 共同判断。
```

长期运行时不得通过改用有符号整数或复用旧编号解决容量问题。若单仓库编号空间、持久化体量或索引规模接近上限，必须通过以下方式扩展：

```text
新增仓库编号
分片仓库
空间编号
纪元编号
快照归档
冷热数据分层
```

扩展策略必须先进入规范和详细设计，不得在代码中临时改变编号含义。

### 2.2.1 版本号语义

版本号用于生命周期失效和稳定句柄判定，不用于每次业务值变化。

版本规则：

```text
创建版本号从 1 开始。
删除、生命周期失效或身份重建时递增版本号。
普通特征值变化、状态变化、任务推进和方法执行不递增节点句柄版本号。
业务变化必须由特征值、状态、动态、因果引用、任务或方法结构承载。
关系写入和读取必须校验源节点与目标节点当前句柄版本。
```

若后续需要记录业务数据版本，应在对应系统中建立特征值版本、状态版本、动态序号或事件版本，不得复用节点 / 主信息 / 关系句柄版本号。

### 2.3 主信息边界

主信息只保存身份、拓扑锚点、值容器或明确声明的非权威缓存。

凡是表达对象属性、观测事实、统计事实、治理状态、判断结论、任务结果、方法结果或因果解释的内容，应优先进入：

```text
特征
特征值
状态
动态
二次判断结果
因果引用
需求
任务
方法
```

主信息字段迁移红线：

```text
未确认主信息系统规范，不迁移旧字段。
未确认字段功能说明，不新增主信息字段。
未确认字段是否权威，不允许进入值容器。
未确认字段能否由关系、索引、特征、状态、动态、因果替代，不允许编码。
```

### 2.4 关系边界

新项目不得把所有关系混成普通父子关系。第一版至少区分：

```text
普通父子
归属关系
引用关系
索引关系
模板关系
运行期临时关系
因果来源关系
```

关系变更必须通过仓库或领域服务入口完成。

### 2.5 入口前置拒绝规则

本项目编码禁止使用事后校验逻辑兜底结构正确性。

所有会写入或改变结构的函数，必须在函数入口排除错误可能：

```text
无效句柄
已删除句柄
错误节点类型
错误关系类型
越权写入方
缺少前置节点
缺少前置关系
不满足生命周期的对象
```

禁止设计以下运行路径：

```text
先写入无效结构，再通过校验函数发现
先创建悬空关系，再通过扫描器修复
先接受错误节点类型，再由后续流程兜底
先把错误状态写入仓库，再由日志或显示提示
```

构建、测试和人工验证可以证明入口拒绝是否生效，但不得成为运行期机器结构正确性的兜底机制。

### 2.6 容错检查诊断规则

容错检查只能作为预编译宏控制的诊断层。

默认构建不得启用容错检查弹窗；需要调试入口拒绝路径时，必须通过项目统一预编译宏启用：

```text
HY_EGO_ENABLE_FAULT_TOLERANCE_CHECK
```

宏关闭时：

```text
不得改变运行行为
不得写诊断日志
不得弹窗
不得引入机器事实变化
```

宏开启后，容错检查触发时必须同时执行：

```text
日志记录
弹窗提醒
```

日志只用于排错，弹窗只用于人读提醒；二者只记录和提示，不改变返回值、结构结果或机器事实。

弹窗意味着逻辑错误已经暴露。该逻辑错误未排除前，不得继续推进依赖该错误路径的实现、验收或完成声明。

日志和弹窗不得作为：

```text
机器事实
业务状态
任务完成依据
需求满足依据
方法成功依据
自动修复入口
```

容错检查不得替代函数入口前置拒绝。正确路径仍然是先拒绝错误输入，再由宏控制是否记录和提醒。

### 2.7 信息节点口径

`基础信息` 是基础信息类信息节点集合，不只等于 `基础信息` 单一节点类型。

第一轮 `基础信息` 口径包括：

```text
基础信息通用节点
存在
场景
特征
抽象状态
抽象动态
二次特征
因果引用
```

`基础信息通用节点` 只作为当前尚未拆入具体基础信息类型时的通用承载入口，不得替代已经明确的存在、场景、特征、抽象状态、抽象动态、二次特征或因果引用节点类型。

实例状态和实例动态不等同于抽象状态和抽象动态：

```text
实例状态：某个存在在某个场景内某个发生时间戳上的临时状态，以场景内临时零散节点存储。
实例动态：某个存在在某个场景内某个发生时间戳上发生的临时变化记录，以场景内临时零散节点存储。
抽象状态：由实例状态抽象、确认或归纳形成的长期基础信息节点。
抽象动态：由实例动态抽象、确认、归档或因果证据提升形成的长期基础信息节点。
```

实例状态和实例动态必须有存在主体、场景归属和发生时间戳，默认不长期独立存储，不得直接作为语素入口对应信息节点。只有抽象状态和抽象动态才可作为长期基础信息类节点被语素入口绑定和跨场景复用；抽象状态和抽象动态本体不携带发生时间戳。

`高级信息` 不是独立节点类型，不得为它新增节点枚举、字段或仓库结构。

现阶段 `高级信息` 指以下既有高级业务节点角色：

```text
需求
任务
方法
```

当语素、名称、索引或其他系统需要绑定“信息节点”时，必须在函数入口接收并验证已有节点。现阶段允许的信息节点范围为：

```text
基础信息类节点：基础信息通用节点、存在、场景、特征、抽象状态、抽象动态、二次特征、因果引用
需求
任务
方法
```

不允许为了等待后续补齐而先创建悬空入口或临时信息节点。

## 3. 领域服务规则

所有业务写入必须经过领域服务。

第一版领域服务按基础信息原子服务、基础信息聚合服务、高级信息服务和入口服务分层。

基础信息原子服务包括：

```text
特征值服务
特征服务
存在服务
场景服务
状态服务
动态服务
二次特征服务
因果服务
```

基础信息聚合服务为：

```text
世界服务
```

高级信息服务包括：

```text
需求服务
任务服务
方法服务
```

入口服务包括：

```text
语素服务
```

世界服务只做基础信息聚合编排，不得绕过基础信息原子服务直接写基础信息结构。

特征值服务只服务于特征系统，不得作为需求、任务、方法的通用值容器。

只有特征服务可直接包含、接收或调用特征值服务。特征值节点的创建、读取和写入必须经特征服务对外提供；其他领域服务不得直接访问特征值服务。

需求服务、任务服务和方法服务等高级服务只允许直接依赖以下基础信息服务：

```text
存在服务
场景服务
特征服务
二次特征服务
状态服务
动态服务
因果服务
```

高级服务需要特征状态材料或特征值节点时，必须通过特征服务获得，不得绕过特征服务直接访问特征值服务或主信息值容器。

需求服务、任务服务和方法服务需要存在、场景、特征、二次特征、状态、动态或因果能力时，必须通过上述基础信息服务获得，不得直接写基础信息仓库结构。

禁止外部模块直接写：

```text
节点仓库
主信息仓库
关系仓库
索引仓库
```

只读查询可以通过领域服务返回只读视图，不暴露可写内部对象。

## 4. 特征和值规则

一阶机器事实优先使用：

```text
I64 状态值
I64 Q10000 评分
计数
版本号
时间戳
节点句柄
主信息句柄
VecI64
VecIU64
特征值句柄
```

普通 bool 不得作为一阶机器事实扩散。需要二态判断时，应由状态值、评分、计数、版本、来源证据或复合裁决生成。

普通 `bool` 可以作为局部函数执行成功 / 失败返回值、入口拒绝结果或临时控制流条件，但不得持久化为业务事实、需求满足、任务完成、方法成功或世界事实。

## 5. 动作和因果规则

线程不是动作来源。

外部输入不是动作来源。

动作来源必须来自明确方法、领域服务写入入口或后续正式定义的动作提交入口。

因果信息是累积信息，不由单条或少量动态直接形成可靠结论。

因果结构必须区分：

```text
轻量因果引用
可靠因果结论
```

第一轮轻量因果引用只表达“已有来源动态作为准入证据，可以创建因果引用节点”。`记录因果引用(来源动态)` 只检查来源动态有效和类型正确，不持久化因果引用到来源动态的单条关系。

动态信息本身作为后续因果评估的证据材料。因果可靠性必须由后续因果聚合、动态样本积累、评分、版本和复判机制形成。

不得把单个轻量因果引用节点解释为稳定因果结论。

可靠因果结论或完整因果评估后续必须围绕动态证据集合追溯：

```text
来源动作或来源方法
输入材料
被改变目标
改变前值
改变后值
验证方式
证据样本集合
评分或版本
```

方法执行权限来自任务授权、任务承接和任务选择方法。方法执行前不再单独创建授权状态节点。

第一轮方法执行入口必须至少排除：

```text
无效任务节点
无效方法节点
缺少任务到方法关系
无效特征值节点
错误特征值节点类型
```

任务选择方法后，方法服务可以通过特征服务请求修改目标特征状态材料。后续若需要更强权限治理，应围绕任务授权关系、任务状态和任务方法关系设计，不恢复方法执行前临时授权壳。

需求目标是目标状态，不是 I64。I64 只可作为第一轮状态服务或特征服务内部的状态值材料，不得作为需求目标的对外语义或主信息字段迁移依据。

## 6. 显示和日志规则

显示层、日志层和控制台输出只供人读。

显示层可以读取结构：

```text
节点
主信息
关系
特征
特征值
需求
任务
方法
动态
因果引用
```

显示层不得写业务事实，不得作为任务完成、需求满足或方法成功的机器判断来源。

日志系统必须遵守 `规范/日志系统规范.md`。

所有日志都放在 `D:\海中鱼巣\日志` 目录下。

逻辑错误报错必须写入独立逻辑错误日志文件。逻辑错误日志只做人读诊断，不得与运行日志或普通调试日志混写，不得参与机器事实、业务状态、任务结算、需求满足、方法成功或自动修复。

运行日志必须写入独立运行日志文件，主要记录系统运行情况。运行日志只做人读运行态观察，不得作为机器事实、业务状态、任务完成、需求满足或方法成功依据。

调试信息可以按排查需要创建调试日志，但调试日志代码必须按切片使用预编译宏控制启用和关闭。调试日志宏默认关闭；宏关闭时不得写调试日志、不得改变运行行为、不得影响验收结果。

调试完成后必须关闭对应切片调试日志代码，移除默认启用宏或构建参数，避免无用日志长期泛滥。

## 7. 持久化规则

第一版运行期核心以内存仓库为准。

数据库和文件持久化只作为：

```text
快照
事件日志
恢复材料
审计材料
只读查询材料
```

数据库不得直接裁决机器事实，不得绕过领域服务写运行期结构。

## 8. 命名规则

函数名、变量名、类内业务成员名必须使用中文。

允许英文的范围：

```text
C++ 关键字
标准库标识
第三方 API
编译系统字段
外部协议固定字段
```

禁止为了方便新增英文业务变量名、英文业务函数名或英文业务枚举名。

## 9. 完成声明规则

任何“完成 / 已实现 / 已支持 / 已接入 / 已闭环”声明必须给出：

```text
输入入口
核心处理
结构承载
输出变化
调用路径
验证方式
```

缺少结构承载或验证方式时，只能声明为：

```text
文档草案
设计方向
代码壳
部分实现
待验证
```

验证方式必须证明入口前置拒绝已经生效。不得用“事后校验发现了错误”替代“函数入口已经排除了错误”。

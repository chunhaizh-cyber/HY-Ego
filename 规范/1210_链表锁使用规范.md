# 1210 链表锁使用规范

## 1. 目标

本文用于固定本项目中 `链表锁` 的统一使用方式、锁顺序、例外边界与禁止事项。

本文回答以下问题：

- 数据仓库中的各条链表应如何加锁
- 业务模块何时应使用 `锁调度器守卫`，何时可直接使用链表模板自带锁
- `*_已加锁` 接口的真实含义是什么
- 多链同时持有时应按什么顺序获取
- 持锁期间允许做什么，不允许做什么
- 如何利用现有诊断能力排查锁等待与潜在死锁

## 3. 当前实现真值来源

当前链表锁规则，必须以以下两处实现为真值来源：

1. `数据仓库模块`
2. `链表模板`

也就是说：

- 锁顺序以 `锁调度器守卫` 的实际排序逻辑为准
- 链表锁的递归/批量使用规则以 `链表模板` 的 `*_已加锁` 约定为准

必须固定：

`当文字注释与锁调度器实际行为冲突时，以实现为准，不以旧注释为准。`


## 4. 链表锁对象与锁域

当前项目中的链式数据仓库，至少包括以下 7 条链：

- `世界链`
- `特征值链`
- `语素链`
- `语言链`
- `需求链`
- `任务链`
- `方法链`

每条链都内含一个：

- `std::shared_mutex 链表锁`

外层统一通过 `锁请求::读(...)` 或 `锁请求::写(...)` 包装该锁，再交由 `锁调度器守卫` 获取。

当前实现中的链表锁域值为：

- `世界链 = 10`
- `特征值链 = 20`
- `语素链 = 30`
- `语言链 = 40`
- `需求链 = 50`
- `任务链 = 60`
- `方法链 = 70`


## 5. 当前有效锁顺序

### 5.1 实际排序规则

`锁调度器守卫` 当前按以下顺序排序并获取锁：

1. 先按 `锁域` 数值升序
2. 同域内按 `锁模式` 排序，当前表现为 `写 > 读 > 互斥`
3. 若前两项相同，再按锁地址排序

因此，当前对 `链表锁` 的实际有效顺序必须固定为：

`世界链 -> 特征值链 -> 语素链 -> 语言链 -> 需求链 -> 任务链 -> 方法链`

### 5.2 实现裁决

必须固定：

- 上述顺序来自 `枚举_锁域` 的数值与 `锁调度器守卫` 的排序实现
- 后续多链同时持有时，必须服从这条顺序
- 不允许按旧文字说明自行发明另一套顺序

### 5.3 混合非链表锁的约束

当前 `数据仓库模块` 文件顶部的人类注释，与 `枚举_锁域` 的实际数值顺序并不完全一致。

因此在当前版本中，必须固定：

- 新代码默认只把 `链表锁` 与 `链表锁` 一起交给同一个 `锁调度器守卫`
- `链表锁` 与 `消息队列锁 / 帧锁 / OpenGL锁 / 参数锁 / 缓存锁` 的混合批量获取，原则上禁止新增
- 若确有必要，必须先明确锁域顺序，再一次性提交给同一个 `锁调度器守卫`
- 不允许手工先拿一部分锁，再把另一部分锁交给调度器补拿


## 6. 统一使用入口

### 6.1 业务层默认入口

业务模块访问链表时，默认必须使用：

- `锁调度器守卫`
- `锁请求::读(...)`
- `锁请求::写(...)`

也就是说，像 `世界树类 / 需求类 / 任务类 / 方法类 / 调度器` 这类上层模块，应统一走：

`锁调度器守卫({ 锁请求::读或写(某链.链表锁, 枚举_锁域::某链, 锁名, 调用点) })`

### 6.2 模板层边界

只有以下一类场景允许直接使用链表模板自带锁：

1. `链表模板` 自身的对外线程安全包装

必须固定：

- 业务模块不得因为“当前只操作单链”而绕过 `锁调度器守卫`
- 不允许为任何具体链条保留长期例外
- `获取锁()` 与 `获取读锁()` 不得作为业务层默认入口

当前若仍存在业务模块直接使用 `链.获取锁()` 或 `链.获取读锁()` 的历史代码，应视为：

- `现存实现偏差`
- `后续应收敛项`

而不是规范许可。

### 6.3 双写锁约束

当前项目虽保留 `双写锁`，但后续新业务代码必须固定为：

- 优先使用 `锁调度器守卫`
- 不再新增依赖 `双写锁` 的跨链业务逻辑


## 7. 读锁、写锁与已加锁接口

### 7.1 读锁适用范围

以下动作默认使用 `读锁`：

- 主键查找
- 当前链扫描
- 只读枚举
- 深拷贝源树
- 只读缓存命中判断

### 7.2 写锁适用范围

以下动作默认使用 `写锁`：

- 添加节点
- 添加子节点
- 删除节点
- 删除子链
- 删除链表
- 修改主信息
- 改写根指针、同层环、全链环、父子关系
- 写入依赖链结构一致性的缓存

### 7.3 `*_已加锁` 的真实含义

所有 `*_已加锁` 接口都必须理解为：

- 调用者已经持有正确的链表锁
- 该接口不会再次为同一把链表锁自动加锁

必须固定：

- 只读型 `*_已加锁` 接口，调用前至少持有该链的 `读锁`
- 改写型 `*_已加锁` 接口，调用前必须持有该链的 `写锁`

### 7.4 非递归约束

`链表锁` 的底层类型是 `std::shared_mutex`，不是递归锁。

因此必须固定：

- 已经持有某条链的锁时，严禁再调用这条链的“自动加锁版”接口
- 已经持有锁时，只允许调用对应的 `*_已加锁` 版本
- 批量处理、递归删除、递归构造、缓存重建等场景，应一次性拿锁后连续调用 `*_已加锁`

可收敛为一句话：

`已持锁时只能走 *_已加锁，不得二次自动加锁。`


## 8. 批量、递归与跨层操作规则

### 8.1 批量与递归

必须固定：

- 批量操作先拿一次锁
- 递归操作先拿一次锁
- 锁内连续调用 `*_已加锁`

禁止写法包括：

- 在 `删除节点_已加锁` 这类递归路径中再调用自动加锁接口
- 在批量循环中每次迭代都重复获取同一条链的锁

### 8.2 跨模块访问同一条链

若一个上层函数已经拿到了某条链的锁，则其调用下游辅助函数时必须二选一：

1. 明确要求下游只接收 `*_已加锁` 版本
2. 在锁外调用下游自动加锁版本

不允许：

- 上层已拿锁
- 下游却再次按默认线程安全接口自动加锁

### 8.3 读后写升级

当前实现未提供同一把 `shared_mutex` 的安全升级语义。

因此必须固定：

- 不允许依赖“先读后升级为写”的隐式写法
- 若后续一定会修改结构，应从一开始就拿 `写锁`
- 若先读后判定需要写，必须先完全释放读锁，再重新走写锁路径


## 9. 持锁期间允许与禁止的动作

### 9.1 允许的动作

持有链表锁期间，只允许做与链结构一致性直接相关的最小动作：

- 指针挂接与摘除
- 父子关系修正
- 同层环与全链环修正
- 主信息最小写回
- 轻量缓存更新
- 轻量句柄拷贝与局部快照收集

### 9.2 禁止的动作

持有链表锁期间，禁止做以下动作：

- 长耗时计算
- 大范围匹配、评分、推理
- 阻塞等待
- 外设调用
- OpenGL 调用
- 消息队列等待
- 跨线程同步等待
- 不必要的日志洪泛

必须固定：

`锁内只做结构与最小一致性写回，重计算与重逻辑必须移到锁外。`


## 10. 诊断与可观测性要求

### 10.1 锁名与调用点

凡是通过 `锁请求` 提交给 `锁调度器守卫` 的锁，必须尽量补齐：

- `锁名`
- `调用点`

不允许长期保留大量空白调用点。

### 10.2 长持锁告警

当前实现中，单次锁获取耗时超过 `100ms` 时会打告警日志。

因此必须固定：

- 长持锁问题不是“风格问题”，而是显性性能与死锁风险信号
- 出现告警后，应优先排查锁内是否夹带了重计算或锁顺序反转

### 10.3 锁轨迹与等待链

当前实现已提供：

- `锁轨迹环形缓冲`
- `WCT` 等待链分析工具

后续排查锁问题时，必须优先使用现有轨迹与等待链能力，而不是只靠猜测。


## 11. 当前项目的主规则与例外

### 11.1 主规则

当前项目主规则必须固定为：

- 上层业务模块访问链表，默认走 `锁调度器守卫`
- 单链简单读写可只提交一把链锁
- 多链同持必须一次性提交并服从锁域顺序
- 递归与批量必须走 `*_已加锁`

### 11.2 现存偏差的处理口径

必须固定：

- 链表锁使用规范不接受“有风险但可保留的例外”
- 只要存在锁顺序、诊断缺失或后续扩散风险，就不应把该写法写成合法例外
- 当前代码中若仍有直接使用 `链.获取锁()` 或 `链.获取读锁()` 的业务实现，只能按历史偏差记录
- 后续重构应把这些实现收敛回 `锁调度器守卫 + 锁请求`

因此：

`规范可以严于现状，现状不得反向改写规范。`


## 12. 可直接落代码的硬约束

后续代码与巡检至少必须满足以下 10 条：

1. 业务层默认通过 `锁调度器守卫 + 锁请求` 获取链表锁。
2. 多链同时持有时，必须服从 `世界链 -> 特征值链 -> 语素链 -> 语言链 -> 需求链 -> 任务链 -> 方法链` 的当前实现顺序。
3. 注释与排序实现冲突时，以 `锁调度器守卫` 的实际排序逻辑为准。
4. 已经持有某条链的锁时，只允许调用该链的 `*_已加锁` 接口。
5. 不允许在已持锁状态下再次调用同链的自动加锁接口。
6. 批量与递归操作必须一次性拿锁后连续调用 `*_已加锁`。
7. 需要改写链结构时，必须直接使用 `写锁`，不得依赖读锁升级。
8. 持链表锁期间只允许做最小结构动作，重计算必须移到锁外。
9. 新增混合 `链表锁 + 非链表锁` 的批量获取前，必须先确认锁域顺序；默认禁止直接新增。
10. 不允许为 `特征值链` 或任何其它具体链条保留规范级例外；现存直锁代码只按历史偏差处理，不按合法写法处理。
11. 所有 `锁请求` 应尽量补齐 `锁名` 与 `调用点`，以便长持锁告警与锁轨迹诊断生效。

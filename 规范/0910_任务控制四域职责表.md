## 3. 总定义

任务控制必须固定理解为：

```text
任务控制
= 筹办任务
+ 维护任务
+ 纠偏任务
+ 收束任务
```

可直接收成一句话：

- `任务控制，是让任务从被提出到被关闭，始终保持可筹办、可运行、可修正、可收束。`

必须固定：

- `筹办 / 维护 / 纠偏 / 收束` 是功能域
- `创建任务 / 细分任务 / 绑定方法 / 插检查点 / 刷新状态 / 替换方法 / 回退 / 收缩 / 写反馈 / 写经验 / 归档` 是执行动作
- 生长和完善只属于 `筹办任务` 内部能力，不再代表整个任务控制


## 4. 四域职责表

### 4.1 筹办任务

目标：

- 把任务立起来，搭成可执行结构

触发条件：

- 新需求进入
- 现有任务尚未成形
- 当前任务缺方法
- 当前任务需要细分或补前置条件

允许动作：

- 创建任务
- 细分任务
- 铺设步骤
- 绑定方法
- 插入检查任务
- 初始化预算

禁止动作：

- 直接归档
- 无因回退
- 无预算的路径扩张

退出条件：

- 已形成可执行任务链
- 已绑定至少一个可尝试方法
- 必要检查点已插入

### 4.2 维护任务

目标：

- 让任务保持可执行、可跟踪、可稳定推进

触发条件：

- 任务已进入执行期
- 当前链路仍有效
- 方法可继续使用
- 无明显偏差但需持续保活

允许动作：

- 刷新状态
- 维护依赖
- 局部检查
- 更新优先级
- 小范围补丁式修正

禁止动作：

- 大规模重建任务树
- 无证据的回退
- 无必要的多分支扩张

退出条件：

- 出现明显偏差，转纠偏
- 已满足结束条件，转收束
- 任务失去可执行性，回筹办

### 4.3 纠偏任务

目标：

- 把跑偏的任务拉回可控轨道

触发条件：

- 方法失败
- 结果不稳定
- 风险上升
- 目标偏离
- 检查点未通过

允许动作：

- 替换方法
- 回退
- 收缩
- 插恢复任务
- 补条件任务
- 局部重筹办

禁止动作：

- 无限制重试
- 越过安全硬边界
- 直接把失败伪装成完成

退出条件：

- 已恢复到可执行状态，转维护
- 需要重新搭建，转筹办
- 已不值得继续，转收束

### 4.4 收束任务

目标：

- 把任务关干净，留下结果和经验

触发条件：

- 已完成目标
- 已达到可接受结果
- 已确定无法继续
- 已决定终止

允许动作：

- 判定完成、失败、取消
- 清理临时节点
- 关闭无效分支
- 写入反馈
- 学习写回
- 归档结果

禁止动作：

- 再次无因扩张
- 在收束阶段偷偷重启主流程
- 跳过反馈和因果登记

退出条件：

- 最终状态已确定
- 反馈已写入
- 必要经验已沉淀
- 临时资源已释放


## 5. 四域与执行动作映射表

| 功能域 | 主动作 | 辅动作 | 说明 |
| --- | --- | --- | --- |
| 筹办 | 创建任务、细分任务、绑定方法 | 插检查点、初始化预算 | 负责把任务搭起来 |
| 维护 | 刷新状态、维护依赖、局部检查 | 更新优先级、小范围修补 | 负责把任务撑住 |
| 纠偏 | 替换方法、回退、收缩 | 插恢复任务、补条件任务、局部重筹办 | 负责把任务拉回来 |
| 收束 | 判定完成/失败/取消、写反馈、归档 | 清尾、关闭分支、学习写回 | 负责把任务关得掉 |

必须固定：

- `细分 / 延长 / 插检查 / 替换方法 / 回退 / 收缩 / 归并 / 结束` 只能作为域内执行动作理解
- 不允许再把 `细分 / 延长 / 回退 / 收缩` 直接叫成任务控制总模式


## 6. 与双根和可靠度的关系

四域都由 `安全值 + 服务值` 双根驱动，但偏重不同：

- `筹办任务`：服务值决定值不值得立、立多大；安全值决定能立到什么边界
- `维护任务`：安全值决定是否必须维持稳定；服务值决定维持投入是否值得
- `纠偏任务`：安全值决定是否必须立刻纠偏；服务值决定纠偏后是否还值得继续推进
- `收束任务`：服务值决定目标是否已经足够满足；安全值决定是否应尽快关闭不稳定残留

方法可靠度贯穿四域：

- `筹办任务`：用于初选方法、路径长度、检查点密度
- `维护任务`：用于判断当前方法是否还稳、是否要刷新绑定
- `纠偏任务`：用于判断当前方法是否失配、该回退多少
- `收束任务`：用于判断结果是否足够稳定、经验是否应写回

# 5130 子规范 需求信息来源自检与身体反馈

编号：5130
日期：20260720
状态：正式子规范

## 1. 定位

本规范定义根特征自检、普通自检、身体反馈和方法自检如何形成需求候选，以及它们与正式需求、任务、动作和结算的边界。

直接上级规范：3100。

直接依赖规范：5150。

相关边界规范：5100、6130。

## 2. 候选生成

1. 根特征自检必须读取正式抽象特征定义和自我实例特征，比较当前状态与定义中的正常或目标状态。
2. 正常值、区间和状态不得来自硬编码函数表、显示文本或控制面板。
3. 自检和身体反馈只生成缺口报告或强类型候选，不得直接写需求树、刷新权重或创建任务。
4. 子存在只报送自身可观察特征和值；自我领域根据主体、目标定义和当前状态决定是否形成候选。
5. 若不能确定主体、目标特征、当前状态或目标状态，材料只能保留为报告或进入观察/诊断闭环。

## 3. 统一提交与去重

候选必须携带主体、目标宿主、目标特征、当前状态材料、目标状态、来源证据、发生时间和版本，并由需求领域统一入口裁决父域、路径、去重、合并和入树。

同一父需求下的同目标候选只能复用或合并；来源报告和归因差异不得作为新需求身份键。来源、自检类型和获取途径只作为关系证据，不进入需求目标字段。

## 4. 方法自检

1. 方法维护总目标收束为方法状态目标，不把动作、条件结果对、运行次数、成功次数和成熟度拆成无边界的平行需求。
2. “方法状态 = 好用”是维护目标；运行账和条件结果对是证据。第一轮不得仅按运行次数或成功次数宣布好用。
3. 执行学习或练习时发现最低可用性缺口，可形成“方法状态 = 可用”等结构化候选，再由统一需求治理入树。
4. 自检不得直接修改方法状态、动作入口、条件结果对或运行账。
5. 代码缺陷、结构损坏和状态机非法只进入开发诊断，不得伪装成方法学习需求。

## 5. 安全与服务边界

1. 影响自我处理任务、学习、应急和安全行动可靠性的自检缺口可作为安全信号，但不等于物理安全已经变化或修复。
2. 安全信号只进入安全根域；服务信号只进入服务根域，二者不得互相兜底。
3. 服务对象的内在状态不可直接读取；服务反馈只能来自外在表达、确认、回执和外在状态变化。
4. 自检报告生成时不得提前结算安全或服务价值；只有问题解决、目标满足、验证和归因成立后才能走叶子任务结算链。
5. 未分类安全候选可挂到安全根治理路径，但仍必须满足原子目标完整性。

## 6. 动作来源和任务去重

正式入树、合并、满足和结算若产生动作动态，动作来源必须是对应需求领域方法；线程只是调度者。

自检候选可以持续提交。同一方法同时只能有一个活动学习或补齐任务的约束，应在正式任务创建入口去重，不得让自检线程因已有任务而停止报送新事实。

## 7. 验证

验证必须证明：抽象定义是目标来源；候选不直接入树；同目标合并；缺主体/状态被拒绝；代码缺陷进入诊断；安全与服务信号不互相兜底；自检完成不直接结算。

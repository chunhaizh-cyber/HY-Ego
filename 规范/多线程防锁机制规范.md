# 多线程防锁机制规范

日期：2026-07-08
状态：正式规范

## 1. 定位

本规范定义 `海中鱼巣` 后续多线程能力的防死锁和锁边界规则。

当前项目只允许先建立核心仓库读写锁边界和锁友好的服务入口，不声明真实线程调度、工作队列、`std::async`、任务线程池、外设等待或并发压力测试已经完成。

锁只保护运行期内存结构一致性，不承载机器事实，不生成动作来源，不替代事务边界、任务授权、方法执行或因果证据。

## 2. 适用范围

本规范适用于：

```text
节点仓库
主信息仓库
关系仓库
索引仓库
领域服务公开入口
后续确认的任务调度、工作队列或外设等待专项
```

当前已落地的最小范围为：

```text
节点、主信息、关系、索引四个核心仓库内部使用读写锁。
读入口返回记录副本、句柄组或快照材料。
写入口独占修改仓库内部结构。
复杂内部路径可拆 `已加锁` 辅助函数，避免递归持锁。
```

## 3. 基本原则

1. 公开仓库入口不得暴露可写内部引用、长期裸指针或可跨锁生命周期持有的可变对象。
2. 读入口只返回只读材料、副本、句柄集合或快照材料。
3. 写入口必须在函数入口完成无效句柄、错误节点类型、越权写入和前置条件拒绝，再进入结构写入。
4. 普通公开入口原则上只持有一个仓库锁；跨仓库组合写入必须由领域服务或后续确认的事务入口编排。
5. 持锁期间不得调用上层领域服务、显示层、控制台输出、日志落盘、弹窗、持久化 I/O、外设等待、用户回调或可能反向进入仓库公开入口的函数。
6. 线程、锁、等待事件、日志标签和控制台命令不得成为动作来源。

## 4. 锁顺序

第一版禁止在普通领域服务中临时发明多仓库同时持锁路径。

确需在后续事务入口中同时持有多个核心仓库锁时，必须先形成详细设计，并按以下顺序获取锁：

```text
节点仓库
主信息仓库
关系仓库
索引仓库
```

禁止反向顺序获取锁。禁止在持有后序仓库锁时再请求前序仓库锁。

如果无法按固定顺序完成，应拆成以下两段：

```text
读取快照 / 句柄材料
释放锁
按领域服务或事务入口重新执行入口拒绝和写入
```

不得用忙等、循环重试或日志轮询作为正常防死锁机制。

## 5. 读写锁规则

核心仓库读写锁采用以下口径：

```text
读路径：共享锁，只读内部结构，返回副本 / 句柄组 / 快照材料。
写路径：独占锁，完成结构新增、删除、重挂、绑定或值写入。
内部辅助：`已加锁` 函数只能由已经持有对应锁的同类仓库内部函数调用。
```

公开入口不得在持有共享锁后升级为独占锁。需要写入时必须在入口一开始就走写入口，或释放读锁后重新执行入口拒绝和写入流程。

`已加锁` 辅助函数不得被领域服务、显示层、测试入口、工具或外部模块调用；不得自行获取同一个仓库锁；不得跨仓库调用其他公开入口。

## 6. 领域服务边界

领域服务负责业务准入和组合编排，不负责把锁暴露给外部调用方。

领域服务读取多个仓库时，应优先读取短生命周期快照或句柄材料，再释放仓库锁后做业务判断。

领域服务写入多个结构时，必须满足以下条件之一：

```text
已有确认的单仓库写入口可以独立完成该结构变化。
已有确认的事务入口按固定锁顺序完成组合写入。
后续专项详细设计已说明锁顺序、失败边界、回滚或读取隔离方式。
```

禁止领域服务通过持有仓库内部引用来绕过仓库锁边界。

## 7. 禁止迁移边界

以下内容当前不得借多线程防锁规范提前迁移或实现：

```text
旧任务线程池
旧动作线程
std::async 执行分派
SQL 投影锁
控制面板线程
WebView 窗口线程
D455 / 体素 / 外设线程
外设等待队列
线程快照作为机器事实
线程事件作为动作来源
```

上述能力后续如需恢复，必须另建专项，明确允许文件、禁止文件、锁边界、事务边界、验证方式和禁止宣称。

## 8. 逻辑错误和诊断

锁顺序违反、递归公开入口持锁、持锁期间调用外部等待、读锁升级写锁、返回可写内部引用，都属于结构实现缺陷。

若宏开启后检测到上述缺陷，必须写逻辑错误日志并弹窗提醒；日志和弹窗只做人读诊断，不得自动修复结构，不得作为业务状态来源。

若宏关闭，诊断层不得改变运行行为；实现和验收仍必须追根因，不得用“未复现死锁”代替防锁规则满足。

## 9. 验收方式

多线程防锁机制的验收必须分级声明：

```text
规范已确认：本文件、项目规则总纲、规则迁移清单和计划索引已同步。
代码已实现：对应仓库或事务入口已按本规范实现锁边界。
验证已通过：构建、默认入口、文档检查、排除项扫描和必要的并发专项验证通过。
系统级完成禁止宣称：不得把仓库锁边界通过扩大为真实线程调度、工作队列或并发压力测试完成。
```

第一版文档治理验收只证明规范已确认。#129 S0a 只证明核心仓库最小读写锁边界已实现并通过当前构建和默认入口验证，不证明真实多线程调度、工作队列或并发压力测试已通过。

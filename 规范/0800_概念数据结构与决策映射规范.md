# 0800 概念数据结构与决策映射规范

## 1. 目标

本文用于固定以下三件事：

- `概念候选` 最少应保存哪些字段
- `稳定概念` 最少应保存哪些字段
- `稳定概念` 如何映射到需求、任务、方法的决策与评分

本文不要求先新增一套独立的大结构。

本文优先原则是：

- `复用现有节点与主信息字段`
- `概念身份靠签名与关系，不靠名称`
- `候选阶段名称可空，但稳定概念应命名，且不能把自然语言词面当真值`

## 2. 现有结构上的推荐落点

按当前工程实现，概念相关信息推荐落在三层：

1. `二次特征节点 / 二次特征主信息类`  
   用于表达：观测结果、结构化关系、候选线索、概念模板命中。

2. `因果节点 / 因果主信息类`  
   用于表达：因果方向、观察次数、稳定度、时间顺序与过程签名。

3. `概念树节点`  
   用于表达：已经稳定下来的概念锚点，以及它与其它概念/特征/因果之间的关系。

因此建议的职责分工是：

- `二次特征` 负责“看到什么关系”
- `因果` 负责“确认关系是否稳定成立”
- `概念树` 负责“把已稳定关系挂成可复用概念节点”

### 2.1 方向差值型二次特征的落点

当二次特征来自 `当前值` 与 `目标值` 的比较时，推荐优先落在 `二次特征主信息类`，并至少保留：

- `当前值`
- `目标值 / 目标线 / 最大值`
- `差值`
- `区间归属`：正区间、负区间、零区间
- `绝对方向`

其中：

- `绝对方向` 可以由 `区间归属` 回推
- 若当前工程强调字段最小化，可把 `绝对方向` 作为计算字段而不是独立主键字段
- 但 `当前值 / 目标值 / 差值 / 区间归属` 这四项不应丢失

### 2.2 状态概念树、复合状态概念与复合特征的落点

当前应进一步区分三类抽象概念：

1. `状态概念`
   对单一特征值或特征值区间的抽象描述，例如 `红色`。

2. `复合状态概念`
   由多个状态概念交集聚合形成的更高层状态概念，例如 `健康`、`病态`。

3. `复合特征概念`
   由多个状态概念组合形成的更高层特征概念，例如 `美丽`、`强壮`。

推荐落点如下：

- `状态概念` 优先挂到 `概念域::状态`
- `复合状态概念` 优先挂到 `概念域::状态`
- `复合特征概念` 优先挂到 `概念域::特征`

其中 `状态概念树` 的内部结构应固定为：

1. `特征类型状态概念根`
   例如：颜色、力量、温度。

2. `稳定区间状态概念`
   例如：红色、强、冰冷。

3. `更细稳定区间状态概念`
   当上层区间还能稳定细分时继续生成。

4. `复合状态概念节点`
   由多个状态概念交集聚合而成，例如：健康、病态。

推荐关系如下：

- `状态概念树` 内部优先用父子层级表达“细分”
- `状态概念 --state_of--> 对应特征概念` 可作为跨域引用
- `复合状态概念` 与其组成状态概念之间应保留组成关系边，例如 `part_of`
- `复合特征概念` 与其组成状态概念或组成特征之间应保留组成关系边，例如 `part_of`

必须固定：

- `状态概念` 不等于运行期状态事实
- `复合状态概念` 也不等于运行期状态事实
- `复合特征概念` 不属于存在概念
- 某个存在只能命中这些概念，不能充当这些概念本身
- `状态概念树` 不是先有叶子再找根，而是先有根再逐层细分
- `复合状态概念` 属于状态域中的聚合分支，不替代单一特征细分主干


## 3. 概念身份规则

### 3.1 不能用名称当身份

概念身份不能主要依赖：

- 名称字符串
- 自然语言标签
- 人工词面命名

因为在早期阶段：

- 候选阶段名称可以为空
- 名称可以变更
- 稳定概念必须至少绑定一个主名称
- 同一概念可以有多个描述版本

### 3.2 概念身份的最小真值

概念身份应主要由以下组合决定：

- `概念域`
- `作用对象签名`
- `被作用对象签名`
- `作用方向`
- `作用维度`
- `适用条件签名`
- `度量签名 / 因果签名`

可收敛为：

- `概念签名 = 结构化关系签名`

名称只是附属解释字段，不是概念主键。

但必须补一条：

- 稳定概念一旦成立，应进入命名流程，不能长期保持无名状态


## 4. 概念候选的最少字段

概念候选是“值得继续观察的关系假设”，推荐至少保存以下字段。

### 4.1 身份字段

1. `候选签名`  
   用于唯一标识当前候选关系。

2. `概念域`  
   例如：二次特征域、因果域。

3. `作用对象`  
   例如：`服务值`。

4. `被作用对象`  
   例如：`安全值`。

5. `作用维度`  
   例如：恢复速度、衰减速度、待机风险、稳定性。

6. `作用方向`  
   例如：正向、负向、条件性正向、未知。

7. `抽象层级`  
   例如：状态概念、复合状态概念、复合特征概念、因果概念。

### 4.2 证据字段

8. `证据次数`  
   当前已观察到多少次同类证据。

9. `首次观察时间`
10. `最近观察时间`
11. `证据链指针集合`  
    至少能回指到相关状态、动态、二次特征或因果节点。

12. `一致性Q`  
    当前证据方向是否稳定一致。

13. `可复现度Q`  
    在相似条件下是否可重复出现。

14. `干扰程度Q`  
    当前样本受到其它主因干扰的程度。

### 4.3 根需求投影字段

15. `根需求投影结果`  
    正向、负向、中性、未知。

16. `净贡献Q`  
    当前候选对根需求的净贡献估计。

17. `决策级别`  
    不参与、弱参与、中参与、强参与。

### 4.4 适用范围字段

18. `适用条件签名`  
    该关系在哪类上下文中成立。

19. `时间窗`
20. `样本范围说明`

### 4.5 状态概念、复合状态概念与复合特征的补充字段

若候选属于 `状态概念`、`复合状态概念` 或 `复合特征概念`，还推荐补充：

1. `来源特征签名`  
   该状态概念源于哪个单一特征。

2. `父状态概念签名`  
   若该状态概念来自上层细分，则记录其父概念。

3. `值区间签名 / 值类别签名`  
   该状态概念对应哪个特征值范围或类别。

4. `组成状态签名集合`  
   若为复合状态概念或复合特征概念，记录其组成状态。

5. `来源存在覆盖数 / 交集支持度`  
   记录有多少不同存在或样本共同支持该交集。

6. `观察期稳定度Q`  
   记录该交集在观察期内的稳定程度。

7. `概念树目标域`  
   标识它最终应挂到 `状态域` 还是 `特征域`。

8. `细分层级`
   根层级、一级细分、二级细分等。

9. `是否终端`
   当前是否还需要继续细分。

10. `生成路径类型`
    单特征细分、跨存在聚合等。

### 4.6 与现有字段的推荐映射

在现有结构中，概念候选优先映射为：

- `二次特征主信息类::度量签名_链键` -> 候选的度量定义
- `二次特征主信息类::主体 / 客体 / 左对象 / 右对象` -> 作用对象与被作用对象
- `二次特征主信息类::标量值` -> 当前一次证据的量化结果
- `二次特征主信息类::观察` -> 观察次数、最近时间、熟悉度、不确定度
- `二次特征主信息类::时间窗` -> 适用时间窗
- `二次特征主信息类::依赖度量签名表` -> 复合关系依赖

若候选来自目标比较型二次特征，还应额外能回指：

- `当前值`
- `目标值`
- `差值`
- `区间归属`

如果当前还缺 `净贡献Q / 一致性Q / 可复现度Q / 干扰程度Q`，应优先作为补充字段加到承载候选的主信息上，而不是另起独立平行结构。


## 5. 稳定概念的最少字段

稳定概念是在候选基础上升级得到的，因此应至少保留候选全部核心字段，并新增以下字段。

### 5.1 稳定身份字段

1. `稳定概念签名`
2. `稳定级别`  
   弱稳定、稳定、强稳定。

3. `概念锚点节点`  
   指向概念树中的概念节点。

4. `主名称`  
   稳定概念至少应绑定一个名称，仅用于解释，不作为身份主键。
   它同时是高级逻辑思维中的稳定引用锚点，也是对外交流对准的基础。

5. `概念树域`  
   状态域、特征域、因果域等。

### 5.2 因果确认字段

6. `因果签名`
7. `观察总次数`
8. `方向一致率`
9. `先因后果成立率`
10. `可复现度Q`
11. `稳定度Q`
12. `置信度Q`

### 5.3 决策字段

13. `根需求投影结果`
14. `净贡献Q`
15. `决策参与级别`
16. `默认作用方式`  
    加分、减分、调权、硬过滤、生成子需求。

17. `作用边界`  
    说明它影响的是恢复、衰减、待机、风险还是其它维度。

18. `失效条件签名`  
    在何种条件下不应继续使用该概念。

19. `降级计数 / 反例计数`
20. `命名状态`
    待命名、已命名、待修订等。
21. `命名优先级`
    用于调节命名任务的先后次序。
22. `命名触发原因集合`
    例如 `高级逻辑复用`、`对外沟通`、`跨模块复用`、`高风险区分`。

### 5.4 状态概念、复合状态概念与复合特征的补充字段

若稳定概念属于 `状态概念`、`复合状态概念` 或 `复合特征概念`，还推荐新增：

1. `来源特征签名`
2. `父状态概念签名`
3. `稳定值区间签名`
4. `组成状态签名集合`
5. `来源存在覆盖数 / 交集支持度`
6. `观察期稳定度Q`
7. `生成路径类型`
8. `细分层级`
9. `是否终端`
10. `关系边集合`  
   例如 `state_of`、`part_of`

### 5.5 与现有字段的推荐映射

在现有结构中，稳定概念优先映射为：

- `概念树节点` -> 稳定概念锚点
- `因果主信息类::条件签名 / 结果签名 / 过程签名` -> 因果身份与过程身份
- `因果主信息类::观察次数` -> 证据总量
- `因果主信息类::稳定度Q` -> 稳定性
- `因果主信息类::首次发生时间 / 最近发生时间` -> 时间证据
- `二次特征主信息类::概念模板` -> 候选/实例与稳定概念锚点之间的连接

如果还缺 `净贡献Q / 决策参与级别 / 失效条件签名 / 反例计数`，建议补到稳定概念承载层，而不是放在自然语言层。


## 6. 概念候选与稳定概念的最少区别

为避免“线索一出现就当真理”，两者必须严格区分：

1. 候选可以没有概念树锚点  
   只要已有结构化签名和证据链即可。

2. 稳定概念必须有概念树锚点  
   否则不能成为可复用判断依据。

3. 候选可以只参与观察，不参与决策。
4. 稳定概念才允许参与决策。
5. 候选的根需求投影可以是未知。
6. 稳定概念的根需求投影必须明确。
7. 候选概念可以暂时无名。
8. 稳定概念应自动生成命名任务，并至少最终绑定一个主名称。


## 7. 概念到需求的映射规则

稳定概念进入需求层时，不是直接变成一句话，而是变成“需求权重修正依据”。

### 7.1 映射目标

概念对需求主要影响：

- 是否生成某类需求
- 某类需求的权重增减
- 某类需求是否提前

### 7.2 推荐映射字段

需求层最少需要读取：

- `根需求投影结果`
- `净贡献Q`
- `决策参与级别`
- `适用条件签名`
- `作用边界`

### 7.3 推荐映射规则

1. 若概念为正向稳定概念  
   则提高与其一致的需求权重。

2. 若概念为负向稳定概念  
   则生成修复型或规避型需求。

3. 若概念只在局部条件成立  
   则只在命中适用条件时参与需求修正。

4. 若概念处于弱稳定状态  
   则只影响排序，不直接生成新需求。

### 7.4 服务概念的例子

若稳定概念成立：

- `服务值对安全值具有正向单调影响`

则可映射为：

- 提高“恢复服务条件”相关需求权重
- 当服务值偏低时，提前服务相关需求
- 当服务值极低时，生成服务恢复型子需求


## 8. 概念到任务的映射规则

稳定概念进入任务层时，主要影响：

- 任务是否生成
- 任务优先级
- 任务是否应被暂缓

### 8.1 推荐映射结果

任务层可把概念映射为：

- `基准优先级修正`
- `局部优先级偏移`
- `是否允许进入当前轮执行`

### 8.2 推荐映射规则

1. 正向概念命中当前条件  
   给一致任务加分。

2. 负向概念命中当前条件  
   给高风险任务减分，必要时暂缓。

3. 条件性概念  
   仅在适用条件命中时生效。

4. 弱参与概念  
   只能影响 `局部优先级偏移`。

5. 强参与概念  
   才允许直接触发“生成任务”或“禁止任务”。

### 8.3 与现有任务结构的对接

按当前实现，建议直接对接：

- `任务主信息::基准优先级`
- `任务主信息::局部优先级偏移`
- `任务执行器` 中的候选排序逻辑

也就是说，概念层不应自己调度任务，而应输出：

- `任务层可消费的加减分与过滤结论`


## 9. 概念到方法的映射规则

稳定概念进入方法层时，主要影响：

- 候选方法评分
- 方法规避过滤
- 条件缺口分析后的子任务方向

### 9.1 方法层最少读取字段

- `净贡献Q`
- `决策参与级别`
- `作用边界`
- `适用条件签名`
- `失效条件签名`

### 9.2 推荐映射规则

1. 若某方法预测结果与正向概念一致  
   给该方法加分。

2. 若某方法预测结果与负向概念一致  
   给该方法减分。

3. 若某方法会触发强负向概念  
   可进入规避过滤。

4. 若概念表明“缺少某条件会持续损害根方向”  
   则把该条件缺口转成优先子任务。

### 9.3 与现有方法评分的对接

按当前实现，建议把概念映射为：

- `评分Q` 加减项
- `avoid_filter_` 的判定依据
- `条件缺口 -> 子需求 -> 子任务` 的优先级修正项

这样可以直接接到现有：

- `评分_方法结果满足需求(...)`
- `结构_筹办方法候选::评分Q`
- `追加候选(...)`
- `avoid_filter_`

而不需要重写整条筹办主链。


## 10. 决策映射的统一格式

为了避免每个模块自己解释概念，推荐把概念输出统一规约为四类结果：

1. `加分`  
   对齐根方向，提升候选优先级。

2. `减分`  
   偏离根方向，但尚未达到禁止级。

3. `调权`  
   影响需求或任务权重，而非直接过滤。

4. `硬过滤`  
   明确与根方向冲突，当前上下文下禁止。

必要时可以加第五类：

5. `生成缺口任务`  
   当前问题不是执行某方法，而是先补齐关键条件。


## 11. “服务越高越好”在数据结构上的落点示例

如果该概念已经稳定成立，建议落点如下：

### 11.1 候选阶段

- 作用对象：`服务值`
- 被作用对象：`安全值`
- 作用维度：`恢复速度 / 衰减速度 / 待机风险`
- 方向：`正向单调`
- 证据：服务变化与安全变化相关的二次特征节点

### 11.2 稳定阶段

- 因果签名：`服务 -> 安全`
- 根需求投影：`正向`
- 净贡献Q：正值
- 决策参与级别：先弱后中，再视证据升级为强
- 概念锚点：概念树中的一个已稳定概念节点，并应绑定主名称

### 11.3 决策映射

- 需求层：提高服务恢复相关需求权重
- 任务层：提高服务恢复任务的局部优先级
- 方法层：给恢复服务条件的方法加分
- 极端低服务时：对高风险方法减分或过滤


## 12. 本文件的约束结论

后续实现中应固定以下结论：

1. 概念身份以结构化签名为主，不以名称为主。
2. 概念候选与稳定概念必须分层保存，不能混用。
3. 优先复用 `二次特征主信息类`、`因果主信息类` 与 `概念树节点` 承载概念信息。
4. `状态概念`、`复合状态概念` 与 `复合特征概念` 都不应落到存在概念域。
5. 状态概念树应先以 `特征类型` 建立最抽象状态概念根，再按稳定区间逐层细分。
6. 状态概念树的当前叶子节点应表示“当前已细分到的最细稳定状态概念”，而不是天然固定不变的终极叶子。
7. `健康`、`病态` 这类复合状态概念应优先挂入状态域的聚合分支，并保留与组成状态的关系边。
8. `美丽`、`强壮` 这类复合特征概念应优先挂入特征域。
9. 目标比较类二次特征应至少保留 `当前值 / 目标值 / 差值 / 区间归属`，必要时可缓存 `绝对方向`。
10. 若字段不足，应优先在现有承载层补字段，而不是新造一套平行概念系统。
11. 稳定概念进入需求、任务、方法层时，应统一输出为 `加分 / 减分 / 调权 / 硬过滤 / 生成缺口任务` 之一。
12. `服务越高越好` 这类概念在结构上必须先有证据、再有因果、再有根需求投影，最后才能进入决策。
13. 候选名称可以为空，但稳定概念必须至少绑定一个主名称。
14. 所有稳定概念都应生成命名任务，因为名称是高级逻辑思维与对外交流对准的基础；命名优先级可按高级逻辑复用、对外沟通、跨模块复用、高风险区分等因素调节。
15. 概念签名、证据链、根需求投影和决策级别不能为空。

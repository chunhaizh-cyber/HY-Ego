# 0940 四域方法树的出生与成长规则

## 1. 目标

本文用于固定以下三件事：

- 四棵树首先是什么
- 它们如何出生、如何成长
- 它们如何在具体任务上实例化为任务树

本文要彻底钉准：

- `筹办 / 维护 / 纠偏 / 收束` 这四棵树，首先是 `域方法树`
- `具体任务树` 不是先天存在，而是域方法树在 `具体任务虚拟存在` 上逐段实例化后的运行结果
- `任务虚拟存在` 是四棵域方法树的统一运行时主入参，不是抽象方法树本体

## 3. 总定义

必须固定：

- 四棵树是 `域方法树`，不是 `具体任务树`
- `具体任务树` 是四棵域方法树在某个具体任务上的逐段实例化结果
- `任务虚拟存在` 只表示“当前正在处理的那个具体任务对象”
- `任务虚拟存在` 是统一主入参，不是四棵抽象方法树的宿主
- `总控` 只负责 `判域 -> 选树 -> 执行 -> 接收结果`，不是第五棵树
- 文档中出现的 `创建任务骨架`、`局部回退`、`写最终反馈`、`归档结果` 等名字，只是语义层的方法片段名，不等于新增底层原语
- 这些语义片段在实现时，必须分解到既有元函数、既有原生学习兜底函数，或由它们组合并已沉淀入树的组合方法片段

可直接收成一句话：

```text
先有四棵方法树，后有一棵任务树。
方法树靠兜底成功发芽，靠稳定复现长大。
```


## 4. 两层出生

### 4.1 系统级出生

四棵域方法树不是从零自发长出来的。

如果完全从零开始，就会落入：

```text
没有方法
-> 需要学习
-> 学习也需要方法
```

因此必须固定：

- 四棵域方法树的树根，来自开发者预置的原生元函数、由既有元函数组合而成的本域基础方法片段，以及本域兜底函数入口首节点

每棵域方法树出生时，至少有三类根节点：

1. 域入口根节点
2. 由既有元函数组合而成的本域基础方法片段首节点
3. 本域兜底入口首节点

所以系统初始化时出生的，不是茂密大树，而是：

```text
根节点 + 少量原生枝条 + 一个兜底出口
```

### 4.2 任务级出生

抽象域方法树在系统初始化时已经出生。

但对某个具体任务来说，真正“活过来”的时刻是：

- 这个任务虚拟存在第一次进入筹办域
- 这个任务虚拟存在第一次进入维护域
- 这个任务虚拟存在第一次进入纠偏域
- 这个任务虚拟存在第一次进入收束域

此时发生的是：

```text
域方法树 x 具体任务虚拟存在 -> 当前域任务子树
```

因此必须区分：

- `方法树先天出生`
- `任务树后天出生`


## 5. 四棵方法树的初始骨架

本节所列内容表示“域方法树应具备的最小语义片段”。

必须固定：

- 本节内容不表示新增底层原语
- 其实现必须分解到既有元函数、既有原生学习兜底函数，或由它们组合并已沉淀入树的组合方法片段

### 5.1 筹办方法树

出生时最少应具备：

- 创建任务骨架
- 判断是否需要细分
- 生成子任务
- 生成补条件任务
- 查询可用方法
- 绑定方法
- 无方法时挂等待学习
- 插入初始检查点

### 5.2 维护方法树

出生时最少应具备：

- 读取当前步骤
- 执行当前方法一步
- 采样结果
- 判断结果是否稳定出现
- 更新方法可靠度
- 判断继续维护、转纠偏、转收束、回筹办

### 5.3 纠偏方法树

出生时最少应具备：

- 识别偏差
- 查询替代方法
- 替换方法
- 局部回退
- 收缩路径
- 回到筹办
- 直接转收束

### 5.4 收束方法树

出生时最少应具备：

- 判定完成、失败、取消
- 写最终反馈
- 更新可靠度
- 清理临时状态
- 关闭无效分支
- 归档结果


## 6. 四域方法树与现有方法信息数据架构的一一对应

### 6.1 基本对应

| 四域方法树中的概念 | 现有方法信息数据架构中的落点 | 说明 |
| --- | --- | --- |
| 域方法树根 | `方法首节点主信息类` | 每个域至少有一个域入口首节点 |
| 普通方法入口 | `方法首节点主信息类` | 一个方法片段总是从首节点开始 |
| 条件筛选 / 条件分叉 | `方法条件节点主信息类` | 是否命中某条分支，在这里判定 |
| 结果落点 / 下一步导向 | `方法结果节点主信息类` | 决定结束、续接、转域、兜底 |
| 兜底入口 | `方法首节点主信息类` | 只是动作句柄指向兜底函数，不是新节点种类 |
| 树边 | `首 -> 条 -> 果 -> 下一首` 的引用关系 | 不再另造独立边模型 |
| 一条方法路径 | 一串 `首 + 条 + 果 + 下一首 ...` 的方法片段链 | 这是运行时匹配和实例化的单位 |

必须固定：

- 四域方法树不得另起独立树节点体系
- 四域方法树必须完全落在现有 `方法首节点主信息类 / 方法条件节点主信息类 / 方法结果节点主信息类` 上

### 6.2 运行时实例化总链

四棵域方法树在运行期，必须按以下结构链路实例化：

```text
匹配域入口首节点
-> 用任务虚拟存在判定条件节点
-> 命中某个结果节点
-> 由结果节点决定：
   - 结束当前片段
   - 跳转同域下一首节点
   - 转入本域兜底首节点
   - 向总控返回下一域建议
```

总控外层链路可直接收成：

```text
总控.处理(任务虚拟存在)
-> 判定当前功能域
-> 执行对应域方法树(任务虚拟存在)
-> 返回当前域结果与下一域建议
```

对应四个统一入口：

```text
执行筹办树(任务虚拟存在)
执行维护树(任务虚拟存在)
执行纠偏树(任务虚拟存在)
执行收束树(任务虚拟存在)
```

必须固定两层依赖边界：

- `运行时主入参`：具体任务虚拟存在，表示“现在处理谁”
- `模块级依赖`：方法注册表、兜底函数集、控制规则、日志、持久化、学习写回通道，表示“拿什么处理它”

这两层不得混成一堆散参数。


## 7. 方法树与任务树的对应关系

必须固定：

- 方法树不是任务树
- 方法树是模板树
- 任务树是方法树在具体任务上的实例化结果

可直接写成：

```text
当前域任务子树 = 实例化(当前域方法树, 具体任务虚拟存在)
总任务树 = 不同时刻实例化出来的各域任务子树的拼接结果
```

所以：

- 筹办方法树实例化，长出任务骨架
- 维护方法树实例化，长出推进子树
- 纠偏方法树实例化，长出修复子树
- 收束方法树实例化，长出关闭子树


## 8. 方法树的成长动作

方法树的成长，不是“凭空长方法”，而是把 `兜底成功路径`、`稳定重复路径`、`因果总结路径` 吸收回本域方法树。

这些成长动作在现有方法节点结构上的唯一允许落点如下。

### 8.1 发芽

发生条件：

- 本域树里没有合适处理路径
- 调用了本域兜底函数
- 兜底函数在某个具体任务上走通了一条局部路径

结果：

- 新增一组 `方法首节点主信息类 + 方法条件节点主信息类 + 方法结果节点主信息类`
- 也就是新增一组 `首 + 条 + 果`

### 8.2 抽枝

发生条件：

- 已有方法能做前半段，但后半段经常缺一步
- 某个检查节点经常需要补入
- 某个恢复动作反复出现

结果：

- 在已有 `方法结果节点主信息类` 后追加新的 `方法首节点主信息类`
- 形成 `结果 -> 下一首` 的续接

### 8.3 分叉

发生条件：

- 条件不同，适合的后续路径不同
- 安全值、服务值、需求值不同
- 方法可靠度不同
- 结果稳定性不同

结果：

- 在同一 `方法首节点主信息类` 下新增 `方法条件节点主信息类` 分支
- 由不同条件节点通向不同结果节点

### 8.4 归并

发生条件：

- 多条路径主干相同，只是局部细节不同
- 因果总结发现它们本质上属于同一类方法
- 可以抽出一个更通用的父节点

结果：

- 抽出共同的 `方法首节点主信息类`
- 原有多条路径退为该首节点下的条件变体或结果变体

### 8.5 修剪

发生条件：

- 某条路径长期不稳定
- 某条路径几乎不用
- 某条路径只在极少条件下可用
- 某条路径在安全上已不再适合

结果：

- 只允许调整既有 `条件节点 / 结果节点` 的优先级、可见性、禁用状态或权重
- 不允许为了修剪再引入第二套结构模型


## 9. 方法树成长的驱动力

必须固定：

- 方法树成长的核心驱动力，不是生命周期，而是 `结果是否稳定出现`

因此判断一条新枝能否长稳，看的是：

- 这条路径被尝试后
- 在相似任务虚拟存在上
- 目标结果是否能稳定复现

可直接压成一句：

```text
兜底成功让新枝出生，稳定复现让新枝长稳。
```


## 10. 四棵方法树分别吸收什么

### 10.1 筹办方法树

主要吸收：

- 新的任务细分方式
- 新的补条件方式
- 新的初始绑定方式
- 新的等待学习挂接方式
- 新的初始检查点布置方式

它长的是：

- `怎么把任务立起来` 的套路

### 10.2 维护方法树

主要吸收：

- 稳定的一步推进链
- 检查点插入模式
- 可靠度更新模式
- 继续维护、转纠偏、转收束、回筹办的判别套路

它长的是：

- `怎么把任务稳稳往前推` 的套路

### 10.3 纠偏方法树

主要吸收：

- 哪种失败先换方法
- 哪种失败先回退
- 哪种偏差需要补条件任务
- 哪种偏差应直接收束

它长的是：

- `怎么把偏掉的任务拉回来` 的套路

### 10.4 收束方法树

主要吸收：

- 哪种完成判定更稳
- 哪种失败收口更干净
- 哪种反馈写法更有价值
- 哪种清理与归档顺序最稳

它长的是：

- `怎么把任务体面地关掉` 的套路


## 11. 工程纪律

必须固定：

- 四棵树是方法组织层，不是四份散乱函数集合
- 四棵树对同一具体任务的运行，都统一吃 `任务虚拟存在` 这个主入参
- 运行期生成的是 `当前域任务子树`，不是去改写抽象方法树本体
- 抽象方法树的成长必须经过写回、验证与稳定性判据，不允许把一次偶然成功直接抬成稳定主干
- 本域优先，本域兜底，本域沉淀，不得跨域偷写别域主干
- 四域方法树不得另起独立树节点体系，必须完全落在现有的 `方法首节点主信息类 / 方法条件节点主信息类 / 方法结果节点主信息类` 上
- 四域方法树中的所有动作名，默认都只是语义层的方法片段名，不得被解释为新增底层原语
- 四域方法树中的所有动作句柄，必须绑定到既有元函数、既有原生学习兜底函数，或由其组合并已沉淀入树的方法片段；不得引入未注册新原语
- 动作句柄统一沿用现有 `结构体_动作句柄` 表示，不另造新的句柄模型
- 若动作句柄绑定的是组合方法片段，则继续通过既有 `动作序列主键` 或已沉淀的方法片段链表达
- 若动作句柄绑定的是原生元函数或兜底函数，则继续通过既有 `本能函数ID` 或已注册 `外部实现主键` 表达

## 13. 收束

本文最后固定为：

> 这4棵树先作为“域方法树”由原生元函数、既有元函数组合片段和本域兜底函数入口出生；当具体任务虚拟存在进入某个功能域时，对应方法树被匹配，并以现有 `方法首节点 / 方法条件节点 / 方法结果节点` 结构实例化，长出该任务的一段任务子树；当兜底路径或运行路径反复稳定成功时，这些路径再被吸收回对应方法树，从而让这4棵方法树不断发芽、抽枝、分叉、归并和修剪。

可再压成最短一句：

> 先有四棵方法树，后有一棵任务树；方法树靠兜底成功发芽，靠稳定复现长大。

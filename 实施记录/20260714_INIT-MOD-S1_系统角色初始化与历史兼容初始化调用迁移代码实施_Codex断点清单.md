## 3. 核心处理与结构承载

系统角色初始化器按以下顺序形成并复核角色：世界根、自我场景、自我实际存在、安全 / 服务两组特征定义、实例槽位、当前值、抽象目标状态和根需求、概念存在 / 动态 / 关系 / 因果四根、方法登记根及三个共享状态。

调用路径固定为：

```text
运行期上下文
-> 唯一运行期业务装配
-> 系统角色初始化器
-> 既有无令牌业务服务 / 组合器与系统角色专用数据操作
-> 结构写入执行器 / 不透明会话
-> 四仓库
-> 权威读回的系统角色清单。
```

概念存在根经存在业务服务创建实际存在；概念动态根经抽象动态形成；概念因果根经该动态形成轻量因果；概念关系根由 `[存在根, 因果根]` 有序组成。方法登记根及共享状态经需求任务方法分层入口形成。最后由系统角色数据操作在同一会话发布 `世界根 -> 自我场景 -> 自我存在` 两条普通父子关系，并从目标方向读回唯一父关系。

四个历史初始化文件原地转换为真实 `ClCompile` 模块，现有调用点只由 `.ixx #include` 迁为 `import`；旧初始化模块未注入新系统角色路径。

## 4. 实施期根因与修复

四个历史文件转换为模块后，直接导入需求初始化的旧调用点不能再依赖文本包含获得世界树公开类型。需求初始化模块改为 `export import 海中鱼巣.领域.初始化.世界树`，恢复原公开可见性且不改变历史初始化行为。

系统角色首次自检曾在 21 个角色全部形成后返回内部不一致。根因是系统实际存在校验把自我存在指向安全 / 服务根需求的合法 `引用` 误判为虚拟来源。修复后，自我存在只允许引用本清单两条根需求；概念存在根继续拒绝任何需求、任务或方法来源引用。预检同时保存权威场景归属关系，重复初始化直接复用该关系，避免兼容读取把合法根需求引用重新分类为虚拟来源。

## 5. 失败收口与发布边界

21 个稳定键在写前全量检查非零、互异、预期节点类型和唯一主键绑定。已有任一角色只有在完整句柄、主键、业务语义和关系全部权威匹配时才可复用；错误类型、错误语义、额外主键、非法引用、冲突父关系或版本漂移均在发布前收口。

两条普通父子关系最后发布；后继失败由不透明会话撤销本次结构。并发同参只允许一次提交和一次幂等读回。宿主对完整候选上下文仍固定返回 `首次发布尚未启用`，不保存当前上下文、不生成租约，也不进入 #249 的首次发布和恢复链所有权。

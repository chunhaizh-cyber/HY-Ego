## 2. 总结结论

1. 当前没有可生成完整权威快照的仓库导出接口，也没有按指定身份恢复、恢复高水位或一次发布运行期上下文的入口。
2. 只保存当前有效的节点、主信息和关系会复活已删除身份；四仓库记录状态、当前版本、仓库编号、下一个编号和节点创建序号必须共同保留。
3. 特征值权威材料并不全在四仓库：I64 原始值版本、VecI64 / VecU64 完整值和容器版本位于特征值服务侧表。
4. 概念图四根登记、生命周期登记、专用关系登记、活动图版本和活动快照位于概念图服务；只恢复仓库会留下无法经现有服务入口读写的半恢复状态。
5. 活动图中的概念、关系和签名原则上是领域结构的派生发布视图，但当前没有从完整仓库确定性重建全部登记与活动快照的正式入口；恢复首版必须把其作为候选证据并重读权威结构复核。
6. 上一候选图、统计缓存、控制面板投影、窗口状态、线程队列、消息幂等表、会话计数和普通回执均为瞬态材料，恢复时必须丢弃或重建。
7. 索引仓库不裁决领域事实，但现有部分稳定键和键占用事实没有其它可枚举的权威登记来源；在所有者专用登记形成前，索引不能简单丢弃后重建，也不能未经领域复核直接恢复。
8. 结构化事件段 HYEVSEG1 只记录审计事件材料，不包含四仓库记录前后镜像、服务侧表或统一提交序号，当前不能用来重放或补齐快照。
9. 现有语素、世界树、两个根需求和概念图初始化都是新建路径；恢复后重新运行会重复创建或因稳定键已占用而失败，必须增加恢复态系统角色清单和重建入口。
10. 当前启动在栈上逐个构造仓库和服务，并先执行新建初始化，没有隔离运行期上下文和恢复分支；恢复必须先构建完整候选上下文，再一次发布，发布后才能启动业务线程、SQL 审计和显示。
11. 性能并行可以用于冻结后不可变材料的分段编码、校验和隔离解析；同一运行期事实的冻结、跨段完整复核和最终发布必须保留明确屏障，不能让并行完成顺序裁决业务状态。
12. 现有仓库快照详细设计已被当前代码事实超出。#202 完成后必须先生成新的持久化恢复流程图、详细设计和依赖门控计划，不能直接进入 C++。

## 4. 四仓库与事务能力

### 4.1 节点仓库

当前记录包含节点编号、节点类型、主信息句柄、记录版本、记录状态和创建序号。仓库构造时接收仓库编号，下个节点编号和下个创建序号从 1 开始。

当前入口只提供创建、当前有效句柄读取、删除、有效性、数量和仓库编号读取。删除保留记录，将状态改为已删除并推进版本；普通读取只接受当前有效版本。

缺口：

~~~text
没有包含已删除记录的全量导出。
没有按指定本地编号、指定版本和指定创建序号导入。
没有恢复下个节点编号和下个创建序号。
没有批量导入、冻结令牌或候选仓库构造入口。
~~~

### 4.2 主信息仓库

当前记录包含主信息编号、版本、状态、拓扑锚点和可选 I64 槽位。仓库构造时接收仓库编号，编号从 1 开始；删除保留记录并推进版本。

当前入口只面向有效记录创建、读取、删除和 I64 槽位读写。缺少仓库编号公开读取、全部状态导出、指定身份导入和高水位恢复。

### 4.3 关系仓库

当前记录包含关系编号、关系类型、来源、目标、顺序、版本和状态。失效将有效关系改为已失效并推进版本；删除将有效关系改为已删除并推进版本；重挂保留关系编号并推进版本。

当前普通读取排除失效 / 删除关系和无效端点。审计读取可接受有效或失效句柄，但仍要求当前端点有效，因此端点删除后历史关系可能无法通过现有入口完整审计。

缺口：

~~~text
没有包含全部状态记录的无损导出。
没有按指定关系编号和版本导入。
没有恢复仓库编号和下个关系编号。
没有区分有效关系端点校验与历史墓碑端点校验的恢复入口。
~~~

### 4.4 索引仓库

当前索引是 uint64 键到节点句柄的单值映射，没有仓库编号、命名空间或格式版本。读取时会排除失效节点，但失效映射仍留在表内；后续绑定看到键已存在时会拒绝新绑定，因此该旧映射仍承担键保留语义。

由此得到：

~~~text
索引命中不能裁决领域事实，恢复后必须重读节点和所有者结构。
当前也不能把索引全部丢弃，因为部分稳定键与键保留事实没有其它可枚举来源。
恢复设计必须先列出键所有者、命名空间和可复用策略，再决定保存、重建或释放。
~~~

### 4.5 写入事务与仓库快照服务

写入事务当前只保存本地事务状态和失败说明，没有锁、写集、补偿、回滚、持久化或领域服务接线，不能作为一致快照边界。

仓库快照服务当前接收布尔式请求材料和格式版本 1，只做只读准入判断；它不解析、不保存四仓库记录，也始终不允许替换运行期仓库。调用方声明的完整性检查不等于服务实际完成的跨仓库复核。

## 5. 服务侧状态与活动发布

### 5.1 特征值与特征

特征值服务使用同一原始值锁保护：

~~~text
主信息仓库中的 I64 值
I64版本记录表_ 中的 I64 原始版本
Vec记录表_ 中的 VecI64 / VecU64 完整值和容器版本
~~~

只恢复四仓库时，I64 数值可能仍在，但版本侧表为空；Vec 数值和版本全部消失。特征服务的结构化序列化候选要求目标身份和现有材料可读，它是值式候选与复核边界，不是向空白隔离上下文导入全部值的接口。

特征服务自身主要保存仓库 / 原子服务引用和互斥量，可在仓库与特征值侧表恢复后重建；实例槽位唯一当前值等结构仍需按正式特征规则复核。

### 5.2 概念图

概念图服务除仓库结构外，还保存：

| 成员 | 性质 | 恢复口径 |
| --- | --- | --- |
| 根登记组_ | 四类根的服务权威登记 | 由系统角色清单恢复并重读类型、稳定键和句柄 |
| 生命周期状态组_ | 活跃 / 冷却 / 退役共享状态角色 | 恢复既有角色，不重新创建 |
| 专用关系登记组_ | 概念图专用关系可达登记 | 从完整关系候选重建并复核所有者 |
| 生命周期关系登记组_ | 生命周期当前关系登记 | 从完整关系和生命周期规则重建 |
| 活动快照_ | 当前已发布活动图、签名和活动版本 | 候选恢复、完整重算 / 复核后一次发布 |
| 上一候选图_ | 固定点计算缓存 | 丢弃 |

活动版本若重启后归 1，会让旧候选与新会话候选发生 ABA 式误接受。首版恢复必须保存单调活动版本，或引入持久化纪元并把纪元纳入候选匹配。

概念签名当前默认格式版本为 1，但完整性判断只要求非零版本；恢复解析必须显式拒绝未知版本，不能把任意非零值当作支持格式。

### 5.3 其它领域服务

语素、需求、任务、方法、存在、场景、状态、动态、二次特征、因果和世界服务当前大多只持仓库 / 原子服务引用，领域事实主要落在四仓库。但这不意味着它们永远无服务侧状态：

~~~text
#203-#208 将增加方法登记、规格和派生索引。
#221-#225 将增加任务当前态、版本、选择、执行请求和结果治理边界。
#216-#220 将增加结构事务和物理删除边界。
~~~

恢复详细设计必须以这些计划实际完成后的接口为准重新扫描；当前计划文本只能作为依赖，不能写成已实现事实。

## 6. 初始化与启动断点

### 6.1 当前初始化行为

当前初始化均面向空仓库：

| 初始化 | 当前行为 | 恢复后重跑风险 |
| --- | --- | --- |
| 语素初始化 | 创建固定语素入口及对应信息结构 | 重复语素与绑定；多数角色没有可恢复稳定键 |
| 世界树初始化 | 创建世界根、场景、自我存在和关系 | 重复世界、自我和场景 |
| 需求初始化 | 创建目标特征 / 槽位 / 值、两个根需求并绑定稳定键 | 既有键导致本次绑定失败；重新创建导致重复根需求 |
| 概念图初始化 | 创建 / 登记四根、名称入口、生命周期状态和支持 | 服务登记为空时可能重复根；既有名称 / 语素结构也可能重复 |
| 自我线程初始化 | 按语素、世界树、两个根需求顺序执行，并发布会话初始化快照 | 没有恢复模式或外部注入的系统角色清单 |

必须新增系统初始化角色清单，至少承载：

~~~text
语素初始化角色与对应信息句柄
世界根、主场景和自我存在
两个根需求及其目标特征、槽位、值、目标状态与稳定键
概念四根、名称入口和生命周期共享状态
必要的仓库编号、格式版本和角色清单版本
~~~

该清单只定位已有权威结构；恢复仍须逐项重读类型、关系和版本，不能凭清单覆盖仓库事实。

### 6.2 当前启动顺序

入口当前大致执行：

~~~text
解析参数并启动日志
-> 在当前栈上构造四仓库与全部服务
-> 启动自我线程并新建语素、世界树和两个根需求
-> 执行概念图新建初始化
-> 运行大量入口自检并继续创建样例结构
-> SQL 数量审计
-> 控制面板或无界面等待
~~~

运行宿主、任务管理、任务工作、缓存统计、事件日志和外设采样线程当前主要在入口自检中被启动 / 停止；正式常驻链仍只有自我线程。恢复设计不得把测试对象生命周期误写成生产启动事实。

### 6.3 目标启动顺序

~~~text
读取明确启动策略：全新初始化或从指定快照恢复
-> 只读加载并严格校验快照头、段表、版本、长度和校验
-> 构建隔离运行期上下文候选
-> 恢复四仓库、侧表、系统角色和服务登记
-> 重建并复核活动图、派生索引和非权威缓存
-> 跨段完整性屏障
-> 一次发布完整运行期上下文
-> 启动自我治理、任务、缓存、事件和外设线程
-> 执行 SQL 审计
-> 启动控制面板
~~~

存在快照但快照无效时，不得静默回退到全新初始化。只有明确全新模式且确认没有要恢复的快照时，才允许执行当前新建初始化。

## 9. 删除、失效与防复活矩阵

| --- | --- | --- | --- |

## 10. 候选读取、并行构建与一次发布

建议的第一轮流程：

1. 调用方明确选择全新模式或恢复模式；不根据文件是否能读成功自动猜测。
2. 快照生成取得统一结构冻结 / 事务许可。当前该能力尚未实现，须等待 #217 实际接口并复核服务侧表是否进入同一冻结域。
3. 在冻结域内快速复制不可变值式材料：四仓库全部记录与高水位、特征值侧表、系统角色清单、概念登记和活动图证据。
4. 释放冻结后按稳定身份规范排序。各独立段可并行编码和校验，但最终段表、清单顺序和摘要必须确定。
5. 恢复先严格校验总头、版本、边界、长度、段唯一性和校验值；通过边界校验后，独立段可并行解析为值式候选。
6. 在隔离上下文按依赖顺序导入：主信息 -> 节点 -> 关系 -> 索引候选 -> 特征值侧表 -> 系统角色 / 服务登记 -> 活动图和派生缓存。
7. 进入完整性屏障，复核有效引用、墓碑不可见、稳定键唯一、特征身份 / 类型 / 版本、世界与两个根需求、概念四根 / 生命周期 / 签名 / 活动版本。
8. 由一个拥有四仓库、服务和线程生命周期的运行期上下文聚合对象一次发布；不得逐仓库或逐服务替换当前对象。
9. 冷启动候选失败时不发布任何上下文；在线替换候选失败时旧上下文保持可用。
10. 发布后才创建和启动线程，再做 SQL 审计和控制面板投影。

### 10.1 并行原则

适合并行：

~~~text
冻结复制后的独立段规范排序、编码和校验。
总头与边界已验证后的独立段解析。
不共享写目标的派生缓存和显示材料重建。
独立段只读一致性检查，结果在最终屏障汇合。
~~~

必须串行或有屏障：

~~~text
取得和释放统一冻结许可。
高水位与记录集合的同一时点复制。
跨段身份和引用完整性裁决。
系统角色与服务登记发布。
运行期上下文唯一发布。
业务线程启动。
~~~

多线程只提高候选处理吞吐，不改变权威顺序，也不能让哪个线程先完成决定记录顺序、候选选择或活动版本。

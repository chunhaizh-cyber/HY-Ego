## 3. 核心处理与结构承载

1. 数据操作模块新增唯一拥有的 `需求正式结算材料`，完整承载结算实例状态、需求归属、来源任务、目标状态、实际状态、动态证据和可选因果引用的值式证据；`需求权威材料.正式结算` 成为正式读回，旧 `当前结算 / 最近结算状态` 只由它兼容投影。
2. `需求结算写入规格` 只接受结论与状态规格值均为 `已满足`；结论 / 规格混合值不能形成写入规格。可选因果句柄只能全零或完整有效，部分句柄在第一笔写入前拒绝。动态与因果均由上游创建，需求服务只复核，需求数据操作只绑定。
3. 需求读取按状态型归属零 / 一 / 多条分支：零条未结算；一条必须完整重建实例状态和固定角色 1—5；多条、未满足值、未知 / 重复 / 错端点角色、错误顺序号或任务来源矛盾均返回内部不一致，不再按时间或关系编号选择“最近记录”。
4. 已有唯一结算按结算状态主键、场景、主体、时间、值、来源任务、目标状态、实际状态、动态和可选因果逐字段比较；完整同义才幂等读回，任一异义均幂等冲突且零写入。
5. 来源任务读回后、进入不透明结构写入会话前，先拒绝结算状态规格的场景或主体不等于任务场景 / 虚拟存在。首次写入在会话内再次复核状态型归属唯一性，写入实例状态、需求归属和角色 1—5 后，从会话完整重建并逐字段读回；只有完整匹配才请求提交。并发占用退出会话后重读，同义返回幂等读回、异义返回幂等冲突；许可拒绝、版本漂移和内部不一致保持分层。
6. 需求服务在形成结算状态规格前复用 `判定目标状态达成`；值不相等只拒绝本次独立结算，已完成任务保持原状；正式结论固定为 `已满足`。
7. 既有通用 `完成任务并独立结算` 在完成任务前读取任务、来源需求并做目标状态达成裁决；值不相等时不调用任务完成或需求结算，两类结构零变化。#225 新任务结果治理路径未修改，继续把未达成迁移为 `待重筹办`。
8. 数据操作层新增只读 `复核需求结算证据`，只复核动态 / 可选因果句柄的当前可读身份和精确节点类型；需求服务以 `复核独立结算证据` 封装该下层只读门禁，既有通用组合入口在任务完成前调用。完整有效但类型错误的动态或因果均明确入口拒绝，任务保持执行中、无实际结果，需求保持未结算；错动态、错因果各自的节点 / 关系 / 索引公开有效数量不变。
9. 运行期业务操作公开转发文件和签名零修改，仍真实转发到收口后的需求任务方法组合入口；新增下层只读复核不向运行期外观增加转发，不形成新写入口。

## 3. 实际接口漂移与裁决

### 3.1 删除不是物理移除

当前 `删除关系` 与 `严格删除关系` 都把关系记录状态推进为 `已删除` 并递增版本，`关系表_` 不执行 `erase`。原流程图、详细设计和 #258 的“物理删除并移除关系表记录”不符合当前代码。

裁决：S1 保留已删除权威记录，只从有效正向、有效反向、有效节点相关和当前审计正向索引移除该关系编号。

### 3.2 删除许可有两条现行路径

旧 `删除关系(..., 令牌)` 通过 `关系令牌入口` 验证共享路径令牌；新 `严格删除关系(..., 令牌)` 验证独占令牌。原文“普通删除继续要求独占”会扩大当前事实。

裁决：S1 不改变任何签名、许可强度或事务 ABI；旧共享兼容删除和严格独占删除均保持原义。

### 3.3 全部关系表写路径

S1 索引维护必须覆盖：

```text
创建关系 / 结构化创建关系
失效关系 / 结构化失效已发布关系
重挂关系 / 重挂节点 / 挂载或重挂节点
结构化重挂已发布关系 / 结构化挂载或重挂节点
删除关系 / 严格删除关系
结构化确认已发布关系变更 / 结构化撤销已发布关系变更
提交节点关系失效包
```

确认入口只复核写后记录和索引后确认能力；撤销入口必须恢复写前记录与索引；批量失效必须先整批复核，再同锁移除有效索引并保留当前失效审计候选。任何路径不得绕过索引维护。

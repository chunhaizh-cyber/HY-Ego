## 12. 第三次具名接口漂移：同名根类别无法在装配 IFC 共存

v0.3 同时固定：

```text
概念活动状态.数据.h 必须复用 概念图算法.h 唯一拥有的 概念根类别 / 概念签名材料。
数据操作.概念活动 不得 import 数据操作.概念图结构。
运行期业务装配必须同时保留既有概念图结构对象并接入新概念活动服务。
```

当前真实接口却存在两个不同实体、同一未限定名称：

| 所有者 | 实体 | 底层类型 | 到达装配的路径 |
| --- | --- | --- | --- |
| `领域/概念图算法.h` | `概念根类别` | `std::uint32_t` | 新 `概念活动状态.数据.h` -> 新数据操作 / 服务 |
| `领域/数据操作.概念图结构.ixx:28` | `概念根类别` | `std::uint8_t` | 既有 `运行期业务装配` 直接 import 旧概念图结构数据操作 / 服务 |

新数据操作本身没有 import 旧概念图结构模块，S1 / S2 也能独立编译；但 `运行期业务装配` 是两条生产依赖第一次汇合的位置。直接让装配看见新活动数据类型时，MSVC 报：

```text
error C1117: 导入模块“海中鱼巣.领域.数据操作.概念图结构”时出现无法恢复的错误：
已定义符号“概念根类别”。
```

尝试仅用模板 / 推导返回类型避免装配重复 `#include` 后，两个不同宽度的同名实体仍经 IFC 同时到达，MSVC 在 `概念活动材料::完整()` 的规范根类别数组处报：

```text
error C4789: 缓冲区“预期类别”(大小为 8 字节)将溢出；将写入 12 字节。
```

该错误说明问题不是缺一个 include，也不能通过隐藏公开签名文本解决：只要装配同时拥有旧概念结构服务和包含规范签名的活动服务，两个同名不同宽度实体就会进入同一模块依赖图。继续实现会迫使执行角色修改 #296 当前禁止的旧概念结构模块，或放弃 `概念图算法.h` 的唯一签名所有权；两者都命中退回条件。

## 2. 当前事实

1. 现有仓库快照服务只是拒绝壳，没有四仓库全状态导出、字节格式、原子文件发布、严格解析、指定身份导入或运行期发布。
2. 完整恢复必须覆盖四仓库全部有效 / 失效 / 删除记录、仓库编号、高水位和节点创建序号，不能只恢复普通可见记录。
3. I64 原始版本、VecI64 / VecU64 完整值与版本、系统角色、领域登记、概念活动候选和活动版本存在四仓库之外的权威状态。
4. 索引不裁决业务事实，但首版必须保存带唯一所有者的候选，再由所有者从权威结构重算复核。
5. HYEVSEG1、缓存、统计、线程、消息、回执、控制面板、SQL 和日志均不裁决恢复。
6. 旧仓库快照详细设计只覆盖四仓库且把所有历史关系端点都要求为当前有效，已不足以限定本批代码范围；本次专用设计取代其恢复实施口径，旧文档继续作为历史材料。

## 3. 默认裁决

### 3.1 格式与冻结

```text
魔数：HYSNAP01
字节序：显式小端
总头版本：1
段表版本：1
快照格式版本：1
必需段：9 个，缺一拒绝，不使用不透明私有服务字节块
损坏检测：逐段 FNV-1a 64 + 跨段摘要，不宣称真实性认证
```

快照请求取得 #217 同域独占许可；全部权威写入口必须参与同一事务域。冻结内只做快速值式复制和有界一致性检查，逐个所有者取锁，不同时持有多个仓库锁；概念图双锁继续固定 `活动图锁_ -> 图写锁_`。释放冻结后才排序、并行编码、校验和文件 I/O。

### 3.2 九段范围

```text
1. 元数据、高水位和版本
2. 主信息记录
3. 节点记录
4. 关系记录
5. 索引候选与所有者
6. 特征原始值
7. 系统角色
8. 领域登记
9. 概念活动候选
```

#226 执行前必须复核 #217、#220、#203-#225 的实际接口并逐字段冻结段 8。出现新的服务侧权威状态时先修订显式格式，禁止复制私有对象内存。

### 3.3 文件发布

正式文件固定在配置的项目 `快照` 根目录，首版正式名为 `权威状态.hysnap`。发布顺序为：

```text
CREATE_NEW 临时文件
-> FlushFileBuffers
-> 当前解析器严格读回
-> MoveFileExW(REPLACE_EXISTING | WRITE_THROUGH)
-> 正式文件最终严格读回
```

提交点前允许的文件失败保持旧快照；当前写入方自读失败属于追根因。运行期快照文件必须保持 git 排除。

### 3.4 隔离恢复与发布

恢复先严格验证总边界，再允许独立段并行解析。四仓库候选按快照指定身份、高水位、墓碑和节点创建序号构造，禁止调用普通业务创建入口猜身份。有效关系要求当前有效端点；失效 / 删除关系可引用同身份墓碑旧版本，但拒绝缺失或未来身份。

侧表、角色、索引和领域登记在隔离上下文中恢复；概念活动图从权威结构确定性重算并与快照候选比较，活动版本精确保留，固定点 `上一候选图_` 清空。

启动策略必须显式为全新或恢复。恢复拒绝不回退全新；全新模式发现正式快照时拒绝且不删除。恢复中排队任务转待重筹办，执行中转等待中，其余状态按正式矩阵保持；线程、队列、消息和回执不恢复。完整候选通过跨段屏障后只允许空宿主首次移动发布，线程随后按固定顺序启动，SQL 审计与控制面板最后读取。

### 3.5 完成边界

首版不包含事件重放、在线热替换、跨版本迁移、进程强杀 / 断电 / 扇区损坏验证、旧任务原地续跑或外部动作恰好一次。

## 6. 阻断矛盾：结算材料不完整的唯一收口

### 6.1 专项规范口径

`规范/因果用途观察与方法学习无环接线规范.md:195` 写明：

```text
待结算推进但正式结算材料不完整 -> 同账冲突。
```

这属于设计内逻辑冲突，要求零结构变化并正常返回冲突结果。

### 6.2 流程图与 O1 详细设计口径

流程图 `流程图/20260717_权威用途观察最小业务合同流程图_v0.1.md:55` 把候选阶段元组非法归为：

```text

```

O1 详细设计进一步固定：

```text
规范/详细设计/权威用途观察最小业务合同详细设计.md:217
成功事件阶段元组非法 -> 追根因；既有账项非法推进 -> 逻辑冲突。

规范/详细设计/权威用途观察最小业务合同详细设计.md:288
已接受成功事件中结算互相矛盾或阶段元组非法 -> 追根因。
```

### 6.3 为什么不能在执行角色中解释消歧

最终阶段的正式结算材料是候选自身合法性的组成部分。若新值来自已经接受的成功事件但正式结算材料不完整，则它既可以按专项规范被解释为“待结算推进冲突”，又可以按流程图 / O1 详细设计被解释为“成功候选阶段非法、内部不一致”。同一输入因此存在两种互斥收口。

#288 的 A07 要求幂等、推进、冲突、不支持和内部不一致互斥；A10 要求所有中途非成功返回二分唯一。执行角色不得自行决定哪份设计优先，也不得修改被审计规范、流程图或详细设计，因此只能判审计不通过并退回。

### 6.4 设计修订所需的最小默认裁决

退回设计时建议按既有流程图与 O1 详细设计的分层收束，但须由设计角色正式写入三件套：

```text
候选自身不合法：
已接受成功事件后，最终阶段缺正式结算材料或材料矛盾
-> 内部不一致并追根因，不进入同账比较。

候选双方各自合法后：
身份 / 核心相同，但终态变化、逆向或其它不允许的阶段转换
-> 同账逻辑冲突，零结构变化。
```

至少须修订专项规范 6.5 和非成功返回表，并复核流程图、O1 详细设计、广域详细设计与 #288 对该边界逐字一致；不得在执行记录中把建议当作已经生效的设计事实。

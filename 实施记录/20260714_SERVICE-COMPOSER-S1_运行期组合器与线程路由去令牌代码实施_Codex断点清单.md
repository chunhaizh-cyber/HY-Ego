## 2. 输入入口与公开结果

线程侧新增封闭 `运行期业务请求` / `运行期业务结果` 变体和同步 `运行期业务请求路由::路由`。第一轮固定八类请求：已有场景创建并接纳实际存在、创建状态并记录动态、创建槽位并发布初始状态、创建语素概念入口、发布概念结构、创建需求并承接任务、召回并提交唯一选择、完成任务并独立结算。

路由返回顶层 `已分派 / 请求类型拒绝 / 请求载荷拒绝 / 业务逻辑内返回 / 内部不一致` 状态，并保留一一对应的领域值式结果。公开请求、结果和路由签名不包含原始结构事务令牌、许可、会话、仓库、执行器或锁。

## 3. 核心处理与结构承载

唯一运行期业务装配按固定顺序拥有：一个内部特征值服务、七个数据操作、十四个业务服务、六个既有组合器和一个顶层运行期业务操作组合器。七个数据操作引用同一接线和同一四仓库；需求任务方法数据操作复用同装配内的存在场景和状态动态数据操作；方法召回服务复用同装配内的需求、任务、方法、特征和状态服务。

调用路径固定为：

```text
线程值式请求
-> 运行期业务请求路由
-> 运行期业务操作组合器
-> 六个既有组合器之一
-> 无令牌业务服务 / 服务专用数据操作
-> 结构写入执行器 / 不透明会话
-> 四仓库
-> 值式领域结果和顶层路由状态。
```

`运行期上下文` 在四仓库之后拥有一个业务装配，增加服务装配完整性和候选业务操作只读入口。上下文 `完整()` 仍固定为假；服务装配已就绪但系统角色初始化材料缺失时，宿主返回 `缺少初始化完成材料`，不发布上下文、不生成租约。

## 4. 实施期根因与修复

完整自检初稿在同一新模块中既导入聚合装配 / 路由，又直接构造并调用多个业务服务。MSVC 为该自检生成了附着到数据操作模块的参数类型符号，而既有业务服务对象导出未附着参数类型符号，最终形成 22 个 `LNK2019`。生产装配最小外壳此前已经成功链接，逐对象 `dumpbin /symbols` 也证明问题只在新自检翻译单元的模块符号身份，不是服务实现缺失。

修复保持生产代码和既有服务零修改：自检只用七个数据操作建立同仓库隔离夹具，十四个业务服务和六个组合器始终由正式 `运行期业务装配` 构造；八类验证全部经候选业务操作和线程路由执行，不再从自检直接调用业务服务。随后 Debug 链接通过。

A08 初次运行只剩幂等分类失败。证据为冲突、许可拒绝和版本漂移均保持原枚举，幂等复核因放在后续概念 / 需求关系增量之后已不再处于首次创建的同一业务语境。将幂等复核前移到首次创建之后立即执行，得到 `已分派 / 幂等读回`；四类证据最终为 `1/2、4/4、4/5、4/6`。

## 5. 失败收口与兼容边界

无效变体和明显无效载荷在路由分派前拒绝，结构数量不变。下层幂等冲突、许可拒绝和版本漂移保留领域结果并映射为业务逻辑内返回，不重试、不改写请求、不回退传统服务。空路由和下层内部不一致保持内部不一致，不能伪装成成功。

旧自我治理路由、概念命名路由、四个历史初始化文件、概念生命周期、统计缓存和概念安全删除均未修改或接入。未创建线程、`std::async`、工作队列、真实动作调度或自动重试；既有生产入口未切换到候选上下文。

## 2. 需求目标状态裁决

需求业务服务新增值式 `需求目标状态达成材料` 和只读 `判定目标状态达成`。入口先重读需求权威材料，再经状态业务服务读取目标抽象状态和实际实例状态；第一轮只在两份完整状态材料的值相等时裁决已达成。

裁决结果保持值式，不返回可写引用、原始令牌、许可、会话、执行器、仓库或锁。创建、父子、结算和既有写入语义均未修改。

## 3. 任务结果组合器与线程路由

新增 `组合.任务结果结算.ixx`。组合器只持需求、任务、方法、状态和动态业务服务引用，按以下顺序重读并复核：

```text
新版回执及完整来源冻结请求
-> 任务、需求、当前选择、方法登记和动作角色关系
-> 实际实例状态与实例动态
-> 动态目标、前后状态、来源动作、场景和主体
-> 需求业务服务目标达成裁决
```

目标未达成时只调用任务业务服务迁移到待重筹办，不绑定实际结果、不完成任务、不结算需求。目标达成时复用任务业务服务现有“实际结果 + 已完成生命周期”原子入口，完成读回完整后再调用需求业务服务独立结算。结算暂不可执行时任务保持已完成，只返回可丢弃、非权威且不落仓的待结算提示。

新增 `路由.任务结果治理.ixx`。路由只持组合器窄引用；显式隔离入口和工作线程值式出队入口都先做新版协议准入，再在队列锁外调用组合器。旧兼容回执被协议写前拒绝，不从消息文本重建业务材料。

## 4. 幂等、冲突和并发

精确重复回执复用任务完成和需求结算幂等读回；已完成任务收到不同实际结果时返回正常当前性漂移，不覆盖权威结果、不回退生命周期。失败、取消、待重筹办等非执行终态不被迟到回执复活。

隔离自检分别执行完成、未达成迁移和独立结算各 100 路竞争。三类路径均只有一个真实提交；完成和结算其余结果为精确幂等，未达成其余结果为精确幂等或正常当前性漂移。

## 5. 实施期根因复核

新自检最初调用了项目不存在的 `汇总自检结果` 辅助，已改为现有中央自检模块采用的手工逐项输出和 `自检单元结果` 汇总模式。

三组各 100 个大型值式治理结果最初使用栈上 `std::array`，Debug 运行以 Windows `0xC00000FD` 栈溢出退出。保持三类 100 路验收不变，改为 `std::vector` 在堆上承载后收口。

A12 最初错误要求冲突回执返回内部不一致；详细设计明确“已完成任务收到不同实际结果”为逻辑内当前性漂移。自检改为验证当前性漂移、拒绝原因、结构数量不变和既有结果不被覆盖。以上三处均只修正自检承载或断言，生产准入未放宽。

# 函数迁移前置专项

编号：HR-001-FM-PRE
日期：2026-07-06
状态：v0.2 / 已确认 / 服务级确认规则已修订 / 未改代码 / 未迁移旧函数

## 1. 任务目标

本专项是下一轮旧能力迁移的函数级前置门禁。它不迁移旧函数体，不新增旧主信息字段，不接控制面板、SQL、D455 或体素，只建立旧函数进入新项目之前必须补齐的说明、判定和验收规则。

目标如下：

```text
1. 为每个候选旧函数建立事实登记模板。
2. 明确旧函数的输入、输出、调用方、写结构、读结构、副作用和机器事实边界。
3. 区分旧函数应保留、改名、拆分、合并、删除或改写为领域服务入口。
4. 要求每个候选函数先说明保留行为、删除行为和新增约束，再归入服务逻辑包；旧函数不再逐个进入用户确认。
5. 将旧函数迁移前置到规范、详细设计、计划和领域服务授权写入门禁之后。
6. 将旧函数只读扫描定位为证据采集，最终迁移确认必须按服务逻辑包、服务专项计划和服务代码实施切片进行。
```

本专项输出的是迁移前服务确认材料，不是代码实施计划。旧函数编号不得直接等同于迁移单位；旧函数事实完整登记后直接汇入服务逻辑包，不再逐个等待用户确认。只有服务逻辑包、服务专项计划或服务代码实施切片确认后，才允许进入对应执行路线。

## 6. 旧函数功能说明模板

每个候选旧函数必须按以下模板登记。函数事实登记不需要逐个等待用户确认；没有完成事实登记并归入服务逻辑包的旧函数，不得作为服务迁移证据进入代码实施。

| 字段 | 填写要求 |
| --- | --- |
| 旧函数编号 | 使用稳定编号，例如 `OF-001`。 |
| 旧项目路径 | 写旧函数所在文件路径；路径未确认时标为 `待确认`。 |
| 旧函数名 | 写旧函数原名，不因命名不合规而提前改名。 |
| 归属逻辑包 | 写候选逻辑包编号和名称；未归并前写 `待归并`。 |
| 当前调用方 | 列出现有调用方和触发路径；未扫描到调用方则写 `未命中调用方`。 |
| 输入 | 写入参、外部依赖、全局状态、线程材料、外设材料。 |
| 输出 | 写返回值、输出参数、结构写入、日志 / 显示输出。 |
| 读结构 | 写读取的节点、主信息、关系、索引、状态、动态、文件、数据库或缓存。 |
| 写结构 | 写修改的结构；无结构写入时明确 `只读`。 |
| 是否写机器事实 | 只能填 `是 / 否 / 待确认`，并给证据。 |
| 是否依赖日志 / 显示 / SQL / 线程 / 外设 | 逐项说明，不得合并成一句。 |
| 保留行为 | 写必须保留的业务含义，不复制旧代码表达。 |
| 删除行为 | 写旧项目中不进入新项目的行为。 |
| 新增约束 | 写入口拒绝、领域服务授权、状态 / 动态时间戳、特征服务封口等新约束。 |
| 目标新结构 | 写节点、主信息、关系、索引、状态、动态、因果引用、缓存或 `无目标结构`。 |

## 7. 函数事实初步判定表

本表只服务逐函数证据分流。函数事实可以给出初步判定，但不得直接触发代码实施；最终迁移确认必须落到逻辑迁移包。

| 判定 | 适用条件 | 必须证据 | 禁止捷径 | 输出材料 |
| --- | --- | --- | --- | --- |
| 保留 | 函数语义已符合当前规则，且不越权写结构。 | 当前项目已有等价入口或旧函数行为可无损映射到已确认结构。 | 禁止保留旧命名、旧字段或旧调用链来绕过中文命名和领域服务授权。 | 保留行为说明、目标入口、验收方式。 |
| 改名 | 函数语义可保留，但名称不符合中文业务命名或新项目术语。 | 旧名、建议新名、命名依据和调用方影响。 | 禁止只改名不补入口拒绝和结构承载。 | 改名映射表、调用方迁移计划。 |
| 拆分 | 旧函数同时承担多个责任，例如扫描、判断、写入、显示或日志混在一起。 | 职责拆分证据、每个新入口的写入方和读取方。 | 禁止把拆分后的显示 / 日志入口继续参与机器事实裁决。 | 拆分后的函数清单、各自所属服务和验证方式。 |
| 合并 | 多个旧函数只是重复包装同一结构入口，且新项目已有统一服务入口。 | 重复语义证据、保留的单一入口和弃用入口清单。 | 禁止为兼容旧调用方保留多个并行写入口。 | 合并目标入口、废弃函数说明。 |
| 删除 | 旧函数只服务旧显示、旧日志、旧 SQL 投影、旧控制面板、旧临时兼容或已被新结构替代。 | 旧函数无新结构承载价值，或其事实来源违反当前规则。 | 禁止把删除函数的返回值、日志或显示文本转写为新机器事实。 | 删除理由、替代入口或无替代说明。 |
| 改写为领域服务入口 | 旧函数表达业务写入或机器事实，但旧实现绕过新项目仓库 / 领域服务边界。 | 业务事实类型、目标结构、授权服务、入口拒绝和验收证据。 | 禁止直接迁移旧函数体，禁止外部调用方裸写仓库或特征值服务。 | 新领域服务入口设计、允许文件、禁止文件、验证剧本。 |

判定顺序：

```text
先判断是否应删除。
再判断是否只是显示 / 日志 / SQL / 控制面板辅助。
再判断是否违反领域服务授权写入。
再判断是否需要拆分或合并。
最后才决定保留、改名或改写为领域服务入口。
```

## 8. 逻辑迁移优先规则

逐函数扫描只作为旧系统事实采集方式。

旧函数编号不得直接等同于迁移单位。迁移确认必须按服务逻辑包进行，代码实施单位必须是新项目结构闭环和领域服务入口，而不是旧函数体。

本专项采用以下分层：

```text
函数 = 证据采集单位。
服务逻辑包 = 迁移确认单位。
领域服务 / 结构闭环 = 代码实施单位。
```

每批函数扫描后必须生成服务逻辑包归并表，归并表至少包含：

```text
逻辑包编号
逻辑能力名称
包含旧函数
旧系统行为摘要
新项目结构承载
目标领域服务
保留行为
删除行为
新增约束
需要补的规范 / 详细设计
验收方式
待确认问题
```

逐函数登记模板中的 `归属逻辑包` 只表示证据路由，不表示最终迁移许可。函数事实本身不逐个确认；未经服务逻辑包、服务专项计划或服务代码实施切片确认，不得进入函数体迁移、字段落地或代码实施。

推荐流程：

```text
逐函数只读扫描
-> 旧函数事实表
-> 服务逻辑包归并表
-> 新项目结构映射表
-> 待确认服务迁移包
-> 确认后补规范 / 详细设计 / 计划或直接生成服务代码实施切片
-> 独立代码实施切片
```

## 8.1 服务级确认规则

为提高迁移效率，从本修订起，旧能力迁移不再逐个函数确认。

确认单位调整为：

```text
服务逻辑包。
服务增强专项计划。
服务代码实施切片。
```

函数事实登记仍必须保留，但只作为服务包证据。一个服务确认可以覆盖多个旧函数、多个旧调用路径和多个旧副作用，只要服务包已经写明：

```text
包含旧函数。
旧系统行为摘要。
新项目结构承载。
目标领域服务。
保留行为。
删除行为。
新增约束。
允许文件。
禁止文件。
验证方式。
禁止扩大声明。
```

服务确认后的推进规则：

```text
服务专项计划确认后，登记对应 S0 当前代码事实扫描到可执行队列。
S0 完成后，如边界清楚，可直接生成该服务的 S1-Sn 代码实施切片。
代码实施切片确认后，由执行窗口按允许文件、禁止文件和验证方式实施。
执行中发现服务包之外的新函数事实、新字段、新结构或新依赖时，登记到待确认问题，不逐函数中断已确认服务切片。
```

仍然禁止：

```text
旧函数体直接复制。
旧函数编号直接等同迁移完成。
旧主信息字段绕过服务包回流。
未确认服务切片就修改 C++。
未写允许文件、禁止文件和验证方式就进入代码实施。
```

## 9. 验收标准

本专项验收只覆盖文档门禁：

```text
1. 已生成函数迁移前置专项文档。
2. 已同步计划索引和项目记忆。
3. 已建立断点清单。
4. 文档中明确只读扫描范围、允许修改文件、禁止修改文件。
5. 文档中包含旧函数功能说明模板。
6. 文档中包含保留 / 改名 / 拆分 / 合并 / 删除 / 改写为领域服务入口判定表。
7. 文档明确不直接迁移旧函数体。
8. 文档明确不新增旧主信息字段。
9. 文档明确不声明旧项目能力迁移完成。
10. 文档明确不接控制面板 / SQL / D455 / 体素。
11. `git diff --check` 通过。
```

后续扫描和归并验收至少必须覆盖：

```text
旧函数模板完整。
每条函数事实都有归属服务逻辑包或 `待归并` 标记。
对应规范和详细设计存在。
服务逻辑包明确目标结构和目标领域服务入口。
保留行为、删除行为和新增约束已在服务包层确认。
禁止修改文件未被触碰。
拒绝路径前后结构数量不变化。
代码实施切片独立建立并通过对应构建 / 入口 / 结构验收。
```

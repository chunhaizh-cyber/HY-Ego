## 4. 实施步骤

### S0：实际接口复核

读取 #228 与 #203-#225 的正式实施记录，逐项确认候选所有权、任务当前态枚举、合法迁移入口、结构化证据和线程构造签名。任一实际接口与本计划假定不一致时不得修改代码，退回设计窗口修订 #229。

### S1：运行期上下文与宿主

新增 `运行期上下文`，按依赖顺序唯一拥有四仓库、事务协调器、领域服务和系统角色。新增 `运行期上下文宿主`，只允许空宿主首次移动发布完整候选；已有上下文逻辑拒绝并保持原上下文。调用方只能取得带上下文身份的共享租约，不缓存候选内部裸指针跨越发布。

### S2：显式启动策略

建立 `全新初始化` 与 `从指定快照恢复` 两种显式策略。全新模式要求请求明确无恢复目标且固定快照不存在，不删除、重命名或覆盖快照；恢复模式任何拒绝都终止启动，不调用全新初始化兜底。

### S3：任务恢复中断收口

在线程和队列尚未构造时读取隔离任务当前态：排队中以结构化恢复中断证据转待重筹办，执行中转等待中；其它非终态保持，完成 / 失败 / 取消保持终态。证据绑定快照生成序号、冻结事务序号和原生命周期版本。不得恢复旧执行请求、消息或回执，不自动继续或重试未知外部副作用。

### S4：跨段完整性屏障

串行复核四仓库全状态、高水位、侧表、系统角色、索引所有者、领域登记、概念活动图、任务中断收口和瞬态队列为空。屏障只返回值式裁决，不写第二份业务事实；外部候选不满足规则逻辑拒绝，当前已验证候选出现内部矛盾追根因。

### S5：首次发布与启动顺序

屏障通过后一次移动发布。发布后重新取得租约并验证仓库、服务和角色均来自同一上下文身份，然后按顺序启动自我线程、任务管理 / 工作线程、缓存统计 / 事件 / 外设线程；SQL 审计和控制面板只在最后读取。出现混合上下文或身份漂移时追根因并停止后续线程启动。

### S6：自检与治理

补齐 RECOVERY-S3-A01 至 A16，完成规范、实施记录、计划归档、治理、提交和推送。

## 8. 失败收口

| 场景 | 分类 | 收口 |
| --- | --- | --- |
| 前置未完成或实际接口漂移 | 依赖门控返回 | 不改代码，退回设计或等待前置 |
| 启动策略含混、全新模式发现快照 | 逻辑内返回 | 不初始化、不删除 / 覆盖快照 |
| 外部候选未通过任务收口或跨段屏障 | 逻辑内返回 | 丢弃整个候选，不发布、不启动线程 |
| 宿主已有上下文 | 逻辑内返回 | 保持旧上下文，不在线替换 |
| 已验证候选无法执行声明的合法迁移 | 追根因解决 | 不发布，停止依赖路径 |
| 发布后发现混合上下文或角色漂移 | 追根因解决 | 停止线程启动，不宣称恢复成功 |

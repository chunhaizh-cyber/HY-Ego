## 5. 实施步骤

### S0：#214 后实际接口复核

从 clean HEAD 逐项记录矩阵 16 个长期成员构造函数、四初始化器构造 / 调用、所有值式接线和令牌重载、入口生产构造点、#246 清理点及 670 占用。任何成员、参数、依赖方向、初始化角色或允许文件漂移时不改代码，退回设计。

### S1：固定长期服务装配

`装配.领域服务` 只接收同一结构事务接线与四仓库引用，按矩阵 01-16 声明值式成员。所有服务引用同一仓库实例；只有正式事务路径服务保存可空接线。模块不得 import / 拥有运行期上下文、仓库、宿主或租约。

### S2：全新候选初始化与角色读回

在候选未发布、宿主为空且无租约时，工厂局部构造四个初始化器，按语素、世界树、根需求、概念图顺序运行，并由方法服务初始化登记根。结果转换为值式 `运行期系统角色材料`，再通过对应服务逐项读回。局部初始化器随后销毁；自我线程不再是新生产候选的初始化权威。

### S3：回改唯一上下文与首次发布

`启动.运行期上下文` 单向 import 服务装配模块，最终成员顺序固定为协调器、接线、主信息、节点、关系、索引、领域服务装配、系统角色材料。`完整()` 同时要求结构核心、16 个服务同仓库身份和角色材料读回成立。宿主只发布完整候选，重复 / 并发发布保持首个实例。

### S4：清理分流

只改 #246 命中的四个领域 / 初始化文件：创建期补偿保留原创建所有者，候选失败时整个未发布上下文丢弃；已发布删除继续走 #219 私有编排授权；内部不一致追根因。不得把六类清理统一成普通失败或公开逐仓库删除。

### S5：自检、治理与同步

`自检.领域接线` 承载 A01-A14、失败注入和并发发布正文，阶段 670 唯一登记。入口无场景 / 循环 / 逐项输出正文。通过验证后归档、精确提交并推送。

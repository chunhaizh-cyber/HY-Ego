## 3. 核心处理

1. 在关系仓库私有结构中增加有效正向、有效反向、有效节点相关、当前审计正向四类关系编号候选索引。
2. 创建、失效、重挂、逻辑删除、结构化撤销、普通父子挂载 / 重挂和批量失效在同一关系仓库独占锁内同步更新关系表和索引；确认入口复核写后记录与索引后只确认能力。
3. 查询先从索引复制候选，再逐项复核关系表当前版本、状态、关系类型、顺序和有效端点。
4. 替换已确认的正向、反向、父子、存在、节点相关、审计、唯一性和删除准备包全表扫描；`有效关系数量` 等未冻结路径保留并登记，不临时扩大。
5. 保持确定性排序、普通父子单父和防环、其它关系的多目标 / 多父语义。
6. 扩展现有 S0 自检模块，新增阶段 800 的全表参考模型等价、逻辑删除 / 撤销 / 批量失效、故障注入、并发原子可见和同环境性能对比。Debug 专项必须读取实际索引桶候选检查数；Release 不暴露候选计数或故障控制符号。

## 4. 结构承载、输出与调用路径

关系表继续是唯一权威；二级索引只保存关系编号候选。通用索引仓库零改义。

```text
领域服务 / 仓库公开查询 -> 关系键 -> 二级索引候选
-> 关系表当前记录复核 -> 有效端点复核 -> 确定性排序 -> 值式结果
```

写路径：

```text
入口拒绝 -> 唯一性候选复核 -> 预备索引项 -> 同锁提交权威记录和索引
-> 最小读回；失败则回退本次项且不返回有效句柄
```

## 7. 失败收口与一致性

新增桶或候选分配失败时不得修改权威记录。提交中任一步失败必须逆序撤销本次新索引项并恢复写前关系记录，不返回有效关系句柄；编号可跳号但不可复用。失效、重挂、逻辑删除、撤销或批量失效后发现旧索引残项属于逻辑错误，停止后续验收。固定 Debug 故障注入必须逐点证明没有可读半索引关系，不能证明时以原编号退回设计。

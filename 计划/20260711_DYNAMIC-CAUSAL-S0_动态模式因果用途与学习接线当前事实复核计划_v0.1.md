## 5. 执行步骤

### S0：前置和提交身份复核

确认 #225 已完成，并记录 #224 / #225、#222 和 #217 的真实提交号、计划归档、实施记录和当前 HEAD。依赖未满足时保持门控，不写扫描结论。

### S1：结果、回合与动态来源链

从任务结果 / 完成 / 结算正式入口反向读取任务尝试、回执、方法、动作、实际状态、来源动态、主体、场景和版本。判断是否已有稳定回合身份及片段顺序来源；没有时只记录 #232 需要的最小非权威片段序号边界，不在本计划发明权威字段。

### S2：特征与二次特征语义

逐入口核实特征值归属、实例槽位、模板特征定义、I64 / Vec 原始材料和版本。确认零 / 一 / 多路径、访问层级与锁边界。核实二次特征角色是否是稳定语义而非读取位置。

### S3：动态、因果和用途现状

列出现有动态聚合、运动基元、因果模板 / 查询、缺失证据候选的全部调用点、输入、核心处理、返回材料和当前性。再核实统计缓存、概念用途观察和方法登记 / 规格 / 召回 / 选择的写入权力。

### S4：假定接口与后续范围复核

对 #232-#235 的每项假定接口逐项标记成立、可在允许范围补齐、需上游修订或禁止。重点证明 #232 能由生产动态模式模块在一次调用期取得一份共享许可，并把同一值式令牌下传到动态、特征、状态和二次特征只读路径；任何路径需要二次取许可、锁内跨仓库读取或绕过令牌时均判为漂移。形成后续允许 / 禁止文件、模块骨架、630 / 640 / 650 阶段占用、锁顺序和非成功二分建议；有实质漂移时 #232 保持门控并退回设计。

### S5：治理收口

生成正式断点清单，归档本计划，更新计划索引、任务队列、当前状态、实施记录、决策和待确认问题，精确提交并推送。

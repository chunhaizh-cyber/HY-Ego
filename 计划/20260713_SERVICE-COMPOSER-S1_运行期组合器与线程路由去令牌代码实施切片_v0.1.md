## 2. 目标

1. 让唯一运行期上下文候选拥有一个运行期业务装配，不再依赖旧“十六个传统服务”矩阵。
2. 让业务装配拥有七个数据操作、十四个无令牌业务服务、六个既有组合器和一个顶层运行期业务操作组合器。
3. 让线程侧通过封闭强类型变体提交八类业务请求，不暴露事务能力。
4. 证明路由只负责类型匹配与分派，业务准入、写入、撤销和读回仍归既有层级所有。
5. 保持系统角色初始化和生产上下文发布后置，不形成半初始化生产入口。

## 6. 实施步骤

### S0：实际接口复核

从 clean `main` 重读 #247、#265-#269、#275 正式模块，逐项冻结：

```text
七个数据操作构造签名和有效性入口
十四个业务服务构造依赖
六个组合器构造签名和八个操作签名
运行期上下文成员顺序和发布拒绝语义
770 阶段占用、入口自检登记方式和工程模块登记方式
```

同时扫描生产 `结构事务令牌|取得.*许可|读取令牌` 当前命中，保存前态供 A10 / A14 对比。任何漂移先退回设计。

### S1：运行期业务操作组合器

新建顶层组合器，只持六个既有组合器引用并导出八个显式同名操作。每个操作只做顶层明显参数拒绝后委托唯一既有入口；不得捕获异常后返回成功，不得把不同领域结果压成 `bool`。

### S2：运行期业务装配

按详细设计第 4.3 节固定顺序构造内部特征值服务、七个数据操作、十四个业务服务、六个组合器和顶层组合器。实现 `完整()`，只检查接线、数据操作有效性和依赖可用性；不读取或创建系统角色，不持宿主 / 租约。

装配只公开顶层业务操作组合器的候选期访问，不公开服务成员、数据操作、执行器或仓库引用。

### S3：线程侧强类型路由

建立八备选项请求变体和一一对应结果变体。路由先做变体 / 顶层参数检查，再分派一次；返回顶层路由状态和具体领域结果。下层逻辑内返回保持原状态；内部不一致保持可识别，不重试、不改写、不切换旧路由。

### S4：运行期上下文候选接线

在四仓库后构造唯一运行期业务装配。增加服务装配完整性读口和候选业务操作访问入口；`完整()` 继续要求后继系统角色初始化完成材料，本批不得返回真。宿主在服务装配已就绪但初始化未完成时返回明确拒绝并保持空。

不得改入口生产装配，不得把路由接入旧自我线程。

### S5：自检与回归

独立自检模块承载 A01-A18：

1. 装配同仓库身份和固定依赖顺序。
2. 八类请求的唯一分派、值式结果和原业务状态守恒。
3. 无效请求数量快照不变。
4. 线程路由公开面无事务能力。
5. 运行期上下文服务装配就绪但仍不可发布。
6. 旧路由、初始化、概念生命周期、统计和删除文件差异为零。

入口只在 Debug 自检路径 import、登记一次 770 并汇总。Release 不登记本自检。

### S6：治理与同步

验证全部通过后：归档本计划，登记 770 和 A01-A18，同步计划索引 / 队列 / 当前状态 / 验收 / 窗口交互，形成正式实施记录，精确提交并推送。随后只把 `#250` 推进到设计窗口重做三件套，不自动实施 690。

## 7. 非成功返回与失败收口

| 场景 | 分类 | 收口 |
| --- | --- | --- |
| S0 接口漂移 | 依赖 / 设计门控返回 | C++ 零修改，退回设计 |
| 请求类型或顶层载荷无效 | 逻辑内返回 | 分派前拒绝，结构和数量不变 |
| 下层入口拒绝、幂等冲突、许可竞争、版本漂移 | 逻辑内返回 | 保留具体领域结果；不重试 |
| 系统角色材料缺失 | 逻辑内返回 | 上下文不可发布，宿主为空 |
| 装配身份 / 依赖不一致 | 追根因解决 | 候选不可用，停止后继 |
| 请求与结果变体错配 | 追根因解决 | 不返回伪结果，停止路由 |
| 下层内部不一致或已发布后读回异常 | 追根因解决 | 沿根因路径停止，不回退旧入口 |

本计划不新增仓库写算法，因此不复制七个数据操作层的撤销逻辑。八类操作的写入失败收口继续由其正式数据操作层和执行器负责；路由不得捕获后静默继续。

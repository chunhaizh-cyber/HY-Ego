## 5. 目标合同

### 5.1 唯一消费和治理输出

```text
任务工作线程原始结果队列
-> 唯一消费者：任务结果治理路由::处理工作线程一条回执
-> 同一协议复核和同一任务结果组合器
-> 领域正式提交与任务域迁移
-> 强类型任务治理回流材料
-> 任务管理上行桥
-> 自我治理消息
```

上行桥删除任务工作线程依赖，只消费治理后材料。显式同步回执可以继续进入 `处理显式回执`，但不能形成第二套治理。

`任务治理回流材料` 至少包含：材料身份 / 规则版本、任务 / 需求 / 方法运行 / 动作 / 执行轮次、冻结和任务快照版本、逐结果唯一分类事实组、事实提交等级、目标达成 / 推进 / 残差 / 验证 / 禁止结果、任务域提交结果及权威读回、后继类型 / 当前性、幂等 / 去重键、时间和过期条件。只有后继为“完成待需求复核”时，才强制携带已发布实际状态、恰一来源动作动态和可选因果；待重筹办、合法等待和结构化缺口分别按其证据合同成立，不得伪造不存在的实际状态或动态。

### 5.2 合法后继

```text
完成待需求复核
待重筹办
合法等待
结构化缺口
```

入口拒绝和内部逻辑错误不是业务后继。内部逻辑错误停止依赖路径，不投递等待或缺口消息。

独立世界事实变化不属于任务业务后继。它使用独立 `世界事实变化触发材料`，至少包含事实稳定身份 / 版本、来源领域、状态 / 动态证据、发生时间、当前性、去重键和过期条件；不得填写虚假任务、需求、方法或执行轮次。该封套仍须经过消息身份、同义去重、异义冲突、版本和当前性门禁后，才能触发自我下一轮状态重判。

### 5.3 方法输出与世界事实

方法输出、运行场景和执行报告先留在动作运行作用域；对应实在域服务按来源、时间、范围、质量、版本和事实提交等级发布正常事实、异常事实或待复核材料。任务域只读取已发布实际结果。报告成功不替代实际回读，输出场景不直接成为世界当前场景。

### 5.4 任务验证和需求复核

任务域按 5120 对方法结果唯一归类，并使用正式特征 / 状态 / 判断收束能力区分：已达成、未达成有推进、未达成无推进、方向相反、材料不足、不可比较、版本过期、禁止结果、内部矛盾。每项四类归属必须由 `执行结果类型`、`附带结果路由状态`、`禁止结果状态`等正式任务特征及来源证据与本轮任务状态共同发布；消息 DTO 只能读回该事实组，不得重新分类。

组合器不再直接提交需求满足。完成任务只形成“完成待需求复核”；自我请求需求域按 5160 复核需求、目标合同、来源任务、实际状态、动态、可选因果和规则版本，只有明确满足才发布唯一正式结算记录。

派生需求“三件套”固定为三组复合值：

```text
目标组 = 目标对象 + 目标特征 + 目标状态 + 目标合同版本
当前与偏差组 = 当前状态 + 结构化偏差 + 事实提交等级 + 当前性
来源证据组 = 来源任务 + 来源方法 + 动作运行身份 + 来源动作动态 + 结果轮次
```

### 5.5 旧旁路封口

原始外设报告、外部材料适配入口和未治理任务回执不得进入自我邮箱。固定安全根兼容路由不得自动创建方法首、动作入口或方法选择；保留自检时必须预先提供由任务 / 方法领域形成的正式材料。

## 6. 机械阶段

### S0 依赖、身份、唯一消费和调用图复核

只读确认 #330 已进入 main、签名与本计划一致；扫描全部 `取出一条任务结果回执`、结果治理、需求结算、自我邮箱、独立世界事实变化和自动方法首调用点。返回 PASS / DRIFT，PASS 经设计接受后才写入。

### S1 收束原始回执唯一消费权

保留治理路由唯一出队；上行桥改为消费治理后材料；显式入口与工作线程入口复用同一协议和组合器。

### S2 形成任务治理回流材料

收束强类型材料、正式任务结果归类事实组、身份 / 幂等 / 当前性 / 过期合同和四类任务业务后继；独立世界事实变化使用不同封套。同义重复读回，异义冲突。

### S3 分层提交世界事实和任务状态

先由对应领域服务提交事实，再读取 #330 完整任务特征快照并裁决目标；使用 #330 入口提交结果归类事实组、完成、待重筹办、等待或结构化缺口。不得直接写需求。

### S4 建立上行封套和需求复核

上行桥形成非权威值式消息；任务回流和独立世界事实变化分别经过封套门禁；自我只请求目标领域；需求域使用正式类型 / 值域 / 版本 / 三态判断，明确满足时才形成唯一正式结算。

### S5 封闭旧旁路

删除或拒绝外部材料直达自我的生产能力；固定安全根自检改为消费预先形成的正式材料，不自动创建方法首。

### S6 回归、验证和任务分支收口

新增 / 修订既有自检，运行第 8 节全部验证；只提交允许文件和专属记录，推送同名任务分支，状态最多为 `分支完成待集成`。

## 7. 验收矩阵

1. 全仓只有治理路由调用原始任务结果出队；上行桥没有任务工作线程依赖。
2. 同一回执只产生一个任务治理结果和一个同义可重放的上行材料；同键异义冲突。
3. 任务域先完成事实 / 结果 / 状态治理，再产生需求复核材料；组合器不直接提交需求满足。
4. 需求比较覆盖相等、不等、推进、材料不足、不可比较、过期和内部矛盾，不比较裸 I64。
5. 完成、待重筹办、等待和结构化缺口逐类满足自己的强类型证据合同；只有完成后继强制实际状态、恰一动态和可选因果。
6. 独立世界事实变化使用无任务身份的独立封套，仍通过身份、去重、版本、当前性和过期门禁，不伪造任务后继。
7. 派生需求三组复合值齐全，父子目标重叠、循环和代码缺陷需求化被拒绝。
8. 原始外设报告、未治理回执和外部材料适配入口不能进入自我邮箱。
9. 无候选方法不自动创建方法首；筹办漏检和完整前置后的结构异常走内部逻辑错误。
10. 不修改 #329、#330、#328、D455、工程、入口、生产装配和中央治理文件。
11. 既有任务结果、需求结算和自我治理自检在新职责链下全部通过。

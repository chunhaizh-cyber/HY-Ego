## 4. 必须审计的冻结合同

### S0：来源、角色与范围

```text
语义身份 = 完整任务句柄 + 非零任务序号。
冻结键只作来源关联；因果完整结构键哈希只作因果内容召回。
格式、身份、不可变核心和阶段负载必须先由既有纯值入口深检。

统计、概念映射、学习、晋级、方法代际和生产线程接线均不进入 O1-S2。
```

### S1：节点 15 与关系 18

逐项验证：

1. `节点类型::用途观察记录 = 15` 只按追加规则定义，不复用、重排或改义既有 0—14。
2. `关系类型::用途观察证据角色 = 18` 只按追加规则定义，不改既有 0—17 语义；但必须新增类型 18 专用唯一性分支，不能沿用普通关系重复规则。
3. 关系 18 的角色只允许 `1—15` 和 `30—32`；每个角色必须在详细设计中给出来源端点、目标端点、顺序号、恰一 / 零一基数、有效 / 失效、历史读回及恢复校验。
4. 未列顺序号、端点类型错误、同源同角色多条当前关系、缺必需角色或阶段角色非法组合必须在写前拒绝；已发布结构出现上述情况属于内部不一致并追根因。
5. 不得用日志、显示、SQL、线程缓存或普通引用集合替代关系 18 的权威角色证据。
6. 对照 `海中鱼巣/核心/关系仓库.cpp:180—200`，证明类型 18 的当前唯一键明确为“关系类型 + 完整源句柄 + 顺序号 / 角色”，不同角色允许复用同一目标；必须用 9+30、9+31、15+32 同目标可共存和同源同角色第二条被拒绝作审计样例。

### S2：15 槽主信息与规范编码

审计详细设计中的 15 槽表是否逐槽固定：槽号、业务含义、整数语义、合法范围、空值规则、格式版本、写入时机、不可变性、读回比较及恢复往返。

必须同时证明：

```text
恰好 15 槽；不依赖 vector 尾部空槽猜测格式。
完整回执、完整结算 DTO、完整用途事件、完整机会证据组和文本均不进入主信息。
完整节点句柄由节点 / 关系结构承载，不裸塞单个 I64 槽。
每条规范记录编码上限为 2 KiB；未知格式、缺槽、额外非空槽、越界值和非法组合写前拒绝。
待结算推进最终不得原地覆盖不可变核心槽；只允许设计明确的阶段槽 / 关系换代路径。
```

### S3：单索引与反向关系双证

确认权威定位只使用一个 O1 专用物理索引命名域；反向关系证据与完整结构读回共同完成双证，不建立第二平行索引或第二事实账。

审计必须区分：

```text
物理索引命中：只取得候选节点。
反向关系命中：读取完整关系记录并筛选类型 18、角色 1、当前有效和目标任务完整句柄，只证明任务到候选观察的结构关联。
完整身份比较：最终裁决完整任务句柄 + 非零任务序号是否相同。
```

索引命中或反向关系命中均不得单独裁决身份。零命中允许创建候选；一命中必须完整读回后裁决幂等 / 推进 / 冲突；多命中或两类证据不一致属于内部不一致，不扫描选优、不静默修复。

必须按当前 `会话.结构写入` / `索引仓库` / `关系仓库` 实际入口逐项证明：

```text
同一不透明会话可按任意 O1 物理主键只读当前候选，从而完成最多 1024 次探测。
同一会话可按目标读取完整关系记录并筛选关系 18、角色 1、当前有效和完整任务句柄。
专用数据操作不取得仓库引用或原始令牌。
```

当前代码如果只有绑定、节点反向主键组或写后绑定匹配，而没有许可内任意主键只读，则 A07 / A08 / A11 必须不通过并退回设计；不得在许可外预扫、二次取得许可或穿透仓库补做。

### S4：FNV 探测与容量

审计新详细设计是否明确 FNV 的输入字段、规范字节顺序、版本 / 命名域、初值、逐步算法及碰撞探测公式，并固定：

```text
最多探测 1024 个物理键。
哈希和探测只定位候选，不替代完整身份比较。
探测耗尽为逻辑内容量拒绝，写入开始前零结构变化。
同一任务最多 1024 条权威用途观察。
全局最多 65536 条，且规范编码总预算最多 128 MiB；任一先到即拒绝新建。
单条规范编码最多 2 KiB。
```

所有计数、字节预算和探测空间必须在第一笔候选写入前完成一致快照与拒绝，不允许写后扫描兜底。幂等读回和合法阶段推进不得被错误计作新增记录；容量计数与恢复重建必须一致。

必须逐项找到同一许可内的当前接口证据，证明可以获得有效节点 15 数量、O1 索引所有者记录数、同一任务角色 1 记录数和 O1 已保留规范字节。全类型节点总数、全局主键总数、许可外扫描或日志汇总均不得替代。任一计数 / 字节接口缺失即 A09 / A11 不通过，退回设计补接口合同，不得把预期新增入口当作当前事实。

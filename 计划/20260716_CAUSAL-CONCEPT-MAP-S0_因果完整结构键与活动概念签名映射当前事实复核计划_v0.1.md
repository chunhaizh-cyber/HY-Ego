## 1. 目标

基于 #283 已完成的权威用途观察结构承载与恢复边界事实矩阵，读取当前因果模式、动态模式、活动概念签名、活动图和生命周期接口，确认 `因果概率候选` 内“原因 `动态模式结构键` + 结果 `动态模式结构键` + 因果规则版本”三元组能否无损映射到活动概念，以及当前性是否可在一个可信读取边界内复核。

本计划沿用上游“完整因果键”简称，但必须把当前事实写窄：仓库当前没有名为“因果完整结构键”的独立正式类型，也没有已裁决的复合主键 / 身份 DTO；只有非权威纯值 `因果概率候选` 中两个命名结构键槽与一个因果规则版本。

本 S0 只形成字段映射矩阵、当前性矩阵、缺口清单和后继设计输入；不实现概念映射，不实现统计投影，不修改概念图、因果模式或任何 C++。

## 2. 流程图与详细设计适用性

本计划是因当前代码事实否定原“直接显式概念映射”假定而新增的只读门禁。现有因果用途流程图和详细设计只用于说明无环目标；#283 只提供结构承载与恢复事实 / 缺口矩阵，二者都不能作为概念映射代码许可，也不能证明权威用途观察结构已经存在。

替代正式依据为 #234 非权威因果概率 DTO 及正式构造算法、#283 S0 正式矩阵、当前概念图算法 / 服务接口和现存概念图详细设计。若字段不能无损表达或当前性不能证明，S0 仍可完成，但结论必须是缺结构签名 / 缺通用原子只读视图，并返回设计角色重做上游三件套；不得在 S0 内补字段、补视图或生成代码计划。

## 5. 必须回答的问题

1. 原因、结果两个 `动态模式结构键` 的全部七字段及因果规则版本能否无损落入当前 `概念签名材料`。
2. 顶层原因 / 结果命名槽与机会内片段顺序分别如何表达；当前是否已有正式角色枚举、角色序号或专用维度，禁止把“结果”误写成必然发生在原因之后。
3. 完整目标、动作句柄如何保持仓库编号 + 节点编号 + 版本号身份；现有通用约束载体虽声明稳定句柄值型，但当前因果构造路径只写目标 / 动作节点类型，是否存在本用途正式维度和构造合同。
4. 概念签名版本、活动图版本和概念完整句柄如何只能来自当前概念图读取，禁止调用方提供。
5. 多次独立读取如何保证活动节点、签名和生命周期来自同一当前性边界。
6. 生命周期转换是否递增活动图版本；若不递增，单看版本为何不足。
7. 删除候选专用 `概念完整当前输入材料` 能否复用为通用视图，还是必须由后继设计新增通用原子只读活动概念视图或扩展因果概念签名结构。
8. 未来唯一匹配如何只形成附加标注，不覆盖、不缩写原因键、结果键或因果规则版本；最终统计 / 学习主键仍由 #283—#285 后继设计裁决。
9. 零匹配、多匹配如何返回材料不足，不选首项、不创建概念、不写关系。
10. 当前 API 不能无损映射时，需要修改哪些上游文件、由哪个后继计划承接。
11. 当前有哪些生产完整比较、稳定哈希、单对象深检或复合三元组身份入口；哈希为何只可召回、不能裁决身份。
12. 原因键规则版本、结果键规则版本与因果规则版本当前如何耦合，概念签名版本为何不得替代任一规则版本。

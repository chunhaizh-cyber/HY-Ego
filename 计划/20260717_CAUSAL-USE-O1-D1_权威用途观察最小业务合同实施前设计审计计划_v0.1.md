## 4. 审计范围

### S0：来源与身份

建立语义身份与字段所有权矩阵，逐项验证：

```text
事件任务 = 冻结调度请求任务 = 冻结任务材料任务 = 回执内嵌请求任务 = 回执目标任务。
冻结任务序号 = 冻结调度请求任务序号 = 回执内嵌请求任务序号 = 回执来源任务序号。
回执内嵌来源执行请求存在，且与冻结来源中的完整任务执行请求逐字段相同。
完整句柄包含仓库编号、节点编号和版本号。
冻结键、请求号、幂等键、回执号、批次号、时间戳、因果键和哈希均未进入身份。
```

### S1：不可变核心与排除字段

逐字段核对九个业务引用和历史概率快照的来源、完整性与比较语义，并确认完整冻结请求、完整回执、完整结算 DTO、机会证据组、待结算提示、文本、缓存及重复状态值没有被写成候选内容。

### S2：阶段、幂等与冲突

先证明新候选与可选既有候选各自属于三种合法阶段元组，且最终阶段沿 `事件.结果来源.需求结算.材料.正式结算->结算状态.状态节点` 深检到完整一致的正式结算状态，`最近结算状态` 只作兼容投影；成功事件的新候选自身非法或已发布既有候选自身非法必须追根因，不进入同账比较。再证明：

```text
同身份 + 同核心 + 同阶段负载 -> 同账幂等。
同身份 + 同核心 + 待结算 -> 合法最终已结算 -> 同账允许推进。
两个候选各自合法后，同身份 + 异核心、终态变化、逆向或其它不允许转换 -> 同账冲突。
同一身份多个已发布当前记录 -> 内部不一致并追根因。
```

不得继续使用“同身份异完整内容一律冲突”的旧口径阻断合法阶段推进。

### S3：固定形状与物理合同后置

确认候选输出只含经批准的固定形状值对象，未保留合法来源输入中的可变长集合、完整冻结请求、完整回执、完整结算 DTO 或完整机会证据组；完整节点句柄和因果完整结构键不得被误归为排除对象。同时明确以下内容仍未裁决：

```text
节点 / 关系枚举。
主信息槽与编码。
物理索引命名域、双索引键和哈希算法。
关系端点、顺序号、基数与阶段历史。
规范字节、单记录上限、总容量和写前预算。
用途数据操作、事务 API、最后发布点。
恢复格式、段号、导入 API、高水位和发布许可。
```

### S4：非成功返回与代码计划结论

审计全部中途非成功返回是否按“逻辑内返回 / 不支持 / 冲突 / 追根因”准确分层；必须逐项核对当前 `用途事件规则版本 = 1` 的状态—具体原因—optional 唯一矩阵，区分合法入口拒绝、合法不支持、上游内部不一致和未列配对 / 非法 optional 组合，并确认成功事件一旦被接受，后续身份、来源、内容、阶段或投影矛盾只能追根因。必须专项证明候选合法性检查先于同账比较：最终阶段缺少 / 矛盾正式结算材料只落内部不一致追根因；两个合法候选间不允许转换只落逻辑冲突。写入开始后的异常只可写成“追根因并停止”，不能在缺正式证明时无条件宣称精确撤销。

完成后最多声明 O1 最小业务合同已经通过或未通过实施前设计审计。不得声明纯值 O1-V1 已实现、权威用途观察已发布、节点 15 / 关系 18 已分配、统计 / 学习 / 晋级 / 代际 / 生产 / 恢复已形成，或旧能力迁移完成。

## 4. 合同并行模型

### 4.1 合同最低字段

每项跨计划接口必须在提供者和消费者计划中用同一语义冻结：

```text
合同编号和版本
提供计划、消费计划
模块 / 文件身份
公开 DTO、规格、结果和值式材料
完整签名或等价行为
ABI / 规则版本
输入输出所有权与生命周期
逻辑内返回与内部逻辑错误
幂等、事务、撤销和可见性边界（适用时）
最终真实接线验证
漂移时退回的唯一接口所有者和受影响消费者
```

提供者拥有实现权，消费者只读消费。消费任务可以 import 计划中尚不存在的提供者模块并形成候选，但不得在本任务内补造同名模块、复制实现或修改提供者文件。

P2 并行批次的共同合同基线固定为：

```text
共同合同基线：NT-P2-CONTRACT/v0.4
唯一合同正文：计划/20260722_NODE-TYPED-MIGRATION_NT-P2_领域载荷与自我投影迁移子计划_v0.4.md
合同组：NT-C1/v0.3、NT-C1Q/v0.1、NT-C2/v0.4、P2A-FREEZE/v2
合同性质：NT-C1/v0.3 为 #339 主线真实消费合同；其它合同均为预冻结待实现合同
提供范围：#339 提供 NT-C1/v0.3；#374 提供 NT-C1Q/v0.1；#340—#342 提供 NT-C2/v0.4 领域条目；#375 提供 P2A-FREEZE/v2
消费范围：#340—#352 按各叶子登记条目只读消费
最终接线：#352 逐模块、DTO、签名、ABI、结果分类和生命周期核对真实提供者
```

本总计划只登记合同基线、所有者和路由，不复制 P2 合同正文。#340—#342 叶子也只能引用上述唯一正文并登记自身提供 / 消费条目；同名 DTO、枚举、签名或 ABI 若与唯一正文不同，以合同漂移退回，不得在叶子或执行窗口重新定义。

## 5. 长期执行通道与最终集成

施工通道固定为：

```text

-> 消费计划按预冻结合同向下实现，不等待提供者实现完成
-> 当前计划 DRIFT 形成允许的 clean WIP 断点后，可检查其它无真实依赖计划

#352 最终计划段
-> 当前长期分支父链已固定 #340—#351、#374、#375 各段结果提交
-> M01—M26、实际接口和最终门控全部成立
-> 唯一完成工程、统一运行器、入口、默认装配和删除集
-> 完整构建、运行、恢复、切换和覆盖审计

独立集成
-> 消费长期通道终端提交及逐计划段提交映射
-> 重跑最终验证并非强制发布 main
```

固定 Git 汇集顺序为：

```text
#339 -> #340 -> #374 -> #375 -> #341 -> #342
-> #343 -> #344 -> #345 -> #346 -> #347 -> #348
-> #349 -> #350 -> #351 -> #352
```

该顺序是长期通道的默认确定性计划段顺序和累计父链。#374 不消费 #340，且二者文件白名单无交集；已经收口的 #340 结果可以作为 #374 的累计父提交，不改变 #374 的业务合同、版本或允许范围。预冻结接口允许消费者在提供者尚未实现时形成源码候选，因此顺序不表示“等待上一个实现完成”的设计依赖；遇到受影响计划 DRIFT 时可跳过并检查后续无真实依赖计划。

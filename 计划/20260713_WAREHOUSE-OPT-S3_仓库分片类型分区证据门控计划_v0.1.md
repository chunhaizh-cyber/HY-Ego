## 1. 输入入口

消费 #257-#259 正式指标和裁决，复核 100,000 关系、至少 8 读 + 2 写的同一可复现负载。当前没有分片实现，也没有证据证明单锁已饱和。

## 2. 分片门槛

连续三轮必须同时满足：关系仓库等待代理值占查询 P95 的 30% 以上；增加线程后吞吐提升低于 20%；热点可按稳定键域划分且不会破坏统一句柄和跨分区关系提交。任一不满足即生成“不分片”记录并完成，阶段 820 不登记。

## 3. 条件设计处理

达到门槛后，本计划第一步仍只生成分片专项详细设计修订，不直接改生产代码。修订必须冻结：

```text
分片键和仓库编号语义
统一句柄跨分片定位
跨分片关系的唯一权威位置
跨分区写集、固定锁序和失败收口
创建 / 失效 / 重挂 / 删除原子边界
快照、恢复、审计和索引重建边界
```

详细设计复核一致后，才允许在本计划追加精确文件范围并实施阶段 820。

## 4. 结构与逻辑边界

分片只能是同一物理核心的容量 / 并发实现，不得为世界、概念、需求、任务、方法建立独立身份和事实裁决。关系仍由一个权威关系记录承载；跨分片投影、路由表和局部索引只提供候选。

多根概念森林、多父共享节点、图关系、反向查询和历史审计必须跨分片保持同一结果。不得因分片把一条边复制成多个权威事实。

## 5. 允许与禁止

默认只允许只读代码及写分片门槛记录、详细设计修订、计划索引、队列和当前状态。未完成二次详细设计和允许文件冻结前，禁止修改任何 C++ / 工程。始终禁止夹带 `#214`、控制面板、SQL、恢复计划、领域业务逻辑或旧能力迁移。

## 6. 验收

不实施分支：指标、三轮结果、门槛判定和结构零改动证据完整。

条件实施分支至少要求：

```text
全量 S1 正确性与故障注入回归；
跨分片创建 / 失效 / 重挂 / 删除无半结构；
普通删除独占、统一版本句柄和事务 ABI 不弱化；
8读+2写吞吐较 S1/S2 再提升至少 50%，P95 至少下降 30%；
单线程与小规模 P95 稳定回归不超过 15%；
连续 20 轮并发、Debug x64 0/0、完整自检、严格规范和提交推送。
```

## 7. 完成边界

证据不足时“不分片”是正式结果。证据达到门槛也只授权先修订详细设计，不自动授权生产分片代码。

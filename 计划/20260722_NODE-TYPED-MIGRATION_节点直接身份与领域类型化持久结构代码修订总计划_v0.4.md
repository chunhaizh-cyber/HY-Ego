## 1. 目标与本次升版裁决

机器逻辑目标保持 v0.1 不变：把当前代码迁移为节点直接稳定身份、正式关系拓扑、领域类型化记录、可重建索引 / 组合投影，以及隔离恢复候选完成后的运行期上下文首次发布。

v0.4 在 v0.3 共享合同修复上继续收口两类缺口。第一，P3 原合同把 M04—M05、M07—M13、M16—M20、M25 的业务合同缺失误写成 #347 可以交付的“缺口记录”，导致最终切换没有真实目标；本版补齐词 / 语素入口、完整因果、任务筹办执行完成、需求独立结算和方法完整结构合同。第二，原施工组织要求 #340—#351 分别新建 R2；本版改为一个长期执行通道逐计划段施工，每段独立冻结 plan blob、段起点、白名单和结果提交，#352 在同一累计父链满足真实门控后作为最终计划段执行。验证仍集中在 #352 和后继独立集成，不因通道连续施工而减少。

## 4. 合同并行模型

### 4.1 合同最低字段

每项跨计划接口必须在提供者和消费者计划中用同一语义冻结：

```text
合同编号和版本
提供计划、消费计划
模块 / 文件身份
公开 DTO、规格、结果和值式材料
完整签名或等价行为
ABI / 规则版本
输入输出所有权与生命周期
逻辑内返回与内部逻辑错误
幂等、事务、撤销和可见性边界（适用时）
最终真实接线验证
漂移时退回的唯一接口所有者和受影响消费者
```

提供者拥有实现权，消费者只读消费。消费任务可以 import 计划中尚不存在的提供者模块并形成候选，但不得在本任务内补造同名模块、复制实现或修改提供者文件。

P2 并行批次的共同合同基线固定为：

```text
共同合同基线：NT-P2-CONTRACT/v0.3
唯一合同正文：计划/20260722_NODE-TYPED-MIGRATION_NT-P2_领域载荷与自我投影迁移子计划_v0.3.md 第 3 节
合同组：NT-C1/v0.3、NT-C2/v0.3
合同性质：NT-C1/v0.3 为 #339 主线真实消费合同；NT-C2/v0.3 为预冻结待实现合同
提供范围：#339 已提供 NT-C1/v0.3；#340 唯一提供共同结果头并与 #341、#342 分别提供 NT-C2/v0.3 的具名条目
消费范围：#340—#351 按各叶子登记的条目只读消费
最终接线：#352 逐模块、DTO、签名、ABI、结果分类和生命周期核对真实提供者
```

本总计划只登记合同基线、所有者和路由，不复制 P2 合同正文。#340—#342 叶子也只能引用上述唯一正文并登记自身提供 / 消费条目；同名 DTO、枚举、签名或 ABI 若与唯一正文不同，以合同漂移退回，不得在叶子或执行窗口重新定义。

## 5. 长期执行通道与最终集成

施工通道固定为：

```text
#339 已完成并作为真实核心接口基线

合同计划 v0.4 发布
-> 一个长期执行顶层任务绑定一个持久 worktree / codex 分支
-> #340—#351 逐计划段自主 S0、独立结果提交、clean 后切换
-> 消费计划按预冻结合同向下实现，不等待提供者实现完成
-> 当前计划 DRIFT 形成允许的 clean WIP 断点后，可检查其它无真实依赖计划

#352 最终计划段
-> 当前长期分支父链已固定 #340—#351 各段结果提交
-> M01—M26、实际接口和最终门控全部成立
-> 唯一完成工程、统一运行器、入口、默认装配和删除集
-> 完整构建、运行、恢复、切换和覆盖审计

独立集成
-> 消费长期通道终端提交及逐计划段提交映射
-> 重跑最终验证并非强制发布 main
```

固定 Git 汇集顺序为：

```text
#339 -> #340 -> #341 -> #342
-> #343 -> #344 -> #345 -> #346 -> #347 -> #348
-> #349 -> #350 -> #351 -> #352
```

该顺序是长期通道的默认确定性计划段顺序和累计父链。预冻结接口允许消费者在提供者尚未实现时形成源码候选，因此顺序不表示“等待上一个实现完成”的设计依赖；遇到受影响计划 DRIFT 时可跳过并检查后续无真实依赖计划。

## 8. 分级验证

### 8.1 #340—#351 计划段候选级验证

- 冻结合同编号、版本、提供者 / 消费者和签名逐项一致；
- 实际差异只含本计划独占代码、数据头、工具和专项自检文件；
- 禁止 import、旧结构、跨域访问、接口复制和共享文件写入静态扫描通过；
- 固定 DTO / 字节夹具 / 拒绝矩阵由叶子计划要求时完成；
- `git diff --check`、暂存白名单和提交后 clean 通过；
- 明确不运行项目 Debug / Release 构建和完整程序，不宣称编译或能力完成。

### 8.2 #352 最终验证

- 全部候选按固定顺序汇集，接口实际签名与合同逐项相等；
- 唯一工程登记、统一自检运行器和默认装配接线完整；
- Debug / Release x64 当轮 Rebuild 和两配置完整自检；
- 跨计划事务、故障撤销、关系 / 领域记录 / 历史 / 4170 全量互证；
- 进程甲冻结退出、进程乙仅恢复启动、规范化再冻结相等；
- 旧格式逐版本成功 / 拒绝、恢复失败宿主不变；
- 旧生产符号、调用点、工程项和物理结构零消费者 / 零命中；
- strict、diff、最终暂存白名单和独立覆盖审计。

完整验证后置只改变验证时点，不减少验收项目。任一合同漂移先退回提供者和受影响消费者；不得由 #352 临时发明接口消冲突。

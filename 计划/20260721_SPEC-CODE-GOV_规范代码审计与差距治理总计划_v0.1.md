## 1. 目标

本计划用“规范审计、代码审计、差距治理”的方式，分层完成两个不能互相替代的目标：

```text
第一层：规范能否定义闭环
-> 正式规范能够唯一回答对象、输入、输出、写入权、消费权、生命周期、状态、并发、拒绝、内部逻辑错误和验证合同

第二层：代码是否已经接通
-> 当前生产入口确实沿规范链路完成材料供给、任务承接、方法执行、领域提交、任务验证、需求复核和连续运行
```

最终必须分别给出两层结论。规范闭环成立不等于代码已接通；自检夹具可运行、日志显示成功、队列存在或单个模块已实现，也不等于生产闭环成立。

### 4.2 明确排除项

1. 本总计划本身不修改 C++、工程文件、自检代码或生产入口。
2. 不从审计记录、缺口清单或聊天结论直接施工。
3. 不把旧项目实现、过期详细设计、既有流程图或旧 #315—#322 计划反向覆盖现行规范。
4. 不把 D455 四路图像流当成四类逻辑产品。
5. 不让 D455 上行桥、自我线程、普通任务工作线程或方法绕过任务域直接写机器事实。
6. 不把自检夹具搬入生产入口充当运行期装配。
7. 不在 DD-01—DD-04 完成前生成或恢复 D455 生产接线施工许可。
8. 不在真实外设门禁通过前宣称 D455、外设、自我循环、自我苏醒或最小生命闭环完成。

## 8. 文件与时期治理

### 8.1 文件状态

```text
当前有效
-> 可作为现行设计或施工依据

待修订
-> 仍有可复用合同，但状态、规范绑定或边界已过时

已归档
-> 核心前提被现行规范否定，只作历史证据

重定向
-> 原路径仅指向归档位置，不含施工正文

缺失
-> 当前没有可实现设计，必须先补齐
```

### 8.2 归档与删除规则

1. 核心业务合同失效的详细设计移入 `规范/详细设计/过期设计/`。
2. 原路径如被大量历史记录引用，可保留明确的非施工重定向；不得同时保留两份可施工正文。
3. 临时草案、重复输出和已被正式产物完全替代且无历史价值的文件，核对引用后再删除。
4. 计划完成后移入 `计划/已完成计划/`；失效计划保留终止原因，不重新派发旧版本。
5. 历史实施记录不因后续规范变化批量改写，只补当前失效或替代关系。

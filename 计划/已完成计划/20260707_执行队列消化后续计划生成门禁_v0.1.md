#108 到 #111：FS-05 到 FS-08 S1-S4 代码实施切片

本计划原用于避免设计窗口凭预期结果提前生成下一批切片。现按用户确认修订为滚动循环门禁：执行窗口每完成一个队列项并形成正式产物后，设计窗口即可只基于该完成项复核和生成后续计划；不得等待整批 #108 到 #118 清空才开始下一轮设计。

## 2. 确认范围

本计划确认后，只允许登记一个文档治理队列项：

```text
滚动复核已完成队列项的实际产物，并按明确依赖生成下一批计划候选。
```

该队列项的目标不是等待整批队列清空，而是在任一队列项完成后读取该项实际实施记录、详细设计、验收结果和新增待确认问题，再决定是否能生成只依赖该项的下一批待确认计划。若计划明确依赖多个队列项，则只等待这些列明依赖项全部完成或真实阻塞。

## 3. 生成下一批计划的前置条件

下一批计划必须至少满足：

```text
1. 至少一个可执行队列项已执行完成，或执行窗口明确记录真实阻塞。
2. 被用于生成计划的完成项必须有对应实施记录、详细设计、扫描记录、验收结果或阻塞记录。
3. 新计划只能引用已完成或已阻塞的队列项作为依据；不得把未完成项的预期结果当作事实。
4. 依赖多个队列项的综合计划，必须列明依赖项，并等待这些依赖项全部完成或真实阻塞。
5. 若涉及 C++，必须已有独立切片所需的允许文件、禁止文件和验证方式。
6. 若执行中产生待确认问题，必须先写入项目记忆并等待确认窗口裁决。
7. 不以旧函数编号作为迁移单位，只以服务逻辑包、服务专项计划或服务代码实施切片作为计划单位。
```

## 4. 明确排除项

本计划不允许：

```text
不直接修改 C++。
不迁移旧函数体。
不新增旧主信息字段。
不把未执行的详细设计专项预设为已完成。
不跳过新计划明确依赖项的实际执行结果。
不把整批队列清空作为普通单项计划生成的默认前置条件。
不从控制面板或数据库旧实现生成迁移计划。
不接 SQL / ADO / WebView2 / D455 / 体素 / 外设。
```

## 5. 控制面板和数据库口径

控制面板和数据库从本轮迁移计划中撤出：

```text
控制面板：后续根据用户明确指令重建，不参与旧能力迁移。
数据库：后续根据用户明确指令重建，不参与旧能力迁移。
```

旧 FS-X 中的控制面板、SQL、ADO 事实只保留为排除证据，不作为自动生成迁移计划的依据。

## 6. 验收方式

本计划自身确认后，只做文档治理验收：

```powershell
git diff --check
if (Test-Path .\tools\check_specs.py) { python .\tools\check_specs.py }
```

后续真正生成下一批计划时，必须再次读取其依据队列项的实际产物；若计划依赖多个队列项，必须逐项列明依赖和完成 / 阻塞证据。

## 7. 完成声明边界

确认本计划只能声明：

```text
下一批计划生成门禁已建立。
控制面板和数据库已从迁移计划中撤出。
后续计划生成改为按已完成队列项滚动触发；跨项综合计划只等待明确依赖项。
```

不得声明：

```text
未完成队列项已经完成。
下一批代码实施切片已确认。
控制面板已重建。
数据库已重建。
旧能力迁移完成。
```

## 2. 策略目标

将后续迁移主线调整为：

```text
服务操作函数矩阵
-> 批次级代码实施包
-> 应用逻辑流程图迁移包
-> 流程图节点到服务调用编排
-> 批次级验收
```

替代当前过细的默认路径：

```text
单个服务小切片
-> 单个确认
-> 单个断点
-> 单个同步
```

## 4. 明确排除项

```text
不直接修改 C++。
不直接生成服务操作函数代码。
不迁移旧函数体。
不新增旧主信息字段。

不绕过领域服务裸写节点仓库、主信息仓库、关系仓库或索引仓库。

```

## 5. 新迁移单位

| 层级 | 新单位 | 用途 |
| --- | --- | --- |
| 证据层 | 旧函数事实 | 发现旧行为、副作用、依赖和排除项 |
| 服务数据层 | 服务操作函数矩阵 | 定义每个服务可创建、读取、绑定、迁移、拒绝和失效阻断哪些结构 |
| 代码实施层 | 批次级代码实施包 | 一次确认多个同阶段、同验证剧本、同禁止范围的服务操作函数 |
| 应用逻辑层 | 流程图迁移包 | 把旧应用逻辑改写为服务调用编排，不按旧函数体迁移 |
| 验收层 | 流程图验收路径 | 验证输入入口、服务调用、结构变化和输出变化 |

```text

```

流程图迁移规则：

```text
旧函数只作为证据节点。
流程图节点必须映射到领域服务入口或只读材料。
流程图边必须映射为调用顺序、关系写入、状态迁移、动态记录或拒绝路径。
流程图不得直接写仓库。
流程图不得把控制台、日志、显示或 SQL 投影作为机器事实。
```

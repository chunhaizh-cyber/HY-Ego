## 1. 目标

消费 #225 正式任务结果与 #234 正式因果概率候选，形成不写仓库的用途事件和结构化结果效用：

```text
执行前显式用途来源冻结 + 三种正式任务结果
-> 复核冻结候选与结果属于同一完整任务回合
-> 分栏保留概率、任务达成、需求结算和观察阶段
-> 输出纯值用途事件候选
```

本切片不创建用途观察节点，不更新统计，不生成学习候选，不修改方法、任务、需求、状态、动态或因果事实。

## 4. 目标结构与公开入口

`材料.用途事件` 唯一声明：

```text
用途事件规则版本。
用途事件处理状态与拒绝原因。
任务达成分量、需求结算分量、观察阶段枚举。
用途来源冻结请求 / 材料：完整因果候选、完整任务执行冻结请求、冻结键、规则版本和冻结时间戳；第一版下游观察主键恰等于该冻结键，不另生成第二稳定键。
规范化任务结果来源：不依赖 #225 组合类型的完整回执、正式结果分类、身份、状态、动态、提交、结算和幂等 / 时间材料。
用途事件请求：用途来源冻结、规范化任务结果来源、规则版本。
结构化结果效用：正式结果、三类分量和独立概率分量。
用途事件材料：完整回执 / 冻结请求、需求、任务、选择记录、方法、动作、目标 / 实际状态、动态与结构化效用。
用途事件结果：状态、拒绝原因和值式事件。
```

公开入口：

```text
用途事件业务服务::冻结用途来源(const 用途来源冻结请求&)
用途事件业务服务::形成结构化结果效用(const 用途事件请求&)
用途事件组合器::形成用途事件候选(const 用途来源冻结材料&, const 任务结果治理请求&, const 任务结果治理结果&)
```

只有 `组合.用途事件` 直接 import `组合.任务结果结算` 并完成规范化适配；材料 / 算法 / 服务不反向依赖 #225 组合模块。服务同步调用纯值算法；不持有服务引用、仓库、许可、会话、锁、线程、回调或持久状态。

## 5. 实施步骤

### S0：执行前接口复核

从 JY-369 设计发布提交逐签名读取 `材料.因果模式`、`组合.任务结果结算`、`任务结果回执协议` 和 `协议.任务执行请求`。确认五个目标文件不存在、850 未占用、工程与入口允许最小登记。任一正式字段、枚举值、模块所有权或阶段漂移，停止代码修改并退回设计。

### S1：材料唯一所有者

新增 `材料.用途事件.ixx`，用强类型枚举表达状态和分量，并定义执行前用途来源冻结、规范化结果来源和值式事件。禁止普通 bool 充当任务达成、需求结算或最终效用事实。DTO 只保存值式材料，不保存指针、引用或运行期对象，也不得 import `组合.任务结果结算`。

### S2：入口准入与回合关联

先由纯值服务形成执行前用途来源冻结：完整候选、完整任务执行冻结请求、非零冻结键、冻结时间戳和规则版本全部有效。冻结材料必须把 `观察主键 = 冻结键` 作为同一字段语义输出，不另接收、派生或随机生成观察键。结果返回后，复核新版回执协议、正式结果已收口，并用 `冻结任务执行请求相同` 确认执行前冻结与结果回执是同一完整任务回合。缺冻结、不同请求或结果后临时补造候选均逻辑返回；不得用候选中碰巧存在的同回合机会倒推“候选已参与选择”。

### S3：身份与结果一致性

新增 `组合.用途事件.ixx` 作为唯一 #225 DTO 适配层，从结果请求 / 结果抽取来源需求、任务、选择记录、方法、动作、目标状态、实际状态、动态、提交与结算材料；逐项复核任务、方法、选择、场景、主体和目标 / 实际句柄一致，再调用纯值服务。状态与 `任务提交 / 需求结算 / 待结算提示` 组合必须符合三种正式映射。

### S4：结构化效用形成

形成强类型任务达成、需求结算和观察阶段，完整保留执行前冻结候选的原因 / 结果键、因果规则版本、历史机会证据、计数与 Q10000。不得计算单一效用分数、阈值、置信度、可靠性、稳定因果或学习建议。

### S5：模块、自检与治理收口

新增 `算法.用途事件`、`服务.用途事件`、`组合.用途事件`、`自检.用途事件`，合计五个真模块。入口只在 Debug 最小登记 850；完整 A01-A16 留在自检模块。完成工程 / filters 唯一登记、Debug / Release 完整 Rebuild、Release 隔离、完整自检和最终 Debug 连续 20 / 20；归档计划并同步治理记录。

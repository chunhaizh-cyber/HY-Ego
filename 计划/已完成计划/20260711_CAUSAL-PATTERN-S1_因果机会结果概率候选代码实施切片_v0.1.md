## 1. 目标

基于 #233 正式 `动态聚类结果` 形成第一版纯值因果概率候选：

```text
原因必须是正式重复聚类 -> 至少两个不同完整回合提供机会。
每个原因支持回合恰有一个原因实例 -> 形成一次机会。
同一回合原因后至少一个结果实例 -> 一次正结果。
同一回合原因后没有结果实例 -> 一次负结果。
Q10000 = floor(正结果数 * 10000 / 机会数)。
```

概率只表达当前输入证据下的非权威比值。它不要求可靠性证明，不自动升格为稳定因果，也不写任务、需求、方法、统计、概念、轻量因果引用或持久化结构。

## 5. 实施步骤

### S0：实际接口复核

从 `b73ad16` 逐签名核对 `动态聚类结果`、`单步片段实例候选`、`重复动态聚类候选`、`规范动态步`、完整键 / 回合比较函数和公开容量。确认阶段 650 未占用、四个目标模块不存在、工程与入口接线无漂移。任一不一致停止并退回设计。

### S1：材料唯一所有者

新增 `材料.因果模式.ixx`，集中声明请求、状态、拒绝原因、逐回合证据、候选和结果。完整性函数只做局部值式结构检查，不读取服务或全局状态。服务 / 算法 / 自检只 import，不重复声明或前置声明这些类型。

### S2：入口准入和正式原因

验证聚类结果成功、容量有效、两键完整不同、版本 1 一致。使用 `动态模式结构键相同` 精确查找原因重复聚类；哈希不得裁决。零命中逻辑拒绝，多于一个同键重复聚类进入内部不一致。结果键允许零命中。复核原因聚类来源实例 / 支持回合与单步实例组一致；正式 DTO 内部矛盾进入内部不一致。

### S3：逐回合机会和结果

只遍历原因重复聚类支持回合。每回合恰有一个原因实例才形成机会；多原因实例逻辑拒绝。收集同一完整回合的结果实例：小于原因序号进入前置证据，等于原因序号逻辑拒绝，大于原因序号进入后续证据。后续组非空计一次正结果，否则计一次负结果。

### S4：概率与确定排序

验证机会数等于支持回合数和证据组大小、正负之和等于机会数。使用 64 位安全整数计算 `floor(正结果数 * 10000 / 机会数)`；#233 容量上界使有效版本 1 输入乘法可证明安全。按正式回合锚点和实例顺序确定排序，不依赖输入排列、桶顺序或并发完成顺序。

### S5：模块、自检和收口

在材料模块之外再新增 `算法.因果模式`、`服务.因果模式`、`自检.因果模式` 三个真模块，合计四个。入口只在 Debug 最小登记阶段 650，不放 A01-A18 正文。完成工程 / filters 唯一登记、Debug / Release 完整 Rebuild、完整自检、Release 隔离、50 轮确定性和最终 Debug 连续 20 轮；随后归档计划并同步治理记录。

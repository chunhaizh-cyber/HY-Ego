# HY-Ego

> 从零开始构建数字生命 —— 工程测试版  
> Windows / Visual Studio 2022 / C++20 Modules（`.ixx`）

HY-Ego 是一个面向“数字生命（Digital Life）”的工程化试验仓库：以 **场景—存在—特征** 为基础信息骨架，向上构建 **世界树 / 概念树 / 二次特征 / 动态 / 因果 / 任务 / 需求 / 方法** 等结构；同时包含 **相机接入、三维场景管理与实时显示线程** 等工程能力，便于在真实数据流中持续迭代验证。

## 任务执行模型（当前约定）

### 1. 方法模板层与任务实例层分离

- 方法是只读模板，负责提供：
  - 方法头结点
  - 方法条件/结果模板
  - 方法自带场景容器
  - 方法参数要求
- 任务是运行实例，负责提供：
  - 任务头结点
  - 任务虚拟存在
  - 任务运行条件场景
  - 步骤节点与结果节点

当前系统语义是：

`读取方法模板 -> 创建任务实例 -> 构造满足要求的运行场景 -> 执行 -> 结果只写回任务侧`

方法模板不应被任务执行过程回写。

### 2. 任务节点结构

当前任务侧以三类核心节点组织：

- `头结点`
  - 持有需求
  - 持有任务虚拟存在
  - 持有任务全局条件场景
  - 缓存当前步骤节点与最近实际结果节点
- `步骤节点`
  - 表示当前要执行的一步
  - 持有步骤条件场景
  - 持有可用方法头结点列表
  - 持有当前选中方法头结点
  - 持有重试、超时、切换方法等运行参数
- `结果节点`
  - 同时承载预测结果与实际结果
  - 通过 `结果角色` 区分 `预测结果` / `实际结果`
  - 记录来源步骤、来源方法、命中预测、命中后动作、评估摘要等信息

### 3. 任务链组织规则

- `任务头结点.子链` 挂步骤节点
- `步骤节点.子链` 挂结果节点
- `步骤节点.下节点` 表示同层并列步骤
- `结果节点.下节点` 表示同一步下的不同预测结果
- `结果节点.子链` 表示命中该结果后的下一步步骤

因此执行主链是：

`任务头 -> 步骤 -> 命中的结果 -> 下一步骤`

### 4. 预测结果与实际结果

- 每个步骤在筹办阶段应先生成一组预测结果节点
- 运行时每执行一次，会在该步骤下追加一个实际结果节点
- 实际结果节点需要回指命中的预测结果节点
- 如果没有命中任何预测结果，则记为 `预测外结果`

`预测外结果` 表示当前方法模板对该情形覆盖不足，是方法待补全或任务选法待修正的证据。

### 5. 预测命中规则

预测结果命中不靠硬编码分支，而是依赖二次特征比较：

- `相等`：实际结果的二次特征与预测结果二次特征一致
- `相似`：实际结果的二次特征与预测结果二次特征在可接受范围内近似

也就是说：

`命中预测 = 二次特征相等 或 二次特征相似`

后续预测匹配、结果归档、方法补全，都应统一建立在二次特征比较之上，而不是散落在各方法内部各自判断。

### 6. 当前筹办规则

当前 `任务执行器` 已经实现“当前条件下的直接筹办”：

- 先从任务需求取出目标状态
- 在方法树里找“结果作用特征类型与需求一致”的方法结果
- 对数值型目标，带入当前值、目标值、预测值做最小方向判断
- 过滤掉会把状态推远或方向相反的方法
- 再用当前场景二次特征检查方法条件是否已满足
- 最后通过规避过滤排除法律、人文、规则等禁止的方法

通过以上规则后，剩余候选方法会被展开成：

- `任务头结点` 下的并列 `步骤节点`

每个步骤再根据方法结果模板补出一组：

- `预测结果节点`

运行时追加：

- `实际结果节点`

并用二次特征做：

`相等或相似命中`

当前版本尚未自动展开“未满足条件的方法前置步骤”，也就是：

`当前先解决直接可执行层，前置条件展开属于下一阶段`

### 7. 当前任务优先级规则

当前任务调度把优先级拆成三层：

- `基准优先级`：来自主任务需求权重，是整棵任务树共享的基础值
- `局部优先级偏移`：来自当前步骤命中分、候选方法评分、局部运行修正
- `调度优先级`：执行器运行时按当前上下文实时刷新后的排队值

当前已经落地的规则是：

- 子任务的调度优先级高于父任务
- 更靠前步骤派生出的子任务，高于更靠后步骤派生出的子任务
- 单步、叶子、无并发子任务的最小任务优先
- 父任务存在可推进的直属子任务时，会显著让位给子任务，但仍保留调度子任务的能力
- 步骤节点、结果节点、子任务头结点的基准优先级都追溯主任务头结点，不按局部树枝各自漂移

也就是说：

`主任务提供基础优先级，步骤与子任务只在这个基础上做局部调度修正`

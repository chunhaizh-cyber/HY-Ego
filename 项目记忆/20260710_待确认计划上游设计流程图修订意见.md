## 3. 真实闭环串联优先目标计划

### REAL-REV-01：前置状态需要同步到最新队列事实

问题：

`计划/待确认计划/20260709_真实闭环串联优先目标计划_v0.1.md` 已写明必须等待 `THREAD-S0` 到 `THREAD-S6` 形成正式实施记录或真实阻塞记录后才能确认入队，这个门禁正确。但计划内 2026-07-10 复核语句仍停留在“THREAD-S2 至 THREAD-S6 已按依赖顺序确认入队”，没有反映当前项目记忆里 #157、#158、#159 已完成，#160 待执行，#161 依赖 #160 的状态。

修订建议：

在计划前置门禁或复核段补一条当前事实：

```text
当前复核口径：THREAD-S0、THREAD-S1、THREAD-S2、THREAD-S3、THREAD-S4 已形成实施记录或验证记录；THREAD-S5 #160 依赖已满足但仍待执行；THREAD-S6 #161 仍依赖 THREAD-S5 完成或形成真实阻塞记录。本计划仍不得确认入队。
```

落地边界：

只更新状态说明，不改变本计划后置待确认结论。

### REAL-REV-02：流程图拒绝分支应显式标为逻辑内返回

问题：

真实闭环流程图中 `任务选择方法关系有效？` 的否分支为 `拒绝：不写动作请求`，`领域入口前置拒绝通过？` 的否分支为 `任务拒绝回执材料`。这两个节点本意符合逻辑内返回，但节点文字没有直接写 `逻辑内返回`，容易被误读成普通失败或业务事实。

修订建议：

将相关节点小修为：

```text
逻辑内返回：任务选择方法关系无效，不写动作请求，结构不变
逻辑内返回：领域入口拒绝，回执只作材料，任务状态不由回执直接裁决
```

若流程图需要验收闭环，可把这两个拒绝节点也连接到 `闭环验收材料` 或新增 `拒绝路径验收材料` 节点，避免拒绝路径在图上悬空。

### REAL-REV-03：完成声明节点应改为后续验收完成声明候选

问题：

流程图末尾节点写 `完成声明：只证明更真实闭环第一轮通过`。虽然上下文已说明这是后续目标，但单看节点容易被读成当前已经完成。

修订建议：

改为：

```text
后续验收完成声明候选：只证明更真实闭环第一轮通过
```

并保持计划和详细设计中的禁止扩大声明。

### REAL-REV-04：方法选择应锁定为固定样例或已选关系，不应暗含自动选择算法

问题：

详细设计中写第一轮目标为“选出一个保守方法入口”，计划中又写“不实现自动选择算法或候选排序”。两者不冲突，但建议把“选出”改得更窄，防止后续实施切片误吸收候选排序或学习算法。

修订建议：

在详细设计或计划 S1/S2 中补充：

```text
第一轮方法选择只允许使用固定验收样例、已存在任务选择方法关系或人工指定方法，不实现候选排序、自动选择、方法学习或成功率裁决。
```

### REAL-REV-05：父计划的允许文件候选应拆成只读复核范围和后续代码候选范围

问题：

计划第 6 节列出后续允许文件候选，含多个领域服务、核心运行队列、入口和工程文件。该计划本身是后置计划，不是代码实施切片；过早放大候选范围会降低后续切片边界清晰度。

修订建议：

拆为两段：

```text
REAL-LOOP-S0 只读复核允许范围：计划、项目记忆、实施记录、相关领域服务和核心入口的只读扫描。
后续代码切片候选范围：每个 S2-S5 独立切片重新列允许文件、禁止文件和验证方式；父计划不授予 C++ 修改权限。
```

## 4. 自我治理循环详细设计生成计划

### SG-REV-01：计划确认联动规则需要写明“详细设计不适用原因”

问题：

项目规则要求确认计划时同步确认对应流程图和详细设计；若无流程图 / 详细设计来源的文档治理计划，必须写明不适用原因和替代依据。`自我治理循环详细设计生成计划` 的目标正是生成 `规范/详细设计/自我治理循环详细设计.md`，当前该文件不存在，这是合理状态，但计划需要显式说明。

修订建议：

在计划依据或确认范围中补充：

```text
本计划的目标产物是 `规范/详细设计/自我治理循环详细设计.md`，因此不存在可预先联动确认的同名详细设计。确认本计划时只联动确认两份流程图、运行宿主与多线程消息队列详细设计、THREAD-S1 至 THREAD-S4 实施记录作为替代依据；自我治理详细设计生成后，后续代码实施仍需另建待确认施工计划。
```

这是 P0 修订项；不补这条时，计划确认链路容易被误判为缺少上游详细设计。

### SG-REV-02：文档治理计划为何仍放待确认，需要给出理由

问题：

按当前规则，建立详细设计通常属于可直接执行的文档治理动作。但该计划仍放在 `计划/待确认计划/`，状态是“待确认 / 文档治理计划”。这不是安全问题，但需要说明为什么这里比普通详细设计更严格。

修订建议：

二选一：

```text
保守路线：保留待确认状态，并写明原因是本计划涉及自我治理循环边界、旧鱼巢自我线程证据改写和后续机器结构候选，需用户确认后再生成详细设计。
直接文档治理路线：若用户确认本计划只生成详细设计且不改变机器结构，可从待确认计划转为可直接执行文档治理项，但仍不得登记 C++ 实施队列。
```

当前建议采用保守路线。

### SG-REV-03：自我治理流程图的当前代码事实应补 THREAD-S4

问题：

`海中鱼巣自我治理循环承载边界流程图` 的说明和计划已承接 THREAD-S1 至 THREAD-S4 实施记录，但“当前代码事实”段主要列出 THREAD-S1、THREAD-S2、THREAD-S3，未明确补入 THREAD-S4 缓存统计线程与事件日志线程第一轮壳。

修订建议：

在流程图“当前代码事实”中补充：

```text
入口.cpp 当前包含 THREAD-S4 验收片段；运行宿主已具备缓存统计线程壳、事件日志线程壳、缓存刷新消息、事件日志消息、局部失败材料和停止收口第一轮材料。该事实不证明完整缓存系统、事件日志持久化恢复、仓库快照恢复、运行期回放、自我循环或自我苏醒完成。
```

同时不要补入 THREAD-S5 / THREAD-S6 为已完成事实，除非执行窗口后续形成正式实施记录或真实阻塞记录。

### SG-REV-04：旧鱼巢证据进入详细设计前应重新做 S0 只读核对

问题：

自我治理计划和流程图引用旧 `D:\鱼巢` 自我线程材料。当前流程图已写明旧代码只作为证据，但后续生成详细设计时，如果直接沿用流程图摘要，仍可能把旧函数事实误读成新项目事实。

修订建议：

在计划 S0 中补充：

```text
生成自我治理循环详细设计前，重新只读核对旧鱼巢自我线程相关文件和旧流程图的关键结论；核对结果只作为逻辑证据和排除边界，不作为海中鱼巣已实现事实，不形成旧函数体迁移许可。
```

### SG-REV-05：治理 mailbox / 自我治理队列与运行消息队列需要先拆边界

问题：

计划把 `治理 mailbox / 有界治理队列 / 冻结批次边界` 列为目标产物，流程图同时已有运行消息队列。后续详细设计必须先裁决两者关系，否则容易把运行消息队列直接升级为自我治理事实容器。

修订建议：

在目标产物中加入一张矩阵：

```text
结构：运行消息队列 / 自我治理消息 / 治理 mailbox / 冻结批次
写入方：运行宿主 / 任务管理上行桥 / 自我治理服务候选
读取方：任务管理线程 / 自我治理循环 / 领域服务复核入口
权威性：请求材料 / 回执材料 / 治理候选 / 不直接裁决机器事实
生命周期：入队 / 冻结 / 归并 / 失效 / 清空
验证方式：入口拒绝、重复拒绝、版本过期拒绝、读回数量快照、清空后权威结构不变
```

### SG-REV-06：执行前许可预判不得成为权限事实

问题：

旧鱼巢流程图中有执行前许可请求和审批，海中鱼巣流程图中改为 `执行前许可预判`。这个方向正确，但详细设计必须继续收窄，防止后续把许可预判落成机器权限事实。

修订建议：

在自我治理详细设计目标中明确：

```text
执行前许可预判只作为治理候选材料；真正的执行权限仍来自任务授权、任务承接和任务选择方法关系。预判结果不得以 bool、日志、摘要、线程状态或消息状态形式持久化为权限事实。
```

### SG-REV-07：失败收口要求应升级为验收矩阵，不只列为待确认项

问题：

计划目标中已经写到读回验证、数量快照、失败停止门禁、失效隔离或事务回滚待确认项。由于自我治理是多步归并路径，这些不应只作为附属待确认项，应进入详细设计验收矩阵。

修订建议：

生成详细设计时至少新增以下验收项：

```text
SG-A1：拒绝入队后结构不变。
SG-A2：冻结批次失败不产生可读半结构。
SG-A3：单条治理消息归并失败后，已写结构必须回滚、失效隔离或由数量快照证明不可读。
SG-A4：任务上行回执转治理消息后，不直接裁决任务完成、需求满足或方法成功。
SG-A5：日志、摘要、控制台、显示刷新和控制面板刷新请求不承载机器事实。
```

# PLAN-DEPENDENCY-PARALLEL 真实依赖门控与无依赖隔离并发规则修订记录

日期：2026-07-21

决策编号：JY-555

状态：设计规则修订 / 文档治理 / 不授权 C++ / 不派发任务

## 1. 用户裁决

计划不再默认采用“先生成依赖门控计划”。只有计划之间存在真实、不可通过合理拆分消除的前后依赖时才登记门控；没有真实依赖时，应尽量隔离业务对象、文件、结构、接口和装配所有权，为不同 worktree 并发执行创造条件。

## 2. 现行问题

修订前 `AGENTS.md` 第 28 条和任务队列通用规则把“先生成依赖门控计划”写成默认路径，容易把以下非依赖因素误写成串行链：

```text
同一主题或同一总计划
同一规范 / 详细设计来源
设计生成顺序或文档顺序
同一目录
验收先后顺序
仅共享工程、入口或统一运行器
```

这与多工作树只并发互相独立切片的既有安全门禁并不矛盾，但缺少“先拆分、再判依赖”的前置规则，导致安全规则可能被误用成默认串行策略。

## 3. 新规则

```text
先按业务对象、允许文件、结构所有权、公开接口和共享装配面拆分
-> 必须消费另一计划尚未形成的正式产物 / 接口
   或存在无法通过拆出共享接线消除的所有权先后
   => 依赖门控待执行
-> 不存在上述关系
   => 独立待执行
   => 共同基线、所有权互斥、共享文件唯一所有者、集成顺序和验证矩阵通过后优先登记并行批次
```

依赖门控计划必须具名写出依赖产物、消费接口、无法隔离原因、前置编号、解除条件、假定接口、执行前复核和漂移退回。接口漂移只退回受影响计划，不得连带冻结无关计划。暂不并行时必须具名说明资源、所有权或验证原因。

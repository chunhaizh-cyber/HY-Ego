## 3. 计划真实派发与集成状态机修复

用户硬门禁固定为：具有执行写范围的施工计划一旦满足设计、依赖、五件套、实际接口、队列、身份和验证条件，设计窗口必须在同一治理续程真实派发，不等待下一轮提醒。

本批只修规则，没有产生施工计划。状态机已修为：

```text
待执行
-> 派发准备提交推送
-> 向唯一独立执行顶层任务实际发送消息并读回
-> 已派发待执行回执与消息证据同步
-> 派发状态提交推送
-> 目标读取状态提交并完成 S0
-> 执行中
-> 分支完成待集成
-> 已预留待集成派发
-> 集成准备提交、消息 / 读回、集成状态提交、集成 S0
-> 集成中
-> 已集成待设计同步
或：主线门禁不成立 / 原角色必须停机
-> 集成分支验证通过待主线发布
-> 另立独立顶层发布 / 集成任务与新身份
-> 重新走集成准备提交、消息 / 读回、状态提交和 S0
-> 已集成待设计同步
-> 已完成 / 已归档
```

修复点：

1. 真实消息前只能保持 `待执行` 或 `已预留待集成派发`，不能预写为已派发。
2. 消息发送成功但仅读回接口不可用时，才允许使用任务工具的持久化投递回执；发送失败、目标缺失或歧义不能晋级。
3. 初次消息类型是“派发准备”，状态提交形成后才发送“派发状态已发布”；接收方在状态提交可读前只读等待。
4. 执行成功统一进入 `分支完成待集成`，不能通过旧的“执行完成待设计同步”直接归档而绕过独立集成。
5. 接口漂移和验证失败经设计裁决后必须先回到 `待执行`，再走完整重派链；不保留绕过派发的“重新待执行”快捷状态。
6. 集成分支验证通过但 main 未发布时进入 `集成分支验证通过待主线发布`；原角色停机并释放占用后只能另立独立顶层发布 / 集成任务，不得把旧验证或旧消息伪装成新派发。

`.codex/skills/hai-zhong-yu-chao-plan/SKILL.md` 和 `AGENTS.md` 已同步上述两阶段派发摘要。

# TASK-RESULT-STABILITY-S0 JY-445 验证失败裁决与独立基线诊断设计

日期：2026-07-19

角色：WT-MAIN 设计角色

## 1. 输入事实

1. WT-310-R1 三份草案已通过两配置 Rebuild、两配置单轮完整自检和本计划专项；严格 Debug 连续组第 19 轮才出现全局失败。
2. 第 19 轮 P0-B3C A01—A12 与 P0-B0 A01—A20 仍全部通过。
3. `任务结果结算通过` 直接取阶段 610 报告；`概念命名第一切片通过` 额外依赖中央批次退出码，因此后者是派生失败。
4. 第 19 轮 stdout 只保存在已结束 PowerShell 进程变量中，工具仅返回末 200 行；TASK-RESULT-S1 A01—A18 已被截掉，叶子证据不可恢复。
5. 阶段 610 早于 #310 沿用的 910，且 TASK-RESULT-S1 使用独立夹具；当前证据不能证明 #310 草案导致失败。
6. 旧连续组存在非零轮次，整体不能登记 20 / 20；18 轮不得续算，一次新运行不得替代缺失轮次。

## 2. 裁决

1. #310 升为 v0.2，状态改为 `验证失败待裁决`。
2. 允许 WT-310-R1 把三份草案和专属实施记录提交为不可集成 WIP 断点；该提交不进入同批集成。
3. 新增 #311 / DQ-203，精确从共同冻结代码基线 `1d95aa5` 建立独立诊断 worktree；最多 60 轮，每轮完整输出先落盘后判断。
4. 基线复现时按具体叶子新建修复计划；基线 60 / 60 未复现时只进入候选提交对照诊断。两条分支都不直接放行 #310。
5. #305 `aee7cc4` 保持只读待同批，不提前集成；#301 / #306 继续门控。

## 3. 范围一致性

#311 不改业务流程和机器结构，因此不新建流程图或详细设计；其替代依据精确引用既有 TASK-RESULT-S1 流程图、详细设计、已完成 #225 计划、当前 #310 v0.2 和本记录。#310 v0.2 只增加验证失败收口，不改变 11 个直接合同、四文件范围或业务验收。

## 4. 禁止宣称

本裁决不证明 TASK-RESULT-S1 的具体失败叶子，不证明 #310 无关或有责，不证明 #310 / #305 完成，不创建集成许可，也不解除任何恢复后继。

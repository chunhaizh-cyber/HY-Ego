## 1. 审计结论

#340—#352 v0.2 不能继续作为最终施工合同。根因不是某个执行窗口单独漏改，而是共同合同在 #339 真实实现进入 main 前按预期结构冻结，之后没有用真实导出面重新闭合：

1. `NT-C1/v0.2` 使用了当前代码不存在的平行 DTO 和签名；真实核心类型是 `节点直接身份记录`、`正式关系记录`、`可重建索引记录` 和嵌套的 `节点直接身份结构写入结果`。
2. v0.2 把正式关系的业务角色与顺序号拆成两个字段；4010、7130 和真实 `正式关系记录` 都规定业务角色由 `std::int64_t 顺序号` 的主题合同表达。
3. 真实事务参与者 `完成发布()` 返回 `void`；v0.2 计划不能补造发布结果。
4. v0.2 共同领域结果没有唯一物理文件和唯一提供者，存在被多个叶子复制定义的风险。
5. #349 v0.2 快照节点段缺少节点编号，不能无损表达完整节点直接身份；#350 又形成平行快照 DTO 和通用字节段，不能证明严格强类型恢复。
6. #352 v0.2 把最终修改 / 删除白名单交给执行 S0 动态形成并等待设计接受，既违反不可变施工合同，也违反设计窗口不指挥执行窗口的现行角色规则。
7. #340—#351 v0.2 叶子把 Markdown 实施记录列入执行写范围，与执行窗口零 Markdown 的现行上限冲突。

因此，v0.2 计划整体退出当前消费，移入 `计划/过期计划/`；不是删除其历史证据。

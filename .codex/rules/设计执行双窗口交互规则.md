# 设计执行集成只读复核四角色交互规则

日期：2026-07-23

状态：Codex 代理角色交互规则；位于 `.codex/rules/`；不是项目机器规范，不构成代码实施许可

## 1. 目标

本入口把计划的设计、实施、集成发布和独立事实复核拆成权限互斥的设计、执行、集成、只读复核四个角色。“制定窗口”统一称为“设计窗口”。设计窗口先判定真实依赖，优先把提供者—消费者条件冻结成精确的待实现接口合同，并隔离文件、结构、接口和装配所有权；没有未合同化真实依赖且满足并行门禁的计划可以由多个执行窗口在独立任务 worktree 中并发形成实现候选，也可以由一个执行窗口在长期执行通道中逐份连续施工。长期执行通道只绑定一次顶层执行身份、持久 worktree 和持久分支，不为每份计划重新建立任务身份、R2、派发状态或执行许可；用户明确指定时，也允许同一物理窗口按正式提交边界串行切换角色。

本规则不属于 `规范/规范目录.md` 登记的项目机器规范，不定义机器事实或代码语义；其约束力来自 `AGENTS.md` 对角色边界、子智能体 S0、执行自主接管、交接状态机、整树切换和任务工具规则的指定入口。相关角色必须遵守本规则，但不得据此覆盖现行机器规范、补造业务规则或扩大计划许可。worktree 路径、分支、冻结、Git 身份、集成实施和回收的技术合同只由 `.codex/rules/多工作树并发与集成规则.md` 定义。

拆分目标：

```text
设计窗口只负责高不确定性的规范、设计、计划和计划修订，不指挥其它角色；正式发布或修订后向已建立消息路由的受影响执行窗口发送结构化设计提醒。
执行窗口负责自主发现自身范围内的成熟计划、完成 S0、实施和验证，并在计划段选择、S0、完成、漂移或失败时以结构化消息提醒设计窗口。
集成窗口负责核对任务提交范围、汇集提交、运行集成验证和发布 main。
只读复核窗口负责在指定提交和指定对象上独立核查事实、缺口与完成声明，不写仓库。
四个角色分别沿当前 Git 事实和计划合同自主推进；设计窗口持续形成后续计划，不等待执行完成，也不承担调度者。
计划索引以列表形式记录计划及每份计划的状态，叶子计划保存不可变施工合同；计划之间只登记具名依赖，不另建层级结构；worktree、分支、HEAD、提交和 clean 状态直接读取 Git。
结构化消息承载设计发布提醒及当前计划段开始、漂移、失败和完成事实；这些消息不再复制到 Markdown 状态文件，也不替代正式仓库与 Git。
```

## 2. 交接证据读取顺序

本节只规定角色接管、状态核对和消息追溯时的证据读取顺序，不定义项目全局权威层级，也不得据此让当前代码、Git 状态、计划或消息覆盖正式规范。项目全局权威顺序、机器语义来源和代码实施依据链唯一由 `AGENTS.md` 定义。

```text
现行正式规范
-> 计划/计划索引.md
-> 目标计划、流程图和详细设计
-> Git 正式提交、当前 worktree / 分支和实际接口
-> 最新结构化执行 / 集成 / 复核消息
-> 聊天摘要和窗口自身上下文
```

该顺序用于先核对正式合同，再判断计划状态、真实依赖、Git 技术身份和实际接口是否一致。计划索引独占计划列表、路径、版本、具名依赖和计划状态；执行通道由本规则、Git 和当前顶层任务身份裁决；Git 直接裁决 worktree、分支、HEAD、提交、dirty state 与远端事实；结构化消息提供当前计划段结果。它们不一致时必须返回具名 DRIFT，不得用聊天上下文或历史 Markdown 覆盖 Git 与计划合同，也不得把实现现状反向升级为规范。

## 3. 文件写入所有权

设计窗口独占规范、流程图、详细设计、计划和计划索引的写入。执行窗口只在目标计划指定的独立任务 worktree 或永久执行通道中自主写当前计划允许的代码、工程和自检，并以结构化消息交接；集成窗口独占集成 worktree 的提交汇集、集成记录和主线发布；只读复核窗口无仓库写入权。

| 文件层 | 唯一写入角色 | 其它角色权限 |
| --- | --- | --- |
| `计划/计划索引.md`、`计划/*.md`、`计划/已完成计划/` | 设计窗口 | 执行窗口只读 |
| 规范、流程图、详细设计和必要设计辅助材料 | 设计窗口 | 执行窗口只读 |
| 计划允许范围内的 C++、工程文件和自检 | 执行窗口 | 设计窗口只读当前事实 |
| 当前计划段 S0、执行结果、漂移、失败或退回事实 | 执行窗口只输出结构化消息和 Git 提交 | 不写任何 Markdown 状态文件 |
| 并行批次集成记录与集成暂存分支 | 集成窗口 | 设计窗口和执行窗口只读；冲突时退回，不代改业务语义 |
| 指定基线上的只读复核结论 | 只读复核窗口只输出消息报告 | 设计窗口读取并裁决；需要改变规则或计划时直接修订对应权威文件 |

设计窗口不得借治理身份修改代码、工程或自检，也不得向执行窗口发出开始、继续、停止、恢复、选择下一项或接受 S0 的指令；但在正式发布计划、修订计划或改变计划状态、合同、依赖与影响范围后，必须按第 9 节向已建立消息路由的受影响执行窗口发送结构化设计提醒。执行窗口不得新建、修改、移动、删除、暂存或提交任何 Markdown，也不得借完成、退回或修复状态修改计划、计划索引、流程图或详细设计；集成窗口不得借冲突处理发明接口、修改计划或扩大任务范围；只读复核窗口不得把报告直接写成计划完成、依赖解除或代码许可。旧状态文件和执行记录只保留历史事实，不再读取、更新或参与当前裁决。

### 3.1 顶层任务树单写角色

顶层任务是平台关系中不属于任何协作代理后代的独立用户可见任务，包括用户直接打开或创建的任务，以及用户授权后由 Codex 任务工具创建并交由用户独立接管的任务。一个顶层任务及其直接、递归创建的全部协作代理共同构成一个顶层任务树。顶层身份按平台父子关系和协作代理关系裁决，不按标题、消息来源、worktree、分支、模型、智能体名称或文件路径裁决。

一个顶层任务树同一时刻最多只能拥有设计、执行、集成中的一种写角色；只读复核是第四种协作角色，但不是写角色。全部有写入权的后代只能继承该写角色；只读后代可以并存，但必须保持零修改、零构建、零运行、零 Git 写入、零活动计划段所有权和零发布占用。顶层任务只读时整树只能只读。worktree、分支、模型、推理等级、名称和递归层级均不能改变角色、范围、所有权或许可。

各角色可以分配给后代的切片固定为：

| 顶层任务树角色 | 同角色写切片 | 只读切片 | 禁止 |
| --- | --- | --- | --- |
| 设计 | 互不相交的规范、流程图、详细设计、计划和治理状态切片 | 规范冲突、代码事实、引用和差异核对 | 执行代码、构建、集成或发布 |
| 执行 | 计划允许且所有权不相交的代码、工程、自检或验证切片 | 接口、差异、日志和证据核对 | 修改任一 Markdown、规范、计划、计划索引或承担集成角色 |
| 集成 | 登记范围内不相交的汇集检查、集成记录和验证切片 | 父链、文件范围、验证输出和主线漂移核对 | 发明接口、修改计划、扩大范围或承担设计角色 |
| 只读 | 无 | 指定提交和问题集上的事实采集 | 任一后代修改、构建、运行或执行 Git 写操作 |

出现两个以上边界明确、无真实依赖且所有权可以隔离的具体子任务时，应优先使用同角色或只读后代并行。不使用时必须具名记录任务不可再分、真实前后依赖、所有权无法隔离、并发槽不足、工具只能串行、协调成本过高或风险不能隔离中的实际原因。不得制造空任务、重复读取、无意义代理链、相互等待或让多个智能体竞争同一原子步骤。

### 3.2 子智能体派发合同与 S0

每次派发协作代理时，派发方必须给出：

```text
顶层任务身份及平台父子 / 协作关系；
继承的设计、执行、集成写角色或只读身份；
具体目标、返回条件和唯一切片所有权；
允许文件、结构、接口、验证或证据范围；
明确禁止项，以及是否允许修改、构建和运行；
Git 暂存、提交和推送权限；
执行通道、活动计划段、worktree 与主线发布占用权限；
冲突、漂移、失败、重叠或新用户裁决的停止条件。
```

协作代理开始写入或执行授权动作前必须完成最小 S0：

```text
确认所属顶层任务树、平台关系和继承角色；
确认目标、允许范围、禁止范围和返回条件；
确认目标路径、分支、HEAD 和 dirty state（适用时）；
确认与其它已派发切片没有所有权交集；
确认 Git、执行通道 / 活动计划段、worktree 和发布占用权限；
向派发方返回 PASS 或具名 DRIFT。
```

未完成 S0 或 S0 为 DRIFT 时不得开始写入。递归派发必须原样传递顶层身份、角色、范围、所有权、Git / 占用权限和停止条件，不得在下级派发中扩大权限。任务消息只负责唤醒，聊天回执不得替代计划合同、Git 事实或完成事实。

### 3.3 所有权、串行收口与违规停止

1. 顶层任务树是执行通道、任务 worktree 和具体切片的正式协作所有者；协作代理只取得派发合同列出的具体切片所有权。
2. 同一文件、结构、接口、验证项目或证据裁决同一时刻只有一个智能体负责。发现交集时必须先重新拆分或改为串行。
3. Git index、暂存、提交、推送、执行通道 / 活动计划段单写所有权迁移和主线发布占用迁移，由顶层智能体或唯一指定的同角色智能体串行完成；收口期间其它后代停止写入。
4. 顶层智能体必须复核后代的实际差异和验证证据；后代返回只证明其受派切片的结果，不证明顶层任务、集成或发布完成。
5. 范围漂移、所有权重叠、角色冲突、计划外文件、验证失败或用户新裁决只停止受影响切片并上报；不得自行扩权。若同一顶层任务树已同时拥有两种写角色或通过 worktree、分支、模型、名称规避角色，整树立即停止新增写入，保留现有 Git 事实并退回设计治理追查根因。
6. 无关 WIP 属于原所有者，任何后代不得暂存、清理、stash、覆盖或带入本任务提交。

## 4. 计划列表状态与过程事实分账

`计划/计划索引.md` 是计划设计可用性的唯一所有者，枚举固定为：

```text
待激活：精确计划版本 / blob 正在等待设计修订，执行窗口本轮跳过。
可执行：精确计划版本 / blob 已完成设计，可由目标执行 worktree 或永久执行通道自主核对。
已退出：该计划版本不再参与后续消费，只作历史导航。
```

`可执行`不是命令，不代表已经开始、S0 PASS、已经形成活动计划段或已经完成。任务过程不设置持久 Markdown 生命周期枚举；执行、漂移、失败、候选完成、集成和发布事实由对应角色的最新结构化消息与 Git 共同证明。执行窗口报告缺口后，设计窗口把旧版本改为 `待激活`，形成新版本 / blob 后恢复 `可执行`，并向已建立消息路由的受影响执行窗口发送结构化设计提醒；执行窗口仍必须通过重读仓库自行确认变化，设计窗口不得把提醒写成继续、停止或选择下一项指令。

## 5. 设计窗口设计发布协议

设计窗口只完成以下工作：

```text
1. 生成或修订规范、流程图、详细设计、知识图谱和计划。
2. 核对依赖必要性、合同化、所有权隔离、允许 / 禁止文件、入口拒绝、失败收口和验证方式。
3. 在计划索引的计划列表中登记稳定路径、版本、计划 blob、具名依赖与计划状态 `待激活 / 可执行 / 已退出`。
4. 需要预冻结接口时，登记模块、DTO、签名、ABI、所有权、生命周期、逻辑内结果、内部错误、提供者、消费者和漂移退回点。
5. 运行文档治理验证，精确提交并非强制推送设计文件。
6. 向已建立消息路由的受影响执行窗口发送 `DESIGN-NOTICE`；发送失败时保留已发布 Git 事实并幂等重试。
7. 继续设计下一份计划，或优先响应已经收到的设计修订请求。
```

到第 6 步本轮设计交接完成。设计窗口不得继续执行下列动作：

```text
向执行窗口发送开始、继续、停止、恢复、选择下一计划或接受 S0 的消息；
把任务消息送达、目标 active、worktree 存在或计划索引更新解释为执行许可；
要求执行窗口等待设计窗口接受 PASS；
因为前一计划尚未执行而停止形成后续无真实依赖计划；
用“派发准备”“派发状态”或具名执行序列控制执行时点。
```

设计窗口可以主动发送第 9 节定义的 `DESIGN-NOTICE`，也可以回答执行窗口提出的计划含义问题、提供正式路径或确认已经发布的设计版本；消息必须明确“仅为设计事实提醒，不构成开始、继续、停止或计划选择指令”。

## 6. 执行窗口自主接管与下一计划选择

执行窗口必须读取尚未处理的设计提醒，但不等待设计消息才开始检查计划；只在目标计划指定的独立任务 worktree 或永久执行通道和文件所有权范围内按下列顺序工作：

```text
读取尚未处理的 DESIGN-NOTICE，只把它作为重读提示
-> 重新读取 Git、当前 worktree / 分支、当前 HEAD、upstream、远端和 dirty state
-> 读取计划索引与目标计划
-> 从自身所有权范围内排除状态为 `待激活` 或 `已退出`、真实依赖未满足和版本 / blob 不一致项
-> 根据计划状态自主选中一份 `可执行` 计划；多个无依赖计划可由不同独立执行任务并行，长期通道内只能逐份选择
-> 核对当前工作区没有第二活动计划段，固定本段起点、计划 blob、允许 / 禁止文件和实际接口
-> 自行完成角色、Git、计划 blob、计划段起点、所有权和接口 S0
-> PASS：向设计窗口发送计划段选择与 S0 PASS 的 EXECUTION-NOTICE，随后直接实施；不等待设计接受
-> DRIFT：停止该计划，发送 S0 DRIFT 与精确修订请求的 EXECUTION-NOTICE，然后检查其它无关 `可执行` 计划
```

执行窗口不得夺取其它执行通道、worktree、文件或结构所有权，也不得把设计窗口的消息、聊天解释或旧 `PASS已接受` 标签当作当前许可。同一长期执行通道串行处理多份计划时，上一份必须正常收口，或把计划允许的代码 WIP 固定成具名提交并恢复 worktree / index clean；随后即可自行检查下一份，不需要设计窗口确认冻结、重新建立 R2 或再次通知。无法形成合法代码断点且工作区仍 dirty 时，通道停在当前计划，不得覆盖现场继续下一份。

## 7. 执行完成与退回协议

执行窗口发现接口漂移、范围越界、验证失败或完成切片时，不得修改任何 Markdown、计划和路由文件。必须：

```text
1. 停止扩大受影响实施范围。
2. 运行计划在当前阶段要求且执行角色获准的验证；完整构建 / 运行 / 闭环验收明确后置时只做候选级最低检查。
3. 只提交执行角色允许的代码、工程和自检，并非强制推送同名任务分支；不得暂存或提交 Markdown。
4. 向设计窗口发送结构化完成、漂移、验证失败或计划修订请求，作为设计修订与事实登记输入。
5. 自主检查其它无关 `可执行` 计划，不等待设计窗口回复；只有当前所需设计产物尚未形成时才跳过该计划。
```

结构化执行消息至少包含计划路由编号和版本、计划 blob、执行通道或任务 worktree、计划段起点、当前分支、实际接口、实际变更文件、验证结果、结果或 WIP 断点提交、远端 / dirty state、完成边界和需要修订的精确合同。消息发送失败时保留 Git 现场并幂等重试；不得为制造持久化回执而写 Markdown。

## 8. 设计窗口响应协议

设计窗口收到执行消息后只执行设计职责：

```text
1. 只读核对消息、Git、计划、实际接口和影响范围。
2. 需要修订时把旧计划版本改为 `待激活`，形成新版本 / blob，并在完成后恢复 `可执行`。
3. 需要新增前置计划时登记真实依赖；可合同化时冻结接口并取消开发启动门控。
4. 运行文档治理验证，精确提交并非强制推送。
5. 向来源执行窗口发送 `DESIGN-NOTICE`，说明已发布状态、版本、blob、合同与影响范围；不得写入继续、停止、接受 S0 或下一项指令。
6. 继续总目标下其它计划设计，不等待执行窗口完成上一项。
```

设计窗口必须回告已发布的设计事实，但不得在响应结尾通知执行窗口“继续下一项”、要求其停止、指定执行顺序或接受 S0。执行窗口收到提醒后仍通过重读正式仓库确认新计划版本和 `可执行` 状态。

### 8.1 计划持续推进闭环

```text
设计窗口持续形成并发布计划
||
执行窗口自主消费自身范围内的可执行计划并返回事实 / 缺口
||
集成窗口按固定输入和主线占用规则自主汇合
-> 设计窗口只根据新事实继续修订规范、设计和后续计划
```

并发的目标是避免空等待：设计窗口不等待代码执行结束才设计下一份；执行窗口不等待设计窗口回复才检查其它无关计划。任何角色都不能因此跨越自身写角色、真实依赖、文件所有权、主线占用或验证边界。

## 9. 消息规则

设计窗口与执行窗口必须在已经建立消息路由的对应顶层任务之间双向提醒。消息只减少对方发现延迟，不改变权威顺序：

```text
DESIGN-NOTICE
来源设计任务与中央设计提交
受影响计划路由、路径、版本、blob 和计划状态
首次发布 / 转待激活 / 修订恢复可执行 / 已退出 / 合同、依赖、范围或角色交互规则变化
影响中的提供者、消费者、活动计划段或候选计划
“仅为设计事实提醒，接收方须重读正式仓库并自行 S0；不构成开始、继续、停止、计划选择或 S0 接受”

EXECUTION-NOTICE
来源执行任务、执行通道或任务 worktree
计划路由、版本、blob、计划段起点、分支、HEAD、upstream 和 dirty state
计划段选择与 S0 PASS / S0 DRIFT / 完成 / 验证失败 / WIP 断点 / 计划修订请求
实际接口、变更文件、验证、结果或 WIP 提交、完成边界和精确缺口
“仅为执行事实与设计输入，不要求设计窗口许可下一步”
```

设计窗口在下列事件已经正式提交后必须发送 `DESIGN-NOTICE`：计划首次转为 `可执行`；受影响版本转为 `待激活` 或 `已退出`；修订版本恢复为 `可执行`；计划 blob、合同、依赖、允许 / 禁止范围或验证合同发生变化；`AGENTS.md` 或本规则中的角色交互、消息和执行边界发生变化。执行窗口在选中计划并完成 S0，以及计划完成、漂移、失败或形成合法 WIP 断点时必须发送 `EXECUTION-NOTICE`。接收方不需要用接受或批准回复，避免消息确认形成第二状态机。

提醒到达执行窗口时不得仅凭消息中断当前计划。若提醒指向当前活动计划的撤回、版本或合同变化，执行窗口立即重读正式仓库和 Git，并依据正式差异停止受影响切片；若只涉及尚未开始的计划，则保留为下一次选项扫描提示，不打断当前活动计划。设计窗口收到执行提醒后按第 8 节核对和修订，不得把消息内容未经核对直接写成正式事实。

“已经建立消息路由”指当前任务上下文中存在可验证的来源任务 / 线程标识或平台可达的对应窗口。不存在对应窗口时，不得为了发送提醒自动创建、派发或唤醒一个新的执行任务；正式计划仍由计划索引公开发现。发送失败不改变已提交设计或 Git 事实的效力，但发送义务保持未完成，发送方必须保留精确消息并幂等重试或在本轮报告中具名说明不可达。

客户端 Steer / Queue、消息送达顺序、目标 active 状态和聊天回执均不得改变当前计划范围、执行通道所有者或活动计划段。

正式并行批次运行期间，设计窗口和执行窗口可以在文件与 Git 身份完全隔离时同时落盘；任一所有权边界失效只停止受影响任务。长期执行通道与 WT-MAIN 必须是不同 worktree / 分支，通道分支允许累计多个计划段提交，但一个通道同一时刻只有一个活动计划段。集成任务取得主线发布占用后，设计窗口保持 WT-MAIN 只读，直到集成角色实际释放占用。

## 10. 同一物理窗口串行双角色

只有用户明确启用时才能进入单窗口串行切换；该模式以 `AGENTS.md` 定义的整棵顶层任务树为单位，全部后代必须先停止并释放旧角色占用，不能让顶层窗口和后代分别保留不同写角色。单窗口模式不合并角色权限：

```text
执行角色完成或退回
-> 只提交计划允许的代码 / 工程 / 自检并发送结构化消息
-> 验证、提交、推送
-> 推送成功后切换设计角色并重读全部正式状态
-> 由设计角色更新路由

设计角色完成设计修订
-> 更新规范、详细设计、计划或计划索引中的唯一权威内容
-> 验证、提交、推送
-> 推送成功后切换执行角色并重读计划索引、目标计划和 Git 事实
```

执行转设计以结构化消息和已推送任务提交为依据；设计转执行只允许由用户明确启用整树串行切换，并以已推送设计提交、不可变计划 blob 和实际 Git/worktree 为重读依据。切换后的执行角色自行选择当前所有权内的 `可执行` 计划并完成 S0，不得把先前设计角色的上下文或消息当作开始命令。需要继续使用子智能体时，只能按新角色重新建立同角色 / 只读所有权；未验证草案不得跨角色继承。

## 11. 模型路由与异常升级

默认建议：

| 工作类型 | 模型 / 推理建议 |
| --- | --- |
| 设计、计划审查、接口漂移和逻辑异常裁决 | Sol + High，或可用的同等高推理配置 |
| 成熟计划常规实施、验证和结构化回执 | Terra + Medium，或可用的同等中等推理配置 |
| 明确、重复、只读的信息整理 | Luna / 低消耗等效配置 |

模型选择不改变角色权限。执行窗口发现接口漂移、计划冲突、事务 / 并发 / 因果边界无法由计划裁决、内部逻辑错误，或验证重复失败且根因不明时，必须停止并退回高推理的设计窗口；不得只通过提高执行窗口模型等级自行改计划或扩大范围。

### 11.1 中等模型可机械执行合同

面向中等推理模型的成熟计划必须按以下四列写清每个阶段：

```text
已验证前置事实
本阶段待实施目标
实施完成的可核对证据
只有前置证据成立后才可判断的完成 / 退回条件
```

硬规则：

1. 待实施目标不存在，默认表示该阶段尚未完成，不表示接口漂移、计划冲突或真实阻塞。
2. 退回条件必须引用已经形成的具体文件、符号、构建输出、具名失败项或接口差异；不得用“证据不足”“代码较长”“尚未形成目标产物”替代前置证据。
3. 同一输出在目标实施前只可作为基线输入，在目标实施后才可作为验收证据；不得把实施前失败套入实施后退回门禁。
4. 执行模型误读明确阶段、跳过授权步骤、把目标当门禁或连续两轮不能按检查表推进，执行窗口返回模型能力不匹配事实；设计窗口只收窄或澄清计划，不指定执行窗口继续或切换模型。
5. 执行轮次因平台边界结束但尚未命中计划的正式完成 / 退回条件时，执行窗口自行保留精确续点并保持 `执行中`。不得仅因轮次结束写 `分支完成待集成`、`接口漂移待设计修订` 或 `验证失败待设计裁决`。

高推理模型进入执行窗口后仍只是执行角色，只能实施已发布计划；不得修改计划或计划索引，也不得借模型等级扩大范围。

## 12. Codex 任务工具规则

```text
设计窗口可以用任务工具读取执行事实、回复执行窗口主动提出的设计问题，或向已经建立消息路由的受影响执行窗口发送 DESIGN-NOTICE。
设计窗口不得用任务工具发送开始、继续、停止、恢复、计划选择、S0 接受或模型切换指令。
执行窗口收到 DESIGN-NOTICE 后仍通过重读正式仓库确认计划版本和 `可执行` 状态；消息不是计划发现、活动计划段建立或实施的前置。
运行中的设计说明消息不得用 Steer 改变当前切片。
线程 / 任务标识只作运行期路由材料，不作为正式计划身份。
消息失败不改变已推送设计文件的效力，也不授权任何角色扩大范围；发送方必须保留精确消息并幂等重试或报告不可达。
协作代理只能继承当前顶层任务树的写角色或保持只读。
```

## 13. worktree 与 Git 的角色入口

本规则不定义 worktree 物理路径、分支格式、冻结基线、Git 身份算法、集成汇集步骤、失败保留或回收条件，只规定它们进入角色交接时的门禁：

1. 设计计划引用既有独立任务 worktree 或永久执行通道时，只登记预期分支、基线、文件 / 结构所有权和验证合同；实际路径与 Git 身份在执行 S0 直接核对，不得据此产生执行许可。
2. 执行窗口在本规则角色 S0 与多工作树规则 Git 身份复核都通过后自主进入当前计划段写状态；长期通道不因尚未开始计划的版本变化自动重建，只有通道基线、累计代码父链或共享接口失配才更换通道代次。集成窗口还必须核对内容冻结基线、分支建立起点和中央路由尾，并自行取得唯一主线发布占用。
3. Git index、暂存、提交、推送和主线发布占用仍由本顶层任务树的唯一指定智能体串行收口；其它后代在收口期间停止写入。
4. 路径、分支、基线、父链、dirty state 或远端状态漂移时，停止受影响计划并发送具名 DRIFT；消息不得把漂移改写为成功。

上述技术合同唯一见 `.codex/rules/多工作树并发与集成规则.md`；本文件不复制其分支命名、身份核对、冻结、集成、验证、失败和回收正文。

## 14. 验证

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

顶层任务树收口还必须核对：全部后代已经结束或明确保持只读；树内没有两种写角色；所有权交集为零或已串行消解；Git 收口只包含本轮目标文件；无关 WIP 未被暂存或改写；提交、推送和完成声明与实际证据一致。

若本轮同时修改技能，还必须按对应技能文件中真实存在的专项验证入口执行；不存在的脚本不得写成强制验证。规则文档验证不证明代码编译、入口拒绝、业务闭环或任何代码能力完成。

## 15. 完成边界

本规则只证明设计、执行、集成和只读复核四角色的写入所有权、自主接管、交接状态、消息边界、主线发布占用、独立复核和计划持续推进规则已经建立。worktree 分配、Git 身份、集成实施、失败保留和回收由多工作树规则唯一裁决。它不证明任何代码计划已经实施、构建通过、业务闭环完成或旧项目能力已经等价重建。

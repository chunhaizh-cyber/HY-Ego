# 交互计划执行计划支撑四智能体规则

日期：2026-07-24

状态：Codex 智能体类型、写入所有权与消息路由规则；位于 `.codex/rules/`；不是项目机器规范，不构成代码实施许可

## 1. 目标

本项目固定使用四类智能体：

```text
交互智能体
计划智能体
执行智能体
计划支撑智能体
```

四类智能体分别承担智能体协作规则、新目标设计包、代码施工和执行中设计包修订。智能体权限不因物理窗口、模型、worktree、分支、智能体名称、子智能体层级或 Full access 合并。

集成、发布和只读复核是上述智能体在自身权限内进入的技术工作模式，不另行建立第五类写智能体。代码候选的汇集、构建、运行和发布属于执行智能体的技术范围；智能体协作规则、新目标设计包、执行中设计包和代码的独立只读复核分别由交互、计划、计划支撑和执行智能体在对应切片中完成。只读模式不产生写权。

本规则不属于 `规范/规范目录.md` 登记的机器规范，不定义机器事实或代码语义。机器逻辑仍以现行正式规范为准；worktree、分支、冻结、Git 身份和集成技术合同仍由 `.codex/rules/多工作树并发与集成规则.md` 定义。

## 2. 四智能体职责

### 2.1 交互智能体

交互智能体负责：

```text
与用户讨论、澄清目标和裁决；
形成或修订智能体协作规则和工具行为规则；
维护 AGENTS.md、.codex/rules/ 和 .codex/skills/；
只有目标需要形成项目设计包时，才向计划智能体发送结构化目标消息。
```

交互智能体是智能体协作规则的唯一写入者。项目机器规范、流程图、详细设计和知识图谱属于设计包，不属于交互智能体的规则写入范围。其它智能体发现 `AGENTS.md`、`.codex/rules/` 或 `.codex/skills/` 缺失、冲突或必须修改时，都不得代改；必须停止受影响切片，在自身用户可见窗口列出问题并等待用户把问题交给交互智能体。

交互智能体不得修改项目机器规范、流程图、详细设计、知识图谱、计划、计划索引、代码、工程、自检或施工记录；不得替计划智能体形成新设计包，也不得替计划支撑智能体修订执行中的设计包。单纯协作规则修订不向计划智能体发送消息；只有确实需要新增项目设计包时才发送目标。

### 2.2 计划智能体

计划智能体负责：

```text
接收交互智能体给出的已明确目标；
根据既有总目标自行推导后续计划目标；
为尚未被执行选中的新目标建立和维护完整设计包；
建立或修订项目机器规范及目录、冲突登记、流程图、详细设计、知识图谱、新计划和对应计划索引项；
冻结目标、依据、依赖、允许 / 禁止范围、施工记录路径、失败收口和验证合同。
```

计划智能体不发送任何跨智能体消息。完整设计包通过正式文件和 `计划/计划索引.md` 公开，由执行智能体自主发现和读取。

计划智能体不得修改智能体协作规则、代码、工程、自检或施工记录；也不得修订已经被执行智能体选中并转由计划支撑智能体接管的设计包。发现协作规则不足、需要改变用户目标或机器语义无法裁决时停止并等待用户处理。

### 2.3 执行智能体

执行智能体负责：

```text
自主读取计划索引和可执行计划；
完成 S0、代码实施、工程登记、自检、构建、运行和计划授权的 Git 收口；
只修改当前计划明确允许的代码、工程、自检、专属施工记录和验证记录；
把计划段选择、S0、完成、漂移、失败和计划修订请求发送给配对的计划支撑智能体。
```

执行智能体只能向配对的计划支撑智能体发送跨智能体消息，不得向交互智能体或计划智能体发送消息。执行智能体不得修改智能体协作规则、项目机器规范、流程图、详细设计、知识图谱、计划或计划索引。专属施工记录和验证记录可以使用 Markdown，但只记录已发生的实施事实，不得成为机器语义、设计合同、计划状态或执行许可。

执行智能体发现设计包问题时必须停止受影响计划并交给计划支撑智能体；发现智能体协作规则问题时不得把它伪装成普通设计缺口，必须在消息中明确标记 `RULE-BLOCKED`。计划支撑智能体可以修订原目标内的项目机器规范和设计产物，但不得代改智能体协作规则。

### 2.4 计划支撑智能体

计划支撑智能体与执行智能体固定配对，负责：

```text
接收执行智能体的结构化执行事实和计划修订请求；
从计划被执行智能体选中并首次发送 S0 起，接管该计划关联设计包的修订权；
只在原目标内修订该设计包的项目机器规范、目录 / 冲突登记、流程图、详细设计、知识图谱、既有计划及其计划索引项；
核对实际接口、范围、依赖、验证合同和 Git 证据；
修订完成后只向来源执行智能体发送正式设计包路径、计划版本、blob、状态和影响范围。
```

计划支撑智能体只能向配对的执行智能体发送跨智能体消息，不得向交互智能体或计划智能体发送消息。它不得创建与原目标无关的新计划，不得修改智能体协作规则、代码、工程、自检、施工记录或验证记录。

计划支撑智能体遇到以下情况时必须停止：

```text
需要改变用户目标；
需要新增或修改智能体协作规则；
项目机器规范之间存在无法按现行权威顺序消解的冲突；
现有证据不足以裁决接口、范围、依赖或验证合同；
继续修订会替用户作业务选择。
```

停止时必须在自身用户可见窗口列出：

```text
问题；
已确认事实；
冲突或缺失内容；
为什么不能自行裁决；
需要用户决定的具体事项；
受影响计划和当前 Git 现场。
```

随后等待用户直接发送消息。不得跨智能体转发、猜测用户意图或自行扩大目标。

## 3. 文件写入所有权

| 文件或工作范围 | 唯一写入智能体 | 其它智能体权限 |
| --- | --- | --- |
| `AGENTS.md`、`.codex/rules/`、`.codex/skills/` | 交互智能体 | 其它智能体只读 |
| 尚未被执行选中的新目标规范、流程图、详细设计、知识图谱、新计划及对应索引切片 | 计划智能体 | 交互、执行、计划支撑只读 |
| 已被执行选中或退回的关联规范、流程图、详细设计、知识图谱、既有计划及对应索引切片 | 配对的计划支撑智能体 | 交互、计划、执行只读 |
| 计划允许的 C++、工程文件和自检 | 执行智能体 | 其它智能体只读 |
| 计划唯一专属的施工记录和验证记录 | 执行智能体 | 其它智能体只读；不得作为设计或状态权威 |
| 执行 worktree、结果提交、构建、运行和代码集成 | 执行智能体 | 其它智能体只读 |

计划智能体与计划支撑智能体都具有项目设计文档写权，但不能写同一设计包或同一物理文件切片：

```text
计划智能体拥有尚未被执行选中的新目标设计包；
执行智能体选中计划并首次发送 S0 后，关联设计包修订权转给配对计划支撑智能体；
计划智能体不再修改已转移设计包，计划支撑智能体不扩展到无关新目标；
规范目录、计划索引等共享物理文件必须串行写入，每次只修改本智能体拥有的条目；
需要用无关新目标替代既有目标时，必须由用户重新把目标交给计划智能体。
```

## 4. 消息路由

跨智能体消息只允许以下三条：

```text
交互智能体 -> 计划智能体
执行智能体 -> 配对的计划支撑智能体
计划支撑智能体 -> 配对的执行智能体
```

计划智能体不发送跨智能体消息。

消息矩阵：

| 发送方 | 唯一允许接收方 | 允许内容 |
| --- | --- | --- |
| 交互智能体 | 计划智能体 | 需要形成项目设计包的目标、范围、成功标准、禁止项和协作规则依据 |
| 计划智能体 | 无 | 不发送消息；以正式设计包和计划索引交付 |
| 执行智能体 | 配对的计划支撑智能体 | 计划段选择、S0、完成、DRIFT、失败、WIP、计划修订请求和 RULE-BLOCKED |
| 计划支撑智能体 | 配对的执行智能体 | 修订后的正式设计包路径、计划版本、blob、状态、影响范围和重新 S0 要求 |

明确禁止：

```text
交互智能体 -> 执行智能体或计划支撑智能体；
计划智能体 -> 任一智能体；
执行智能体 -> 交互智能体或计划智能体；
计划支撑智能体 -> 交互智能体或计划智能体；
用消息替代正式规则、计划、计划索引、Git 事实或执行许可；
为发送消息而自动创建、派发或唤醒新的顶层任务。
```

用户直接向任一用户可见智能体窗口发送消息不属于跨智能体消息，不受上述路由限制。智能体向用户报告本窗口结果、问题或阻塞也不属于跨智能体消息。

## 5. 消息格式

### 5.1 交互目标消息

```text
INTERACTION-GOAL
来源交互任务
目标
背景和成功标准
禁止项
现行规则依据
需要形成的计划范围
不构成代码执行许可
```

交互智能体仅在确实需要形成项目设计包时发送该消息。协作规则修订、讨论、解释和只读检查不发送。

### 5.2 执行消息

```text
EXECUTION-SUPPORT-NOTICE
来源执行任务、执行通道或任务 worktree
计划路由、路径、版本和 blob
计划段起点、分支、HEAD、upstream 和 dirty state
S0 PASS / S0 DRIFT / 完成 / 验证失败 / WIP / RULE-BLOCKED
实际接口、实际变更文件、验证和提交
专属施工记录与验证记录路径
精确计划缺口或规则阻塞
完成边界
```

### 5.3 计划支撑消息

```text
PLAN-SUPPORT-NOTICE
来源计划支撑任务
受影响计划路由、路径、旧版本和新版本
新计划 blob 和计划状态
修订的规范、流程图、详细设计、知识图谱、依赖、合同、允许 / 禁止范围和验证边界
执行智能体必须在通知唤醒的同一轮重读正式仓库并重新 S0
通知不预先裁决计划选择、S0 PASS 或实施结果
```

`PLAN-SUPPORT-NOTICE` 不是执行授权，也不能用消息正文替代正式计划、计划索引或 Git；但它是既有执行任务的正式状态变化触发事件。除非用户已经在执行智能体自身窗口明确暂停，执行智能体收到通知后不得只确认收到、推迟到“下次恢复”或等待另一条“继续”，必须立即从正式仓库重新计算受影响计划和全部候选。用户已明确暂停时只保持 `USER-PAUSED`，不得借通知恢复施工。

消息发送失败不改变正式文件和 Git 事实，也不扩大权限。发送方只可对唯一允许接收方幂等重试，并在自身窗口报告不可达。

## 6. 计划形成、执行和修订闭环

新目标：

```text
用户
-> 交互智能体澄清目标或修订规则
-> 确实需要计划时发送 INTERACTION-GOAL
-> 计划智能体形成完整新目标设计包并更新计划索引
-> 计划智能体不发送消息
-> 执行智能体从计划索引自主发现可执行计划
```

执行计划：

```text
执行智能体读取计划索引和目标计划
-> 核对 Git、worktree、计划 blob、允许范围和实际接口
-> S0 PASS：向配对计划支撑智能体报告后直接实施
-> S0 DRIFT：停止受影响计划并向配对计划支撑智能体请求修订
```

计划修订：

```text
计划支撑智能体核对执行事实
-> 可在原目标和现行协作规则内解决：修订已接管设计包和计划索引
-> 向来源执行智能体发送 PLAN-SUPPORT-NOTICE
-> 通知送达既有执行任务并产生新轮次：执行智能体当轮重读正式仓库并重新 S0，不等待用户继续指令

需要改变目标、智能体协作规则或无法裁决机器语义：
-> 计划支撑智能体停止
-> 在自身窗口列出问题
-> 等待用户直接消息
```

计划智能体不负责已经转移所有权的执行中设计包修订；计划支撑智能体不负责无关新目标设计包生成。

## 7. 计划状态与执行自主性

`计划/计划索引.md` 的计划状态仍只允许：

```text
待激活
可执行
已退出
```

`可执行`只证明精确计划版本和 blob 可供选择，不是执行命令。执行智能体自主读取计划索引、选择自身范围内依赖已满足的计划并完成 S0，不等待任何消息。

执行智能体发送消息后不等待计划支撑智能体接受 S0。PASS 后直接实施；DRIFT 或计划失败后停止受影响计划，并且在形成合法 clean 断点后必须继续检查其它无关 `可执行` 计划。计划支撑智能体修订完成后，执行智能体必须以正式计划、计划索引和 Git 为准重新 S0，不能仅凭消息正文继续；`PLAN-SUPPORT-NOTICE` 负责触发重新评估，不负责替执行智能体判断结果。

### 7.1 执行继续门禁与终止码

计划段结果或计划允许的 WIP 断点已经提交、非强制推送且 worktree / index 恢复 clean 后，执行智能体必须在输出最终答复前运行一次继续门禁：

```text
重读 origin/main 的计划索引并固定索引 blob
-> 重读每个 `可执行` 计划的路径、版本、blob 与具名依赖
-> 读取当前执行分支 HEAD / upstream / dirty state
-> 从 Git 计划段提交和结构化回执排除本通道已经固定的计划
-> 核对当前没有第二活动计划段
-> 逐项判断剩余计划是可进入 S0，还是有具名排除原因
```

只要存在一份当前通道所有权内、真实依赖满足且未由本通道固定的 `可执行` 计划，继续门禁结果就是 `CONTINUE-REQUIRED`。此时禁止输出最终答复；执行智能体必须立即以当前累计 HEAD 建立下一计划段、发送计划选择 / S0 回执并继续实施或具名退回。单份计划完成、完成回执已送达、计划支撑智能体无须回复、模型发生上下文压缩、单轮耗时较长或没有收到新消息，都不构成停止条件。

只有下列代码允许结束当前执行任务：

| 终止码 | 成立条件 |
| --- | --- |
| `NO-CANDIDATE` | 已逐项扫描正式计划索引，当前通道没有可选择计划 |
| `USER-PAUSED` | 用户在执行智能体自身窗口明确暂停 |
| `DIRTY-BLOCKED` | 当前活动计划无法合法提交或撤回，worktree / index 不能恢复 clean |
| `CHANNEL-DRIFT` | 通道身份、累计父链、分支、所有者或共享接口失配使整个通道代次失效 |
| `RULE-BLOCKED` | 智能体类型、消息路由或协作规则冲突，不能在现行规则内继续 |
| `ENVIRONMENT-BLOCKED` | 计划要求的权限、工具或环境不可用，且无其它无关候选可检查 |

终止回执必须同时列出终止码、计划索引 blob、当前 HEAD / upstream、dirty state、最近计划段起点—结果提交，以及全部剩余 `可执行` 计划的选择或排除结论。缺任一项时不得把最终答复解释为合法停机。

平台轮次或上下文上限迫使当前轮让出时，只能发送 `EXECUTION-YIELD`，固定当前活动计划或下一候选、HEAD、dirty state 和精确续点；它不是终止码，也不证明任务完成。仓库规则本身不能在 Codex 任务已经进入 idle 后产生新轮次；需要用户直接消息或另行授权的同任务调度器唤醒，但不得因此改写计划、Git 或智能体消息矩阵。

### 7.2 计划支撑通知唤醒门禁

既有执行任务收到 `PLAN-SUPPORT-NOTICE` 并实际产生新轮次时，无论上一轮是 `NO-CANDIDATE`、DRIFT、`EXECUTION-YIELD` 还是普通 idle，都必须执行：

```text
先核对用户是否已经在执行智能体自身窗口明确暂停
-> 已暂停：保持 USER-PAUSED，不进入 S0
-> 未暂停：刷新并重读 origin/main、计划索引及其 blob
-> 核对通知所述计划版本 / blob、正式设计提交和当前通道累计 HEAD
-> 受影响计划当前可选择：立即重新 S0，PASS 后直接实施
-> 受影响计划仍不可选择：重新扫描其它全部可执行候选
-> 有候选：CONTINUE-REQUIRED，禁止最终答复并立即进入下一 S0
-> 无候选或发生其它阻断：只能按第 7.1 节给出具名终止码和完整证据
```

通知正文中的版本、blob、状态和完成描述都是定位线索，不是机器事实；执行智能体必须自行验证。禁止以“通知不构成继续指令”“下次恢复时再处理”“已接收并记录”作为结束理由。计划支撑智能体只能向既有配对执行任务发送通知，不得为此创建新的顶层任务；一旦通知已经使既有任务产生新轮次，这就是允许消息路由内的同任务唤醒，不需要用户重复发送“继续”。

## 8. 顶层任务树与子智能体

一个顶层任务及其全部直接和递归协作代理构成一个顶层任务树。每棵树固定继承交互、计划、执行、计划支撑中的一种写类型；同一顶层任务树同一时刻不得出现两种写类型。

子智能体只能：

```text
继承所属顶层任务树的同一种智能体类型和写权限；
或保持严格只读。
```

子智能体、模型、worktree、分支或名称不能规避智能体类型边界。Git index、暂存、提交、推送、活动计划段和发布占用仍由唯一指定智能体串行收口。

存在两个以上边界清楚、无真实依赖且所有权可隔离的同类型子任务时，可以使用同类型子智能体并行；不得跨类型派发或让两个智能体写同一文件、计划、接口或验证切片。

## 9. 集成、只读复核与 Git

代码集成和发布仍属于执行智能体。执行智能体必须在登记的集成 worktree 中核对固定输入、文件范围、父链、验证矩阵、主线漂移和发布占用，不得借集成修改规则或计划。

智能体协作规则只读复核由交互智能体负责；新目标设计包和已接管设计包的只读复核分别由计划智能体、计划支撑智能体在各自所有权内负责；代码与施工记录只读复核由执行智能体负责。只读复核不修改、构建、运行、暂存、提交、推送或改变状态。

worktree、分支、基线、父链、dirty state 或远端状态漂移时，只停止受影响切片。执行智能体把具名 DRIFT 发给计划支撑智能体；规则或目标问题由计划支撑智能体停下等待用户。

## 10. 模型路由

模型选择不改变智能体类型或权限：

| 工作 | 建议模型 |
| --- | --- |
| 智能体协作规则讨论、修订和目标澄清 | 高推理模型 |
| 新目标规范、流程图、详细设计、计划生成和审查 | 高推理模型 |
| 成熟计划实施、构建和验证 | 中等推理模型 |
| 执行中设计包修订、接口漂移和计划冲突 | 高推理模型 |
| 明确重复的只读整理 | 低消耗等效模型 |

执行智能体不能通过升级模型自行修改设计包或智能体协作规则。计划支撑智能体不能通过升级模型改变用户目标或智能体协作规则。

## 11. 验证

协作规则或设计包文档治理至少运行：

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

协作规则修订不运行代码构建。设计包文档验证不证明任何代码计划已经实施、构建通过或业务闭环完成。施工记录只能记录执行智能体实际运行的验证，不得补写未发生结果。

## 12. 完成边界

本规则只证明以下协作边界已经建立：

```text
交互智能体独占智能体协作规则修订；
计划智能体拥有未被执行选中的新目标完整设计包且不发送消息；
执行智能体实施代码并维护专属施工 / 验证记录，只向计划支撑智能体发送消息；
计划支撑智能体接管执行中设计包修订并只向执行智能体发送消息；
无法由现行规则解决的问题停下并等待用户。
```

它不证明任何计划已形成、代码已实施、构建已通过、候选已集成或项目能力已经完成。

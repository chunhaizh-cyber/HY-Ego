# 多工作树并发与集成规则

日期：2026-07-24

状态：Codex 代理协作 / Git 工程规则；位于 `.codex/rules/`；不是项目机器规范

## 1. 目标

本规则只定义 worktree 路径与分支、冻结基线、计划 blob、Git 身份、并行隔离、长期执行通道、集成实施、主线发布占用的技术条件、技术失败和回收合同。智能体类型、S0、消息路由和整树切换由 `AGENTS.md` 与 `.codex/rules/设计执行双窗口交互规则.md` 定义；任务过程不写入 Markdown 状态机。

本规则不属于 `规范/规范目录.md` 登记的项目机器规范，不定义机器事实或代码语义；其约束力来自 `AGENTS.md` 对多工作树、并行批次、集成和回收的指定入口。凡进入上述工程 / 协作活动的智能体都必须遵守本规则，但不得据此覆盖现行机器规范、扩大计划范围或取得新的智能体权限。

执行代码可以采用两种互斥的技术形态：

```text
一个 main 主线发布分支
一个 main 集成工作树
并行形态：每个并发任务一个独立分支和 worktree
连续形态：一个长期执行顶层任务、一个持久分支和 worktree，顺序承载多个计划段
每个文件和结构所有权在同一批次内唯一
集成后统一验证再发布 origin/main
```

多 worktree 只提高互相独立或已经由预冻结接口合同解耦切片的吞吐量；长期执行通道用于避免为连续计划重复创建任务、R2、分支和 worktree。两种形态都不取消计划、未合同化真实依赖、接口合同复核、允许 / 禁止文件、验证和完成声明门禁。计划形成或修订时必须先尝试冻结提供者—消费者合同、隔离计划范围和所有权，不能先假定串行，再把本可按合同并行的计划登记成完成依赖链。

## 2. 工作树类型

下表中的智能体名称只绑定工作树用途，不重新定义智能体权限：

| 类型 | 分支 | 用途 | 登记写入身份 |
| --- | --- | --- | --- |
| 主集成工作树 | `main` | 智能体协作规则、新目标设计包、执行中设计包修订，以及已验证集成结果的非强制快进与最终发布 | 同一时刻只允许取得对应文件所有权的交互、计划、计划支撑或集成模式执行智能体之一写入 |
| 任务工作树 | `codex/<计划编号>-<主题>` | 执行一个计划索引中的 `可执行` 计划 | 只允许该计划指定的执行智能体写入 |
| 长期执行通道工作树 | `codex/execution-<通道主题>-r<代次>` | 同一执行顶层任务逐份消费多个 `可执行` 计划；每个计划形成独立计划段 | 只允许该通道的执行顶层任务写入；同一时刻只允许一个活动计划段 |
| 集成暂存工作树 | `codex/integration-<批次号>` | 汇集同批次任务提交并运行集成验证 | 只允许该批次进入集成模式的执行智能体写入 |

主集成工作树不得由规范硬编码盘符或旧仓库名，必须以当前 Git 顶层和 `git worktree list --porcelain` 解析。任务工作树、长期执行通道工作树和集成暂存工作树必须位于主集成工作树目录之外，可以由 Codex 应用或 `git worktree` 建立；预期分支、冻结基线和文件 / 结构所有权写入目标计划，实际绝对路径、HEAD、upstream 和 dirty state 在 S0 直接读取 Git。

不得让两个 worktree 检出同一分支。不得在 detached HEAD 上实施计划。

### 2.1 worktree 不改变智能体类型

worktree 只隔离文件和 Git 身份，不产生或改变智能体类型、切片所有权、活动计划段和发布许可。顶层任务树、子智能体、唯一切片所有者、S0、Git 串行、整树切换和跨顶层交接统一服从 `AGENTS.md` 与 `.codex/rules/设计执行双窗口交互规则.md`；本规则只核对其绑定的 worktree 技术身份。

## 3. worktree 身份核对证据

本节只规定 worktree 技术身份的核对证据，不是项目全局权威顺序，也不裁决任务生命周期：

```text
计划/计划索引.md 中的计划列表、计划版本、计划状态和具名依赖
目标计划中的计划 blob、预期分支、允许范围、接口与验证矩阵
Git worktree、分支、HEAD、upstream、提交、dirty state 和远端实际状态
最新 `EXECUTION-SUPPORT-NOTICE` / `PLAN-SUPPORT-NOTICE`
```

任务消息、聊天摘要、任务标题和显示状态不构成 Git 身份证据。计划合同与 Git 事实冲突时停止写入，并按交互规则返回具名 DRIFT；不得用消息正文覆盖 Git 事实。

计划中的预期 worktree / 分支只表达施工合同，不自动表示物理工作树已经创建、执行智能体已经选中计划或取得占用。写入前的技术身份核对必须覆盖 Git worktree、路径、分支、HEAD、upstream、远端、dirty state、计划 blob、活动计划段起点和唯一所有者；当前计划段事实由结构化消息与 Git 共同证明。

## 4. 并行批次技术准入

真实依赖、计划拆分和智能体边界由 `AGENTS.md` 与交互规则裁决。本规则只对拟采用并行形态的两个或以上计划执行下列技术准入；全部成立后才能建立并发 worktree：

1. 每个计划的未合同化正式依赖在同一冻结基线提交上均已满足；提供者实现尚未形成但接口已经按本节冻结时，不构成开发启动依赖。
2. 每个计划都有完整版本、允许文件、禁止文件、实际接口复核和验证命令。
3. 批次内允许写文件集合两两无交集。
4. 批次内结构所有者、业务写入口、公开 ABI 和阶段号登记所有权两两无冲突。
5. 提供者—消费者关系已经由同一版本合同冻结，或者确实不存在；合同至少固定模块 / 文件、DTO、签名或等价行为、ABI / 规则版本、所有权、返回 / 错误、提供计划、消费计划和退回点。未合同化关系或无法隔离的装配先后仍禁止并发。
6. 共享文件已明确归属唯一任务或后继集成切片。
7. 已固定集成顺序、集成验证矩阵、失败退回路径和进入集成模式的执行智能体所有权。
8. 每个并发任务计划已冻结唯一任务分支、基线提交、执行所有者和文件 / 结构所有权；实际绝对路径由 Git 解析。共同代码基线可以暂时不含提供者实现，但必须含相同版本的正式计划合同。采用长期执行通道时不适用本条的“每任务 worktree”要求，改按第 7.2 节逐计划段串行。

任一条件不成立时只允许串行推进，不能用“文件看起来不同”或聊天中的接口名称代替正式合同。合同并行只允许形成实现候选；提供者、消费者和真实装配尚未汇合前不得声明编译、能力或集成完成。

## 5. 共享文件边界

以下文件默认属于中央治理或集成共享面：

```text
AGENTS.md
.codex/rules/**
.codex/skills/**
规范/**
流程图/**
详细设计/**
计划/**
其它中央设计 Markdown
海中鱼巣.vcxproj
海中鱼巣.vcxproj.filters
海中鱼巣/入口.cpp
中央自检运行器、统一阶段登记和其它计划明确列出的共享装配文件
```

中央治理文件按交互规则分别由交互智能体、计划智能体或计划支撑智能体修改；三者不能越过各自协作规则或设计包所有权。`实施记录/**` 中由计划唯一指定的施工记录和验证记录不属于中央设计文件，由执行智能体随对应计划候选维护；其它 Markdown 不因文件扩展名取得执行写权。代码共享面在一个并行批次内只能由一个执行任务拥有；若多个切片都需要修改工程、入口或中央运行器，对应设计包的合法写入智能体必须：

```text
合并为同一串行切片
或把私有实现与共享接线拆成不同计划
或取消该批次的并行资格
```

进入集成模式的执行智能体不得借解决冲突发明业务逻辑、扩大接口或修改计划外语义。

## 6. 冻结清单

每个并行批次必须在目标计划中冻结：

| 字段 | 要求 |
| --- | --- |
| 批次号 | 全局唯一 |
| 基线提交 | 完整 Git 提交号，且是各任务分支共同祖先 |
| 计划身份 | 计划路由编号、版本和路径 |
| 计划 blob | 冻结基线或中央计划提交中不可变施工合同的 Git blob；任务 worktree S0 或长期通道计划段 S0 必须引用同一 blob |
| 计划索引引用 | 任务编号、计划版本、计划状态和具名依赖 |
| 文件所有权 | 规范化绝对或仓库相对路径集合 |
| 结构所有权 | 仓库、服务、模块、DTO、阶段和公开接口 |
| 接口证据 | 执行前实际签名，或正式预冻结接口合同版本及提供者 / 消费者映射 |
| 验证矩阵 | 任务候选级与最终集成级命令、成功条件；允许把完整构建和运行验收后置，但不得省略合同 / 白名单 / 静态边界检查 |
| 集成关系 | 独立任务候选固定汇集顺序；长期通道固定各计划段结果提交与累计父链 |
| 退回条件 | 具名冲突、漂移、失败和责任智能体 |

独立任务 worktree 的冻结计划 blob、冻结文件、冻结接口或批次依赖发生正式变更后，原任务技术身份自动失效；执行智能体停止并只向配对计划支撑智能体发送具名漂移，等待计划索引出现新的 `可执行` 版本后重新 S0。任务工作树不得自行合并新 `main` 继续执行，也不得用其它 Markdown 的显示状态解释跨越。

长期执行通道按影响范围处理变更：活动计划 blob 变化时立即停止该计划段；尚未开始计划的 blob 变化只废弃该计划段旧合同，不废弃整个通道；通道建立基线、累计代码父链、共享接口或已提交文件所有权失配时，才使整个通道代次失效。新计划合同可以从中央计划提交按路径和 blob 读取，不要求把 Markdown 合并进执行分支；但计划段 S0 必须证明当前通道 HEAD 与该计划假定的代码基线相容。

## 7. 任务工作树接管与分支结果

智能体类型、消息路由和 `分支完成待集成` 事实格式由交互规则唯一定义。本规则只规定任务工作树的 Git 条件：

1. 从目标计划冻结的基线创建或接管唯一任务分支，不得 detached HEAD，不得与其它 worktree 共用分支。
2. 写入前核对绝对路径、分支、HEAD、共同基线、所有者和 dirty state 均与登记一致。
3. 实际差异必须落在冻结文件 / 结构所有权内；计划外文件和其它任务所有文件不得进入 index。
4. 完成计划规定的任务级验证后，只暂存本任务文件，提交并非强制推送到同名远端任务分支。
5. 分支结果、提交号、父链、文件范围、验证证据和残余 dirty state 由执行智能体写入计划唯一专属施工记录，并通过 `EXECUTION-SUPPORT-NOTICE` 只交给配对计划支撑智能体。施工记录不得定义机器语义、改变计划状态或替代结构化消息；没有设计包缺口时，消息只报告 Git 事实，不自动触发设计包修改。

任务分支提交只形成隔离分支结果，不自动启动集成，也不证明计划进入 `main`。

### 7.1 Windows 自主接管权限门禁

当 Codex 或其它执行平台把任务 worktree 与主仓库 Git 元数据放在不同目录时，执行智能体首次自主接管前必须在 S0 实际验证当前环境能同时写入任务文件、任务分支索引和主仓库 `.git/worktrees` 所需元数据。受限环境不能完成上述 Git 写入，或执行过程中会被不可预期的交互审批阻断时，该技术身份不可执行；执行智能体只向配对计划支撑智能体返回具名环境 DRIFT，并继续检查其它无关计划，不等待其它智能体命令。

具体平台配置键、沙箱名称和审批选项属于自主接管时的运行环境事实，由执行智能体在结构化消息中报告；不在本长期入口固定。无论平台权限多高，都不改变项目授权：任务工作树的差异和 index 只能包含目标计划允许文件，推送目标只能是同名任务分支，并必须执行精确暂存、staged diff、禁止命令和完整验证门禁。删除、移动、强推、主工作树写入、`origin/main` 推送、计划未允许的 merge / rebase / cherry-pick 一律禁止。

### 7.2 长期执行通道与计划段

长期执行通道的固定身份是：

```text
一个长期执行顶层任务
+ 一个持久 worktree
+ 一个持久 codex/* 分支
+ 一个通道建立基线
+ 多个逐份消费的计划段
```

通道身份只登记一次；计划段不是新的执行权、任务身份或 worktree。每个计划段在开始时自行冻结：

```text
任务 / DQ / 计划版本 / 计划 blob
中央计划提交
通道当前 HEAD（计划段起点）
允许 / 禁止文件和结构所有权
实际接口或预冻结合同
本段验证矩阵
结果提交或允许的 WIP 断点提交
```

长期通道遵守以下硬规则：

1. 同一时刻只有一个活动计划段；不允许两个计划共享 dirty state 或 index。
2. 执行顶层任务可以在当前计划段内把互不重叠的代码 / 自检切片交给同类型子智能体，但 Git 暂存、提交、推送和计划段切换仍由通道顶层智能体串行收口。
3. 计划正常完成时形成独立结果提交；计划发生 DRIFT 或验证失败时，只有计划明确允许保留代码 WIP，才能形成具名 WIP 断点提交。WIP 断点不等于候选完成。
4. worktree / index 恢复 clean 后才能选择下一份计划。无法合法提交且仍 dirty 时，整个通道停在当前计划，不得覆盖、stash 或带入下一计划。
5. 通道 HEAD 可以随着计划段提交累计前进；已结束计划固定的是其段起点和结果提交，不要求通道永久停在该提交。
6. 执行智能体读取计划索引中的计划列表，根据计划状态和具名依赖自主选中计划，再由本规则与 Git 核对自身执行通道；其它智能体不得另行发布执行顺序。当前计划 DRIFT 后可以跳过并检查列表中的其它无真实依赖 `可执行` 计划。
7. 每次计划段 S0 都重新核对当前 HEAD、计划 blob、白名单、合同和无第二活动段；无需其它智能体接受，也不创建逐计划 R2、派发状态或执行许可。
8. `EXECUTION-SUPPORT-NOTICE` 必须携带通道编号、计划段起点、结果 / WIP 提交、专属施工记录路径和 dirty state，并且只发送给配对计划支撑智能体。计划支撑智能体只有在存在设计包缺口时才修订已接管设计包。

### 7.3 本仓库永久执行智能体通道

本仓库固定使用下列 Codex 执行智能体通道；该绑定属于代理规则，不写入项目规范、详细设计、叶子计划或计划索引：

```text
窗口标题：执行当前计划任务
顶层任务：019f8d82-e587-7522-a62d-204f902c8455
持久分支：codex/execution-current-plan-r1
```

物理 worktree 路径、HEAD、upstream、dirty state 和远端一致性每次从 `git worktree list --porcelain` 与目标 worktree 的 Git 结果直接读取，不持久化绝对路径。更换电脑时允许为同一持久分支重新建立 clean worktree；不得因此改写计划列表、复制状态文件或建立逐计划执行身份。执行智能体只从计划索引列表中按计划状态和具名依赖选中计划，每次仍须完成本段 S0。

## 8. 漂移与失败

独立任务 worktree 或长期执行通道的活动计划段发现接口漂移、文件交集、冻结基线失效或计划外逻辑错误时：

1. 停止扩大实施范围。
2. 撤回未验证代码草案，或按计划明确许可在代码 / 工程 / 自检白名单内保留具名 WIP。
3. 只有计划允许保留 WIP 时才形成断点提交；执行智能体可以同步更新该计划唯一专属施工记录，记录实际断点、验证和 Git 现场，不得新建或修改其它 Markdown。
4. 只提交并推送允许的代码断点和已许可产物；无法合法提交时保持当前工作区并停止，不切换下一计划。
5. 执行智能体在专属施工记录中固定技术失败类型和证据，并按交互规则只向配对计划支撑智能体回传；计划支撑智能体只在现行协作规则与原目标内修订已接管设计包，不另建状态机。
6. 纯断点是否纳入 `main` 仍按目标计划的集成路径处理。

执行智能体不得为追上 `main` 自行 merge、rebase、cherry-pick 或修改计划。长期通道中尚未开始的计划若需要更新计划合同，只从中央计划提交读取精确 blob；若当前通道 HEAD 已不满足该合同的代码前提，则该计划段返回具名基线 DRIFT，不得靠 Git 改写跨越。

## 9. 集成协议

### 9.1 集成自主接管与身份核对

执行智能体进入集成模式时的消息边界由交互规则唯一定义。本规则读取已经存在的集成编号、绝对路径、分支、内容冻结基线、分支建立起点、中央治理尾白名单、输入提交、所有者、允许范围、验证矩阵和失败收口，只裁决这些字段对应的 Git 身份是否成立；其它智能体不得据此指挥执行智能体开始集成或接受 S0。

集成技术身份必须机械区分三段，不得再把它们压成一个“集成起点”：

| 字段 | 唯一含义 |
| --- | --- |
| 内容冻结基线 | 冻结计划 blob、固定输入、父链、允许 / 禁止范围和验证矩阵的代码 / 合同基线；集成接管前不得变化 |
| 分支建立起点 | 新集成 worktree 与分支实际创建时的唯一 HEAD；负责路径、分支、reflog 和身份连续性，不要求后续中央治理提交形成后仍等于当前 main |
| 中央治理尾 | 从分支建立起点到取得主线发布占用前追加的连续 main 规则 / 规范 / 流程图 / 详细设计 / 计划提交；不得改变内容冻结基线、固定输入或目标计划合同 |

集成 S0 必须直接比较分支建立起点与当前 `origin/main`。两者之间若只有不改变内容冻结基线、固定输入和目标计划合同的中央治理提交，可以先在 clean 集成分支 ff-only 同步；出现计划合同、代码、工程、自检或固定输入变化时，旧集成身份失效并从当前 `origin/main` 重建。不得用历史状态文件解释跨越。

进入集成模式的执行智能体在创建或写入前必须核对：

```text
目标路径是否不存在，或已存在且精确属于同一集成编号和所有者。
目标分支是否不存在，或已存在且 HEAD、父链和分支建立起点完全一致。
worktree / index 是否 clean。
任务提交父链、远端分支和文件范围是否符合冻结清单。
WT-MAIN 是否 clean，且 main 与 origin/main 相等。
分支建立起点是否为当前 main 的祖先；两者之间是否只有允许的中央治理尾变化。
内容冻结基线、计划 blob、固定输入、父链、允许 / 禁止范围和验证矩阵在当前 main 是否仍与登记一致。
```

已存在路径 / 分支的身份、分支建立起点、所有者、父链或 reflog 任一不一致，或者当前 main 不是建立起点的后代、中央治理尾含未登记文件、内容冻结合同发生变化时，技术结论固定为“旧候选身份失效”。旧候选保持只读并登记为失效，不得 reset、delete、切换基线、覆盖分支或继续复用；执行智能体只向配对计划支撑智能体发送具名 DRIFT。计划支撑智能体能在现行规则与目标内修订既有计划时形成新版本，否则停止并等待用户；新身份具备后由执行智能体自主重新 S0 才可恢复写入。当前 main 仅比建立起点多出合法中央治理尾时不构成漂移，也不得要求为了恢复等值而改写历史或重复重建身份。

### 9.2 汇集、验证与主线发布占用

进入集成模式的执行智能体必须在独立集成暂存工作树中按冻结顺序处理任务提交：

```text
读取 origin/main、目标计划、计划索引和任务分支提交
-> 验证每个任务提交范围、父提交和任务级证据
-> 执行智能体自行完成集成 S0，PASS 后原子取得唯一主线发布占用，不等待其它智能体接受
-> 再次确认 main == origin/main 且仍是 S0 所见路由尾的合法后代
-> 在 clean 集成分支执行唯一一次 ff-only 同步到当前 origin/main，把同步后 HEAD 固定为集成执行起点
-> 按冻结顺序集成
-> 任一冲突立即停止，不作语义性手工消冲突
-> 运行批次规定的完整集成验证
-> 主线发布门禁成立且执行智能体的集成模式继续有效：让主集成工作树 main 非强制快进到集成结果，推送 origin/main，只向配对计划支撑智能体返回 `已集成` 的 `EXECUTION-SUPPORT-NOTICE`
-> 主线发布门禁不成立或当前执行智能体必须停机：推送同名集成分支，固定提交与验证证据，释放主线发布占用，只向配对计划支撑智能体返回 `集成分支验证通过待主线发布` 的 `EXECUTION-SUPPORT-NOTICE`；对应计划状态保持不变，等待后续执行智能体在集成模式中自主核对
```

`集成分支验证通过待主线发布` 只证明固定集成分支已通过当次验证并已推送，不证明 main 已发布。新的执行智能体集成任务不得复用已停机任务身份，必须重新核对 origin/main 漂移、集成分支身份与父链、固定输入、工作树 clean 状态和本次发布验证矩阵，自行 S0 并取得唯一占用后才能非强制推进并推送 main；其它智能体不发出开始命令。

集成工作树完成身份核对、自行 S0 PASS，并由执行智能体在 `EXECUTION-SUPPORT-NOTICE` 中声明取得本批次唯一主线发布占用时，占用才成立。取得占用后，执行智能体只能先按本节把 clean 分支 ff-only 同步到当前 origin/main；同步结果是本轮集成执行起点，不改变内容冻结基线或固定输入。从此刻到成功发布或正式退回释放之间，技术冻结条件为：

```text
主集成工作树文件、index、main 和 origin/main 不得被其它写路径推进。
集成暂存工作树保持目标计划身份、分支建立起点证据、集成执行起点和唯一执行所有者。
其它任务 worktree 或长期执行通道不得改变本批次冻结输入，也不得要求 main 前进。
紧急规则或计划治理必须等待执行智能体释放占用，再由本规则核对释放后的 Git 状态；不得用跨路由消息指挥停机。
```

主集成工作树在集成发布期间必须保持 clean，且 `main == origin/main == 集成执行起点`，直到最终由同一执行智能体非强制推进到已验证结果。远端 `main` 在占用后被其它路径推进、ff-only 同步失败、出现合并冲突、任务提交超范围、集成验证失败或主工作树不干净时停止发布，不得 force。集成发布或退回后必须在 `EXECUTION-SUPPORT-NOTICE` 中明确主线发布占用已经释放。

集成失败不得通过修改测试、复用旧二进制、删除验收项或发明接口收口。执行智能体的失败报告必须区分任务分支代码问题、计划合同问题和规则问题：代码问题留在执行范围内追根因，计划问题交给配对计划支撑智能体，规则问题由计划支撑智能体停下等待用户。

## 10. 验证层级

多 worktree 不自动降低验证等级：

1. 独立任务 worktree 和长期执行通道的每个计划段都执行目标计划规定的全部任务级验证。合同并行批次可以把任务级验证收束为合同签名、静态依赖、固定夹具、文件白名单和差异卫生，并明确不作编译 / 能力完成声明。
2. 集成暂存工作树至少重跑受影响配置的完整构建、完整自检、严格规范和批次交叉回归。
3. 涉及事务、所有权、运行期装配、持久化、恢复或并发语义的批次，集成层必须执行计划中最严格的连续运行与并发门禁。
4. 只有正式计划在冻结为 `可执行` 前明确采用分级验证时，才能减少任务级重复验证或把完整构建 / 运行后置；最终集成必须补齐所有被后置的 Debug / Release、完整自检、真实提供者接线、恢复、并发和静态边界验证，执行智能体不得临时继续降级。

## 11. 提交、推送与回收

| 工作树 | 自动提交目标 | 自动推送目标 |
| --- | --- | --- |
| 主集成工作树的交互智能体 | `main`，仅 `AGENTS.md`、`.codex/rules/`、`.codex/skills/` | `origin/main` |
| 主集成工作树的计划智能体 | `main`，仅尚未被执行选中的新目标设计包 | `origin/main` |
| 主集成工作树的计划支撑智能体 | `main`，仅已经被执行选中或退回的关联设计包 | `origin/main` |
| 任务工作树执行智能体 | 当前 `codex/<计划编号>-<主题>`；包括计划允许的代码 / 工程 / 自检及专属施工记录 | 同名远端任务分支 |
| 长期执行通道执行智能体 | 当前 `codex/execution-<通道主题>-r<代次>`；每个计划段及其专属施工记录独立提交 | 同名远端长期通道分支 |
| 集成暂存工作树中的执行智能体 | 当前 `codex/integration-<批次号>` | 默认不单独发布；验证完成后由主集成工作树快进并发布 `origin/main` |

所有推送均禁止 force 和 force-with-lease。任务分支不得直接推送 `origin/main`。

只有在 `origin/main` 已包含集成结果、目标 worktree clean、没有未集成提交、没有尚待续作的活动计划段，且最新结构化消息明确释放所有权后，才能回收独立任务 worktree、长期执行通道 worktree和对应本地分支。长期通道还必须证明全部已开始计划段均已有结果 / WIP 提交映射。存在 dirty state、未集成提交、未读取断点或远端独有提交时禁止删除。远端任务 / 通道分支删除不属于自动收尾，必须由明确的回收动作授权。

## 12. 完成声明边界

任务分支提交只证明单任务候选进入隔离分支；长期执行通道中的计划段提交只证明该段候选或具名 WIP 已固定在累计分支。集成验证和 `origin/main` 推送只证明对应批次已进入主线。各智能体消息、计划状态和最终完成声明边界由交互规则裁决。

不得把允许多 worktree 宣称为：

```text
允许多个智能体写同一文件
允许多个业务所有者并存
允许绕过依赖并行
允许执行智能体修改中央治理
允许任务分支直接成为生产权威
允许降低验证或跳过集成回归
允许同一顶层任务树通过子智能体同时取得多个写类型
```

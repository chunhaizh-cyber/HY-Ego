# 海中鱼巣制定计划

## 角色与权威入口

只在已登记的设计角色中使用本技能。遵守 `AGENTS.md`；派发、S0 和角色切换读取 `规范/设计执行双窗口交互规范.md`；worktree、分支、冻结与集成读取 `规范/多工作树并发与集成规范.md`。

设计任务树可以把互不重叠的设计或只读切片交给同角色 / 只读子智能体，但不得把任何后代切换成执行或集成写角色。顶层统一复核差异，并串行完成 Git 和任务派发收口。

不得硬编码仓库盘符、旧仓库名或固定 worktree。先用 Git 顶层、`git worktree list --porcelain` 和 `项目记忆/并行工作树登记表.md` 解析当前中央工作树及目标任务身份；三者不一致时停止写入并报告 DRIFT。

本技能只做设计治理：可以读取代码事实并形成计划，不修改 C++ / 工程 / 自检，不运行 C++ 构建、程序或业务自检。

## 前置读取

先读取 `AGENTS.md` 要求的共同入口，再按任务读取：

```text
目标正式规范与相邻边界
有效流程图与详细设计
代码差距或设计审查记录
当前代码 / 接口 S0 事实
计划/计划索引.md
项目记忆/任务状态台账.md
项目记忆/Codex任务队列.md
项目记忆/并行工作树登记表.md
项目记忆/窗口交互记录.md
```

聊天、旧项目规则、过期计划和旧函数只能作为来源证据。代码实施单位必须来自现行规范定义的业务对象、领域服务、结构闭环和有效设计，不能按旧函数一对一迁移。

缺少正式规范、有效设计、当前接口事实或必要差距记录时，先在设计角色内补齐允许的设计产物；无法补齐时停止，并登记具名缺口，不能凭假设生成可执行代码计划。

## 计划形成算法

按以下顺序工作：

```text
确认目标规范与完成边界
-> 核对当前代码和实际接口
-> 核对流程图、详细设计、差距记录的范围一致性
-> 按业务对象、文件、结构、公开接口和装配面拆分
-> 判断真实依赖与所有权隔离
-> 形成计划正文及机械阶段
-> 登记索引导航、队列依赖、工作树技术身份和唯一任务状态
-> 验证、精确提交并非强制推送派发准备状态
-> 对就绪计划向独立执行任务实际发送消息并读回
-> 同步派发状态、再次验证提交推送
-> 目标读取派发状态提交并完成 S0
```

函数只是事实采集单位；计划范围由现行规范与有效设计裁决。

### 真实依赖

只有下列情况可以登记依赖：

- 当前切片必须消费尚未形成、且尚未冻结为精确待实现合同的具名正式产物或接口。
- 文件、结构、公开接口或装配所有权无法通过拆分隔离，存在不可消除的先后。

同主题、同总计划、生成顺序、文档顺序、目录或验收顺序都不是依赖。仅共享工程、入口、统一运行器或装配文件时，优先拆成私有实现切片和唯一共享接线切片。

提供者实现尚未形成，但模块 / 文件、公开 DTO、完整签名或等价行为、ABI / 规则版本、输入输出所有权、逻辑内返回、内部错误、提供计划、消费计划和漂移退回点已经冻结时，该关系改记为“预冻结待实现接口合同”，不再作为等待上游实现完成的开发启动依赖。提供者和消费者可在独立 worktree 并行形成候选；消费者不得补造同名接口或修改提供者文件，双方在最终真实接线验证前不得宣称接口已实现、代码已编译或能力已完成。

依赖门控计划必须写明：

```text
已验证当前事实
具名前置计划 / 队列和产物
待消费的文件、符号、签名与行为
无法隔离的原因
解除条件
执行前复核命令
接口漂移的精确退回点
```

依赖未满足或接口漂移时，只登记受影响计划的精确门禁，不冻结无关计划，不把预期产物写成当前事实，也不生成泛化的“等待后续处理”。

### 并发与所有权

无未合同化真实依赖的计划独立登记。两个以上计划满足共同计划合同基线、文件 / 结构 / 接口所有权互斥、共享文件唯一所有者、固定集成顺序和分级验证矩阵时，优先登记并行批次；共同代码基线可以暂时缺少提供者实现。不能并行时写明具体所有权、资源或验证原因。

在任何任务 worktree 写入前，登记任务编号、计划版本、绝对路径、`codex/*` 分支、冻结基线、文件 / 结构所有权、接口证据、验证矩阵和集成顺序。

## 计划正文

完整计划放在 `计划/`。代码实施切片至少包含：

```text
设计状态、计划 / 队列身份和唯一任务状态引用
依据：正式规范、流程图、详细设计、差距 / S0 记录
已验证前置事实
实施范围与明确排除项
目标服务、结构和公开接口
预冻结待实现接口合同、提供者与消费者（适用时）
允许文件与禁止文件
机械步骤 S0/S1/...
入口拒绝、内部错误和失败收口
验证命令与成功条件
实施后证据
完成声明边界
```

面向常规执行模型的每一阶段明确区分：已验证前置事实、待实施目标、实施后可核对证据、只有前置成立后才能判断的完成 / 退回条件。

新增或改名代码文件时，读取 `规范/代码文件建立归属与模块命名规范.md`，列出文件分类、项目项类型、`include` / `import` 路径、工程登记、生产 / 自检依赖方向和 `入口.cpp` 不恶化边界。

计划、流程图、详细设计和设计记录必须互相列明路径且范围一致。纯规则或只读治理确实不需要流程图 / 详细设计时，写明不适用原因和替代正式依据。

## 登记与真实派发

计划形成后，叶子正文冻结为不可变施工合同，只写设计状态和任务状态定位键，不写当前待执行、派发、S0、执行或集成状态。计划索引只登记路径、版本和静态依赖；任务队列只登记顺序、直接依赖和解除条件；工作树登记只写路径、分支、基线、计划 blob、所有权和 Git 技术状态；窗口交互记录只追加已经发生且与台账同名互证的任务生命周期迁移事件。每个任务的当前生命周期只在 `项目记忆/任务状态台账.md` 维护一行。只有具名依赖尚未满足时，台账登记 `依赖门控待执行`；其余完整实施计划登记 `待执行`。

计划登记不是派发。具有执行写范围的计划同时满足下列条件后，不等待下一轮用户确认：

```text
依赖已满足，或提供者—消费者关系已经合法合同化
流程图、详细设计、计划和记录范围一致
实际接口或预冻结待实现接口合同复核通过
已登记队列、执行身份、worktree、分支、基线和所有权
文档治理验证通过
中央治理精确提交并非强制推送成功
```

随后按交互规范完成两段式真实派发：先推送台账保持 `待执行` 的派发准备提交，再通过 Codex 任务工具向唯一、独立、用户可见的执行顶层任务实际发送消息并读回接收状态；消息明确要求目标等待派发状态提交。发送与读回成功后，只在任务状态台账原子迁移为 `已派发待执行回执`，绑定派发状态版本、计划 blob、目标身份和 worktree，并在窗口交互记录追加事件；再次验证、提交并非强制推送，最后通知目标读取该派发状态提交。目标只有读到该提交、核对同一计划 blob 并完成 S0，且设计窗口在新状态版本接受 S0 后才能写入。不得使用本设计任务树的子智能体承担执行角色，也不得把预留 worktree、目标标题、消息草稿或“下一步”当作已经派发。

派发消息至少包含计划 / 队列编号、计划版本、任务状态版本、派发准备提交、权威路径、计划 blob、允许 / 禁止范围、任务 worktree / 分支 / 冻结基线、必须重读的材料、验证、等待派发状态提交和返回条件；状态迁移后追加派发状态提交号。消息只负责唤醒，执行窗口仍须从仓库完成 S0。

唯一目标缺失、歧义、工具不可用或消息发送失败时，保持 `待执行`，在当轮设计记录、技术登记或任务工具持久化证据中具名保存缺失身份与失败原因；窗口交互记录不追加未发生的状态迁移。不得宣称已经派发或执行已经开始。只有消息发送已经成功、只是读回接口不可用时，才可按交互规范使用任务工具返回的持久化投递回执继续同步派发状态。具名门禁解除后立即恢复派发链。

## 验证与停止

设计治理至少运行：

```powershell
git diff --check
python .\tools\check_specs.py --strict
```

只暂存本轮设计文件，按 `AGENTS.md` 精确提交并非强制推送。目标分支、主线发布占用、远端前进、同文件 WIP、验证或推送任一不安全时停止，不扩大范围。

本技能形成并派发计划，不证明代码已修改、构建通过、任务已集成或能力已闭环。

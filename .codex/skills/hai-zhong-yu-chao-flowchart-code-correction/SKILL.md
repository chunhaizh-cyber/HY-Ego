# 海中鱼巣流程图驱动代码纠偏

## 权威引用与门禁

1. 只在当前顶层任务树拥有设计写角色时使用；遵守 `AGENTS.md`，不复制其中的角色、派发、Git 或执行状态机。其它角色只返回只读事实，不落盘治理产物。
2. 先解析当前 Git 顶层并读取 `AGENTS.md` 指定的共同入口，再读取目标正式规范、有效流程图与详细设计、现行计划、必要设计记录以及目标代码和调用点。代码事实只取自冻结的当前提交与工作树。
3. 流程图和详细设计只能在正式规范内解释实现，不能覆盖规范；流程图、映射表、偏差清单和用户确认都不能单独授权修改代码。需要形成、登记或派发施工计划时转用 `hai-zhong-yu-chao-plan`；计划就绪后由该技能在同一治理续程完成真实派发，不得停在文档产物或预留状态。
4. 现状流程图只表达当前代码事实。实施记录和验证输出只能交叉核对来源，不得把目标、旧状态或未发生的验证画成当前事实。
5. 施工流程图必须同时具名绑定现行正式规范、有效详细设计和现行计划，并保持三者范围一致；任一依据缺失、失效、冲突或未覆盖目标时，不生成施工图或可执行代码纠偏计划，退回相应设计治理。
6. 本技能只读代码并写设计治理产物；不修改 C++、工程或自检，不运行构建、程序或业务自检。

## 专用算法

```text
确认设计角色、目标、用途与唯一文件所有者
-> 冻结提交、工作树状态、覆盖文件、函数、调用点和行号口径
-> 逐行读取当前代码并建立完整映射
-> 由映射生成现状流程图及配对 HTML
-> 建立输入契约 / 调用语境表
-> 对每个中途非成功结果做二分审查
-> 对照正式规范、有效流程图 / 详细设计和现行计划形成偏差清单
-> 分流为修现状图 / 修施工图 / 修详细设计 / 补规范或契约 / 形成代码纠偏计划
-> 仅在正式依据完整且范围一致时生成或修订施工流程图
-> 机械核对产物、按现行设计治理规则验证并收口
```

对每个非成功结果依次核对：输入是外部、测试或候选材料，还是正式上游已经保证有效的结构；协议是否明确允许拒绝、空候选或等待；是否已进入创建、绑定、写关系、写状态、记录动态、结算、读回或发布；是否产生可读半结构。

分类固定为：

- `逻辑内返回`：协议允许的入口拒绝、候选为空、等待、请求材料返回或人读材料返回，且不产生可读半结构。
- `追根因解决`：正式上游已保证输入有效，或前置通过后内部结构处理结果不符合预期。

结构不变化不能单独证明是逻辑内返回。同一判断点兼有两种调用语境时拆为两个节点；缺少输入契约 / 调用语境表时，不得把中途非成功结果定性为逻辑内返回。

## 产物合同

按目标范围建立或核对以下产物；现状图与施工图必须使用不同标题和文件名，Markdown 与 HTML 必须包含相同 Mermaid 主体：

```text
流程图/YYYYMMDD_<主题>现状流程图_v0.1.md
流程图/YYYYMMDD_<主题>现状流程图_v0.1.html
项目记忆/设计记录/YYYYMMDD_<主题>逐行代码映射表.md
项目记忆/设计记录/YYYYMMDD_<主题>输入契约与调用语境表.md
项目记忆/设计记录/YYYYMMDD_<主题>非成功返回二分审查表.md
项目记忆/设计记录/YYYYMMDD_<主题>现状施工偏差清单.md
流程图/YYYYMMDD_<主题>施工流程图_v0.1.md（有完整正式依据时）
流程图/YYYYMMDD_<主题>施工流程图_v0.1.html（有完整正式依据时）
计划/YYYYMMDD_<主题>代码纠偏或文档修订计划_v0.1.md（需要且具备形成条件时）
```

新的设计审查产物写入 `项目记忆/设计记录/`，不写入 `实施记录/`。已有有效产物按其现行路径原地修订；版本、命名和登记按仓库现行治理规则处理。

### 现状图精确元数据

每份现状流程图开头逐项填写，不适用项明确写 `无 / 不适用`，不得省略：

```text
图类型：现状流程图
代码版本：<提交号；有未提交改动时注明“基于当前工作区”>
覆盖文件：
覆盖函数：
逐行映射表：
输入契约 / 调用语境表：
非成功返回二分审查表：
偏差清单：
依据实施记录：
验证输出：
不得作为施工许可：
不得宣称：
```

工作树存在未提交改动时，只能声明当前工作区事实，不能声明已经成为提交、远端或稳定基线事实。

### 施工图精确元数据

每份施工流程图开头逐项填写：

```text
图类型：施工流程图
绑定正式规范：
绑定计划：
绑定详细设计：
允许文件：
禁止文件：
预期结构变化：
执行前复核：
验证方式：
不得宣称：
```

施工图只表达具名依据内的待实施目标，不得把目标路径写成当前代码事实；施工图自身不改变计划范围、执行许可或机器语义。

### 逐行代码映射

覆盖所审查函数或代码片段内每一个非空、非纯格式代码行。空行、纯括号和仅排版行可以并入相邻节点，但必须落入明确行号范围；文件头规则注释、人读说明注释和预编译宏必须说明是否影响运行路径。

| 字段 | 填写合同 |
| --- | --- |
| 文件 | 当前代码文件路径 |
| 行号 | 当前工作树行号或连续范围；脏工作树须注明基线 |
| 代码行为 | 用中文概括该行实际行为，不写目标行为 |
| 流程图节点 | 对应 Mermaid 节点编号 |
| 行为类型 | 入口拒绝 / 读取 / 写入 / 读回 / 纯计算 / 调用 / 返回 / 输出 / 诊断 / 预编译 |
| 结构变化 | 写明是否及如何改变节点、主信息、关系、索引、特征、状态、动态、因果、需求、任务或方法 |
| 权威性 | 权威事实 / 非权威材料 / 人读诊断 / 临时局部变量 |
| 非成功口径 | 不适用 / 逻辑内返回 / 追根因解决 |
| 纠偏建议 | 仅在与正式依据不一致时写明后续治理方向，不直接下达代码许可 |

### 输入契约与非成功返回

输入契约 / 调用语境表：

| 字段 | 填写合同 |
| --- | --- |
| 入口 | 被审查函数、服务入口或流程节点 |
| 调用方 | 外部、测试、领域服务、仓库入口、线程消息或其它正式来源 |
| 输入字段 | 被判断、写入或传递的关键字段 |
| 输入来源 | 外部材料 / 测试材料 / 候选材料 / 正式流程材料 |
| 上游是否保证有效 | 正式流程进入前是否已经承诺该字段有效 |
| 允许逻辑内返回 | 字段无效时是否有正式依据允许拒绝 |
| 追根因触发条件 | 上游保证有效、内部处理异常或结构承载不一致时的具名根因对象 |
| 结构变化 | 当前分支实际改变的机器结构；无则写无 |
| 验证方式 | 代码读取、只读扫描、测试、构建或人工审查；未执行项明确标注 |

非成功返回二分审查表：

| 字段 | 填写合同 |
| --- | --- |
| 节点 | 流程图节点编号 |
| 代码位置 | 文件与当前工作树行号 |
| 返回条件 | 当前代码判断与结果 |
| 输入语境 | 外部材料 / 测试材料 / 候选材料 / 正式流程契约 |
| 是否上游保证有效 | 是 / 否，并给出依据 |
| 结构是否变化 | 是 / 否，并列明结构 |
| 设计是否允许 | 是 / 否，并回指正式规范或有效设计 |
| 二分口径 | 逻辑内返回 / 追根因解决 |
| 追根因对象或逻辑返回理由 | 具名根因对象或允许返回依据 |
| 计划动作 | 修现状图 / 修施工图 / 修详细设计 / 修代码 / 补规范或补契约 |

偏差清单至少包含：

```text
编号 | 当前代码事实 | 正式规范 / 有效流程图 / 详细设计 / 计划口径 | 偏差类型 | 唯一纠偏归属 | 是否需要代码计划
```

## 验证与停止

1. 核对两类图标识和元数据完整，现状事实与施工目标没有混写。
2. 核对逐行映射无非空有效行遗漏，全部图节点、表项和证据路径可互相追溯。
3. 核对 Markdown / HTML 文件成对且 Mermaid 主体一致，偏差均有唯一治理归属。
4. 运行仓库现行文档治理检查：`git diff --check` 和 `python .\tools\check_specs.py --strict`。只使用当前仓库实际存在且由现行规则要求的工具，不虚构专项脚本。

出现角色不符、写入所有权重叠、目标代码在扫描期间变化、当前事实无法冻结、逐行覆盖不完整、正式依据缺失或冲突、现状与施工混写、偏差无唯一归属或验证失败时，停止受影响切片并返回具名证据，不补造规范、详细设计、计划、接口或完成事实。

本技能产物只证明当前代码映射、设计审查和纠偏路由已经形成；不证明代码已修改、构建通过、计划已登记或派发、变更已集成，或目标能力已经闭环。
